# Vite 环境变量文件、Mode 模式与 NODE_ENV

## 概述

Vite 通过 **环境变量文件（`.env` 文件）**、**Mode 模式** 和 **`NODE_ENV`** 三者协同工作，为不同环境（开发、测试、预发布、生产等）提供灵活的配置管理。理解它们之间的关系，是掌握 Vite 工程化配置的关键。

---

## 一、环境变量文件（.env 文件）

### 1.1 什么是环境变量文件

环境变量文件是以 `.env` 为名称的文件，存放在项目根目录（或 `envDir` 指定的目录）中，用于定义在不同环境下需要使用的变量。

### 1.2 文件类型与命名规则

Vite 支持以下几种环境变量文件：

| 文件名 | 加载时机 | 优先级 | 是否提交 Git | 说明 |
|--------|----------|--------|-------------|------|
| `.env` | 所有模式 | 最低（4） | ✅ 是 | 通用默认配置 |
| `.env.local` | 所有模式 | 较高（2） | ❌ 否 | 本地覆盖，含敏感信息 |
| `.env.[mode]` | 指定模式 | 较低（3） | ✅ 是 | 模式专属配置 |
| `.env.[mode].local` | 指定模式 | 最高（1） | ❌ 否 | 模式专属本地覆盖 |

### 1.3 加载优先级

当多个文件同时存在时，优先级从高到低：

```
.env.[mode].local    ← 优先级最高（1）
.env.local           ← 优先级（2）
.env.[mode]          ← 优先级（3）
.env                 ← 优先级最低（4）
```

**规则**：高优先级文件中的同名变量会覆盖低优先级文件中的值。

### 1.4 加载机制详解

Vite 在启动时会根据当前的 `mode` 加载对应文件。以 `mode = production` 为例：

```
加载顺序：
1. .env                    → 始终加载
2. .env.local              → 始终加载
3. .env.production         → 因为 mode = production
4. .env.production.local   → 因为 mode = production
```

最终合并时，后加载的文件中同名变量覆盖先加载的：

```javascript
// 合并结果（伪代码）
const env = {
  ...loadFile('.env'),
  ...loadFile('.env.local'),
  ...loadFile('.env.production'),
  ...loadFile('.env.production.local')
}
```

### 1.5 环境变量文件的语法

```bash
# .env 文件使用简单的 KEY=VALUE 格式

# ✅ 注释以 # 开头
VITE_APP_TITLE=My Application

# ✅ 字符串值（不需要引号）
VITE_API_URL=https://api.example.com

# ✅ 使用双引号包裹（值中包含空格时）
VITE_APP_NAME="My Cool App"

# ✅ 使用单引号包裹
VITE_APP_DESC='A wonderful application'

# ✅ 多行值（使用双引号 + 换行）
VITE_MULTILINE="line1
line2
line3"

# ✅ 空值
VITE_EMPTY=

# ✅ 值中包含引号（使用另一种引号包裹，或转义）
VITE_MESSAGE="It's a test"
VITE_QUOTE='He said "hello"'

# ❌ 不能在值后面加注释
VITE_HOST=localhost # this is a comment  → 整个 "localhost # this is a comment" 会被当作值

# ❌ 不能使用 shell 变量替换
VITE_PATH=$HOME/app  → 不会被解析

# ❌ 不支持命令替换
VITE_TIME=$(date)    → 不会被解析
```

### 1.6 客户端暴露规则

> **核心规则：只有以 `VITE_` 开头的变量才会暴露给客户端代码。**

```bash
# .env.production

# ✅ 以 VITE_ 开头 → 暴露给客户端
VITE_API_URL=https://api.example.com
VITE_APP_TITLE=My App
VITE_ENABLE_ANALYTICS=true

# ❌ 不以 VITE_ 开头 → 仅服务端可访问，客户端无法获取
DATABASE_URL=mongodb://localhost:27017/mydb
SECRET_KEY=my-secret-key
API_TOKEN=abc123
```

在客户端代码中访问：

```javascript
// src/main.js
console.log(import.meta.env.VITE_API_URL)        // → "https://api.example.com"
console.log(import.meta.env.VITE_APP_TITLE)       // → "My App"
console.log(import.meta.env.DATABASE_URL)          // → undefined（不可访问）
```

> 变量前缀可通过 `envPrefix` 配置项自定义，详见 [envPrefix-环境变量前缀.md](./envPrefix-环境变量前缀.md)。

### 1.7 典型项目结构

```
project/
├── .env                        # 所有环境共享的默认变量
├── .env.local                  # 本地开发覆盖（不提交 Git）
├── .env.development            # 开发环境变量
├── .env.development.local      # 开发环境本地覆盖（不提交 Git）
├── .env.staging                # 预发布环境变量
├── .env.staging.local          # 预发布本地覆盖（不提交 Git）
├── .env.production             # 生产环境变量
├── .env.production.local       # 生产环境本地覆盖（不提交 Git）
├── src/
├── vite.config.js
└── package.json
```

### 1.8 各文件典型内容

```bash
# .env — 所有环境共享
VITE_APP_TITLE=My Application
VITE_APP_VERSION=1.0.0
```

```bash
# .env.development — 开发环境
VITE_API_URL=http://localhost:3000

```

```bash
# .env.staging — 预发布环境
VITE_API_URL=https://staging-api.example.com
VITE_ENABLE_MOCK=false
VITE_ENABLE_DEVTOOLS=false
```

```bash
# .env.production — 生产环境

VITE_ENABLE_MOCK=false
VITE_ENABLE_DEVTOOLS=false
VITE_ENABLE_ANALYTICS=true
```

```bash
# .env.local — 本地覆盖（开发者各自维护）

```

### 1.9 `import.meta.env` 内置变量

Vite 会自动注入以下内置变量，无需手动定义：

| 变量 | 类型 | 说明 |
|------|------|------|
| `import.meta.env.MODE` | `string` | 当前应用的 [mode](#二mode-模式) |
| `import.meta.env.BASE_URL` | `string` | 部署应用时的基本 URL，由 `base` 配置项决定 |
| `import.meta.env.PROD` | `boolean` | 当前是否为生产环境（`NODE_ENV === 'production'`） |
| `import.meta.env.DEV` | `boolean` | 当前是否为开发环境（`NODE_ENV !== 'production'`） |
| `import.meta.env.SSR` | `boolean` | 当前是否运行在 Server-Side Rendering 环境 |

```javascript
// 在任意客户端代码中使用
if (import.meta.env.DEV) {
  console.log('当前为开发环境')
}

if (import.meta.env.PROD) {
  console.log('当前为生产环境')
}

console.log('运行模式:', import.meta.env.MODE)       // 例如 "development"
console.log('基础路径:', import.meta.env.BASE_URL)   // 例如 "/"
```

## 二、Mode 模式

### 2.1 什么是 Mode

Mode（模式）是 Vite 用来区分不同运行环境的核心概念。它决定了加载哪些 `.env.[mode]` 文件，以及 `import.meta.env.MODE` 的值。

### 2.2 默认模式

| 命令 | 默认 mode | 说明 |
|------|-----------|------|
| `vite`（即 `vite dev`） | `development` | 开发服务器 |
| `vite build` | `production` | 生产构建 |
| `vite preview` | `production` | 预览构建产物 |

```bash
# 以下命令使用 mode = development
npx vite
npx vite dev

# 以下命令使用 mode = production
npx vite build
npx vite preview
```

### 2.3 自定义模式

通过 `--mode` 参数指定：

```bash
# 使用 staging 模式启动开发服务器
npx vite --mode staging

# 使用 staging 模式构建
npx vite build --mode staging

# 使用 test 模式构建
npx vite build --mode test
```

使用自定义模式时，Vite 会加载对应的 `.env.staging` 或 `.env.test` 文件。

### 2.4 在配置文件中设置 mode

```javascript
// vite.config.js
import { defineConfig } from 'vite'

export default defineConfig({
  // 固定 mode
  mode: 'development'
})
```

更常见的做法是根据环境动态设置：

```javascript
// vite.config.js
import { defineConfig } from 'vite'

export default defineConfig(({ mode }) => {
  return {
    // mode 在此处已经被确定（来自 --mode 参数或默认值）
    // 可以根据 mode 做条件配置
    server: {
      port: mode === 'development' ? 3000 : 8080
    }
  }
})
```

### 2.5 Mode 与环境变量文件的关系

```
mode = 'development'
  → 加载 .env
  → 加载 .env.local
  → 加载 .env.development
  → 加载 .env.development.local

mode = 'production'
  → 加载 .env
  → 加载 .env.local
  → 加载 .env.production
  → 加载 .env.production.local

mode = 'staging'
  → 加载 .env
  → 加载 .env.local
  → 加载 .env.staging
  → 加载 .env.staging.local
```

### 2.6 `package.json` 中配置常用模式

```json
{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "build:staging": "vite build --mode staging",
    "build:test": "vite build --mode test",
    "preview": "vite preview",
    "preview:staging": "vite preview --mode staging"
  }
}
```

### 2.7 使用 `loadEnv` 手动加载环境变量

在 `vite.config.js` 中，可以使用 `loadEnv` 函数手动加载指定模式的环境变量：

```javascript
// vite.config.js
import { defineConfig, loadEnv } from 'vite'

export default defineConfig(({ mode }) => {
  // 加载当前 mode 对应的环境变量
  // 第二个参数指定 envDir（默认为项目根目录）
  const env = loadEnv(mode, process.cwd())

  console.log(env.VITE_API_URL) // 可以访问所有 VITE_ 前缀的变量

  return {
    // 使用环境变量做配置
    server: {
      proxy: {
        '/api': {
          target: env.VITE_API_URL,
          changeOrigin: true
        }
      }
    }
  }
})
```

`loadEnv` 签名：

```typescript
function loadEnv(
  mode: string,         // 模式名称
  envDir: string,       // 环境变量文件目录
  prefixes?: string | string[]  // 变量前缀，默认 'VITE_'
): Record<string, string>
```

```javascript
// 加载所有前缀的变量（包括非 VITE_ 开头的）
const env = loadEnv(mode, process.cwd(), '')

// 加载多种前缀的变量
const env = loadEnv(mode, process.cwd(), ['VITE_', 'CUSTOM_'])
```

## 三、NODE_ENV

### 3.1 什么是 NODE_ENV

`NODE_ENV` 是 Node.js 生态系统中的一个特殊环境变量，用于标识当前的运行时环境。它的值影响 Vite 的构建优化行为和开发体验。

### 3.2 NODE_ENV 的自动设置规则

Vite 会在不同命令下自动设置 `NODE_ENV`：

| 命令 | `NODE_ENV` 值 | 说明 |
|------|---------------|------|
| `vite dev` | `'development'` | 开发模式，不压缩代码，保留 source map |
| `vite build` | `'production'` | 生产模式，代码压缩、tree-shaking 等优化 |

### 3.3 NODE_ENV 与 Mode 的区别（重要）

> **这是最容易混淆的概念！**

| 对比项 | `NODE_ENV` | `mode` |
|--------|-----------|--------|
| **作用** | 控制构建行为（压缩、优化） | 决定加载哪些 `.env` 文件 |
| **默认值** | `vite dev` → `development`，`vite build` → `production` | `vite dev` → `development`，`vite build` → `production` |
| **能否自定义** | 不建议手动修改 | 可通过 `--mode` 自定义 |
| **影响范围** | 构建优化、代码压缩 | `.env` 文件加载、`import.meta.env.MODE` |
| **对应内置变量** | `import.meta.env.DEV` / `import.meta.env.PROD` | `import.meta.env.MODE` |

**关键区别图示：**

```
                        ┌─────────────────┐
                        │   vite build     │
                        │  --mode staging  │
                        └────────┬────────┘
                                 │
                 ┌───────────────┴───────────────┐
                 │                               │
          NODE_ENV = 'production'          mode = 'staging'
                 │                               │
    ┌────────────┴────────────┐      ┌──────────┴──────────┐
    │ 控制构建行为：           │      │ 决定加载的文件：      │
    │ • 代码压缩              │      │ • .env               │
    │ • Tree-shaking          │      │ • .env.local         │
    │ • 去除开发代码           │      │ • .env.staging       │
    │ • Source map 关闭       │      │ • .env.staging.local │
    └─────────────────────────┘      └─────────────────────┘
```

### 3.4 在 `.env` 文件中设置 NODE_ENV

```bash
# .env.production
VITE_API_URL=https://api.example.com

# 可以在 .env 文件中设置 NODE_ENV，覆盖默认值
NODE_ENV=production
```

```bash
# .env.development
VITE_API_URL=http://localhost:3000
NODE_ENV=development
```

> **注意**：`NODE_ENV` 在 `.env` 文件中设置时，不应该加 `VITE_` 前缀。它是 Vite 内部识别的特殊变量，即使不加前缀也会被读取。

### 3.5 NODE_ENV 对构建的影响

```javascript
// 当 NODE_ENV = 'production' 时（vite build）
// Vite 自动启用以下优化：
{
  build: {
    minify: 'esbuild',          // 代码压缩
    sourcemap: false,            // 不生成 source map（默认）
    cssCodeSplit: true,          // CSS 代码分割
    rollupOptions: {
      // tree-shaking 由 Rollup 自动处理
    }
  }
}

// 当 NODE_ENV = 'development' 时（vite dev）
// Vite 行为：
{
  build: {
    minify: false,               // 不压缩
    sourcemap: true,              // 生成 source map
  }
}
```

### 3.6 在代码中使用 NODE_ENV

通过 `import.meta.env.DEV` 和 `import.meta.env.PROD` 访问：

```javascript
// src/utils/logger.js

// 开发环境打印日志，生产环境静默
export function logger(message, ...args) {
  if (import.meta.env.DEV) {
    console.log(`[DEV] ${message}`, ...args)
  }
}

// 仅在生产环境执行
if (import.meta.env.PROD) {
  // 初始化分析工具
  initAnalytics()
}
```

> **注意**：`import.meta.env.DEV` 和 `import.meta.env.PROD` 的值由 `NODE_ENV` 决定，而非 `mode`。

```javascript
// 当 vite build --mode staging 时：
import.meta.env.MODE   // → 'staging'
import.meta.env.PROD   // → true（因为 NODE_ENV = 'production'）
import.meta.env.DEV    // → false

// 也就是说：虽然是 staging 模式，但构建行为等同于生产模式
```

## 四、三者协作关系

### 4.1 完整工作流程

```
┌──────────────────────────────────────────────────────────┐
│                    执行 vite build --mode staging          │
└─────────────────────┬────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────────────┐
│ Step 1: 确定 mode = 'staging'                            │
│   来源: --mode 参数（或默认 production）                    │
└─────────────────────┬────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────────────┐
│ Step 2: 确定 NODE_ENV = 'production'                     │
│   来源: .env 文件中的设置 或 默认值（build → production）  │
└─────────────────────┬────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────────────┐
│ Step 3: 加载环境变量文件（按优先级从低到高）                │
│   .env → .env.local → .env.staging → .env.staging.local  │
└─────────────────────┬────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────────────┐
│ Step 4: 合并所有变量，高优先级覆盖低优先级                  │
└─────────────────────┬────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────────────┐
│ Step 5: 根据 NODE_ENV 决定构建优化策略                     │
│   NODE_ENV=production → 压缩、tree-shaking 等             │
└─────────────────────┬────────────────────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────────────────────┐
│ Step 6: 注入 import.meta.env                              │
│   .MODE = 'staging'                                      │
│   .PROD = true                                           │
│   .DEV = false                                           │
│   .BASE_URL = '/'                                        │
│   .VITE_xxx = 从 .env.* 文件中读取的值                     │
└──────────────────────────────────────────────────────────┘
```

### 4.2 关系总结表

| 场景 | 命令 | `mode` | `NODE_ENV` | 加载的 `.env` 文件 | `DEV` | `PROD` |
|------|------|--------|-----------|-------------------|-------|--------|
| 本地开发 | `vite` | `development` | `development` | `.env` + `.env.local` + `.env.development` + `.env.development.local` | `true` | `false` |
| 生产构建 | `vite build` | `production` | `production` | `.env` + `.env.local` + `.env.production` + `.env.production.local` | `false` | `true` |
| 预发布构建 | `vite build --mode staging` | `staging` | `production` | `.env` + `.env.local` + `.env.staging` + `.env.staging.local` | `false` | `true` |
| 测试环境构建 | `vite build --mode test` | `test` | `production` | `.env` + `.env.local` + `.env.test` + `.env.test.local` | `false` | `true` |
| 测试运行（vitest） | `vitest` | `test` | `development` | `.env` + `.env.local` + `.env.test` + `.env.test.local` | `true` | `false` |

```
├── src/
│   ├── utils/
│   │   └── env.js                # 环境变量工具
│   ├── api/
│   │   └── request.js            # API 请求配置
```

#### 环境变量工具函数

```javascript
// src/utils/env.js

/**
 * 获取环境变量的类型安全工具
 * 封装 import.meta.env 的访问逻辑
 */
export const env = {
  /** 当前模式 */
  get mode() {
    return import.meta.env.MODE
  },
  /** API 地址 */
  get apiUrl() {
    return import.meta.env.VITE_API_URL || ''
  },
  /** API 前缀 */
  get apiPrefix() {
    return import.meta.env.VITE_API_PREFIX || '/api'
  },
  /** 是否启用 Mock */
  get enableMock() {
    return import.meta.env.VITE_ENABLE_MOCK === 'true'
  },
  /** 应用标题 */
  get appTitle() {
    return import.meta.env.VITE_APP_TITLE || 'App'
  },
  /** 是否为开发环境 */
  get isDev() {
    return import.meta.env.DEV
  },
  /** 是否为生产环境 */
  get isProd() {
    return import.meta.env.PROD
  }
}
```

#### API 请求配置

```javascript
// src/api/request.js
import axios from 'axios'
import { env } from '../utils/env'

const request = axios.create({
  baseURL: `${env.apiUrl}${env.apiPrefix}`,
  timeout: env.isDev ? 60000 : 10000
})

// 开发环境打印请求日志
if (env.isDev) {
  request.interceptors.request.use((config) => {
    console.log(`[API Request] ${config.method?.toUpperCase()} ${config.url}`)
    return config
  })
}

export default request
```

### 5.2 条件编译与代码剔除

Vite 在生产构建时会通过 dead code elimination 自动剔除不可达的代码分支：

```javascript
// src/main.js

// 开发环境专属代码 — 生产构建时会被完全剔除
if (import.meta.env.DEV) {
  // 这些代码不会出现在生产包中

  setupPerformanceMonitor()

  initAnalytics()
}
```

```javascript

initAnalytics()
```

### 5.3 动态配置（vite.config.js 中使用环境变量）

```javascript
// vite.config.js
import { defineConfig, loadEnv } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd())

  return {
    plugins: [vue()],

    // 根据环境变量配置代理
    server: {
      proxy: {
        '/api': {
          target: env.VITE_API_URL,
          changeOrigin: true,
          rewrite: (path) => path.replace(/^\/api/, env.VITE_API_PREFIX || '/api')
        }
      }
    },

    // 生产环境额外配置
    build: {
      // 仅在生产构建时生成 sourcemap
      sourcemap: mode === 'production' ? false : true
    },

    // 定义全局常量
    define: {
      __APP_VERSION__: JSON.stringify(env.VITE_APP_VERSION || '0.0.0')
    }
  }
})
```

## 六、常见问题与注意事项

### 6.1 环境变量值始终是字符串

```javascript
// .env
VITE_ENABLE_FEATURE=true
VITE_PORT=3000
VITE_DEBUG=false

// ❌ 错误：直接比较 boolean
if (import.meta.env.VITE_ENABLE_FEATURE === true) {
  // 永远不会执行！因为 'true' !== true
}

// ✅ 正确：与字符串比较
if (import.meta.env.VITE_ENABLE_FEATURE === 'true') {
  // 正确执行
}

// ✅ 正确：转换为数字
const port = Number(import.meta.env.VITE_PORT)
```

### 6.2 环境变量变更需要重启

```bash
# 修改 .env 文件后，需要重启开发服务器
# Vite 不会热更新环境变量的变更

# 按 Ctrl+C 停止服务器后重新启动
npx vite
```

### 6.3 安全性：不要在前端暴露敏感信息

```bash
# ❌ 危险：敏感信息暴露给客户端
VITE_DB_PASSWORD=my-secret-password
VITE_API_SECRET=sk-xxxxxxxxxxxx

# ✅ 正确：敏感信息只在服务端使用

# ✅ 正确：前端只保留公开信息
VITE_API_URL=https://api.example.com
VITE_APP_TITLE=My App
```

> 所有以 `VITE_` 开头的变量都会被打包进客户端代码，任何人都可以在浏览器中看到。

### 6.4 `.env.local` 文件的正确使用

```bash
# .env.local 适合存放开发者个人的配置
# 不同开发者的 .env.local 内容不同，不应提交到 Git

# .gitignore 中应包含：
```

### 6.5 `mode` 和 `NODE_ENV` 不匹配的情况


执行 `vite`（mode = development）时，最终 `VITE_API_URL` 的值为：

```
最终值 = http://192.168.1.100:3000

1. .env               → VITE_API_URL=https://production.example.com
```

## 七、最佳实践总结

### 7.1 文件管理

1. **`.env`** — 放所有环境共享的默认值，提交 Git
2. **`.env.[mode]`** — 放环境专属配置，提交 Git
3. **`.env.local`** — 放个人本地覆盖，加入 `.gitignore`
4. **`.env.[mode].local`** — 放环境专属的本地覆盖，加入 `.gitignore`

```bash
VITE_1=https://...            # 数字开头
```

## 八、在 `.env` 文件中错误设置 NODE_ENV 会怎样

### 8.1 核心结论

> **Vite 会强制覆盖 `.env` 文件中的 `NODE_ENV`**，所以错误配置不会导致灾难性后果。

```
vite dev   → NODE_ENV 强制为 'development'（无论 .env 文件怎么写）
vite build → NODE_ENV 强制为 'production' （无论 .env 文件怎么写）
```

### 8.2 场景分析

#### 场景 1：`.env.development` 中写了 `NODE_ENV=production`

```bash
# .env.development
VITE_API_URL=http://localhost:3000
NODE_ENV=production    ← 写了也白写
```

运行 `vite dev`：

```
你期望的：NODE_ENV = production（按 .env 文件的值）
实际结果：NODE_ENV = development（Vite 强制覆盖）

import.meta.env.MODE  → 'development'  ✅
import.meta.env.DEV   → true           ✅（没有变成 false）
import.meta.env.PROD  → false          ✅（没有变成 true）
```

**结论**：开发服务器正常运行，热更新正常，调试代码正常执行。`.env.development` 中的 `NODE_ENV=production` 被忽略。

#### 场景 2：`.env.production` 中写了 `NODE_ENV=development`

```bash
# .env.production
VITE_API_URL=https://api.example.com
NODE_ENV=development    ← 写了也白写
```

运行 `vite build`：

```
你期望的：NODE_ENV = development（按 .env 文件的值）
实际结果：NODE_ENV = production（Vite 强制覆盖）

import.meta.env.MODE  → 'production'  ✅
import.meta.env.DEV   → false         ✅（没有变成 true）
import.meta.env.PROD  → true          ✅（没有变成 false）
```

**结论**：生产构建正常运行，代码正常压缩，tree-shaking 正常工作，调试代码正常被剔除。

### 8.3 陷阱：`loadEnv()` 会原样读取 `.env` 文件中的值

虽然 Vite 内部行为不受影响，但 **`loadEnv()` 函数会原样读取 `.env` 文件中的 NODE_ENV 值**，这在 `vite.config.js` 中可能造成误判：

```javascript
// vite.config.js
import { defineConfig, loadEnv } from 'vite'

export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd(), '')

  // ⚠️ 这里读到的 NODE_ENV 是 .env 文件中的值！
  // 如果 .env.production 中写了 NODE_ENV=development
  // 这里 env.NODE_ENV 就是 'development'
  console.log(env.NODE_ENV) // → 'development'（来自 .env 文件的原始值，不是 Vite 强制设置的值）

  return {}
})
```

如果基于 `loadEnv` 读取的 `NODE_ENV` 做判断，**就会出问题**：

```javascript
// ❌ 危险：基于 loadEnv 的 NODE_ENV 做决策
export default defineConfig(({ mode }) => {
  const env = loadEnv(mode, process.cwd(), '')

  // 如果 .env.production 中 NODE_ENV=development
  // mode = 'production' 但 env.NODE_ENV = 'development'
  if (env.NODE_ENV === 'production') {
    // 这个分支不会执行！因为读到的是 .env 文件中的 'development'
    console.log('生产配置')
  }

  return {}
})
```

正确做法：

```javascript
// ✅ 正确：使用 mode 或 command 判断
export default defineConfig(({ mode, command }) => {
  // 方式 1：用 command 判断
  if (command === 'build') {
    // 构建时逻辑
  }

  // 方式 2：用 mode 判断
  if (mode === 'production') {
    // 生产环境逻辑
  }

  return {}
})
```

### 8.4 影响范围总结

| 对比项 | `.env` 文件中的 `NODE_ENV` | Vite 强制设置的 `NODE_ENV` |
|--------|--------------------------|---------------------------|
| 影响 `import.meta.env.DEV/PROD` | ❌ 不影响 | ✅ 决定性 |
| 影响构建优化（压缩/Tree-shaking） | ❌ 不影响 | ✅ 决定性 |
| 影响代码剔除（dead code elimination） | ❌ 不影响 | ✅ 决定性 |
| 影响开发服务器行为（HMR 等） | ❌ 不影响 | ✅ 决定性 |
| `loadEnv()` 读取的值 | ✅ 读到的是文件中的值 | — |

### 8.5 最佳实践

> **永远不要在 `.env` 文件中手动设置 `NODE_ENV`**。Vite 会根据命令（`vite dev` / `vite build`）自动设置正确的值，手动设置既无效又容易在 `vite.config.js` 中通过 `loadEnv` 造成误判。

```bash
# ❌ 不要这样做
# .env.development
NODE_ENV=production

# .env.production
NODE_ENV=development

# ✅ 不要写 NODE_ENV，让 Vite 自动管理
# .env.development
VITE_API_URL=http://localhost:3000

# .env.production
VITE_API_URL=https://api.example.com
```
