# 微前端架构

## 一、什么是微前端

### 1.1 基本定义

**微前端** = **微**（微小、独立）+ **前端**

简单来说，就是将一个**大型前端应用**拆分成**多个小型、独立的前端应用**，每个应用可以**独立开发、独立部署、独立运行**，最终组合成一个完整的应用。

这是一种将**微服务架构**思想延伸到**前端开发**的架构模式。

### 1.2 形象理解

想象一个大型购物网站：

```
传统单体前端应用                     微前端架构
─────────────────────────────────────────────────────────
一个大型工厂                          一个商业综合体
├── 所有部门在同一栋楼               ├── 独立店铺（各自管理）
│   ├── 任何改动都要重新部署整个      │   ├── 服装店（独立装修）
│   ├── 一个部门出错影响全部          │   ├── 餐厅（独立营业）
│   └── 难以引入新技术                │   └── 电影院（独立排片）
                                      │
一个页面包含所有功能                  ↓
用户访问一个巨大应用                  用户访问统一入口
                                      后台是多个独立应用组合
```

### 1.3 架构对比

```
┌─────────────────── 传统单体前端 ───────────────────┐
│                                                           │
│  Shopping Website                                        │
│  ├── Header/                                             │
│  ├── Product List                                        │
│  ├── Shopping Cart                                       │
│  ├── User Profile                                        │
│  ├── Payment                                             │
│  └── Footer                                              │
│                                                           │
│  一个项目 → 一个团队 → 一套技术栈 → 一起部署              │
└───────────────────────────────────────────────────────────┘

┌─────────────────── 微前端架构 ───────────────────┐
│                                                           │
│  Main App（主应用/基座）                               │
│  ├── Header（主应用提供）                              │
│  ├── Product List（子应用 A）← Vue2 开发              │
│  ├── Shopping Cart（子应用 B）← React 开发            │
│  ├── User Profile（子应用 C）← Angular 开发          │
│  ├── Payment（子应用 D）← Vue3 开发                  │
│  └── Footer（主应用提供）                             │
│                                                           │
│  多个项目 → 多个团队 → 多种技术栈 → 独立部署            │
└───────────────────────────────────────────────────────────┘
```

### 1.4 微前端的工作原理

```
┌─────────────────────────────────────────────────────────┐
│                    用户浏览器                            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│              主应用（Main App / Shell）                  │
│  ┌─────────────────────────────────────────────────┐   │
│  │  路由管理                                         │   │
│  │  应用注册                                         │   │
│  │  生命周期管理                                     │   │
│  │  应用间通信                                       │   │
│  └─────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────┤
│         子应用 A          子应用 B         子应用 C      │
│      （Vue2）         （React）        （Angular）       │
│         ↓                 ↓                ↓            │
│    独立开发、独立部署、独立运行                          │
└─────────────────────────────────────────────────────────┘
```

## 二、微前端的作用/优势

### 2.1 技术栈无关

```
场景：公司历史项目使用不同技术栈

传统单体应用：
┌─────────────────────────────────┐
│  必须统一技术栈                  │
│  要么全部 Vue，要么全部 React    │
│  新技术难以引入                  │
└─────────────────────────────────┘

微前端架构：
┌─────────────────────────────────────────────────────────┐
│  主应用（Vue3）                                          │
│  ├── 子应用 A（Vue2）← 老项目，无需重构                 │
│  ├── 子应用 B（React）← 新项目，使用新技术              │
│  ├── 子应用 C（Angular）← 特定团队擅长的技术            │
│  └── 子应用 D（Svelte）← 实验性技术                     │
│                                                          │
│  各个团队可以选择最适合自己的技术栈！                    │
└─────────────────────────────────────────────────────────┘
```

### 2.2 独立开发、独立部署

```
场景：电商网站，多个团队协作

传统单体应用：
┌─────────────────────────────────────────────────────────┐
│  所有代码在一个仓库                                     │
│                                                         │
│  商品团队：我要改商品列表功能                           │
│  └── 需要协调其他团队                                  │
│  └── 需要整体测试                                      │
│  └── 需要整体部署（其他功能也被部署）                  │
│                                                         │
│  购物车团队：我想修个 bug                               │
│  └── 需要等待商品团队完成                              │
│  └── 冒着影响其他功能的风险                            │
└─────────────────────────────────────────────────────────┘

微前端架构：
┌─────────────────────────────────────────────────────────┐
│  主应用：只负责路由和应用集成                           │
│                                                         │
│  商品子应用：                                            │
│  └── 独立仓库，自己开发                                 │
│  └── 完成后独立部署，不影响其他子应用                   │
│  └── 每天可以发布多次                                  │
│                                                         │
│  购物车子应用：                                          │
│  └── 独立仓库，自己开发                                 │
│  └── 紧急 bug 立即修复，立即部署                        │
│  └── 与其他团队零耦合                                  │
└─────────────────────────────────────────────────────────┘
```

### 2.3 增量升级

```
场景：老项目需要技术升级

传统方式：
┌─────────────────────────────────────────────────────────┐
│  jQuery 项目 → Vue 项目                                 │
│                                                         │
│  ❌ 停止新功能开发                                      │
│  ❌ 花费半年重构整个项目                                │
│  ❌ 风险极高，一次性替换                                │
│  ❌ 可能引入新 bug                                      │
└─────────────────────────────────────────────────────────┘

微前端方式：
┌─────────────────────────────────────────────────────────┐
│  jQuery 项目                                            │
│  ↓                                                      │
│  第一步：接入微前端基座                                 │
│  ├── 将现有功能作为子应用 A                             │
│  └── 新功能用 Vue 开发，作为子应用 B                    │
│  ↓                                                      │
│  第二步：逐步迁移                                       │
│  ├── 将用户中心迁移到 Vue（子应用 C）                   │
│  ├── 其他功能保持不变                                  │
│  └── 用户无感知                                        │
│  ↓                                                      │
│  第三步：完全迁移                                       │
│  └── 所有功能都迁移到 Vue                               │
│                                                         │
│  ✅ 新功能继续开发                                      │
│  ✅ 逐步替换，风险可控                                  │
│  ✅ 出问题只影响一部分功能                              │
└─────────────────────────────────────────────────────────┘
```

### 2.4 团队自治

```
传统单体应用团队结构：
┌─────────────────────────────────────────────────────────┐
│  前端团队（20人）                                        │
│  ├── 所有代码在一个项目                                 │
│  ├── 需要频繁沟通协调                                   │
│  ├── 代码冲突频繁                                       │
│  └── 发布需要统一安排                                   │
└─────────────────────────────────────────────────────────┘

微前端架构团队结构：
┌─────────────────────────────────────────────────────────┐
│  基座团队（2人） ← 维护主应用                           │
│                                                         │
│  商品团队（5人） ← 负责商品子应用                       │
│  ├── 自己选择技术栈                                     │
│  ├── 自己决定发布时机                                   │
│  └── 对自己负责的模块全权负责                           │
│                                                         │
│  订单团队（4人） ← 负责订单子应用                       │
│  ├── 独立开发                                           │
│  ├── 独立测试                                           │
│  └── 独立部署                                           │
│                                                         │
│  支付团队（3人） ← 负责支付子应用                       │
│  ├── 有自己的代码规范                                   │
│  └── 有自己的发布流程                                   │
│                                                         │
│  支付团队（6人） ← 负责用户子应用                       │
└─────────────────────────────────────────────────────────┘
```

### 2.5 故障隔离

```
传统单体应用：
┌─────────────────────────────────────────────────────────┐
│  一个模块的 JavaScript 错误                              │
│  ↓                                                      │
│  可能导致整个页面崩溃                                    │
│  ↓                                                      │
│  所有功能都无法使用                                      │
└─────────────────────────────────────────────────────────┘

微前端架构：
┌─────────────────────────────────────────────────────────┐
│  购物车子应用报错                                         │
│  ↓                                                      │
│  只影响购物车功能                                        │
│  ↓                                                      │
│  商品浏览、下单等其他功能正常使用                         │
│  ↓                                                      │
│  用户体验影响最小                                        │
└─────────────────────────────────────────────────────────┘
```

## 三、使用场景

### 3.1 ✅ 适合使用微前端的场景

| 场景 | 说明 | 例子 |
|------|------|------|
| **大型企业应用** | 功能复杂，多团队协作 | ERP、CRM、OA 系统 |
| **渐进式重构** | 老项目需要逐步升级 | jQuery → Vue、Angular → React |
| **跨团队协作** | 不同团队负责不同业务模块 | 电商平台的商品、订单、支付 |
| **技术栈整合** | 需要整合使用不同技术栈的项目 | 收购其他公司的产品 |
| **SaaS 平台** | 需要快速集成第三方应用 | 企业管理平台 |
| **独立部署需求** | 各模块发布频率差异大 | 营销页（日发）+ 核心功能（周发） |

### 3.2 ❌ 不适合使用微前端的场景

| 场景 | 原因 |
|------|------|
| **小型项目** | 架构成本高于收益 |
| **简单的展示网站** | 不需要复杂架构 |
| **单一团队维护** | 增加复杂度，无必要 |
| **应用间需要深度交互** | 微前端隔离反而增加复杂度 |
| **对性能极度敏感** | 多应用加载有一定性能损耗 |

### 3.3 真实案例

```
┌─────────────────────────────────────────────────────┐
│  公司          │  微前端应用                         │
├─────────────────────────────────────────────────────┤
│  阿里巴巴       │  云速搭、语雀、飞猪                 │
│  京东           │  资管系统、物流系统                 │
│  美团           │  美团外卖、酒店旅游                 │
│  字节跳动       │  抖音开放平台、企业级应用           │
│  微软           │  Microsoft 365（Office 在线版）    │
└─────────────────────────────────────────────────────┘
```

## 四、技术方案对比

### 4.1 主流微前端框架对比

| 框架 | 核心原理 | 技术栈限制 | 学习成本 | 成熟度 |
|------|----------|------------|----------|--------|
| **qiankun** | 基于 single-spa，HTML Entry | 基本无限制 | ⭐⭐ 中 | ⭐⭐⭐⭐⭐ 高 |
| **single-spa** | JS Entry，路由劫持 | 需要适配 | ⭐⭐⭐ 中高 | ⭐⭐⭐⭐⭐ 高 |
| **Micro-App** | Web Components，类 WebComponent | 基本无限制 | ⭐ 低 | ⭐⭐⭐ 中 |
| **wujie** | iframe + WebComponent | 基本无限制 | ⭐ 低 | ⭐⭐⭐ 中 |
| **Module Federation** | Webpack 5 原生支持 | 需要模块化 | ⭐⭐ 中 | ⭐⭐⭐⭐ 高 |

### 4.2 各方案特点


## 五、实现步骤 - qiankun 方案

### 5.1 为什么选择 qiankun

```
qiankun 的优势：
┌─────────────────────────────────────────────────────┐
│  1. 基于 single-spa，提供更开箱即用的 API            │
│  2. HTML Entry 方式，接入成本最低                    │
│  3. 技术栈无关，接任何框架都能用                    │
│  4. 样式隔离，避免样式冲突                          │
│  5. JS 沙箱，避免全局变量污染                       │
│  6. 资源预加载，提升用户体验                        │
└─────────────────────────────────────────────────────┘
```

### 5.2 整体架构设计

```
微前端项目结构
├── main-app/              ← 主应用（基座）
│   ├── src/
│   │   ├── App.vue
│   │   ├── main.js
│   │   └── micro-fe.js    ← 微前端配置
│   └── package.json
│
├── sub-app-vue/           ← 子应用 A（Vue3）
│   ├── src/
│   │   ├── App.vue
│   │   └── main.js
│   └── package.json
│
└── sub-app-react/         ← 子应用 B（React）
    ├── src/
    │   ├── App.jsx
    │   └── index.js
    └── package.json
```

### 5.3 创建主应用（基座）

```bash

# 2. 安装 qiankun
pnpm add qiankun

# 3. 安装 Vue Router

```

```typescript
// src/micro-fe.ts
import { registerMicroApps, start } from 'qiankun';

// 注册子应用
registerMicroApps([
  {
    name: 'sub-vue',           // 子应用名称
    entry: '//localhost:5174', // 子应用地址
    container: '#subapp-container', // 挂载容器
    activeRule: '/sub-vue',    // 激活规则（路由）
    props: {
      // 传递给子应用的数据
      token: 'main-app-token',
      userInfo: { name: '张三' }
    }
  },
  {
    name: 'sub-react',
    entry: '//localhost:5175',
    container: '#subapp-container',
    activeRule: '/sub-react',
    props: {
      token: 'main-app-token'
    }
  }
]);

// 启动 qiankun
start({
  sandbox: {
    // 样式隔离
    strictStyleIsolation: false,
    // JS 沙箱
    experimentalStyleIsolation: true
  }
});
```

### 5.4 创建 Vue 子应用

```bash

cd ..

cd sub-app-vue
pnpm create vite@latest . --template vue-ts

# 2. 修改 vite.config.ts
```

```typescript
// vite.config.ts
import { defineConfig } from 'vite';
import vue from '@vitejs/plugin-vue';

export default defineConfig({
  plugins: [vue()],
  server: {
    port: 5174, // 不同的端口
    cors: true  // 允许跨域

  origin: 'http://localhost:5174' // qiankun 需要
});
```

```typescript
// src/main.ts
import { createApp } from 'vue';
import App from './App.vue';
import router from './router';

let app: any;

// ✅ 必须导出生命周期钩子
export async function bootstrap() {
  console.log('Vue 子应用 bootstrap');
```

```typescript
// 方式二：使用全局事件总线
// 创建一个简单的 EventBus
class EventBus {
  private events: Record<string, Function[]> = {};

  on(event: string, callback: Function) {
    if (!this.events[event]) {
      this.events[event] = [];
    }
    this.events[event].push(callback);
  }

  emit(event: string, data: any) {
    if (this.events[event]) {
      this.events[event].forEach(callback => callback(data));
    }
  }

  off(event: string, callback: Function) {
    if (this.events[event]) {
      this.events[event] = this.events[event].filter(cb => cb !== callback);
    }
  }
}

export const eventBus = new EventBus();
```

## 八、常见问题

### 8.1 常见问题及解决方案

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **子应用样式丢失** | 沙箱隔离导致 | 关闭严格样式隔离或使用 CSS Modules |
| **路由跳转不生效** | 路由模式不一致 | 统一使用 history 模式，配置 base |
| **子应用加载失败** | 跨域或端口问题 | 子应用配置 CORS，检查 entry 地址 |
| **全局变量污染** | 子应用依赖全局变量 | 使用沙箱，或通过 props 传递 |
| **性能问题** | 加载多个应用 | 使用懒加载、预加载策略 |
| **开发环境热更新失效** | qiankun 限制 | 子应用独立开发，不用 qiankun |

## 九、总结

### 9.1 微前端核心要点

```
┌─────────────────────────────────────────────────────┐
│  微前端的优势                                        │
├─────────────────────────────────────────────────────┤
│  ✅ 技术栈无关，各团队自由选择                       │
│  ✅ 独立开发、独立部署、独立运行                     │
│  ✅ 增量升级，老项目平滑迁移                         │
│  ✅ 团队自治，提高开发效率                           │
│  ✅ 故障隔离，降低风险                               │
│  ✅ 按需加载，提升性能                               │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  微前端的挑战                                        │
├─────────────────────────────────────────────────────┤
│  ❌ 增加架构复杂度                                   │
│  ❌ 应用间通信相对复杂                               │
│  │ 性能有一定损耗                                   │
│  │ 调试相对困难                                      │
│  │ 需要额外的框架学习成本                            │
└─────────────────────────────────────────────────────┘
```

### 9.2 何时选择微前端

```
选择微前端如果你需要：

✅ 大型应用，多团队协作
✅ 老项目需要技术升级
✅ 不同模块使用不同技术栈
✅ 各模块独立部署需求强
✅ 需要整合多个独立系统

不适合微前端：

❌ 小型项目，简单应用
❌ 单一团队维护
❌ 应用间需要深度交互
❌ 对性能极度敏感
```
