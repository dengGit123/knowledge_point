# 前端工程 Monorepo 架构实践

## 一、什么是前端 Monorepo

### 1.1 基本定义

**前端 Monorepo** 是将多个前端相关项目（应用、组件库、工具包等）放在同一个 Git 仓库中进行管理的架构模式。

### 1.2 核心理念

```

│                     传统前端项目架构                             │
├─────────────────────────────────────────────────────────────────┤

│   ├── src/              ├── src/              ├── src/             │
```

### 2.2 组件库的高效开发

```
┌─────────────────────────────────────────────────────────────────┐
│  组件库开发的典型场景                                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  场景：开发一个 Button 组件，需要在多个应用中使用                │
│                                                                 │
│  Polyrepo 方式：                                                │
│  ┌───────────────────────────────────────────────────────┐      │
│  │ 1. 在 components 仓库开发 Button 组件                 │      │
│  │ 2. 执行 npm publish 发布到 npm                         │      │
│  │ 3. 等待构建完成（2-5 分钟）                             │      │
│  │ 4. 在 web 应用中执行 npm install @company/ui           │      │
│  │ 5. 在 admin 应用中执行 npm install @company/ui         │      │
│  │ 6. 重启两个应用的开发服务器                             │      │
│  │ 7. 发现组件有 bug                                       │      │
│  │ 8. 重复以上步骤...                                      │      │
│  └───────────────────────────────────────────────────────┘      │
│                                                                 │
│  Monorepo 方式：                                                │
│  ┌───────────────────────────────────────────────────────┐      │
│  │ 1. 在 packages/ui/src/Button.vue 修改组件             │      │
│  │ 2. 保存文件，热更新立即生效                             │      │
│  │ 3. web 和 admin 应用实时看到更新                        │      │
│  │ 4. 发现 bug，立即修改，实时预览                         │      │
│  │ 5. 开发完成后，执行 pnpm build 统一构建                │      │
│  │ 6. 执行 pnpm publish 发布                              │      │
│  └───────────────────────────────────────────────────────┘      │
│                                                                 │
│  对比结果：                                                    │
│  • 开发效率提升 10 倍+                                         │
│  • 即时反馈，调试更方便                                        │
│  • 减少发布次数，降低出错风险                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4 原子提交与代码审查

```
┌─────────────────────────────────────────────────────────────────┐
│  场景：修改组件库 API，影响所有使用该组件的应用                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Polyrepo 方式：                                                │
│  1. 在 components 仓库修改 API                                  │
│  2. 发布 v2.0.0 到 npm                                          │
│  3. 在 web 仓库更新依赖，修改适配代码                            │
│  4. 在 admin 仓库更新依赖，修改适配代码                          │
│  5. 分别提交 PR，分别审查                                       │
│  6. 分别测试，分别发布                                         │
│  7. 任何一步出错，版本就不同步                                  │
│                                                                 │
│  Monorepo 方式：                                                │
│  1. 一次 PR 包含所有改动：                                      │
│     ├── packages/ui/              组件 API 修改                  │
│     ├── apps/web/src/             Web 应用适配                  │
│     └── apps/admin/src/           Admin 应用适配                │
│  2. 一次 Code Review，审查所有关联改动                         │
│  3. 一次测试，确保所有应用正常                                 │
│  4. 合并后所有应用同步更新                                     │
│                                                                 │
│  优势：                                                        │
│  • 避免版本不同步导致的 bug                                     │
│  • 一次审查，看到完整的改动范围                                 │
│  • 降低协调成本                                                │
│  • 可以使用 Changeset 自动生成 CHANGELOG                       │
└─────────────────────────────────────────────────────────────────┘
```

## 三、前端 Monorepo 适用场景

### 3.1 ✅ 适合场景

| 场景 | 说明 | 典型案例 |
|------|------|----------|
| **组件库 + 应用** | 同时维护组件库和使用组件的应用 | Ant Design + Ant Design Pro |
| **多应用共享代码** | 多个前端应用共享业务组件、Hooks、工具 | 电商前台 + 管理后台 + 商家后台 |
| **微前端架构** | 基座 + 多个子应用 | qiankun 主应用 + 子应用 |
| **全栈项目** | Next.js/Nuxt.js 项目，前后端一体 | Next.js SaaS 项目 |
| **npm 包开发** | 开发多个相关的 npm 包 | VueUse、@tanstack/query |
| **设计系统** | 组件库 + 文档站点 + Storybook | Element Plus、Naive UI |
| **多租户系统** | 不同租户的应用大部分代码相同 | SaaS 平台 + 白标产品 |
| **文档站 + 库** | 库代码 + 文档代码 + 示例代码 | Vite 文档站、Vitest 文档 |

### 3.2 ❌ 不适合场景

| 场景 | 原因 |
|------|------|
| **完全无关的项目** | 没有代码共享，Monorepo 反而增加复杂度 |
| **超大型仓库** | Git clone、IDE 性能、构建时间会成为瓶颈 |
| **不同团队/公司** | 权限隔离困难，协作成本高 |
| **发布周期差异大** | 一个项目每天发版，另一个一年发一次，会互相干扰 |

## 四、技术选型

### 4.1 包管理器选择

| 特性 | pnpm | npm | Yarn |
|------|------|-----|------|
| **Workspace 支持** | ✅ 原生支持 | ✅ 原生支持 | ✅ 原生支持 |
| **磁盘空间** | ⭐⭐⭐⭐⭐ 硬链接，最节省 | ⭐⭐ 每个项目独立 | ⭐⭐⭐ 软链接 |
| **安装速度** | ⭐⭐⭐⭐⭐ 最快 | ⭐⭐ 较慢 | ⭐⭐⭐⭐ 较快 |
| **依赖提升** | 可配置 | 默认提升 | 可配置 |
| **Monorepo 支持** | ⭐⭐⭐⭐⭐ 最佳 | ⭐⭐⭐ 一般 | ⭐⭐⭐⭐ 较好 |
| **社区活跃度** | ⭐⭐⭐⭐⭐ 很高 | ⭐⭐⭐⭐⭐ 很高 | ⭐⭐⭐⭐ 较高 |

**推荐：pnpm**

```bash
# pnpm 的优势
# 1. 节省磁盘空间
node_modules/
├── .pnpm/
│   └── react@18.2.0/
│       └── node_modules/
│           └── react/          ← 只存储一份
├── react → .pnpm/react@18.2.0/node_modules/react
└── app1/node_modules/react → .pnpm/react@18.2.0/node_modules/react
    app2/node_modules/react → .pnpm/react@18.2.0/node_modules/react

# 2. 更严格的依赖管理
# 只有 package.json 中声明的依赖才能被访问
# 防止"幻影依赖"问题

# 3. 更快的安装速度
# 硬链接机制减少文件复制
```

### 4.2 构建工具选择

| 工具 | 定位 | 适用场景 |
|------|------|----------|
| **Turborepo** | 构建加速 + 缓存 | 大型项目，追求构建速度 |
| **Nx** | 全功能工程化平台 | 复杂项目，需要完整工具链 |
| **Lerna** | 版本管理 | 老项目维护 |
| **无（只用 pnpm）** | 简单场景 | 小型项目 |

**推荐方案：**

```
┌─────────────────────────────────────────────────────────────────┐
│  项目规模                  │  推荐工具                          │
├─────────────────────────────────────────────────────────────────┤
│  小型（< 5 个包）            │  pnpm workspace                   │
│  中型（5-20 个包）          │  pnpm + Turborepo                 │
│  大型（20+ 个包）           │  pnpm + Turborepo / Nx            │
└─────────────────────────────────────────────────────────────────┘
```

```bash
// 日期工具
export function formatDate(date: Date, format = 'YYYY-MM-DD'): string {
  const year = date.getFullYear();
  const month = String(date.getMonth() + 1).padStart(2, '0');
  const day = String(date.getDate()).padStart(2, '0');

  return format
    .replace('YYYY', String(year))
    .replace('MM', month)
    .replace('DD', day);
}

// 数字工具

  return num.toLocaleString('zh-CN', {

// 防抖
export function debounce<T extends (...args: any[]) => any>(
  fn: T,
  delay: number
): (...args: Parameters<T>) => void {
  let timer: ReturnType<typeof setTimeout> | null = null;
  return function (this: any, ...args: Parameters<T>) {
    if (timer) clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), delay);
  };
}

// 节流
export function throttle<T extends (...args: any[]) => any>(
  fn: T,
  delay: number
): (...args: Parameters<T>) => void {
  let lastTime = 0;
  return function (this: any, ...args: Parameters<T>) {
    const now = Date.now();
    if (now - lastTime >= delay) {
      lastTime = now;
      fn.apply(this, args);
    }
  };
}
```

## 十、常见问题与解决方案

### 10.1 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **workspace:* 引用失败** | 包名不正确 | 检查 package.json 的 name 字段 |
| **TypeScript 找不到模块** | paths 配置问题 | 配置 tsconfig.json 的 paths 和 references |
| **构建顺序错误** | 依赖关系未声明 | 在 turbo.json 中配置 dependsOn |
| **循环依赖** | 包之间相互引用 | 使用 madge 检测，重构代码结构 |
| **HMR 不生效** | Vite 配置问题 | 检查 Vite 配置的 optimizeDeps |
| **样式冲突** | CSS 全局污染 | 使用 CSS Modules 或 CSS-in-JS |

### 10.2 依赖管理最佳实践

```

├─────────────────────────────────────────────────────────────────┤
```

**2. 业务逻辑 Hook**

```typescript
// packages/hooks/src/useButton/useFormButton.ts
import { ref } from 'vue'

export interface UseFormButtonOptions {
  /** 提交函数 */
  handler: () => Promise<void> | void
  /** 验证函数，返回 true 表示验证通过 */
  validate?: () => boolean | Promise<boolean>
  /** 成功后的回调 */
  onSuccess?: () => void
  /** 失败后的回调 */
  onError?: (error: unknown) => void
  /** 是否显示错误提示 */
  showError?: boolean
}

export function useFormButton(options: UseFormButtonOptions) {
  const loading = ref(false)
  const error = ref<unknown>(null)

  const handleClick = async () => {
    // 验证
    if (options.validate) {
      const valid = await options.validate()
      if (!valid) return
    }

    loading.value = true
    error.value = null

    try {
      await options.handler()
      options.onSuccess?.()
    } catch (err) {
      error.value = err
      options.onError?.(err)

      if (options.showError) {
        // 这里可以接入提示组件
        console.error('操作失败:', err)
      }
    } finally {
      loading.value = false
    }
  }

  return {
    loading,
    error,
    handleClick
  }
}
```

```typescript
// packages/hooks/src/useButton/useConfirmButton.ts
import { ref } from 'vue'

export interface UseConfirmButtonOptions {
  /** 确认后执行的函数 */
  onConfirm: () => Promise<void> | void
  /** 确认提示内容 */
  message?: string
  /** 确认弹窗标题 */
  title?: string
}

export function useConfirmButton(options: UseConfirmButtonOptions) {
  const loading = ref(false)
  const showConfirm = ref(false)

  const handleClick = () => {
    showConfirm.value = true
  }

  const handleConfirm = async () => {
    loading.value = true
    try {
      await options.onConfirm()
      showConfirm.value = false
    } finally {
      loading.value = false
    }
  }

  const handleCancel = () => {
    showConfirm.value = false
  }

  return {
    loading,
    showConfirm,
    handleClick,
    handleConfirm,
    handleCancel,
    message: options.message || '确定要执行此操作吗？',
    title: options.title || '确认操作'
  }
}
```

**提取逻辑 Hook：**

```typescript
// packages/hooks/src/useTable/useTableData.ts
import { ref, computed } from 'vue'

export interface UseTableDataOptions<T> {
  /** 数据加载函数 */
  fetcher: (params: PaginationParams) => Promise<PaginatedData<T>>
  /** 初始页码 */
  initialPage?: number
  /** 每页数量 */
  pageSize?: number
}

export function useTableData<T>(options: UseTableDataOptions<T>) {
  const data = ref<T[]>([])
  const total = ref(0)
  const current = ref(options.initialPage || 1)
  const pageSize = ref(options.pageSize || 10)
  const loading = ref(false)

  const pagination = computed(() => ({
    current: current.value,
    pageSize: pageSize.value,
    total: total.value
  }))

  const fetch = async () => {
    loading.value = true
    try {
      const result = await options.fetcher({
        page: current.value,
        pageSize: pageSize.value
      })
      data.value = result.items
      total.value = result.total
    } finally {
      loading.value = false
    }
  }

  const changePage = (page: number) => {
    current.value = page
    fetch()
  }

  const changePageSize = (size: number) => {
    pageSize.value = size
    current.value = 1
    fetch()
  }

  // 初始加载
  fetch()

  return {
    data,
    loading,
    pagination,
    fetch,
    changePage,
    changePageSize,
    refresh: fetch
  }
}
```

### 12.4 场景：API 请求封装共享

```typescript
// packages/api-client/src/request.ts
import type { AxiosInstance, AxiosRequestConfig, AxiosResponse } from 'axios'
import axios from 'axios'

export interface RequestConfig extends AxiosRequestConfig {
  showError?: boolean
}

export class HttpClient {
  private instance: AxiosInstance

  constructor(baseURL: string) {
    this.instance = axios.create({
      baseURL,
      timeout: 10000
    })

    this.setupInterceptors()
  }

  private setupInterceptors() {
    // 请求拦截器
    this.instance.interceptors.request.use(
      (config) => {
        const token = localStorage.getItem('token')
        if (token) {
          config.headers.Authorization = `Bearer ${token}`
        }
        return config
      },
      (error) => Promise.reject(error)
    )

    // 响应拦截器
    this.instance.interceptors.response.use(
      (response) => response.data,
      (error) => {
        if (error.response?.status === 401) {
          // 处理未授权
          localStorage.removeItem('token')
          window.location.href = '/login'
        }
        return Promise.reject(error)
      }
    )
  }

  get<T = any>(url: string, config?: RequestConfig): Promise<T> {
    return this.instance.get(url, config)
  }

  post<T = any>(url: string, data?: any, config?: RequestConfig): Promise<T> {
    return this.instance.post(url, data, config)
  }

  put<T = any>(url: string, data?: any, config?: RequestConfig): Promise<T> {
    return this.instance.put(url, data, config)
  }

  delete<T = any>(url: string, config?: RequestConfig): Promise<T> {
    return this.instance.delete(url, config)
  }
}

// 创建实例
export const httpClient = new HttpClient(import.meta.env.VITE_API_BASE_URL)
```

### 13.2 组件复用速查表

| 场景 | 样式 | 逻辑 | 放置位置 | 实现方式 |
|------|------|------|----------|----------|


```
Error: Circular dependency detected
@monorepo/ui → @monorepo/components → @monorepo/ui

✅ 解决方案

2. 重构代码结构
3. 提取公共依赖到单独的包
```

### 14.2 TypeScript 相关错误

#### 错误 3：找不到模块类型定义

```
❌ 错误信息
TS2307: Cannot find module '@monorepo/ui' or its corresponding type declarations

✅ 解决方案
```

```json
// tsconfig.json
{
  "compilerOptions": {
    // 添加路径映射
    "baseUrl": ".",
    "paths": {
      "@monorepo/*": ["packages/*/src"]
    }
  },
  // 添加项目引用
  "references": [
    { "path": "./packages/ui" },
    { "path": "./packages/utils" },
    { "path": "./packages/hooks" }
  ]
}
```


## 十六、最佳实践

### 16.1 包设计原则

```
┌─────────────────────────────────────────────────────────────────┐
│  单一职责原则                                                   │
├─────────────────────────────────────────────────────────────────┤
│  每个 package 只做一件事，做好一件事                            │
│  ├── ui/       只负责 UI 组件                                   │
│  ├── hooks/   只负责业务逻辑                                   │
│  └── utils/   只负责工具函数                                   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  依赖方向原则                                                   │
├─────────────────────────────────────────────────────────────────┤
│  依赖关系应该是单向的，避免循环依赖                             │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐                    │
│  │  apps   │ →  │packages │ →  │packages │                    │
│  │         │    │ /ui     │    │ /utils  │                    │
│  └─────────┘    └─────────┘    └─────────┘                    │
│                                                                 │
│  ❌ 禁止：packages → apps                                       │
│  ❌ 禁止：packages/ui → packages/components                     │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  版本管理原则                                                   │
├─────────────────────────────────────────────────────────────────┤
│  内部包使用 workspace:*，发布时自动替换                         │
│  ├── 开发时："@monorepo/ui": "workspace:*"                     │
│  ├── 发布后："@monorepo/ui": "^1.2.3"                          │
│  └── 统一使用 Changesets 管理版本                               │
└─────────────────────────────────────────────────────────────────┘
```
