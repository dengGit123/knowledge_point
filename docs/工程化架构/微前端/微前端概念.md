# 微前端概念

## 一、什么是微前端

**微前端（Micro-Frontends）**是一种将单页应用（SPA）拆分为多个可独立开发、测试和部署的小型前端应用的架构模式。每个微前端应用可以由不同的团队使用不同的技术栈开发和维护。

```
┌─────────────────────────────────────────────────────────────┐
│                      浏览器                                  │
│  ┌─────────────┬─────────────┬─────────────┬─────────────┐ │
│  │   微前端 A  │   微前端 B  │   微前端 C  │   基座应用  │ │
│  │  (Vue 3)   │  (React)   │  (Angular)  │   (主应用)   │ │
│  │  团队 A     │  团队 B     │  团队 C     │   团队 Core  │ │
│  └─────────────┴─────────────┴─────────────┴─────────────┘ │
│                      ↓↓ 通信 ↓↓                            │
│                   共享状态 / 路由                           │
└─────────────────────────────────────────────────────────────┘
```

### 1.1 核心理念

微前端借鉴了微服务的架构思想，将其应用到前端领域：

| 特性 | 说明 |
|------|------|
| **技术栈无关** | 不同子应用可使用不同框架（Vue、React、Angular等） |
| **独立开发** | 各团队独立开发、互不干扰 |
| **独立部署** | 子应用可独立上线，无需整个应用重新部署 |
| **运行时组合** | 在浏览器运行时动态加载和组合各个子应用 |

## 二、为什么需要微前端

### 2.1 传统单体应用的痛点

```
问题场景：一个大型电商平台

┌────────────────────────────────────────────┐
│           单体前端应用                       │
│                                            │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  │
│  │首页  │  │商品  │  │订单  │  │支付  │  │
│  │团队  │  │团队  │  │团队  │  │团队  │  │
│  └──────┘  └──────┘  └──────┘  └──────┘  │
│         ↓ 全部耦合在同一项目                │
│                                            │
│  问题：                                     │
│  ❌ 构建时间长（10分钟+）                    │
│  ❌ 一个模块出错，整体无法上线               │
│  ❌ 技术栈统一，升级困难                     │
│  ❌ 代码冲突频繁                            │
└────────────────────────────────────────────┘
```

### 2.2 微前端带来的价值

| 价值点 | 说明 |
|--------|------|
| **团队自治** | 不同团队独立负责各自模块，减少沟通成本 |
| **增量升级** | 可以逐步替换旧系统，而非一次性重写 |
| **技术自由** | 新模块可用新技术，旧模块保持稳定 |
| **故障隔离** | 一个子应用故障不影响整体可用性 |
| **独立部署** | 小团队可以快速迭代，无需等待其他团队 |

## 三、微前端的核心技术方案

### 3.1 方案对比

| 方案 | 代表框架 | 技术栈限制 | 复杂度 | 适用场景 |
|------|----------|------------|--------|----------|
| **基于路由分发** | iframe、Web Component | 无限制 | ⭐️ | 简单场景、跨系统集成 |
| **运行时集成** | qiankun、single-spa | 需统一协议 | ⭐️⭐️⭐️ | 完整的微前端架构 |
| **编译时集成** | Module Federation | 无限制 | ⭐️⭐️ | 模块联邦、代码共享 |
| **EM 单体仓库** | Turborepo、pnpm workspace | 建议统一 | ⭐️⭐️ | 统一技术栈的巨石应用 |

### 3.2 方案一：基于路由分发

#### 3.2.1 iframe 方案

```html
<!-- 基座应用 -->
<iframe
  src="https://app-a.com"
  id="micro-app-a"
  sandbox="allow-scripts allow-same-origin"
></iframe>
```

```javascript
// 通信方式
// iframe → 父页面
window.parent.postMessage({ type: 'UPDATE_USER', data: user }, '*')

// 父页面 → iframe
document.getElementById('micro-app-a').contentWindow.postMessage(
  { type: 'CHANGE_LANG', lang: 'en' },
  'https://app-a.com'
)
```

**优点：**
- 天然隔离 CSS 和 JS
- 技术栈完全无关
- 浏览器原生支持

**缺点：**
- 性能较差（重复加载资源）
- 用户体验不流畅（无法共享状态）
- 弹窗、遮罩层处理复杂

#### 3.2.2 Web Component 方案

```javascript
// 定义微前端组件
class MicroAppA extends HTMLElement {
  connectedCallback() {
    this.attachShadow({ mode: 'open' })
    const shadow = this.shadowRoot

    // 加载子应用
    fetch('https://app-a.com/entry.js')
      .then(res => res.text())
      .then(code => eval(code))
  }
}

customElements.define('micro-app-a', MicroAppA)
```

```html
<!-- 使用 -->
<micro-app-a></micro-app-a>
```

### 3.3 方案二：运行时集成（qiankun）

[qiankun](https://qiankun.umijs.org/) 是基于 single-spa 的微前端实现，开箱即用。

```typescript
// 基座应用 - 注册子应用
import { registerMicroApps, start } from 'qiankun'

// 注册子应用
registerMicroApps([
  {
    name: 'vue-app',
    entry: '//localhost:7100',
    container: '#subapp-viewport',
    activeRule: '/vue',
  },
  {
    name: 'react-app',
    entry: '//localhost:7101',
    container: '#subapp-viewport',
    activeRule: '/react',
  }
])

// 启动 qiankun
start({
  sandbox: {
    strictStyleIsolation: true, // 样式隔离
    experimentalStyleIsolation: true,
  }
})
```

```typescript
// 子应用 - Vue 3 示例
// src/main.ts
import { createApp } from 'vue'
import App from './App.vue'
import router from './router'

let app: any

// 🔥 必须导出生命周期钩子
export async function bootstrap() {
  console.log('Vue app bootstraped')
}

export async function mount(props: any) {
  app = createApp(App)
  app.use(router)
  app.mount('#app')
}

export async function unmount() {
  app.unmount()
}

// 独立运行环境
if (!window.__POWERED_BY_QIANKUN__) {
  mount({})
}
```

**核心机制：**

| 机制 | 说明 |
|------|------|
| **HTML Entry** | 解析 HTML 获取 JS/CSS 资源 |
| **JS 沙箱** | 隔离全局变量，防止污染 |
| **样式隔离** | Scoped CSS 或 Shadow DOM |
| **生命周期** | bootstrap、mount、unmount |

### 3.4 方案三：模块联邦（Module Federation）

[Webpack 5 Module Federation](https://webpack.js.org/concepts/module-federation/) 允许多个独立构建的代码在运行时共享依赖。

```javascript
// app-a/webpack.config.js - 提供模块
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'app_a',
      filename: 'remoteEntry.js',
      exposes: {
        './Button': './src/components/Button',
        './Header': './src/components/Header',
      },
      shared: ['react', 'react-dom'],
    }),
  ],
}
```

```javascript
// app-b/webpack.config.js - 消费模块
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'app_b',
      remotes: {
        appA: 'app_a@http://localhost:3000/remoteEntry.js',
      },
      shared: ['react', 'react-dom'],
    }),
  ],
}
```

```tsx
// app-b 使用远程组件
import React from 'react'
const Button = React.lazy(() => import('appA/Button'))

function App() {
  return (
    <div>
      <React.Suspense fallback="Loading Button...">
        <Button />
      </React.Suspense>
    </div>
  )
}
```

## 四、微前端的关键问题与解决

### 4.1 应用间通信

```typescript
// 方案一：全局状态管理（Pinia/Vuex）
// 基座应用初始化全局 store
import { createPinia } from 'pinia'
const pinia = createPinia()

// 通过 props 传递给子应用
export function beforeMount(props) {
  props.setGlobalState({ user: currentUser, token: 'xxx' })
}

// 方案二：自定义事件总线
class EventBus {
  constructor() {
    this.events = {}
  }

  on(event, callback) {
    if (!this.events[event]) this.events[event] = []
    this.events[event].push(callback)
  }

  emit(event, data) {
    if (this.events[event]) {
      this.events[event].forEach(cb => cb(data))
    }
  }
}

window.microEventBus = new EventBus()
```

### 4.2 样式隔离

| 方案 | 实现方式 | 效果 |
|------|----------|------|
| **动态 CSS** | 加载时添加前缀，卸载时移除 | 简单有效 |
| **Shadow DOM** | 天然隔离 | 兼容性问题 |
| **CSS Modules** | 编译时生成唯一类名 | 需要构建支持 |
| **Scoped CSS** | Vue/React 自带 | 框架依赖 |

```css
/* 动态 CSS 示例 */
/* 原始 */
.button { color: red; }

/* 转换后 */
.app-name-button { color: red; }
```

### 4.3 公共依赖加载

```javascript
// qiankun 预加载共享依赖
start({
  prefetch: 'all', // 预加载所有子应用资源
  singular: false,  // 是否单实例
  getTemplate(tpl) {
    // 可以在这里处理 HTML 模板
    return tpl
  }
})

// CDN 共享依赖
import { externals } from './config'

externals.forEach(({ name, url }) => {
  const script = document.createElement('script')
  script.src = url
  document.head.appendChild(script)
})
```

## 六、常见问题 FAQ

### Q1：什么时候应该使用微前端？

**适合使用：**
- 大型团队（10+ 人）协作的项目
- 需要整合多个旧系统
- 不同业务模块需要独立上线
- 想尝试新技术但需保持旧系统稳定

**不适合使用：**
- 小型项目（< 5 人团队）
- 业务关联紧密的模块
- 对性能要求极高的场景

### Q2：微前端会影响性能吗？

会有一定影响，但可以通过以下方式优化：
- 预加载关键资源
- 共享公共依赖（react、vue 等）
- 使用缓存策略
- 按需加载子应用

### Q3：如何处理登录状态？

**推荐方案：** 基座应用统一处理登录

```typescript
// 基座应用路由守卫
router.beforeEach((to, from, next) => {
  const token = getToken()

  if (!token && to.path !== '/login') {
    next('/login')
  } else if (token) {
    // 登录状态通过 props 传给子应用
    next()
  }
})
```
