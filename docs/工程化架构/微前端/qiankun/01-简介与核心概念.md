# qiankun 简介与核心概念

> 📖 官方文档：[qiankun](https://qiankun.umijs.org/zh) | [GitHub 仓库](https://github.com/umijs/qiankun)

## 一、概述

**qiankun**（读作「乾坤」）是阿里 UmiJS 团队开源的一款微前端框架，它基于 [single-spa](https://single-spa.js.org/) 二次封装，提供了一套**开箱即用**的微前端解决方案。

用大白话说：如果你把一个庞大的前端应用拆成好几个独立的小应用（比如「商品应用」「订单应用」「用户应用」），qiankun 就是那个负责把这些小应用「拼装」到一个页面里、并管理它们加载、卸载、隔离的「总调度」。

> 💡 **名字由来**：「qiankun」意为「乾坤」，取自《易经》「乾坤其易之缊邪」——寓意「能大能小、包容万物」，象征它能将各类应用聚合到一个体系下运行。

## 二、为什么需要 qiankun

[single-spa](https://single-spa.js.org/) 是微前端的鼻祖，但直接使用它门槛很高，需要手动处理很多事。qiankun 在它之上补齐了这些能力：

| 痛点 | single-spa | qiankun |
| --- | --- | --- |
| JS 沙箱 | 需自行实现 | ✅ ProxySandbox 自动隔离全局变量 |
| 样式隔离 | 需自行处理 | ✅ Shadow DOM / Scoped CSS |
| 应用加载方式 | JS Entry（手动维护入口 JS 清单） | ✅ HTML Entry（给个 HTML 地址即可） |
| 资源预加载 | 手动实现 | ✅ 内置 `prefetch` |
| 全局状态通信 | 自行实现 | ✅ `initGlobalState` |
| 接入成本 | 高 | 低，改几行配置即可 |

其中最关键的升级是 **HTML Entry**，它是 qiankun 区别于 single-spa 的核心思想，下面单独讲解。

## 三、HTML Entry —— qiankun 的核心思想

### 3.1 什么是 HTML Entry

传统 single-spa 采用 **JS Entry**：你需要告诉主应用子应用入口 JS 文件的地址清单（`scripts`），并手动维护这些 JS 的依赖关系、加载顺序、`publicPath`。子应用一旦改了打包产物，主应用的配置就得跟着改，耦合非常严重。

qiankun 改用 **HTML Entry**：主应用只需要一个子应用的 **HTML 地址**（即子应用部署后的访问 URL），剩下的 qiankun 全自动处理：

1. 用 `fetch` 把子应用的 HTML 文档拉下来
2. 解析 HTML，提取其中所有的 `` `<script>` ``、`` `<style>` ``、`` `<link>` `` 资源
3. 按顺序执行这些脚本，并从中识别出子应用导出的生命周期函数（`bootstrap` / `mount` / `unmount`）
4. 把子应用渲染到主应用指定的容器节点

```text
┌───────────────┐    ① fetch HTML    ┌─────────────────┐
│    主应用      │ ─────────────────▶ │  子应用 index.html │
│               │                    │  <script> ...    │
│  entry:       │ ◀───────────────── │  <style>  ...    │
│  //子应用URL  │   ②解析 ③执行脚本   │  <link>   ...    │
│               │   ④渲染到容器       └─────────────────┘
└───────────────┘
```

### 3.2 HTML Entry 的好处

- **解耦**：主应用不关心子应用怎么打包、产物叫什么名字，只要子应用的 HTML 能被正常访问即可
- **独立部署**：子应用可独立构建、独立部署，完全不影响主应用
- **技术栈无关**：子应用用 Vue、Angular、原生 JS 都行，只要导出约定的生命周期钩子

> ⚠️ **注意：** 这也是为什么 qiankun 要求子应用的静态资源**必须支持跨域（CORS）**——因为主应用要通过 `fetch` 跨域去拉取子应用的 HTML、JS、CSS。开发环境需要在子应用的 dev server 配置 CORS 响应头。

## 四、核心特性一览

| 特性 | 作用 | 一句话理解 |
| --- | --- | --- |
| **HTML Entry** | 通过 HTML 入口加载应用 | 给个网址就能加载，不用管打包细节 |
| **JS 沙箱** | Proxy 隔离全局变量 | 子应用写的 `window.xxx` 不会污染别人 |
| **样式隔离** | Shadow DOM / Scoped CSS | 子应用的 CSS 不会影响主应用和其他子应用 |
| **预加载 prefetch** | 空闲时预加载资源 | 用户还没点，资源就偷偷下好了，秒开 |
| **全局状态 initGlobalState** | 应用间通信 | 主子应用共享一份数据，变化能互相通知 |

## 五、与其它微前端方案对比

| 方案 | 隔离性 | 接入成本 | 通信能力 | 适用场景 |
| --- | --- | --- | --- | --- |
| **iframe** | 强（天然隔离） | 低 | 困难（postMessage） | 隔离要求高、交互少 |
| **qiankun** | 中（JS + 样式沙箱） | 中 | `initGlobalState` | 通用，中大型应用拆分 |
| **模块联邦** | 弱（共享运行时） | 中（需 webpack5） | 共享模块 | 同技术栈、需共享依赖 |
| **Web Components** | 中 | 高（需改造） | 自定义事件 | 组件级复用 |

> 💡 **提示：** 如果你的子应用之间需要大量共享依赖、同属一个技术栈，模块联邦可能比 qiankun 更合适；如果只是想把几个独立应用「拼」到一个后台里，qiankun 是更通用的选择。

## 六、适用场景与不适用场景

### 6.1 适合 qiankun

- **后台管理系统**：多个团队各负责一个业务模块，需统一登录、统一菜单、统一布局
- **老应用渐进式迁移**：旧的 Vue2 应用与新 Vue3 应用并存，逐步替换
- **多技术栈共存**：不同模块由不同团队用不同框架开发

### 6.2 不太适合 qiankun

- **对首屏极致性能要求的 C 端页面**：qiankun 有 `fetch` HTML 的运行时开销
- **应用间需要大量实时、高频通信**：qiankun 的全局状态通信能力有限，不适合做高频数据流
- **多个子应用需同时渲染并频繁交互**：多实例场景下的样式隔离、沙箱有坑，需谨慎

## 七、qiankun 的基本工作流程

用一张图概括 qiankun 从注册到运行的全过程：

```text
                  主应用启动
                      │
            ┌─────────▼──────────┐
            │ registerMicroApps  │  注册所有子应用配置
            └─────────┬──────────┘
                      │
                 start()  启动 qiankun，监听路由变化
                      │
        ┌─────────────▼──────────────┐
        │  浏览器 URL 变化            │
        │  匹配 activeRule 规则       │
        └─────────────┬──────────────┘
                      │ 命中某子应用
        ┌─────────────▼──────────────┐
        │  ① beforeLoad              │
        │  ② fetch 子应用 HTML        │
        │  ③ 解析并执行脚本            │
        │  ④ 调用子应用 bootstrap      │
        │  ⑤ 调用子应用 mount（挂载）   │
        │  ⑥ afterMount              │
        └────────────────────────────┘
                      │
              用户路由切走 → beforeUnmount → unmount → afterUnmount
```

## 八、总结

- qiankun = single-spa + **HTML Entry** + JS 沙箱 + 样式隔离 + 预加载 + 全局状态
- **核心思想**：HTML Entry 让子应用接入只需一个 URL，主子应用彻底解耦
- **三大隔离**：JS 沙箱（全局变量）、样式隔离（CSS）、路由前缀隔离
- **适用场景**：中后台系统、渐进式迁移、多技术栈共存
