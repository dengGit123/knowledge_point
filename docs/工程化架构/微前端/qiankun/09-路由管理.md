# qiankun 路由管理

> 📖 官方文档：[activeRule](https://qiankun.umijs.org/zh/api#registermicroappsapps-lifecycles) | [主应用路由页加载微应用](https://qiankun.umijs.org/zh/faq)

qiankun 通过监听浏览器 URL 变化来激活/卸载子应用。本文讲解 `activeRule` 匹配、子应用路由 base 配置、主应用路由页加载子应用，以及路由 404 问题。

## 一、qiankun 路由机制

qiankun 启动后，劫持了 `history.pushState` / `history.replaceState` 和 `popstate` 事件。URL 一旦变化，qiankun 会遍历所有注册子应用的 `activeRule`，匹配上的激活，不再匹配的卸载：

```text
用户操作改 URL（pushState / 点击后退 / 手动输入）
        │
        ▼
qiankun 监听到变化
        │
        ▼
遍历所有子应用的 activeRule
        │
   ┌────┴────────────┐
   ▼                 ▼
匹配成功 → mount    不再匹配 → unmount
```

所以主应用**不需要手动控制子应用的加载**，只需配好 `activeRule`，剩下的交给 qiankun。

## 二、activeRule 激活规则

`activeRule` 决定「什么 URL 下激活该子应用」。字符串做**前缀匹配**，函数返回 `true` 表示激活，数组满足其一即激活：

```ts
// 前缀匹配：访问 /sub-vue 及其子路径都激活
activeRule: '/sub-vue'

// hash 模式
activeRule: '/sub-vue/#/home'

// 多前缀
activeRule: ['/sub-vue', '/vue-app']

// 自定义函数
activeRule: (location) => location.pathname.startsWith('/sub-vue')
```

> 💡 **提示：** 完整的 `activeRule` 匹配规则（含动态参数 `:userId`、hash 规则等）见 [03-主应用配置](./03-主应用配置.md#12-activerule-匹配规则)。

## 三、路由模式：history vs hash

### 3.1 history 模式

URL 形如 `https://app.com/sub-vue/home`，`activeRule` 用 pathname 前缀：

```ts
activeRule: '/sub-vue'
```

> ⚠️ **注意：** history 模式下，刷新页面或直接访问子路由会向服务器请求 `/sub-vue/home`，服务器需配置 fallback 到 `index.html`，否则 404。

### 3.2 hash 模式

URL 形如 `https://app.com/#/sub-vue/home`，`activeRule` 需带上 hash：

```ts
activeRule: '/#/sub-vue'
```

hash 模式不需要服务器配置，但 URL 不够美观。主应用和子应用的 hash 模式要统一，否则匹配混乱。

> 💡 **提示：** 推荐主子应用都用 history 模式，URL 清晰、SEO 友好。

## 四、子应用 router base 配置

子应用被主应用加载后，它的路由需要知道「自己的前缀是什么」，否则子应用路由会从根路径 `/` 开始，与主应用冲突。

### 4.1 主应用传递 routerBase

```ts
registerMicroApps([{
  name: 'sub-vue',
  entry: '//localhost:7105',
  container: '#subapp-viewport',
  activeRule: '/sub-vue',
  props: { routerBase: '/sub-vue' }   // ⭐ 把前缀传给子应用
}])
```

### 4.2 Vue3 子应用

```ts
import { createRouter, createWebHistory } from 'vue-router'

export async function mount(props) {
  const { routerBase } = props

  const router = createRouter({
    // qiankun 环境用传入的前缀，独立运行用根路径
    history: createWebHistory(
      window.__POWERED_BY_QIANKUN__ ? routerBase : '/'
    ),
    routes
  })
  // ...
}
```

### 4.3 Vue2 子应用

```ts
import VueRouter from 'vue-router'

export async function mount(props) {
  const { routerBase } = props

  const router = new VueRouter({
    base: window.__POWERED_BY_QIANKUN__ ? routerBase : '/',
    mode: 'history',
    routes
  })
  // ...
}
```

> ⚠️ **关键：** `activeRule` 和 `routerBase` 必须一致。`activeRule: '/sub-vue'` 对应 `routerBase: '/sub-vue'`。不一致会导致「子应用激活了但内部路由匹配不到」的 404。

## 五、在主应用路由页面加载子应用

实际项目里，子应用容器常放在主应用的某个**路由页面**中（如 `/portal` 页面里嵌子应用）。此时要保证「子应用加载时，主应用这个路由页面也已加载」。

### 5.1 配置步骤（Vue + vue-router）

```ts
// ① 主应用路由：path 加通配符 *
const routes = [
  {
    path: '/portal/*',                    // ⭐ 通配符，匹配 /portal 下的所有子路径
    name: 'portal',
    component: () => import('../views/Portal.vue')
  }
]
```

```ts
// ② 子应用 activeRule 包含主应用路由 path
registerMicroApps([{
  name: 'app1',
  entry: '//localhost:8080',
  container: '#container',
  activeRule: '/portal/app1'              // ⭐ 包含 /portal
}])
```

```vue
<!-- ③ 在 Portal.vue 的 mounted 中调用 start（避免重复） -->
<template>
  <div>
    <div id="container"></div>
  </div>
</template>

<script setup lang="ts">
import { onMounted } from 'vue'
import { start } from 'qiankun'

onMounted(() => {
  // ⭐ 防止重复 start
  if (!window.qiankunStarted) {
    window.qiankunStarted = true
    start()
  }
})
</script>
```

### 5.2 为什么要这样

- `path: '/portal/*'`：让 vue-router 把 `/portal/app1` 这样的 URL 也渲染成 `Portal.vue`（容器才存在）
- `activeRule: '/portal/app1'`：qiankun 在该 URL 下激活子应用
- 在 `mounted` 调 `start`：此时容器 DOM 已渲染，`start` 能正确挂载

> ⚠️ **注意：** 若 `Portal.vue` 有子路由，需另外注册一个使用该组件的路由。`start()` 全局只能调用一次，务必用标志位防重复。

## 六、路由跳转与 404

### 6.1 history 模式刷新 404

直接访问 `https://app.com/sub-vue/home`，服务器找不到该路径对应文件，返回 404。需配置服务器将所有路由 fallback 到 `index.html`：

```nginx
# Nginx 配置
location / {
  try_files $uri $uri/ /index.html;
}
```

### 6.2 子应用内部路由 404

原因通常是 `activeRule` 与 `routerBase` 不一致：

```ts
// ❌ 不一致：activeRule 是 /sub-vue，但 routerBase 是 /vue
{ activeRule: '/sub-vue', props: { routerBase: '/vue' } }
// 访问 /sub-vue/home：qiankun 激活了，但子应用路由 base 是 /vue，匹配不到 /home

// ✅ 一致
{ activeRule: '/sub-vue', props: { routerBase: '/sub-vue' } }
```

### 6.3 主子应用路由跳转

| 场景 | 方式 |
| --- | --- |
| 主应用 → 子应用 | `history.pushState` 或 `<a href="/sub-vue">` |
| 子应用 → 子应用 | 子应用内 `router.push('/sub-vue2/...')`（需带完整前缀） |
| 子应用 → 主应用页面 | `history.pushState(state, '', '/main-page')` |

> 💡 **提示：** 子应用跳转到其他子应用时，要用**完整路径**（含前缀），如 `router.push('/sub-vue2/home')`，而不是 `/home`。

## 七、主从模式 vs 前缀模式

### 7.1 前缀模式（推荐）

主应用只匹配 `activeRule` 前缀，子应用内部路由由子应用自己管理：

```text
主应用路由表：/home, /about, /portal/*（容器页）
子应用 activeRule：/sub-vue
子应用内部路由：/sub-vue/home, /sub-vue/list（子应用自治）
```

- **优点**：主子应用路由解耦，子应用可独立开发
- **缺点**：需要协调 `activeRule` 与 `routerBase` 一致

### 7.2 主从模式

子应用路由作为主应用路由的子集，主应用路由表里直接写子应用路由：

```text
主应用路由表：/sub-vue/home, /sub-vue/list（直接包含子应用路由）
```

- **优点**：路由统一管理
- **缺点**：主应用耦合子应用路由，子应用路由变化主应用要跟着改

> 💡 **提示：** 微前端的核心价值是解耦，推荐**前缀模式**，让子应用路由自治。

## 八、总结

- qiankun 劫持 `pushState`/`popstate`，URL 变化时按 `activeRule` 激活/卸载子应用
- `activeRule` 做前缀匹配，与子应用 `routerBase` 必须一致
- history 模式需服务器 fallback 到 `index.html`，否则刷新 404
- 容器在主应用路由页时：`path` 加 `*`、`activeRule` 包含路由 path、在 `mounted` 调 `start`
- 推荐前缀模式，主子应用路由解耦
