# qiankun 样式隔离

> 📖 官方文档：[sandbox 配置](https://qiankun.umijs.org/zh/api#startopts)

主应用和多个子应用共享同一个 `document`，CSS 是全局生效的。样式隔离让每个应用的 CSS 只作用于自己，避免相互覆盖。本文讲解 qiankun 的三种样式隔离情况与注意事项。

## 一、为什么需要样式隔离

没有样式隔离时，全局 CSS 会互相覆盖：

```css
/* 子应用 A 的全局样式 */
.btn { color: red; }

/* 子应用 B 的全局样式 */
.btn { color: blue; }

/* 主应用的全局样式 */
* { box-sizing: border-box; }
```

三者都挂在同一个 document 上，后加载的覆盖先加载的，`.btn` 颜色、`box-sizing` 行为都不可控。样式隔离要解决的就是「谁的样式只影响谁」。

## 二、qiankun 的三种样式隔离情况

qiankun 的样式隔离能力取决于 `sandbox` 配置：

| 情况 | 配置 | 隔离范围 | 机制 |
| --- | --- | --- | --- |
| **默认** | `sandbox: true`（默认） | 单实例下**子应用之间** | 切换时卸载旧子应用样式 |
| **Shadow DOM** | `strictStyleIsolation: true` | 子应用 ↔ 主应用、子应用之间 | Shadow DOM 完全隔离 |
| **Scoped CSS** | `experimentalStyleIsolation: true` | 子应用样式不漏出到外部 | 选择器加前缀改写 |

> ⚠️ **关键认知：** 默认沙箱**只保证单实例场景下子应用之间的样式隔离**，**不保证**主应用与子应用之间、多实例场景下的隔离。要隔离主子应用，必须额外开启 `strictStyleIsolation` 或 `experimentalStyleIsolation`。

## 三、strictStyleIsolation —— Shadow DOM

```ts
start({
  sandbox: {
    strictStyleIsolation: true
  }
})
```

### 3.1 原理

qiankun 为每个子应用的容器包裹一层 **Shadow DOM**。Shadow DOM 内部的样式天然与外部隔离，子应用的 CSS 出不去，外部 CSS 也进不来：

```text
┌─ 主应用 document ─────────────────────────┐
│                                          │
│   ┌─ #subapp-viewport ────────────────┐  │
│   │  ╔═ Shadow Root (隔离边界) ═══════╗│  │
│   │  ║  <style>子应用样式</style>     ║│  │
│   │  ║  <div id="app">子应用内容</div>║│  │
│   │  ╚════════════════════════════════╝│  │
│   └────────────────────────────────────┘  │
└──────────────────────────────────────────┘
```

### 3.2 优缺点

| 优点 | 缺点 |
| --- | --- |
| 样式完全隔离 | 弹窗类组件挂到 `body` 会逃出 Shadow DOM，样式丢失 |
| 主子应用互不影响 | 事件冒泡被 retarget，某些全局事件监听异常 |
| | 第三方 UI 库需大量适配 |

### 3.3 弹窗逃逸问题

Vue3 的 `<Teleport>`、第三方 UI 库的 Modal/Message/Popover 默认挂到 `document.body`，会逃出 Shadow DOM，导致样式丢失：

```vue
<!-- ❌ Teleport 到 body，逃出 Shadow DOM -->
<Teleport to="body">
  <div class="modal">弹窗</div>
</Teleport>

<!-- ✅ 挂到子应用容器内，留在 Shadow DOM 中 -->
<Teleport to="#app">
  <div class="modal">弹窗</div>
</Teleport>
```

> ⚠️ **注意：** 官方明确指出 Shadow DOM 隔离「不是一个可以无脑使用的方案」，大部分应用需要适配才能在 Shadow DOM 中正常运行。如非必要，优先用 `experimentalStyleIsolation`。

## 四、experimentalStyleIsolation —— Scoped CSS

```ts
start({
  sandbox: {
    experimentalStyleIsolation: true
  }
})
```

### 4.1 原理

qiankun 在运行时**改写**子应用的所有样式规则，给每条选择器加上一个特殊前缀 `div[data-qiankun-{应用名}]`，限定其作用范围：

```css
/* 改写前（子应用原始样式） */
.app-main {
  font-size: 14px;
}
.container .title {
  color: #333;
}

/* 改写后（假设应用名为 sub-vue） */
div[data-qiankun-sub-vue] .app-main {
  font-size: 14px;
}
div[data-qiankun-sub-vue] .container .title {
  color: #333;
}
```

qiankun 给子应用容器加了 `data-qiankun-sub-vue` 属性，于是这些样式只匹配容器内的元素，**不会漏出到主应用和其他子应用**。

### 4.2 不被改写的规则

以下规则**不会被改写**，仍可能造成冲突：

| 规则 | 原因 | 影响 |
| --- | --- | --- |
| `@keyframes` | 动画名是全局的 | 动画名冲突 |
| `@font-face` | 字体名是全局的 | 字体名冲突 |
| `@import` | 引入外部样式 | 外部样式不受控 |
| `@page` | 打印相关 | 较少涉及 |

> ⚠️ **注意：** 若子应用大量使用 `@keyframes` 动画或自定义字体，需手动加命名空间前缀避免冲突。

### 4.3 body 样式失效

由于 `body` 不是 `div[data-qiankun-xxx]` 的后代，加前缀后 `body { ... }` 永远匹配不到。所以子应用的 `body` 全局样式**既不污染主应用、自己也不生效**：

```css
/* 子应用写的 body 样式，改写后匹配不到，不生效 */
body { background: #f0f0f0; }
/* 改写后：div[data-qiankun-sub-vue] body { ... } ← body 不是它的后代 */

/* ✅ 应改为容器内选择器 */
#app { background: #f0f0f0; }
```

## 五、方案对比

| 方案 | 隔离强度 | 兼容性 | 适配成本 | 推荐度 |
| --- | --- | --- | --- | --- |
| **默认**（`sandbox: true`） | 弱（仅子应用间） | 好 | 无 | 单实例简单场景 |
| **experimentalStyleIsolation** | 中 | 好 | 低 | ⭐⭐⭐⭐⭐ 推荐 |
| **strictStyleIsolation** | 强（完全隔离） | 差 | 高 | 隔离要求极高时 |

> 💡 **提示：** `strictStyleIsolation` 和 `experimentalStyleIsolation` **不要同时开启**，选其一即可。

## 六、主子应用样式隔离

默认情况下，**主应用样式可能影响子应用**（因为子应用在主应用 DOM 树内，主应用的全局样式如 `* {}`、`body {}` 会作用到子应用）。要彻底隔离主子：

1. **主应用避免写「杀伤性」全局样式**：

```css
/* ❌ 影响所有后代，包括子应用 */
* { box-sizing: border-box; }
body { margin: 0; }
#app { font-size: 14px; }

/* ✅ 限定在主应用自身作用域 */
.main-app * { box-sizing: border-box; }
.main-app { font-size: 14px; }
```

2. **子应用开启 `experimentalStyleIsolation`**，防止子应用样式漏出影响主应用

3. **极端情况用 `strictStyleIsolation`**（Shadow DOM 完全隔离，但需适配弹窗）

## 七、注意事项与最佳实践

### 7.1 子应用不要写全局选择器

```css
/* ❌ 污染全局 */
body { background: #fff; }
* { box-sizing: border-box; }
a { color: #409eff; }

/* ✅ 限定在子应用根容器内 */
#app { background: #fff; }
#app * { box-sizing: border-box; }
#app a { color: #409eff; }
```

### 7.2 弹窗组件不要挂 body

子应用的弹窗、抽屉、消息提示等，配置为挂载到子应用容器内，而非 `document.body`。否则在 Shadow DOM 下样式丢失，在 Scoped CSS 下也可能溢出。

### 7.3 动画与字体加命名空间

```css
/* 加应用名前缀，避免 @keyframes 冲突 */
@keyframes sub-vue-fade { /* ... */ }

/* 字体名也加前缀 */
@font-face {
  font-family: 'sub-vue-icon';
  src: url('./icon.woff2');
}
```

### 7.4 第三方 UI 库

Element Plus、Ant Design Vue 等组件库的样式是全局的，接入时：

- 开启 `experimentalStyleIsolation` 防止其样式漏出
- 注意组件库的弹窗是否挂 body（按 7.2 处理）
- 避免使用 `strictStyleIsolation`（Shadow DOM 下组件库易出问题）

## 八、总结

- 默认沙箱只隔离单实例子应用之间，**不隔离主子应用**
- `experimentalStyleIsolation`：给样式加 `div[data-qiankun-xxx]` 前缀，推荐使用
- `strictStyleIsolation`：Shadow DOM 完全隔离，但弹窗、事件有坑，适配成本高
- `@keyframes`、`@font-face` 不被改写，需手动加命名空间
- 子应用避免写 `body`、`*` 等全局样式，弹窗不要挂 `body`
