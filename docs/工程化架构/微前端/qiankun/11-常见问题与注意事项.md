# qiankun 常见问题与注意事项

> 📖 官方文档：[常见问题 FAQ](https://qiankun.umijs.org/zh/faq)

本文汇总 qiankun 接入与运行中的高频问题、踩坑点与最佳实践。遇到问题先看这里。

## 一、问题排查总览

```text
子应用没出来 / 报错？
│
├─ 报 LOADING_SOURCE_CODE：未导出生命周期？  → 见 二
├─ 报 NOT_MOUNTED：Target container not existed？ → 见 三
├─ 资源 404（JS/CSS）？                    → 见 四
├─ 字体/图片 404？                         → 见 五
├─ Vue Router redefine $router？           → 见 六
├─ 跨域报错？                              → 见 七
├─ 脚本执行错误？                          → 见 八
└─ 运营商脚本干扰？                        → 见 九
```

## 二、LOADING_SOURCE_CODE：未导出生命周期

```
Application died in status LOADING_SOURCE_CODE:
You need to export the functional lifecycles in xxx entry
```

qiankun 无法从子应用 entry js 中识别出生命周期钩子。逐条排查：

| 检查项 | 要求 |
| --- | --- |
| 生命周期导出 | 入口 js 必须导出 `bootstrap`/`mount`/`unmount` |
| webpack 配置 | `output.library` + `libraryTarget: 'umd'` |
| `output.globalObject` | 若配置了，值必须是 `'window'`，或移除用默认值 |
| `package.json` 的 `name` | 全局唯一，与主应用注册 `name` 一致 |
| 入口 js 加载顺序 | 必须是 HTML 中**最后一个**加载的脚本 |
| 生产环境 | 检查 `index.html` 和 entry js 是否正常返回（非 404.html） |
| webpack5 + 模块联邦 | 需异步导出生命周期（见下） |

### webpack5 模块联邦的特殊处理

```ts
// index 文件
const promise = import('./bootstrap')

export const bootstrap = () => promise.then(m => m.bootstrap())
export const mount = () => promise.then(m => m.mount())
export const unmount = () => promise.then(m => m.unmount())
```

### AMD/CommonJS 兼容问题

若主子应用用了 AMD/CommonJS，`umd` 会失效，将 `libraryTarget` 改为 `'window'`，或直接把生命周期挂到 `window` 上。

> 💡 **提示：** 若仍报错，尝试把子应用 `output.library` 配成与主应用注册的 `name` **完全一致**（不带 `-[name]` 后缀）。

## 三、NOT_MOUNTED：Target container not existed

```
Application died in status NOT_MOUNTED:
Target container with #container not existed after xxx mounted!
```

子应用加载后，容器 DOM 节点不存在了。常见原因：

### 3.1 子应用根 id 与其他 DOM 冲突

子应用用 `document.querySelector('#app')` 查找时，可能查到了主应用的 `#app`。应限定查找范围：

```ts
// ❌ 全局查找，可能找错
app.mount(document.querySelector('#app'))

// ✅ 在主应用传入的 container 内查找
function render(props = {}) {
  const { container } = props
  app.mount(container ? container.querySelector('#app') : '#app')
}
```

### 3.2 子应用使用了 document.write

某些地图 SDK（高德 1.x、腾讯 2.x）内部用 `document.write`，会清空页面。解决：升级 SDK 版本，或把地图 SDK 放主应用加载。

### 3.3 start 调用时机不对

容器在异步渲染（如路由页面）中，`start()` 在容器存在前就调用了。应在容器 DOM 就绪后（Vue 的 `mounted`、组件 `onMounted`）再 `start()`。详见 [09-路由管理](./09-路由管理.md#五、在主应用路由页面加载子应用)。

### 3.4 路由过渡效果

Vue 主应用若给容器所在路由配了过渡动画，可能导致 mounting 过程中容器短暂消失。解决：去掉过渡或换过渡效果。

## 四、资源 404：publicPath 问题

子应用被主应用加载后，动态资源（JS/CSS/图片）路径错误导致 404。

### 4.1 运行时 publicPath（推荐）

在子应用入口 js **最顶部**添加：

```js
if (window.__POWERED_BY_QIANKUN__) {
  // eslint-disable-next-line no-undef
  __webpack_public_path__ = window.__INJECTED_PUBLIC_PATH_BY_QIANKUN__
}
```

qiankun 会在 `bootstrap` 前注入这个变量。

### 4.2 静态 publicPath

```js
// webpack / vue.config.js
output: {
  publicPath: '//localhost:7105'   // 绝对地址，生产改为 CDN 域名
}
```

> 💡 **提示：** 运行时方案灵活（适配多环境），静态方案简单。生产推荐静态配 CDN。

## 五、字体 / 图片 404

qiankun 把子应用的外链样式改成**内联**，但 CSS 中的字体、背景图是相对路径，CSS 打包后无法动态修正。

### 5.1 方案一：资源上 CDN（推荐）

所有图片、字体上传 CDN，CSS 中直接引用 CDN 绝对地址。

### 5.2 方案二：小文件转 base64（推荐）

用 `url-loader` 把小体积字体/图片打包成 base64 内联：

```js
// webpack
module: {
  rules: [{
    test: /\.(png|jpe?g|gif|webp|woff2?|eot|ttf|otf)$/i,
    use: [{ loader: 'url-loader', options: {} }]
  }]
}
```

vue-cli5 用 `asset/inline` 替代：

```js
chainWebpack: (config) => {
  config.module.rule('fonts').type('asset/inline').set('generator', {})
  config.module.rule('images').type('asset/inline').set('generator', {})
}
```

### 5.3 方案三：大文件用 file-loader 注入完整路径

```js
const publicPath = process.env.NODE_ENV === 'production'
  ? 'https://cdn.example.com/'
  : 'http://localhost:7105'

module: {
  rules: [{
    test: /\.(png|jpe?g|gif|webp)$/i,
    use: [{
      loader: 'file-loader',
      options: { name: 'img/[name].[hash:8].[ext]', publicPath }
    }]
  }]
}
```

> 💡 **提示：** 实际项目常组合方案二和三：小文件 base64，大文件注入路径前缀。

## 六、Vue Router 报错：Cannot redefine $router

```
Uncaught TypeError: Cannot redefine property: $router
```

### 原因

qiankun 用 Proxy 代理 `window`。VueRouter 模块加载时会执行：

```js
if (inBrowser && window.Vue) {
  window.Vue.use(VueRouter)  // 访问 window.Vue
}
```

若主应用把 Vue 通过 CDN external 挂到了 `window.Vue`，子应用访问 `window.Vue` 时会先查自己的沙箱，再查父应用，导致 `Vue.use(VueRouter)` 被重复调用，`$router` 重定义报错。

### 解决

```js
// 方案一：主应用不用 CDN external 加载 Vue（推荐）
// 用打包工具加载 Vue 模块，不挂 window.Vue

// 方案二：主应用改名 window.Vue
window.Vue2 = window.Vue
delete window.Vue
```

## 七、CORS 跨域

qiankun 通过 `fetch` 跨域获取子应用资源，**子应用静态资源必须支持跨域**。

```js
// webpack dev server
devServer: {
  headers: { 'Access-Control-Allow-Origin': '*' }
}

// vite dev server
server: { cors: true }
```

生产环境 Nginx：

```nginx
location / {
  add_header Access-Control-Allow-Origin *;
}
```

> ⚠️ **注意：** 第三方脚本若无法加跨域头，下载到本地由自己服务器 serve 再配跨域。

## 八、import-html-entry 脚本执行错误

```
[import-html-entry]: error occurs while excuting xxx script http://xxx/x.js
```

第一行只是辅助信息（指出哪个 js 报错），**真正的异常在第二行**。排查方向：

1. 报错 js 是否有语法错误（少分号、未初始化变量等）
2. 是否依赖了主应用提供的全局变量，但主应用未初始化
3. 兼容性问题（js 在当前环境有语法兼容问题）

## 九、运营商脚本干扰

运营商可能往页面注入脚本，若未标 `async`，失败后会 block 后续脚本。用 `getTemplate` 过滤：

```ts
start({
  getTemplate(tpl) {
    // 移除异常脚本
    return tpl.replace(/bad-script\.js/, '')
  }
})
```

## 十、最佳实践清单

### 10.1 主应用

- [ ] `registerMicroApps` 的 `name` 与子应用 `package.json` 的 `name` 一致且唯一
- [ ] `activeRule` 与子应用 `routerBase` 一致
- [ ] 容器在路由页时，`start()` 在 `mounted` 后调用且不重复
- [ ] 注册 `addGlobalUncaughtErrorHandler` 做错误兜底，避免白屏
- [ ] 合理配置 `prefetch` 平衡速度与流量

### 10.2 子应用

- [ ] 导出 `bootstrap`/`mount`/`unmount`（async 函数）
- [ ] 打包配 `library` + `libraryTarget: 'umd'`（webpack）
- [ ] 入口顶部加运行时 publicPath
- [ ] dev server 开启 CORS
- [ ] `unmount` 清理实例、定时器、事件监听
- [ ] 避免写 `body`、`*` 等全局样式
- [ ] 弹窗不挂 `document.body`
- [ ] 支持独立运行（`!__POWERED_BY_QIANKUN__`）

### 10.3 通用

- [ ] 主子应用路由都用 history 模式，服务器配 fallback
- [ ] 资源上 CDN，字体图片用 base64 或注入路径
- [ ] 主应用不 external Vue 到 `window`
- [ ] Vite 子应用操作 `window` 用 `qiankunWindow`
- [ ] 微应用按业务域拆分，不宜过细

## 十一、总结

- 生命周期问题 → 检查导出 + webpack umd 配置 + name 唯一
- 容器问题 → 限定 `container.querySelector` 查找范围 + `start` 时机
- 资源 404 → 运行时/静态 publicPath + 字体图片 base64/CDN
- Vue Router 报错 → 主应用不 external Vue
- 跨域 → dev server / Nginx 配 CORS
- 核心：主应用统一调度，子应用自包含、可独立运行、卸载必清理
