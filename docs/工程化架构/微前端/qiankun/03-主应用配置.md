# qiankun 主应用配置

> 📖 官方文档：[API - 基于路由配置](https://qiankun.umijs.org/zh/api#registermicroappsapps-lifecycles) | [手动加载微应用](https://qiankun.umijs.org/zh/api#loadmicroappapp-configuration)

本文详细讲解主应用侧的全部 API：`registerMicroApps`、`start`、`loadMicroApp` 等。子应用侧的接入改造见 [04-子应用接入](./04-子应用接入.md)。

## 一、registerMicroApps —— 注册子应用

```ts
import { registerMicroApps } from 'qiankun'

registerMicroApps(apps, lifeCycles?)
```

注册微应用的基础信息。当浏览器 URL 变化时，qiankun 会自动检查每个子应用的 `activeRule`，符合规则的应用被激活。

### 1.1 apps 配置项

每个 `RegistrableApp` 的字段如下：

| 字段 | 类型 | 必选 | 说明 |
| --- | --- | --- | --- |
| `name` | `string` | 是 | 子应用名称，**全局唯一**，需与子应用打包配置一致 |
| `entry` | `string` \| `object` | 是 | 子应用入口（HTML 地址或对象配置） |
| `container` | `string` \| `HTMLElement` | 是 | 子应用挂载容器，如 `'#root'` 或 DOM 元素 |
| `activeRule` | `string` \| `function` \| `array` | 是 | 激活规则，URL 匹配时激活该子应用 |
| `loader` | `(loading: boolean) => void` | 否 | loading 状态变化回调 |
| `props` | `object` | 否 | 传给子应用的数据 |

#### entry 的两种写法

```ts
// 写法一：字符串（最常用）—— 子应用部署后的访问地址
entry: '//localhost:7105'

// 写法二：对象 —— 手动指定 html/scripts/styles
// 此时 publicPath 会被设置为 '/'
entry: {
  html: '<div id="app"></div>',           // html 内容字符串
  scripts: ['//localhost:7105/main.js'],  // 手动指定脚本
  styles: ['//localhost:7105/style.css']  // 手动指定样式
}
```

### 1.2 activeRule 匹配规则

`activeRule` 支持字符串、函数、数组。**字符串做前缀匹配**，函数返回 `true` 表示激活，数组满足其一即激活。

```ts
// ① 字符串前缀匹配
activeRule: '/app1'
//   ✅ https://app.com/app1
//   ✅ https://app.com/app1/anything
//   🚫 https://app.com/app2

// ② 动态参数
activeRule: '/users/:userId/profile'
//   ✅ https://app.com/users/123/profile
//   ✅ https://app.com/users/123/profile/sub
//   🚫 https://app.com/users//profile       (参数为空)

// ③ hash 模式
activeRule: '/pathname/#/hash'
//   ✅ https://app.com/pathname/#/hash
//   ✅ https://app.com/pathname/#/hash/nested
//   🚫 https://app.com/pathname#/another-hash

// ④ 数组（满足其一即激活）
activeRule: ['/app1', '/app2']

// ⑤ 函数（自定义判断逻辑）
activeRule: (location) => location.pathname.startsWith('/app1')
```

> 💡 **提示：** `activeRule` 与子应用路由前缀 `routerBase` 要保持一致，否则会出现「子应用激活了但路由匹配不到」的 404 问题。详见 [09-路由管理](./09-路由管理.md)。

### 1.3 props —— 传递数据给子应用

```ts
registerMicroApps([{
  name: 'sub-vue',
  entry: '//localhost:7105',
  container: '#subapp-viewport',
  activeRule: '/sub-vue',
  props: {
    routerBase: '/sub-vue',          // 路由前缀
    token: 'xxx',                    // 静态数据
    onLogin: (user) => { /* ... */ } // 回调函数
  }
}])
```

子应用在 `mount(props)` 中接收。这是主应用向子应用传数据的主要方式，详见 [06-应用间通信](./06-应用间通信.md)。

### 1.4 lifeCycles —— 全局生命周期钩子

```ts
registerMicroApps(apps, {
  beforeLoad:   (app) => console.log('加载前', app.name),
  beforeMount:  (app) => console.log('挂载前', app.name),
  afterMount:   (app) => console.log('挂载后', app.name),
  beforeUnmount:(app) => console.log('卸载前', app.name),
  afterUnmount: (app) => console.log('卸载后', app.name)
})
```

每个钩子也支持数组形式（按顺序执行）：

```ts
beforeMount: [
  (app) => { /* 第一件事 */ },
  (app) => { /* 第二件事 */ }
]
```

> 💡 **提示：** 这些是主应用侧的钩子，用于埋点、日志、loading 控制；子应用自身的生命周期（`bootstrap`/`mount`/`unmount`）见 [05-生命周期](./05-生命周期.md)。

> 🔗 **进阶：** 若子应用列表来自后端接口（按权限下发、可配置、菜单驱动），见 [12-动态注册子应用](./12-动态注册子应用.md)。

## 二、start —— 启动 qiankun

```ts
import { start } from 'qiankun'

start(opts?)
```

`Options` 全部配置项如下：

| 配置项 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |
| `prefetch` | `boolean` \| `'all'` \| `string[]` \| `function` | `true` | 预加载策略 |
| `sandbox` | `boolean` \| `object` | `true` | JS 沙箱与样式隔离 |
| `singular` | `boolean` \| `function` | `true` | 是否单实例模式 |
| `fetch` | `Function` | `window.fetch` | 自定义 fetch 方法 |
| `getPublicPath` | `(entry) => string` | - | 自定义 publicPath |
| `getTemplate` | `(tpl) => string` | - | 修改子应用 HTML 模板 |
| `excludeAssetFilter` | `(assetUrl) => boolean` | - | 过滤不需 qiankun 处理的资源 |

### 2.1 prefetch —— 预加载策略

```ts
// 默认 true：第一个子应用 mount 完成后，预加载其他子应用资源
start({ prefetch: true })

// start 后立即预加载所有子应用
start({ prefetch: 'all' })

// 第一个 mounted 后，只预加载指定子应用
start({ prefetch: ['sub-vue', 'sub-vue2'] })

// 完全自定义首屏 / 次屏应用
start({
  prefetch: (apps) => ({
    criticalAppNames: ['sub-vue'],   // 首屏应用（立即加载）
    minorAppsName: ['sub-vue2']      // 次屏应用（空闲加载）
  })
})
```

详见 [10-加载与预加载策略](./10-加载与预加载策略.md)。

### 2.2 sandbox —— 沙箱与样式隔离

```ts
start({
  sandbox: {
    strictStyleIsolation: true,        // Shadow DOM 严格样式隔离
    experimentalStyleIsolation: true   // 实验性样式隔离（选择器改写）
  }
})
```

- 默认 `sandbox: true`：开启 JS 沙箱，保证**单实例**场景下子应用之间的样式隔离
- 但**无法保证**主应用与子应用之间、**多实例**场景下的样式隔离
- 两种样式隔离方案二选一，详见 [08-样式隔离](./08-样式隔离.md)

> ⚠️ **注意：** `strictStyleIsolation` 和 `experimentalStyleIsolation` 不要同时开启。Shadow DOM 隔离有较多适配成本，不是无脑可用的方案。

### 2.3 singular —— 单实例模式

```ts
start({ singular: true })   // 默认：同一时间只渲染一个子应用
start({ singular: false })  // 多实例：允许多个子应用同时渲染
```

- **单实例**（默认）：路由切换时，旧子应用先 `unmount`，新子应用再 `mount`
- **多实例**：多个子应用可同时存在，但样式隔离、沙箱要额外配置，坑较多

### 2.4 fetch —— 自定义请求方法

```ts
// 例如加鉴权头、统一错误处理
start({
  fetch: (url, ...args) => {
    return window.fetch(url, {
      ...args,
      headers: { 'Authorization': `Bearer ${getToken()}` }
    })
  }
})
```

### 2.5 excludeAssetFilter —— 资源过滤

指定某些动态加载的资源不被 qiankun 劫持处理（比如想让某些脚本走原生加载）：

```ts
start({
  excludeAssetFilter: (assetUrl) => {
    // 这些资源 qiankun 不处理
    return assetUrl.includes('amap.com')   // 高德地图等
  }
})
```

### 2.6 getTemplate —— 修改 HTML 模板

用于过滤子应用 HTML 中的异常脚本（如运营商注入的脚本）：

```ts
start({
  getTemplate(tpl) {
    // 移除模板中指定的异常脚本
    return tpl.replace(/some-bad-script\.js/, '')
  }
})
```

## 三、setDefaultMountApp —— 设置默认子应用

```ts
import { setDefaultMountApp } from 'qiankun'

// 主应用启动后默认进入 /homeApp 对应的子应用
setDefaultMountApp('/homeApp')
```

用户访问主应用根路径时，自动跳转到指定的子应用。

## 四、runAfterFirstMounted —— 首次挂载后执行

```ts
import { runAfterFirstMounted } from 'qiankun'

// 第一个子应用 mount 后执行（如启动监控、埋点）
runAfterFirstMounted(() => {
  startMonitor()
})
```

## 五、loadMicroApp —— 手动加载子应用

适用于**不与路由关联**、需要手动控制加载/卸载的场景（如一个页面内同时放多个子应用组件）。

```ts
import { loadMicroApp } from 'qiankun'

const microApp = loadMicroApp({
  name: 'sub-vue',
  entry: '//localhost:7105',
  container: '#some-container',
  props: { brand: 'qiankun' }
}, {
  // configuration 配置项与 start 一致
  sandbox: true,
  singular: false   // ⚠️ loadMicroApp 场景 singular 默认 false，与 start 不同！
})
```

返回的 `MicroApp` 实例方法：

| 方法 | 说明 |
| --- | --- |
| `mount()` | 挂载子应用 |
| `unmount()` | 卸载子应用 |
| `update(customProps)` | 更新传给子应用的 props |
| `getStatus()` | 获取当前状态（见下表） |

`getStatus()` 返回的状态值：

```text
NOT_LOADED → LOADING_SOURCE_CODE → NOT_BOOTSTRAPPED → BOOTSTRAPPING
→ NOT_MOUNTED → MOUNTING → MOUNTED → UNMOUNTING → (回到 NOT_MOUNTED)
→ UPDATING / UNLOADING / SKIP_BECAUSE_BROKEN / LOAD_ERROR
```

### loadMicroApp 的 update 能力

若希望主应用能 `update` 子应用 props，子应用需额外导出 `update` 钩子：

```ts
// 子应用
export async function mount(props) {
  renderApp(props)
}

export async function update(props) {
  renderPatch(props)   // 增量更新，而非整体重渲染
}
```

### registerMicroApps vs loadMicroApp

| 维度 | `registerMicroApps` | `loadMicroApp` |
| --- | --- | --- |
| 触发方式 | 路由变化自动激活 | 手动调用 |
| 子应用类型 | 带路由的完整应用 | 不带路由的业务组件 |
| `singular` 默认 | `true`（单实例） | `false`（多实例） |
| 适用场景 | 后台多页面切换 | 页面内嵌组件、面板 |

> 💡 **提示：** 官方建议微应用不宜拆分过细，按**业务域**拆分。判断标准：若两个微应用频繁通信，说明它们服务于同一业务，合并成一个更合适。

## 六、prefetchApps —— 手动预加载

仅在 `loadMicroApp` 手动加载场景下使用；基于路由自动激活的场景直接配 `start({ prefetch })` 即可。

```ts
import { prefetchApps } from 'qiankun'

prefetchApps([
  { name: 'app1', entry: '//localhost:7001' },
  { name: 'app2', entry: '//localhost:7002' }
])
```

## 七、全局错误处理

```ts
import { addGlobalUncaughtErrorHandler, removeGlobalUncaughtErrorHandler } from 'qiankun'

const handler = (event) => {
  console.error('[qiankun] 未捕获异常:', event)
  // 可在此做降级处理，如显示错误页
}

addGlobalUncaughtErrorHandler(handler)

// 不再需要时移除
removeGlobalUncaughtErrorHandler(handler)
```

> 💡 **提示：** 子应用加载失败（如 entry 404、生命周期导出错误）都会通过这个全局处理器抛出，生产环境务必注册一个，避免白屏无提示。

## 八、总结

| API | 作用 |
| --- | --- |
| `registerMicroApps` | 注册子应用 + 全局生命周期钩子 |
| `start` | 启动 qiankun，配置沙箱/预加载/单实例等 |
| `setDefaultMountApp` | 设置默认进入的子应用 |
| `runAfterFirstMounted` | 首次挂载后执行副作用 |
| `loadMicroApp` | 手动加载子应用（多实例、组件级） |
| `prefetchApps` | 手动预加载资源 |
| `addGlobalUncaughtErrorHandler` | 全局错误兜底 |
