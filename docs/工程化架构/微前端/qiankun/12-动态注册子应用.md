# qiankun 动态注册子应用

> 📖 官方文档：[registerMicroApps](https://qiankun.umijs.org/zh/api#registermicroappsapps-lifecycles) | [loadMicroApp](https://qiankun.umijs.org/zh/api#loadmicroappapp-configuration)
> 🔗 关联：[03-主应用配置](./03-主应用配置.md) | [09-路由管理](./09-路由管理.md) | [10-加载与预加载策略](./10-加载与预加载策略.md)

[03-主应用配置](./03-主应用配置.md) 里，子应用列表是**写死在主应用代码里**的（静态注册）。但真实的中后台项目里，子应用的「名称、地址、激活路由、是否启用」往往由**运营后台或权限中心**下发——这就需要**动态注册**：主应用启动时先请求接口拿到子应用列表，再调用 `registerMicroApps`。

本文是 03 的进阶实战篇，聚焦「子应用列表来自后端」这个场景。

## 一、为什么需要动态注册

静态注册（把 apps 写死在代码里）有三个局限：

| 局限 | 说明 |
| --- | --- |
| 增删子应用要发版 | 新上一个子应用，主应用要改代码、打包、重新部署 |
| 无法按权限/租户差异化 | 不同角色看到的子应用不同，写死无法实现 |
| 环境地址易错 | 测试/预发/生产地址不同，硬编码容易配错 |

动态注册把「子应用清单」变成**运行时数据**，主应用代码只负责「按清单调度」，实现了主应用与子应用列表的解耦。

```text
静态注册：                    动态注册：
┌─────────────┐              ┌─────────────┐
│ apps 写死在   │              │ 请求接口 →    │
│ 代码里        │              │ 拿到 apps    │
└──────┬──────┘              └──────┬──────┘
       │                            │
       ▼                            ▼
  registerMicroApps            registerMicroApps
```

## 二、动态注册三步走

核心是：**请求配置 → 注册 → 启动**。推荐在 `start()` 之前完成全部注册（qiankun 的标准用法，避免路由激活时机问题）。

```ts
import { registerMicroApps, start } from 'qiankun'

async function setupQiankun() {
  // ① 从后端获取子应用列表
  const apps = await fetchMicroApps()

  // ② 动态注册（apps 结构与静态注册完全一致）
  registerMicroApps(apps)

  // ③ 启动
  start()
}

setupQiankun()
```

> ⚠️ **注意：** 顺序别反——`registerMicroApps` 应在 `start` 之前、且全部子应用注册完再 start。start 之后才注册的子应用，要等下一次路由变化才会被激活，且会触发重复注册警告。

## 三、后端配置数据约定

后端返回的每个子应用对象，字段就是 [registerMicroApps 的标准配置](./03-主应用配置.md#一、registermicroapps-——-注册子应用)：

```json
// GET /api/micro-apps 响应
[
  {
    "name": "sub-vue",
    "entry": "//cdn.example.com/sub-vue/",
    "activeRule": "/sub-vue",
    "container": "#subapp-viewport",
    "props": { "routerBase": "/sub-vue" }
  },
  {
    "name": "sub-list",
    "entry": "//cdn.example.com/sub-list/",
    "activeRule": "/list",
    "container": "#subapp-viewport",
    "props": { "routerBase": "/list" }
  }
]
```

| 字段 | 后端下发要点 |
| --- | --- |
| `name` | 全局唯一，必须与子应用打包配置一致 |
| `entry` | 建议下发**线上 CDN 地址**，开发环境可用 localhost |
| `activeRule` | 路由前缀，需与子应用 `routerBase` 对应 |
| `container` | 主应用里的挂载容器，通常固定 |
| `props` | 传给子应用的数据（token、主题、回调等） |

> 💡 **提示：** `container` 通常与子应用无关（就是主应用那个挂载点），可以让主应用在注册前补上，后端只关心业务相关的 `name/entry/activeRule`。

## 四、按权限过滤子应用

最常见的动态场景：**不同用户能看到不同的子应用**。拿到全量列表后，按当前用户权限过滤：

```ts
import { registerMicroApps, start } from 'qiankun'

async function setupQiankun() {
  const [allApps, userInfo] = await Promise.all([
    fetchMicroApps(),
    fetchUserInfo()
  ])

  // 只注册当前用户有权限的子应用
  const apps = allApps.filter(app =>
    userInfo.permissions.includes(app.name)   // 用 app.name 当权限码
  )

  registerMicroApps(apps)
  start()
}
```

过滤发生在**注册之前**，没注册的子应用不会加载任何资源，权限控制是彻底的（而不只是把菜单藏起来）。

## 五、用子应用列表驱动导航菜单

动态注册后，主应用的左侧菜单不应再写死，而应由 `apps` 动态生成：

```vue
<!-- 主应用 Menu.vue -->
<template>
  <nav class="menu">
    <router-link
      v-for="app in microApps"
      :key="app.name"
      :to="app.activeRule"
    >
      {{ app.title }}
    </router-link>
  </nav>
</template>

<script setup lang="ts">
import { ref } from 'vue'

// 后端下发的子应用元信息
interface MicroAppConfig {
  name: string
  entry: string
  activeRule: string
  title?: string
  props?: Record<string, unknown>
}

const microApps = ref<MicroAppConfig[]>([])

async function init() {
  const apps = await fetchMicroApps()
  microApps.value = apps        // 驱动菜单渲染
  registerMicroApps(apps)       // 驱动子应用注册
  start()
}

init()
</script>
```

> 💡 **提示：** 让后端在下发数据里带上 `title`、`icon` 等菜单元信息，主应用直接渲染，实现「运营在后台配一个子应用 → 菜单和路由自动出现」。

## 六、动态增删子应用（运行时变更）

⚠️ 这是 qiankun 的一个**已知短板**：`registerMicroApps` 内部按 `name` 去重，重复注册同名应用会被跳过并打印警告，且 **qiankun 没有提供反注册（unregister）API**。所以「运行时热增删子应用」的能力有限。三种应对策略：

```text
需求                            推荐做法
──────────────────────────────────────────────────────
启动时按权限决定加载哪些          动态 registerMicroApps（本文第二~四节）
偶尔增删（可接受刷新页面）         更新配置后 location.reload() 重走启动流程
不刷新、运行时挂载/卸载            loadMicroApp 手动管理
```

```ts
// 不刷新页面、按需挂载卸载 → 用 loadMicroApp（详见 03 第五节）
import { loadMicroApp } from 'qiankun'

const handle = loadMicroApp({
  name: 'sub-vue',
  entry: '//cdn.example.com/sub-vue/',
  container: '#panel'
})

// 用完卸载
await handle.unmount()
```

> ⚠️ **注意：** 不要试图用「清空数组 + 再次 registerMicroApps」来模拟反注册，qiankun 内部维护的清单无法这样清掉，只会产生重复注册警告和路由混乱。

## 七、注意事项与最佳实践

### 7.1 启动时机

`registerMicroApps` 应在 `start` 之前完成；若主应用挂载容器在路由页里，`start()` 还要等该容器 `mounted` 之后调用，否则找不到挂载点。

```ts
// Vue3：在根组件 onMounted 后启动
import { onMounted } from 'vue'

onMounted(async () => {
  const apps = await fetchMicroApps()
  registerMicroApps(apps)
  start()
})
```

### 7.2 配置接口失败的降级

接口可能超时或挂掉，要有兜底，避免主应用白屏：

```ts
// 本地维护一份最小可用清单
const FALLBACK_APPS = [
  { name: 'sub-home', entry: '//cdn.example.com/home/', activeRule: '/home', container: '#subapp-viewport', props: { routerBase: '/home' } }
]

async function fetchMicroApps() {
  try {
    const res = await fetch('/api/micro-apps')
    if (!res.ok) throw new Error('bad response')
    return await res.json()
  } catch (e) {
    console.error('[qiankun] 子应用配置拉取失败，使用本地兜底', e)
    return FALLBACK_APPS
  }
}
```

### 7.3 activeRule 与配置校验

后端下发的数据不可全信，注册前做基本校验，避免缺字段或规则冲突导致子应用抢路由：

```ts
const apps = rawApps.filter(app => {
  const ok = app.name && app.entry && app.activeRule
  if (!ok) console.warn('[qiankun] 非法子应用配置，已忽略', app)
  return ok
})
```

### 7.4 entry 用线上地址

动态注册的 `entry` 一般是线上 CDN 地址（不是 localhost）。配合 [10-加载与预加载策略](./10-加载与预加载策略.md) 的 `prefetch`，能进一步提速。

## 八、完整实战示例

一个 Vue3 主应用，从接口拉子应用 → 按权限过滤 → 注册启动 → 兜底错误处理：

```ts
// main.ts
import { createApp } from 'vue'
import {
  registerMicroApps,
  start,
  addGlobalUncaughtErrorHandler
} from 'qiankun'
import App from './App.vue'
import { router } from './router'

const app = createApp(App)
app.use(router)
app.mount('#app')

async function bootstrapMicroApps() {
  const [apps, user] = await Promise.all([
    fetchMicroApps(),                          // 内部带降级，见 7.2
    fetchUserInfo().catch(() => ({ permissions: [] }))
  ])

  const visible = apps
    .filter(a => a.name && a.entry && a.activeRule)   // 校验
    .filter(a => user.permissions.includes(a.name))   // 权限

  // 共享给菜单组件（生产环境建议放进 Pinia/Vuex，避免挂在 window）
  window.__MICRO_APPS__ = visible

  registerMicroApps(visible)
  start({ prefetch: true })
}

addGlobalUncaughtErrorHandler(err => {
  console.error('[qiankun] 加载异常', err)
})

bootstrapMicroApps()
```

> 💡 **提示：** 子应用列表通过 `window.__MICRO_APPS__` 或状态管理共享给菜单组件，避免重复请求。生产环境推荐用 Pinia/Vuex。

## 九、总结

- 动态注册 = **请求配置 → registerMicroApps → start**，应在 start 前完成
- 后端下发 `name/entry/activeRule`，主应用补 `container`、按权限过滤
- 导航菜单应由 `apps` 驱动，实现「配置即菜单」
- qiankun **无反注册 API**，运行时增删用 `loadMicroApp`，或刷新重走启动流程
- 接口要降级、配置要校验、`start` 时机要对
