### 一、概述

> 📖 [Nginx gzip 模块](https://nginx.org/en/docs/http/ngx_http_gzip_module.html) ｜ [Vite 压缩插件](https://github.com/anncwb/vite-plugin-compression)

前端打包后的 JS/CSS 文件通常较大（几百 KB 到几 MB），直接传输会**拖慢首屏加载**。压缩（gzip / brotli）能在传输前把文件体积**缩小到原来的 1/3 ~ 1/5**，显著提升加载速度。

大白话：你要寄一箱棉被（很大的 JS 文件），很占地方、运费贵。压缩就像「**抽真空打包**」——把棉被压扁，运过去又快又省。到了对方那里（浏览器）再自动「拆包」还原。

| 你将学到 | 说明 |
| --- | --- |
| 为什么压缩 | 体积大、传输慢 |
| gzip | 最通用的压缩方式 |
| brotli | 更强的新一代压缩 |
| Nginx 配置 | 开启 gzip |
| Vite 预压缩 | 构建时压缩，更快 |

> 💡 **提示：** 开启 gzip 是**性价比最高的前端性能优化**之一——几行配置，体积降 70%，加载快很多。

---

### 二、为什么需要压缩

#### 1. 现代前端文件很大

```javascript
// 一个普通 Vue 项目打包后
dist/assets/index.js      // 350 KB（未压缩）
dist/assets/vendor.js     // 800 KB（未压缩）
```

未压缩，用户首次加载要下载 1MB+，**网络慢时首屏要好几秒**。

#### 2. 压缩的效果

```
index.js:   350 KB  ──gzip──►  120 KB  （降 66%）
vendor.js:  800 KB  ──gzip──►  260 KB  （降 67%）
brotli 还能比 gzip 再小 15%~20%
```

#### 3. 浏览器和服务器如何协作

```
浏览器请求，告诉服务器「我支持 gzip」：
  请求头：Accept-Encoding: gzip, br

服务器压缩后返回：
  响应头：Content-Encoding: gzip
  响应体：压缩后的字节

浏览器自动解压还原，用户无感知
```

> 💡 **提示：** 压缩/解压是**浏览器和服务器自动协商**的，前端代码不用管，只需在服务器开启压缩。

---

### 三、压缩什么、不压缩什么

| 该压缩 | 不该压缩 |
| --- | --- |
| HTML、CSS、JS | 已经压缩过的文件（jpg/png/mp4/zip） |
| JSON、XML、SVG | 字体文件（woff2 已压缩） |
| 纯文本 | 二进制媒体 |

> ⚠️ **注意：** **不要压缩图片、视频、zip**！它们本身就是压缩格式，再压缩不仅没效果，还浪费 CPU。

### 四、Nginx 开启 gzip

#### 基础配置

在 `nginx.conf` 的 `http` 块，或单个 server 里：

```nginx
# 开启 gzip
gzip on;

# 压缩级别（1-9，1 最快压缩比低，9 最慢压缩比高，推荐 5-6）
gzip_comp_level 6;

# 最小压缩文件大小（小于此值不压缩，单位 KB）
gzip_min_length 1k;

# 压缩的文件类型
gzip_types
    text/plain
    text/css
    text/javascript
    application/javascript
    application/json
    application/xml
    image/svg+xml;

# 给代理服务器也缓存压缩结果
gzip_vary on;
```

#### 完整示例

```nginx
http {
    gzip on;
    gzip_comp_level 6;
    gzip_min_length 1k;
    gzip_types text/plain text/css application/javascript application/json image/svg+xml;
    gzip_vary on;

    server {
        listen 80;
        # ...
    }
}
```

#### 生效与验证

```bash
sudo nginx -s reload

# ✅ 验证：看响应头是否有 Content-Encoding: gzip
curl -I -H "Accept-Encoding: gzip" https://example.com/index.js
# 应返回：content-encoding: gzip
```

> 💡 **提示：** `gzip_comp_level` 不是越大越好。**6 是性能和压缩比的平衡点**。设到 9 会让服务器 CPU 飙升，但体积只小一点点。

### 五、实时压缩 vs 预压缩

Nginx gzip 有两种工作方式：

#### 1. 实时压缩（gzip on）

每次请求，Nginx **临时压缩**返回。

- 优点：零配置，文件更新自动生效
- 缺点：每次请求都消耗 CPU

#### 2. 预压缩（gzip_static）

提前把 `.gz` 文件准备好，Nginx 直接返回现成的，**不临时压缩**。

```nginx
# 开启：如果存在 index.js.gz，直接返回它，不临时压缩
gzip_static on;
```

> 💡 **提示：** **预压缩性能更好**（不耗运行时 CPU）。配合 Vite 构建时生成 .gz 文件，是生产环境推荐做法。

### 六、Vite 构建时生成压缩文件

用 `vite-plugin-compression` 在打包时就生成 `.gz` / `.br` 文件：

#### 1. 安装

```bash
npm install vite-plugin-compression -D
```

#### 2. 配置 vite.config.js

```javascript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import compression from 'vite-plugin-compression'

export default defineConfig({
  plugins: [
    vue(),
    // ✅ 生成 gzip 文件
    compression({
      algorithm: 'gzip',
      ext: '.gz',
      threshold: 10240,   // 大于 10KB 才压缩
      deleteOriginFile: false,  // 保留原文件
    }),
    // ✅ 生成 brotli 文件（可选，压缩率更高）
    compression({
      algorithm: 'brotliCompress',
      ext: '.br',
    }),
  ],
})
```

打包后 `dist` 里会多出 `.gz` / `.br` 文件：

```
dist/assets/index.js
dist/assets/index.js.gz    ← 预压缩好的
dist/assets/index.js.br    ← brotli 预压缩
```

#### 3. Nginx 使用预压缩文件

```nginx
# ✅ 优先返回预压缩的 .gz
gzip_static on;

# brotli 需要额外模块 ngx_brotli（见下文）
```

### 七、Brotli（更强的压缩）

Brotli 是 Google 推出的新一代压缩算法，**比 gzip 压缩率更高**（同等质量小 15%~20%），现代浏览器都支持。

#### Brotli 的使用

- 需要给 Nginx 装 **ngx_brotli** 模块（或用自带 brotli 的版本）
- 通常配合 Vite 预压缩生成 `.br` 文件

```nginx
# 启用 brotli（需 ngx_brotli 模块）
brotli on;
brotli_comp_level 6;
brotli_types text/plain text/css application/javascript application/json image/svg+xml;
```

#### gzip vs brotli

| 对比 | gzip | brotli |
| --- | --- | --- |
| 压缩率 | 好 | **更好（小 15%~20%）** |
| 浏览器支持 | 全部 | 现代浏览器（95%+） |
| 兼容性 | 无敌 | 极好 |
| 服务器支持 | Nginx 内置 | 需装模块 |

> 💡 **提示：** **最佳实践：brotli 优先，gzip 兜底**。Nginx 配置好，自动按浏览器能力选最优压缩。

### 八、常见问题与注意事项

#### 1. 开了 gzip 没生效

- `gzip_types` 没包含该文件类型
- 文件小于 `gzip_min_length`
- 请求头没带 `Accept-Encoding`（用 curl 要加 `-H "Accept-Encoding: gzip"`）
- `nginx -s reload` 没 reload

#### 2. gzip_static 不生效

`gzip_static on` 需要文件实际存在 `.gz`。确保 Vite 打包生成了 `.gz`，且和原文件在同一目录。

#### 3. 压缩级别设太高

`gzip_comp_level 9` 会让 CPU 暴涨，**推荐 5-6**。

#### 4. 压缩了图片

别把 `image/jpeg` `image/png` 加进 `gzip_types`——它们本身已压缩，gzip 无效还费 CPU。

#### 5. Content-Encoding 重复

如果上游已经压缩、Nginx 又压缩，会重复。一般静态资源自己管，不会有这问题。

### 九、实际应用场景

| 场景 | 配置 |
| --- | --- |
| 普通 Nginx 部署 | `gzip on` + `gzip_types` |
| 追求极致性能 | Vite 预压缩 + `gzip_static` + brotli |
| 对象存储部署 | 上传时压缩，CDN 直接分发 .gz |
| 大流量站点 | brotli 优先，省带宽 |

### 十、总结

| 要点 | 说明 |
| --- | --- |
| 作用 | 文件体积降 60%~80%，加速加载 |
| gzip | 最通用，Nginx 内置 |
| brotli | 更强，现代浏览器支持 |
| 该压 | HTML/CSS/JS/JSON/SVG |
| 不该压 | 图片/视频/字体/zip |
| 实时 vs 预压缩 | 预压缩（gzip_static）性能更好 |
| 推荐级别 | gzip_comp_level 6 |

一句话：**前端部署必开 gzip，配 Vite 预压缩 + gzip_static 性能最佳，有条件上 brotli**。
