### 一、概述

> 📖 [Nginx 反向代理](https://nginx.org/en/docs/beginners_guide.html#proxy) ｜ [浏览器同源策略（MDN）](https://developer.mozilla.org/zh-CN/docs/Web/Security/Same-origin_policy)

前端项目部署后，几乎都要**调用后端 API**。而前端和后端往往不在同一个域名/端口下，直接请求会触发**跨域（CORS）问题**。用 Nginx 反向代理，能把前端和后端「**统一到同一个域名下**」，从根本上规避跨域。

大白话：跨域是浏览器的「**门禁**」——你的前端在 `a.com`，后端在 `b.com`，浏览器不让前端直接调 b.com。反向代理像在 `a.com` 开个「**中转窗口**」：前端调 `a.com/api/xxx`，Nginx 偷偷转发到后端 `b.com/xxx`，浏览器看到的是同源请求，不拦了。

| 你将学到 | 说明 |
| --- | --- |
| 跨域问题 | 为什么会跨域 |
| 反向代理 | Nginx proxy_pass 转发 |
| 解决跨域 | 反向代理 vs CORS |
| 实战配置 | 前端 + API 同域名 |

> 💡 **提示：** 反向代理是前端部署**最常用的跨域解法**，比让后端配 CORS 更省事、更通用。

### 二、跨域问题

#### 1. 什么是跨域

浏览器的**同源策略**：脚本默认只能请求**同源**（协议 + 域名 + 端口都相同）的接口，不同源就叫跨域。

```
前端： https://www.example.com
后端： https://api.example.com        ← 域名不同，跨域
后端： http://www.example.com:8080     ← 协议、端口不同，跨域
后端： https://www.example.com         ← ✅ 同源
```

#### 2. 跨域的表现

```javascript
// 前端在 www.example.com，请求 api.example.com
fetch('https://api.example.com/users')
// ❌ 浏览器报错：CORS policy，请求被拦
```

#### 3. 解决跨域的两种思路

| 方案 | 做法 | 优劣 |
| --- | --- | --- |
| **CORS** | 后端响应头允许跨域 | 后端要改，每个接口都配 |
| **反向代理** | Nginx 把前后端统一到同源 | ✅ 前端搞定，后端不用改 |

> 💡 **提示：** 反向代理让前端和 API「看起来」是同一个网站，**浏览器根本感知不到跨域**，所以不拦。这是最干净的方案。

### 三、反向代理原理

#### 1. 正向 vs 反向代理

| 类型 | 代理谁 | 例子 |
| --- | --- | --- |
| **正向代理** | 代理**客户端**（客户端知道要访问真实服务器） | VPN、翻墙 |
| **反向代理** | 代理**服务端**（客户端不知道真实服务器是谁） | Nginx 转发到后端 |

#### 2. 反向代理工作流程

```
浏览器 ──请求 www.example.com/api/users──► Nginx（www.example.com）
                                              │ proxy_pass
                                              ▼
                                          后端服务器
                                       http://localhost:8080/users
```

浏览器以为自己请求的是 `www.example.com`（同源），实际上 Nginx 把 `/api` 开头的请求**转发**给了后端。浏览器全程不知道后端的存在，所以**没有跨域问题**。

### 四、Nginx 反向代理配置

#### 核心：proxy_pass

```nginx
server {
    listen 80;
    server_name www.example.com;
    root /usr/share/nginx/html/myapp;

    # ✅ /api/ 开头的请求，转发到后端
    location /api/ {
        proxy_pass http://localhost:8080/;   # ✅ 注意结尾斜杠
    }

    # 其他请求走前端
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

#### proxy_pass 的斜杠陷阱（重要！）

结尾**有没有斜杠**，转发行为完全不同：

```nginx
# 配置1：proxy_pass 带 /（替换 /api/）
location /api/ {
    proxy_pass http://localhost:8080/;
}
# 请求 /api/users → 转发到 http://localhost:8080/users
#                                  ↑ /api/ 被替换掉

# 配置2：proxy_pass 不带 /（保留 /api/）
location /api/ {
    proxy_pass http://localhost:8080;
}
# 请求 /api/users → 转发到 http://localhost:8080/api/users
#                                  ↑ /api/ 被保留
```

| 配置 | 请求 `/api/users` 转发到 |
| --- | --- |
| `proxy_pass http://backend/;`（带 /） | `http://backend/users`（去掉 /api） |
| `proxy_pass http://backend;`（不带 /） | `http://backend/api/users`（保留 /api） |

> ⚠️ **注意：** **proxy_pass 带不带斜杠是 90% 反向代理 bug 的根源**。后端接口路径不带 `/api` 前缀就用带斜杠的；带 `/api` 前缀就用不带斜杠的。务必和后端确认！

### 五、完整的代理配置（生产级）

```nginx
server {
    listen 80;
    server_name www.example.com;
    root /usr/share/nginx/html/myapp;
    index index.html;

    # ✅ API 反向代理
    location /api/ {
        proxy_pass http://localhost:8080/;

        # 把客户端真实信息传给后端
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时设置
        proxy_connect_timeout 30s;
        proxy_read_timeout 60s;
    }

    # WebSocket 支持（如有需要）
    location /ws/ {
        proxy_pass http://localhost:8080/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # 前端路由
    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

#### proxy_set_header 的作用

后端拿到的请求是 Nginx 转发的，默认看不到客户端真实 IP。这些 header 把真实信息传给后端：

| 头 | 作用 |
| --- | --- |
| `Host` | 原始主机名 |
| `X-Real-IP` | 客户端真实 IP |
| `X-Forwarded-For` | 代理链路 IP |
| `X-Forwarded-Proto` | 原始协议（http/https） |

> 💡 **提示：** 反向代理后，后端用 `request.getRemoteAddr()` 拿到的是 Nginx 的 IP。要拿真实客户端 IP，得读 `X-Forwarded-For` 头。

### 六、前端配合：请求基础路径

前端代码里，**API 请求用相对路径**（同源）：

```javascript
// ✅ 用相对路径 /api，让 Nginx 代理
axios.get('/api/users')

// ❌ 不要写死后端地址（否则又跨域了）
axios.get('http://localhost:8080/users')
```

#### Vite 开发环境的代理

开发时（vite dev server）也要配代理，保持和生产一致：

```javascript
// vite.config.js
export default defineConfig({
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
      },
    },
  },
})
```

> 💡 **提示：** **开发和生产用同样的 `/api` 前缀 + 代理**，前端代码不用区分环境，最省心。

### 七、反向代理 vs CORS

| 对比 | 反向代理 | CORS |
| --- | --- | --- |
| 谁改 | 前端（Nginx） | 后端 |
| 前端代码 | 相对路径 `/api` | 直接写后端地址 |
| 后端改动 | 不用 | 加 CORS 响应头 |
| Cookie | 同源，自动带 | 要配 `credentials` |
| 灵活性 | 高 | 中 |
| 推荐 | **生产首选** | 后端对外开放时 |

#### 什么时候用 CORS 而非代理

- 后端 API 要**对外开放**（如第三方调用、移动端）→ 用 CORS
- 前端和后端**同源部署**（同一域名）→ 反向代理
- 第三方 SDK、CDN 资源 → 服务端用 CORS

详见 [[../java/网络/HTTP协议]] 的跨域部分。

### 八、常见问题与注意事项

#### 1. 502 Bad Gateway

Nginx 转发失败（后端没启动、端口错、后端挂了）。检查后端服务是否正常、`proxy_pass` 地址对不对。

#### 2. 接口路径多/少了 /api

`proxy_pass` 斜杠问题（见上文）。和后端确认接口到底带不带 `/api` 前缀。

#### 3. Cookie / 登录态丢失

跨域代理后 Cookie 可能因域名不同丢失。配合 `proxy_set_header` + 后端配置 Cookie 的 domain，或用 Token（见 [[../java/网络/Cookie与Session与Token]]）。

#### 4. WebSocket 代理

普通 `proxy_pass` 不支持 WebSocket。要加 `Upgrade` / `Connection` 头（见上文配置）。

#### 5. 生产 vs 开发路径不一致

开发用 vite proxy，生产用 Nginx proxy。**前端代码统一用 `/api` 相对路径**，两边代理配置对齐即可。

### 九、实际应用场景

| 场景 | 配置 |
| --- | --- |
| 前端 + 后端同域名 | `/api/` 代理到后端 |
| 多个微服务 | `/api/user/`、`/api/order/` 代理到不同后端 |
| WebSocket | 加 Upgrade 头 |
| 负载均衡 | `proxy_pass` 到 upstream 多台（见负载均衡） |
| 接口加缓存/限流 | 在代理层做 |

```nginx
# 多后端微服务示例
location /api/user/   { proxy_pass http://localhost:8081/; }
location /api/order/  { proxy_pass http://localhost:8082/; }
location /api/pay/    { proxy_pass http://localhost:8083/; }
```

### 十、总结

| 要点 | 说明 |
| --- | --- |
| 跨域根因 | 浏览器同源策略 |
| 反向代理 | Nginx 把 API 和前端统一同源 |
| 核心指令 | `proxy_pass` |
| 斜杠陷阱 | 带 `/` 替换路径，不带 `/` 保留路径 |
| 前端配合 | 请求用相对路径 `/api` |
| 推荐度 | 生产环境首选跨域方案 |

一句话：**前端请求 `/api/xxx`，Nginx 用 `proxy_pass` 转发到后端，跨域问题消失**。
