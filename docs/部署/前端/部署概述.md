### 一、概述

> 📖 [Web 部署（MDN）](https://developer.mozilla.org/zh-CN/docs/Learn/Common_questions/Tools_and_setup/How_do_you_host_your_website_on_the_web) ｜ [Nginx 文档](https://nginx.org/en/docs/)

前端部署，指的是把**构建打包后的前端代码（一堆静态文件）放到服务器上，让用户能通过浏览器访问**。它是前端开发流程的最后一环，也是项目真正「上线」的关键。

大白话：你在本地 `npm run build` 之后，得到一个 `dist` 文件夹（里面是 HTML、CSS、JS、图片）。部署就是把这个文件夹**搬到一台有公网 IP 的服务器上**，配一个 Web 服务器（如 Nginx）对外提供访问——用户访问域名时，服务器把这些文件发给浏览器。

| 你将学到 | 说明 |
| --- | --- |
| 前端部署的本质 | 静态资源的托管 |
| 部署流程 | 打包 → 上传 → 配置 → 访问 |
| 常见部署方式 | Nginx / CDN / Docker / 静态托管平台 |
| 本系列导航 | 各篇知识点 |

> 💡 **提示：** 前端部署的核心难点不在「搬文件」，而在**路由刷新、缓存、跨域、HTTPS、加速**这些细节。本系列逐个击破。

### 二、前端部署的本质：静态资源托管

#### 1. 前端项目打包后是什么

```
npm run build  →  dist/
                    ├── index.html        ← 入口 HTML
                    ├── assets/
                    │   ├── index-a3f2.js   ← 带 hash 的 JS
                    │   ├── index-b8c1.css  ← 带 hash 的 CSS
                    │   └── logo-d4e5.png
                    └── favicon.ico
```

打包产物是**纯静态文件**（HTML/CSS/JS/图片），没有任何后端逻辑。

#### 2. 部署的本质

部署 = 把这些静态文件放到服务器，用 **Web 服务器软件**（Nginx、Apache）监听端口，**当浏览器请求时把对应文件返回**。

```
浏览器请求 https://example.com
        │
        ▼
   服务器（Nginx 监听 443）
        │ 找到 index.html
        ▼
   返回 HTML → 浏览器解析 → 再请求 JS/CSS → 渲染页面
```

> 💡 **提示：** 前端部署和后端部署的本质区别：**前端是「静态文件」，后端是「运行中的程序」**。前端不需要运行环境（Node），只需要一个能返回文件的服务器。

### 三、标准部署流程

```
① 本地开发、构建打包
   npm run build  →  得到 dist/

② 把 dist 上传到服务器
   scp / rsync / CI 自动化

③ 服务器配置 Web 服务器（Nginx）
   指向 dist 目录、配置域名、HTTPS、压缩、缓存...

④ 配置域名解析
   域名 → 服务器 IP（DNS）

⑤ 访问验证
   浏览器打开域名，确认页面正常
```

#### 一条最小命令链

```bash
# 本地打包
npm run build

# 上传到服务器（scp）
scp -r dist/* root@your-server:/usr/share/nginx/html/

# 服务器上重启 Nginx（其实改文件不用重启）
ssh root@your-server "nginx -s reload"
```

### 四、常见部署方式对比

| 方式 | 说明 | 适用 |
| --- | --- | --- |
| **Nginx / Apache** | 自建服务器，自己装 Web 服务器 | **最主流**，本系列重点 |
| **静态托管平台** | Vercel、Netlify、GitHub Pages、Cloudflare Pages | 个人项目、小项目，零配置 |
| **对象存储** | 阿里云 OSS、腾讯云 COS、AWS S3 | 纯静态，配合 CDN |
| **CDN** | 把静态资源分发到全球节点 | 追求加速、大流量 |
| **Docker** | 容器化部署（Nginx + dist 打成镜像） | 标准化、配合 K8s |
| **传统虚拟主机** | FTP 上传文件 | 老旧方式，逐渐淘汰 |

> 💡 **提示：** 企业项目最常见组合：**Nginx（自建）或 对象存储 + CDN**，配合 Docker + CI/CD 自动化。

### 五、Web 服务器：为什么是 Nginx

前端部署几乎都用 **Nginx**，因为：

| 优势 | 说明 |
| --- | --- |
| 高性能 | 能扛高并发，内存占用低 |
| 静态资源强 | 专为返回静态文件优化 |
| 反向代理 | 能转发 API 请求，解决跨域 |
| 配置简单 | 几行配置就能跑 |
| 功能全 | HTTPS、压缩、缓存、负载均衡一体 |

> 💡 **提示：** Nginx 是前端部署的「**瑞士军刀**」。掌握 Nginx 配置，是前端工程师走向全栈/运维的必备技能。详见 [[Nginx部署]]。

### 六、前端部署的几个核心难点

部署不是「文件传上去就行」，这几个问题必须处理好：

| 难点 | 问题 | 解决 | 文档 |
| --- | --- | --- | --- |
| **SPA 路由刷新 404** | history 路由刷新报 404 | `try_files` 回退到 index.html | [[SPA路由部署]] |
| **加载慢** | 文件大，传输慢 | gzip/brotli 压缩 | [[压缩配置]] |
| **更新不生效** | 浏览器用旧缓存 | hash 文件名 + 缓存策略 | [[缓存策略]] |
| **跨域** | 前端调后端接口跨域 | 反向代理 | [[反向代理与跨域]] |
| **不安全** | HTTP 明文 | HTTPS + 证书 | [[HTTPS配置]] |
| **全球慢** | 距离服务器远 | CDN 加速 | [[CDN部署]] |

### 九、部署前的检查清单

部署前对照检查，避免常见坑：

- [ ] `npm run build` 成功，`dist` 生成正常
- [ ] 资源路径正确（相对路径 `/` vs `./`，见 [[CDN部署]] 的 publicPath）
- [ ] 环境变量（API 地址）配置正确（生产环境地址）
- [ ] 路由模式（history 刷新是否 404）
- [ ] 跨域是否解决（API 代理）
- [ ] HTTPS 是否配置
- [ ] 压缩、缓存是否开启
- [ ] 域名解析是否生效

### 十、常见问题与注意事项

#### 1. 部署后白屏

常见原因：
- 资源路径错（`publicPath` 配置不对）→ 检查 `vite.config.js` 的 `base`
- JS 报错 → F12 看控制台
- 路由配置错 → 检查 history 模式

#### 2. 接口 404 / 跨域

前端请求 API 失败，通常是**反向代理没配**或**API 地址错**。见 [[反向代理与跨域]]。

#### 3. 更新代码用户没看到新版本

浏览器缓存 + CDN 缓存。用 hash 文件名 + 合理缓存策略。见 [[缓存策略]]。

#### 4. 服务器放哪里

- `/usr/share/nginx/html/`（Nginx 默认）
- `/var/www/html/`（Apache 默认）
- 自定义目录（配置里 root 指定）

#### 5. 需要装 Node 吗

**生产服务器不需要 Node**！打包产物是静态文件，Nginx 直接返回即可。Node 只在**构建时**需要（可以在本地或 CI 环境构建）。

> 💡 **提示：** 很多人误以为「Vue 项目部署要装 Node」，其实不用。除非你要在服务器上做 SSR（服务端渲染），那才需要 Node 运行环境。
