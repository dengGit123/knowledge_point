### 一、概述

> 📖 [HTTP 缓存（MDN）](https://developer.mozilla.org/zh-CN/docs/Web/HTTP/Caching) ｜ [Vite 构建产物](https://vitejs.dev/guide/build.html)

前端部署有两个永恒主题：**「加载要快」**（用缓存）和**「更新要生效」**（别让用户用旧版本）。缓存策略就是平衡这两者——**让不变的静态资源长期缓存，让会变的入口文件每次都拿最新的**。

大白话：你更新了代码、重新部署，结果用户看到的还是旧页面——因为浏览器用了**缓存**。但完全不缓存，每次都要重新下载，又慢。聪明的做法：**给文件名加 hash，内容变了 hash 就变，浏览器自然下载新的；内容没变的就让它缓存着**。

| 你将学到 | 说明 |
| --- | --- |
| 缓存的价值 | 加速、省流量 |
| 强缓存 vs 协商缓存 | 两种缓存机制 |
| hash 文件名 | 解决「更新不生效」的关键 |
| Nginx 缓存配置 | 实战配置 |
| 入口 HTML 的特殊处理 | 永远不缓存 |

> 💡 **提示：** 前端缓存的精髓一句话：**带 hash 的静态资源长缓存，index.html 永不缓存**。

---

### 二、为什么需要缓存

#### 1. 不缓存的坏处

每次访问都重新下载所有 JS/CSS，**首屏慢、浪费带宽**。

#### 2. 缓存的坏处

更新代码后，**浏览器用旧缓存，用户看不到新版本**。

#### 3. 解决思路

让**内容不变的文件**长期缓存，让**会变的入口**每次都验证。靠 **hash 文件名** 完美解决。

---

### 三、强缓存 vs 协商缓存

浏览器缓存分两层，详见 [[../java/网络/HTTP协议]] 的缓存章节，这里结合前端部署讲。

#### 1. 强缓存

命中强缓存时，**浏览器直接用本地副本，连请求都不发**（状态码 `200 from disk/memory cache`）。

控制头：

| 头 | 说明 |
| --- | --- |
| `Cache-Control: max-age=31536000` | 缓存 1 年（秒） |
| `Cache-Control: immutable` | 永不重新验证 |
| `Expires` | 过期时间（老方案） |

#### 2. 协商缓存

强缓存过期后，浏览器**发请求问服务器「变了吗」**，没变返回 `304 Not Modified`，变了返回 `200` + 新内容。

控制头：

| 头 | 说明 |
| --- | --- |
| `Last-Modified` / `If-Modified-Since` | 按修改时间 |
| `ETag` / `If-None-Match` | 按内容指纹（更精确） |

#### 流程对比

```
强缓存流程：
  浏览器 ── 有缓存且未过期？ ── 是 ── 直接用（200 from cache，不发请求）
                          └ 否 ── 走协商缓存

协商缓存流程：
  浏览器 ── 问服务器「变了吗」 ── 没变 ── 304（用本地缓存）
                              └ 变了 ── 200（返回新内容）
```

### 四、hash 文件名：解决更新不生效（核心！）

#### 1. 问题

如果文件名固定（如 `app.js`），更新后浏览器仍用缓存的旧 `app.js`，看不到新版本。

#### 2. hash 文件名的解法

Vite / Webpack 打包时，给文件名**加上内容的 hash**：

```
内容变了 → hash 变 → 文件名变 → 浏览器认为是「新文件」→ 下载新的
内容没变 → hash 不变 → 文件名不变 → 浏览器用缓存
```

```
第一次打包：  index-a3f2b1.js
改了代码后：  index-c8d9e4.js   ← hash 变了，新文件名
```

> 💡 **提示：** 这就是为什么 Vite 默认产物是 `assets/index-[hash].js`。**hash 文件名让「文件名」成为内容的指纹**，完美解决了缓存与更新的矛盾。

#### 3. 为什么能放心长缓存

因为内容变了文件名就变，**旧的缓存文件永远不会被错误地用于新版本**。所以带 hash 的文件可以放心缓存一年。

#### Vite 的产物结构

```
dist/
├── index.html                      ← ❗ 不带 hash（每次都要最新的）
└── assets/
    ├── index-a3f2b1.js             ← 带 hash（长缓存）
    ├── index-b8c1d2.css            ← 带 hash（长缓存）
    └── logo-d4e5f6.png             ← 带 hash（长缓存）
```

### 五、正确的缓存策略

基于 hash 文件名，前端缓存策略就一句话：**静态资源长缓存，HTML 不缓存**。

| 文件 | 策略 | 原因 |
| --- | --- | --- |
| **index.html** | **永不缓存**（no-cache） | 它引用了带 hash 的 JS/CSS，必须每次拿最新的，才能找到新的 hash 文件 |
| **带 hash 的 JS/CSS/图片** | **长缓存 1 年**（immutable） | 内容变了 hash 就变，旧缓存不会被误用 |
| 接口数据 | 不缓存或协商缓存 | 实时数据 |

#### 为什么 HTML 不能缓存

```
index.html 里写着：<script src="/assets/index-a3f2b1.js">
更新后变成：    <script src="/assets/index-c8d9e4.js">

如果 HTML 被缓存，用户看到的还是旧 HTML（引用旧 hash 的 JS）→ 看不到新代码
所以 HTML 必须每次拿最新的！
```

### 六、Nginx 缓存配置实战

```nginx
server {
    listen 80;
    root /usr/share/nginx/html/myapp;
    index index.html;

    # ✅ 入口 HTML：不缓存（每次都验证）
    location = /index.html {
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }

    # ✅ 带 hash 的静态资源：长缓存 1 年
    location ~* \.(js|css|png|jpg|jpeg|gif|svg|ico|woff2?)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    location / {
        try_files $uri $uri/ /index.html;
    }
}
```

#### 配置解析

| 配置 | 作用 |
| --- | --- |
| `expires 1y` | 强缓存 1 年（`Cache-Control: max-age=31536000`） |
| `immutable` | 告诉浏览器「这文件永不变」，连刷新都不验证 |
| `no-cache` | 每次都要验证（协商缓存），不用强缓存 |
| `no-store` | 完全不缓存 |

> 💡 **提示：** `immutable` 是关键——告诉浏览器「带 hash 的文件内容永不变」，即使用户按 F5 刷新，也不会重新验证，**极致加速**。

#### 验证

```bash
# HTML 应该是 no-cache
curl -I https://example.com/index.html
# cache-control: no-cache, no-store, must-revalidate

# JS 应该是长缓存
curl -I https://example.com/assets/index-a3f2b1.js
# cache-control: public, immutable
# expires: ...（1年后）
```

### 七、更新部署的正确姿势

更新前端代码时，确保**只覆盖带 hash 的新文件，别动 HTML 的缓存逻辑**：

```bash
# ✅ 增量部署：新 hash 文件覆盖，旧的可保留（用户老的 HTML 还可能引用）
rsync -avz dist/ /usr/share/nginx/html/myapp/

# 旧文件要不要删？
# - 短期内别删：用户浏览器里缓存的旧 HTML 还可能引用旧 hash 文件
# - 过段时间再清理：等所有用户都访问过新 HTML 了
```

> ⚠️ **注意：** 部署新版本后，**别立即删除旧的 hash 文件**——否则正在用旧版本的用户会突然加载失败（他们的 HTML 还引用着旧 hash 文件名）。等几分钟~几小时再清理。

### 八、Vite 相关配置

Vite 默认就给 JS/CSS 加 hash。文件名格式可在 `vite.config.js` 调整：

```javascript
import { defineConfig } from 'vite'

export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        // 自定义带 hash 的文件名格式
        chunkFileNames: 'assets/[name]-[hash].js',
        entryFileNames: 'assets/[name]-[hash].js',
        assetFileNames: 'assets/[name]-[hash].[ext]',
      },
    },
  },
})
```

> 💡 **提示：** 默认配置已经够用，除非有特殊需求（如固定某些文件名）才需要改。

### 九、常见问题与注意事项

#### 1. 更新代码用户看不到新版本

- HTML 被强缓存了 → HTML 配 `no-cache`
- CDN 缓存了 → 部署后刷新 CDN 缓存（见 [[CDN部署]]）

#### 2. immutable 的前提

`immutable` 只对**带 hash 的文件**安全。给不带 hash 的文件（如 index.html）设 immutable 会锁死旧版本。

#### 3. 调试时缓存烦人

开发环境关闭缓存（Vite dev server 默认不缓存）。浏览器 DevTools → Network → 勾选「Disable cache」调试时禁用缓存。

#### 4. CDN 缓存叠加

静态资源上 CDN 后，CDN 也会缓存。带 hash 文件可让 CDN 长缓存，但 HTML 要短缓存。见 [[CDN部署]]。

#### 5. Service Worker 的坑

如果用了 PWA（Service Worker），它会有自己的缓存层，更新逻辑更复杂（要更新 SW 版本）。这里不展开。

### 十、实际应用场景

| 场景 | 策略 |
| --- | --- |
| Vite/Vue/React 部署 | hash 静态资源长缓存，HTML 不缓存 |
| 频繁更新的活动页 | HTML no-cache，资源 immutable |
| 不带 hash 的老项目 | 用协商缓存（ETag），避免强缓存 |
| CDN 部署 | 同策略，CDN 节点也按此缓存 |

### 十一、总结

| 要点 | 说明 |
| --- | --- |
| 缓存价值 | 加速、省流量 |
| 强缓存 | 直接用本地，不发请求 |
| 协商缓存 | 问服务器变了吗，304 复用 |
| **hash 文件名** | 内容变→文件名变，解决更新不生效 |
| **核心策略** | 带 hash 资源长缓存（immutable），HTML 不缓存 |
| Nginx | `expires 1y` + `immutable`；HTML `no-cache` |

一句话：**带 hash 的静态资源 `immutable` 长缓存 1 年，index.html `no-cache` 每次验证**——这就是前端缓存的黄金法则。
