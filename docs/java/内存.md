# JVM 内存模型

> 官方文档：[The Java Virtual Machine Specification - Runtime Data Areas](https://docs.oracle.com/javase/specs/jvms/se21/html/jvms-2.html#jvms-2.5) ｜ 菜鸟教程：[Java 内存模型](https://www.runoob.com/java/jvm-memory-model.html)

---

## 一、概述

Java 之所以"一次编写，到处运行"，并且能**自动管理内存**（不需要像 C/C++ 那样手动 `malloc`/`free`），核心就在于 **JVM（Java 虚拟机）** 在运行程序时，会在内存中划分出若干区域，分别存放不同类型的数据。

> 💡 **通俗理解**：JVM 就像一个"厂房"，里面划分了不同的"车间"：
> - **堆**是"成品仓库"——所有 `new` 出来的对象都堆在这里。
> - **栈**是"工人的工作台"——每个线程（工人）有自己的工作台，上面放着正在处理的方法和局部变量。
> - **方法区**是"图纸资料室"——存放所有类的"设计图纸"（类信息）、常量、静态变量。
> - **程序计数器**是"工人做到第几步的小本本"——记录当前执行到哪一行代码。

理解 JVM 内存模型，是理解**对象存储、垃圾回收、内存溢出（OOM）、并发**的基础，也是 Java 面试的高频考点。

## 二、运行时数据区域（核心）

JVM 在执行 Java 程序时，会把它管理的内存划分为以下几个区域（基于 JDK 8+ 的规范）：

```
┌──────────────────────────────────────────────────────────────┐
│                    JVM 运行时数据区                            │
├──────────────────────┬───────────────────────────────────────┤
│                      │   程序计数器（PC Register）  线程私有   │
│                      │                                       │
│                      ├───────────────────────────────────────┤
│                      │   虚拟机栈（VM Stack）      线程私有   │
│   线程隔离区域        │   ┌─────────────┐                     │
│ （每个线程独有一份）  │   │  栈帧（方法）│ ← 方法调用入栈       │
│                      │   └─────────────┘                     │
│                      ├───────────────────────────────────────┤
│                      │   本地方法栈（Native Stack）线程私有   │
├──────────────────────┼───────────────────────────────────────┤
│                      │                       ┌─────────────┐ │
│   线程共享区域        │   堆（Heap）          │   新生代     │ │
│ （所有线程共用一份）  │   （对象实例存放处）   │ Eden S0 S1  │ │
│                      │                       ├─────────────┤ │
│                      │                       │   老年代     │ │
│                      │                       └─────────────┘ │
│                      ├───────────────────────────────────────┤
│                      │   方法区（Method Area / 元空间）       │
│                      │   类信息、常量、静态变量、JIT 代码      │
└──────────────────────┴───────────────────────────────────────┘
```

### 1. 程序计数器（Program Counter Register）

| 项目 | 说明 |
|------|------|
| **作用** | 记录当前线程正在执行的字节码**行号 / 地址** |
| **线程私有** | ✅ 是（每个线程一个，互不干扰） |
| **特点** | **唯一一个不会发生 OOM（内存溢出）的区域** |
| **原因** | 如果线程执行的是 Java 方法，记录字节码地址；如果是 Native 方法，则为空（Undefined） |

> 💡 **为什么需要它？** CPU 在多线程间切换执行，切换回来时必须知道"刚才执行到哪一行了"，程序计数器就是用来记住这个位置的。

### 2. Java 虚拟机栈（VM Stack）

每个方法在被调用时，都会创建一个**栈帧（Stack Frame）** 压入虚拟机栈；方法执行完毕后栈帧出栈。**线程私有**。

栈帧包含四个部分：

```
┌─────────────────────────────────┐
│            栈帧（Stack Frame）   │
├─────────────────────────────────┤
│ ① 局部变量表                    │ ← 存方法参数和方法内的局部变量
│   （基本类型直接存值，引用类型存地址）│   （slot 为单位）
├─────────────────────────────────┤
│ ② 操作数栈                      │ ← 计算的临时工作区（如 a + b 的中间结果）
├─────────────────────────────────┤
│ ③ 动态链接                      │ ← 指向运行时常量池中方法的引用
├─────────────────────────────────┤
│ ④ 方法返回地址                  │ ← 方法结束后跳回调用处的位置
└─────────────────────────────────┘
```

**两种异常**：

| 异常 | 触发条件 | 典型场景 |
|------|---------|---------|
| `StackOverflowError` | 栈深度超过虚拟机允许的最大深度 | **递归没有终止条件**、递归太深 |
| `OutOfMemoryError` | 栈扩展时无法申请到足够内存 | 创建过多线程 |

```java
// ❌ 经典的死递归，会导致 StackOverflowError
public void recursion() {
    recursion(); // 没有终止条件，栈帧无限压入，最终栈溢出
}
```

### 3. 本地方法栈（Native Method Stack）

与虚拟机栈作用类似，区别在于：虚拟机栈为 **Java 方法**服务，本地方法栈为 **Native 方法**（用 C/C++ 实现的本地方法，如 `Object.hashCode()`、`System.currentTimeMillis()` 底层）服务。**线程私有**。

> 💡 在 HotSpot 虚拟机中，本地方法栈和虚拟机栈是**合二为一**的。

### 4. Java 堆（Heap）

| 项目 | 说明 |
|------|------|
| **作用** | 存放**几乎所有对象实例和数组**（`new` 出来的东西都在这里） |
| **线程共享** | ✅ 是 |
| **GC 主战场** | ✅ 垃圾收集器管理的主要区域（又称 GC 堆） |
| **异常** | `OutOfMemoryError: Java heap space` |

堆是垃圾收集器管理的最大一块区域，通常被划分为**新生代**和**老年代**：

```
┌──────────────────────── 堆（Heap）────────────────────────────┐
│                          │                                   │
│       新生代（Young）     │         老年代（Old）              │
│  ┌────────┬──────┬──────┐│                                   │
│  │  Eden  │  S0  │  S1  ││                                   │
│  │ (8)    │ (1)  │ (1)  ││                                   │
│  └────────┴──────┴──────┘│                                   │
│   新生代 = Eden : S0 : S1 = 8 : 1 : 1                        │
│   新对象先放 Eden，GC 后存活对象进 Survivor，多次存活进老年代 │
└───────────────────────────────────────────────────────────────┘
```

### 5. 方法区（Method Area）

| 项目 | 说明 |
|------|------|
| **作用** | 存放**类信息、常量、静态变量、JIT 编译后的代码**等 |
| **线程共享** | ✅ 是 |
| **别名** | 别称"非堆"（Non-Heap），与堆逻辑上分离 |

**方法区的实现演变（重要考点）**：

| JDK 版本 | 方法区实现 | 位置 | 异常 |
|---------|-----------|------|------|
| JDK 7 及以前 | **永久代（PermGen）** | JVM 进程内存的一部分，有固定上限 | `OOM: PermGen space` |
| **JDK 8+** | **元空间（Metaspace）** | **使用本地内存（直接内存）**，默认只受系统内存限制 | `OOM: Metaspace` |

> ⚠️ **注意（易错点）**：JDK 8 用**元空间**取代了永久代，并且元空间使用的是**本地内存（机器的物理内存）**，不再是 JVM 堆内存的一部分。这就是为什么 JDK 8 之后很少再看到 `PermGen space` 溢出，而可能看到 `Metaspace` 溢出（比如动态加载了太多类）。

**字符串常量池的位置变化**：

| JDK 版本 | 字符串常量池位置 |
|---------|----------------|
| JDK 6 | 方法区（永久代） |
| JDK 7+ | **移到了堆中** |

```java
// JDK 8 下，字符串常量池在堆中，因此 intern() 会影响堆
String s = new String("a"); // 堆中创建对象，常量池中（若不存在）也有 "a"
```

### 6. 运行时常量池（Runtime Constant Pool）

是方法区的一部分，存放**编译期生成的各种字面量和符号引用**，在类加载后进入运行时常量池。除了编译期内容，运行期间也可以动态放入（如 `String.intern()`）。

### 五大区域对比总表

| 区域 | 线程私有 | 存储内容 | 异常类型 |
|------|:-------:|---------|---------|
| 程序计数器 | ✅ | 字节码行号 | 无（唯一不溢出） |
| 虚拟机栈 | ✅ | 栈帧、局部变量 | StackOverflowError / OOM |
| 本地方法栈 | ✅ | Native 方法服务 | StackOverflowError / OOM |
| 堆 | ❌ | 对象实例、数组 | OOM: Java heap space |
| 方法区 | ❌ | 类信息、常量、静态变量 | OOM: Metaspace（JDK 8+） |

## 三、对象的创建过程（new 关键字背后发生了什么）

当我们写下 `Person p = new Person("张三");` 时，JVM 做了这些事：

```
① 类加载检查
   检查 Person 类是否已加载、解析、初始化，没有则先执行类加载
        │
        ▼
② 分配内存
   在堆中为新对象划分内存。两种方式：
   • 指针碰撞：内存规整时，移动分界指针即可
   • 空闲列表：内存碎片化时，维护可用块列表查找
        │
        ▼
③ 内存空间初始化为零值
   将分配的内存（不含对象头）置零，使实例变量无需赋初值就能用默认值
        │
        ▼
④ 设置对象头
   设置 Mark Word（哈希码、GC 分代年龄、锁信息）和类型指针
        │
        ▼
⑤ 执行 <init> 构造方法
   执行程序员编写的构造方法，给成员变量赋真实值
        │
        ▼
⑥ 引用指向对象
   栈中的引用变量 p 指向堆中刚创建的对象
```

> 💡 **对应到前面学过的知识**：第③步解释了为什么**成员变量有默认值**（int=0、引用=null），因为内存先被零值初始化了；第⑤步就是**构造函数**做的事，详见 [构造函数.md](./构造函数.md)。

## 四、对象的内存布局

在堆中创建的一个对象，由三部分组成：

```
┌──────────────────────────────────────────┐
│                对象（Object）             │
├──────────────────────────────────────────┤
│  ① 对象头（Header）                      │
│     • Mark Word：哈希码、GC 年龄、锁标志  │
│     • 类型指针（Klass Pointer）：指向类元数据│
├──────────────────────────────────────────┤
│  ② 实例数据（Instance Data）             │
│     • 真正存储的成员变量值                │
├──────────────────────────────────────────┤
│  ③ 对齐填充（Padding）                   │
│     • 凑成 8 字节的整数倍（HotSpot 要求）  │
└──────────────────────────────────────────┘
```

- **对象头**：包含 `Mark Word`（存储运行时数据，如哈希码、GC 分代年龄、锁状态标志——这是 **synchronized 锁升级**的基础）和**类型指针**（指向方法区中该类的元数据，从而知道这个对象是什么类型）。
- **实例数据**：对象真正存储的有效信息，即各成员变量的值。
- **对齐填充**：HotSpot 要求对象大小必须是 8 字节的整数倍，不够就补齐。

## 五、对象的引用类型（四种引用）

JDK 1.2 引入了四种引用级别，强度从高到低：

| 引用类型 | 说明 | GC 回收时机 | 典型应用 |
|---------|------|------------|---------|
| **强引用（Strong）** | 普通赋值 `Object o = new Object()` | **只要强引用还在，绝不回收**（即使 OOM 也不回收） | 日常绝大多数对象 |
| **软引用（SoftReference）** | `SoftReference<Object>` | **内存不足时**才回收 | 缓存（如图片缓存） |
| **弱引用（WeakReference）** | `WeakReference<Object>` | **下一次 GC 时**必然回收（无论内存够不够） | `WeakHashMap`、防止内存泄漏 |
| **虚引用（PhantomReference）** | `PhantomReference<Object>` | 随时可能被回收，**唯一作用**是对象被回收时收到系统通知 | 跟踪对象被 GC 的状态，配合 `ReferenceQueue` |

```java
import java.lang.ref.*;

// 强引用：最常见
Object strong = new Object();

// 软引用：内存不足时才回收，适合做缓存
SoftReference<byte[]> soft = new SoftReference<>(new byte[1024 * 1024]);

// 弱引用：下次 GC 必回收
WeakReference<Object> weak = new WeakReference<>(new Object());

// 虚引用：必须配合引用队列，get() 永远返回 null
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantom = new PhantomReference<>(new Object(), queue);
```

> 💡 **软引用 vs 弱引用的区别（常考）**：软引用尽可能久地保留（直到内存不足），弱引用只能存活到下一次 GC。所以软引用适合做**缓存**，弱引用适合做**临时关联**（如 `WeakHashMap` 的 key 用弱引用，避免 Map 持有对象导致无法回收）。

## 六、垃圾回收（Garbage Collection, GC）简介

JVM 自动管理内存的核心就是**垃圾回收**。

### 1. 如何判断对象"已死"（可被回收）

| 方法 | 说明 | 是否采用 |
|------|------|:-------:|
| 引用计数法 | 给对象加引用计数，为 0 则可回收 | ❌（无法解决循环引用） |
| **可达性分析算法** | 从 GC Roots 出发，不可达的对象即为可回收 | ✅ JVM 采用 |

**GC Roots（垃圾回收的根节点）** 包括：
- 虚拟机栈中**局部变量表**引用的对象（方法中的局部变量）
- 方法区中**类静态属性**引用的对象（static 变量）
- 方法区中**常量**引用的对象
- 本地方法栈中 **JNI（Native 方法）** 引用的对象

```
        GC Roots（根）
        ├── obj1 ──→ obj2 ──→ obj3     （可达，存活）
        │
        └── obj4

   obj5 ──→ obj6     （从 GC Roots 不可达，且互相循环引用）
                  ↑   仍会被判定为可回收 ✅（可达性分析能解决循环引用）
```

### 2. 垃圾回收算法

| 算法 | 过程 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| **标记-清除** | 先标记可回收对象，再统一清除 | 简单 | 产生**内存碎片** | 老年代 |
| **标记-复制** | 将存活对象复制到另一半，清空原来 | 无碎片、高效 | 浪费一半内存 | **新生代** |
| **标记-整理** | 标记后将存活对象向一端移动，清理边界外 | 无碎片 | 移动成本高 | 老年代 |
| **分代收集** | 新生代用复制算法，老年代用标记-清除/整理 | 综合最优 | —— | 现代主流 |

> 💡 **为什么新生代用复制算法？** 因为新生代对象**朝生夕灭**（绝大多数很快变成垃圾），存活对象少，复制成本低；而老年代存活率高，复制不划算，用标记-整理。

### 3. 分代回收机制

- **新生代（Young）**：分为 Eden、Survivor 0、Survivor 1（比例 8:1:1）。新对象分配在 Eden；Minor GC 后存活对象进 Survivor；经过多次 GC（默认 15 次）仍存活，进入老年代。
- **老年代（Old）**：存放长期存活的对象和大对象。Major GC / Full GC 回收。

| GC 类型 | 回收区域 | 频率 | 速度 |
|--------|---------|------|------|
| Minor GC（Young GC） | 新生代 | 频繁 | 快 |
| Major GC / Full GC | 老年代 / 整个堆 | 较少 | 慢（会 STW 停顿） |

### 4. 常见垃圾收集器

| 收集器 | 作用区域 | 特点 |
|--------|---------|------|
| Serial / Serial Old | 新生代 / 老年代 | 单线程，适合客户端 |
| Parallel Scavenge / Parallel Old | 新生代 / 老年代 | 多线程并行，注重**吞吐量**（JDK 8 默认） |
| CMS（Concurrent Mark Sweep） | 老年代 | 注重**低停顿**，已逐步淘汰 |
| **G1（Garbage First）** | 整堆（分 Region） | JDK 9+ 默认，兼顾吞吐与停顿 |
| ZGC / Shenandoah | 整堆 | 超低延迟（< 10ms），大内存友好 |

## 七、内存溢出（OOM）的常见类型与排查

| 异常信息 | 出错区域 | 常见原因 |
|---------|---------|---------|
| `StackOverflowError` | 虚拟机栈 | 递归太深（无终止条件） |
| `OutOfMemoryError: Java heap space` | 堆 | 创建大量对象、内存泄漏、大集合不清 |
| `OutOfMemoryError: Metaspace` | 方法区（元空间） | 动态加载过多类（如大量动态代理） |
| `OutOfMemoryError: Direct buffer memory` | 直接内存 | NIO 的 ByteBuffer 用尽直接内存 |
| `OutOfMemoryError: unable to create new native thread` | 系统 | 创建线程数过多 |

> ⚠️ **内存溢出（OOM）vs 内存泄漏（Memory Leak）**：
> - **内存泄漏**：对象不再使用，但仍然被引用着无法被 GC 回收（如静态集合一直 add 不 remove）。泄漏积累会导致 OOM。
> - **内存溢出**：内存不够用，程序无法继续分配内存。内存泄漏是 OOM 的常见原因之一。

## 八、直接内存（Direct Memory）

直接内存**不是** JVM 运行时数据区的一部分，也不是 Java 虚拟机规范中定义的内存区域，但也被频繁使用：

- **NIO** 引入的基于通道和缓冲区的 IO 方式，通过 `ByteBuffer.allocateDirect()` 分配**堆外内存**，避免在 Java 堆和 Native 堆之间来回复制数据，提升 IO 性能。
- 不受堆大小限制，但受机器物理内存限制，可能 `OOM: Direct buffer memory`。

```java
import java.nio.ByteBuffer;

// 分配直接内存（堆外内存），NIO 高性能 IO 场景使用
ByteBuffer buffer = ByteBuffer.allocateDirect(1024 * 1024); // 1MB
```

## 九、栈与堆的协作（综合示例）

通过一个具体例子，把栈和堆串起来理解：

```java
public class MemoryDemo {
    public static void main(String[] args) {      // main 栈帧入栈
        int x = 10;                                // x 存在 main 栈帧的局部变量表
        Student s = new Student("张三", 18);        // s（引用）在栈，Student 对象在堆
        s.study();                                 // study 栈帧入栈
    }                                              // 方法结束，栈帧出栈
}

class Student {
    String name;   // name 引用在堆的对象中，指向堆中字符串对象
    int age;       // age 直接存在堆的对象实例数据中

    public Student(String name, int age) {
        this.name = name;
        this.age = age;
    }

    public void study() {                          // study 栈帧入栈
        int hours = 2;                             // hours 在 study 栈帧局部变量表
        System.out.println(name + " 学习了 " + hours + " 小时");
    }                                              // study 栈帧出栈，hours 随之消失
}
```

```
   线程的虚拟机栈                          堆（Heap）
  ┌──────────────┐                  ┌─────────────────────────┐
  │ main 栈帧    │                  │ Student 对象            │
  │  x = 10      │     ──引用──→    │  name ──→ "张三"(String)│
  │  s ─────────────────────────→  │  age = 18               │
  ├──────────────┤                  └─────────────────────────┘
  │ study 栈帧   │
  │  hours = 2   │
  │  this ────┐  │
  └──────────┼──┘
             └──→（指向堆中的 Student 对象）
```

> 💡 **关键结论**：
> - **局部变量**（方法内的 `x`、`hours`、引用 `s`）存在**栈**中。
> - **对象实例**（`new Student`、`"张三"` 字符串）存在**堆**中。
> - 栈中的 `s` 只是一个**引用（地址）**，指向堆中的真实对象。

## 十、面试常见问题

### Q1：简述 JVM 的运行时数据区域

分为五块：**程序计数器、虚拟机栈、本地方法栈**（三者线程私有）；**堆、方法区**（两者线程共享）。程序计数器是唯一不会 OOM 的区域；堆存放对象实例，是 GC 主战场；方法区在 JDK 8 后由元空间（Metaspace）实现，使用本地内存。

### Q2：堆和栈有什么区别？

| 维度 | 栈 | 堆 |
|------|----|----|
| 存储内容 | 局部变量、栈帧、方法调用 | 对象实例、数组 |
| 线程 | 私有 | 共享 |
| 生命周期 | 随方法调用自动释放 | 等 GC 回收 |
| 异常 | StackOverflowError | OOM: Java heap space |
| 访问速度 | 快（仅次于寄存器） | 较慢 |
| 空间 | 小 | 大 |

### Q3：JDK 8 为什么用元空间替换永久代？

1. 永久代大小固定（`-XX:MaxPermSize`），容易 OOM（`PermGen space`），尤其在使用大量动态类（反射、代理、JSP）的场景。
2. 元空间使用**本地内存**，大小只受系统限制，降低了 OOM 风险。
3. 合并了 JRockit 和 HotSpot 的设计（JRockit 本就没有永久代）。

### Q4：为什么 JDK 7 把字符串常量池从方法区移到堆中？

1. 永久代的 GC 效率低（Full GC 才回收），导致常量池中大量字符串无法及时回收。
2. 移到堆中后，可以在 Minor GC 时就被回收，提升内存利用效率。
3. `intern()` 方法在堆中实现更灵活。

### Q5：对象创建的过程是怎样的？

类加载检查 → 分配内存 → 内存初始化为零值 → 设置对象头 → 执行构造方法 `<init>` → 返回引用。这解释了成员变量为什么有默认值（内存被零值初始化）。

### Q6：怎么判断一个对象可以被回收？什么是 GC Roots？

JVM 用**可达性分析算法**：从 GC Roots 出发，沿着引用链向下搜索，不可达的对象即为可回收。GC Roots 包括：栈中局部变量引用的对象、方法区静态属性引用的对象、常量引用的对象、Native 方法引用的对象。它能解决引用计数法无法处理的**循环引用**问题。

### Q7：强引用、软引用、弱引用、虚引用有什么区别？

强度依次递减。**强引用**永不回收（除非 OOM）；**软引用**内存不足时回收（适合缓存）；**弱引用**下次 GC 必回收（如 WeakHashMap）；**虚引用**形同虚设，只为对象被回收时收到通知。

### Q8：Minor GC 和 Full GC 的区别？

Minor GC 只回收新生代，频率高、速度快；Full GC 回收整个堆（含老年代和方法区），频率低、速度慢、会 STW（Stop The World）停顿。应尽量减少 Full GC。

## 十一、总结

| 知识点 | 核心要点 |
|--------|---------|
| 五大区域 | 程序计数器、虚拟机栈、本地方法栈（私有）；堆、方法区（共享） |
| 不溢出的区域 | 程序计数器（唯一） |
| 对象存储 | 实例在**堆**，局部变量在**栈** |
| 方法区演变 | 永久代（JDK 7-）→ 元空间（JDK 8+，本地内存） |
| 对象创建 | 加载检查→分配内存→零值初始化→设对象头→构造方法 |
| 对象布局 | 对象头（Mark Word + 类型指针）+ 实例数据 + 对齐填充 |
| 四种引用 | 强 > 软 > 弱 > 虚 |
| 判活算法 | 可达性分析（GC Roots），非引用计数 |
| 分代 GC | 新生代复制算法，老年代标记-清除 / 整理 |

**记住三句话**：

1. **`new` 出来的都在堆，局部变量都在栈，类信息和常量在方法区。**
2. **JDK 8 之后没有永久代，方法区用元空间实现，元空间在本地内存。**
3. **判断对象死亡用可达性分析（从 GC Roots 出发），垃圾回收分代进行（新生代复制、老年代整理）。**
