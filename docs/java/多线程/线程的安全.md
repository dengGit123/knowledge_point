# 线程的安全

> 📖 官方教程：[Memory Consistency Properties / Atomic Access (Oracle)](https://docs.oracle.com/javase/tutorial/essential/concurrency/atomic.html)
> 📖 官方 API：[java.util.concurrent.atomic (Java Platform SE 21)](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/atomic/package-summary.html)

## 一、概述

"线程安全"指的是：**多线程同时访问某段代码/某个对象时，无论调度顺序如何，都能得到正确的结果**。线程不安全的代码，在并发下会出现数据错乱、丢失更新、读到脏值等问题。

本文从理论（三大特性）、到工具（原子类、CAS）、再到典型问题（ABA、ThreadLocal、死锁），系统讲清楚"线程安全"这件事。具体的锁机制用法见 [线程的同步](./线程的同步.md)。

---

## 二、线程安全三大特性

这是理解所有线程安全问题的核心理论框架。

### 1. 原子性（Atomicity）

操作**不可分割**——要么全部执行完，要么完全不执行，中间不会被其他线程打断。

- **破坏示例**：`i++` 看着是一行，实际是"读取 → 加 1 → 写回"三步，并发下会丢失更新。
- **保证手段**：`synchronized`、`Lock`、`AtomicXxx` 原子类。

### 2. 可见性（Visibility）

一个线程对共享变量的修改，**其他线程能立即看到**。

- **破坏示例**：每个 CPU 核有自己的缓存（工作内存），线程 A 改了变量但还留在自己的缓存里，线程 B 读到的是旧值。
- **保证手段**：`volatile`、`synchronized`、`final`。

### 3. 有序性（Ordering）

程序执行的顺序符合预期，**防止编译器/CPU 为了优化而"指令重排序"**，导致出乎意料的结果。

- **破坏示例**：双重检查单例中，对象的"分配内存 → 初始化 → 赋值给引用"可能被重排序，导致别的线程拿到"半初始化"的对象。
- **保证手段**：`volatile`（禁止重排序）、`synchronized`、`happens-before` 规则。

| 特性 | 含义 | 常见破坏 | 保证手段 |
| --- | --- | --- | --- |
| 原子性 | 操作不可分割 | `i++` | `synchronized`/`Lock`/`AtomicXxx` |
| 可见性 | 修改对其他线程立即可见 | CPU 缓存不一致 | `volatile`/`synchronized`/`final` |
| 有序性 | 防止指令重排序 | 双重检查单例的坑 | `volatile`/`synchronized` |

## 三、Java 内存模型（JMM）简介

三大特性（尤其可见性、有序性）的根源，是 **Java 内存模型（JMM）**：

- 每个线程有自己的**工作内存**（对应 CPU 缓存/寄存器），共享变量存在**主内存**中；
- 线程不能直接读写主内存，必须先把变量**复制到工作内存**，修改后再**刷回主内存**；
- 这就导致了**可见性问题**（修改没及时刷回）和**重排序问题**（编译器/CPU 优化打乱顺序）。

JMM 通过 **happens-before 规则** 定义了"哪些操作的结果对另一些操作可见"，是判断线程安全与否的理论依据。

> 💡 **提示：** `volatile`、`synchronized` 等关键字，本质上都是通过 JMM 规定的**内存屏障**来保证可见性和有序性的。

## 四、原子类（Atomic）

`java.util.concurrent.atomic` 包提供了**基于 CAS** 的原子类，**无需加锁**就能实现原子操作。

### 1. 常用原子类

| 类型 | 示例 | 用途 |
| --- | --- | --- |
| 基本类型 | `AtomicInteger` | 原子 int |
| 基本类型 | `AtomicLong` | 原子 long |
| 基本类型 | `AtomicBoolean` | 原子 boolean |
| 引用类型 | `AtomicReference` | 原子对象引用 |
| 数组类型 | `AtomicIntegerArray` | 原子数组 |
| 字段更新器 | `AtomicIntegerFieldUpdater` | 原子更新某字段 |
| 累加器（JDK 8） | `LongAdder` | 高并发计数（优于 AtomicLong） |

### 2. 用法示例

```java
import java.util.concurrent.atomic.AtomicInteger;

AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet(); // 等价于 ++count，原子操作
count.getAndIncrement(); // 等价于 count++
count.compareAndSet(0, 1); // CAS：期望值为 0 则设为 1
```

## 五、CAS 原理

**CAS（Compare And Swap，比较并交换）** 是原子类的底层实现，是一种**乐观锁**思想。

### 1. 工作过程

```
CAS(V, expected, newValue):
  如果 V 的当前值 == expected:
      把 V 更新为 newValue，返回 true（成功）
  否则:
      返回 false（说明已被其他线程修改，重试）
```

整个"比较+交换"由 **CPU 的原子指令**（如 x86 的 `cmpxchg`）保证原子性。`AtomicInteger` 内部就是用 `Unsafe.compareAndSwapInt` 实现。

### 2. 自旋

CAS 失败（说明有竞争）时通常不会阻塞，而是**循环重试**（自旋），直到成功。这避免了线程切换的开销，但在竞争激烈时自旋会消耗 CPU。

### 3. CAS 的缺点

- **自旋开销**：竞争激烈时空转浪费 CPU；
- **只能保证单个变量的原子性**：多个变量一起原子操作还是得靠锁；
- **ABA 问题**（见下）。

## 六、ABA 问题

CAS 只比较"值是否相等"，但值可能经历了"A → B → A"的变化，CAS 检查时以为没变过，实际上已被改过——这就是 **ABA 问题**。

- **举例**：栈/链表操作中，节点 A 被弹出又压回，CAS 看到"A 还在"就操作，其实中间结构已变。
- **解决**：用 `AtomicStampedReference`，给值附加一个**版本号**——比较时连版本号一起比，`A1 → B2 → A3`，版本号不同就不会误判。

> 💡 **提示：** 大部分场景（如简单计数）ABA 没有实际影响；涉及"复合结构/指针复用"的场景才需要用 `AtomicStampedReference` 防范。

## 七、ThreadLocal

`ThreadLocal` 不是用来"同步"的，而是反其道而行——**给每个线程一份独立的变量副本**，从根本上**避免共享**，从而避免竞争。

```
                ThreadLocal<T> tl = new ThreadLocal<>();

  线程 A                线程 B              线程 C
   │                     │                   │
   ▼                     ▼                   ▼
┌─────────┐         ┌─────────┐         ┌─────────┐
│ tl: "A" │         │ tl: "B" │         │ tl: "C" │   各自独立副本
└─────────┘         └─────────┘         └─────────┘
```

### 1. 用法

```java
public class Context {
    // 每个线程一份 SimpleDateFormat，避免线程安全问题
    private static final ThreadLocal<java.text.SimpleDateFormat> dateFormat =
            ThreadLocal.withInitial(() -> new java.text.SimpleDateFormat("yyyy-MM-dd"));

    public static String format(java.util.Date date) {
        return dateFormat.get().format(date);
    }

    // 用户上下文示例
    private static final ThreadLocal<String> currentUser = new ThreadLocal<>();
    public static void set(String u) { currentUser.set(u); }
    public static String get() { return currentUser.get(); }
    public static void clear() { currentUser.remove(); }
}
```

### 2. 典型应用

- **数据库连接 / 事务**：每个线程一个 Connection，避免争用；
- **非线程安全对象的复用**：如 `SimpleDateFormat`，给每个线程一份；
- **用户上下文**：保存当前请求的登录用户信息，链路传递。

### 3. 内存泄漏注意

`ThreadLocal` 内部通过 `ThreadLocalMap` 存储：**key 是弱引用，value 是强引用**。在线程长期存活（如线程池）时，若用完不 `remove()`，value 可能无法回收，导致**内存泄漏**。

> ⚠️ **注意：** 在**线程池环境**下，使用完 `ThreadLocal` **必须在 `finally` 中调用 `remove()`** 清理，否则不仅可能内存泄漏，还可能"串数据"——线程被复用，下一个任务读到上一个任务的残留值。

## 八、死锁

### 1. 什么是死锁

多个线程**互相持有对方需要的锁**，谁也不撒手，导致**永久阻塞**。

### 2. 死锁的四个必要条件

| 条件 | 含义 |
| --- | --- |
| 互斥 | 资源同一时刻只能被一个线程占有 |
| 占有并等待 | 持有资源的同时等待新的资源 |
| 不可剥夺 | 不能强行夺走线程已占有的资源 |
| 循环等待 | 多个线程形成首尾相接的等待链 |

### 3. 死锁示例

```java
public class DeadLockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        // 线程 1：先锁 A 再锁 B
        new Thread(() -> {
            synchronized (lockA) {
                System.out.println("T1 持有 A，等待 B");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lockB) { System.out.println("T1 获得 B"); }
            }
        }, "T1").start();

        // 线程 2：先锁 B 再锁 A，形成循环等待
        new Thread(() -> {
            synchronized (lockB) {
                System.out.println("T2 持有 B，等待 A");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lockA) { System.out.println("T2 获得 A"); }
            }
        }, "T2").start();
    }
}
```

### 4. 如何避免

| 方法 | 说明 |
| --- | --- |
| 固定加锁顺序 | 所有线程**按相同顺序**获取多把锁（最常用，破坏"循环等待"） |
| 设置超时 | 用 `tryLock(time)`，超时放弃，避免无限等待 |
| 一次性获取所有锁 | 用 `tryLock` 同时尝试获取，失败则全部释放重试 |
| 减小锁粒度 | 缩小同步范围，缩短持锁时间 |

### 5. 如何排查

> 💡 **提示：** 排查死锁可用 **`jstack`**（打印线程堆栈，会直接标出"Found Java-level deadlock"），或用 **jconsole / jvisualvm** 的死锁检测功能。

## 十、总结

| 知识点 | 关键内容 |
| --- | --- |
| 三大特性 | 原子性、可见性、有序性 |
| JMM | 工作内存/主内存模型，happens-before |
| 原子类/CAS | 乐观锁，无锁原子操作，自旋 |
| ABA | 用 `AtomicStampedReference` 加版本号解决 |
| ThreadLocal | 线程隔离，注意线程池下 `remove()` |
| 死锁 | 四个必要条件，固定加锁顺序避免 |

> 💡 **提示：** 解决线程安全的思路有三条：**加锁（排队）**、**无锁原子类（CAS）**、**避免共享（ThreadLocal / 不可变对象）**。优先用并发工具类（`AtomicXxx`、`ConcurrentHashMap`、`BlockingQueue`），它们已经帮你处理好了大部分安全问题。
