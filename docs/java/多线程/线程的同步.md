# 线程的同步

> 📖 官方教程：[Thread Interference / Synchronization / Guarded Blocks (Oracle)](https://docs.oracle.com/javase/tutorial/essential/concurrency/guardmeth.html)
> 📖 官方 API：[ReentrantLock (Java Platform SE 21)](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/locks/ReentrantLock.html)

## 一、概述

"同步"指的是**控制多个线程对共享资源的访问顺序**，避免它们同时操作同一份数据而出错。当多个线程**读写共享数据**时，就需要同步机制来保证正确性。

本文讲解 Java 最核心的几种同步工具：`synchronized`、`ReentrantLock`、`volatile`、以及线程间协作的 `wait()` / `notify()`。线程安全背后的"三大特性"理论，见 [线程的安全](./线程的安全.md)。

---

## 二、为什么需要同步

看一段经典的**卖票问题**：多线程下 `ticket--` 不是原子操作，会出现卖出同一张票、甚至卖出负数票的情况。

```java
// 线程不安全：多个线程同时操作 ticket
class Ticket implements Runnable {
    private int ticket = 100;
    public void run() {
        while (ticket > 0) {
            try { Thread.sleep(10); } catch (InterruptedException e) {}
            System.out.println(Thread.currentThread().getName()
                    + " 卖出 " + ticket--);
        }
    }
}
```

`ticket--` 实际是"**读取 → 减 1 → 写回**"三步，多个线程交叉执行就会出问题。**同步机制**就是为了让"读-改-写"这种操作"一气呵成"，不被别的线程打断。

---

## 三、synchronized 关键字

`synchronized` 是 **JVM 层面**的内置锁，可修饰方法或代码块。它保证同一时刻**只有一个线程**能进入被保护的区域。

### 1. 三种用法

```java
class Ticket implements Runnable {
    private int ticket = 100;
    private final Object lock = new Object();

    public void run() {
        while (true) {
            // 同步代码块：锁指定对象
            synchronized (lock) {
                if (ticket <= 0) break;
                System.out.println(Thread.currentThread().getName()
                        + " 卖出 " + ticket--);
            }
        }
    }

    // 同步方法：锁的是 this（当前实例）
    public synchronized void sell() {
        if (ticket > 0) System.out.println(ticket--);
    }

    // 静态同步方法：锁的是 Class 对象（Ticket.class）
    public static synchronized void staticSell() { /* ... */ }
}
```

### 2. 锁的是谁（重要）

| 使用形式 | 锁对象 |
| --- | --- |
| 同步代码块 `synchronized(obj)` | 括号里指定的 `obj` |
| 普通同步方法 | `this`（当前实例） |
| 静态同步方法 | 对应的 `Class` 对象 |

> ⚠️ **注意：** `synchronized` 锁的是**对象**，不是"代码"。**必须用同一个锁对象**才能互斥——两个线程用不同的锁对象，同步就形同虚设。

### 3. 可重入性

`synchronized` 是**可重入锁**：同一个线程获得锁后，可以再次进入同一个锁保护的其他代码，不会自己把自己锁死。

```java
synchronized (lock) {
    methodA(); // methodA 内部也有 synchronized(lock)，同一把锁可再次进入
}
```

### 4. 自动释放

进入 `synchronized` 时加锁，**正常退出或抛异常时自动释放锁**，不需要手动 `unlock`——这是它比 `Lock` 用起来省心的地方。

## 四、ReentrantLock（显式锁）

`ReentrantLock` 是 `java.util.concurrent.locks` 包提供的**显式锁**，比 `synchronized` 更灵活。

```java
import java.util.concurrent.locks.ReentrantLock;

class Ticket implements Runnable {
    private int ticket = 100;
    private final ReentrantLock lock = new ReentrantLock(true); // true = 公平锁

    public void run() {
        while (true) {
            lock.lock(); // 加锁
            try {
                if (ticket <= 0) break;
                System.out.println(Thread.currentThread().getName()
                        + " 卖出 " + ticket--);
            } finally {
                lock.unlock(); // 必须在 finally 中释放锁
            }
        }
    }
}
```

### 常用方法

| 方法 | 作用 |
| --- | --- |
| `lock()` | 获取锁（阻塞等待） |
| `unlock()` | 释放锁 |
| `tryLock()` | 尝试获取锁，立即返回成功与否 |
| `tryLock(time, unit)` | 限时尝试获取锁 |
| `lockInterruptibly()` | 可被中断地获取锁 |
| `newCondition()` | 获取条件变量，替代 `wait/notify` |

> ⚠️ **注意：** `lock()` 之后**必须**在 `try-finally` 的 `finally` 里 `unlock()`。如果加锁后抛异常又没释放锁，会导致其他线程永远拿不到锁（死锁）。而且 `lock()` 要放在 `try` **外面**，否则未加锁成功就 `unlock` 会抛 `IllegalMonitorStateException`。

## 五、synchronized vs ReentrantLock

| 对比项 | synchronized | ReentrantLock |
| --- | --- | --- |
| 实现 | JVM 关键字（隐式） | API 类（显式 `lock/unlock`） |
| 锁释放 | 自动（出代码块/异常） | 手动（必须 `finally unlock`） |
| 响应中断 | 否 | `lockInterruptibly()` 可响应 |
| 超时获取 | 否 | `tryLock(time)` 支持 |
| 公平锁 | 只能非公平 | 可配置公平/非公平 |
| 条件变量 | `wait/notify`（单条件） | 多个 `Condition` |
| 性能 | JDK 6 后优化，差距很小 | 灵活性高 |

> 💡 **提示：** 简单场景用 `synchronized`（省心、不易出错）；需要"可中断、超时、公平、多条件"等高级功能时用 `ReentrantLock`。

## 六、volatile 关键字

`volatile` 是一种**轻量级同步机制**，主要解决**可见性**和**有序性**，但**不保证原子性**。

### 1. 两大作用

| 特性 | volatile 是否保证 | 原理 |
| --- | --- | --- |
| 可见性 | ✅ 保证 | 强制读写主内存，禁用工作内存缓存 |
| 有序性 | ✅ 保证 | 插入内存屏障，禁止指令重排序 |
| 原子性 | ❌ 不保证 | `i++` 仍是非原子操作 |

### 2. 典型应用：状态标志位

```java
class Worker implements Runnable {
    private volatile boolean running = true; // 保证可见性

    public void run() {
        while (running) {
            // 持续工作
        }
        System.out.println("线程结束");
    }

    public void shutdown() {
        running = false; // 其他线程修改后，工作线程能立即看到
    }
}
```

### 3. 不保证原子性

```java
// 错误：volatile 不能让 i++ 变成原子操作
private volatile int count = 0;
count++; // 仍是 读取-修改-写入 三步，并发下仍出错

// 正确：需要原子性请用 AtomicInteger 或 synchronized
private final java.util.concurrent.atomic.AtomicInteger count =
        new java.util.concurrent.atomic.AtomicInteger();
count.incrementAndGet();
```

> 💡 **更多：** volatile 背后的"工作内存/主内存"模型、指令重排序、happens-before 等理论，见 [线程的安全](./线程的安全.md)。

## 七、线程间协作：wait() / notify()

`wait()` 和 `notify()` 是 `Object` 的方法，用于**线程间通信**：一个线程等待某个条件，另一个线程把条件准备好后通知它。

### 1. 基本规则

- **必须在 `synchronized` 块内调用**，且**锁对象要和调用对象一致**；
- `wait()` 会**释放锁**并进入等待，被唤醒后要**重新获取锁**才能继续；
- `notify()` 唤醒一个等待线程，`notifyAll()` 唤醒所有。

```java
synchronized (lock) {
    while (!condition) {   // 用 while 防止虚假唤醒
        lock.wait();       // 释放 lock 的锁，等待；唤醒后重新获取锁
    }
    // 条件满足，执行逻辑
}
```

> ⚠️ **注意：** 等待条件**必须用 `while` 循环判断**，不能用 `if`。因为线程被唤醒时，条件可能已经被别的线程消费掉了（虚假唤醒 / 多线程竞争），醒来后必须**再次检查**条件。

### 2. 经典应用：生产者-消费者

```java
import java.util.LinkedList;
import java.util.Queue;

public class ProducerConsumer {
    private final Queue<Integer> queue = new LinkedList<>();
    private final int MAX = 5;
    private final Object lock = new Object();

    // 生产者
    public void produce() throws InterruptedException {
        synchronized (lock) {
            while (queue.size() == MAX) {
                lock.wait(); // 满了，等待消费者消费
            }
            queue.add(1);
            System.out.println("生产，当前数量 " + queue.size());
            lock.notifyAll(); // 通知消费者
        }
    }

    // 消费者
    public void consume() throws InterruptedException {
        synchronized (lock) {
            while (queue.isEmpty()) {
                lock.wait(); // 空了，等待生产者生产
            }
            queue.poll();
            System.out.println("消费，当前数量 " + queue.size());
            lock.notifyAll(); // 通知生产者
        }
    }
}
```

> 💡 **提示：** `notifyAll()` 比 `notify()` 更安全——`notify()` 只随机唤醒一个，可能唤醒"同类"线程导致死等；实际开发中**推荐 `notifyAll()`**。`ReentrantLock` 配合 `Condition` 可以实现更精细的"分别唤醒生产者/消费者"。

## 八、同步的代价与原则

1. **同步有开销**：加锁、争用、上下文切换都会降低性能，**只在必要的临界区加锁**，范围越小越好。
2. **避免死锁**：多把锁要按**固定顺序**获取，详见 [线程的安全](./线程的安全.md)。
3. **能用并发工具类就别手写锁**：`AtomicInteger`、`ConcurrentHashMap`、`BlockingQueue` 等已经封装好了，更安全高效。

---

## 九、总结

| 工具 | 特点 | 适用场景 |
| --- | --- | --- |
| `synchronized` | JVM 内置锁，自动释放 | 简单互斥，最常用 |
| `ReentrantLock` | 显式锁，灵活（中断/超时/公平/多条件） | 需要高级控制 |
| `volatile` | 轻量，保证可见性+有序性，不保证原子 | 状态标志位 |
| `wait/notify` | 线程间协作等待/唤醒 | 生产者-消费者 |

> 💡 **提示：** 同步的本质是"**让多个线程排好队，别抢同一份资源**"。选择工具的原则：**简单优先 `synchronized`，标志位用 `volatile`，高级控制用 `ReentrantLock`，能并发工具类就别手写**。
