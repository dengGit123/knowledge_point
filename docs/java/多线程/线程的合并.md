# 线程的合并

> 📖 官方 API：[Thread.join (Java Platform SE 21)](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/lang/Thread.html#join())
> 📖 官方教程：[Joins (Oracle)](https://docs.oracle.com/javase/tutorial/essential/concurrency/join.html)

## 一、概述

"合并"线程，指的是让**当前线程阻塞等待另一个线程执行完毕**，再继续往下走。核心方法是 `Thread.join()`。

它常用于"**等子任务全部完成后再汇总结果**"的场景——比如主线程派了几个子线程去计算，主线程要等它们都算完，再把结果加起来。

> 💡 **理解名字：** "join（加入/合并）"可以理解为：把另一个线程的执行"合并进"当前线程的时间线——当前线程在这里停下，等那个线程跑完，两条线再合到一起继续。

---

## 二、基本用法

```java
Thread t1 = new Thread(() -> {
    try { Thread.sleep(1000); } catch (InterruptedException e) {}
    System.out.println("t1 完成");
}, "t1");
t1.start();

t1.join(); // main 线程在这里阻塞，等 t1 结束
System.out.println("main 继续（t1 已结束）");
```

执行顺序：`t1.start()` → `t1.join()` 阻塞 main → t1 跑完打印"t1 完成" → `join()` 返回 → main 继续。

---

## 三、三种重载形式

| 方法 | 行为 |
| --- | --- |
| `join()` | **无限等待**，直到目标线程结束 |
| `join(long millis)` | 最多等 `millis` 毫秒，超时就不再等 |
| `join(long millis, int nanos)` | 等待到毫秒+纳秒（精度有限） |

```java
t1.join();        // 一直等 t1 结束
t1.join(500);     // 最多等 500 毫秒，超时就不等了
```

> ⚠️ **注意：** 带时间的 `join()` 超时后**当前线程会继续往下执行**，而**目标线程仍在后台继续跑**——超时只是"我不想再等了"，并不停止目标线程。

---

## 四、join() 期间的状态

调用 `t.join()` 后，当前线程进入 **`WAITING`**（无限等）；调用 `t.join(ms)` 进入 **`TIMED_WAITING`**（限时等）。等到 t 结束（或超时）后，当前线程回到 `RUNNABLE`。

> 💡 **本质：** `join()` 的底层实现其实就是**在当前线程上调用 `wait()` 等待**（等待目标线程对象这个"锁"被 notify）。所以 `join()` 也能被 `interrupt()` 打断。

---

## 五、经典场景：等待多个子线程完成

```java
public class JoinDemo {
    public static void main(String[] args) throws InterruptedException {
        final int N = 3;
        int[] results = new int[N];
        Thread[] threads = new Thread[N];

        for (int i = 0; i < N; i++) {
            final int idx = i;
            threads[i] = new Thread(() -> {
                try { Thread.sleep(500); } catch (InterruptedException e) {}
                results[idx] = idx * 10; // 子任务产出结果
            });
            threads[i].start();
        }

        // 依次 join 每个子线程，等它们全部完成
        for (Thread t : threads) {
            t.join();
        }

        // 这里所有子线程一定都结束了，可以安全地汇总结果
        int sum = 0;
        for (int r : results) sum += r;
        System.out.println("汇总结果：" + sum); // 0 + 10 + 20 = 30
    }
}
```

> 💡 **提示：** `results` 数组在这里是安全的——因为 `join()` 保证了"在 main 汇总时，所有子线程对 `results[idx]` 的写入都已完成"（happens-before 关系），不需要额外加锁。

## 六、join() 与中断

`join()` 期间如果当前线程被 `interrupt()`，会抛 `InterruptedException` 并停止等待：

```java
try {
    t1.join();
} catch (InterruptedException e) {
    // 等待被打断
    Thread.currentThread().interrupt();
}
```

---

## 七、join() vs sleep() vs wait()

| 方法 | 谁等 | 等什么 | 是否释放锁 | 状态 |
| --- | --- | --- | --- | --- |
| `t.join()` | 调用者线程 | 目标线程 `t` 执行结束 | （底层用 wait） | WAITING / TIMED_WAITING |
| `Thread.sleep(ms)` | 当前线程 | 一段时间 | 不释放 | TIMED_WAITING |
| `obj.wait()` | 当前线程 | 被 `notify`/超时唤醒 | **释放** `obj` 的锁 | WAITING / TIMED_WAITING |

---

## 八、join() vs CountDownLatch

"等一批线程完成"这件事，`join()` 和 `CountDownLatch` 都能做到，但适用场景不同：

| 对比项 | `join()` | `CountDownLatch` |
| --- | --- | --- |
| 适用对象 | 等待**具体的 Thread 对象**结束 | 等待**若干个事件**完成（不一定是线程） |
| 灵活性 | 线程必须真的结束 | 任意位置 `countDown()`，任务内部通知 |
| 线程池场景 | 不适用（线程池线程不"结束"） | **适用**（见 [线程池](./线程池.md)） |
| 复用 | 不可复用 | 不可复用（`CyclicBarrier` 才可） |

> 💡 **提示：** 在线程池环境下，线程会被复用、不会真正"结束"，所以 `join()` 通常不适用，应该用 `Future.get()` 或 `CountDownLatch`。

---

## 九、常见问题与注意事项

1. **`join()` 是阻塞的**：调用它的线程会被挂起，直到目标结束或超时，别在关键路径上无脑 `join` 影响吞吐。
2. **超时不等于停止目标线程**：`join(ms)` 超时后目标线程仍在跑，要停止得用 `interrupt()`。
3. **不要 `join()` 自己**：`Thread.currentThread().join()` 会让当前线程永远等待自己结束，导致死等。
4. **能被中断**：处理 `InterruptedException` 时记得重置中断标志。
5. **线程池中慎用**：池化线程不结束，用 `Future.get()` 或 `CountDownLatch` 替代。

---

## 十、总结

| 知识点 | 关键内容 |
| --- | --- |
| 作用 | 当前线程阻塞等待目标线程结束 |
| 重载 | `join()` / `join(ms)` 无限等 vs 限时等 |
| 状态 | WAITING / TIMED_WAITING |
| 底层 | 基于 `wait()` 实现 |
| 典型场景 | 主线程等子线程完成后汇总 |

> 💡 **提示：** `join()` 解决的是"**先后顺序**"问题——让当前线程**排到**目标线程**后面**。需要"等多个任务完成"时，优先考虑 `CountDownLatch`（尤其在线程池里）。
