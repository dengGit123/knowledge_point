# 线程池

> 📖 官方教程：[Thread Pools (Oracle)](https://docs.oracle.com/javase/tutorial/essential/concurrency/pools.html)
> 📖 官方 API：[ThreadPoolExecutor (Java Platform SE 21)](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/util/concurrent/ThreadPoolExecutor.html)

## 一、概述

实际开发中**几乎不会手动 `new Thread()`**，而是用**线程池**统一创建和管理线程。线程池能在任务到来时**复用已有线程**，避免频繁创建销毁的开销，并提供统一的管理、监控、限流能力。

本文是并发编程的"重中之重"，讲解线程池的核心参数、工作流程、拒绝策略，以及为什么不用 `Executors`、如何合理配置线程数，最后补充常用的并发工具类。

---

## 二、为什么用线程池

- **降低资源消耗**：复用已创建的线程，避免频繁创建销毁；
- **提高响应速度**：任务到达时无需等待线程创建；
- **便于统一管理**：统一调优、监控、限流；
- **避免无限创建线程**：防止线程过多导致 OOM 或系统崩溃。

---

## 三、ThreadPoolExecutor 七大参数

创建线程池最核心的类是 `ThreadPoolExecutor`，它有 7 个参数：

```java
public ThreadPoolExecutor(
    int corePoolSize,                      // 1. 核心线程数
    int maximumPoolSize,                   // 2. 最大线程数
    long keepAliveTime,                    // 3. 非核心线程空闲存活时间
    TimeUnit unit,                         // 4. 时间单位
    BlockingQueue<Runnable> workQueue,     // 5. 任务队列
    ThreadFactory threadFactory,           // 6. 线程工厂
    RejectedExecutionHandler handler       // 7. 拒绝策略
)
```

| 参数 | 说明 |
| --- | --- |
| `corePoolSize` | 核心线程数，即使空闲也保留（除非设置 `allowCoreThreadTimeOut`） |
| `maximumPoolSize` | 线程池最大线程数 |
| `keepAliveTime` | 非核心线程空闲后的存活时间 |
| `unit` | `keepAliveTime` 的单位（`TimeUnit.SECONDS` 等） |
| `workQueue` | 阻塞队列，如 `LinkedBlockingQueue`、`ArrayBlockingQueue`、`SynchronousQueue` |
| `threadFactory` | 创建线程的工厂（可自定义线程名） |
| `handler` | 拒绝策略（任务太多处理不过来时） |

## 四、线程池工作流程（核心！）

提交任务后，线程池按以下顺序处理，记住一个口诀：**核心线程 → 队列 → 非核心线程 → 拒绝策略**。

```
                 提交任务 (execute)
                       │
                       ▼
            ┌───────────────────────┐
            │ 当前线程数 < 核心线程数？│
            └──────────┬────────────┘
                   是  │  否
           ┌───────────┘  └────────────┐
           ▼                            ▼
   创建核心线程执行任务          ┌─────────────────┐
                                │ 队列是否已满？    │
                                └────────┬────────┘
                                    否  │   是
                            ┌───────────┘   └─────────────┐
                            ▼                             ▼
                       放入队列等待           ┌─────────────────────┐
                                              │ 当前线程 < 最大线程数？│
                                              └──────────┬──────────┘
                                                    是   │    否
                                        ┌──────────────┘  └─────────────┐
                                        ▼                                ▼
                                创建非核心线程执行                  执行拒绝策略
```

**关键点（容易记错）**：任务先尝试进**核心线程**，满了不是直接开新线程，而是先进**队列**，**队列也满了**才开**非核心线程**（直到最大线程数），再不够才触发**拒绝策略**。

---

## 五、四种拒绝策略

当线程池的线程数达到最大、队列也满了，新任务被提交时执行拒绝策略：

| 策略 | 行为 |
| --- | --- |
| `AbortPolicy`（默认） | 抛出 `RejectedExecutionException` |
| `CallerRunsPolicy` | 由**提交任务的线程**自己执行该任务（起到限流、降速作用） |
| `DiscardPolicy` | 静默丢弃新任务 |
| `DiscardOldestPolicy` | 丢弃**队列中最旧**的任务，再尝试提交新任务 |

> 💡 **提示：** `CallerRunsPolicy` 常用于"背压"——让提交者线程自己干活，从而自然降低提交速度，避免任务堆积。

## 六、为什么不推荐 Executors

阿里《Java 开发手册》强制规定**禁用 `Executors` 创建线程池**，应直接用 `new ThreadPoolExecutor(...)`。原因是 `Executors` 的工厂方法存在 OOM 风险：

| Executors 方法 | 实际配置 | 风险 |
| --- | --- | --- |
| `newFixedThreadPool` | 队列用 `LinkedBlockingQueue`（**无界**） | OOM（队列无限增长） |
| `newSingleThreadExecutor` | 同上，无界队列 | OOM（队列） |
| `newCachedThreadPool` | 最大线程数 `Integer.MAX_VALUE` | OOM（创建过多线程） |
| `newScheduledThreadPool` | 队列无界 | OOM（队列） |

> ⚠️ **注意：** 手动用 `new ThreadPoolExecutor(...)` 显式指定每个参数（尤其用**有界队列**、合理的**最大线程数**），才能控制资源、避免 OOM，也便于排查问题。

## 七、完整使用示例

```java
import java.util.concurrent.*;

public class PoolDemo {
    public static void main(String[] args) {
        ThreadPoolExecutor pool = new ThreadPoolExecutor(
            2,                              // 核心线程数
            4,                              // 最大线程数
            30, TimeUnit.SECONDS,           // 非核心线程空闲存活时间
            new ArrayBlockingQueue<>(10),   // 有界任务队列
            Executors.defaultThreadFactory(),// 默认线程工厂
            new ThreadPoolExecutor.AbortPolicy() // 拒绝策略
        );

        // 提交 Runnable（无返回值）
        pool.execute(() -> System.out.println("执行任务"));

        // 提交 Callable（有返回值，返回 Future）
        Future<String> future = pool.submit(() -> {
            Thread.sleep(500);
            return "任务结果";
        });
        try {
            System.out.println(future.get()); // 阻塞拿结果
        } catch (Exception e) {
            e.printStackTrace();
        }

        pool.shutdown(); // 平缓关闭：不再接新任务，已提交任务执行完
        // pool.shutdownNow(); // 立即停止：尝试中断正在执行的任务
    }
}
```

`execute` vs `submit`：

| 方法 | 接收 | 返回值 | 异常处理 |
| --- | --- | --- | --- |
| `execute(Runnable)` | `Runnable` | 无 | 抛未捕获异常会触发 UncaughtExceptionHandler |
| `submit(Callable/Runnable)` | `Runnable` / `Callable` | `Future` | 异常被封装在 `Future` 中，`get()` 时抛出 |

> ⚠️ **注意：** 生产环境的线程池一般随 JVM 长期存活，**不关闭**；只在应用/模块退出时才 `shutdown()`。

## 八、如何合理设置线程数

线程数不是越多越好，要看任务的类型：

| 任务类型 | 推荐核心线程数 | 说明 |
| --- | --- | --- |
| CPU 密集型 | `N + 1`（N = CPU 核数） | 线程过多反而频繁切换，1 个用来处理偶发的缺页/调度 |
| IO 密集型 | `2N` 或更多 | 线程多在等 IO，CPU 空闲，可多开线程填满 |

> 💡 **提示：** 一个经验公式：**线程数 = N × U × (1 + 等待时间/计算时间)**，其中 N 是 CPU 核数，U 是目标 CPU 利用率。实际还要结合压测调整。

```java
int n = Runtime.getRuntime().availableProcessors(); // 获取 CPU 核数
```

## 九、常用并发工具类

`java.util.concurrent` 还提供了几个高频的"协调工具"，配合线程池使用。

### 1. CountDownLatch（一次性倒计时）

让一个或多个线程**等待，直到其他线程执行完毕**。计数器**不可重置**（一次性）。

```java
final int N = 3;
CountDownLatch latch = new CountDownLatch(N); // 计数 3

for (int i = 0; i < N; i++) {
    new Thread(() -> {
        System.out.println(Thread.currentThread().getName() + " 完成");
        latch.countDown(); // 计数减 1
    }).start();
}

latch.await(); // 等待计数归零
System.out.println("所有子任务完成，继续主流程");
```

### 2. CyclicBarrier（可复用屏障）

让**一组线程相互等待**，全部到达屏障点后**一起继续**，**可重复使用**。

```java
CyclicBarrier barrier = new CyclicBarrier(3, () ->
        System.out.println("全部到达屏障，开始下一阶段"));

for (int i = 0; i < 3; i++) {
    new Thread(() -> {
        System.out.println(Thread.currentThread().getName() + " 到达");
        try { barrier.await(); } catch (Exception e) {}
    }).start();
}
```

### 3. Semaphore（信号量 / 限流）

控制**同时访问某资源的线程数量**。

```java
Semaphore semaphore = new Semaphore(3); // 只允许 3 个线程同时访问

new Thread(() -> {
    try {
        semaphore.acquire(); // 获取许可
        System.out.println("访问资源");
        Thread.sleep(1000);
    } catch (InterruptedException e) {
    } finally {
        semaphore.release(); // 释放许可
    }
}).start();
```

### 4. 并发容器

- **`ConcurrentHashMap`**：线程安全的 HashMap，性能远优于 `Hashtable`；
- **`BlockingQueue`（如 `ArrayBlockingQueue` / `LinkedBlockingQueue`）**：阻塞队列，常用于"生产者-消费者"和线程池的任务队列；
- **`CopyOnWriteArrayList`**：写时复制的 List，读多写少场景。

### CountDownLatch vs CyclicBarrier

| 对比项 | CountDownLatch | CyclicBarrier |
| --- | --- | --- |
| 可复用 | 否 | 是 |
| 计数方式 | `countDown()` 递减 | `await()` 等待 |
| 主语 | 一个线程等 N 个 | N 个线程互相等 |
| 侧重点 | 等待任务完成 | 线程同步到某一点 |

## 十、总结

| 知识点 | 关键内容 |
| --- | --- |
| 七大参数 | core / max / keepAlive / unit / queue / factory / handler |
| 工作流程 | 核心 → 队列 → 非核心 → 拒绝策略 |
| 拒绝策略 | Abort / CallerRuns / Discard / DiscardOldest |
| Executors | 不推荐（OOM 风险），用 ThreadPoolExecutor |
| 线程数 | CPU 密集型 N+1，IO 密集型 2N |
| 并发工具 | CountDownLatch / CyclicBarrier / Semaphore / 并发容器 |

> 💡 **提示：** 线程池是面试和实战的核心。务必背熟"**七大参数 + 工作流程口诀（核心→队列→非核心→拒绝）**"，并记住"**不用 Executors，手动 new ThreadPoolExecutor 并配 有界队列**"这条生产规范。
