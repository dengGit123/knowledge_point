# 线程的调度

> 📖 官方教程：[Pause Execution with Sleep / Interference (Oracle)](https://docs.oracle.com/javase/tutorial/essential/concurrency/)
> 📖 官方 API：[Thread (Java Platform SE 21)](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/lang/Thread.html)

## 一、概述

写多线程代码时，常有人疑惑：为什么我的线程**执行顺序每次都不一样**？为什么"先 `start()` 的线程反而后执行完"？这些现象的根源在于——**线程什么时候执行、执行多久，由操作系统（OS）的线程调度器决定，程序几乎无法精确控制**。

本文讲清楚 Java 线程是怎么被调度的、有哪些调度策略、以及为什么不能依赖"执行顺序"。

---

## 二、Java 线程与系统线程

从 JDK 1.2 起，Java 线程采用 **1:1 模型**：每个 Java 线程直接对应一个操作系统线程。因此：

- **线程调度最终交给操作系统**的线程调度器；
- JVM 只负责创建/销毁线程、管理状态映射，**不参与"哪个线程现在上 CPU"的决策**（虚拟线程除外）。

所以理解 Java 线程调度，本质上要理解操作系统的线程调度机制。

---

## 三、两大调度策略

主流操作系统（Windows、Linux、macOS）对普通线程采用：**抢占式调度 + 时间片轮转**。

### 1. 抢占式调度（Preemptive）

一个线程**不会一直独占 CPU**。当发生以下情况时，它会被"抢占"（强制暂停，让出 CPU）：

- 分给它的**时间片用完**；
- 有**更高优先级**的线程进入就绪状态；
- 线程自己**阻塞**（如 IO、`sleep`、等锁）或主动 `yield()`。

> 💡 **提示：** 因为是"抢占式"，所以**没有"线程 A 执行完才轮到线程 B"的保证**，任何线程都可能在执行到一半时被切走。

### 2. 时间片轮转（Round-Robin）

每个处于就绪状态的线程会被分到一个**时间片（通常几毫秒）**。用完就轮到下一个就绪线程，大家**轮流**上 CPU。切换极快（微秒级），人感觉所有线程"同时"在跑——这正是**单核也能并发**的原理。

```
单核时间片轮转示意：
  ┌──T1──┐┌──T2──┐┌──T3──┐┌──T1──┐┌──T2──┐...
  各线程轮流获得一小段 CPU 时间
```

## 四、优先级（Priority）

Java 允许给线程设置优先级，**给调度器一个提示**：

```java
Thread t = new Thread(() -> {});
t.setPriority(Thread.MAX_PRIORITY); // 10，最大
```

| 常量 | 值 | 含义 |
| --- | --- | --- |
| `Thread.MIN_PRIORITY` | 1 | 最低优先级 |
| `Thread.NORM_PRIORITY` | 5 | 默认优先级 |
| `Thread.MAX_PRIORITY` | 10 | 最高优先级 |

> ⚠️ **注意：**
>
> - 优先级**只是建议**，**不保证**高优先级线程一定先执行或执行更久；
> - 不同操作系统对 Java 优先级的映射不同，跨平台表现可能不一致；
> - **不能依赖优先级来保证执行顺序或公平性**。

---

## 五、让出 CPU：yield()

`Thread.yield()` 是一个"提示"：当前线程表示"我愿意让出 CPU"，让调度器去调度其他线程。

```java
Thread.yield(); // 提示让出，但不保证一定让
```

> ⚠️ **注意：** `yield()` **只是建议**，调度器可以**完全忽略**它。调用后线程仍处于 `RUNNABLE`（就绪）状态，可能立刻又被选中继续执行。它和 `sleep(0)` 行为类似，但语义上更明确。**不要用 `yield()` 做同步或控制流程。**

---

## 六、为什么"执行顺序不可预测"

由于抢占式 + 时间片轮转的调度方式：

- 多个线程**谁先跑、跑到哪被切换**，是 OS 安排的，每次运行结果可能不同；
- 这正是**线程安全问题难以复现**的原因——只有当线程恰好以某种顺序交叉执行时才暴露 bug；
- **先 `start()` 不等于先执行完**：`start()` 只是请求调度，真正何时被调度由 OS 决定。

```java
// 下面的输出顺序完全不可控
for (int i = 0; i < 3; i++) {
    int idx = i;
    new Thread(() -> System.out.println("线程 " + idx)).start();
}
// 可能输出 0 1 2，也可能 2 0 1 ...
```

---

## 七、不要依赖"时序假设"

这是多线程编程最重要的原则之一：

- ❌ 不能假设"先 `start()` 的线程一定先执行/先结束"；
- ❌ 不能假设"几个语句会连续执行不被打断"；
- ❌ 不能假设"靠 `yield()` / 优先级就能保证顺序"；
- ✅ 想让线程**协作**，必须使用**同步机制**：

| 协作需求 | 用什么 |
| --- | --- |
| 等某线程结束再继续 | `join()`（见 [线程的合并](./线程的合并.md)） |
| 等某个条件成立 | `wait()` / `notify()`（见 [线程的同步](./线程的同步.md)） |
| 等一批任务完成 | `CountDownLatch`（见 [线程池](./线程池.md)） |
| 多线程互斥访问临界区 | `synchronized` / `Lock`（见 [线程的同步](./线程的同步.md)） |

---

## 八、协作式 vs 抢占式（补充）

| 调度方式 | 说明 | 特点 |
| --- | --- | --- |
| 协作式 | 线程主动让出 CPU（如早期 Green Thread） | 一个不主动让，别人就饿死 |
| 抢占式 | OS 强制切换（现代主流） | 公平性好，线程不会一直霸占 CPU |

Java 早期（JDK 1.1）用过绿色线程（协作式），从 JDK 1.2 起改为基于原生线程的抢占式调度。

---

## 九、总结

| 知识点 | 关键内容 |
| --- | --- |
| 调度主体 | 操作系统（Java 是 1:1 模型） |
| 调度策略 | 抢占式 + 时间片轮转 |
| 优先级 | 只是提示，不保证顺序 |
| yield() | 提示让出 CPU，可被忽略 |
| 执行顺序 | 不可预测，不能依赖时序假设 |
| 协作靠同步 | join / wait-notify / CountDownLatch / 锁 |

> 💡 **提示：** 一句话总结——**线程状态是"程序视角的快照"（`getState()`），线程调度是"操作系统视角的安排"——程序只能影响状态，调度由 OS 说了算**。所以，永远不要用"我以为它会按顺序执行"来写多线程代码，要用同步机制来保证协作。
