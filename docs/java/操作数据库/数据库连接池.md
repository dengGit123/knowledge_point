### 一、概述

> 📖 [Druid GitHub](https://github.com/alibaba/druid) ｜ [HikariCP GitHub](https://github.com/brettwooldridge/HikariCP)

数据库连接池是**管理数据库连接的「池子」**，预先创建一批连接重复使用，避免每次操作都新建/销毁连接。它是生产环境**必备**的组件。

大白话：每次连数据库都要「**握手、认证、建连接**」，很耗时。如果每次请求都新建连接、用完就扔，就像「**每次打电话都重新拉一根电话线，挂了就剪断**」，浪费又慢。

连接池像「**共享单车桩**」——提前备好一堆连接（单车），要用时借一个，用完还回去，**反复使用**，不用每次都造新的。

| 你将学到 | 说明 |
| --- | --- |
| 为什么用连接池 | 不用连接池的问题 |
| 连接池原理 | 池化思想、核心参数 |
| Druid | 阿里出品，带监控 |
| HikariCP | 性能极致，Spring Boot 默认 |

> 💡 **提示：** **生产环境绝不能直接用 DriverManager.getConnection()**，必须用连接池。这是基本常识。

### 二、为什么需要连接池

#### 1. 不用连接池的问题

```java
// ❌ 每次操作都新建连接
Connection conn = DriverManager.getConnection(url, user, pwd);  // 很慢！
// 用...
conn.close();   // 销毁
```

每次 `getConnection` 要：
- TCP 三次握手（见 [[../网络/TCP协议]]）
- 数据库认证
- 建立会话

整个过程**耗时几十到几百毫秒**。高并发下频繁建连，**数据库扛不住**。

#### 2. 连接池的解决思路

```
① 启动时预创建 N 个连接，放进池子
② 要用时从池里借一个（极快，毫秒级）
③ 用完归还到池子（不是真关闭）
④ 连接复用，不用反复建连
```

#### 连接池的好处

| 好处 | 说明 |
| --- | --- |
| 性能 | 复用连接，省去建连开销 |
| 稳定 | 控制最大连接数，防止数据库被打爆 |
| 管理 | 统一监控、超时控制 |

### 三、连接池的核心参数

任何连接池都有这几个关键参数：

| 参数 | 说明 | 典型值 |
| --- | --- | --- |
| **初始大小**（initialSize） | 启动时创建的连接数 | 5 |
| **最小空闲**（minIdle） | 池中保持的最小空闲连接 | 5 |
| **最大连接**（maxActive） | 池中**最多**多少连接 | 20 |
| **获取超时**（maxWait） | 等连接的超时时间（毫秒） | 60000 |
| 连接最长寿命（maxLifetime） | 连接存活最长时间 | 30 分钟 |
| 空闲超时（idleTimeout） | 空闲连接多久回收 | 10 分钟 |

> 💡 **提示：** **maxActive 是最重要的参数**——设小了请求排队，设大了数据库扛不住。要根据数据库性能和并发量调优。

### 四、Druid（阿里出品，国内主流）

Druid 是阿里巴巴开源的连接池，**国内最流行**，特点是自带**强大的 SQL 监控**。

#### 1. 引入依赖

```xml
<dependency>
    <groupId>com.alibaba</groupId>
    <artifactId>druid</artifactId>
    <version>1.2.20</version>
</dependency>
```

#### 2. 基本使用

```java
import com.alibaba.druid.pool.DruidDataSource;
import javax.sql.DataSource;
import java.sql.*;
import java.util.Properties;

public class DruidDemo {
    public static void main(String[] args) throws SQLException {
        DruidDataSource ds = new DruidDataSource();
        ds.setUrl("jdbc:mysql://localhost:3306/test?useSSL=false&serverTimezone=UTC");
        ds.setUsername("root");
        ds.setPassword("123456");
        ds.setInitialSize(5);      // 初始连接数
        ds.setMinIdle(5);          // 最小空闲
        ds.setMaxActive(20);       // 最大连接数
        ds.setMaxWait(60000);      // 获取超时

        // ✅ 从连接池获取连接（不再是 DriverManager）
        try (Connection conn = ds.getConnection();
             PreparedStatement ps = conn.prepareStatement("SELECT * FROM user WHERE id = ?");
             ResultSet rs = ps.executeQuery()) {
            ps.setInt(1, 1);
            while (rs.next()) {
                System.out.println(rs.getString("name"));
            }
        }
        // conn.close() 其实是归还连接池，不是真关闭
    }
}
```

#### 3. Druid 的监控能力

Druid 自带 **StatViewServlet** 和 **Web 监控页面**，能看到：
- 所有执行过的 SQL 及耗时
- 慢查询
- 连接池状态

> 💡 **提示：** Druid 的 SQL 监控是它最大的卖点，排查慢查询、定位问题非常方便。

### 五、HikariCP（性能之王，Spring Boot 默认）

HikariCP 是性能最快的连接池，**Spring Boot 2.0+ 默认内置**。

#### 1. 引入依赖

```xml
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
    <version>5.1.0</version>
</dependency>
```

#### 2. 基本使用

```java
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;
import java.sql.*;

public class HikariDemo {
    public static void main(String[] args) throws SQLException {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:mysql://localhost:3306/test");
        config.setUsername("root");
        config.setPassword("123456");
        config.setMaximumPoolSize(20);   // 最大连接数
        config.setMinimumIdle(5);
        config.setConnectionTimeout(30000);   // 获取超时
        config.setIdleTimeout(600000);        // 空闲超时
        config.setMaxLifetime(1800000);       // 最长寿命

        HikariDataSource ds = new HikariDataSource(config);

        try (Connection conn = ds.getConnection()) {
            // 使用连接
        }
    }
}
```

#### HikariCP 为什么快

| 优化点 | 说明 |
| --- | --- |
| 字节码精简 | 代码量小，加载快 |
| ConcurrentBag | 自研无锁集合，获取连接快 |
| FastList | 优化过的 List |
| 严格的状态管理 | 避免不必要的同步开销 |

### 六、Spring Boot 整合（最常见场景）

Spring Boot 让连接池配置变得极简，**application.yml** 一行搞定：

```yaml
spring:
  datasource:
    # 用 HikariCP（Spring Boot 默认）
    url: jdbc:mysql://localhost:3306/test?useSSL=false&serverTimezone=UTC
    username: root
    password: 123456
    driver-class-name: com.mysql.cj.jdbc.Driver
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 30000
```

```java
// ✅ 直接注入 DataSource / JdbcTemplate 使用
@Service
public class UserService {
    @Autowired
    private JdbcTemplate jdbcTemplate;

    public List<User> list() {
        return jdbcTemplate.query("SELECT * FROM user",
            (rs, i) -> new User(rs.getInt("id"), rs.getString("name")));
    }
}
```

> 💡 **提示：** Spring Boot **默认用 HikariCP**。要换成 Druid，引入 `druid-spring-boot-starter` 即可，配置方式类似。

### 八、连接池的工作原理（池化思想）

```
应用 ──getConnection()──► [连接池] ──有空闲连接──► 返回连接
                            │
                          没有空闲连接
                            │
                            ▼
                     未达 maxActive？──是──► 新建连接返回
                            │
                           否
                            ▼
                       等待（maxWait）
                            │
                       超时还没拿到 ──► 抛异常

应用 ──close()──► 连接归还池子（不是真关闭）
```

> 💡 **提示：** 连接池模式下，`conn.close()` **不是真关闭**，而是把连接归还到池里复用。这正是连接池省资源的原理。

### 九、常见问题与注意事项

#### 1. 连接泄漏（borrow 不还）

```java
// ❌ 借了连接不关，池子耗尽，后续请求全卡住
Connection conn = ds.getConnection();
// 忘了 close
```

**永远 try-with-resources 关闭连接**（实际上是归还）。

#### 2. 连接数设置不当

- maxActive 太小：请求排队，响应慢
- maxActive 太大：数据库连接数打满，拖垮数据库

经验值：从 10~20 起步，根据压测调整。

#### 3. 连接失效

数据库重启或网络断开后，池里的旧连接失效。好的连接池会做**心跳检测**（testOnBorrow / keepAlive）剔除坏连接。

#### 4. maxWait 超时

连接拿不到超过 maxWait 抛异常。通常是 maxActive 太小或连接泄漏导致。

#### 5. 配合事务

连接池和事务（[[事务管理]]）配合无碍，事务是 Connection 级别的。但要注意：**一个事务要在同一个连接内完成**，框架会保证这一点。
