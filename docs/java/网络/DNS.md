### 一、概述

> 📖 [DNS（MDN）](https://developer.mozilla.org/zh-CN/docs/Glossary/DNS) ｜ [域名系统（维基百科）](https://zh.wikipedia.org/wiki/%E5%9F%9F%E5%90%8D%E7%B3%BB%E7%BB%9F)

DNS（Domain Name System，域名系统）是互联网的「**电话簿**」。它负责把人类易记的**域名**（如 `www.baidu.com`）翻译成机器能识别的 **IP 地址**（如 `14.215.177.38`）。

大白话：你想找百度，肯定记不住它服务器的 IP（一串数字），但记得住 `www.baidu.com`。DNS 就是那个帮你「**按名字查号码**」的服务。

| 角色 | 说明 |
| --- | --- |
| 域名 | 人类友好的名字（`www.example.com`） |
| IP 地址 | 机器通信用的地址（`93.184.216.34`） |
| DNS | 把域名 → IP 的翻译服务 |

> 💡 **提示：** 浏览器、所有网络程序底层只认 IP，不认域名。**任何一次访问域名，第一步都是 DNS 解析**，查出 IP 后才能建立连接。

---

### 二、为什么需要 DNS

- **好记**：域名比 IP 好记太多（你记不住 `14.215.177.38`，但记得住 baidu.com）
- **灵活**：服务器换 IP 了，只要改 DNS 记录指向新 IP，**用户无感知**（域名不变）
- **负载均衡**：一个域名可以对应多个 IP，DNS 轮询分流
- **高可用**：同一域名可指向不同地区的服务器

---

### 三、域名的层级结构

域名是**分层**的，从右到左层级递减：

```
        www.example.com.
        │    │     │  │
        │    │     │  └─ 根域（root，通常省略，用一个点表示）
        │    │     └──── 顶级域（TLD）：com、cn、org
        │    └────────── 二级域：example
        └─────────────── 三级域（子域/主机名）：www
```

| 层级 | 例子 | 说明 |
| --- | --- | --- |
| 根域 | `.` | 全球 13 组根 DNS 服务器 |
| 顶级域（TLD） | `.com` `.cn` `.org` `.net` | 国家/类别 |
| 二级域 | `example.com` | 你注册的域名 |
| 子域/主机 | `www.example.com` `mail.example.com` | 自定义 |

> 💡 **提示：** `www.example.com` 和 `example.com` 技术上是两条不同的记录，但通常都指向同一个网站。

---

### 四、DNS 的分层服务器

DNS 是**分布式**的，没有一台服务器存所有域名，而是分层管理：

| 服务器类型 | 作用 |
| --- | --- |
| **本地 DNS 服务器** | 你电脑/路由器配置的 DNS（如 `8.8.8.8`、运营商 DNS） |
| **根 DNS 服务器** | 管理顶级域，告诉你要去哪查 `.com` |
| **顶级域（TLD）服务器** | 管理 `.com` `.cn` 等，告诉你要去哪查 `example.com` |
| **权威 DNS 服务器** | 实际存储某域名记录的服务器（如阿里云、Cloudflare 托管的） |

---

### 五、DNS 解析流程（重点）⭐

当你在浏览器输入 `www.example.com`，DNS 解析经过这些步骤：

```
  浏览器/系统缓存？
     │ 否
     ▼
  ① 本地 DNS 服务器（递归查询者）开始查
     │
     │ ② 问根 DNS：www.example.com 在哪？
     │ ◄─ 根说：去问 .com 的 TLD 服务器
     ▼
  ③ 问 .com TLD 服务器
     │ ◄─ TLD 说：去问 example.com 的权威服务器
     ▼
  ④ 问 example.com 的权威服务器
     │ ◄─ 权威服务器给出最终 IP
     ▼
  ⑤ 本地 DNS 把 IP 返回给浏览器，并缓存结果
```

#### 两种查询方式

- **递归查询**：客户端把活全交给本地 DNS，让它「一查到底」并返回最终结果（客户端→本地 DNS 这一段是递归）
- **迭代查询**：被问的服务器不直接给最终答案，而是「指路」——告诉你「去问下一个」（本地 DNS → 根/TLD/权威 这几段是迭代）

> 💡 **提示：** 一句话区分：**客户端对本地 DNS 是递归（我要最终结果），本地 DNS 对各级服务器是迭代（你给我指路我自己查）**。

### 六、多级缓存

为了速度，DNS 结果会**到处缓存**，减少查询次数：

```
查 IP 时按顺序找缓存：
1. 浏览器 DNS 缓存
2. 操作系统 DNS 缓存
3. 本机 hosts 文件
4. 本地 DNS 服务器缓存
   ↓ 都没有，才发起上面的迭代查询
```

只要有缓存命中，就直接用，不用走完整流程。

> ⚠️ **注意：** 缓存也是 DNS 解析故障的常见原因——**域名改了 IP，但旧缓存还没过期**，导致访问的还是旧服务器。这就是为什么改完 DNS 要「等生效」。

#### TTL（Time To Live）

每条 DNS 记录都有 **TTL**（存活时间），单位秒。缓存到期前都用旧记录，到期后重新查询。

- TTL 大：缓存久、查询少、速度快，但**改记录后生效慢**
- TTL 小：改记录生效快，但查询多、稍慢

> 💡 **提示：** 计划迁移服务器时，**提前把 DNS 的 TTL 调小**（如 60 秒），迁移完再调回，能让切换快速生效。

### 七、常见 DNS 记录类型

DNS 不只存「域名→IP」，还有多种记录类型：

| 记录类型 | 名称 | 作用 |
| --- | --- | --- |
| **A** | Address | 域名 → **IPv4** 地址（最常用） |
| **AAAA** | Address | 域名 → **IPv6** 地址 |
| **CNAME** | Canonical Name | 把一个域名**别名**指向另一个域名（常用于 CDN） |
| **MX** | Mail Exchange | 邮件服务器地址（收发邮件用） |
| **TXT** | Text | 任意文本（常用于域名验证、SPF） |
| **NS** | Name Server | 该域名由哪个 DNS 服务器解析 |
| **PTR** | Pointer | IP → 域名（**反向解析**，A 的逆操作） |

#### CNAME 举例

```
博客托管在 GitHub Pages：
blog.example.com  CNAME → username.github.io
username.github.io A → GitHub 的 IP

好处：GitHub 换 IP，你不用改自己的 DNS，自动跟随。
```

### 八、DNS 用的什么协议？

- **默认用 UDP**（端口 53）：查询报文小，UDP 快，适合一问一答
- **报文大或区域传送时用 TCP**：UDP 限制响应不超过 512 字节（实际约 1232 字节），超过用 TCP
- **DoH/DoT 加密 DNS**：传统 DNS 是明文，`DNS over HTTPS` / `DNS over TLS` 给 DNS 查询加密，防窃听和劫持

> 💡 **提示：** DNS 默认走 UDP——这正是 [[UDP协议]] 的典型应用场景之一（小报文、要快、丢一次重查）。

### 九、实际操作：查看与排查 DNS

```bash
# 查询域名 IP（最常用）
nslookup www.baidu.com

# 详细的解析过程（包含从哪台服务器查的）
dig www.baidu.com

# 查看 DNS 缓存与解析路径（Mac/Windows 命令不同）
dig +trace www.baidu.com

# 清除本机 DNS 缓存
# Windows:  ipconfig /flushdns
# Mac:      sudo dscacheutil -flushcache; sudo killall -HUP mDNSResponder
```

#### 修改本机 hosts

系统会**先查 hosts 文件**再查 DNS，可手动指定域名→IP（本地测试常用）：

```
# /etc/hosts (Mac/Linux) 或 C:\Windows\System32\drivers\etc\hosts (Windows)
127.0.0.1  www.myproject.com
192.168.1.100  api.test.com
```

> 💡 **提示：** 本地开发时，改 hosts 把测试域名指向本机或测试服务器，能**不污染 DNS 地用域名访问**，非常实用。

### 十、常见问题与注意事项

#### 1. 域名解析失败（打不开网页）

排查顺序：
1. 是不是只有这个域名不行？（该域名 DNS 故障）
2. 所有域名都不行？（本机/路由器 DNS 配置问题，试试换 `8.8.8.8` / `114.114.114.114`）
3. 清缓存重试

#### 2. DNS 劫持

运营商或黑客篡改 DNS 响应，把域名指向错误 IP（如广告、钓鱼）。防御：用可信 DNS、DoH/DoT 加密、HTTPS 验证服务器身份。

#### 3. DNS 污染

在某些网络环境下，DNS 查询被干扰返回错误结果。了解即可。

#### 4. 一个域名多个 IP

DNS 可对同一域名返回多个 IP，实现**负载均衡**（轮询）和**就近访问**（智能 DNS 按用户地区返回最近的服务器 IP）。

#### 5. 为什么有时候网站很慢是 DNS 的锅？

DNS 解析是访问网站的第一步。如果 DNS 服务器响应慢，会拖慢整个访问。换更快的公共 DNS（如 `8.8.8.8`、`223.5.5.5`）有时能提速。

### 十一、总结

| 要点 | 说明 |
| --- | --- |
| 作用 | 域名 → IP 的翻译（互联网电话簿） |
| 分层 | 根 → 顶级域 → 权威服务器 |
| 查询 | 客户端对本地 DNS 递归，本地 DNS 对各级迭代 |
| 缓存 | 浏览器/系统/路由/本地 DNS 多级缓存，TTL 控制过期 |
| 记录 | A(IPv4)、AAAA(IPv6)、CNAME(别名)、MX(邮件) 等 |
| 协议 | 默认 UDP（端口 53），大响应用 TCP，可加密（DoH/DoT） |

每次上网的第一步都是 DNS 解析。理解它，就理解了「域名怎么变成 IP」这关键一跳。
