### 一、概述

> 📖 [UDP（维基百科）](https://zh.wikipedia.org/wiki/%E7%94%A8%E6%88%B7%E6%95%B0%E6%8D%AE%E6%8A%A5%E5%8D%8F%E8%AE%AE) ｜ [RFC 768（UDP 规范）](https://datatracker.ietf.org/doc/html/rfc768)

UDP（User Datagram Protocol，用户数据报协议）是传输层的另一个重要协议，与 TCP 并列。它提供**无连接、不可靠、面向数据报**的传输，主打一个「快」。

大白话：UDP 像「寄明信片」——写好地址贴好邮票扔进邮筒就完事，不等回执、不管有没有送到。可能会丢、可能晚到、可能顺序乱，但**简单、快速、开销小**。

| 特性 | 说明 |
| --- | --- |
| 无连接 | 发送前不需要建立连接 |
| 不可靠 | 不保证送达、不保证顺序 |
| 面向数据报 | 以整个数据报为单位，保留边界 |
| 开销小 | 首部只有 8 字节（TCP 是 20 字节） |
| 支持广播/多播 | 一对多发送 |

---

### 二、UDP 报文首部结构

UDP 首部非常简单，只有 **8 字节**，四个字段各 2 字节：

```
┌──────────────┬──────────────┐
│  源端口 (16位) │ 目的端口 (16位) │
├──────────────┼──────────────┤
│  长度   (16位) │  校验和 (16位)  │
└──────────────┴──────────────┘
       ↓
   [ 数据部分 ]
```

| 字段 | 含义 |
| --- | --- |
| **源端口** | 发送方端口（可选，全 0 表示不需要回信） |
| **目的端口** | 接收方端口 |
| **长度** | 整个 UDP 报文（首部+数据）的字节数 |
| **校验和** | 检测数据是否出错（可选，全 0 表示不校验） |

> 💡 **提示：** 对比 TCP 20 字节的头，UDP 头只有 8 字节，**没有序列号、没有确认号、没有窗口**——这就是它「轻」的原因。

---

### 三、UDP 的特点详解

#### 1. 无连接

UDP 不像 TCP 要先三次握手。**想发就发**，发送方和接收方不需要预先建立任何关系。

#### 2. 不可靠

UDP **不保证**：
- 数据能到达（可能丢包）
- 数据只到达一次（可能重复）
- 数据按顺序到达（可能乱序）

它发送后**完全不管结果**，没有确认、没有重传。

#### 3. 面向数据报（保留边界）

这是 UDP 和 TCP 的重要区别：

- **TCP 面向字节流**：数据无边界，会产生粘包/拆包
- **UDP 面向数据报**：一个数据报就是一个完整单位，**发一个收一个**，边界清晰

```
发送：  [报文A] [报文B]
TCP 收：可能合并成 [AB]，或拆成 [A][B]...（无边界）
UDP 收：一定是 [报文A] 然后单独 [报文B]（有边界）
```

> 💡 **提示：** 用 UDP 不会有粘包问题，因为每个数据报都是独立的、完整的。

#### 4. 高效

- 头部小（8 字节）
- 没有连接管理开销
- 没有确认/重传机制
- 实时性好（不等确认，直接发）

### 四、UDP vs TCP 对比（核心！）

| 对比项 | UDP | TCP |
| --- | --- | --- |
| 连接性 | 无连接 | 面向连接（三次握手） |
| 可靠性 | 不可靠 | 可靠（确认+重传+排序） |
| 数据形式 | 数据报（有边界） | 字节流（无边界） |
| 首部大小 | 8 字节 | 20 字节 |
| 速度 | 快 | 慢 |
| 顺序性 | 不保证 | 保证有序 |
| 流量/拥塞控制 | 无 | 有 |
| 通信方式 | 一对一/多播/广播 | 仅一对一 |
| 资源占用 | 少 | 多 |

**选型原则：**
- **丢了无所谓、追求实时** → UDP（直播、游戏、语音）
- **一个字节都不能错** → TCP（文件、网页、转账）

---

### 五、UDP 的典型应用

UDP 因为快、能广播，被大量用在**实时、高频、允许少量丢失**的场景：

| 应用 | 为什么用 UDP |
| --- | --- |
| **DNS 域名解析** | 查询报文小、要求快，丢一次重查即可 |
| **DHCP** | 分配 IP 时设备还没 IP，没法建立 TCP 连接 |
| **视频/语音通话** | 实时性优先，丢几帧画面花一下不影响，等重传反而卡 |
| **直播流媒体** | 同上，强调实时 |
| **在线游戏** | 玩家位置每秒更新几十次，旧数据丢了不要重传 |
| **TFTP** | 简单文件传输协议 |
| **SNMP** | 网络管理，简单轻量 |

> 💡 **提示：** 你可能会问「视频会议这么重要为什么还用不可靠的 UDP？」——因为对实时通信来说，**晚到的数据等于没到**。一个迟到 2 秒的画面包，重传来了也没意义，不如直接丢。这正是 UDP 的优势场景。

---

### 六、UDP 的「不可靠」能弥补吗？

UDP 本身不可靠，但**如果业务需要可靠，可以在应用层自己实现确认+重传**——这正是 QUIC（HTTP/3 的底层协议）做的事。

#### QUIC 协议（了解）

Google 开发的 **QUIC** 建立在 UDP 之上，自己实现了类似 TCP 的可靠传输 + 加密，但**比 TCP 快**（建连快、无队头阻塞）。HTTP/3 就是基于 QUIC 的。

```
传统：HTTP/2 over TCP
未来：HTTP/3 over QUIC over UDP
```

> 💡 **提示：** 「UDP 不可靠所以不如 TCP」是误区。需要可靠时，在 UDP 上层补上可靠性，反而能获得更快的连接建立和更好的现代网络适应性。

---

### 七、UDP 的广播与多播

UDP 的一大独门优势是**一对多**发送，TCP 做不到：

| 方式 | 说明 |
| --- | --- |
| **单播（Unicast）** | 一对一 |
| **广播（Broadcast）** | 发给本网段**所有**机器（`255.255.255.255`） |
| **多播（Multicast）** | 发给**加入某个组**的机器（多播地址 `224.0.0.0` ~ `239.255.255.255`） |

应用：
- **广播**：局域网设备发现（投屏、智能家居配网）
- **多播**：直播分发，路由器复制数据给订阅者，省带宽

### 八、常见问题与注意事项

#### 1. UDP 会丢包吗？为什么会丢？

会。原因包括：
- 网络拥堵，路由器/接收方缓冲区满了，新包被直接丢弃
- 接收方处理不过来

UDP 应用如果需要可靠，要**自己实现重传机制**。

#### 2. UDP 数据报多大合适？

单个 UDP 报文理论最大约 65507 字节，但**建议不超过 MTU（约 1472 字节）**。超过会被 IP 层分片，任一分片丢失整个报文就废了，丢包率大增。

> ⚠️ **注意：** 不要用 UDP 传大块数据而不做分片重传，丢包会非常严重。

#### 3. UDP 适合做聊天吗？

技术上可以，但聊天消息不能丢，需要应用层自己加确认+重传+排序，**等于重造 TCP**。所以多数聊天软件底层仍用 TCP（或长连接），只有语音/视频部分用 UDP。

#### 4. TCP vs UDP 谁更省带宽？

UDP 头部小，**单次传输开销更小**。但 UDP 不保证送达，要可靠就得重传，反而可能更费。要看具体场景。

---

### 九、总结

| 要点 | 说明 |
| --- | --- |
| UDP 特点 | 无连接、不可靠、快、面向数据报 |
| 首部 | 仅 8 字节，极轻量 |
| 数据报 vs 字节流 | UDP 有边界（无粘包），TCP 无边界 |
| 优势 | 快、开销小、支持广播/多播 |
| 适用 | 实时性高、允许丢包（DNS、直播、游戏、语音） |
| 弥补不可靠 | 应用层自实现重传（如 QUIC） |

UDP 的精髓是「**用可靠性换速度和实时性**」。理解了它和 TCP 的取舍，就掌握了传输层选型的核心逻辑。
