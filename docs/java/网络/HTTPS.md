### 一、概述

> 📖 [HTTPS（MDN）](https://developer.mozilla.org/zh-CN/docs/Glossary/HTTPS) ｜ [TLS（维基百科）](https://zh.wikipedia.org/wiki/%E4%BC%A0%E8%BE%93%E5%B1%82%E5%AE%89%E5%85%A8%E6%80%A7)

HTTPS（HyperText Transfer Protocol Secure）就是 **HTTP + SSL/TLS**——在 HTTP 之下加了一层加密，让数据在传输过程中**不被窃听、不被篡改、能确认对方身份**。

大白话：HTTP 像「明信片」，内容谁都能看；HTTPS 像「密封的保密信」，只有收信人能拆开，而且信封上有「防伪证书」证明对方确实是他声称的那个人。

| 问题 | HTTP | HTTPS 如何解决 |
| --- | --- | --- |
| 窃听（内容泄露） | 明文，裸奔 | **加密**传输 |
| 篡改（内容被改） | 无法发现 | **摘要校验**（MAC） |
| 伪装（钓鱼网站） | 无法验证 | **数字证书**验证身份 |

> 💡 **提示：** HTTPS 不是新协议，而是 HTTP 跑在 SSL/TLS 之上。平时说的 SSL 和 TLS 基本是一回事，TLS 是 SSL 的继任者（SSL 已淘汰，现在都用 TLS）。

---

### 二、为什么要加密——三大风险

网络数据传输面临三大安全威胁，HTTPS 逐一解决：

| 威胁 | 说明 | HTTPS 防御手段 |
| --- | --- | --- |
| **窃听** | 中间人偷看传输内容 | 加密（对称 + 非对称） |
| **篡改** | 中间人修改传输内容 | 消息认证码（MAC） |
| **伪装** | 假冒服务器骗你 | 数字证书 + CA 认证 |

#### 中间人攻击（MITM）

如果不加密，中间的路由器、运营商、黑客都能做「中间人」：

```
你以为：  浏览器 ◄══════════► 银行服务器
实际：   浏览器 ──► [黑客中间人] ──► 银行服务器
                偷看、修改、伪造
```

HTTPS 的加密 + 证书机制，让中间人**看不懂、改不了、装不成**。

---

### 三、加密的两大类型

要理解 HTTPS，先搞懂两种加密：

#### 1. 对称加密

加密和解密用**同一把密钥**。

```
加密： 明文 + 密钥K → 密文
解密： 密文 + 密钥K → 明文
```

- ✅ **优点**：速度快，适合加密大量数据
- ❌ **缺点**：密钥怎么安全地告诉对方是个难题（密钥泄露就完蛋）

#### 2. 非对称加密（公钥加密）

有一对密钥：**公钥**（公开）和 **私钥**（保密），成对生成、数学关联。

```
规则：公钥加密，只能私钥解密；反之私钥加密（签名），公钥能验证。
```

- ✅ **优点**：公钥可公开，解决了密钥分发问题
- ❌ **缺点**：速度慢，不适合加密大量数据

> 💡 **提示：** HTTPS 的精妙之处在于**两者结合**——用非对称加密安全地交换一把「对称密钥」，之后用这把对称密钥加密所有数据。**既安全又快**。

### 四、数字证书与 CA

#### 问题：怎么确认公钥是真的？

非对称加密虽好，但有个致命问题：你怎么确定收到的公钥真的是银行的，而不是黑客冒充的？

**解决：数字证书 + CA（证书颁发机构）。**

#### 数字证书

服务器不直接发公钥，而是发一张**由 CA 签名的证书**，证书里包含：

- 服务器域名
- 服务器的公钥
- 有效期
- 颁发机构（CA）
- CA 的数字签名

#### CA（证书颁发机构）

CA 是**大家都信任的第三方**（像公安局发身份证）。操作系统/浏览器内置了一批可信 CA 的根证书。只要证书是这些 CA 签的，浏览器就认。

#### 验证流程

```
服务器发证书 → 浏览器用内置 CA 公钥验证证书签名
            → 签名有效 + 域名匹配 + 未过期 → 信任，取出公钥
            → 否则 → 报「证书错误」警告
```

> ⚠️ **注意：** 看到浏览器的「您的连接不是私密连接」警告，往往就是证书过期、域名不匹配或自签名证书。**生产环境绝不能让用户习惯性忽略证书警告**。

### 五、HTTPS 完整握手流程（TLS）

HTTPS 通信分两阶段：先**握手**协商出对称密钥，再用对称密钥**加密通信**。

```
   客户端                                        服务端
     │                                            │
     │ ① Client Hello                             │
     │   （支持的 TLS 版本、加密套件、随机数A）       │
     │ ────────────────────────────────────────► │
     │                                            │
     │ ② Server Hello + 证书 + 公钥               │
     │   （选定加密套件、随机数B、数字证书）          │
     │ ◄──────────────────────────────────────── │
     │                                            │
     │ ③ 验证证书 → 生成预主密钥，用公钥加密发回     │
     │ ────────────────────────────────────────► │
     │                                            │
     │ 双方用 (随机数A + 随机数B + 预主密钥)        │
     │ 计算出相同的「会话密钥」（对称密钥）           │
     │                                            │
     │ ══════ 之后用会话密钥对称加密通信 ══════════ │
```

#### 关键点

1. **随机数 + 预主密钥**：双方各持一个随机数，客户端再生成「预主密钥」用服务器公钥加密发过去。双方独立算出**同一个会话密钥**（非对称只用于这一次交换）。
2. **会话密钥是临时的**：每次连接都不同，用完即弃，提升安全性（前向安全）。
3. **之后用对称加密**：握手完成，所有数据用会话密钥加密，性能好。

> 💡 **提示：** 这就是「**非对称加密换密钥，对称加密传数据**」的经典组合——非对称解决密钥分发，对称解决速度。

### 六、数字签名：防篡改的原理

数字证书里的「CA 签名」是怎么防篡改的？

#### 签名与验证过程

```
【CA 给证书签名】
证书内容 → 用 Hash 函数算摘要 → CA 用自己私钥加密摘要 → 得到「数字签名」
（签名附在证书上）

【浏览器验证签名】
收到的证书内容 → 用同样 Hash 算摘要A
用内置 CA 公钥解密签名 → 得到摘要B
比较 摘要A == 摘要B？
  相等 → 证书没被篡改 ✅
  不等 → 证书被改过 ❌
```

**为什么能防篡改？** 因为黑客改了证书内容，摘要就对不上；而重新签名需要 CA 的私钥，黑客没有。

> 💡 **提示：** 摘要（Hash）是把任意长度数据变成固定长度的「指纹」，单向不可逆。常见的有 SHA-256。

### 七、对称加密 vs 非对称加密 总结

| 对比项 | 对称加密 | 非对称加密 |
| --- | --- | --- |
| 密钥 | 加解密同一把 | 公钥/私钥一对 |
| 速度 | 快 | 慢 |
| 安全性 | 密钥分发难 | 公钥可公开 |
| 用途 | 加密大量数据 | 密钥交换、数字签名 |
| HTTPS 中的角色 | 握手后加密通信数据 | 握手阶段交换对称密钥、签名 |

### 八、HTTP vs HTTPS

| 对比项 | HTTP | HTTPS |
| --- | --- | --- |
| 端口 | 80 | 443 |
| 安全性 | 明文，不安全 | 加密，安全 |
| 证书 | 不需要 | 需要 CA 颁发的证书 |
| 性能 | 略快 | 握手有额外开销（优化后影响很小） |
| SEO | 弱势 | 搜索引擎优先收录 |
| 现状 | 已不推荐 | **强制标配** |

> 💡 **提示：** 现代浏览器对 HTTP 网站会标「不安全」警告。**所有网站都应上 HTTPS**，证书现在有免费方案（如 Let's Encrypt）。

### 九、常见问题与注意事项

#### 1. HTTPS 一定慢吗？

握手确实比 HTTP 多几个 RTT，但有优化手段：
- **Session 复用**：重复连接复用之前的密钥，跳过部分握手
- **TLS 1.3**：简化握手到 1 个 RTT，甚至支持 0-RTT
- 现代硬件加解密极快，**实际影响几乎可忽略**

#### 2. 自签名证书能用吗？

自己生成、未经 CA 签名的证书，浏览器不信任会报警。**仅适合内网测试**，公网必须用正规 CA 证书。

#### 3. 单向认证 vs 双向认证

- **单向认证**（常见）：客户端验证服务端身份（普通网站）
- **双向认证（mTLS）**：双方互相验证证书（银行、内部微服务、高安全场景）

#### 4. 证书过期

证书有有效期（通常 1 年）。**过期后用户会看到警告**，需及时续签。运维要监控证书到期时间。

#### 5. HTTPS 能被破解吗？

理论上能，但计算上不可行（破解 RSA/椭圆曲线需天文算力）。实际风险更多来自：私钥泄露、弱算法（如淘汰的 SHA-1）、用户忽略证书警告。

---

### 十、总结

| 要点 | 说明 |
| --- | --- |
| HTTPS = | HTTP + TLS/SSL 加密层 |
| 解决三大风险 | 窃听（加密）、篡改（MAC）、伪装（证书） |
| 加密组合 | 非对称换密钥 + 对称传数据 |
| 数字证书 | CA 签名，证明公钥归属，防中间人 |
| 端口 | HTTPS 默认 443，HTTP 默认 80 |
| 趋势 | 全站 HTTPS，免费证书（Let's Encrypt） |

理解 HTTPS 的核心是三件事：**非对称加密怎么换密钥、对称加密怎么传数据、数字证书怎么防冒充**。
