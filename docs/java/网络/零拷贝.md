### 一、概述

> 📖 [Zero-copy（维基百科）](https://en.wikipedia.org/wiki/Zero-copy) ｜ [FileChannel.transferTo（Java）](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/nio/channels/FileChannel.html#transferTo(long,long,java.nio.WritableByteChannel))

零拷贝（Zero-Copy）是一种**提升 IO 性能的技术**。它通过减少数据在内核空间和用户空间之间的**来回拷贝**，大幅降低 CPU 和内存开销。

大白话：传统读文件再发到网络，数据要被「搬运工（CPU）」来来回回搬好几次。零拷贝就是**让快递直接从仓库发走，不用先搬回前台再发**，省下大量搬运功夫。

| 关键点 | 说明 |
| --- | --- |
| 「拷贝」指什么 | 数据在内存中的复制（内核态 ↔ 用户态） |
| 「零」是相对的 | 不是完全一次都不拷贝，是尽量减少 |
| 解决什么 | 减少数据拷贝次数 + 减少上下文切换 |
| 典型场景 | 大文件传输、Kafka、Netty |

> 💡 **提示：** 零拷贝是 Kafka 高吞吐、Nginx 高性能的核心技术之一。

---

### 二、先理解：传统 IO 的四次拷贝

用传统方式「**读文件 + 发到网络**」，数据要经历：

```
应用程序发起 read() 和 write()：

① read() 把数据从 硬盘 → 内核缓冲区        （DMA 拷贝）
② read() 把数据从 内核缓冲区 → 用户缓冲区    （CPU 拷贝）
③ write() 把数据从 用户缓冲区 → Socket缓冲区 （CPU 拷贝）
④ write() 把数据从 Socket缓冲区 → 网卡       （DMA 拷贝）
```

```
┌─────────┐                ┌──────────────┐                ┌──────────┐
│  硬盘    │ ──DMA①──►  内核缓冲区 ──CPU②──►  用户缓冲区       │
│         │                                        │        │
│         │       内核缓冲区 ◄──CPU③──  用户缓冲区  │         │
│         │  Socket缓冲区 ──DMA④──►  网卡           │         │
└─────────┘                └──────────────┘                └──────────┘
```

#### 代价

| 拷贝次数 | 4 次（2 次 CPU + 2 次 DMA） |
| --- | --- |
| 上下文切换 | 4 次（每次 read/write 各 2 次用户态↔内核态切换） |
| CPU 占用 | 2 次拷贝要 CPU 干活 |
| 数据冗余 | 同一份数据在内存里存了好几份 |

> 💡 **提示：** DMA（Direct Memory Access）是硬件帮 CPU 搬数据，不占 CPU；但 CPU 拷贝那两次是真消耗 CPU 的。

### 三、mmap（内存映射）

#### 原理

`mmap` 把**内核缓冲区**和**用户缓冲区映射到同一块物理内存**，省掉「内核缓冲区 → 用户缓冲区」这次 CPU 拷贝。

```
传统：内核缓冲区 ──CPU拷贝──► 用户缓冲区
mmap：内核缓冲区 = 用户缓冲区（同一块内存，省去这次拷贝）
```

#### mmap 的拷贝次数

```
① 硬盘 → 内核缓冲区（=用户映射区）  （DMA 拷贝）
② 用户映射区 → Socket 缓冲区          （CPU 拷贝）
③ Socket 缓冲区 → 网卡                （DMA 拷贝）
```

- 拷贝：**3 次**（1 次 CPU + 2 次 DMA），比传统少 1 次 CPU 拷贝
- 上下文切换：4 次

> 💡 **提示：** mmap 适合「**读文件后还要在程序里处理数据**」的场景，因为用户程序能直接访问映射的内存。

---

### 四、sendfile（真正的零拷贝核心）

#### 原理

`sendfile` 让数据**完全不进入用户空间**——直接在内核里从「读缓冲区」搬到「Socket 缓冲区」，应用程序只下个命令，不接触数据。

```
① 硬盘 → 内核读缓冲区          （DMA 拷贝）
② 内核读缓冲区 → 网卡          （DMA 拷贝，靠 SG-DMA 硬件支持）
   （CPU 只把「数据位置/长度」描述符传给 Socket，不实际拷贝数据）
```

#### sendfile 的拷贝次数

- 拷贝：**2 次**（全是 DMA 拷贝，**0 次 CPU 拷贝**！这就是「零拷贝」的由来）
- 上下文切换：**2 次**（只有 sendfile 一个系统调用）

> 💡 **提示：** 现代 Linux 的 sendfile 配合 SG-DMA（Scatter-Gather DMA）网卡，能把 CPU 拷贝次数降到 **0**。这是最高效的文件传输方式。

#### sendfile 的限制

- 数据**不经过用户程序**，所以程序无法在中间修改数据
- 适合「**把文件原样发出去**」的场景（如静态文件下载、Kafka 消息传输）

---

### 五、三种方式对比

| 方式 | CPU 拷贝 | 总拷贝 | 上下文切换 | 数据能否被程序处理 |
| --- | --- | --- | --- | --- |
| 传统 read+write | 2 次 | 4 次 | 4 次 | ✅ 可以 |
| **mmap + write** | 1 次 | 3 次 | 4 次 | ✅ 可以 |
| **sendfile** | 0 次 | 2 次 | 2 次 | ❌ 不可以 |

> 💡 **提示：** 选型：要处理数据用 mmap；纯转发文件用 sendfile。

### 六、Java 中的零拷贝

Java NIO 提供了几种使用零拷贝的方式：

#### 1. FileChannel.transferTo / transferFrom（对应 sendfile）

```java
import java.io.FileInputStream;
import java.io.FileOutputStream;
import java.nio.channels.FileChannel;

public class TransferToDemo {
    public static void main(String[] args) throws Exception {
        try (FileInputStream fis = new FileInputStream("source.big");
             FileOutputStream fos = new FileOutputStream("target.big");
             FileChannel inChannel = fis.getChannel();
             FileChannel outChannel = fos.getChannel()) {

            // ✅ 底层用 sendfile，零拷贝高效传输
            inChannel.transferTo(0, inChannel.size(), outChannel);
            // 或：outChannel.transferFrom(inChannel, 0, inChannel.size());
        }
    }
}
```

#### 2.transferTo 发送到网络（Socket）

```java
try (FileInputStream fis = new FileInputStream("bigfile.zip");
     FileChannel fileChannel = fis.getChannel();
     Socket socket = new Socket("127.0.0.1", 8080)) {

    // ✅ 文件直接零拷贝发到网络 Socket
    fileChannel.transferTo(0, fileChannel.size(),
        /* 网络输出通道 */ channels.newChannel(socket.getOutputStream()));
}
```

> 💡 **提示：** `transferTo` 在 Linux 底层就是 `sendfile` 系统调用，**0 次 CPU 拷贝**。

#### 3. MappedByteBuffer（对应 mmap）

```java
import java.nio.MappedByteBuffer;
import java.nio.channels.FileChannel;
import java.io.RandomAccessFile;

try (RandomAccessFile raf = new RandomAccessFile("data.txt", "r");
     FileChannel channel = raf.getChannel()) {

    // ✅ 把文件映射到内存，像操作内存一样读写文件（底层 mmap）
    MappedByteBuffer mapped = channel.map(
        FileChannel.MapMode.READ_ONLY, 0, channel.size());

    byte firstByte = mapped.get(0);   // 直接读，不经 read()
}
```

#### 4. DirectByteBuffer（堆外内存）

```java
import java.nio.ByteBuffer;

// ✅ 分配堆外内存，避免「堆 → 内核」那次拷贝
ByteBuffer buffer = ByteBuffer.allocateDirect(1024);
```

> ⚠️ **注意：** `allocateDirect` 分配的是堆外内存，能减少一次 GC 时的拷贝，但分配/回收成本高，适合**长期复用的大缓冲区**，不要频繁分配。

### 七、实际应用场景

| 场景 | 用到的零拷贝 | 原因 |
| --- | --- | --- |
| **Kafka** | sendfile（消费者拉消息） | 海量消息高吞吐，零拷贝是关键 |
| **Nginx** | sendfile | 静态文件高效分发 |
| **大文件下载** | transferTo | 省去 CPU 拷贝 |
| **Netty** | FileRegion（封装 transferTo）、CompositeByteBuf | 网络传输优化 |
| **RocketMQ** | mmap | CommitLog 文件读写 |

> 💡 **提示：** Kafka 用零拷贝（sendfile）实现了**百万级 TPS** 的消息传输，这是它吞吐量高的核心原因之一。

### 八、常见问题与注意事项

#### 1. 「零拷贝」真的是零次拷贝吗？

不是字面意义的零。它指的是 **CPU 拷贝次数为零**（剩下的都是 DMA 硬件拷贝，不占 CPU）。相对传统的多次 CPU 拷贝，性能提升巨大。

#### 2. sendfile 和 mmap 怎么选？

- **数据要被程序读出来用** → mmap（程序能访问映射内存）
- **数据只是转发不处理** → sendfile（更快，因为根本不进用户态）

#### 3. transferTo 有大小限制吗？

单次 transferTo 传输量有限制（Linux 上约 2GB），大文件要循环调用分批传输。

#### 4. DirectByteBuffer 何时用？

高频 IO 场景，缓冲区长期复用、数据量大时，用堆外内存减少拷贝。但要注意手动管理（或靠 Cleaner 回收），避免内存泄漏。

#### 5. 零拷贝和 [[IO多路复用]] 什么关系？

都是高性能 IO 技术，但解决不同问题：
- 多路复用：解决「**一个线程管多连接**」（连接管理）
- 零拷贝：解决「**单次数据传输的拷贝开销**」（数据传输）

Netty 同时用到了两者。

### 九、总结

| 要点 | 说明 |
| --- | --- |
| 传统 IO | 4 次拷贝（2 CPU + 2 DMA）+ 4 次上下文切换 |
| **mmap** | 内存映射，省 1 次 CPU 拷贝，数据可被程序处理 |
| **sendfile** | 数据不进用户态，0 次 CPU 拷贝，纯转发最快 |
| Java 用法 | `transferTo`（sendfile）、`MappedByteBuffer`（mmap）、`allocateDirect` |
| 典型应用 | Kafka、Nginx、Netty、大文件传输 |

零拷贝的核心价值：**把本该 CPU 干的搬运活，交给 DMA 硬件，让 CPU 去处理更有用的逻辑**。
