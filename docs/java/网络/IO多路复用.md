### 一、概述

> 📖 [epoll（维基百科）](https://zh.wikipedia.org/wiki/Epoll) ｜ [C10K Problem](https://en.wikipedia.org/wiki/C10k_problem)

IO 多路复用（IO Multiplexing）是**高性能网络编程的核心技术**。它让**一个线程同时监控大量连接（fd）**，哪个有事件（可读/可写/新连接）就处理哪个，从而用极少线程扛住海量连接。

大白话：你是一个学校门卫（一个线程），同时要管 1000 个学生（连接）有没有寄快递。

- 笨办法（BIO）：每个学生配一个门卫盯着——门卫不够用
- **多路复用**：一个门卫站在传达室，盯着「取件通知板」，哪个学生名字亮了就去处理哪个

`select`、`poll`、`epoll` 就是这块「通知板」的三种实现，**epoll 是性能最好的那个**，也是 Redis、Nginx、Netty、NIO 高性能的底层秘密。

| 技术 | 说明 |
| --- | --- |
| **select** | 早期方案，有数量限制、效率低 |
| **poll** | 改进 select 的数量限制，但效率仍一般 |
| **epoll** | Linux 上的高效方案，**现代高性能服务的基石** |

> 💡 **提示：** Java NIO 的 `Selector` 在 Linux 底层就是用 epoll 实现的。理解 epoll，就理解了 NIO 为什么快。

---

### 二、为什么需要多路复用

回顾 [[IO模型]] 的痛点：

```
BIO 模式下，处理 10000 个连接需要 10000 个线程
  → 线程内存爆炸（每个线程约 1MB 栈 = 10GB）
  → 线程上下文切换开销巨大
  → 大部分线程都在阻塞等待，浪费
```

多路复用的思路：**别给每个连接配线程**。让一个线程通过一个「系统调用」，**一次性问操作系统「我这一堆连接里，哪些有事了？」**，然后只处理有事的那几个。

---

### 三、select

`select` 是最早的 IO 多路复用方案（1983 年）。

#### 工作方式

```
① 把要监控的所有连接（fd）打包成一个集合，传给 select
② select 阻塞，直到有连接有事（或超时）
③ select 返回，告诉你「有事了」（但不说具体是哪个，要你自己遍历检查）
④ 遍历整个集合，找出有事的连接处理
```

#### 缺点

| 缺点 | 说明 |
| --- | --- |
| **数量限制** | 默认最多监控 1024 个连接（FD_SETSIZE） |
| **性能差** | 每次调用都要把全部 fd 集合**从用户态拷贝到内核态** |
| **遍历低效** | 返回后要**遍历全部 fd** 才知道哪个有事（O(n)） |
| **重复传参** | 每次都要重新传入全部 fd |

> ⚠️ **注意：** select 的 1024 限制和 O(n) 遍历，让它根本扛不住高并发，现在基本被淘汰。

---

### 四、poll

`poll` 是 select 的改进版。

#### 改进了什么

- **去掉了 1024 的数量限制**（用链表代替固定数组）
- 其余机制和 select 几乎一样

#### 仍有缺点

- 仍然每次要拷贝全部 fd 到内核
- 仍然要遍历全部 fd（O(n)）
- 本质性能没有质的飞跃

> 💡 **提示：** poll 只是修了 select 的「数量上限」，没解决「拷贝开销大 + 遍历慢」的核心问题。真正革命性的是 epoll。

---

### 五、epoll ⭐

`epoll` 是 Linux 2.6 引入的高效方案，是当前**高性能网络编程的事实标准**。

#### 三大核心操作

| 操作 | 作用 |
| --- | --- |
| `epoll_create` | 创建一个 epoll 实例 |
| `epoll_ctl` | 注册/修改/删除要监控的 fd（**只在注册时操作一次**） |
| `epoll_wait` | 等待事件，返回**有事件的 fd 列表** |

#### epoll 为什么快（三大改进）

**① 注册一次，不用重复传参**
select 每次都要把全部 fd 拷到内核；epoll 用 `epoll_ctl` 注册一次，之后内核自己维护。

**② 内核维护就绪列表，返回时只给有事的**
select 返回后要遍历全部 fd 找有事的；epoll 内核维护一个「就绪链表」，`epoll_wait` **直接返回有事件的 fd**，复杂度 O(1) 级别。

**③ 不受数量限制**
epoll 能轻松管理几十万连接（C100K 甚至 C1000K）。

```
select/poll：  我有10000个连接 → 你全检查一遍 → 告诉我有人事（具体是谁你自己找）→ O(n)
epoll：       我注册了10000个 → 哪个有事内核记着 → 直接把有事的几个给你 → O(1)
```

#### epoll 的两种触发模式

| 模式 | 说明 | 特点 |
| --- | --- | --- |
| **LT（水平触发，默认）** | 只要 fd 有数据可读，每次 `epoll_wait` 都会通知你 | 安全，不漏数据，但可能重复通知 |
| **ET（边缘触发）** | 只在状态**变化**时通知一次（从无到有） | 高效，但必须一次读完，否则丢通知 |

> 💡 **提示：** LT 模式更易用（没读完下次还会通知）；ET 模式性能更高但容易漏数据，必须配合非阻塞 IO 一次读到 EAGAIN。Nginx 默认用 epoll 的 ET 模式。

### 六、三者对比

| 对比项 | select | poll | epoll |
| --- | --- | --- | --- |
| 连接数限制 | 1024 | 无 | 无 |
| 每次调用拷贝 | 全部 fd | 全部 fd | 无（注册时一次） |
| 返回后查找方式 | 遍历全部 O(n) | 遍历全部 O(n) | 直接给就绪的 O(1) |
| 性能随连接数 | 下降明显 | 下降明显 | **几乎不变** |
| 平台 | Linux/Mac/Windows | Unix/Linux | **仅 Linux** |
| 适用场景 | 已淘汰 | 少见 | **高并发首选** |

> 💡 **提示：** epoll 是 Linux 独有的。Mac 用 `kqueue`，Windows 用 `IOCP`，思路类似但 API 不同。Java NIO 的 `Selector` 会根据平台自动选对应的底层实现。

### 七、Reactor 反应堆模式

基于 IO 多路复用，业界总结出了 **Reactor 模式**——这是 Netty、Nginx、Redis 等高性能组件的通用架构。

#### 核心思想

把「事件分离 + 事件分发 + 事件处理」分开：

```
                 ┌─────────────┐
  连接1 ────────►│             │── 读取事件 ──► 读处理器
  连接2 ────────►│  Reactor    │
  连接3 ────────►│ (多路复用器) │── 新连接事件 ──► 接受处理器
  连接N ────────►│             │
                 └─────────────┘
   ① 一个线程用 epoll 监听所有连接
   ② 有事件就分发到对应的处理器
```

#### 三种 Reactor 变体

**① 单 Reactor 单线程**

```
一个线程干所有事（监听 + 读写 + 业务）
```
- 简单，但无法利用多核，handler 慢了全卡住
- Redis 6.0 之前就是这种

**② 单 Reactor 多线程**

```
一个线程负责监听 + 接收，业务处理交给线程池
```
- 利用多核处理业务
- 但单 Reactor 仍是瓶颈

**③ 主从 Reactor 多线程**（主流，Netty 默认）

```
主 Reactor：只负责接收新连接
从 Reactor（多个）：负责已连接的读写
线程池：负责业务处理
```

```
新连接 ──► 主Reactor(accept) ──► 分配给 ──► 从Reactor1(读写)
                                  ├──► 从Reactor2(读写)
                                  └──► 从ReactorN(读写)
                                                         │
                                                      业务线程池
```

> 💡 **提示：** 主从 Reactor 多线程模式是**工业级高性能服务器的标配**。Netty 默认就是这个架构（`bossGroup` + `workerGroup`）。

---

### 八、Reactor 与 Proactor

| 模式 | 对应 IO 模型 | 特点 |
| --- | --- | --- |
| **Reactor** | 同步（NIO） | OS 通知「数据准备好了」，应用**自己来读** |
| **Proactor** | 异步（AIO） | OS 不仅通知，还**帮应用把数据读好**放进缓冲区 |

- Linux 下主要是 Reactor（基于 epoll + NIO）
- Windows 的 IOCP 是 Proactor 风格

---

### 九、实际应用

| 软件 | 用的什么 |
| --- | --- |
| **Nginx** | epoll + ET 模式 |
| **Redis** | epoll + 单线程 Reactor（6.0 后引入多线程 IO） |
| **Netty** | 对 epoll/Selector 的封装 + 主从 Reactor |
| **Java NIO** | Selector（Linux 下是 epoll） |

> 💡 **提示：** 这些高性能组件能用「单线程/少线程」扛住百万连接，秘密全是 **IO 多路复用（epoll）+ Reactor 模式**。

### 十、常见问题与注意事项

#### 1. select/poll/epoll 本质区别一句话

select/poll：每次都要把所有连接告诉内核，内核遍历后告诉你有事的，你自己再遍历找；epoll：连接注册一次，内核直接把有事的给你。

#### 2. epoll 一定比 select 好吗？

不一定。**连接数少且都很活跃**时，epoll 的优势不明显，甚至 select 更简单。epoll 优势在「**连接数多 + 大量空闲连接**」的场景。

#### 3. 惊群问题（Thundering Herd）

多线程同时 `epoll_wait` 同一个 fd，事件来时多个线程被唤醒争抢，只有一个能处理，其余白醒。Nginx 用锁解决此问题。了解即可。

#### 4. epoll 是同步还是异步？

epoll 本身是**同步**的（IO 多路复用属于同步 IO，因为「读数据」还是应用自己做的）。它只是「**事件通知**」是高效的。真正的异步是 AIO（Proactor）。

#### 5. Java 里怎么用 epoll？

不用直接调 epoll，Java 的 `Selector`（NIO）已封装。Netty 进一步封装并提供更易用的 API。详见 [[Netty框架]]。

### 十一、总结

| 要点 | 说明 |
| --- | --- |
| 核心思想 | 一个线程同时监控大量连接 |
| select | 有 1024 限制、O(n) 遍历、每次拷贝全部 fd |
| poll | 去掉数量限制，但仍 O(n) |
| **epoll** | 注册一次、直接给就绪 fd、O(1)、高性能首选 |
| 触发模式 | LT（水平，安全）、ET（边缘，高效） |
| Reactor | 基于多路复用的事件分发模式，主从多线程是主流 |

**一句话串联**：NIO（[[IO模型]]）的 `Selector` 底层用 epoll，epoll 让一个线程管海量连接，配合 Reactor 模式（主从多线程），就成了 Netty（[[Netty框架]]）这样的高性能网络框架。
