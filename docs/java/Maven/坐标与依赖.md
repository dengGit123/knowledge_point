### 一、概述

> 📖 [依赖机制简介](https://maven.apache.org/guides/introduction/introduction-to-the-dependency-mechanism.html) ｜ [POM 依赖参考](https://maven.apache.org/pom.html#Dependencies)

Maven 用**坐标**定位 jar，用**依赖**声明「我要用哪个 jar」。理解坐标和依赖，是看懂 `pom.xml` 的基础。

大白话：**坐标是 jar 的「身份证号」**，依赖是你在 `pom.xml` 里写的「**我要用这个 jar**」。但 jar 之间的关系错综复杂——A 依赖 B，B 依赖 C（传递依赖），还可能两个 jar 引入了同一个 D 的不同版本（冲突）。本文把这些讲透。

| 你将学到 | 说明 |
| --- | --- |
| GAV 坐标 | 怎么唯一定位一个 jar |
| 依赖范围 scope | compile / test / provided / runtime 等 |
| 传递依赖 | A → B → C 的传递规则 |
| 依赖冲突 | 版本冲突怎么发生、怎么解决 |
| SNAPSHOT vs Release | 快照版和发布版的区别 |

---

### 二、坐标（Coordinate）

#### 1. GAV 三要素

Maven 用三个元素唯一定位一个 jar，合称 **GAV**：

| 元素 | 含义 | 示例 | 类比 |
| --- | --- | --- | --- |
| `groupId` | 组织 / 公司 / 域名反写 | `com.alibaba` | 你家在哪个城市 |
| `artifactId` | 项目 / 模块名 | `fastjson` | 哪个小区 |
| `version` | 版本号 | `1.2.83` | 几号楼几单元 |

合起来 `com.alibaba:fastjson:1.2.83` 在**全球唯一**确定一个 jar。

```xml
<dependency>
    <groupId>com.alibaba</groupId>          <!-- 谁 -->
    <artifactId>fastjson</artifactId>       <!-- 什么东西 -->
    <version>1.2.83</version>               <!-- 哪个版本 -->
</dependency>
```

#### 2. 自己项目的坐标

自己的项目也要有 GAV，写在 `pom.xml` 顶部：

```xml
<groupId>com.example</groupId>
<artifactId>my-project</artifactId>
<version>1.0.0</version>
```

> 💡 **提示：** 当别的项目要引用你这个项目时，就写你的 GAV。所以 `groupId` 一般用域名反写（公司官网域名倒过来），保证全球唯一。

#### 3. 版本号规范

| 版本形式 | 含义 | 示例 |
| --- | --- | --- |
| Release（发布版） | 稳定、不再变 | `1.0.0`、`6.1.5` |
| SNAPSHOT（快照版） | 开发中、会持续更新 | `1.0.0-SNAPSHOT`、`6.1.5-SNAPSHOT` |
| `LATEST` | 远程仓库最新版（**不推荐**用） | `LATEST` |
| `RELEASE` | 远程仓库最新发布版（**不推荐**用） | `RELEASE` |

#### SNAPSHOT vs Release（重要）

```xml
<!-- Release：定了就不变，Maven 只下载一次，缓存本地 -->
<version>6.1.5</version>

<!-- SNAPSHOT：开发中，Maven 默认每天去远程仓库检查更新 -->
<version>6.1.5-SNAPSHOT</version>
```

| 特性 | Release | SNAPSHOT |
| --- | --- | --- |
| 是否可变 | ❌ 固定不变 | ✅ 持续更新 |
| 下载策略 | 下一次后永久缓存 | 定期（默认每天）检查更新 |
| 适用阶段 | 正式发布 | 团队内部联调开发 |
| 仓库 | 一般放 release 仓库 | 一般放 snapshot 仓库 |

> ⚠️ **注意：** **正式发布的项目千万别依赖别人的 SNAPSHOT 版本**——它会随时变，导致你的构建不可复现。SNAPSHOT 只用于团队内部多模块联调。

### 三、依赖范围（scope）

这是依赖里**最容易踩坑**也最常考的点。scope 决定一个依赖「**在什么阶段有效、是否传递**」。

#### 六种 scope

| scope | 对主程序 classpath | 对测试 classpath | 是否参与打包 | 典型例子 |
| --- | --- | --- | --- | --- |
| **compile**（默认） | ✅ | ✅ | ✅ | Spring、MyBatis |
| **test** | ❌ | ✅ | ❌ | JUnit、Mockito |
| **provided** | ✅ | ✅ | ❌ | Servlet API、Lombok |
| **runtime** | ❌ | ✅ | ✅ | MySQL 驱动、JDBC 实现 |
| **system** | ✅ | ✅ | ❌ | 本地手动 jar（**少用**） |
| **import** | — | — | — | 只能在 `dependencyManagement` 用 |

#### 各 scope 详解

```xml
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-core</artifactId>
    <version>6.1.5</version>
    <!-- 不写 scope 默认就是 compile -->
</dependency>

<!-- test：只在测试时用，不会打包进 jar -->
<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
    <scope>test</scope>
</dependency>

<!-- provided：编译、测试时需要，但运行时由容器（Tomcat）提供 -->
<dependency>
    <groupId>jakarta.servlet</groupId>
    <artifactId>jakarta.servlet-api</artifactId>
    <version>6.0.0</version>
    <scope>provided</scope>
</dependency>

<!-- runtime：编译不需要，运行时需要（一般通过反射/接口加载） -->
<dependency>
    <groupId>mysql</groupId>
    <artifactId>mysql-connector-java</artifactId>
    <version>8.0.33</version>
    <scope>runtime</scope>
</dependency>
```

#### 怎么选 scope（决策表）

| 场景 | 选哪个 |
| --- | --- |
| 项目核心库（Spring、业务工具） | `compile` |
| 单元测试框架（JUnit、Mockito） | `test` |
| 容器/环境已提供的（Servlet API、JSP API） | `provided` |
| 通过接口调用、运行时反射加载（JDBC 驱动、SLF4J 实现） | `runtime` |
| Lombok（编译期生成代码，运行时不需要） | `provided` |

> 💡 **提示：** `provided` 的精髓是「**运行环境已经有了，别重复打包**」。比如 Tomcat 自带 Servlet API，你的 war 包再打一份反而冲突。

### 四、传递依赖（Transitive Dependency）

#### 1. 什么是传递依赖

```
你的项目
  └─ 依赖 spring-core 6.1.5
        └─ spring-core 又依赖 spring-jcl 6.1.5
              └─ spring-jcl 又依赖 commons-logging...
```

你只声明了 `spring-core`，但 Maven 会**自动把 spring-jcl、commons-logging 都下载并加到 classpath**——这就是传递依赖。

好处：**不用手动处理「依赖的依赖」**，Maven 全自动。

#### 2. 传递依赖的范围规则

传递依赖的 scope 受「直接依赖 scope」和「传递依赖自身 scope」**共同决定**，规则较复杂，简化记忆：

| 直接依赖(行) ／ 传递依赖(列) | compile | provided | runtime | test |
| --- | --- | --- | --- | --- |
| **compile** | compile | ❌ | runtime | test |
| **provided** | provided | ❌ | provided | test |
| **runtime** | runtime | ❌ | runtime | test |
| **test** | test | ❌ | test | test |

> 💡 **提示：** 不用死记。记住关键规律：**`provided` 和 `system` 的依赖不传递**；其余的大多是「取两者中范围较小的一个」。

### 五、依赖冲突与解决

#### 1. 冲突怎么发生

```
你的项目
  ├─ 依赖 A（A 依赖 commons-logging 1.1）
  └─ 依赖 B（B 依赖 commons-logging 1.2）
```

两个 jar 都引入了 `commons-logging`，但版本不同——**到底用哪个？**

#### 2. Maven 的解决原则

**原则一：最短路径优先（ shortest path）**

```
你的项目 ─► A ─► commons-logging 1.1      （路径长度 2）
你的项目 ─► B ─► commons-logging 1.2      （路径长度 2，相同长度）
```

路径相同时，**先声明的优先**（pom.xml 里写在前的赢）。

```
你的项目 ─► C ─► D ─► commons-logging 1.0  （路径长度 3）
你的项目 ─► A ─► commons-logging 1.1       （路径长度 2）← 用这个
```

C 引入的版本路径更长（3 层），被 A 的（2 层）**淘汰**。

**原则二：先声明优先（nearer / first declared）**

路径长度相同时，**在 `pom.xml` 中先写的依赖胜出**。

#### 3. 手动解决冲突

当自动判定不是你想要的版本，用 `<exclusions>` 排除掉不要的版本：

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>A</artifactId>
    <version>1.0.0</version>
    <exclusions>
        <!-- 排除掉 A 带过来的旧版本 commons-logging -->
        <exclusion>
            <groupId>commons-logging</groupId>
            <artifactId>commons-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>

<!-- 直接声明想要的版本 -->
<dependency>
    <groupId>commons-logging</groupId>
    <artifactId>commons-logging</artifactId>
    <version>1.2</version>
</dependency>
```

#### 4. 查看依赖树（排查冲突必备）

```bash
mvn dependency:tree
```

输出会列出所有依赖及其传递关系：

```
com.example:my-project:jar:1.0.0
+- org.springframework:spring-core:jar:6.1.5:compile
|  \- org.springframework:spring-jcl:jar:6.1.5:compile
+- com.example:A:jar:1.0.0:compile
|  \- (commons-logging:commons-logging:jar:1.1:compile - omitted for conflict with 1.2)
```

`omitted for conflict` 就是被冲突淘汰的版本。

> 💡 **提示：** IDEA 装个 **Maven Helper** 插件，可视化查看依赖树、一键排除冲突，排查神器。

#### 5. 依赖冲突的典型症状

| 症状 | 可能原因 |
| --- | --- |
| `NoSuchMethodError` / `ClassNotFoundException` | 同一 jar 多版本，运行时用了旧版本 |
| `NoClassDefFoundError` | 缺少某个依赖，或被错误排除 |
| 编译通过运行报错 | 编译期和运行期 classpath 不一致（scope 配错） |

### 六、可选依赖（optional）与排除依赖（exclusions）

#### 1. optional：可选依赖

```xml
<!-- 在 A 的 pom.xml 里，声明对 X 的依赖是 optional -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>X</artifactId>
    <version>1.0.0</version>
    <optional>true</optional>   <!-- 标记为可选 -->
</dependency>
```

效果：**A 自己能用 X，但当别的项目依赖 A 时，X 不会传递过去**。

| 对比 | `optional=true` | `<exclusions>` |
| --- | --- | --- |
| 谁来控制 | **被依赖方**（A 的作者）声明 | **依赖方**（你）主动排除 |
| 目的 | 「我这功能可选，你要用自己加」 | 「我不要你带来的这个传递依赖」 |

#### 2. exclusions：排除依赖

见上一节「手动解决冲突」。

> 💡 **提示：** 常见搭配——日志框架冲突。Spring 默认带 `commons-logging`，而项目想用 `logback` + `slf4j`，所以要排除掉 Spring 的 `commons-logging`，再引入 `jcl-over-slf4j` 桥接。

### 七、统一管理依赖版本（dependencyManagement）

多模块项目里，要保证所有子模块用同一版本的依赖，靠 `dependencyManagement`：

```xml
<!-- 父 pom.xml 里只声明版本，不真正引入 -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework</groupId>
            <artifactId>spring-core</artifactId>
            <version>6.1.5</version>   <!-- 版本统一在这管 -->
        </dependency>
    </dependencies>
</dependencyManagement>
```

```xml
<!-- 子模块里只写坐标，不写 version -->
<dependency>
    <groupId>org.springframework</groupId>
    <artifactId>spring-core</artifactId>
    <!-- 不写 version，自动继承父 pom 的 6.1.5 -->
</dependency>
```

详见 [[pom配置/dependencyManagement配置]]。

### 八、常见问题与注意事项

#### 1. `provided` 和 `runtime` 的区别记混

- `provided`：编译 + 测试要，**打包不要**（运行环境提供）
- `runtime`：编译不要，**测试 + 打包要**（运行时反射加载）

#### 2. 传递依赖「不见了」

某依赖不传递，检查它的 scope 是不是 `provided` / `system`，或 `optional=true`。

#### 3. 本地仓库有 jar，但项目找不到

- scope 配成了 `test`，但你在主程序里用了
- `mvn clean install` 重新安装
- IDEA 没刷新，`Reload Project`

#### 4. 版本冲突导致运行报错

用 `mvn dependency:tree` 查依赖树，用 `<exclusions>` 排除旧版本，或用 `dependencyManagement` 强制统一版本。

#### 5. 用 `LATEST` / `RELEASE` 动态版本

**不推荐**——构建不可复现，可能今天能跑明天就挂。固定写明确版本号。

### 九、总结

| 要点 | 说明 |
| --- | --- |
| 坐标 GAV | `groupId:artifactId:version` 唯一定位 jar |
| scope | 控制依赖在哪些阶段/路径有效 |
| 传递依赖 | A → B → C 自动传递，`provided`/`system` 不传递 |
| 冲突原则 | 最短路径优先 → 路径相同先声明优先 |
| 解决冲突 | `<exclusions>` 排除 + 手动声明版本 |
| 版本管理 | 多模块用 `dependencyManagement` 统一版本 |
| SNAPSHOT | 开发中的可变版本，Release 是稳定发布版 |
