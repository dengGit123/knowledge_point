### 一、概述

> 📖 [生命周期简介](https://maven.apache.org/guides/introduction/introduction-to-the-lifecycle.html) ｜ [Maven 命令行参考](https://maven.apache.org/ref/current/maven-embedder/cli.html)

Maven 的**生命周期（Lifecycle）**是一组有序的构建阶段（phase）。你执行某个阶段时，它**前面的所有阶段会自动先执行**——这就是为什么一句 `mvn install` 能自动完成清理、编译、测试、打包、安装的全流程。

大白话：**生命周期是一条「流水线」**，流水线上有很多工位（phase）。你只要喊一声「停在第 5 个工位」，前面 1~5 号工位会依次自动干完。真正在每个工位干活的是**插件**（工人），生命周期只是规定了工位顺序。

| 你将学到 | 说明 |
| --- | --- |
| 三大生命周期 | clean / default / site |
| phase（阶段） | 流水线上的工位 |
| phase 与 goal | 工位与工人任务的关系 |
| mvn 常用命令 | 日常构建命令 |
| 常用参数 | `-D`、`-P`、`-pl`、`-am` 等 |

> 💡 **提示：** 理解生命周期后，你就明白为什么 `mvn clean install` 能一条龙干活，以及为什么命令的顺序很重要。

---

### 二、三大生命周期

Maven 有**三套互相独立**的生命周期：

| 生命周期 | 作用 | 典型 phase |
| --- | --- | --- |
| **clean** | 清理项目（删 `target/`） | `pre-clean` → `clean` → `post-clean` |
| **default** | 构建主流程（编译、测试、打包、发布） | `validate` → `compile` → `test` → `package` → `install` → `deploy` |
| **site** | 生成项目站点文档 | `site` → `site-deploy` |

> ⚠️ **注意：** 这三套生命周期**互相独立**。执行 `clean` 生命周期不会触发 `default` 的 phase，除非你在命令里同时写（如 `mvn clean package`）。

---

### 三、default 生命周期（最常用）

`default` 是最长、最常用的生命周期，包含十几个 phase。日常只关心加粗的几个：

| phase | 作用 | 绑定的插件 goal |
| --- | --- | --- |
| `validate` | 校验项目、配置是否正确 | — |
| `initialize` | 初始化构建状态 | — |
| **`generate-sources`** | 生成源代码 | — |
| **`compile`** | **编译主程序**（`src/main/java` → `target/classes`） | `compiler:compile` |
| `process-classes` | 处理编译后的 class | — |
| **`generate-test-sources`** | 生成测试源码 | — |
| **`test-compile`** | **编译测试代码**（`src/test/java`） | `compiler:testCompile` |
| `process-test-classes` | 处理测试 class | — |
| **`test`** | **运行单元测试**（不会被打包失败的测试中断） | `surefire:test` |
| `prepare-package` | 打包前准备 | — |
| **`package`** | **打包**（jar / war / ear） | `jar:jar` / `war:war` |
| **`verify`** | 集成测试校验 | `failsafe:integration-test` |
| **`install`** | **安装到本地仓库**（`~/.m2`） | `install:install` |
| **`deploy`** | **发布到远程仓库**（私服） | `deploy:deploy` |

#### 记忆口诀

```
编译(compile) → 测试编译(test-compile) → 测试(test)
   → 打包(package) → 安装(install) → 发布(deploy)
```

#### 关键规律：执行某个 phase，前面的都自动跑

```bash
mvn compile      # 只跑到 compile
mvn test         # 跑到 test（自动先 compile、test-compile）
mvn package      # 跑到 package（自动先 compile、test...）
mvn install      # 跑到 install（全流程，最后装到本地仓库）
```

> 💡 **提示：** 这就是为什么 `mvn install` 一条命令能完成「编译 + 测试 + 打包 + 安装」——因为它会从生命周期的第一个 phase 一路执行到 `install`。

### 四、phase 与 goal 的关系

| 概念 | 是什么 | 类比 |
| --- | --- | --- |
| **phase（阶段）** | 生命周期上的一个工位 | 流水线的工位编号 |
| **goal（目标）** | 插件的一个具体任务 | 工人在工位上干的具体活 |
| **plugin（插件）** | 提供 goal 的工具 | 工人 |

一个 phase 可以绑定**多个 goal**（多个插件任务）。比如 `package` 阶段，对 jar 项目会绑定 `maven-jar-plugin:jar`。

#### 直接执行插件 goal（绕过生命周期）

格式：`mvn 插件:目标` 或 `mvn 插件GroupId:artifactId:版本:goal`

```bash
mvn compiler:compile           # 直接调编译插件
mvn surefire:test              # 直接调测试插件
mvn dependency:tree            # 查看依赖树
mvn tomcat7:run                # 用 Tomcat 插件启动项目
```

#### goal 写法说明

`compiler:compile` 这种简写，靠「插件前缀（prefix）」找插件。完整写法是 `groupId:artifactId:version:goal`，但通常用前缀更简洁。

### 五、常用命令大全

#### 1. 基础构建命令

```bash
mvn clean                # 清理（删除 target/ 目录）
mvn validate             # 校验项目正确性
mvn compile              # 编译主程序
mvn test                 # 运行测试
mvn test-compile         # 只编译测试代码（不运行）
mvn package              # 打包成 jar/war
mvn install              # 打包并安装到本地仓库
mvn deploy               # 发布到远程仓库（私服）
```

#### 2. 组合命令（最常用）

```bash
mvn clean package        # 清理 + 重新打包（日常构建）
mvn clean install        # 清理 + 重新打包 + 装本地仓库（多模块开发）
mvn clean deploy         # 清理 + 重新打包 + 发私服（发布）
mvn clean package -Dmaven.test.skip=true   # 跳过测试打包
```

> ⚠️ **注意：** 命令顺序有意义。`mvn package clean` 会先打包、再清理（白干了）。习惯写 `clean` 在前。

#### 3. 跳过测试的两种方式

| 命令 | 效果 |
| --- | --- |
| `mvn package -DskipTests` | **编译**测试代码，但**不运行**测试 |
| `mvn package -Dmaven.test.skip=true` | **既不编译也不运行**测试（更快） |

```bash
# 推荐日常快速构建用：
mvn clean package -DskipTests
```

> 💡 **提示：** 生产环境发布**不要跳过测试**！测试是质量保障，跳过容易出事。

#### 4. 查看依赖相关信息

```bash
mvn dependency:tree                          # 查看完整依赖树
mvn dependency:tree -Dincludes=org.springframework   # 只看 Spring 相关依赖
mvn dependency:analyze                       # 分析依赖（哪些没用到、哪些隐式使用）
mvn dependency:list                          # 列出所有已解析的依赖
mvn help:effective-pom                       # 查看生效的完整 POM（含继承的）
mvn help:effective-settings                  # 查看生效的 settings
```

#### 5. 创建项目

```bash
# 生成普通 Java 项目
mvn archetype:generate -DgroupId=com.example -DartifactId=my-app \
    -DarchetypeArtifactId=maven-archetype-quickstart -DinteractiveMode=false

# 生成 Web 项目
mvn archetype:generate -DgroupId=com.example -DartifactId=my-web \
    -DarchetypeArtifactId=maven-archetype-webapp -DinteractiveMode=false
```

#### 6. 多模块相关命令

```bash
mvn clean install -pl module-a         # 只构建 module-a 模块
mvn clean install -pl module-a -am     # 构建 module-a 及它依赖的其他模块
mvn clean install -pl module-a -amd    # 构建 module-a 及依赖它的模块
```

详见 [[多模块/多模块实战]]。

### 六、常用命令行参数

| 参数 | 作用 | 示例 |
| --- | --- | --- |
| `-D<属性>=<值>` | 设置系统属性（覆盖 pom 里的属性） | `-DskipTests` |
| `-P<profileId>` | 激活 profile（多环境构建） | `-Pprod` |
| `-pl <模块>` | 只构建指定模块（projects list） | `-pl module-a` |
| `-am` | 同时构建依赖的模块（also make） | `-am` |
| `-amd` | 同时构建依赖该模块的模块（also make dependents） | `-amd` |
| `-rf <模块>` | 从指定模块恢复构建（resume from） | `-rf :module-c` |
| `-o` | 离线模式（不联网，只用本地仓库） | `-o` |
| `-U` | 强制检查 SNAPSHOT 更新 | `-U` |
| `-f <pom文件>` | 指定 pom.xml 路径 | `-f sub/pom.xml` |
| `-q` / `-X` | 安静模式 / 调试模式（详细输出） | `-X` |
| `-T 4` | 多线程构建（4 线程） | `-T 1C`（每核 1 线程） |

#### 常用组合示例

```bash
# 生产环境打包（激活 prod profile，跳过测试）
mvn clean package -Pprod -DskipTests

# 强制更新所有 SNAPSHOT 依赖
mvn clean install -U

# 离线模式快速构建（适合网络不好时）
mvn clean package -o

# 多线程加速构建
mvn clean install -T 1C
```

### 七、多模块项目中的反应堆（Reactor）

在多模块项目里执行 `mvn`，Maven 会构建一个「**反应堆**」——分析所有模块的依赖关系，决定**构建顺序**。

```
mvn clean install （在父项目根目录执行）
       │
       ▼
  反应堆分析模块依赖
       │
       ▼
  按依赖顺序构建：common → service → web
       （被依赖的模块先构建）
```

> 💡 **提示：** 反应堆自动排序，所以你不用担心「先 build 哪个模块」。详见 [[多模块/多模块实战]]。

### 八、常见问题与注意事项

#### 1. 测试失败导致构建中断

`mvn package` 默认会运行测试，**任何一个测试失败就中断**。调试时可用 `-DskipTests` 临时跳过。

#### 2. `clean` 一定要加吗

不强制，但**强烈建议**。不加 `clean`，旧的编译产物可能残留，导致奇怪的 bug（比如删了类但 `target/` 里还有）。养成 `mvn clean xxx` 的习惯。

#### 3. `install` 和 `deploy` 的区别

- `install`：装到**本地**仓库（`~/.m2`），只有你这台机器能用
- `deploy`：发布到**远程**仓库（私服），整个团队能用

#### 4. 离线模式 `-o` 报错

本地仓库里没有所需的依赖，离线模式下无法下载。去掉 `-o`，联网下载一次。

#### 5. `mvn` 命令找不到（command not found）

环境变量没配好，检查 `MAVEN_HOME` 和 `PATH`。详见 [[Maven概述]] 安装部分。

#### 6. 命令行中文乱码

Windows 下测试输出乱码，加 `-Dfile.encoding=UTF-8`，或在 IDEA 的 Maven Runner 里设置 VM Options。

| 要点 | 说明 |
| --- | --- |
| 三大生命周期 | clean（清理）/ default（构建）/ site（文档） |

| 核心规律 | 执行某 phase 时，前面所有 phase 自动先执行 |
| phase vs goal | phase 是工位，goal 是插件的具体任务 |
