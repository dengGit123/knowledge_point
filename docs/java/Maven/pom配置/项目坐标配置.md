### 一、概述

> 📖 [POM 坐标（groupId/artifactId/version）](https://maven.apache.org/pom.html#Maven_Coordinates) ｜ [打包方式 packaging](https://maven.apache.org/pom.html#Packaging)

项目的**坐标（GAV）**是 `pom.xml` 里**最核心、唯一必须配置**的部分——它决定了项目在 Maven 世界里的「身份证号」。本篇讲清坐标各元素的含义、`packaging` 打包方式、以及 `name` 等项目描述信息怎么配。

大白话：**坐标就是「项目叫什么、在哪个组织、第几版」**。这三个信息合起来，能在全球唯一定位你的项目。别的项目要引用你，就写你的坐标。

| 你将学到 | 说明 |
| --- | --- |
| GAV 三要素 | `groupId` / `artifactId` / `version` 怎么填 |
| `packaging` | 打包方式（jar/war/pom/ear） |
| `name` 等 | 项目描述信息 |
| 必须性 | 哪些必须、哪些可选 |

---

### 二、坐标 GAV 详解

#### 1. `groupId`：组织 / 公司

项目所属的**组织或团队**，通常用**域名反写**保证全球唯一。

```xml
<groupId>com.alibaba</groupId>          <!-- 阿里巴巴 -->
<groupId>org.springframework</groupId>  <!-- Spring 官方 -->
<groupId>com.example</groupId>          <!-- 你自己（example.com 反写） -->
```

| 写法 | 含义 |
| --- | --- |
| `com.company` | 公司官网 `company.com` 反写 |
| `com.company.project` | 加上项目/部门细分 |
| `org.apache` | 开源组织（apache.org） |

> 💡 **提示：** 个人项目或学习用，随便起一个如 `com.github.你的用户名` 或 `com.example` 即可。**企业项目必须用公司域名反写**，避免和别人的冲突。

#### 2. `artifactId`：项目 / 模块名

项目或模块的**具体名字**，通常小写、用 `-` 分隔。

```xml
<artifactId>fastjson</artifactId>        <!-- fastjson 库 -->
<artifactId>spring-core</artifactId>     <!-- Spring 核心 -->
<artifactId>user-service</artifactId>    <!-- 用户服务模块 -->
```

约定：

- 小写字母
- 多词用 `-` 分隔（如 `user-service`，不用驼峰或下划线）
- 生成的 jar 包名就是 `artifactId-version.jar`（如 `spring-core-6.1.5.jar`）

#### 3. `version`：版本号

```xml
<version>1.0.0</version>                <!-- Release 发布版 -->
<version>1.0.0-SNAPSHOT</version>       <!-- Snapshot 快照版 -->
<version>1.0.0-RELEASE</version>        <!-- 显式标 RELEASE 后缀 -->
```

版本号约定（语义化版本 SemVer）：

```
主版本.次版本.修订号
   1   .  0   .  0
```

| 版本段 | 何时升 | 含义 |
| --- | --- | --- |
| 主版本（major） | 不兼容的大改动 | 1.x → 2.x，API 可能不兼容 |
| 次版本（minor） | 新增功能、向下兼容 | 1.1 → 1.2 |
| 修订号（patch） | 修 bug | 1.0.0 → 1.0.1 |

> ⚠️ **注意：** 关于 SNAPSHOT（快照版）和 Release（发布版）的区别，详见 [[../坐标与依赖]] 的「SNAPSHOT vs Release」。

### 三、完整的坐标配置示例

```xml
<project>
    <modelVersion>4.0.0</modelVersion>

    <!-- ===== GAV 坐标（必须） ===== -->
    <groupId>com.example</groupId>
    <artifactId>my-project</artifactId>
    <version>1.0.0</version>

    <!-- ===== 打包方式 ===== -->
    <packaging>jar</packaging>
</project>
```

这个项目在 Maven 世界里的完整坐标就是：

```
com.example:my-project:1.0.0
```

打出来的包是 `my-project-1.0.0.jar`。

### 四、`packaging`：打包方式

| 取值 | 含义 | 典型场景 |
| --- | --- | --- |
| **`jar`**（默认） | 打成 jar 包 | 普通后端项目、工具库 |
| **`war`** | 打成 war 包 | 传统 Web 项目（部署到 Tomcat） |
| **`pom`** | 不打包，只作配置 | **父项目 / 聚合项目** |
| **`ear`** | 企业级应用包 | Java EE（少见） |
| **`maven-plugin`** | 打成 Maven 插件 | 自己开发 Maven 插件 |
| **`bundle`** | OSGi bundle | OSGi 应用（少见） |

```xml
<!-- 普通 jar 项目 -->
<packaging>jar</packaging>

<!-- Web 项目，部署到外部 Tomcat -->
<packaging>war</packaging>

<!-- 父项目（只管理子模块，自己不打包） -->
<packaging>pom</packaging>
```

> 💡 **提示：** `packaging` 决定了项目会绑定哪些插件、走哪条构建流程。比如 `war` 会额外绑定 `maven-war-plugin`，`pom` 则完全跳过编译打包。

> ⭐ **重点：多模块项目的父项目必须配 `packaging=pom`**。详见 [[../多模块/父模块]]。

### 五、项目描述信息（可选）

```xml
<!-- 项目展示名（更友好，可含中文/空格） -->
<name>My Project</name>

<!-- 项目详细描述 -->
<description>这是一个用户管理系统</description>

<!-- 项目主页地址 -->
<url>https://github.com/example/my-project</url>

<!-- 开始年份 -->
<inceptionYear>2024</inceptionYear>

<!-- 许可证 -->
<licenses>
    <license>
        <name>Apache License, Version 2.0</name>
        <url>https://www.apache.org/licenses/LICENSE-2.0.txt</url>
    </license>
</licenses>

<!-- 开发者 -->
<developers>
    <developer>
        <id>dyc</id>
        <name>张三</name>
        <email>zhangsan@example.com</email>
    </developer>
</developers>

<!-- 代码仓库地址 -->
<scm>
    <url>https://github.com/example/my-project</url>
    <connection>scm:git:git://github.com/example/my-project.git</connection>
</scm>

<!-- 组织信息 -->
<organization>
    <name>Example Inc.</name>
    <url>https://example.com</url>
</organization>
```

这些信息**纯属描述性**，不影响构建，主要作用：

- 在 Maven Central 等仓库的页面上展示
- 生成项目站点文档（`mvn site`）
- `name` 会显示在 IDE 项目列表、Maven 仓库里

> 💡 **提示：** 日常内部项目，配个 `name` 和 `description` 就够了。`licenses`、`developers`、`scm` 这些主要是**开源发布到中央仓库**时才需要填全。

### 六、是否必须配置（汇总表）

| 标签 | 是否必须 | 说明 |
| --- | --- | --- |
| `groupId` | ✅ 必须 | 继承父 pom 时可省略 |
| `artifactId` | ✅ 必须 | 永远必须 |
| `version` | ✅ 必须 | 继承父 pom 时可省略 |
| `packaging` | ❌ 可选 | 不写默认 `jar` |
| `name` | ❌ 可选 | 描述用 |
| `description` | ❌ 可选 | 描述用 |
| `url` | ❌ 可选 | 描述用 |
| `licenses` / `developers` / `scm` | ❌ 可选 | 发布到中央仓库时用 |

> 💡 **提示：** **继承父 pom 的子模块**，连 `groupId` 和 `version` 都能省略（自动继承父 pom 的）。`artifactId` 不能省（每个子模块要有自己的名字）。详见 [[../多模块/子模块]]。

### 七、坐标命名的最佳实践

#### 1. groupId 用域名反写

```xml
<!-- ✅ 正确：公司域名反写 -->
<groupId>com.example</groupId>

<!-- ❌ 不规范：随便起，可能和别人冲突 -->
<groupId>myproject</groupId>
```

#### 2. artifactId 简洁、小写、用分隔符

```xml
<!-- ✅ 正确 -->
<artifactId>user-service</artifactId>
<artifactId>order-api</artifactId>

<!-- ❌ 不规范 -->
<artifactId>UserService</artifactId>      <!-- 不用大写 -->
<artifactId>user_service</artifactId>     <!-- 不用下划线 -->
```

#### 3. version 区分 Snapshot 和 Release

```xml
<!-- 开发阶段（团队内部联调） -->
<version>1.0.0-SNAPSHOT</version>

<!-- 正式发布 -->
<version>1.0.0</version>
```

#### 4. 多模块项目命名

```
父项目 artifactId：my-app（packaging=pom）
子模块：
  my-app-common      （公共模块）
  my-app-user-api    （用户接口）
  my-app-user-service（用户服务）
  my-app-web         （启动模块）
```

用「父名-子名」的命名，一眼看出归属关系。

### 八、常见问题与注意事项

#### 1. `groupId` 和 `package`（Java 包名）的关系

两者**没有强制关系**，但**通常保持一致或相关**。比如 `groupId=com.example`，Java 代码包名一般也是 `com.example.xxx`。

#### 2. 改了版本号，本地仓库的旧 jar 还在

本地仓库会同时保留多个版本的 jar。`mvn install` 新版本不会删旧版本。手动清理本地仓库对应目录即可。

#### 3. `artifactId` 里用了大写

能跑，但不规范。生成的 jar 名也会带大写（`MyProject-1.0.0.jar`），不符合社区习惯。坚持小写。

#### 4. 忘记配 `packaging` 导致打成 jar 而非 war

Web 项目要部署到 Tomcat 必须 `packaging=war`。不配默认 jar，部署会失败。

#### 5. 父项目忘了配 `packaging=pom`

父项目默认是 jar，Maven 会尝试编译打包，但父项目没有源码，报错。父项目必须 `packaging=pom`。

### 九、总结

| 要点 | 说明 |
| --- | --- |
| GAV | `groupId`（组织）+ `artifactId`（项目名）+ `version`（版本） |
| 必须配置 | `modelVersion` + GAV（继承时可省部分） |
| packaging | jar（默认）/ war / pom（父项目）/ maven-plugin |
| 描述信息 | `name`/`description`/`url` 等可选，发布到中央仓库才填全 |
| 命名规范 | groupId 域名反写，artifactId 小写用 `-`，version 用语义化 |
