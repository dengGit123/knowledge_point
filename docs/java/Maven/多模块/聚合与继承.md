### 一、概述

> 📖 [Project Aggregation vs Project Inheritance](https://maven.apache.org/guides/introduction/introduction-to-the-pom.html#Project_Inheritance_vs_Project_Aggregation)

「**聚合**」和「**继承**」是 Maven 多模块的**两个核心机制**，也是最容易混淆的概念。本文专门对比两者，帮你彻底分清。

大白话：

- **聚合**是「**一起干活**」——父项目把子模块们叫到一起，`mvn install` 一条命令全建。靠的是父 pom 的 `<modules>`。
- **继承**是「**抄作业**」——子模块复用父 pom 的配置，省得每个都写一遍。靠的是子 pom 的 `<parent>`。

两者是**完全独立**的机制，只是实际项目里常常一起用。

| 你将学到 | 说明 |
| --- | --- |
| 聚合 | 目的、实现、方向 |
| 继承 | 目的、实现、方向 |
| 区别对比 | 一张表看清 |
| 关系 | 独立但常配对 |

### 二、聚合（Aggregation）

#### 定义

**聚合**：在一个 pom 里声明「我包含哪些子模块」，从而能用**一条命令构建所有子模块**。

#### 实现

**父 pom** 写 `<modules>`：

```xml
<project>
    <artifactId>my-app</artifactId>
    <packaging>pom</packaging>

    <modules>
        <module>my-app-common</module>
        <module>my-app-service</module>
    </modules>
</project>
```

#### 目的

- **批量构建**：`mvn install` 一条命令构建所有模块
- **统一管理构建流程**：反应堆自动排序

#### 方向：父 → 子

是**父项目**知道有哪些子模块（父指向子）。子模块本身**不知道**自己被聚合了。

> 💡 **提示：** 详见 [[../pom配置/modules配置]]。

---

### 三、继承（Inheritance）

#### 定义

**继承**：子模块复用父 pom 的配置（版本号、依赖标准、插件配置等），避免重复。

#### 实现

**子 pom** 写 `<parent>`：

```xml
<project>
    <parent>
        <groupId>com.example</groupId>
        <artifactId>my-app-parent</artifactId>
        <version>1.0.0</version>
        <relativePath>../pom.xml</relativePath>
    </parent>

    <artifactId>my-app-common</artifactId>
</project>
```

#### 目的

- **复用配置**：版本号、依赖、插件配置只写一次
- **统一管理**：一处修改，处处生效

#### 方向：子 → 父

是**子模块**知道自己的父 pom 是谁（子指向父）。父 pom 本身**不知道**有哪些子模块继承它。

> 💡 **提示：** 详见 [[父模块]] 和 [[子模块]]。

---

### 四、聚合 vs 继承（核心对比）

| 对比项 | 聚合 | 继承 |
| --- | --- | --- |
| 目的 | **一起构建**（批量打包） | **复用配置**（少写重复） |
| 谁配 | 父 pom 配 `<modules>` | 子 pom 配 `<parent>` |
| 方向 | 父 → 子（父知道子） | 子 → 父（子知道父） |
| 关键标签 | `<modules>` | `<parent>` |
| 是否要求 `packaging=pom` | ✅ 父必须 pom | ✅ 父必须 pom |
| 能否脱离对方独立用 | ✅ 能 | ✅ 能 |

#### 一句话区分

```
聚合：父 pom 问「我有哪些孩子？」     → 为了"一起 build"
继承：子 pom 问「我爹是谁？」         → 为了"抄配置"
```

---

### 五、两者的关系：独立，但常配对

聚合和继承是**两个独立的机制**，理论上可以单独使用，组合出四种情况：

| 情况 | 说明 | 实际常见吗 |
| --- | --- | --- |
| 既聚合又继承 | 父写 modules，子写 parent | ✅ 最常见（标准多模块项目） |
| 只聚合不继承 | 父写 modules，子不写 parent | ⚠️ 偶尔（批量构建无关项目） |
| 只继承不聚合 | 子写 parent，父不写 modules | ⚠️ 常见（如继承 spring-boot-starter-parent） |
| 都不用 | 各自独立 | ❌ 那就是单模块项目 |

#### 情况一：既聚合又继承（标准多模块）

```
my-app/                      ← 父 pom：packaging=pom + <modules> + 共享配置
├── pom.xml
├── my-app-common/           ← 子 pom：<parent> 指向父
│   └── pom.xml
└── my-app-service/          ← 子 pom：<parent> 指向父
    └── pom.xml
```

```xml
<!-- 父 pom：聚合 + 提供继承配置 -->
<packaging>pom</packaging>
<modules>
    <module>my-app-common</module>
    <module>my-app-service</module>
</modules>
<!-- + properties / dependencyManagement / pluginManagement -->

<!-- 子 pom：继承父 -->
<parent>
    <artifactId>my-app</artifactId>
    <relativePath>../pom.xml</relativePath>
</parent>
<artifactId>my-app-common</artifactId>
```

> 💡 **提示：** 这就是 99% 的企业多模块项目的写法。完整实战见 [[多模块实战]]。

#### 情况二：只聚合不继承

```
batch-build/                 ← 父 pom：只有 <modules>
├── pom.xml
├── project-a/               ← 不写 <parent>
└── project-b/               ← 不写 <parent>
```

子项目互不相干，只是借父项目「**一起构建**」。

#### 情况三：只继承不聚合

```xml
<!-- 任意 Spring Boot 单模块项目 -->
<project>
    <parent>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-parent</artifactId>
        <version>3.2.0</version>
    </parent>
    <artifactId>my-single-app</artifactId>
</project>
```

继承远程父 pom 的配置，但父 pom（spring-boot-starter-parent）**不聚合**你的项目。

---

### 六、为什么实际项目「聚合 + 继承」一起用

| 单独聚合 | 单独继承 | 聚合 + 继承 |
| --- | --- | --- |
| 能批量构建 | 能复用配置 | ✅ 两者兼得 |
| ❌ 配置要各写各的（重复） | ❌ 要逐个模块手动 build | ✅ 一处配置 + 一条命令 |

**聚合解决「构建效率」，继承解决「配置复用」，两者各管一摊，配合使用最完美。**

### 七、图解两者的协作

```
         my-app (父 pom)
         packaging=pom
         ┌─────────────────────────┐
         │ 1. <modules>  → 聚合     │ ← 父知道有哪些子（一起 build）
         │ 2. properties            │
         │ 3. dependencyManagement │ ← 共享配置（给子继承）
         │ 4. pluginManagement     │
         └─────────────────────────┘
              ▲              │
   继承（子抄配置）          聚合（父叫子一起干活）
              │              ▼
    ┌─────────┴──────────────────────┐
    │                                │
 my-app-common                   my-app-service
 <parent> my-app </parent>       <parent> my-app </parent>
 ① 继承父的配置                   ① 继承父的配置
 ② 被父聚合（一起 build）         ② 被父聚合（一起 build）
```

### 八、常见问题与注意事项

#### 1. 把聚合和继承当成一回事

两者**方向相反、目的不同**。聚合是父→子（构建），继承是子→父（配置）。

#### 2. 以为父 pom 写了 `<modules>` 子模块就继承了

不是。`<modules>` 只管「**一起构建**」，继承要子模块自己写 `<parent>`。

#### 3. 以为子模块写了 `<parent>` 就能被聚合

不是。继承只是「抄配置」。要让父项目一键构建子模块，父 pom 必须写 `<modules>`。

#### 4. 父 pom 不是 `packaging=pom`

无论聚合还是继承，**父 pom 都必须 `packaging=pom`**。

#### 5. 忘了把聚合和继承配齐

常见错误：父 pom 写了 `<modules>`，但子模块没写 `<parent>`（或反之）。结果要么配置没复用、要么没法一键构建。

---

### 九、总结

| 要点 | 聚合 | 继承 |
| --- | --- | --- |
| 目的 | 一起构建 | 复用配置 |
| 谁配 | 父 pom `<modules>` | 子 pom `<parent>` |
| 方向 | 父 → 子 | 子 → 父 |
| 关系 | 独立机制，常与继承配对 | 独立机制，常与聚合配对 |

**核心记忆**：聚合管「**一起 build**」，继承管「**抄配置**」，实际项目两者一起用。
