### 一、概述

> 📖 [Project Inheritance（项目继承）](https://maven.apache.org/guides/introduction/introduction-to-the-pom.html#Project_Inheritance) ｜ [父 POM 示例](https://maven.apache.org/ref/current/maven-model-builder/super-pom.html)

**父模块（父 pom）**是多模块项目里的「**配置中心**」。它把所有子模块共享的配置（依赖版本、插件配置、属性）集中放一处，子模块**继承**它，避免每个子模块重复配置。

大白话：父 pom 像家族的「**家训**」——规定「全家都用 JDK 17、Spring 6.1.5、UTF-8 编码」。子模块遵循家训，**不用各自再写一遍**。要改规矩，改家训一处即可。

| 你将学到 | 说明 |
| --- | --- |
| 父模块是什么 | 配置中心，被子模块继承 |
| 怎么定义父模块 | `packaging=pom` + 共享配置 |
| 父 pom 放什么 | properties/dependencyManagement/pluginManagement |
| 父子关系 | 子模块怎么继承（见 [[子模块]]） |

---

### 二、父模块的两个核心特征

#### 特征一：`packaging=pom`

父模块本身**不产出 jar/war**，只是配置载体，所以必须：

```xml
<packaging>pom</packaging>
```

#### 特征二：放「共享配置」

父 pom 里放**所有子模块都要用**的配置，子模块自动继承。

---

### 三、一个典型的父 pom

```xml
<project>
    <modelVersion>4.0.0</modelVersion>

    <!-- ① 父模块自己的坐标 -->
    <groupId>com.example</groupId>
    <artifactId>my-app-parent</artifactId>
    <version>1.0.0</version>
    <packaging>pom</packaging>           <!-- ⭐ 父模块必须 pom -->

    <!-- ② 聚合（可选，详见 [[../pom配置/modules配置]]） -->
    <modules>
        <module>my-app-common</module>
        <module>my-app-service</module>
    </modules>

    <!-- ③ 共享属性（子模块继承） -->
    <properties>
        <maven.compiler.source>17</maven.compiler.source>
        <maven.compiler.target>17</maven.compiler.target>
        <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>

        <!-- 版本号集中管理 -->
        <spring.version>6.1.5</spring.version>
        <junit.version>4.13.2</junit.version>
    </properties>

    <!-- ④ 依赖版本管理（子模块继承版本标准） -->
    <dependencyManagement>
        <dependencies>
            <dependency>
                <groupId>org.springframework</groupId>
                <artifactId>spring-core</artifactId>
                <version>${spring.version}</version>
            </dependency>
            <dependency>
                <groupId>junit</groupId>
                <artifactId>junit</artifactId>
                <version>${junit.version}</version>
                <scope>test</scope>
            </dependency>
        </dependencies>
    </dependencyManagement>

    <!-- ⑤ 插件版本管理（子模块继承插件版本） -->
    <build>
        <pluginManagement>
            <plugins>
                <plugin>
                    <groupId>org.apache.maven.plugins</groupId>
                    <artifactId>maven-compiler-plugin</artifactId>
                    <version>3.13.0</version>
                    <configuration>
                        <source>17</source>
                        <target>17</target>
                    </configuration>
                </plugin>
            </plugins>
        </pluginManagement>
    </build>
</project>
```

### 四、父 pom 通常放什么（共享配置清单）

| 配置 | 放父 pom 的方式 | 子模块如何继承 |
| --- | --- | --- |
| **版本号属性** | `properties` | 直接用 `${变量}` |
| **依赖版本标准** | `dependencyManagement` | 子模块只写坐标，省略 version |
| **插件版本/配置** | `pluginManagement` | 子模块只写插件坐标，省略 version |
| **编译/JDK 配置** | `properties` 或 `pluginManagement` | 自动生效 |
| **仓库地址** | `repositories` | 自动继承 |
| **发布配置** | `distributionManagement` | 自动继承 |

> 💡 **提示：** 记住一个原则——**「子模块都要用、且配置相同」的放父 pom；「只有某个子模块用」的放子 pom**。

#### 什么不该放父 pom

- 各子模块**特有**的依赖（放各自 pom 的 `dependencies`）
- 各子模块**特有**的插件（放各自 pom 的 `plugins`）
- 只有一个模块用的属性

---

### 五、`dependencyManagement` 和 `pluginManagement` 在父 pom 的作用

这是父 pom 的**两大核心价值**，都是「**只管标准，不真正激活**」：

#### `dependencyManagement`（管依赖版本）

父 pom 声明版本标准 → 子模块 `dependencies` 里只写坐标、不写 version → 自动用父的标准。

详见 [[../pom配置/dependencyManagement配置]]。

#### `pluginManagement`（管插件版本）

父 pom 声明插件版本和默认配置 → 子模块 `plugins` 里只写插件坐标 → 自动继承版本和配置。

详见 [[../pom配置/plugins插件配置]]。

> ⭐ **这两个 Management 是父 pom 的灵魂**——让多模块项目的版本「一处定义，处处生效」。

### 六、父模块的坐标约定

多模块项目里，父模块和子模块的坐标通常有关系：

| 项 | 父模块 | 子模块 |
| --- | --- | --- |
| `groupId` | `com.example` | `com.example`（继承，可省） |
| `artifactId` | `my-app-parent`（或 `my-app`） | `my-app-common` / `my-app-service` |
| `version` | `1.0.0` | `1.0.0`（继承，可省） |

约定：

- 父 artifactId 常加 `-parent` 后缀，或直接用项目名
- 子模块 artifactId 用「`父名-模块名`」便于识别归属
- 所有模块 version 统一（便于一起升级）

### 七、父模块的两种典型场景

#### 场景一：父模块也是聚合模块（最常见）

父 pom **同时**配 `packaging=pom`、`modules`（聚合）和共享配置（继承）：

```
my-app/                 ← 父 pom（既是聚合又是继承父）
├── pom.xml             ← packaging=pom + <modules> + 共享配置
├── my-app-common/
├── my-app-service/
└── my-app-web/
```

这是**最常见**的多模块结构。

#### 场景二：父模块只做继承，不聚合

父 pom 只提供共享配置，**不含 `modules`**，分散在各处的子模块各自继承它。典型例子是 **`spring-boot-starter-parent`**——它是个「**远程父 pom**」，无数 Spring Boot 项目继承它，但它并不聚合这些项目。

```xml
<!-- 任意 Spring Boot 项目的 pom.xml -->
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.2.0</version>
</parent>
```

> 💡 **提示：** `spring-boot-starter-parent` 是理解「继承父 pom」的最好例子——它内部用 `properties` + `dependencyManagement`（引 `spring-boot-dependencies` BOM）+ `pluginManagement`，统一了 Spring Boot 全家桶的版本和插件配置。

### 八、父模块是否必须配置

| 场景 | 是否用父模块 |
| --- | --- |
| 单模块项目 | ❌ 不用 |
| 多模块项目 | ⭐ 强烈建议（统一配置） |
| 用 Spring Boot | ✅ 继承 `spring-boot-starter-parent` |

### 九、常见问题与注意事项

#### 1. 父 pom 忘了 `packaging=pom`

报错或生成空 jar。父/聚合模块**必须** `packaging=pom`。

#### 2. 把所有依赖都堆在父 pom 的 `dependencies`（而非 `dependencyManagement`）

这会让**所有子模块都被强制引入这些依赖**，导致模块臃肿。**版本标准放 `dependencyManagement`，子模块按需在 `dependencies` 里引入**。

#### 3. 父 pom 改了配置，子模块没生效

- 子模块要先 `Reload Project`
- 确认子模块的 `parent` 配置正确（见 [[子模块]]）
- 父 pom 要先 `mvn install` 到本地仓库（如果是远程继承）

#### 4. 父 pom 配置和子 pom 冲突

**子 pom 的配置优先级更高**（特殊覆盖一般）。但建议避免冲突，保持父 pom 为唯一标准。

#### 5. 多层继承

父 pom 可以再继承「爷爷 pom」（如自定义公司基础 pom 继承 spring-boot-starter-parent）。但**层级不宜太深**，否则排查问题困难。
