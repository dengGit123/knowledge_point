### 一、概述

> 📖 [Java Networking（Oracle 官方教程）](https://docs.oracle.com/javase/tutorial/networking/)

「通信」这个词在 Java 里，指的是**两台计算机（或两个进程）之间通过网络交换数据**。

大白话理解：你的程序想跟另一台机器上的程序说话，就得遵循一套双方都懂的「规则」（协议），通过一根「管道」（连接）把数据传过去。这整套机制，就是网络通信。

Java 在 `java.net` 包里提供了丰富的通信 API，从最底层的 Socket，到上层的 HTTP 客户端，覆盖了几乎所有通信场景。

本文是整个「通信」系列的入门篇，先把**通信的核心概念、网络分层模型、常见协议**讲清楚，后续每篇文档再深入具体技术。

---

### 二、核心概念

#### 1. IP 地址

IP 地址是网络上每台设备的「门牌号」，用来唯一标识一台主机。

- **IPv4**：如 `192.168.1.100`，4 个字节，约 43 亿个地址
- **IPv6**：如 `2001:0db8:85a3::8a2e:0370:7334`，为了解决 IPv4 地址不够用而设计
- **127.0.0.1**：本地回环地址，代表「本机自己」，也叫 `localhost`

> 💡 **提示：** 测试时常用 `127.0.0.1`，客户端和服务端都跑在同一台机器上，省去联网配置。

#### 2. 端口（Port）

一台机器上可能同时运行很多程序，光有 IP 地址不够，还得知道把数据交给哪个程序——这就是端口的作用。

- 端口范围：`0 ~ 65535`
- `0 ~ 1023`：系统保留端口（如 HTTP 用 80，HTTPS 用 443，MySQL 用 3306）
- `1024 ~ 49151`：注册端口
- `49152 ~ 65535`：动态/私有端口

**类比：** IP 地址是一栋大楼的地址，端口是这栋楼里的房间号。你要找的人在大楼的某个房间里。

#### 3. 协议（Protocol）

协议就是通信双方约定好的「对话规则」——数据按什么格式发、按什么顺序发、出错怎么处理。

最常见的两类传输层协议：

| 协议 | 全称 | 特点 | 类比 |
| --- | --- | --- | --- |
| **TCP** | 传输控制协议 | 可靠、有序、面向连接、三次握手 | 打电话，要先拨通才能聊，对方没听清会重说 |
| **UDP** | 用户数据报协议 | 不可靠、无连接、速度快、不保证顺序 | 寄明信片，发出去就不管了，可能丢、可能晚到 |

#### 4. Socket（套接字）

Socket 是通信的**端点**，是应用层和传输层之间的抽象接口。

- 一条 TCP 连接由「源 IP + 源端口 + 目标 IP + 目标端口」四元组唯一确定
- 程序通过 Socket 收发数据，就像通过一个「插座」接通网络

> 💡 **提示：** Socket 不是一个协议，而是一套编程接口（API），底层可以跑 TCP，也可以跑 UDP。

### 三、网络分层模型

理解网络分层，是理解「为什么 Java 通信 API 长这样」的关键。

#### 1. OSI 七层模型（理论模型）

```
┌──────────────┐
│  应用层       │  ← HTTP、FTP、SMTP、DNS（你的程序在这一层）
├──────────────┤
│  表示层       │  ← 数据加密、压缩、序列化
├──────────────┤
│  会话层       │  ← 建立/管理/断开会话
├──────────────┤
│  传输层       │  ← TCP、UDP（Java 的 Socket API 工作在这里）
├──────────────┤
│  网络层       │  ← IP、路由（IP 地址在这层起作用）
├──────────────┤
│  数据链路层    │  ← MAC 地址、帧
├──────────────┤
│  物理层       │  ← 网线、光纤、电信号
└──────────────┘
```

#### 2. TCP/IP 四层模型（实际使用的模型）

实际开发中更常用的是 TCP/IP 四层模型，它是 OSI 的简化版：

| TCP/IP 层 | 对应 OSI 层 | 典型协议 | Java 相关 |
| --- | --- | --- | --- |
| **应用层** | 应用、表示、会话层 | HTTP、HTTPS、DNS、FTP、SMTP | `HttpClient`、`URL` |
| **传输层** | 传输层 | TCP、UDP | `Socket`、`ServerSocket`、`DatagramSocket` |
| **网络层** | 网络层 | IP、ICMP | `InetAddress` |
| **网络接口层** | 数据链路、物理层 | 以太网、Wi-Fi | 操作系统/网卡负责，Java 基本不直接操作 |

#### 3. 数据封装与解封装

发送数据时，每经过一层，都会给数据**加上一层「头部」**（封装）；接收时则逐层**拆掉头部**（解封装）。这就像一封信被层层套上信封寄出，到了对方再层层拆开。

```
应用层：    [ 应用数据 ]
传输层：    [ TCP头 | 应用数据 ]
网络层：    [ IP头 | TCP头 | 应用数据 ]
链路层：    [ 帧头 | IP头 | TCP头 | 应用数据 | 帧尾 ]
```

### 四、Java 中的通信 API 一览

Java 把网络通信封装成了不同层次的 API，从低到高：

| 层次 | 类/接口 | 所在包 | 适用场景 |
| --- | --- | --- | --- |
| 地址解析 | `InetAddress` | `java.net` | 根据主机名查 IP |
| URL 读取 | `URL`、`URLConnection` | `java.net` | 简单读取网络资源 |
| TCP 通信 | `Socket`、`ServerSocket` | `java.net` | 自定义协议、长连接 |
| UDP 通信 | `DatagramSocket`、`DatagramPacket` | `java.net` | 广播、实时性要求高的场景 |
| HTTP 通信（老） | `HttpURLConnection` | `java.net` | 兼容老项目 |
| HTTP 通信（新） | `HttpClient`、`HttpRequest`、`HttpResponse` | `java.net.http` | Java 11+ 推荐，支持异步、HTTP/2 |
| WebSocket | `WebSocket`、`@ServerEndpoint` | `jakarta.websocket` | 服务端主动推送、实时通信 |
| RPC | `Remote`、`RMI` | `java.rmi` | 调用远程对象的方法 |

#### 一个最小示例：用 InetAddress 查询主机 IP

```java
import java.net.InetAddress;

public class InetAddressDemo {
    public static void main(String[] args) throws Exception {
        // 根据 主机名 查询 IP 地址
        InetAddress address = InetAddress.getByName("www.baidu.com");

        System.out.println("主机名: " + address.getHostName());   // www.baidu.com
        System.out.println("IP 地址: " + address.getHostAddress()); // 例如 14.215.177.38

        // 测试本机是否能连通（3 秒超时）
        boolean reachable = address.isReachable(3000);
        System.out.println("是否可达: " + reachable);
    }
}
```

> 💡 **提示：** `InetAddress` 是几乎所有 Java 网络通信的基石，Socket、URL 底层都会用到它。

### 五、TCP vs UDP：如何选择

这是面试高频题，也是实际开发的核心选型决策。

| 对比项 | TCP | UDP |
| --- | --- | --- |
| 连接方式 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认、重传、排序） | 不可靠（尽力而为） |
| 传输速度 | 相对慢 | 快 |
| 数据顺序 | 保证顺序 | 不保证顺序 |
| 头部开销 | 大（20 字节） | 小（8 字节） |
| 通信方式 | 一对一 | 一对一、一对多（广播/多播） |
| Java 核心 API | `Socket` / `ServerSocket` | `DatagramSocket` |
| 典型应用 | HTTP、文件传输、邮件 | DNS、视频直播、游戏、语音 |

**选择建议：**
- 数据**不能丢**、**必须有序**（如转账、文件下载、网页）→ 选 **TCP**
- 数据**允许少量丢失**、**追求实时**（如直播、在线游戏、心跳检测）→ 选 **UDP**

### 六、通信的完整流程（以 TCP 为例）

理解一次完整的 TCP 通信分几步，是掌握后续 Socket 编程的关键：

```
       服务端                              客户端
   ┌───────────┐                      ┌───────────┐
   │ 1. 创建    │                      │ 1. 创建    │
   │ ServerSocket│                     │ Socket     │
   │ 绑定端口    │                      │ 指定IP+端口 │
   └─────┬─────┘                      └─────┬─────┘
         │                                  │
   ┌─────▼─────┐                            │
   │ 2. accept()│ ◄──── 三次握手建立连接 ────┤
   │ 阻塞等待   │                            │
   └─────┬─────┘                      ┌─────▼─────┐
         │                            │ 3. 获取    │
   ┌─────▼─────┐                      │ 输出流     │
   │ 3. 获取    │ ◄──── 发送数据 ──────┤ 发送数据   │
   │ 输入流     │                      └─────┬─────┘
   │ 接收数据   │ ──── 返回数据 ─────► ┌─────▼─────┐
   └─────┬─────┘                      │ 获取输入流 │
         │                            │ 接收数据   │
   ┌─────▼─────┐                      └─────┬─────┘
   │ 4. close()│ ◄──── 四次挥手断开 ────┤ 4. close()│
   └───────────┘                      └───────────┘
```

**关键点：**
1. 服务端要先 `bind`（绑定端口）并 `accept`（等待连接），这是「被动方」
2. 客户端主动 `connect`（连接），这是「主动方」
3. 连接建立后，双方通过**输入流/输出流**读写数据
4. 通信结束，双方都要 `close` 释放资源

### 七、常见问题与注意事项

#### 1. 端口被占用（`BindException`）

```java
// ❌ 端口已被其他程序占用，会抛出 BindException
ServerSocket server = new ServerSocket(8080);
```

> ⚠️ **注意：** 端口被占用通常是因为上一次程序没正常退出，或被其他进程占用。可用 `lsof -i:8080`（Mac/Linux）或 `netstat -ano | findstr 8080`（Windows）排查。

#### 2. 防火墙 / 网络不通

- `ConnectException: Connection refused`：目标机器没有程序监听该端口，或被防火墙拦截
- 测试时建议先关掉本机防火墙，或换用 `127.0.0.1`

#### 3. 中文乱码

网络传输的是字节，读写文本时必须**指定字符编码**，否则不同系统的默认编码会导致乱码：

```java
// ✅ 显式指定 UTF-8 编码
BufferedReader reader = new BufferedReader(
    new InputStreamReader(inputStream, "UTF-8")
);
```

#### 4. 资源一定要关闭

Socket、流都是系统资源，用完必须关闭，否则会导致**句柄泄漏**（`Too many open files`）：

```java
// ✅ 使用 try-with-resources 自动关闭
try (Socket socket = new Socket("127.0.0.1", 8080)) {
    // 通信逻辑
}
```
