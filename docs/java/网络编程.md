## 一、概述

### 1. 什么是网络编程

**网络编程**是指编写程序，使运行在不同计算机上的进程能够相互进行数据通信。

通俗地说：一台电脑上的程序，通过网络把数据发给另一台电脑上的程序，并接收对方返回的数据——这套机制就是网络编程。

```
  计算机 A                          计算机 B
┌────────────┐      网络        ┌────────────┐
│ 进程 P1    │ ◄══════════════► │ 进程 P2    │
│ (客户端)   │   数据传输        │ (服务端)   │
└────────────┘                  └────────────┘
```

### 2. 网络通信三要素

任何一次网络通信，都离不开以下三个核心要素：

| 要素       | 作用                       | 类比                 |
| ---------- | -------------------------- | -------------------- |
| **IP 地址** | 在网络中定位一台主机       | 小区地址（某某路 1 号）|
| **端口号**  | 在主机中定位一个进程       | 门牌号（1 号楼 101）  |
| **协议**    | 规定数据如何组织、如何传输 | 交流语言（中文/英文） |

> 💡 **提示：** 一台电脑上可以同时运行很多联网程序（浏览器、QQ、微信），IP 只能定位到电脑，必须靠**端口号**才能精确找到是哪个程序在收发数据。端口号范围是 `0 ~ 65535`，其中 `0 ~ 1023` 为系统保留端口（如 HTTP 用 80），`1024 ~ 49151` 为注册端口，`49152 ~ 65535` 为动态/私有端口。

### 3. 为什么要学网络编程

- **理解互联网本质**：浏览器、聊天软件、游戏都建立在网络通信之上。
- **企业开发必备**：分布式系统、微服务、RPC 调用底层全是网络编程。
- **面试高频考点**：TCP/UDP、三次握手、HTTP 状态码是后端面试"必考题"。
- **进阶基石**：掌握 BIO（阻塞 IO）后才能更好理解 NIO、Netty、Reactor 模型。

## 二、网络分层模型

为了让纷繁复杂的网络通信有章可循，国际标准化组织提出了分层模型：每层各司其职，下层为上层提供服务。

### 1. OSI 七层模型 vs TCP/IP 四层模型

| OSI 七层模型 | TCP/IP 四层模型 | 对应协议示例                 | 作用简述                 |
| ------------ | ---------------- | ---------------------------- | ------------------------ |
| 应用层       | 应用层           | HTTP、FTP、DNS、SMTP         | 为应用程序提供网络服务   |
| 表示层       | ↑（合并到应用层）| JPEG、ASCII、SSL/TLS         | 数据格式转换、加密压缩   |
| 会话层       | ↑（合并到应用层）| RPC、Socket API              | 建立、管理、终止会话     |
| 传输层       | 传输层           | **TCP、UDP**                 | 端到端的可靠/不可靠传输  |
| 网络层       | 网络层           | **IP、ICMP、ARP**            | 路由选择、寻址           |
| 数据链路层   | 网络接口层       | 以太网、PPP                  | 相邻节点间的帧传输       |
| 物理层       | ↑（合并到接口层）| 光纤、网线、电信号           | 比特流的透明传输         |

> 💡 **提示：** 实际开发中，**TCP/IP 四层模型**用得最多。Java 网络编程主要工作在**传输层**（用 Socket 编程，操作 TCP/UDP）和**应用层**（封装 HTTP 协议）。

### 2. 数据封装与解封装

发送数据时，每一层都会在数据前面加上自己的"头部"（封装）；接收数据时，每一层又逐层剥掉头部（解封装）。

```
应用层数据
   │  + TCP 头
   ▼
TCP 段 (Segment)
   │  + IP 头
   ▼
IP 数据报 (Packet)
   │  + 帧头/帧尾
   ▼
以太网帧 (Frame) ──► 物理线路传输
```

## 三、TCP 协议（重点）

### 1. TCP 是什么

**TCP（Transmission Control Protocol，传输控制协议）** 是一种**面向连接、可靠、基于字节流**的传输层通信协议。

可以把它想象成**打电话**：必须先拨通建立连接，确认对方在线，然后才能开始讲话，讲完还要挂断。

### 2. TCP 的四大特性

| 特性         | 含义                                                                 |
| ------------ | -------------------------------------------------------------------- |
| **面向连接** | 通信前必须先建立连接（三次握手），通信结束要断开连接（四次挥手）       |
| **可靠传输** | 通过确认应答、超时重传、校验和等机制保证数据不丢失、不重复、按序到达   |
| **字节流**   | 数据以无边界的字节流形式传输，没有"消息"的概念（这正是粘包问题的根源） |
| **全双工**   | 双方都可以同时发送和接收数据                                         |

### 3. 三次握手（建立连接）

TCP 建立连接需要三次交互，称为**三次握手（Three-Way Handshake）**。

```
   客户端                                服务端
     │                                     │
     │ ──① SYN, seq=x ──────────────────► │  客户端：我想连接 (SYN_SENT)
     │                                     │  服务端：收到，我也确认 (SYN_RCVD)
     │ ◄──② SYN+ACK, seq=y, ack=x+1 ────── │
     │                                     │
     │ ──③ ACK, ack=y+1 ─────────────────► │  双方进入 ESTABLISHED，连接建立
     │                                     │
     │◄═══════ 可以互发数据了 ═══════════►│
```

**每一步的作用：**

1. **第一次握手**：客户端发送 SYN 报文（seq=x），进入 `SYN_SENT` 状态，表示请求建立连接。
2. **第二次握手**：服务端收到后，回复 SYN+ACK 报文（seq=y, ack=x+1），进入 `SYN_RCVD` 状态，表示同意连接并确认。
3. **第三次握手**：客户端收到后，再发一个 ACK 报文（ack=y+1），双方都进入 `ESTABLISHED` 状态，连接正式建立。

### 4. 四次挥手（断开连接）

TCP 是全双工的，断开连接时双方都要各自关闭，所以需要四次交互。

```
   客户端                                服务端
     │                                     │  (双方都在 ESTABLISHED)
     │ ──① FIN, seq=u ──────────────────► │  客户端：我没数据了 (FIN_WAIT_1)
     │                                     │  服务端：收到你的关闭请求 (CLOSE_WAIT)
     │ ◄──② ACK, ack=u+1 ───────────────── │
     │   (客户端进入 FIN_WAIT_2)           │
     │                                     │
     │     ...... 服务端继续发送剩余数据 ......│
     │                                     │
     │ ◄──③ FIN, seq=w ────────────────── │  服务端：我也没数据了 (LAST_ACK)
     │                                     │
     │ ──④ ACK, ack=w+1 ─────────────────► │  客户端：再见 (TIME_WAIT → CLOSED)
     │   (服务端立即 CLOSED)               │
     │   (客户端等待 2*MSL 后 CLOSED)      │
```

**为什么需要 TIME_WAIT？**

> 💡 **提示：** 客户端在第四次挥手后不会立即关闭，而是进入 `TIME_WAIT` 状态，等待 `2 * MSL`（最大报文段生存时间）后才真正关闭。原因有二：
> 1. **保证最后的 ACK 能到达**：如果服务端没收到最后的 ACK，会重发 FIN，客户端在 TIME_WAIT 期间还能重发 ACK。
> 2. **让旧连接的报文失效**：防止本次连接的残留报文干扰下一个复用相同端口的新连接。

### 5. 为什么握手三次而挥手四次？（高频面试题）

| 阶段     | 次数 | 原因                                                                 |
| -------- | ---- | -------------------------------------------------------------------- |
| 三次握手 | 3 次 | 建立连接时服务端的 SYN 和 ACK 可以合并到一次发送（因为没有数据传输） |
| 四次挥手 | 4 次 | 断开时，服务端收到 FIN 后可能还有数据没发完，所以 ACK 和 FIN 必须分开 |

> 💡 **通俗解释：** 握手时服务端"爽快"，直接一句"行，我同意"（SYN+ACK 合并）；挥手时服务端"依依不舍"，先回一句"我知道你要走了"，等把手头的话说完，再说"我也走了吧"（ACK 和 FIN 分开发）。

### 6. 流量控制与拥塞控制（简要）

- **流量控制**：接收方通过**滑动窗口**告诉发送方"我还能收多少"，防止发送方发太快把接收方淹没。
- **拥塞控制**：发送方根据网络拥塞情况动态调整发送速率，包含**慢启动、拥塞避免、快重传、快恢复**四个算法。

## 四、UDP 协议

### 1. UDP 是什么

**UDP（User Datagram Protocol，用户数据报协议）** 是一种**无连接、不可靠、面向数据报**的传输层协议。

可以把它想象成**寄明信片**：写好地址直接扔进邮筒，不保证对方一定收到，也不保证顺序，但胜在简单快捷。

### 2. UDP 的特性

| 特性         | 含义                                           |
| ------------ | ---------------------------------------------- |
| **无连接**   | 发送数据前不需要建立连接，拿来就发             |
| **不可靠**   | 不保证送达、不保证顺序、不保证不重复           |
| **面向数据报**| 以完整的数据报为单位收发，有明确的消息边界     |
| **开销小**   | 头部只有 8 字节（TCP 头部 20 字节），效率高     |
| **支持广播** | 可以一对多广播、多播                           |

### 3. UDP 的适用场景

- **音视频直播/通话**：偶尔丢几帧画面无伤大雅，但对实时性要求极高。
- **DNS 查询**：请求小、响应快，建立 TCP 连接开销不划算。
- **在线游戏**：玩家的位置信息需要实时同步，丢一两个包不如发新的。
- **物联网上报**：海量设备、小数据包，UDP 更省资源。

## 五、TCP vs UDP 对比（重点）

| 对比项       | TCP                          | UDP                        |
| ------------ | ---------------------------- | -------------------------- |
| 连接性       | 面向连接（三次握手）         | 无连接                     |
| 可靠性       | 可靠（确认、重传、排序）     | 不可靠（尽力而为）         |
| 传输形式     | 字节流（无边界）             | 数据报（有边界）           |
| 速度         | 较慢                         | 快                         |
| 头部开销     | 20 字节起                    | 8 字节                     |
| 拥塞/流量控制| 有                           | 无                         |
| 通信方式     | 一对一                       | 一对一、一对多、多对多     |
| 适用场景     | 文件传输、网页、邮件         | 直播、语音、DNS、游戏      |
| Java 核心 API| `Socket` / `ServerSocket`    | `DatagramSocket` / `DatagramPacket` |

> 💡 **选择口诀：** **要可靠选 TCP，要速度选 UDP。** 老板让你做文件下载/转账——TCP；做直播/语音——UDP。

## 六、Java 网络编程 API（重点）

### 1. InetAddress：表示 IP 地址

`InetAddress` 用于封装 IP 地址和主机名。

```java
import java.net.InetAddress;

public class InetAddressDemo {
    public static void main(String[] args) throws Exception {
        // ✅ 获取本机的 InetAddress
        InetAddress local = InetAddress.getLocalHost();
        System.out.println("本机名: " + local.getHostName());       // dengMac
        System.out.println("本机 IP: " + local.getHostAddress());   // 192.168.1.100

        // ✅ 根据主机名获取远程地址
        InetAddress baidu = InetAddress.getByName("www.baidu.com");
        System.out.println("百度 IP: " + baidu.getHostAddress());

        // ✅ 判断是否可达（类似 ping，3 秒超时）
        boolean reachable = baidu.isReachable(3000);
        System.out.println("百度是否可达: " + reachable);
    }
}
```

### 2. 基于 TCP 的 Socket 编程

Java 中 TCP 通信使用两个核心类：

| 类              | 角色   | 说明                                   |
| --------------- | ------ | -------------------------------------- |
| `ServerSocket`  | 服务端 | 监听端口，`accept()` 阻塞等待客户端连接 |
| `Socket`        | 客户端 | 主动连接服务器，获取输入/输出流          |

**通信流程图：**

```
        客户端 (Socket)                         服务端 (ServerSocket)
        ────────────────                        ──────────────────────
  ┌─ 1. new Socket(ip, port) ─────┐    ┌─ 1. new ServerSocket(port) ─┐
  │   主动发起连接                │    │   绑定端口开始监听          │
  └───────────────────────────────┘    └─────────────────────────────┘
                                              │
                                              ▼ 2. accept() 阻塞等待
        ◄══════════ TCP 三次握手 ════════════►   返回一个 Socket
        │                                          │
        ▼ 3. getOutputStream()                     ▼ 3. getInputStream()
        ──── 发送数据 ──────────────────────────► 收到数据
        ◄──── 接收数据 ◄────────────────────────── 发送数据
        │                                          │
        ▼ 4. close()                               ▼ 4. close()
                  四次挥手，连接关闭
```

#### （1）服务端：TcpServer

```java
import java.io.*;
import java.net.ServerSocket;
import java.net.Socket;

public class TcpServer {
    public static void main(String[] args) {
        // ✅ try-with-resources 自动关闭资源，避免泄漏
        try (ServerSocket serverSocket = new ServerSocket(8888)) {
            System.out.println("服务端已启动，监听 8888 端口...");

            // accept() 会阻塞，直到有客户端连接
            Socket clientSocket = serverSocket.accept();
            System.out.println("客户端已连接: " + clientSocket.getInetAddress());

            // 获取输入输出流
            BufferedReader in = new BufferedReader(
                    new InputStreamReader(clientSocket.getInputStream()));
            PrintWriter out = new PrintWriter(clientSocket.getOutputStream(), true);

            // 读取客户端消息
            String msg = in.readLine();
            System.out.println("收到客户端: " + msg);

            // 回复客户端
            out.println("你好客户端，我已收到你的消息: " + msg);

            clientSocket.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

#### （2）客户端：TcpClient

```java
import java.io.*;
import java.net.Socket;

public class TcpClient {
    public static void main(String[] args) {
        // ✅ 连接本机 8888 端口
        try (Socket socket = new Socket("127.0.0.1", 8888)) {
            BufferedReader in = new BufferedReader(
                    new InputStreamReader(socket.getInputStream()));
            PrintWriter out = new PrintWriter(socket.getOutputStream(), true);

            // 发送消息给服务端
            out.println("Hello Server!");

            // 读取服务端回复
            String reply = in.readLine();
            System.out.println("收到服务端: " + reply);
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

> ⚠️ **注意：** 先启动 `TcpServer`，再启动 `TcpClient`。否则客户端会抛出 `ConnectException: Connection refused`。

#### （3）多线程服务端（支持并发）

上面的服务端只能处理一个客户端。实际项目中需要为每个客户端分配一个线程：

```java
import java.io.*;
import java.net.ServerSocket;
import java.net.Socket;

public class MultiThreadServer {
    public static void main(String[] args) throws IOException {
        try (ServerSocket serverSocket = new ServerSocket(8888)) {
            System.out.println("多线程服务端启动...");
            while (true) {
                Socket client = serverSocket.accept(); // 每来一个连接
                // ✅ 交给新线程处理，主线程继续等待下一个连接
                new Thread(new ClientHandler(client)).start();
            }
        }
    }
}

// 处理单个客户端的任务
class ClientHandler implements Runnable {
    private final Socket socket;

    ClientHandler(Socket socket) {
        this.socket = socket;
    }

    @Override
    public void run() {
        try (BufferedReader in = new BufferedReader(
                new InputStreamReader(socket.getInputStream()));
             PrintWriter out = new PrintWriter(socket.getOutputStream(), true)) {

            String line;
            // 持续读取，直到客户端断开（readLine 返回 null）
            while ((line = in.readLine()) != null) {
                System.out.println("来自 " + socket.getInetAddress() + ": " + line);
                out.println("echo: " + line); // 回显
            }
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

> ⚠️ **注意：** 这种"一个连接一个线程"的模型（**BIO 模型**）在连接数大时线程爆炸，性能急剧下降。这正是 NIO（多路复用）要解决的问题，详见后文「八、NIO 在网络编程中的应用」。

### 3. 基于 UDP 的 Socket 编程

UDP 通信使用 `DatagramSocket`（收发数据的"邮局"）和 `DatagramPacket`（装数据的"信封"）。

#### （1）接收端：UdpReceiver

```java
import java.net.DatagramPacket;
import java.net.DatagramSocket;

public class UdpReceiver {
    public static void main(String[] args) throws Exception {
        // ✅ 在 9999 端口接收数据报
        try (DatagramSocket socket = new DatagramSocket(9999)) {
            byte[] buf = new byte[1024];
            DatagramPacket packet = new DatagramPacket(buf, buf.length);

            System.out.println("接收端等待数据...");
            socket.receive(packet); // 阻塞等待

            String msg = new String(packet.getData(), 0, packet.getLength());
            System.out.println("收到: " + msg
                    + " 来自 " + packet.getAddress() + ":" + packet.getPort());
        }
    }
}
```

#### （2）发送端：UdpSender

```java
import java.net.DatagramPacket;
import java.net.DatagramSocket;
import java.net.InetAddress;

public class UdpSender {
    public static void main(String[] args) throws Exception {
        // ✅ 不指定端口，系统自动分配
        try (DatagramSocket socket = new DatagramSocket()) {
            String msg = "Hello UDP!";
            byte[] data = msg.getBytes();
            // 把数据封装成数据报，目标地址 127.0.0.1:9999
            DatagramPacket packet = new DatagramPacket(
                    data, data.length, InetAddress.getByName("127.0.0.1"), 9999);

            socket.send(packet); // 发送，无需建立连接
            System.out.println("已发送: " + msg);
        }
    }
}
```

## 七、HTTP 协议简介

### 1. HTTP 是什么

**HTTP（HyperText Transfer Protocol，超文本传输协议）** 是基于 TCP 的应用层协议，采用**请求-响应**模型，是万维网数据通信的基础。

> 💡 **提示：** HTTP/1.1 和 HTTP/2 基于 TCP，而 HTTP/3 改用了基于 UDP 的 QUIC 协议，但绝大多数场景仍是 TCP。

### 2. HTTP 报文结构

**请求结构：**

```
POST /api/login HTTP/1.1          ← 请求行：方法 路径 版本
Host: www.example.com             ┐
Content-Type: application/json    │ 请求头（多个键值对）
Content-Length: 36                ┘
                                  
{"username":"admin","pwd":"123"}  ← 请求体（可选）
```

**响应结构：**

```
HTTP/1.1 200 OK                   ← 状态行：版本 状态码 状态描述
Content-Type: application/json    ┐
Content-Length: 28                │ 响应头
                                 ┘
{"code":0,"msg":"success"}        ← 响应体
```

### 3. 常见 HTTP 状态码

| 状态码 | 含义                   | 典型场景                       |
| ------ | ---------------------- | ------------------------------ |
| 200    | OK 成功                | 请求正常处理完成               |
| 301    | Moved Permanently 永久重定向 | 网站换域名，老地址永久跳新地址 |
| 302    | Found 临时重定向       | 登录后跳转到首页               |
| 304    | Not Modified 资源未修改 | 协商缓存命中，用本地缓存       |
| 400    | Bad Request 请求错误   | 参数格式错误                   |
| 401    | Unauthorized 未认证    | 没登录就访问需要登录的接口     |
| 403    | Forbidden 拒绝访问     | 已登录但无权限                 |
| 404    | Not Found 找不到资源   | URL 不存在                     |
| 500    | Internal Server Error 服务器内部错误 | 代码抛异常                 |
| 502    | Bad Gateway 网关错误   | 反向代理拿不到后端响应         |
| 503    | Service Unavailable 服务不可用 | 服务过载或维护中        |

> 💡 **状态码分类口诀：** **1xx 信息、2xx 成功、3xx 重定向、4xx 客户端错、5xx 服务端错。**

## 八、NIO 在网络编程中的应用

前面 BIO（阻塞 IO）模型存在致命问题：**一个连接占用一个线程**。当并发连接数达到上万时，线程开销会让服务器不堪重负。

Java NIO 引入了**多路复用（Multiplexing）** 机制：一个线程可以同时管理成千上万个连接。

### 1. NIO 网络编程三大核心

| 组件       | 作用                                                   |
| ---------- | ------------------------------------------------------ |
| `Channel`  | 双向数据通道（如 `SocketChannel`），替代流             |
| `Buffer`   | 数据容器，所有读写都经过 Buffer                        |
| `Selector` | 多路复用器，一个线程监听多个 Channel 的事件（连接、读、写）|

### 2. 工作原理图

```
                   ┌─────────────┐
                   │  Selector   │  ← 一个线程
                   │ (多路复用器)│
                   └──────┬──────┘
        ┌──────────┬──────┴───────┬──────────┐
        ▼          ▼              ▼          ▼
   ┌─────────┐┌─────────┐   ┌─────────┐┌─────────┐
   │Channel 1││Channel 2│...│Channel N│  上万个连接
   │ 可读    ││ 可写    │   │ 连接就绪│  只在有事件时才处理
   └─────────┘└─────────┘   └─────────┘
```

### 3. 引出 Netty

直接用 NIO API 开发复杂且容易出错（半包处理、断线重连、线程模型等）。**Netty** 是基于 NIO 封装的高性能网络框架，提供了优雅的 API 和完善的线程模型（Reactor），是 Java 网络编程的事实标准。

> 💡 **延伸阅读：** 关于 Buffer、Channel、Selector 的详细用法，以及零拷贝、Reactor 模型等内容，请参考同目录下的 **[NIO.md](./NIO.md)**。

## 九、注意事项与陷阱

### 1. 资源必须关闭

`Socket`、`ServerSocket`、流用完必须关闭，否则会导致**文件描述符泄漏**，最终无法新建连接。

```java
// ❌ 错误：手动关闭，容易遗漏（异常时不会关闭）
Socket socket = new Socket("127.0.0.1", 8888);
// ... 使用 ...
socket.close(); // 如果上面抛异常，这行执行不到

// ✅ 正确：try-with-resources，自动关闭
try (Socket socket = new Socket("127.0.0.1", 8888)) {
    // ... 使用 ...
}
```

### 2. 端口占用异常

启动服务端时报 `java.net.BindException: Address already in use`，说明端口被占用（可能是上次程序没退干净）。

```java
// ✅ 处理端口占用：可设置 SO_REUSEADDR 允许端口复用
ServerSocket serverSocket = new ServerSocket();
serverSocket.setReuseAddress(true);
serverSocket.bind(new InetSocketAddress(8888));
```

### 3. TCP 粘包/拆包问题（重要）

TCP 是**字节流**协议，**没有消息边界**。发送方连续发两条 `"Hello"` 和 `"World"`，接收方可能一次收到 `"HelloWorld"`（粘包），也可能分三次收到 `"Hel"`、`"loWo"`、`"rld"`（拆包）。

```
发送方连续发送:    [Hello] [World]
                ↓ TCP 字节流，无边界 ↓
接收方可能收到:    [HelloWorld]      ← 粘包（多条粘在一起）
接收方可能收到:    [Hel][loWo][rld]  ← 拆包（一条被拆开）
```

**常见的解决方案：**

| 方案         | 做法                                       | 优缺点                       |
| ------------ | ------------------------------------------ | ---------------------------- |
| 消息定长     | 每条消息固定长度，不足补空格               | 简单但浪费带宽               |
| 分隔符       | 用特殊字符（如 `\n`）作为消息结束标志      | 简单，但消息体不能含分隔符   |
| 长度字段     | 消息头中用固定字节表示消息体长度（最常用） | 灵活高效，Netty/自定义协议首选 |

> ⚠️ **注意：** UDP 是数据报协议，**有消息边界，不会粘包**。所以一次 `send` 对应一次 `receive`，收到的就是一个完整的数据报。

### 4. 阻塞 IO 的线程模型问题

BIO 中 `accept()`、`read()` 都是阻塞的，导致必须为每个连接分配一个线程。连接数一多，线程切换开销巨大。这是催生 NIO、Netty 的根本原因。

```java
// ❌ 阻塞：没有连接时一直卡在这里
Socket client = serverSocket.accept();
String line = in.readLine(); // 没数据时一直卡住
```

## 十、面试常见问题

### Q1：TCP 和 UDP 的区别是什么？

**答：** TCP 面向连接、可靠、字节流、一对一，适合文件传输和网页；UDP 无连接、不可靠、数据报、支持广播，适合直播、DNS、游戏。详见第五节对比表。

### Q2：详细描述三次握手过程。

**答：** ① 客户端发 SYN（seq=x），进入 SYN_SENT；② 服务端回 SYN+ACK（seq=y, ack=x+1），进入 SYN_RCVD；③ 客户端再发 ACK（ack=y+1），双方进入 ESTABLISHED，连接建立。

### Q3：为什么是三次握手，而不是两次或四次？

**答：** 三次是**最少**次数，能保证双方都确认了对方的收发能力：两次无法防止历史失效的连接请求被服务端误以为是新连接（导致资源浪费）；四次没必要，因为建立连接时服务端的 SYN 和 ACK 可以合并发送。

### Q4：描述四次挥手过程，为什么需要 TIME_WAIT？

**答：** 主动关闭方发 FIN，被动方回 ACK（此时被动方进入 CLOSE_WAIT，可能继续发数据），被动方数据发完后也发 FIN，主动方回 ACK 并进入 TIME_WAIT。TIME_WAIT 等待 2*MSL 的目的：① 确保最后的 ACK 能到达，若对方重发 FIN 还能再补一次；② 让本次连接的报文在网络中失效，避免干扰复用相同端口的新连接。

### Q5：为什么握手三次而挥手四次？

**答：** 建立连接时服务端收到 SYN 后可以直接把 ACK 和自己的 SYN 合并发送；但断开连接时，被动方收到 FIN 后可能还有数据没发完，所以先回 ACK，等数据发完再单独发 FIN，因此需要四次。

### Q6：什么是 TCP 粘包/拆包？如何解决？

**答：** TCP 是字节流协议，没有消息边界，多条消息可能被合并接收（粘包）或一条消息被拆成多次（拆包）。常用解决方案：消息定长、分隔符、在消息头中用长度字段标识消息体长度（业界主流，Netty 提供 `LengthFieldBasedFrameDecoder`）。

### Q7：简述 Socket 通信的完整流程。

**答：** 服务端创建 `ServerSocket` 绑定端口并 `accept()` 阻塞等待；客户端创建 `Socket` 连接服务端（触发三次握手）；连接建立后双方通过输入输出流互发数据；通信结束各自 `close()`（触发四次挥手）。

### Q8：HTTP 常见状态码有哪些？301 和 302 的区别？

**答：** 200 成功，404 找不到，500 服务器内部错误等。**301 永久重定向**（搜索引擎会缓存新地址），**302 临时重定向**（每次仍访问老地址再跳）。304 表示协商缓存命中，直接用本地缓存。

### Q9：一个 ServerSocket 监听 8080，能建立多少个 TCP 连接？

**答：** 理论上 TCP 连接由四元组（源 IP、源端口、目的 IP、目的端口）唯一确定。客户端 IP+端口组合近乎无限，所以单机可建立的连接数非常大（百万级），实际受限于内存（每个连接占用 buffer）、文件描述符数量（`ulimit -n`）和线程数。这也是高并发服务器要用 NIO 的原因。

### Q10：BIO、NIO、AIO 的区别？

**答：** **BIO**（阻塞 IO）一个连接一个线程，连接不做事也占线程；**NIO**（非阻塞 IO）一个线程用 Selector 管理多个连接，只有事件发生时才处理；**AIO**（异步 IO）由操作系统完成后回调通知，Java 中 AIO 在 Linux 下实际仍是 epoll 模拟，业界主流用 NIO + Netty。

## 十一、总结

网络编程是 Java 后端开发的基石。本文核心要点回顾：

1. **三要素**：IP（找主机）、端口（找进程）、协议（定规则）。
2. **TCP**：面向连接、可靠、字节流；建立靠**三次握手**，断开靠**四次挥手**。
3. **UDP**：无连接、不可靠、数据报、高效，适合实时性要求高的场景。
4. **Java API**：TCP 用 `Socket`/`ServerSocket`，UDP 用 `DatagramSocket`/`DatagramPacket`。
5. **HTTP**：基于 TCP 的请求-响应协议，掌握常见状态码含义。
6. **陷阱**：资源关闭、端口占用、**TCP 粘包拆包**、BIO 线程模型瓶颈。
7. **进阶**：NIO 多路复用解决高并发问题，进而引出 Netty 框架。
