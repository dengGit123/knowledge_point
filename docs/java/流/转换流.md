### 一、概述

> 📖 [InputStreamReader（Java）](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/io/InputStreamReader.html) ｜ [OutputStreamWriter（Java）](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/io/OutputStreamWriter.html)

转换流是**连接字节流和字符流的桥梁**：它把「字节流」包装成「字符流」，并在转换过程中**指定字符编码**。

大白话：字节流只认字节（不懂数据是什么文字）；转换流像一个「**翻译官**」，把字节按指定编码（如 UTF-8）翻译成字符，让程序能正确处理文本，**彻底解决乱码问题**。

两个转换流：

| 转换流 | 方向 | 作用 |
| --- | --- | --- |
| `InputStreamReader` | 字节输入流 → 字符输入流 | 把字节解码成字符（读） |
| `OutputStreamWriter` | 字符输出流 → 字节输出流 | 把字符编码成字节（写） |

> 💡 **提示：** 转换流的核心价值不是「转换」本身，而是**能在转换时指定编码**。这是处理中文、跨平台文本乱码的关键武器。详见 [[字符编码]]。

### 二、为什么需要转换流

#### 1. 字节流读文本的乱码问题

`FileInputStream` 是字节流，读出来的 `byte` 要转成 `char` 才能显示。但**不同编码下，一个字符占的字节数不同**：

```
UTF-8：一个汉字占 3 字节
GBK：  一个汉字占 2 字节
```

如果用字节流一个一个字节读再强转 char，中文必然乱码：

```java
// ❌ 字节流直接读中文，乱码
FileInputStream fis = new FileInputStream("中文.txt");
int b;
while ((b = fis.read()) != -1) {
    System.out.print((char) b);   // 乱码！
}
```

#### 2. FileReader 的编码缺陷（JDK 11 前）

`FileReader` 是字符流，但它**用的是系统默认编码**，不能指定：

```java
// JDK 11 前：FileReader 无法指定编码，跨平台可能乱码
new FileReader("a.txt");   // 用系统默认编码（Windows 可能是 GBK）
```

转换流 `InputStreamReader` 解决了这个问题——**可以显式指定编码**：

```java
// ✅ 转换流指定 UTF-8 编码，杜绝乱码
new InputStreamReader(new FileInputStream("a.txt"), "UTF-8");
```

> 💡 **提示：** JDK 11 起，`FileReader` 也支持指定编码了（`new FileReader(file, charset)`），但转换流仍是更通用、更底层的方案。

### 三、基本用法

#### 1. InputStreamReader（读文本，指定编码）

```java
import java.io.*;

public class InputReaderDemo {
    public static void main(String[] args) throws IOException {
        // ✅ 字节流 → 字符流，指定 UTF-8
        try (InputStreamReader isr = new InputStreamReader(
                new FileInputStream("a.txt"), "UTF-8")) {

            int ch;
            while ((ch = isr.read()) != -1) {
                System.out.print((char) ch);   // 正确显示中文
            }
        }
    }
}
```

#### 2. 配合缓冲流（实战推荐组合）

```java
// ✅ 黄金组合：字节流 + 转换流(指定编码) + 缓冲流(按行读)
try (BufferedReader reader = new BufferedReader(
        new InputStreamReader(
            new FileInputStream("a.txt"), StandardCharsets.UTF_8))) {

    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
}
```

#### 3. OutputStreamWriter（写文本，指定编码）

```java
import java.io.*;
import java.nio.charset.StandardCharsets;

public class OutputWriterDemo {
    public static void main(String[] args) throws IOException {
        try (OutputStreamWriter osw = new OutputStreamWriter(
                new FileOutputStream("b.txt"), StandardCharsets.UTF_8)) {

            osw.write("你好，世界！");   // 按 UTF-8 编码写入
        }
    }
}
```

### 四、InputStreamReader 的工作原理

转换流读取时，**不是「一个字节一个字节」解码，而是「按编码规则读够字节再解码成一个字符」**：

```
UTF-8 读「你」：
  读 3 个字节 (0xE4 0xBD 0xA0) → 解码成「你」一个字符 → 返回
```

```
┌─────────────┐   读字节      ┌──────────────┐   解码    ┌──────┐
│ FileInputStream│ ────────► │ InputStreamReader│ ──────► │ char │
│  (字节流)     │            │ (转换+解码器)    │         │      │
└─────────────┘            └──────────────┘         └──────┘
```

> 💡 **提示：** 这就是为什么转换流能正确读多字节字符——它**按编码规则**读取，而不是按字节。

### 五、指定编码的几种方式

```java
import java.nio.charset.Charset;
import java.nio.charset.StandardCharsets;

// 方式1：字符串编码名
new InputStreamReader(in, "UTF-8");
new InputStreamReader(in, "GBK");

// 方式2：StandardCharsets 常量（推荐，避免拼写错误，编译期检查）
new InputStreamReader(in, StandardCharsets.UTF_8);
new InputStreamReader(in, StandardCharsets.ISO_8859_1);

// 方式3：Charset 对象
new InputStreamReader(in, Charset.forName("GB18030"));

// 查看当前 JVM 默认编码
System.out.println(Charset.defaultCharset());
```

> 💡 **提示：** **优先用 `StandardCharsets.UTF_8`**，而非字符串 `"UTF-8"`——前者编译期检查、无拼写错误风险、性能略好。

#### 常见编码

| 编码 | 说明 | 中文字节 |
| --- | --- | --- |
| `UTF-8` | 国际通用，**首选** | 3 字节 |
| `GBK` | 中文 Windows 默认 | 2 字节 |
| `ISO-8859-1` | 西欧语言，不支持中文 | 1 字节 |
| `US-ASCII` | 纯英文 | 1 字节 |

详见 [[字符编码]]。

### 六、InputStreamReader 是 Reader 的子类

转换流**继承了字符流的能力**，可以无缝替换字符流使用：

```
Reader（字符输入根）
  └── InputStreamReader（转换流，桥接字节）
        └── FileReader（子类，固定读文件 + 默认编码）
```

`FileReader` 其实就是 `InputStreamReader` 的简化版（绑死了 `FileInputStream` + 默认编码）。所以：

- `FileReader` = `InputStreamReader` + `FileInputStream` + 默认编码
- 要**指定编码**，必须用 `InputStreamReader`

### 七、实际应用场景

| 场景 | 用法 |
| --- | --- |
| 读取任意编码的文本文件 | `InputStreamReader` + 指定编码 |
| 网络流读文本（HTTP 响应） | `InputStreamReader(socket.getInputStream(), "UTF-8")` |
| 处理中文 GBK 文件 | `InputStreamReader(in, "GBK")` |
| 跨平台写文件保证编码一致 | `OutputStreamWriter(out, UTF_8)` |
| HTTP 请求/响应处理 | 配合 HttpClient 的响应流 |

```java
// 网络读取文本，指定编码避免乱码（参考 [[Socket通信]]）
BufferedReader reader = new BufferedReader(
    new InputStreamReader(socket.getInputStream(), StandardCharsets.UTF_8));
```

### 八、常见问题与注意事项

#### 1. 编码不一致 → 乱码

**乱码的根本原因只有一个：编码和解码用了不同的编码表。**

```
写入：UTF-8 编码（「你」= 3 字节）
读取：用 GBK 解码（按 2 字节切）→ 乱码
```

解决：读写**必须用同一个编码**，且优先 UTF-8。

#### 2. 系统默认编码坑

不同操作系统默认编码不同：
- Windows（中文版）：GBK
- Mac/Linux：UTF-8

依赖默认编码的代码（不指定编码的 `FileReader`），换台机器就乱码。**生产代码必须显式指定编码**。

#### 3. 编码名写错抛异常

```java
// ❌ 不支持的编码名，抛 UnsupportedEncodingException
new InputStreamReader(in, "UTF-88");

// ✅ 用 StandardCharsets 常量，杜绝拼写错误
new InputStreamReader(in, StandardCharsets.UTF_8);
```

#### 4. 转换流能「转换」但效率

转换流每次解码有一定开销，所以**通常再套一层缓冲流**：

```java
// ✅ 转换流 + 缓冲流，兼顾编码与性能
new BufferedReader(new InputStreamReader(in, UTF_8));
```

### 九、总结

| 要点 | 说明 |
| --- | --- |
| 作用 | 字节流 ↔ 字符流的桥梁，转换时指定编码 |
| 核心价值 | **解决乱码**（显式指定编码） |
| 输入转换 | `InputStreamReader`（字节→字符，读） |
| 输出转换 | `OutputStreamWriter`（字符→字节，写） |
| 黄金组合 | `BufferedReader(InputStreamReader(字节流, 编码))` |
| 编码选择 | 优先 `StandardCharsets.UTF_8` |

一句话：**只要涉及文本编码，就用转换流显式指定编码**，把乱码扼杀在摇篮里。
