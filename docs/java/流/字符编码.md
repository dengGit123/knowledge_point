### 一、概述

> 📖 [字符编码（维基百科）](https://zh.wikipedia.org/wiki/%E5%AD%97%E7%AC%A6%E7%BC%96%E7%A0%81) ｜ [Java Charset（Oracle）](https://docs.oracle.com/en/java/javase/17/docs/api/java.base/java/nio/charset/Charset.html)

字符编码是**把「字符」和「字节」相互转换的规则**。它不是流，但和流（尤其是字符流、[[转换流]]）密不可分——**几乎所有中文乱码问题，根因都是编码**。

大白话：计算机只认 0 和 1（字节），不认「你」「好」这些字。字符编码就是一本「**字典型对照表**」——规定「你」这个字对应哪几个字节。读的时候按这本字典「**解码**」（字节→字符），写的时候「**编码**」（字符→字节）。用错字典，字就乱套了。

| 概念 | 说明 |
| --- | --- |
| 字符集（Charset） | 字符的集合（收录了哪些字） |
| 编码（Encoding） | 字符 → 字节的规则 |
| 解码（Decoding） | 字节 → 字符的规则 |

> 💡 **提示：** 乱码只有一个原因：**编码和解码用了不同的「字典」（编码表）**。解决乱码，就是让读写用同一个编码。

### 二、常见编码详解

#### 1. ASCII

最早的编码，**只覆盖英文**：
- 用 1 个字节（实际 7 位）表示一个字符
- 共 128 个字符：英文字母、数字、标点、控制符
- **不支持中文**

#### 2. ISO-8859-1（Latin-1）

ASCII 的扩展，覆盖西欧语言，**1 字节 1 字符**，仍**不支持中文**。

> 💡 **提示：** HTTP 头部默认用 ISO-8859-1，所以 HTTP 头里不能直接放中文（中文要 URL 编码）。

#### 3. GBK / GB2312 / GB18030

**中文编码**（中国国家标准）：
- GB2312：早期，覆盖常用汉字
- **GBK**：扩展，覆盖几乎所有汉字（中文 Windows 默认）
- GB18030：更全，少数民族文字
- **中文占 2 字节**

#### 4. Unicode（统一码）

目标是**收录全世界所有字符**的字符集（解决了「各国有各国的编码」的混乱）。它给每个字符分配一个唯一编号（码点），如「你」= `U+4F60`。

> ⚠️ **注意：** Unicode 是「**字符集**」（规定有哪些字、编号是多少），它本身**不是编码方式**。具体怎么存成字节，由 UTF-* 实现。

#### 5. UTF-8（最重要！）

Unicode 的**编码实现**之一，也是**当今最主流的编码**：
- **变长编码**：英文 1 字节，中文 3 字节，生僻字 4 字节
- 兼容 ASCII（英文部分和 ASCII 完全一样）
- 节省空间、容错性好

| 字符 | UTF-8 字节数 |
| --- | --- |
| 英文 `A` | 1 字节 |
| 中文 `你` | 3 字节 |
| Emoji 😀 | 4 字节 |

> 💡 **提示：** **现代项目统一用 UTF-8**。它是 Web、JSON、Java 源文件、数据库的事实标准。

#### 6. UTF-16

Unicode 的另一种编码实现，Java `char` 内部就是基于 UTF-16（大部分字符 2 字节）。

### 三、编码 / 解码过程

```
【写入（编码）】
字符 "你"  ──编码表(UTF-8)──►  字节 0xE4 0xBD 0xA0  ──► 存到文件/网络

【读取（解码）】
文件/网络  ──►  字节 0xE4 0xBD 0xA0  ──解码表(UTF-8)──►  字符 "你"
```

**核心规则：编码用什么表，解码就必须用什么表，否则乱码。**

```
✅ 写入用 UTF-8，读取用 UTF-8  →  正确显示
❌ 写入用 UTF-8，读取用 GBK    →  乱码
```

### 四、乱码的常见场景与解决

#### 1. 读文件乱码

```java
// ❌ FileReader 用系统默认编码，跨平台可能乱码
new FileReader("a.txt");

// ✅ 显式指定编码（JDK 11+）
new FileReader("a.txt", StandardCharsets.UTF_8);

// ✅ 用转换流指定编码（通用）
new InputStreamReader(new FileInputStream("a.txt"), StandardCharsets.UTF_8);
```

详见 [[转换流]]。

#### 2. 网络传输乱码

HTTP 请求/响应没指定编码，或前后端编码不一致。

```java
// ✅ 读 HTTP 响应指定编码
new InputStreamReader(responseStream, StandardCharsets.UTF_8);
```

#### 3. 数据库乱码

数据库、表、连接的编码要统一为 `utf8mb4`（支持 emoji）。

#### 4. 控制台乱码

JVM 编码和控制台编码不一致。启动参数加 `-Dfile.encoding=UTF-8`，并设置 IDE/终端为 UTF-8。

### 五、Java 中的编码操作

#### 1. 字符串 ↔ 字节转换

```java
import java.nio.charset.StandardCharsets;

String text = "你好";

// 编码：字符串 → 字节
byte[] bytes = text.getBytes(StandardCharsets.UTF_8);
//   UTF-8 下：6 个字节

// 解码：字节 → 字符串
String decoded = new String(bytes, StandardCharsets.UTF_8);
```

> ⚠️ **注意：** **永远不要用无参的 `getBytes()` / `new String(bytes)`**——它们用系统默认编码，跨平台就乱。始终显式指定 UTF-8。

#### 2. 查看与设置默认编码

```java
import java.nio.charset.Charset;

// 查看当前 JVM 默认编码
System.out.println(Charset.defaultCharset());

// 启动时设置：-Dfile.encoding=UTF-8
```

#### 3. StandardCharsets 常量（推荐）

```java
StandardCharsets.UTF_8        // ✅ 编译期检查，无拼写错误
StandardCharsets.ISO_8859_1
StandardCharsets.US_ASCII
StandardCharsets.UTF_16
```

#### 4. 判断编码是否支持

```java
System.out.println(Charset.isSupported("GBK"));   // true
```

### 六、UTF-8 为什么是主流

| 优势 | 说明 |
| --- | --- |
| 兼容 ASCII | 英文数据无成本迁移 |
| 变长节省空间 | 英文 1 字节，不像 UTF-16 固定 2 字节 |
| 容错好 | 错一个字节不会波及整个文本 |
| 全球通用 | 覆盖所有语言，Web 标准编码 |
| 支持完整 | 包括 emoji（4 字节） |

> 💡 **提示：** 选用 `utf8mb4` 而非 `utf8`（MySQL 的 `utf8` 是阉割版，只支持 3 字节，存不了 emoji）。

### 七、编码与流的关系

字符流的本质就是「**字节流 + 编码**」：

```
FileInputStream（字节流）+ UTF-8 解码器 = 能正确读中文的字符流
```

```
字符流 = 字节流 + 编码（Charset）
```

- `FileReader` / `FileWriter`：内置编码（默认或指定）
- `InputStreamReader` / `OutputStreamWriter`：[[转换流]]，可指定编码
- `String.getBytes(charset)` / `new String(bytes, charset)`：手动编解码

理解编码，才能真正驾驭字符流。

### 八、常见问题与注意事项

#### 1. 乱码怎么办？三步排查

1. 数据是**用什么编码写的**？
2. 你是**用什么编码读的**？
3. 让两者**一致**（统一 UTF-8）

#### 2. 乱码能修复吗？

如果只是「读错编码」但字节没变，可以重新解码修复：

```java
// 用错误编码读出了乱码 s，但底层字节还在
// 修复：先把乱码按错误编码还原成字节，再用正确编码解码
String fixed = new String(garbledStr.getBytes("ISO-8859-1"), "UTF-8");
```

但如果字节已经被破坏（如用了不支持中文的 ISO-8859-1 解码并丢弃了字节），**无法恢复**。

#### 3. getBytes() 不带编码的坑

```java
// ❌ Windows 上用 GBK，Mac 上用 UTF-8，同一段代码结果不同
"你好".getBytes();

// ✅ 统一 UTF-8
"你好".getBytes(StandardCharsets.UTF_8);
```

#### 4. 字节数 ≠ 字符数

```java
"你好".length();                        // 2（字符数）
"你好".getBytes(UTF_8).length;          // 6（UTF-8 字节数）
"你好".getBytes("GBK").length;          // 4（GBK 字节数）
```

处理固定长度协议（如 [[Socket通信]]）时，注意按**字节**算长度。

### 九、总结

| 要点 | 说明 |
| --- | --- |
| 编码 | 字符 → 字节 |
| 解码 | 字节 → 字符 |
| 乱码根因 | 编码和解码用了不同的表 |
| 主流编码 | **UTF-8**（统一用它） |
| 中文编码 | GBK（2 字节）、UTF-8（3 字节） |
| Java 操作 | `getBytes(charset)` / `new String(bytes, charset)` |
| 字符流本质 | 字节流 + 编码 |

一句话定调：**所有项目统一 UTF-8，所有编解码显式指定 UTF-8，乱码问题 99% 消失**。
