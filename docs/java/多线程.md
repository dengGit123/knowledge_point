# 多线程

> 官方文档：[Java Concurrency (Oracle Java Tutorials)](https://docs.oracle.com/javase/tutorial/essential/concurrency/)
> 参考教程：[菜鸟教程 - Java 多线程编程](https://www.runoob.com/java/java-multithreading.html)

## 一、概述

### 1. 进程 vs 线程

| 对比项   | 进程 (Process)             | 线程 (Thread)               |
| -------- | -------------------------- | --------------------------- |
| 定义     | 程序的一次执行过程         | 进程内的一个执行单元        |
| 资源     | 系统分配资源的基本单位     | CPU 调度的基本单位          |
| 内存     | 拥有独立的内存空间         | 共享所在进程的内存空间      |
| 通信     | 进程间通信 (IPC) 较复杂    | 线程间通信较简单（共享内存）|
| 切换开销 | 大                         | 小                          |

通俗理解：把**进程**比作一个工厂（有独立的厂房、仓库），**线程**就是工厂里的工人（共用厂房和仓库，但各自做不同的活）。

### 2. 并发 (Concurrency) vs 并行 (Parallelism)

| 概念 | 英文        | 含义                                           | 类比                       |
| ---- | ----------- | ---------------------------------------------- | -------------------------- |
| 并发 | Concurrency | 多个任务交替执行（同一时间段内）               | 一个人交替切两个西瓜       |
| 并行 | Parallelism | 多个任务真正同时执行（同一时刻，需多核 CPU）   | 两个人各自切一个西瓜       |

```
并发 (单核)              并行 (多核)
时间 →                   时间 →
┌────┐┌────┐┌────┐       ┌──────────┐
│ T1 ││ T2 ││ T1 │       │ T1 ─────│
└────┘└────┘└────┘       ├──────────┤
  交替执行                │ T2 ─────│
                          └──────────┘
                           真正同时执行
```

### 3. 为什么需要多线程

- **充分利用多核 CPU**：现代 CPU 多为多核，单线程无法发挥其性能。
- **提升响应速度**：耗时任务放后台线程，主线程（如 UI）不卡顿。
- **异步处理**：如批量发邮件、日志记录等可异步并行完成。
- **简化建模**：复杂任务可拆分为多个线程协作。

> ⚠️ **注意：** 多线程并非银弹。线程创建和切换有开销，且会带来线程安全问题、死锁等复杂问题。

## 二、线程的创建方式

Java 创建线程有**三种**经典方式。

### 1. 继承 Thread 类

重写 `run()` 方法，调用 `start()` 启动。

```java
// ✅ 方式一：继承 Thread 类
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " 正在执行");
    }
}

public class Demo {
    public static void main(String[] args) {
        MyThread t1 = new MyThread();
        MyThread t2 = new MyThread();
        t1.start(); // 启动新线程
        t2.start();
    }
}
```

> ⚠️ **注意：** Java 单继承，继承了 Thread 就不能再继承其他类，扩展性差。

### 2. 实现 Runnable 接口（推荐）

实现 `run()` 方法，交给 `Thread` 启动。

```java
// ✅ 方式二：实现 Runnable 接口（推荐）
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " 正在执行");
    }
}

public class Demo {
    public static void main(String[] args) {
        Runnable r = new MyRunnable();
        new Thread(r, "线程-A").start();
        new Thread(r, "线程-B").start();

        // Java 8 Lambda 写法更简洁
        new Thread(() -> System.out.println("Lambda 线程执行"), "线程-C").start();
    }
}
```

**推荐理由**：避免单继承局限；同一个 `Runnable` 可被多个线程共享，适合资源共享场景。

### 3. 实现 Callable + FutureTask

`Callable` 有**返回值**且可**抛出受检异常**，用 `FutureTask` 包装后交给 `Thread`。

```java
import java.util.concurrent.Callable;
import java.util.concurrent.FutureTask;
import java.util.concurrent.ExecutionException;

// ✅ 方式三：Callable + FutureTask
class MyCallable implements Callable<Integer> {
    @Override
    public Integer call() throws Exception {
        int sum = 0;
        for (int i = 1; i <= 100; i++) sum += i;
        return sum; // 有返回值
    }
}

public class Demo {
    public static void main(String[] args) throws ExecutionException, InterruptedException {
        FutureTask<Integer> future = new FutureTask<>(new MyCallable());
        new Thread(future, "计算线程").start();

        Integer result = future.get(); // 阻塞，直到任务完成并返回结果
        System.out.println("计算结果：" + result); // 5050
    }
}
```

### 三种方式对比

| 对比项      | 继承 Thread       | 实现 Runnable         | 实现 Callable         |
| ----------- | ----------------- | --------------------- | --------------------- |
| 返回值      | 无 (`void run()`) | 无 (`void run()`)     | 有 (`V call()`)       |
| 异常        | 不能抛受检异常    | 不能抛受检异常        | 可抛受检异常          |
| 继承局限    | 受单继承限制       | 无                    | 无                    |
| 资源共享    | 不便              | 方便共享              | 方便共享              |
| 推荐程度    | 不推荐            | 推荐                  | 推荐（需要返回值时）  |

> 💡 **提示：** 实际开发中，绝大多数场景用 **`Runnable` 或 `Callable` 配合线程池**，而不是直接 `new Thread()`。

## 三、start() vs run() 的区别（重点）

这是面试高频考点。

| 对比项       | `start()`                       | `run()`                          |
| ------------ | ------------------------------- | -------------------------------- |
| 作用         | 启动新线程，并自动调用 `run()`  | 普通方法调用                      |
| 是否新开线程 | 是                              | 否（在当前线程执行）              |
| 调用次数     | 一个 Thread 只能调用一次        | 可多次调用                        |
| 重复调用     | 抛 `IllegalThreadStateException`| 正常                              |

```java
Thread t = new Thread(() -> System.out.println("执行"));
t.start(); // ✅ 启动新线程
t.start(); // ❌ 重复 start，抛 IllegalThreadStateException

// 直接调用 run()，不开新线程（在 main 线程执行）
Thread t2 = new Thread(() -> System.out.println("执行"));
t2.run(); // ❌ 这样写没有开启新线程，只是普通方法调用
```

> 💡 **提示：** `start()` 底层会调用本地方法启动新线程，由 JVM 在新线程中回调 `run()`。

## 四、线程的生命周期

Java 6 种线程状态定义在 `java.lang.Thread.State` 枚举中。

```
                          ┌───────────┐
                          │    NEW    │  new Thread() 创建但未 start()
                          └─────┬─────┘
                                │ start()
                                ▼
                          ┌───────────┐
                ┌────────▶│ RUNNABLE  │◀─────────┐
                │         │ (就绪/运行)│           │
                │         └──┬──┬──┬──┘           │
                │  wait()    │  │  │  时间到/notify
                │            ▼  │  │              │
        ┌───────┴────┐  ┌──────┐ │  ┌────────────┴──┐
        │  WAITING   │  │BLOCKED│ │  │ TIMED_WAITING │
        │ (无限等待) │  │(阻塞) │ │  │  (限时等待)   │
        └────────────┘  └──────┘ │  └───────────────┘
                                 │
                                 │ run() 结束
                                 ▼
                          ┌───────────┐
                          │TERMINATED │  线程终止，不可再启动
                          └───────────┘
```

| 状态           | 说明                                              | 触发条件                            |
| -------------- | ------------------------------------------------- | ----------------------------------- |
| `NEW`          | 新建但未启动                                      | `new Thread()` 后                   |
| `RUNNABLE`     | 可运行（包含操作系统层面的就绪和运行）            | 调用 `start()`                      |
| `BLOCKED`      | 阻塞，等待 monitor 锁                             | 进入 `synchronized` 块未抢到锁      |
| `WAITING`      | 无限等待，需被其他线程显式唤醒                    | `wait()`、`join()`、`LockSupport.park()` |
| `TIMED_WAITING`| 限时等待，时间到自动唤醒                          | `sleep(ms)`、`wait(ms)`、`join(ms)` |
| `TERMINATED`   | 终止                                              | `run()` 执行完毕                    |

> ⚠️ **注意：** Java 没有「Running」状态，RUNNABLE 涵盖了就绪和运行。另外 `sleep` 和 IO 阻塞在 Java 层面仍是 RUNNABLE（取决于实现）。

## 五、线程常用方法

| 方法                                | 作用                                             | 是否释放锁 |
| ----------------------------------- | ------------------------------------------------ | ---------- |
| `sleep(long ms)`                    | 让当前线程睡眠指定毫秒（Thread 静态方法）        | 否         |
| `join()` / `join(long)`             | 等待目标线程执行结束                              | 是（内部用 wait） |
| `yield()`                           | 提示调度器让出 CPU（不保证）                      | 否         |
| `wait()` / `wait(ms)`               | 使当前线程等待（Object 方法，需持有 monitor 锁）  | 是         |
| `notify()` / `notifyAll()`          | 唤醒等待在该对象上的线程                          | -          |
| `setDaemon(true)`                   | 设置为守护线程                                    | -          |
| `setPriority(int)`                  | 设置优先级（1~10）                                | -          |
| `interrupt()`                       | 中断线程（设置中断标志）                          | -          |

### wait/notify vs sleep 的区别（重点）

| 对比项      | `wait()` / `notify()`          | `sleep()`                  |
| ----------- | ------------------------------ | -------------------------- |
| 所属类      | `Object`                       | `Thread`（静态方法）       |
| 是否释放锁  | **释放锁**                     | **不释放锁**               |
| 使用位置    | 必须在 `synchronized` 块内     | 任意位置                   |
| 唤醒方式    | `notify`/`notifyAll` 或超时    | 时间到自动                 |
| 作用对象    | 对象的 monitor                 | 当前线程                   |

```java
// ✅ wait/notify 必须在 synchronized 内，且锁对象要与调用对象一致
synchronized (lock) {
    while (!condition) {
        lock.wait(); // 释放 lock 锁并等待
    }
    // 条件满足后执行
}
```

### 守护线程 (Daemon)

守护线程是为用户线程服务的后台线程，**当所有用户线程结束时，守护线程会随之结束**（如 GC 线程）。

```java
Thread t = new Thread(() -> {
    while (true) {
        // 后台任务
    }
});
t.setDaemon(true); // 必须在 start() 之前设置
t.start();
```

> ⚠️ **注意：** `setDaemon()` 必须在 `start()` 前调用，否则抛 `IllegalThreadStateException`。

## 六、线程安全问题与同步

### 1. 线程安全三大特性

| 特性   | 含义                                             | 破坏示例              |
| ------ | ------------------------------------------------ | --------------------- |
| 原子性 | 操作不可分割，要么全执行要么不执行               | `i++`（非原子操作）   |
| 可见性 | 一个线程对共享变量的修改，其他线程能立即看到     | 工作内存/主内存不一致 |
| 有序性 | 程序执行顺序符合预期（防止指令重排序）           | 双重检查单例的坑      |

经典的**卖票问题**（多线程下 `ticket--` 不原子）：

```java
// ❌ 线程不安全：多个线程同时操作 ticket 可能出现负数或重复
class Ticket implements Runnable {
    private int ticket = 100;
    public void run() {
        while (ticket > 0) {
            try { Thread.sleep(10); } catch (InterruptedException e) {}
            System.out.println(Thread.currentThread().getName() + " 卖出 " + ticket--);
        }
    }
}
```

### 2. synchronized 关键字

`synchronized` 是 JVM 层面的锁，可修饰方法或代码块。

```java
class Ticket implements Runnable {
    private int ticket = 100;
    private final Object lock = new Object();

    public void run() {
        while (true) {
            // ✅ 同步代码块：锁指定对象
            synchronized (lock) {
                if (ticket <= 0) break;
                System.out.println(Thread.currentThread().getName()
                        + " 卖出 " + ticket--);
            }
        }
    }

    // ✅ 同步方法：锁的是 this（当前对象）
    public synchronized void sell() { /* ... */ }

    // ✅ 静态同步方法：锁的是 Class 对象（Ticket.class）
    public static synchronized void staticSell() { /* ... */ }
}
```

`synchronized` 锁的对象：

| 使用形式         | 锁对象                  |
| ---------------- | ----------------------- |
| 同步代码块       | 括号里指定的对象        |
| 普通同步方法     | `this`（当前实例）      |
| 静态同步方法     | 对应的 `Class` 对象     |

### 3. Lock / ReentrantLock

`ReentrantLock` 是 `java.util.concurrent.locks` 提供的显式锁。

```java
import java.util.concurrent.locks.ReentrantLock;

class Ticket implements Runnable {
    private int ticket = 100;
    private final ReentrantLock lock = new ReentrantLock(true); // true 表示公平锁

    public void run() {
        while (true) {
            lock.lock(); // 加锁
            try {
                if (ticket <= 0) break;
                System.out.println(Thread.currentThread().getName()
                        + " 卖出 " + ticket--);
            } finally {
                lock.unlock(); // ✅ 必须在 finally 中释放锁
            }
        }
    }
}
```

常用方法：

| 方法                  | 作用                                       |
| --------------------- | ------------------------------------------ |
| `lock()`              | 获取锁（阻塞等待）                         |
| `unlock()`            | 释放锁                                     |
| `tryLock()`           | 尝试获取锁，立即返回成功与否               |
| `tryLock(time, unit)` | 限时尝试获取锁                             |
| `lockInterruptibly()` | 可被中断地获取锁                           |
| `newCondition()`      | 获取条件变量，替代 `wait/notify`           |

> ⚠️ **注意：** `lock()` 之后必须 `try-finally` 中 `unlock()`，否则一旦异常锁不会被释放导致死锁。

### 4. synchronized vs Lock 对比

| 对比项       | synchronized              | ReentrantLock                       |
| ------------ | ------------------------- | ----------------------------------- |
| 实现         | JVM 关键字（隐式）        | API 类（显式 `lock/unlock`）        |
| 锁释放       | 自动（出代码块/异常）     | 手动（必须 `finally unlock`）       |
| 响应中断     | 否                        | `lockInterruptibly()` 可响应         |
| 超时获取     | 否                        | `tryLock(time)` 支持                |
| 公平锁       | 只能非公平                | 可配置公平/非公平                    |
| 条件变量     | `wait/notify`（单条件）   | 多个 `Condition`                    |
| 性能         | JDK 6 后优化，差距很小    | 灵活性高                            |

## 七、volatile 关键字

`volatile` 是轻量级同步机制，主要解决**可见性**和**有序性**。

### 1. 两大作用

| 特性     | volatile 是否保证 | 原理                                     |
| -------- | ----------------- | ---------------------------------------- |
| 可见性   | ✅ 保证           | 强制读写主内存，禁用 CPU 缓存            |
| 有序性   | ✅ 保证           | 插入内存屏障，禁止指令重排序             |
| 原子性   | ❌ 不保证         | `i++` 仍是非原子操作                     |

### 2. 典型应用：状态标志位

```java
// ✅ 用 volatile 做停止标志，保证可见性
class Worker implements Runnable {
    private volatile boolean running = true;

    public void run() {
        while (running) {
            // 持续工作
        }
        System.out.println("线程结束");
    }

    public void shutdown() {
        running = false; // 其他线程修改后，工作线程能立即看到
    }
}
```

### 3. 不保证原子性

```java
// ❌ volatile 不能让 i++ 变成原子操作
private volatile int count = 0;
count++; // 实际是 读取-修改-写入 三步，并发下仍出错

// ✅ 需要原子性请用 AtomicInteger 或 synchronized
private final AtomicInteger count = new AtomicInteger();
count.incrementAndGet();
```

### volatile vs synchronized

| 对比项     | volatile            | synchronized            |
| ---------- | ------------------- | ----------------------- |
| 原子性     | 不保证              | 保证                    |
| 可见性     | 保证                | 保证                    |
| 有序性     | 保证                | 保证                    |
| 是否阻塞   | 否                  | 是                      |
| 性能       | 轻量                | 相对较重                |
| 适用场景   | 状态标志、单次读写 | 复合操作临界区           |

## 八、线程池（重点中的重点）

### 1. 为什么用线程池

- **降低资源消耗**：复用已创建的线程，避免频繁创建销毁。
- **提高响应速度**：任务到达时无需等待线程创建。
- **便于统一管理**：统一调优、监控、限流。
- **避免无限创建线程**：防止线程过多导致 OOM 或系统崩溃。

### 2. ThreadPoolExecutor 七大参数

```java
public ThreadPoolExecutor(
    int corePoolSize,                      // 1. 核心线程数
    int maximumPoolSize,                   // 2. 最大线程数
    long keepAliveTime,                    // 3. 非核心线程空闲存活时间
    TimeUnit unit,                         // 4. 时间单位
    BlockingQueue<Runnable> workQueue,     // 5. 任务队列
    ThreadFactory threadFactory,           // 6. 线程工厂
    RejectedExecutionHandler handler       // 7. 拒绝策略
)
```

| 参数          | 说明                                                             |
| ------------- | ---------------------------------------------------------------- |
| corePoolSize  | 核心线程数，即使空闲也保留（除非设置 `allowCoreThreadTimeOut`）  |
| maximumPoolSize | 线程池最大线程数                                                |
| keepAliveTime | 非核心线程空闲后的存活时间                                       |
| unit          | keepAliveTime 的单位（`TimeUnit.SECONDS` 等）                    |
| workQueue     | 阻塞队列，如 `LinkedBlockingQueue`、`ArrayBlockingQueue`、`SynchronousQueue` |
| threadFactory | 创建线程的工厂（可自定义线程名）                                 |
| handler       | 拒绝策略（任务太多处理不过来时）                                 |

### 3. 线程池工作流程

```
                 提交任务 (execute)
                       │
                       ▼
            ┌──────────────────────┐
            │ 当前线程数 < 核心线程数？│
            └──────────┬───────────┘
                   是  │  否
           ┌───────────┘  └────────────┐
           ▼                            ▼
   创建核心线程执行任务          ┌─────────────────┐
                                │ 队列是否已满？    │
                                └────────┬────────┘
                                    否  │   是
                            ┌───────────┘   └─────────────┐
                            ▼                             ▼
                       放入队列等待           ┌─────────────────────┐
                                              │ 当前线程 < 最大线程数？│
                                              └──────────┬──────────┘
                                                    是   │    否
                                        ┌──────────────┘  └─────────────┐
                                        ▼                                ▼
                                创建非核心线程执行                  执行拒绝策略
```

**核心流程口诀**：核心线程 → 队列 → 非核心线程 → 拒绝策略。

### 4. 四种拒绝策略

| 策略                | 行为                                       |
| ------------------- | ------------------------------------------ |
| `AbortPolicy`（默认）| 抛出 `RejectedExecutionException`          |
| `CallerRunsPolicy`  | 由提交任务的线程自己执行该任务              |
| `DiscardPolicy`     | 静默丢弃新任务                              |
| `DiscardOldestPolicy`| 丢弃队列中最旧的任务，再尝试提交新任务     |

### 5. Executors 工厂方法为何不推荐

阿里《Java 开发手册》强制规定**禁用 `Executors` 创建线程池**，原因如下：

| Executors 方法            | 实际配置                                   | 风险        |
| ------------------------- | ------------------------------------------ | ----------- |
| `newFixedThreadPool`      | 队列用 `LinkedBlockingQueue`（无界）       | OOM（队列） |
| `newSingleThreadExecutor` | 同上，无界队列                             | OOM（队列） |
| `newCachedThreadPool`     | 最大线程数 `Integer.MAX_VALUE`             | OOM（线程） |
| `newScheduledThreadPool`  | 队列无界                                   | OOM（队列） |

> ⚠️ **注意：** 应使用 `new ThreadPoolExecutor(...)` 手动指定参数，便于控制资源和排查问题。

### 6. 手写 ThreadPoolExecutor 完整示例

```java
import java.util.concurrent.*;

public class PoolDemo {
    public static void main(String[] args) {
        ThreadPoolExecutor pool = new ThreadPoolExecutor(
            2,                              // 核心线程数
            4,                              // 最大线程数
            30, TimeUnit.SECONDS,           // 非核心线程空闲存活时间
            new ArrayBlockingQueue<>(10),   // 有界任务队列
            Executors.defaultThreadFactory(),// 默认线程工厂
            new ThreadPoolExecutor.AbortPolicy() // 拒绝策略
        );

        // 提交 Runnable（无返回值）
        pool.execute(() -> System.out.println("执行任务"));

        // 提交 Callable（有返回值，返回 Future）
        Future<String> future = pool.submit(() -> {
            Thread.sleep(500);
            return "任务结果";
        });
        try {
            System.out.println(future.get());
        } catch (Exception e) {
            e.printStackTrace();
        }

        pool.shutdown(); // 关闭线程池（不再接受新任务，已提交任务执行完）
    }
}
```

> 💡 **提示：** 线程池关闭用 `shutdown()`（平缓关闭）或 `shutdownNow()`（立即停止）。生产环境线程池一般随 JVM 长期存活，不关闭。

### CPU 密集型 vs IO 密集型线程数

| 任务类型   | 推荐核心线程数              | 说明                         |
| ---------- | --------------------------- | ---------------------------- |
| CPU 密集型 | `N + 1`（N = CPU 核数）     | 线程过多反而频繁切换         |
| IO 密集型  | `2N` 或更多                 | 线程多在等待 IO，可多开      |

## 九、常用并发工具类

### 1. CountDownLatch（一次性倒计时）

让一个或多个线程等待，直到其他线程执行完毕。计数器**不可重置**。

```java
import java.util.concurrent.CountDownLatch;

public class Demo {
    public static void main(String[] args) throws InterruptedException {
        final int N = 3;
        CountDownLatch latch = new CountDownLatch(N); // 计数 3

        for (int i = 0; i < N; i++) {
            new Thread(() -> {
                System.out.println(Thread.currentThread().getName() + " 完成");
                latch.countDown(); // 计数减 1
            }).start();
        }

        latch.await(); // 等待计数归零
        System.out.println("所有子任务完成，继续主流程");
    }
}
```

### 2. CyclicBarrier（可复用屏障）

让一组线程相互等待，到达屏障点后一起继续执行，**可重复使用**。

```java
import java.util.concurrent.CyclicBarrier;

public class Demo {
    public static void main(String[] args) {
        CyclicBarrier barrier = new CyclicBarrier(3, () ->
                System.out.println("全部到达屏障，开始下一阶段"));

        for (int i = 0; i < 3; i++) {
            new Thread(() -> {
                System.out.println(Thread.currentThread().getName() + " 到达");
                try { barrier.await(); } catch (Exception e) {}
            }).start();
        }
    }
}
```

### 3. Semaphore（信号量，限流）

控制同时访问某资源的线程数量。

```java
import java.util.concurrent.Semaphore;

public class Demo {
    public static void main(String[] args) {
        Semaphore semaphore = new Semaphore(3); // 只允许 3 个线程同时访问

        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                try {
                    semaphore.acquire(); // 获取许可
                    System.out.println(Thread.currentThread().getName() + " 访问资源");
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                } finally {
                    semaphore.release(); // 释放许可
                }
            }).start();
        }
    }
}
```

### 4. ConcurrentHashMap

线程安全的 HashMap，通过分段锁（JDK 7）/ CAS + synchronized（JDK 8）保证并发安全，性能远优于 `Hashtable` 和 `Collections.synchronizedMap`。

### CountDownLatch vs CyclicBarrier

| 对比项    | CountDownLatch          | CyclicBarrier           |
| --------- | ----------------------- | ----------------------- |
| 可复用    | 否                      | 是                      |
| 计数方式  | countDown() 递减        | await() 等待            |
| 主语      | 一个线程等 N 个         | N 个线程互相等          |
| 侧重点    | 等待任务完成            | 线程同步到某一点        |

## 十、原子类与 CAS

### 1. 原子类

`java.util.concurrent.atomic` 包提供基于 CAS 的原子类，无需加锁即可实现原子操作。

| 类型           | 示例                  | 用途                  |
| -------------- | --------------------- | --------------------- |
| 基本类型       | `AtomicInteger`       | 原子 int              |
| 基本类型       | `AtomicLong`          | 原子 long             |
| 基本类型       | `AtomicBoolean`       | 原子 boolean          |
| 引用类型       | `AtomicReference`     | 原子对象引用          |
| 数组类型       | `AtomicIntegerArray`  | 原子数组              |
| 字段更新器     | `AtomicIntegerFieldUpdater` | 原子更新某字段   |
| 累加器（JDK 8）| `LongAdder`           | 高并发计数（优于 AtomicLong） |

```java
import java.util.concurrent.atomic.AtomicInteger;

AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet(); // 等价于 ++count，原子操作
count.getAndIncrement(); // 等价于 count++
count.compareAndSet(0, 1); // CAS：期望值为 0 则设为 1
```

### 2. CAS 原理

**CAS (Compare And Swap)**：比较并交换，是一种**乐观锁**思想。

```
CAS(V, expected, newValue):
  如果 V 的当前值 == expected:
      将 V 更新为 newValue，返回 true
  否则:
      返回 false（说明已被其他线程修改，重试）
```

整个操作由 CPU 的 `cmpxchg` 指令保证原子性。`AtomicInteger` 内部用 `Unsafe` 的 `compareAndSwapInt` 实现。

### 3. ABA 问题

一个值从 A → B → A，CAS 检查时以为没变过，实际上已被修改过。

- **解决**：用 `AtomicStampedReference`，给值加版本号（`A1 → B2 → A3`，CAS 比较时连版本号一起比较）。

> 💡 **提示：** CAS 的缺点：自旋开销大（竞争激烈时）；只能保证单个变量的原子性。

## 十一、ThreadLocal

### 1. 是什么

`ThreadLocal` 为每个使用该变量的线程提供**独立的副本**，实现线程间数据隔离。

```
                ThreadLocal<T> tl = new ThreadLocal<>();

  线程 A                线程 B              线程 C
   │                     │                   │
   ▼                     ▼                   ▼
┌─────────┐         ┌─────────┐         ┌─────────┐
│ tl: "A" │         │ tl: "B" │         │ tl: "C" │   各自独立副本
└─────────┘         └─────────┘         └─────────┘
```

### 2. 典型应用

- **数据库连接 / 事务**：每个线程一个 Connection，避免争用。
- **SimpleDateFormat**：非线程安全，用 ThreadLocal 给每个线程一份。
- **用户上下文**：保存当前请求的登录用户信息。

```java
public class Context {
    // ✅ 每个线程一份 SimpleDateFormat，避免线程安全问题
    private static final ThreadLocal<SimpleDateFormat> dateFormat =
            ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

    public static String format(Date date) {
        return dateFormat.get().format(date);
    }

    // 用户上下文示例
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();
    public static void set(User u) { currentUser.set(u); }
    public static User get() { return currentUser.get(); }
    public static void clear() { currentUser.remove(); } // ✅ 用完务必 remove
}
```

### 3. 内存泄漏注意

`ThreadLocal` 内部通过 `ThreadLocalMap`（key 为弱引用，value 为强引用）存储。线程长期存活时（如线程池），若不 `remove()`，value 可能无法回收导致**内存泄漏**。

> ⚠️ **注意：** 在线程池环境下，使用完 `ThreadLocal` **必须在 `finally` 中调用 `remove()`** 清理。

## 十二、死锁

### 1. 死锁的四个必要条件

| 条件         | 含义                                       |
| ------------ | ------------------------------------------ |
| 互斥         | 资源同一时刻只能被一个线程占有             |
| 占有并等待   | 持有资源的同时等待新的资源                 |
| 不可剥夺     | 不能强行夺走线程已占有的资源               |
| 循环等待     | 多个线程形成首尾相接的等待链               |

### 2. 死锁示例

```java
public class DeadLockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        // ❌ 线程 1：先锁 A 再锁 B
        new Thread(() -> {
            synchronized (lockA) {
                System.out.println("T1 持有 A，等待 B");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lockB) {
                    System.out.println("T1 获得 B");
                }
            }
        }, "T1").start();

        // ❌ 线程 2：先锁 B 再锁 A，形成循环等待
        new Thread(() -> {
            synchronized (lockB) {
                System.out.println("T2 持有 B，等待 A");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lockA) {
                    System.out.println("T2 获得 A");
                }
            }
        }, "T2").start();
    }
}
```

### 3. 如何避免

| 方法                 | 说明                                                     |
| -------------------- | -------------------------------------------------------- |
| 固定加锁顺序         | 所有线程按相同顺序获取多把锁（最常用）                   |
| 设置超时             | 用 `tryLock(time)`，超时放弃，避免无限等待               |
| 一次性获取所有锁     | 用 `tryLock` 同时尝试获取，失败则全部释放重试            |
| 减小锁粒度           | 缩小同步范围，缩短持锁时间                               |

> 💡 **提示：** 排查死锁可用 `jstack` 打印线程堆栈，或用 jconsole/jvisualvm 的死锁检测功能。

## 十三、面试常见问题

### Q1：创建线程有几种方式？哪种推荐？

三种：继承 `Thread`、实现 `Runnable`、实现 `Callable`。推荐 `Runnable`/`Callable` 配合线程池，因为：避免单继承局限、便于资源共享、方便统一管理和复用。

### Q2：start() 和 run() 的区别？

`start()` 启动一个新线程，由 JVM 在新线程中回调 `run()`；直接调用 `run()` 只是普通方法调用，不开启新线程。一个 Thread 只能 `start()` 一次，重复调用抛 `IllegalThreadStateException`。

### Q3：sleep() 和 wait() 的区别？

- `sleep()` 是 `Thread` 静态方法，**不释放锁**；`wait()` 是 `Object` 方法，**释放锁**。
- `wait()` 必须在 `synchronized` 块内调用，由 `notify/notifyAll` 唤醒或超时。
- `sleep()` 可在任意位置调用，时间到自动唤醒。

### Q4：synchronized 和 Lock 的区别？

| 维度     | synchronized       | Lock              |
| -------- | ------------------ | ----------------- |
| 释放     | 自动               | 手动 finally 释放 |
| 中断     | 不响应             | 可响应（`lockInterruptibly`）|
| 超时     | 不支持             | 支持（`tryLock(time)`）|
| 公平     | 非公平             | 可配置            |
| 条件     | 单一 wait/notify   | 多 Condition      |

### Q5：volatile 的作用？

保证**可见性**（强制读写主存）和**有序性**（禁止指令重排序），但**不保证原子性**。常用于状态标志位。需要原子性时配合 `AtomicXxx` 或 `synchronized`。

### Q6：线程池有哪些核心参数？工作流程是怎样的？

七大参数：corePoolSize、maximumPoolSize、keepAliveTime、unit、workQueue、threadFactory、handler。

工作流程：提交任务时，若当前线程数 < corePoolSize，创建核心线程；否则尝试放入队列；队列满后，若线程数 < maximumPoolSize，创建非核心线程；若已达最大线程数，执行拒绝策略。

### Q7：线程池的拒绝策略有哪些？

四种：`AbortPolicy`（默认，抛异常）、`CallerRunsPolicy`（调用者线程执行）、`DiscardPolicy`（静默丢弃）、`DiscardOldestPolicy`（丢弃最旧任务再提交）。

### Q8：为什么不推荐用 Executors 创建线程池？

`newFixedThreadPool` 和 `newSingleThreadExecutor` 用无界队列，可能 OOM；`newCachedThreadPool` 最大线程数为 `Integer.MAX_VALUE`，可能因创建过多线程 OOM。应手动用 `ThreadPoolExecutor` 指定参数。

### Q9：什么是死锁？如何避免？

多个线程互相持有对方需要的锁，导致永久阻塞。四个必要条件：互斥、占有并等待、不可剥夺、循环等待。避免方法：固定加锁顺序、使用 `tryLock` 设置超时、减小锁粒度。

### Q10：CAS 是什么？有什么问题？

CAS (Compare And Swap) 是乐观锁机制，比较当前值与期望值，相等则更新，由 CPU 指令保证原子性。缺点：竞争激烈时自旋开销大；存在 ABA 问题（用 `AtomicStampedReference` 加版本号解决）。
