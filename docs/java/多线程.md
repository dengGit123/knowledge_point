> 📖 官方教程：[Java Concurrency (Oracle Java Tutorials)](https://docs.oracle.com/javase/tutorial/essential/concurrency/)
> 📖 官方 API：[Thread.State (Java Platform SE 21)](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/lang/Thread.State.html)
> 📖 参考教程：[菜鸟教程 - Java 多线程编程](https://www.runoob.com/java/java-multithreading.html)

## 一、概述

**多线程**是指一个程序中**同时运行多个线程**，让多个任务"齐头并进"地执行。Java 从设计之初就内置了对多线程的支持（核心类是 `java.lang.Thread`），多线程也是 Java 高并发开发和面试的重中之重。

### 为什么需要多线程

- **充分利用多核 CPU**：现代 CPU 多为多核，单线程只能用到其中一个核，多线程能把多个核都用起来。
- **提升响应速度**：把耗时任务（如下载、计算）放到后台线程，主线程（如 UI、接口主流程）不会被卡住。
- **异步处理**：批量发邮件、记录日志、生成报表等可以异步并行完成，不必让用户干等。
- **简化建模**：把复杂任务拆成多个"各司其职"的线程来协作，逻辑更清晰。

> ⚠️ **注意：** 多线程**不是银弹**。线程的创建、销毁、切换都有开销，还会带来**线程安全**、**死锁**等复杂问题。用得好是提速，用不好是"埋坑"。

## 二、进程 vs 线程

### 1. 先分清"程序、进程、线程"

- **程序（Program）**：躺在硬盘上的代码，是**静态**的。比如你电脑里的 `WeChat.exe`，不运行时它只是个文件。
- **进程（Process）**：程序**运行起来**后的实例，是**动态**的。双击打开微信，操作系统就给它创建了一个进程，分配了内存等资源。可以同时打开多个微信（多个进程）。
- **线程（Thread）**：进程内部的**执行单元**。一个进程里至少有一个线程（主线程），也可以有多个。线程是 **CPU 调度的基本单位**——真正在 CPU 上跑代码的是线程，不是进程。

### 2. 一个通俗类比

把**进程**比作一个**工厂**，**线程**就是工厂里的**工人**：

- 工厂（进程）有自己独立的厂房和仓库（**独立内存空间**），不同工厂之间互不干涉。
- 工人（线程）住在同一个工厂里，**共用**这个工厂的厂房和仓库（**共享进程内存**），但每个工人手上干的活可能不同。
- 多招几个工人（多线程），工厂的活就能**并行**地干，效率更高。
- 工人之间因为**共用仓库**，可能会**抢同一个工具**（线程安全问题），需要排队或上锁。

### 3. 进程与线程对比

| 对比项     | 进程 (Process)                   | 线程 (Thread)                      |
| ---------- | -------------------------------- | ---------------------------------- |
| **定义**   | 程序运行后的实例                 | 进程内的一个执行单元               |
| **资源**   | 操作系统**分配资源**的基本单位   | **CPU 调度**的基本单位             |
| **内存**   | 拥有**独立**的内存空间           | **共享**所在进程的内存空间         |
| **数量**   | 一个程序可对应多个进程           | 一个进程可包含多个线程（至少 1 个）|
| **通信**   | 进程间通信（IPC）较复杂、较慢    | 线程间可直接读写共享内存，简单快   |
| **切换开销**| **大**（要切换内存空间、页表等）| **小**（内存不变，只切换执行上下文）|
| **健壮性** | 一个进程崩溃通常不影响其他进程   | 一个线程崩溃可能导致整个进程挂掉   |

### 4. 几个关键点

- **每个 Java 程序启动时都有至少一个线程**：执行 `main` 方法的那个，叫**主线程（main 线程）**。
- **为什么线程切换比进程快**：进程切换要更换内存空间（页表、TLB 刷新）开销大；线程切换时内存空间不变（同属一个进程），只需保存/恢复寄存器和栈，所以**轻量**得多——线程也常被称为"轻量级进程（LWP）"。
- **Java 线程与系统线程的关系**：从 JDK 1.2 起，Java 线程采用 **1:1 模型**——**每个 Java 线程直接对应一个操作系统线程**，调度交给操作系统。JDK 21 起正式引入的**虚拟线程（Virtual Thread）**则允许多个轻量级线程映射到少量平台线程上（M:N），适合超高并发场景。

> 💡 **提示：** 线程"共享内存"是双刃剑——好处是通信方便（直接读写共享变量），坏处是**多个线程同时改同一份数据会出问题**，这正是后面"线程安全"要解决的核心。

## 三、并发 (Concurrency) vs 并行 (Parallelism)

这两个词极易混淆，但含义不同。一句话区分：

> **并发是"同时处理"多件事，并行是"同时做"多件事。**

### 1. 并发（Concurrency）

多个任务在**同一时间段内**都有进展，但**不一定是同一时刻**在执行。强调的是"**能处理**多个任务的能力（结构）"。

- 在**单核 CPU** 上，并发靠**时间片快速切换**实现：CPU 在任务 A、B 之间飞速来回切换，由于切换极快，人感觉它们"同时"在跑，其实任何一刻都只有一个任务在执行。
- 类比：**一个厨师同时张罗几道菜**——灶上炖汤的间隙去切菜，切菜的间隙去摆盘，来回切换，"看起来"同时在做。

### 2. 并行（Parallelism）

多个任务在**同一时刻**真正同时执行。强调的是"**真正同时**执行（能力）"。

- 需要**多核 CPU**（或多个处理器）才能实现——每个核跑一个任务，物理上同时进行。
- 类比：**几个厨师各做各的菜**——每人一口锅，同一瞬间真的在同时炒。

```
并发 Concurrency —— 单核靠"快速切换"实现（任意时刻只跑 1 个）
时间 →
   ┌──T1──┐┌──T2──┐┌──T1──┐┌──T2──┐┌──T1──┐
   交替推进，"看起来"同时

并行 Parallelism —— 多核"真正同时"（同一时刻跑多个）
         核1: ┌──────────T1──────────┐
         核2: ┌──────────T2──────────┐
         核3: ┌──────────T3──────────┐
```

### 3. 对比

| 对比项   | 并发 (Concurrency)             | 并行 (Parallelism)           |
| -------- | ------------------------------ | ---------------------------- |
| **本质** | 多个任务在时间段内交替推进     | 多个任务在同一时刻同时执行   |
| **硬件** | 单核也能并发                   | 必须多核/多处理器            |
| **侧重** | **结构**：程序能处理多任务     | **执行**：硬件真能同时算     |
| **关系** | 目标/设计                      | 手段/能力                    |

### 4. 二者的关系（重要）

并发和并行**不是对立**的：

- 并发是一种**程序结构**（把任务设计成可独立推进的多个部分）；
- 并行是一种**执行能力**（有多核来真正同时跑）。

> 💡 **结论：** 一个设计良好的**并发**程序，在**多核** CPU 上运行时，那些可独立推进的任务会被分配到不同核心上**真正并行**执行。Go 语言有一句名言很好地概括：**"并发是关于同时处理很多事情，并行是关于同时做很多事情。"**

## 四、线程的创建方式

> 💡 **本质：** 在 Java 中，**真正创建线程只有一条路——`new Thread().start()`**。所谓的"几种方式"，其实是**把"要执行的代码（任务）"交给线程的几种不同写法**。

### 方式一：继承 Thread 类

`Thread` 本身就是一个类，继承它并重写 `run()` 方法，把任务写在 `run()` 里。

```java
// ✅ 方式一：继承 Thread，重写 run()
class MyThread extends Thread {
    @Override
    public void run() {
        // 这个方法的代码就是新线程要执行的任务
        System.out.println(Thread.currentThread().getName() + " 正在执行");
    }
}

public class Demo {
    public static void main(String[] args) {
        MyThread t1 = new MyThread();
        MyThread t2 = new MyThread();
        t1.start(); // ✅ start() 才会真正启动新线程
        t2.start();
    }
}
```

> ⚠️ **注意：** Java 是**单继承**，继承了 `Thread` 就不能再继承别的类，扩展性差，所以**不推荐**。

### 方式二：实现 Runnable 接口（推荐）

实现 `Runnable` 接口的 `run()` 方法，再用 `Thread` 包装启动。把"任务"和"线程"解耦。

```java
// ✅ 方式二：实现 Runnable（推荐）
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println(Thread.currentThread().getName() + " 正在执行");
    }
}

public class Demo {
    public static void main(String[] args) {
        Runnable r = new MyRunnable();
        new Thread(r, "线程-A").start(); // 把任务 r 交给线程执行
        new Thread(r, "线程-B").start(); // 同一个 r 可被多个线程共享

        // Java 8+ Lambda 写法更简洁（Runnable 是函数式接口）
        new Thread(() -> System.out.println("Lambda 线程执行"), "线程-C").start();
    }
}
```

**推荐理由**：

- **避免单继承局限**：实现接口不影响继承其他类。
- **任务与线程分离**：`Runnable` 只描述"做什么"，由谁执行可以灵活搭配。
- **资源共享**：同一个 `Runnable` 实例可被多个线程共享（如卖票场景）。

### 方式三：实现 Callable + FutureTask（需要返回值时）

`Runnable` 的 `run()` 没有**返回值**、也**不能抛出受检异常**。如果任务需要返回结果，用 `Callable`。

> ⚠️ **注意：** `Callable` **不能直接交给 `Thread`**（`Thread` 只认 `Runnable`），必须先用 `FutureTask` 包装一层，而 `FutureTask` 同时实现了 `Runnable` 和 `Future`，这样才能交给 `Thread`。

```java
import java.util.concurrent.Callable;
import java.util.concurrent.FutureTask;
import java.util.concurrent.ExecutionException;

// ✅ 方式三：Callable + FutureTask
class MyCallable implements Callable<Integer> {
    @Override
    public Integer call() throws Exception {   // call() 有返回值，可抛受检异常
        int sum = 0;
        for (int i = 1; i <= 100; i++) sum += i;
        return sum;                            // 返回计算结果
    }
}

public class Demo {
    public static void main(String[] args) throws ExecutionException, InterruptedException {
        // 1. 用 FutureTask 包装 Callable
        FutureTask<Integer> future = new FutureTask<>(new MyCallable());
        // 2. 把 FutureTask（它实现了 Runnable）交给 Thread
        new Thread(future, "计算线程").start();

        // 3. future.get() 会阻塞当前线程，直到任务执行完并拿到结果
        Integer result = future.get();
        System.out.println("计算结果：" + result); // 5050
    }
}
```

### 方式四：线程池（实际开发最常用）

实际项目中**几乎不会手动 `new Thread()`**，而是用**线程池**来统一创建和管理线程。线程池既能执行 `Runnable`，也能执行 `Callable`。

```java
import java.util.concurrent.*;

ExecutorService pool = Executors.newFixedThreadPool(3); // （示例，生产环境用 ThreadPoolExecutor）

pool.execute(() -> System.out.println("Runnable 任务，无返回值"));          // 提交 Runnable
Future<String> f = pool.submit(() -> "Callable 任务，有返回值");          // 提交 Callable
System.out.println(f.get());                                               // 拿返回值
```

> ⚠️ **注意：** `Executors` 工具方法存在 OOM 风险，生产环境应直接用 `new ThreadPoolExecutor(...)`（详见第九章）。

### 四种方式对比

| 对比项     | 继承 Thread       | 实现 Runnable    | 实现 Callable     | 线程池               |
| ---------- | ----------------- | ---------------- | ----------------- | -------------------- |
| **返回值** | 无 (`void run()`) | 无 (`void run()`)| 有 (`V call()`)   | 有/无均可            |
| **异常**   | 不能抛受检异常    | 不能抛受检异常   | 可抛受检异常      | 取决于任务类型       |
| **继承局限**| 受单继承限制      | 无               | 无                | 无                   |
| **资源共享**| 不便              | 方便             | 方便              | 方便                 |
| **线程管理**| 手动创建销毁      | 手动创建销毁     | 手动创建销毁      | **复用、统一管理**   |
| **推荐度** | 不推荐            | 推荐             | 推荐（要返回值）  | **强烈推荐**         |

## 五、创建之后使用（启动与常用操作）

创建线程只是准备好了"工人和任务"，怎么启动、怎么配合、怎么优雅地停掉，是这一章的重点。

### 1. 启动：start() vs run()（高频考点）

| 对比项       | `start()`                          | `run()`                           |
| ------------ | ---------------------------------- | --------------------------------- |
| **作用**     | 启动一个**新线程**，由它回调 `run()`| 就是**普通方法调用**              |
| **是否开新线程**| **是**                             | **否**（在调用者线程里执行）      |
| **调用次数** | 一个 `Thread` 只能调用**一次**     | 可多次调用                        |
| **重复调用** | 抛 `IllegalThreadStateException`   | 正常                              |

```java
Thread t = new Thread(() -> System.out.println("执行线程：" + Thread.currentThread().getName()));

// ❌ 直接调 run()：没有开新线程，代码在 main 线程里同步执行
t.run();   // 输出：执行线程：main

// ✅ 调 start()：启动新线程，由新线程回调 run()
t.start(); // 输出：执行线程：Thread-0

// ❌ 同一个 Thread 不能重复 start()
t.start(); // 抛 IllegalThreadStateException
```

> 💡 **提示：** `start()` 底层会调用**本地方法（native）**，由 JVM 在操作系统层面创建新线程，再由新线程回调 `run()`。日常**只调 `start()`**，绝不手动调 `run()`。

### 2. 线程命名与获取当前线程

```java
// 创建时通过构造方法命名
Thread t = new Thread(() -> {}, "我的线程");

// 或用 setName() 设置
t.setName("worker-1");

// 获取当前正在执行的线程（在任意代码里都能用）
Thread current = Thread.currentThread();
String name = current.getName();   // 如 "main"、"worker-1"
```

> 💡 **提示：** 给线程起个有意义的名字，**排查问题时非常重要**（堆栈、日志、jstack 都会显示线程名）。

### 3. 睡眠：Thread.sleep()

让**当前线程**暂停指定毫秒，**让出 CPU**，但不释放锁。

```java
try {
    Thread.sleep(2000); // 当前线程睡 2 秒（1 秒 = 1000 毫秒）
} catch (InterruptedException e) {
    // 睡眠期间若被 interrupt()，会抛此异常
    Thread.currentThread().interrupt(); // 建议重新设置中断标志
}
```

### 4. 等待其他线程结束：join()

让当前线程**阻塞等待**目标线程执行完毕，再继续往下走。常用于"等所有子任务完成后再汇总"。

```java
Thread t1 = new Thread(() -> {
    try { Thread.sleep(1000); } catch (InterruptedException e) {}
    System.out.println("t1 完成");
}, "t1");
t1.start();

t1.join();  // main 线程在这里等 t1 结束
// t1.join(500); // 也可以限时等待 500ms，超时就不再等

System.out.println("main 继续（t1 已结束）");
```

### 5. 让出 CPU：Thread.yield()

**提示**调度器"我愿意让出 CPU"，但**只是建议，调度器可以无视**。线程仍处于 RUNNABLE（就绪）状态。

```java
Thread.yield(); // 提示让出，不保证一定让
```

### 6. 中断线程：interrupt()（重要且易错）

Java **没有强制停止线程**的安全 API，正确做法是用**协作式中断**：一个线程通过 `interrupt()` 发出"请停止"的请求，目标线程**自己检查并决定何时退出**。

```java
Thread t = new Thread(() -> {
    // ✅ 推荐模式：在循环里检查中断标志
    while (!Thread.currentThread().isInterrupted()) {
        // 干活...
    }
    System.out.println("收到中断，正常收尾后退出");
}, "worker");
t.start();

t.interrupt(); // 请求中断：把 t 的中断标志设为 true
```

> ⚠️ **注意（经典坑）：** 如果线程正卡在 `sleep()` / `wait()` / `join()` 中被 `interrupt()`，会立刻抛出 `InterruptedException`，**并且中断标志会被自动清除**（变回 false）。所以捕获异常后通常要**重新设置中断标志**或向上抛，否则上层无法感知中断：
>
> ```java
> try {
>     Thread.sleep(5000);
> } catch (InterruptedException e) {
>     // 异常被抛出时，中断标志已被清除
>     Thread.currentThread().interrupt(); // ✅ 重新设置，让上层能检测到
> }
> ```

### 7. 优先级

```java
t.setPriority(Thread.MAX_PRIORITY); // 10，最大
// 范围：Thread.MIN_PRIORITY(1) ~ Thread.MAX_PRIORITY(10)，默认 NORM_PRIORITY(5)
```

> ⚠️ **注意：** 优先级**只是给调度器一个提示**，**不保证**高优先级线程一定先执行或执行更多。不能依赖优先级来保证执行顺序。

### 8. 守护线程（Daemon）

为**用户线程**服务的后台线程。**当所有用户线程都结束时，守护线程会随之被强制结束**（最典型的守护线程就是 GC 垃圾回收线程）。

```java
Thread t = new Thread(() -> {
    while (true) {
        // 后台任务，如心跳、监控
    }
});
t.setDaemon(true); // ✅ 必须在 start() 之前设置
t.start();
```

> ⚠️ **注意：** `setDaemon(true)` **必须在 `start()` 之前调用**，否则抛 `IllegalThreadStateException`。

### 9. sleep() vs wait() 的区别（高频考点）

| 对比项       | `sleep()`                   | `wait()`                          |
| ------------ | --------------------------- | --------------------------------- |
| **所属类**   | `Thread`（静态方法）        | `Object`（每个对象都有）          |
| **是否释放锁**| **不释放锁**                | **释放锁**                        |
| **使用位置** | 任意位置                    | 必须在 `synchronized` 块内        |
| **唤醒方式** | 时间到自动唤醒              | `notify()`/`notifyAll()` 或超时   |
| **用途**     | 让线程"睡一会"              | 线程间协作（等某个条件）          |

```java
// ✅ wait/notify 必须在 synchronized 内，且锁对象要和调用对象一致
synchronized (lock) {
    while (!condition) {
        lock.wait(); // 释放 lock 的锁，进入等待；被唤醒后重新获取锁再继续
    }
    // 条件满足，执行逻辑
}
```

## 六、线程的生命周期及状态、调度

### 1. 六种状态

Java 把线程的状态定义在 `java.lang.Thread.State` 枚举中，**共 6 种**。注意：Java **没有单独的"运行中（Running）"状态**，RUNNABLE 已经包含了"就绪"和"运行"两种情况。

```
                          ┌───────────┐
                          │    NEW    │  new Thread() 创建但未 start()
                          └─────┬─────┘
                                │ start()
                                ▼
                          ┌───────────┐
                ┌────────▶│ RUNNABLE  │◀─────────┐
                │         │(就绪+运行) │           │
                │         └──┬──┬──┬──┘           │
                │  wait()    │  │  │  时间到/notify
                │            ▼  │  │              │
        ┌───────┴────┐  ┌──────┐ │  ┌────────────┴──┐
        │  WAITING   │  │BLOCKED│ │  │ TIMED_WAITING │
        │ (无限等待) │  │(阻塞) │ │  │  (限时等待)   │
        └────────────┘  └──────┘ │  └───────────────┘
                                 │
                                 │ run() 正常结束/抛异常
                                 ▼
                          ┌───────────┐
                          │TERMINATED │  终止，不可再启动
                          └───────────┘
```

### 2. 各状态详解

| 状态            | 说明                                              | 触发条件                                       |
| --------------- | ------------------------------------------------- | ---------------------------------------------- |
| `NEW`           | 已创建但**未启动**                                | `new Thread()` 后、未调 `start()`              |
| `RUNNABLE`      | **可运行**（含 OS 层的就绪和运行）                | 调用了 `start()`                               |
| `BLOCKED`       | 阻塞，等待 **`synchronized` 的 monitor 锁**       | 进入 `synchronized` 块/方法但没抢到锁          |
| `WAITING`       | **无限等待**，需被其他线程显式唤醒                | `wait()`、`join()`、`LockSupport.park()`       |
| `TIMED_WAITING` | **限时等待**，时间到自动唤醒                      | `sleep(ms)`、`wait(ms)`、`join(ms)`            |
| `TERMINATED`    | 终止                                              | `run()` 执行完毕或抛出未捕获异常               |

### 3. 状态转换的关键规律

- **`start()` 之前**：`NEW`，调用后进入 `RUNNABLE`。
- **抢 `synchronized` 锁失败**：进入 `BLOCKED`；抢到锁回到 `RUNNABLE`。
- **`Lock`（如 `ReentrantLock`）抢锁失败**：进入的是 `WAITING`（用 `LockSupport.park()` 实现），**而不是** `BLOCKED`——这是 `synchronized` 和 `Lock` 在状态上的一个区别。
- **`sleep` / `wait(ms)` / `join(ms)`**：进入 `TIMED_WAITING`。
- **传统阻塞式 IO（如 `Socket.read()`）**：在**操作系统层面**线程被挂起，但 JVM **感知不到**这种挂起，所以 Java 层面仍显示 `RUNNABLE`。
- **`run()` 结束**：进入 `TERMINATED`，**不可重新 `start()`**。

> ⚠️ **注意（修正常见误区）：** `Thread.sleep()` 会让线程进入 **`TIMED_WAITING`**（不是 RUNNABLE）。只有**传统阻塞式 IO** 在 Java 层面才显示 RUNNABLE。

### 4. 线程是如何被调度的（重点）

> 简单说：**Java 线程什么时候执行、执行多久，由操作系统决定，程序几乎无法精确控制。**

Java 线程是 1:1 映射到操作系统线程的，所以**线程调度最终交给操作系统（OS）的线程调度器**。主流操作系统（包括 Windows、Linux）对普通线程采用的是**抢占式调度 + 时间片轮转**：

**① 抢占式调度（Preemptive）**

一个线程不会一直独占 CPU。当发生以下情况时，它会被"抢占"（强制暂停）：

- 分给它的**时间片用完**；
- 有**更高优先级**的线程进入就绪状态；
- 线程自己**阻塞**（如 IO、`sleep`、等锁）或主动 `yield()`。

**② 时间片轮转（Round-Robin）**

每个处于就绪状态的线程会被分到一个**时间片（通常几毫秒）**。用完就轮到下一个就绪线程，大家**轮流**上 CPU。切换极快（微秒级），人感觉所有线程"同时"在跑——这正是**单核也能并发**的原理。

**③ 优先级（Priority）只是建议**

```java
thread.setPriority(Thread.MAX_PRIORITY); // 10
```

优先级范围 `1（MIN）~ 10（MAX）`，默认 `5（NORM）`。**高优先级只是"更有可能"被优先调度**，但**不保证**——具体仍由 OS 决定，且不同操作系统对 Java 优先级的映射不同。

**④ 结论与启示**

- **执行顺序不可预测**：多个线程谁先跑、跑到哪被切换，是 OS 安排的，每次运行结果可能不同（这也是线程安全问题难以复现的原因）。
- **不要依赖"时序假设"**：比如不能假设"先 `start()` 的线程一定先执行完"。
- **`yield()` 不可靠**：它只是"提示"调度器让出 CPU，调度器可以**忽略**这个提示。
- **想让线程协作，要用同步机制**（`join`、`wait/notify`、`CountDownLatch`、锁等），而不是寄希望于"它自然会按顺序执行"。

> 💡 **一句话总结：** 线程状态是**程序视角的快照**（`Thread.getState()`），线程调度是**操作系统视角的安排**——程序只能影响状态，调度由 OS 说了算。

## 七、线程安全问题与同步

### 1. 线程安全三大特性

| 特性   | 含义                                             | 破坏示例              |
| ------ | ------------------------------------------------ | --------------------- |
| 原子性 | 操作不可分割，要么全执行要么不执行               | `i++`（非原子操作）   |
| 可见性 | 一个线程对共享变量的修改，其他线程能立即看到     | 工作内存/主内存不一致 |
| 有序性 | 程序执行顺序符合预期（防止指令重排序）           | 双重检查单例的坑      |

经典的**卖票问题**（多线程下 `ticket--` 不原子）：

```java
// ❌ 线程不安全：多个线程同时操作 ticket 可能出现负数或重复
class Ticket implements Runnable {
    private int ticket = 100;
    public void run() {
        while (ticket > 0) {
            try { Thread.sleep(10); } catch (InterruptedException e) {}
            System.out.println(Thread.currentThread().getName() + " 卖出 " + ticket--);
        }
    }
}
```

### 2. synchronized 关键字

`synchronized` 是 JVM 层面的锁，可修饰方法或代码块。

```java
class Ticket implements Runnable {
    private int ticket = 100;
    private final Object lock = new Object();

    public void run() {
        while (true) {
            // ✅ 同步代码块：锁指定对象
            synchronized (lock) {
                if (ticket <= 0) break;
                System.out.println(Thread.currentThread().getName()
                        + " 卖出 " + ticket--);
            }
        }
    }

    // ✅ 同步方法：锁的是 this（当前对象）
    public synchronized void sell() { /* ... */ }

    // ✅ 静态同步方法：锁的是 Class 对象（Ticket.class）
    public static synchronized void staticSell() { /* ... */ }
}
```

`synchronized` 锁的对象：

| 使用形式         | 锁对象                  |
| ---------------- | ----------------------- |
| 同步代码块       | 括号里指定的对象        |
| 普通同步方法     | `this`（当前实例）      |
| 静态同步方法     | 对应的 `Class` 对象     |

### 3. Lock / ReentrantLock

`ReentrantLock` 是 `java.util.concurrent.locks` 提供的显式锁。

```java
import java.util.concurrent.locks.ReentrantLock;

class Ticket implements Runnable {
    private int ticket = 100;
    private final ReentrantLock lock = new ReentrantLock(true); // true 表示公平锁

    public void run() {
        while (true) {
            lock.lock(); // 加锁
            try {
                if (ticket <= 0) break;
                System.out.println(Thread.currentThread().getName()
                        + " 卖出 " + ticket--);
            } finally {
                lock.unlock(); // ✅ 必须在 finally 中释放锁
            }
        }
    }
}
```

常用方法：

| 方法                  | 作用                                       |
| --------------------- | ------------------------------------------ |
| `lock()`              | 获取锁（阻塞等待）                         |
| `unlock()`            | 释放锁                                     |
| `tryLock()`           | 尝试获取锁，立即返回成功与否               |
| `tryLock(time, unit)` | 限时尝试获取锁                             |
| `lockInterruptibly()` | 可被中断地获取锁                           |
| `newCondition()`      | 获取条件变量，替代 `wait/notify`           |

> ⚠️ **注意：** `lock()` 之后必须 `try-finally` 中 `unlock()`，否则一旦异常锁不会被释放导致死锁。

### 4. synchronized vs Lock 对比

| 对比项       | synchronized              | ReentrantLock                       |
| ------------ | ------------------------- | ----------------------------------- |
| 实现         | JVM 关键字（隐式）        | API 类（显式 `lock/unlock`）        |
| 锁释放       | 自动（出代码块/异常）     | 手动（必须 `finally unlock`）       |
| 响应中断     | 否                        | `lockInterruptibly()` 可响应         |
| 超时获取     | 否                        | `tryLock(time)` 支持                |
| 公平锁       | 只能非公平                | 可配置公平/非公平                    |
| 条件变量     | `wait/notify`（单条件）   | 多个 `Condition`                    |
| 性能         | JDK 6 后优化，差距很小    | 灵活性高                            |

## 八、volatile 关键字

`volatile` 是轻量级同步机制，主要解决**可见性**和**有序性**。

### 1. 两大作用

| 特性     | volatile 是否保证 | 原理                                     |
| -------- | ----------------- | ---------------------------------------- |
| 可见性   | ✅ 保证           | 强制读写主内存，禁用 CPU 缓存            |
| 有序性   | ✅ 保证           | 插入内存屏障，禁止指令重排序             |
| 原子性   | ❌ 不保证         | `i++` 仍是非原子操作                     |

### 2. 典型应用：状态标志位

```java
// ✅ 用 volatile 做停止标志，保证可见性
class Worker implements Runnable {
    private volatile boolean running = true;

    public void run() {
        while (running) {
            // 持续工作
        }
        System.out.println("线程结束");
    }

    public void shutdown() {
        running = false; // 其他线程修改后，工作线程能立即看到
    }
}
```

### 3. 不保证原子性

```java
// ❌ volatile 不能让 i++ 变成原子操作
private volatile int count = 0;
count++; // 实际是 读取-修改-写入 三步，并发下仍出错

// ✅ 需要原子性请用 AtomicInteger 或 synchronized
private final AtomicInteger count = new AtomicInteger();
count.incrementAndGet();
```

### volatile vs synchronized

| 对比项     | volatile            | synchronized            |
| ---------- | ------------------- | ----------------------- |
| 原子性     | 不保证              | 保证                    |
| 可见性     | 保证                | 保证                    |
| 有序性     | 保证                | 保证                    |
| 是否阻塞   | 否                  | 是                      |
| 性能       | 轻量                | 相对较重                |
| 适用场景   | 状态标志、单次读写 | 复合操作临界区           |

## 九、线程池（重点中的重点）

### 1. 为什么用线程池

- **降低资源消耗**：复用已创建的线程，避免频繁创建销毁。
- **提高响应速度**：任务到达时无需等待线程创建。
- **便于统一管理**：统一调优、监控、限流。
- **避免无限创建线程**：防止线程过多导致 OOM 或系统崩溃。

### 2. ThreadPoolExecutor 七大参数

```java
public ThreadPoolExecutor(
    int corePoolSize,                      // 1. 核心线程数
    int maximumPoolSize,                   // 2. 最大线程数
    long keepAliveTime,                    // 3. 非核心线程空闲存活时间
    TimeUnit unit,                         // 4. 时间单位
    BlockingQueue<Runnable> workQueue,     // 5. 任务队列
    ThreadFactory threadFactory,           // 6. 线程工厂
    RejectedExecutionHandler handler       // 7. 拒绝策略
)
```

| 参数          | 说明                                                             |
| ------------- | ---------------------------------------------------------------- |
| corePoolSize  | 核心线程数，即使空闲也保留（除非设置 `allowCoreThreadTimeOut`）  |
| maximumPoolSize | 线程池最大线程数                                                |
| keepAliveTime | 非核心线程空闲后的存活时间                                       |
| unit          | keepAliveTime 的单位（`TimeUnit.SECONDS` 等）                    |
| workQueue     | 阻塞队列，如 `LinkedBlockingQueue`、`ArrayBlockingQueue`、`SynchronousQueue` |
| threadFactory | 创建线程的工厂（可自定义线程名）                                 |
| handler       | 拒绝策略（任务太多处理不过来时）                                 |

### 3. 线程池工作流程

```
                 提交任务 (execute)
                       │
                       ▼
            ┌──────────────────────┐
            │ 当前线程数 < 核心线程数？│
            └──────────┬───────────┘
                   是  │  否
           ┌───────────┘  └────────────┐
           ▼                            ▼
   创建核心线程执行任务          ┌─────────────────┐
                                │ 队列是否已满？    │
                                └────────┬────────┘
                                    否  │   是
                            ┌───────────┘   └─────────────┐
                            ▼                             ▼
                       放入队列等待           ┌─────────────────────┐
                                              │ 当前线程 < 最大线程数？│
                                              └──────────┬──────────┘
                                                    是   │    否
                                        ┌──────────────┘  └─────────────┐
                                        ▼                                ▼
                                创建非核心线程执行                  执行拒绝策略
```

**核心流程口诀**：核心线程 → 队列 → 非核心线程 → 拒绝策略。

### 4. 四种拒绝策略

| 策略                | 行为                                       |
| ------------------- | ------------------------------------------ |
| `AbortPolicy`（默认）| 抛出 `RejectedExecutionException`          |
| `CallerRunsPolicy`  | 由提交任务的线程自己执行该任务              |
| `DiscardPolicy`     | 静默丢弃新任务                              |
| `DiscardOldestPolicy`| 丢弃队列中最旧的任务，再尝试提交新任务     |

### 5. Executors 工厂方法为何不推荐

阿里《Java 开发手册》强制规定**禁用 `Executors` 创建线程池**，原因如下：

| Executors 方法            | 实际配置                                   | 风险        |
| ------------------------- | ------------------------------------------ | ----------- |
| `newFixedThreadPool`      | 队列用 `LinkedBlockingQueue`（无界）       | OOM（队列） |
| `newSingleThreadExecutor` | 同上，无界队列                             | OOM（队列） |
| `newCachedThreadPool`     | 最大线程数 `Integer.MAX_VALUE`             | OOM（线程） |
| `newScheduledThreadPool`  | 队列无界                                   | OOM（队列） |

> ⚠️ **注意：** 应使用 `new ThreadPoolExecutor(...)` 手动指定参数，便于控制资源和排查问题。

### 6. 手写 ThreadPoolExecutor 完整示例

```java
import java.util.concurrent.*;

public class PoolDemo {
    public static void main(String[] args) {
        ThreadPoolExecutor pool = new ThreadPoolExecutor(
            2,                              // 核心线程数
            4,                              // 最大线程数
            30, TimeUnit.SECONDS,           // 非核心线程空闲存活时间
            new ArrayBlockingQueue<>(10),   // 有界任务队列
            Executors.defaultThreadFactory(),// 默认线程工厂
            new ThreadPoolExecutor.AbortPolicy() // 拒绝策略
        );

        // 提交 Runnable（无返回值）
        pool.execute(() -> System.out.println("执行任务"));

        // 提交 Callable（有返回值，返回 Future）
        Future<String> future = pool.submit(() -> {
            Thread.sleep(500);
            return "任务结果";
        });
        try {
            System.out.println(future.get());
        } catch (Exception e) {
            e.printStackTrace();
        }

        pool.shutdown(); // 关闭线程池（不再接受新任务，已提交任务执行完）
    }
}
```

> 💡 **提示：** 线程池关闭用 `shutdown()`（平缓关闭）或 `shutdownNow()`（立即停止）。生产环境线程池一般随 JVM 长期存活，不关闭。

### CPU 密集型 vs IO 密集型线程数

| 任务类型   | 推荐核心线程数              | 说明                         |
| ---------- | --------------------------- | ---------------------------- |
| CPU 密集型 | `N + 1`（N = CPU 核数）     | 线程过多反而频繁切换         |
| IO 密集型  | `2N` 或更多                 | 线程多在等待 IO，可多开      |

## 十、常用并发工具类

### 1. CountDownLatch（一次性倒计时）

让一个或多个线程等待，直到其他线程执行完毕。计数器**不可重置**。

```java
import java.util.concurrent.CountDownLatch;

public class Demo {
    public static void main(String[] args) throws InterruptedException {
        final int N = 3;
        CountDownLatch latch = new CountDownLatch(N); // 计数 3

        for (int i = 0; i < N; i++) {
            new Thread(() -> {
                System.out.println(Thread.currentThread().getName() + " 完成");
                latch.countDown(); // 计数减 1
            }).start();
        }

        latch.await(); // 等待计数归零
        System.out.println("所有子任务完成，继续主流程");
    }
}
```

### 2. CyclicBarrier（可复用屏障）

让一组线程相互等待，到达屏障点后一起继续执行，**可重复使用**。

```java
import java.util.concurrent.CyclicBarrier;

public class Demo {
    public static void main(String[] args) {
        CyclicBarrier barrier = new CyclicBarrier(3, () ->
                System.out.println("全部到达屏障，开始下一阶段"));

        for (int i = 0; i < 3; i++) {
            new Thread(() -> {
                System.out.println(Thread.currentThread().getName() + " 到达");
                try { barrier.await(); } catch (Exception e) {}
            }).start();
        }
    }
}
```

### 3. Semaphore（信号量，限流）

控制同时访问某资源的线程数量。

```java
import java.util.concurrent.Semaphore;

public class Demo {
    public static void main(String[] args) {
        Semaphore semaphore = new Semaphore(3); // 只允许 3 个线程同时访问

        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                try {
                    semaphore.acquire(); // 获取许可
                    System.out.println(Thread.currentThread().getName() + " 访问资源");
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                } finally {
                    semaphore.release(); // 释放许可
                }
            }).start();
        }
    }
}
```

### 4. ConcurrentHashMap

线程安全的 HashMap，通过分段锁（JDK 7）/ CAS + synchronized（JDK 8）保证并发安全，性能远优于 `Hashtable` 和 `Collections.synchronizedMap`。

### CountDownLatch vs CyclicBarrier

| 对比项    | CountDownLatch          | CyclicBarrier           |
| --------- | ----------------------- | ----------------------- |
| 可复用    | 否                      | 是                      |
| 计数方式  | countDown() 递减        | await() 等待            |
| 主语      | 一个线程等 N 个         | N 个线程互相等          |
| 侧重点    | 等待任务完成            | 线程同步到某一点        |

## 十一、原子类与 CAS

### 1. 原子类

`java.util.concurrent.atomic` 包提供基于 CAS 的原子类，无需加锁即可实现原子操作。

| 类型           | 示例                  | 用途                  |
| -------------- | --------------------- | --------------------- |
| 基本类型       | `AtomicInteger`       | 原子 int              |
| 基本类型       | `AtomicLong`          | 原子 long             |
| 基本类型       | `AtomicBoolean`       | 原子 boolean          |
| 引用类型       | `AtomicReference`     | 原子对象引用          |
| 数组类型       | `AtomicIntegerArray`  | 原子数组              |
| 字段更新器     | `AtomicIntegerFieldUpdater` | 原子更新某字段   |
| 累加器（JDK 8）| `LongAdder`           | 高并发计数（优于 AtomicLong） |

```java
import java.util.concurrent.atomic.AtomicInteger;

AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet(); // 等价于 ++count，原子操作
count.getAndIncrement(); // 等价于 count++
count.compareAndSet(0, 1); // CAS：期望值为 0 则设为 1
```

### 2. CAS 原理

**CAS (Compare And Swap)**：比较并交换，是一种**乐观锁**思想。

```
CAS(V, expected, newValue):
  如果 V 的当前值 == expected:
      将 V 更新为 newValue，返回 true
  否则:
      返回 false（说明已被其他线程修改，重试）
```

整个操作由 CPU 的 `cmpxchg` 指令保证原子性。`AtomicInteger` 内部用 `Unsafe` 的 `compareAndSwapInt` 实现。

### 3. ABA 问题

一个值从 A → B → A，CAS 检查时以为没变过，实际上已被修改过。

- **解决**：用 `AtomicStampedReference`，给值加版本号（`A1 → B2 → A3`，CAS 比较时连版本号一起比较）。

> 💡 **提示：** CAS 的缺点：自旋开销大（竞争激烈时）；只能保证单个变量的原子性。

## 十二、ThreadLocal

### 1. 是什么

`ThreadLocal` 为每个使用该变量的线程提供**独立的副本**，实现线程间数据隔离。

```
                ThreadLocal<T> tl = new ThreadLocal<>();

  线程 A                线程 B              线程 C
   │                     │                   │
   ▼                     ▼                   ▼
┌─────────┐         ┌─────────┐         ┌─────────┐
│ tl: "A" │         │ tl: "B" │         │ tl: "C" │   各自独立副本
└─────────┘         └─────────┘         └─────────┘
```

### 2. 典型应用

- **数据库连接 / 事务**：每个线程一个 Connection，避免争用。
- **SimpleDateFormat**：非线程安全，用 ThreadLocal 给每个线程一份。
- **用户上下文**：保存当前请求的登录用户信息。

```java
public class Context {
    // ✅ 每个线程一份 SimpleDateFormat，避免线程安全问题
    private static final ThreadLocal<SimpleDateFormat> dateFormat =
            ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

    public static String format(Date date) {
        return dateFormat.get().format(date);
    }

    // 用户上下文示例
    private static final ThreadLocal<User> currentUser = new ThreadLocal<>();
    public static void set(User u) { currentUser.set(u); }
    public static User get() { return currentUser.get(); }
    public static void clear() { currentUser.remove(); } // ✅ 用完务必 remove
}
```

### 3. 内存泄漏注意

`ThreadLocal` 内部通过 `ThreadLocalMap`（key 为弱引用，value 为强引用）存储。线程长期存活时（如线程池），若不 `remove()`，value 可能无法回收导致**内存泄漏**。

> ⚠️ **注意：** 在线程池环境下，使用完 `ThreadLocal` **必须在 `finally` 中调用 `remove()`** 清理。

## 十三、死锁

### 1. 死锁的四个必要条件

| 条件         | 含义                                       |
| ------------ | ------------------------------------------ |
| 互斥         | 资源同一时刻只能被一个线程占有             |
| 占有并等待   | 持有资源的同时等待新的资源                 |
| 不可剥夺     | 不能强行夺走线程已占有的资源               |
| 循环等待     | 多个线程形成首尾相接的等待链               |

### 2. 死锁示例

```java
public class DeadLockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        // ❌ 线程 1：先锁 A 再锁 B
        new Thread(() -> {
            synchronized (lockA) {
                System.out.println("T1 持有 A，等待 B");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lockB) {
                    System.out.println("T1 获得 B");
                }
            }
        }, "T1").start();

        // ❌ 线程 2：先锁 B 再锁 A，形成循环等待
        new Thread(() -> {
            synchronized (lockB) {
                System.out.println("T2 持有 B，等待 A");
                try { Thread.sleep(100); } catch (Exception e) {}
                synchronized (lockA) {
                    System.out.println("T2 获得 A");
                }
            }
        }, "T2").start();
    }
}
```

### 3. 如何避免

| 方法                 | 说明                                                     |
| -------------------- | -------------------------------------------------------- |
| 固定加锁顺序         | 所有线程按相同顺序获取多把锁（最常用）                   |
| 设置超时             | 用 `tryLock(time)`，超时放弃，避免无限等待               |
| 一次性获取所有锁     | 用 `tryLock` 同时尝试获取，失败则全部释放重试            |
| 减小锁粒度           | 缩小同步范围，缩短持锁时间                               |

> 💡 **提示：** 排查死锁可用 `jstack` 打印线程堆栈，或用 jconsole/jvisualvm 的死锁检测功能。

## 十四、面试常见问题

### Q1：创建线程有几种方式？哪种推荐？

四种：继承 `Thread`、实现 `Runnable`、实现 `Callable`、线程池。本质都是把"任务"交给线程（真正创建线程只有 `new Thread().start()`）。推荐 `Runnable`/`Callable` 配合线程池，因为：避免单继承局限、便于资源共享、方便统一管理和复用。

### Q2：start() 和 run() 的区别？

`start()` 启动一个新线程，由 JVM 在新线程中回调 `run()`；直接调用 `run()` 只是普通方法调用，不开启新线程。一个 Thread 只能 `start()` 一次，重复调用抛 `IllegalThreadStateException`。

### Q3：sleep() 和 wait() 的区别？

- `sleep()` 是 `Thread` 静态方法，**不释放锁**；`wait()` 是 `Object` 方法，**释放锁**。
- `wait()` 必须在 `synchronized` 块内调用，由 `notify/notifyAll` 唤醒或超时。
- `sleep()` 可在任意位置调用，时间到自动唤醒。

### Q4：synchronized 和 Lock 的区别？

| 维度     | synchronized       | Lock              |
| -------- | ------------------ | ----------------- |
| 释放     | 自动               | 手动 finally 释放 |
| 中断     | 不响应             | 可响应（`lockInterruptibly`）|
| 超时     | 不支持             | 支持（`tryLock(time)`）|
| 公平     | 非公平             | 可配置            |
| 条件     | 单一 wait/notify   | 多 Condition      |

### Q5：volatile 的作用？

保证**可见性**（强制读写主存）和**有序性**（禁止指令重排序），但**不保证原子性**。常用于状态标志位。需要原子性时配合 `AtomicXxx` 或 `synchronized`。

### Q6：线程有哪些状态？如何调度？

6 种状态：`NEW`、`RUNNABLE`（含就绪和运行）、`BLOCKED`、`WAITING`、`TIMED_WAITING`、`TERMINATED`。

调度由操作系统完成，采用**抢占式 + 时间片轮转**：每个就绪线程分到一个时间片，用完轮转；高优先级线程"更可能"先执行但不保证；线程何时被切换由 OS 决定，程序无法精确控制，因此不能用"执行顺序假设"来保证线程协作，必须使用同步机制。

### Q7：线程池有哪些核心参数？工作流程是怎样的？

七大参数：corePoolSize、maximumPoolSize、keepAliveTime、unit、workQueue、threadFactory、handler。

工作流程：提交任务时，若当前线程数 < corePoolSize，创建核心线程；否则尝试放入队列；队列满后，若线程数 < maximumPoolSize，创建非核心线程；若已达最大线程数，执行拒绝策略。

### Q8：线程池的拒绝策略有哪些？

四种：`AbortPolicy`（默认，抛异常）、`CallerRunsPolicy`（调用者线程执行）、`DiscardPolicy`（静默丢弃）、`DiscardOldestPolicy`（丢弃最旧任务再提交）。

### Q9：为什么不推荐用 Executors 创建线程池？

`newFixedThreadPool` 和 `newSingleThreadExecutor` 用无界队列，可能 OOM；`newCachedThreadPool` 最大线程数为 `Integer.MAX_VALUE`，可能因创建过多线程 OOM。应手动用 `ThreadPoolExecutor` 指定参数。

### Q10：什么是死锁？如何避免？

多个线程互相持有对方需要的锁，导致永久阻塞。四个必要条件：互斥、占有并等待、不可剥夺、循环等待。避免方法：固定加锁顺序、使用 `tryLock` 设置超时、减小锁粒度。

### Q11：CAS 是什么？有什么问题？

CAS (Compare And Swap) 是乐观锁机制，比较当前值与期望值，相等则更新，由 CPU 指令保证原子性。缺点：竞争激烈时自旋开销大；存在 ABA 问题（用 `AtomicStampedReference` 加版本号解决）。

## 十五、总结

### 核心知识点回顾

| 知识点          | 关键内容                                                 |
| --------------- | -------------------------------------------------------- |
| 进程 vs 线程    | 进程是资源单位，线程是 CPU 调度单位；线程共享进程内存    |
| 并发 vs 并行    | 并发是"同时处理"，并行是"同时执行"（需多核）            |
| 线程创建        | Thread / Runnable / Callable + FutureTask / 线程池       |
| start vs run    | start 开新线程，run 普通调用                              |
| 线程状态        | NEW / RUNNABLE / BLOCKED / WAITING / TIMED_WAITING / TERMINATED |
| 线程调度        | OS 抢占式 + 时间片轮转，优先级不保证                     |
| 同步机制        | synchronized（JVM）vs ReentrantLock（API）               |
| volatile        | 保证可见性、有序性，不保证原子性                         |
| 线程池          | 七大参数 + 核心流程（核心→队列→非核心→拒绝）            |
| 拒绝策略        | Abort / CallerRuns / Discard / DiscardOldest             |
| 并发工具        | CountDownLatch / CyclicBarrier / Semaphore               |
| 原子类 / CAS    | AtomicInteger 等，CAS 乐观锁，ABA 问题                   |
| ThreadLocal     | 线程隔离，注意 remove 防内存泄漏                          |
| 死锁            | 四个必要条件，固定加锁顺序避免                            |

### 三大特性 vs 对应工具

| 特性   | 保证手段                                |
| ------ | --------------------------------------- |
| 原子性 | `synchronized`、`Lock`、`AtomicXxx`     |
| 可见性 | `volatile`、`synchronized`、`final`     |
| 有序性 | `volatile`、`synchronized`、`happens-before` |

> 💡 **提示：** 多线程是 Java 面试的重中之重，掌握以上内容能覆盖绝大部分考点。重点吃透：线程池参数与流程、synchronized 与 Lock、volatile 三大特性、线程状态与调度、死锁。
