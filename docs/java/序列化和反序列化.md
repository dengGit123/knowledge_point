# 序列化与反序列化（Serialization）

> 官方文档：[Java Serialization (Oracle)](https://docs.oracle.com/javase/tutorial/jndi/objects/serial.html)、[Serializable 接口 API](https://docs.oracle.com/javase/8/docs/api/java/io/Serializable.html)、[菜鸟教程 Java 序列化](https://www.runoob.com/java/java-serialization.html)

## 一、概述

### 1.1 什么是序列化与反序列化

- **序列化（Serialization）**：把 Java **对象**转换成**字节序列**（一串 byte），方便存到磁盘、或在网络中传输。
- **反序列化（Deserialization）**：序列化的逆过程，把**字节序列**重新还原成 Java **对象**。

```
        序列化                                  反序列化
┌──────────────┐  ──────────►  ┌──────────────┐  ──────────►  ┌──────────────┐
│  Java 对象    │              │  字节序列      │              │  Java 对象    │
│  new User()  │              │  byte[] / 文件 │              │  和原来一样   │
└──────────────┘              └──────────────┘              └──────────────┘
   内存里的对象              可传输 / 可存储的形式              换一台机器还原对象
```

> 💡 **类比：** 序列化就像"**打包快递**"——把一个立体的对象拆解、压扁装进箱子（字节流），方便运输；反序列化就是"**拆箱组装**"，到目的地后按原样把它拼回来。

### 1.2 为什么需要序列化

Java 对象存在于 **JVM 内存**里，JVM 一关就没了。但很多场景需要把对象"搬走"或"存起来"：

| 场景 | 说明 | 例子 |
|-----|------|------|
| **网络传输** | 对象要从一个 JVM 传到另一个 JVM | RPC 调用、RMI、消息队列 |
| **持久化存储** | 把对象存到磁盘，下次再读回来 | Session 持久化、缓存落盘 |
| **缓存** | 对象存到 Redis 等外部缓存 | 分布式缓存共享对象 |
| **深拷贝** | 通过"序列化→反序列化"复制一个全新对象 | 工具类深拷贝 |
| **进程间通信** | Android 的 Intent、Binder 传对象 | 跨进程传递 |

```java
// 场景：调用方 A 想把 user 对象传给服务方 B（两台不同机器）
// ✅ 先序列化成字节流 → 网络发送 → B 收到后反序列化还原对象
User user = new User("张三", 20);
byte[] bytes = serialize(user);        // A 端序列化
network.send(bytes);                    // 网络传输（传不了对象，只能传字节）
User u2 = deserialize(network.recv()); // B 端反序列化
```

> 💡 **提示：** 网络、磁盘只能传输/存储**字节**，传不了 Java 对象。所以"对象要离开 JVM"就必须先序列化成字节。

### 1.3 整体结构

```
                         序列化体系
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
  Java 原生序列化        第三方序列化框架        跨语言格式
        │                   │                   │
  Serializable 接口     Kryo / Hessian        JSON
  Externalizable 接口   Protobuf / Avro       MessagePack
  ObjectOutputStream    FST / FST             XML
  ObjectInputStream
```

本文先讲 **Java 原生序列化**（面试重点、理解原理的基础），再讲它的**缺点和替代方案**（实际开发主流）。

## 二、Java 原生序列化：基本用法

Java 原生序列化的核心是 `java.io.Serializable` 接口和两个流类：`ObjectOutputStream`（序列化）、`ObjectInputStream`（反序列化）。

### 2.1 第一步：实现 Serializable 接口

```java
import java.io.Serializable;

// ✅ 想要可序列化，实现 Serializable 接口即可
public class User implements Serializable {

    private static final long serialVersionUID = 1L;   // 版本号（很重要，后面讲）

    private String name;
    private int age;

    public User() {}

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    @Override
    public String toString() {
        return "User{name='" + name + "', age=" + age + "}";
    }

    // 省略 getter/setter
}
```

> 💡 **关键点：** `Serializable` 是一个**标记接口（Marker Interface）**——它**没有任何方法**，只是给类贴上一个"我可序列化"的标签。JVM 看到这个标签才允许序列化这个对象。

### 2.2 第二步：序列化对象（写出去）

用 `ObjectOutputStream` 把对象转成字节流，写到文件或字节数组：

```java
import java.io.*;

public class SerializeDemo {
    public static void main(String[] args) throws IOException {
        User user = new User("张三", 20);

        // ✅ 序列化到文件
        try (ObjectOutputStream oos = new ObjectOutputStream(
                new FileOutputStream("user.dat"))) {
            oos.writeObject(user);   // 把对象写出去
        }
        System.out.println("序列化成功，已写入 user.dat");

        // ✅ 序列化到字节数组（内存中，常用于网络传输）
        ByteArrayOutputStream baos = new ByteArrayOutputStream();
        try (ObjectOutputStream oos = new ObjectOutputStream(baos)) {
            oos.writeObject(user);
        }
        byte[] bytes = baos.toByteArray();   // 得到字节流
    }
}
```

> ⚠️ **注意：** 没实现 `Serializable` 的对象调用 `writeObject` 会抛出 `NotSerializableException`。

### 2.3 第三步：反序列化（读回来）

用 `ObjectInputStream` 把字节流还原成对象：

```java
import java.io.*;

public class DeserializeDemo {
    public static void main(String[] args) throws IOException, ClassNotFoundException {
        // ✅ 从文件反序列化
        try (ObjectInputStream ois = new ObjectInputStream(
                new FileInputStream("user.dat"))) {
            User user = (User) ois.readObject();   // 读回对象，要强转
            System.out.println("反序列化得到：" + user);
            // 输出：反序列化得到：User{name='张三', age=20}
        }
    }
}
```

> ⚠️ **注意：**
> - `readObject()` 返回 `Object`，必须**强制类型转换**成目标类型。
> - 它抛出受检异常 `ClassNotFoundException`（字节流里记录的类找不到时）和 `IOException`，必须处理。

### 2.4 完整流程图

```
  序列化端                                         反序列化端
┌────────────┐   writeObject()   ┌──────────┐   readObject()   ┌────────────┐
│ User 对象   │ ───────────────► │ 字节流     │ ──────────────► │ User 对象   │
│ 张三/20    │                  │ 010101... │                 │ 张三/20    │
└────────────┘                  └──────────┘                  └────────────┘
   实现了                                     文件/网络/内存         还原成功 ✅
  Serializable
```

## 三、serialVersionUID（版本号，必懂）

### 3.1 什么是 serialVersionUID

`serialVersionUID` 是序列化时的**版本号**，用来保证"序列化的对象"和"反序列化的类"是**兼容的**。

```java
private static final long serialVersionUID = 1L;
```

### 3.2 它的作用：版本兼容性检查

```
序列化时：字节流里会记录当时类的 serialVersionUID
反序列化时：对比"字节流里的 UID" 和 "当前类的 UID"
   ├─ 一致 → 反序列化成功 ✅
   └─ 不一致 → 抛出 InvalidClassException ❌
```

### 3.3 不显式定义会怎样？

```java
// ❌ 危险：没有显式定义 serialVersionUID
public class User implements Serializable {
    private String name;
    private int age;
}
```

如果不写 `serialVersionUID`，**编译器会根据类的结构（字段、方法等）自动计算一个**。问题是：

```
今天序列化了 User 对象存到磁盘 → serialVersionUID = 9527（自动算的）
明天你给 User 加了一个字段 email → 类结构变了 → serialVersionUID 变成 9528
后天反序列化 → 9527 ≠ 9528 → ❌ InvalidClassException，反序列化失败！
```

> ⚠️ **注意：这就是为什么必须显式定义 `serialVersionUID`。** 显式定义后，即使类结构变化（加字段、改方法），UID 不变，**旧的字节流依然能反序列化**（新增字段用默认值，兼容性好）。

### 3.4 推荐做法

```java
public class User implements Serializable {
    // ✅ 显式定义，固定不变
    private static final long serialVersionUID = 1L;

    // IDEA 可以设置提醒：有可序列化类没定义 UID 就警告
}
```

| 情况 | serialVersionUID 是否显式定义 | 后果 |
|-----|:-----------------------:|------|
| 类结构不变 | 都行 | 正常 |
| 加/删字段 | ✅ 显式定义 | **兼容**，新字段用默认值 |
| 加/删字段 | ❌ 没定义 | **不兼容**，反序列化失败 |

> 💡 **提示：** IDEA 中可以开启"Serializable class without serialVersionUID"检查，自动提示生成。**面试常问"为什么序列化要写 serialVersionUID"，答案就是上面的版本兼容。**

## 四、transient 和 static 关键字

### 4.1 transient：标记字段不参与序列化

有些字段不希望被序列化（比如密码、临时计算结果），用 `transient` 修饰：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private int age;
    transient private String password;   // 🔒 不序列化密码

    public User(String name, int age, String password) {
        this.name = name;
        this.age = age;
        this.password = password;
    }
}
```

```java
// 序列化
User user = new User("张三", 20, "123456");
serialize(user);   // password 不会被写入

// 反序列化
User u = deserialize();
System.out.println(u.password);   // ❌ null（transient 字段反序列化后是默认值）
```

### 4.2 反序列化后 transient 字段的值

`transient` 字段反序列化后是**类型的默认值**（不是原值）：

| 字段类型 | 反序列化后的值 |
|---------|--------------|
| `int` / `long` / `double` 等基本类型 | `0` |
| `boolean` | `false` |
| `String` / 对象 | `null` |

### 4.3 static：静态字段不参与序列化

**`static` 字段属于"类"不属于"对象"，所以序列化对象时，静态字段不会被序列化。**

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    public static int count = 100;   // 静态字段，不序列化
}
```

```java
// 即使改了 count，序列化/反序列化后 count 取的是【当前 JVM 中类的 count 值】
User.count = 999;
serialize(user);          // 序列化时 count 根本没写进去
User.count = 0;           // 改了
User u = deserialize();
System.out.println(User.count);   // 0（取类当前的值，跟序列化无关）
```

> 💡 **记忆：序列化的是"对象的状态"，而 static 属于"类的状态"，transient 是"明确不要"。** 两者都不参与对象序列化。

### 4.4 transient vs static

| 对比 | `transient` | `static` |
|-----|------------|---------|
| 作用对象 | 实例字段 | 静态字段 |
| 不序列化的原因 | 主动标记"不要" | 本就属于类不属于对象 |
| 反序列化后值 | 类型默认值 | 类当前的值 |

## 五、序列化的特性与坑

### 5.1 序列化不调用构造器

**反序列化时，对象的构造器不会被调用**（默认 `Serializable` 方式）——对象是直接从字节流"装配"出来的：

```java
public class User implements Serializable {
    private String name;
    public User() {
        System.out.println("构造器被调用了！");
    }
}
// 反序列化时，控制台【不会】打印"构造器被调用了！"
// 对象是绕过构造器，由 JVM 直接创建的
```

> ⚠️ **注意：** 这意味着构造器里的初始化逻辑（如默认值设置、注册）反序列化时**不会执行**，需要在 `readObject` 里手动处理。

### 5.2 同一对象多次序列化：只存一份引用

```java
User user = new User("张三", 20);
List<User> list = new ArrayList<>();
list.add(user);
list.add(user);   // 同一个对象加两次

serialize(list);   // 字节流里 user 只存了一份，第二个是引用
// ✅ 反序列化后 list 里的两个元素仍然是同一个对象（引用相等）
```

这个特性避免了循环引用导致的无限递归，也保证了对象图的一致性。

### 5.3 引用类型字段也要可序列化

```java
public class Order implements Serializable {
    private static final long serialVersionUID = 1L;
    private User user;          // ✅ User 也实现了 Serializable，没问题
    private Connection conn;    // ❌ Connection 没实现 Serializable，会报错
}
```

如果对象里引用了**不可序列化**的字段，序列化时会抛 `NotSerializableException`。解决：把该字段标为 `transient`。

### 5.4 继承中的序列化（重点）

这是面试高频考点：

```
规则：
  ① 父类实现了 Serializable → 子类自动可序列化（继承下来）
  ② 子类实现了 Serializable，但父类没有 → 父类的字段不会被序列化，
     反序列化时【父类的无参构造器会被调用】来初始化父类部分
```

```java
// 父类：没实现 Serializable
public class Animal {
    String type = "动物";          // 父类字段
    public Animal() {              // ⚠️ 父类必须有可访问的无参构造器
        System.out.println("Animal 构造器");
    }
}

// 子类：实现了 Serializable
public class Dog extends Animal implements Serializable {
    private static final long serialVersionUID = 1L;
    String name = "旺财";          // 子类字段，会被序列化
}

// 序列化 Dog
Dog d = new Dog();
serialize(d);

// 反序列化
Dog d2 = deserialize();
// 输出：Animal 构造器（父类无参构造被调用！）
// d2.type = "动物"（父类字段用构造器初始化的值，不是序列化恢复的，因为父类没序列化）
// d2.name = "旺财"（子类字段正常恢复）
```

> 💡 **提示：** 反序列化子类时，**父类（未实现 Serializable）的无参构造器一定会被调用**，所以父类**必须提供无参构造器**，否则反序列化失败。

## 六、自定义序列化

有时默认序列化不满足需求（如加密敏感字段、只序列化部分字段），可以自定义。

### 6.1 实现 writeObject / readObject

在类里定义这两个 private 方法，JVM 序列化时会**自动回调**它们（这是通过反射实现的特殊约定）：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private int age;
    transient private String password;   // 默认不序列化

    // ✅ 自定义序列化逻辑
    private void writeObject(ObjectOutputStream out) throws IOException {
        out.defaultWriteObject();          // 先做默认序列化（name、age）
        out.writeUTF(encrypt(password));   // 再手动序列化密码（加密后）
    }

    // ✅ 自定义反序列化逻辑（顺序要和 writeObject 对应）
    private void readObject(ObjectInputStream in)
            throws IOException, ClassNotFoundException {
        in.defaultReadObject();            // 先做默认反序列化
        this.password = decrypt(in.readUTF()); // 手动解密恢复密码
    }

    private String encrypt(String s) { return "ENC(" + s + ")"; }
    private String decrypt(String s) { return s.replace("ENC(", "").replace(")", ""); }
}
```

> 💡 **提示：** `writeObject`/`readObject` 必须是 **private**、**精确的签名**，否则不会被当成回调。`defaultWriteObject()`/`defaultReadObject()` 用于保留默认行为。

### 6.2 readResolve：保护单例

反序列化会**创建新对象**，这会破坏单例模式：

```java
// 单例类
public class Singleton implements Serializable {
    private static final long serialVersionUID = 1L;
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() { return INSTANCE; }
}

// ❌ 反序列化会创建一个新对象，破坏单例
Singleton s1 = Singleton.getInstance();
byte[] bytes = serialize(s1);
Singleton s2 = deserialize(bytes);
System.out.println(s1 == s2);   // false！单例被破坏
```

**解决：实现 `readResolve()` 方法，返回单例实例：**

```java
public class Singleton implements Serializable {
    private static final long serialVersionUID = 1L;
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() { return INSTANCE; }

    // ✅ 反序列化时，用这个方法返回的对象代替新创建的对象
    private Object readResolve() {
        return INSTANCE;   // 返回唯一的单例
    }
}

// 现在 s1 == s2 为 true，单例保住了 ✅
```

## 七、Externalizable 接口

`Externalizable` 是 `Serializable` 的子接口，提供**完全自定义**的序列化方式。

### 7.1 用法

```java
import java.io.*;

public class User implements Externalizable {
    private String name;
    private int age;

    // ⚠️ 必须有 public 无参构造器（反序列化时要用）
    public User() {}

    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // ✅ 完全手动控制：写什么、怎么写
    @Override
    public void writeExternal(ObjectOutput out) throws IOException {
        out.writeUTF(name);
        out.writeInt(age);
    }

    // ✅ 完全手动控制：读什么、怎么读（顺序必须和 write 对应）
    @Override
    public void readExternal(ObjectInput in) throws IOException, ClassNotFoundException {
        this.name = in.readUTF();
        this.age = in.readInt();
    }
}
```

### 7.2 Serializable vs Externalizable

| 对比 | `Serializable` | `Externalizable` |
|-----|---------------|-----------------|
| 实现难度 | 简单（标记接口即可） | 复杂（要实现两个方法） |
| 序列化内容 | 自动序列化所有非 transient/static 字段 | **完全手动**，自己决定写什么 |
| 性能 | 一般（要写类信息） | **较好**（只写你指定的） |
| 是否调用构造器 | ❌ 不调用 | ✅ 调用 public 无参构造器 |
| 安全性 | 一般 | 更可控（可加密、过滤） |
| 适用场景 | 绝大多数情况 | 需要极致性能或完全控制时 |

> 💡 **提示：** 日常用 `Serializable` 就够了。只有对性能、体积有极致要求，或需要完全控制序列化内容时，才用 `Externalizable`。

## 八、序列化的安全性问题（重要！）

### 8.1 反序列化漏洞

反序列化是**危险的**——因为你从字节流还原对象时，**不知道里面是什么**。如果攻击者构造恶意的字节流，反序列化时会**执行恶意代码**（RCE 远程代码执行）。

```
著名的反序列化漏洞：
  - Apache Commons Collections 反序列化漏洞（2015）
  - Fastjson 反序列化漏洞（多次）
  - Log4j（本质也是反序列化/JNDI 注入）

攻击原理：
  攻击者发送恶意字节流 → 服务端 readObject →
  触发某些类的特殊方法（如 readObject、compare）→ 执行任意命令
```

### 8.2 防护措施

```java
// ❌ 危险：直接反序列化不可信的数据
ObjectInputStream ois = new ObjectInputStream(untrustedInputStream);
Object obj = ois.readObject();   // 可能被攻击

// ✅ 防护1：用 ObjectInputFilter 白名单过滤（Java 9+）
ois.setObjectInputFilter(info -> {
    Class<?> clazz = info.serialClass();
    if (clazz != null && ALLOWED.contains(clazz.getName())) {
        return ObjectInputFilter.Status.ALLOWED;
    }
    return ObjectInputFilter.Status.REJECTED;
});

// ✅ 防护2：不要反序列化不可信的数据
// ✅ 防护3：用更安全的格式（如 JSON）代替 Java 序列化
// ✅ 防护4：保持依赖库更新，避免有漏洞的版本
```

> ⚠️ **注意：** **永远不要对"外部传入的不可信数据"做 Java 原生反序列化。** 这就是为什么现代系统很少用 Java 原生序列化传输数据，而是用 JSON。

## 九、为什么实际开发很少用 Java 原生序列化？

Java 原生序列化有明显的缺点：

| 缺点 | 说明 |
|-----|------|
| **跨语言差** | 字节流只有 Java 能读，Python/Go/前端读不了 |
| **体积大** | 字节流里包含大量类信息，比 JSON/Protobuf 大很多 |
| **性能差** | 反射 + 流程复杂，速度慢 |
| **不安全** | 反序列化漏洞风险 |
| **强耦合** | 类路径必须一致，改包名就反序列化失败 |

> 💡 **结论：** Java 原生序列化**理解原理即可**，实际开发（尤其是网络传输、跨语言场景）用**第三方框架**。

## 十、主流替代方案

### 10.1 各方案对比

| 方案 | 跨语言 | 体积 | 性能 | 可读性 | 适用场景 |
|-----|:------:|:----:|:----:|:------:|---------|
| **JSON**（Jackson/Gson） | ✅ | 中 | 中 | ✅ 人可读 | **Web API、前后端交互（最常用）** |
| **Protobuf**（Google） | ✅ | **小** | **快** | ❌ 二进制 | gRPC、高性能 RPC |
| **Hessian** | ✅ | 小 | 快 | ❌ | Dubbo 默认序列化 |
| **Kryo** | ❌（Java） | 很小 | **很快** | ❌ | 纯 Java 高性能场景 |
| **Avro** | ✅ | 小 | 快 | ❌ | 大数据（Hadoop/Kafka） |
| **MessagePack** | ✅ | 小 | 快 | ❌ | 类 JSON 但更紧凑 |
| Java 原生 | ❌ | **大** | 慢 | ❌ | 不推荐 |

### 10.2 JSON 序列化（最常用）

JSON 是跨语言的、人可读的，是 Web 开发的主流。用 **Jackson**（Spring 默认）或 **Gson**：

```java
// Jackson 序列化/反序列化
import com.fasterxml.jackson.databind.ObjectMapper;

ObjectMapper mapper = new ObjectMapper();

// 对象 → JSON 字符串（序列化）
User user = new User("张三", 20);
String json = mapper.writeValueAsString(user);
// {"name":"张三","age":20}

// JSON 字符串 → 对象（反序列化）
User u = mapper.readValue(json, User.class);
```

```java
// Gson 序列化/反序列化
import com.google.gson.Gson;

Gson gson = new Gson();
String json = gson.toJson(user);        // 序列化
User u = gson.fromJson(json, User.class);  // 反序列化
```

> 💡 **提示：** 前后端交互、RESTful API **都用 JSON**。它跨语言、可读、调试方便，是事实标准。

### 10.3 Protobuf（高性能 RPC）

Google 的 Protobuf 用 `.proto` 文件定义结构，编译成各语言代码，二进制紧凑、速度极快，是 **gRPC** 的基础：

```protobuf
// user.proto
syntax = "proto3";
message User {
    string name = 1;
    int32 age = 2;
}
```

```java
// 编译后用生成的类
User.UserProto proto = User.UserProto.newBuilder()
    .setName("张三").setAge(20).build();
byte[] bytes = proto.toByteArray();        // 序列化（很小）
User.UserProto u = User.UserProto.parseFrom(bytes); // 反序列化
```

### 10.4 选型建议

```
前后端 / 跨语言 API   → JSON（Jackson）
高性能 RPC / 微服务    → Protobuf（gRPC）
纯 Java 内部高性能     → Kryo
大数据（Kafka/Hadoop） → Avro
Dubbo 服务            → Hessian（默认）
```

## 十一、常见面试题

### Q1：序列化和反序列化是什么？为什么要用？

```
序列化：对象 → 字节流；反序列化：字节流 → 对象。
用途：网络传输、持久化存储、缓存、深拷贝。
因为网络/磁盘只能传字节，不能直接传对象。
```

### Q2：Serializable 接口有方法吗？为什么实现它就能序列化？

```
没有任何方法，它是【标记接口】。
JVM 通过反射检查对象是否实现了这个接口，
是 → 允许序列化；否 → 抛 NotSerializableException。
```

### Q3：serialVersionUID 有什么用？必须定义吗？

```
版本兼容性检查。序列化时记录 UID，反序列化时对比，
不一致就抛 InvalidClassException。
强烈建议显式定义——否则类结构一变，UID 自动变，旧数据无法反序列化。
```

### Q4：transient 和 static 修饰的字段会被序列化吗？

```
都不会。
- transient：主动标记"不要序列化"，反序列化后是类型默认值。
- static：属于类不属于对象，序列化的是对象，所以不序列化。
```

### Q5：反序列化时会调用构造器吗？

```
Serializable 方式：不调用（对象由 JVM 直接装配）。
Externalizable 方式：会调用 public 无参构造器。
特殊：父类未实现 Serializable 时，反序列化子类会调用父类的无参构造器。
```

### Q6：为什么单例模式实现 Serializable 后会失效？怎么解决？

```
反序列化会创建新对象，破坏单例。
解决：实现 readResolve() 方法，返回单例实例。
```

### Q7：Serializable 和 Externalizable 的区别？

```
Serializable：标记接口，自动序列化所有字段，简单，不调用构造器。
Externalizable：要实现 writeExternal/readExternal，完全手动控制，
               性能更好，调用无参构造器。
```

### Q8：为什么实际开发不用 Java 原生序列化？

```
跨语言差、体积大、性能差、有安全漏洞（反序列化 RCE）。
实际用 JSON（前后端）、Protobuf（RPC）等替代。
```

### Q9：常见的序列化框架有哪些？

```
JSON：Jackson、Gson、Fastjson
二进制：Protobuf、Hessian、Kryo、Avro、MessagePack
```

## 十二、快速参考

### 原生序列化核心代码

```java
// 1. 实现接口 + 版本号
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private int age;
    transient private String password;   // 不序列化
    // public 无参构造器（父类未实现 Serializable 时必需）
}

// 2. 序列化
try (ObjectOutputStream oos = new ObjectOutputStream(new FileOutputStream("f.dat"))) {
    oos.writeObject(user);
}

// 3. 反序列化
try (ObjectInputStream ois = new ObjectInputStream(new FileInputStream("f.dat"))) {
    User u = (User) ois.readObject();
}

// 4. 自定义序列化
private void writeObject(ObjectOutputStream out) throws IOException { ... }
private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException { ... }
private Object readResolve() { return INSTANCE; }   // 保护单例
```

### 核心概念速记

```
Serializable：标记接口，无方法，贴上才能序列化
serialVersionUID：版本号，必须显式定义
transient：不序列化该字段（反序列化后是默认值）
static：不参与序列化（属于类）
反序列化不调用构造器（Serializable 方式）
父类未实现 Serializable → 反序列化子类时调用父类无参构造器
readResolve()：防止反序列化破坏单例
Externalizable：完全手动控制序列化，调用无参构造
```

### 方案选型

```
Java 原生序列化：理解原理，实际少用
前后端 / API：JSON（Jackson / Gson）
RPC / 微服务：Protobuf（gRPC）
纯 Java 高性能：Kryo
大数据：Avro
```

### 自检清单

```
□ 要序列化的类实现了 Serializable 吗？
□ 显式定义了 serialVersionUID 吗？
□ 敏感字段加 transient 了吗？
□ 引用的字段类型都可序列化吗（或标 transient）？
□ 单例类实现 Serializable 时，写了 readResolve() 吗？
□ 父类没实现 Serializable 时，提供了 public 无参构造器吗？
□ 没有反序列化不可信的外部数据吧（安全）？
□ 网络传输优先用 JSON/Protobuf 而非原生序列化吧？
```
