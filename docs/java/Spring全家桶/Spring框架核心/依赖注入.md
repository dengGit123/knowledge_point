### 二、三种注入方式对比

依赖注入有三种写法，**强烈推荐构造器注入**。

#### 1. 构造器注入（推荐 ⭐）

```java
@Service
public class UserService {
    private final UserDao userDao;

    // ✅ 推荐写法：通过构造器注入
    // （只有一个构造器时 @Autowired 可省略）
    public UserService(UserDao userDao) {
        this.userDao = userDao;
    }
}
```

#### 2. Setter 注入

```java
@Service
public class UserService {
    private UserDao userDao;

    @Autowired
    public void setUserDao(UserDao userDao) {
        this.userDao = userDao;
    }
}
```

#### 3. 字段注入（不推荐）

```java
@Service
public class UserService {
    @Autowired  // ❌ 简单但不推荐：无法声明 final、难以单测
    private UserDao userDao;
}
```

| 对比 | 构造器注入 | Setter 注入 | 字段注入 |
| --- | --- | --- | --- |
| 推荐度 | ⭐ **首选** | 可选 | 不推荐 |
| 不可变性 | 可声明 `final` | 不行 | 不行 |
| 循环依赖 | **会直接报错**（暴露问题） | 可绕过 | 可绕过 |
| 单元测试 | 不用 Spring 也能 new | 要 setter | 反射才能注入 |
| 强依赖表达 | 清晰（构造参数=必填） | 弱（可选依赖） | 不清晰 |

> 💡 **为什么推荐构造器注入：** ①依赖一目了然（看构造器就知道这个类需要什么）；②字段能设为 `final`，线程安全；③**循环依赖会在启动时直接暴露**而不是潜伏成运行时 bug。Spring 官方文档也推荐这种方式。

### 三、`@Autowired` 的注入流程

`@Autowired` 标注的依赖，Spring 按这个顺序找 Bean：

```
① 按类型（byType）找
     │
     ├─ 找到 1 个 ──► 直接注入 ✅
     ├─ 找到多个 ──► 进入 ②
     └─ 找到 0 个 ──► 报错（除非 required=false）

② 按名称（byName） narrowing
     │
     └─ 字段/参数名 是否匹配某个 Bean 名？
         ├─ 匹配上 ──► 注入它 ✅
         └─ 还是有歧义 ──► 报错，需 @Qualifier 明确
```

举个「多实现」的例子：

```java
// UserDao 有两个实现，都注册成 Bean
@Repository("mysqlUserDao")
public class UserDaoMysqlImpl implements UserDao { }

@Repository("oracleUserDao")
public class UserDaoOracleImpl implements UserDao { }

@Service
public class UserService {
    @Autowired
    private UserDao mysqlUserDao;  // 按类型找到 2 个 → 按字段名匹配到 mysqlUserDao
}
```

> ⚠️ **注意：** 靠「字段名匹配」太隐晦、容易改坏。多个实现时，**显式用 `@Qualifier` 更可靠**。

### 四、`@Qualifier` / `@Resource` / `@Value`

#### `@Qualifier`：按名称精确指定

```java
@Service
public class UserService {
    @Autowired
    @Qualifier("mysqlUserDao")  // ✅ 显式指定注入哪个实现
    private UserDao userDao;
}
```

#### `@Resource`：JSR-250 标准注解

```java
@Service
public class UserService {
    @Resource(name = "mysqlUserDao")  // 先按 name，找不到再按 type
    private UserDao userDao;
}
```

| 对比 | `@Autowired` | `@Resource` |
| --- | --- | --- |
| 来源 | Spring 自带 | JSR-250 标准（`javax.annotation`） |
| 默认匹配 | **按类型** | **按名称** |
| 配合 `@Qualifier` | 需要 | 不需要（自带 name） |
| 推荐 | Spring 项目首选 | 需要跨框架通用时用 |

> 💡 **提示：** Spring 项目里统一用 `@Autowired` + `@Qualifier` 即可，避免两种风格混用。

#### `@Value`：注入配置值

```java
@Service
public class SmsService {
    @Value("${sms.appkey:默认key}")  // 从配置读，冒号后是默认值
    private String appKey;

    @Value("${sms.timeout}")
    private int timeout;
}
```

`@Value` 读的是配置文件里的值（配合 Spring Boot 的 `application.yml`），详见 [[../SpringBoot/配置文件与Profile]]。

### 五、注入集合：一次拿所有实现

`@Autowired` 标注在 `List`、`Map`、数组上时，Spring 会注入**该类型的所有 Bean**——这是个很实用的特性：

```java
@Service
public class PayService {

    @Autowired
    private List<PayStrategy> strategies;  // 注入所有 PayStrategy 实现到 List

    @Autowired
    private Map<String, PayStrategy> strategyMap;  // key=Bean名，value=实例

    public void pay(String type) {
        // 配合策略模式，按类型选不同支付实现
        PayStrategy strategy = strategyMap.get(type + "PayStrategy");
        strategy.pay();
    }
}
```

> 💡 **提示：** 这是**策略模式**的标准玩法——新增一种支付方式只需加一个实现类，调用方代码不用改。

### 六、循环依赖（面试高频）

#### 1. 什么是循环依赖

A 依赖 B，B 又依赖 A：

```
UserService ──依赖──► OrderService
     ▲                    │
     └──────依赖──────────┘
```

#### 2. Spring 怎么解决（singleton + setter/字段注入）

对于 **singleton** 作用域、且用 **setter 或字段注入**的循环依赖，Spring 通过**三级缓存**解决：

| 缓存 | 名字 | 存什么 |
| --- | --- | --- |
| 一级 | `singletonObjects` | **完全初始化好**的 Bean（成品） |
| 二级 | `earlySingletonObjects` | **提前暴露**的半成品 Bean（已实例化、未注入完） |
| 三级 | `singletonFactories` | 生成半成品的**工厂**（用来支持 AOP 提前代理） |

流程（以 A↔B 为例）：

```
① 创建 A，实例化后把「A 的工厂」放三级缓存
② 给 A 注入依赖 B，但 B 还没有 → 去创建 B
③ 创建 B，实例化后给 B 注入依赖 A
④ 找 A：一级没有、二级没有，从三级工厂拿到「早期 A」→ 放入二级缓存
⑤ B 拿到 A，B 完成创建，放入一级缓存
⑥ 回到 A，拿到完成的 B，A 完成创建，放入一级缓存
```

> 💡 **一句话理解：** 容器先把「没装配完的 A」提前暴露出来，让 B 先用着，等 B 好了再回来补全 A。

#### 3. 哪些循环依赖解决不了

> ⚠️ **注意：** 以下情况 Spring **无法自动解决**，会启动报错：
> - **构造器注入**的循环依赖（实例化时就需要对方，没法提前暴露）。
> - **prototype** 作用域的循环依赖（每次都新建，缓存机制不适用）。

正因为构造器注入的循环依赖会「**直接报错**」而不是潜伏，它才被推荐——逼你在设计阶段消除不合理的相互依赖。

### 七、常见问题与注意事项

> ⚠️ **注意：**
> - `@Autowired` 找不到 Bean 时默认会报错；设 `@Autowired(required = false)` 可让依赖可选（不推荐常用）。
> - Spring Boot 2.6+ **默认禁止循环依赖**（启动直接报错），这是为了规范设计。如有遗留代码，可临时 `spring.main.allow-circular-references=true` 开启，但应尽快重构。
> - 构造器注入时若必须保留循环依赖，可以用 `@Lazy` 让其中一方延迟加载——但本质还是设计问题，应优先解耦。

> 💡 **提示：** 解决循环依赖的根因往往是「职责划分不清」。A 和 B 互相依赖，通常说明该把公共逻辑抽到第三个类 C 里，让 A、B 都依赖 C。

---

### 八、总结

- **推荐构造器注入**：依赖清晰、字段可 `final`、循环依赖早暴露。
- `@Autowired` 按类型找，多个时按名称/`@Qualifier` 缩小范围。
- `@Resource` 是标准注解，默认按名称；`@Value` 注入配置值。
- 注入 `List`/`Map` 可一次性拿到某接口的所有实现（策略模式利器）。
- **循环依赖**：singleton + setter/字段注入靠三级缓存解决；构造器注入和 prototype 无法解决，会报错。
