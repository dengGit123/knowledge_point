### 一、概述

> 📖 [Spring Boot - Auto-configuration](https://docs.spring.io/spring-boot/docs/current/reference/html/using.html#using.auto-configuration)

**自动配置（Auto Configuration）** 是 Spring Boot 最核心的魔法——你引入一个依赖（比如 MySQL 驱动），数据源、事务管理器就自动配好了，**不用写任何配置代码**。这篇揭开它的黑盒。

大白话：自动配置像是一个「**智能管家**」——它检查你家（项目）里有什么设备（依赖），自动帮你接好水电煤。家里有 MySQL，它就配好数据源；没有 Redis，它就跳过 Redis 配置。**按需装配，绝不强加。**

| 你将学到 | 说明 |
| --- | --- |
| `@SpringBootApplication` 的组成 | 三个注解的合体 |
| 自动配置怎么加载 | `spring.factories` 与新版的 `.imports` 文件 |
| 条件装配 | `@ConditionalOnXxx` 系列注解 |
| 排查问题 | `--debug` 看装配报告 |

> 📌 本文是 [[SpringBoot入门]] 和 [[起步依赖]] 的深入。

### 二、`@SpringBootApplication` 的三合一

启动类上的这个注解，其实是三个注解的「合体」：

```java
@SpringBootApplication  // ← 等价于下面三个 ↓
public class Application { }

// 拆解开来：
@SpringBootConfiguration   // 本质就是 @Configuration，标记这是个配置类
@EnableAutoConfiguration   // ← 开启自动配置（核心）
@ComponentScan             // 组件扫描，扫描启动类所在包及子包
```

| 注解 | 作用 |
| --- | --- |
| `@SpringBootConfiguration` | 标记配置类（容器里可以定义 `@Bean`） |
| `@ComponentScan` | 扫描 `@Component`/`@Service`/`@Controller` 等 |
| **`@EnableAutoConfiguration`** | **自动配置的入口**，下文详解 |

---

### 三、自动配置是怎么加载的

`@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 触发加载。它会去读取一个「**自动配置类清单**」文件，把里面列的配置类全部加载（再按条件过滤）。

> ⚠️ **版本差异（重要）：** 清单文件的位置在 Spring Boot 2.7 前后变了：

| 版本 | 清单文件位置 |
| --- | --- |
| 2.7 **之前** | `META-INF/spring.factories`，键为 `org.springframework.boot.autoconfigure.EnableAutoConfiguration` |
| 2.7 **及之后** | `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`，每行一个配置类全名 |

例如新版里，这个文件长这样（每行一个自动配置类）：

```
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
org.springframework.boot.autoconfigure.redis.RedisAutoConfiguration
... (一百多个)
```

> 💡 **提示：** Spring Boot 启动时会加载清单里**所有**配置类，但**不会全部生效**——靠下一节的「条件装配」按需过滤。

### 四、核心：条件装配（`@ConditionalOnXxx`）

每个自动配置类都用「**条件注解**」声明「**在什么情况下我才生效**」。这是「按需装配」的关键。

| 注解 | 生效条件 |
| --- | --- |
| `@ConditionalOnClass(Xxx.class)` | classpath 里有 `Xxx` 类 |
| `@ConditionalOnMissingClass(...)` | classpath 里**没有**某类 |
| `@ConditionalOnBean(Xxx.class)` | 容器里已经有 `Xxx` Bean |
| `@ConditionalOnMissingBean(Xxx.class)` | 容器里**还没有** `Xxx` Bean |
| `@ConditionalOnProperty(prefix="x", name="enabled")` | 配置里有指定属性 |
| `@ConditionalOnWebApplication` | 是 Web 应用 |
| `@ConditionalOnNotWebApplication` | 不是 Web 应用 |

看一个真实例子（`DataSourceAutoConfiguration` 的简化版）：

```java
@AutoConfiguration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })  // 有数据库相关类才生效
@EnableConfigurationProperties(DataSourceProperties.class)             // 绑定 spring.datasource.* 配置
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean   // ✅ 用户没自定义数据源时，才创建默认数据源
    public DataSource dataSource(DataSourceProperties props) {
        return props.initializeDataSourceBuilder().build();
    }
}
```

**逻辑链**：classpath 有 MySQL 驱动（满足 `@ConditionalOnClass`）+ 配置了 `spring.datasource.url` → 自动创建数据源 Bean；若用户自己定义了 `DataSource` Bean（`@ConditionalOnMissingBean` 不满足），就跳过默认的。

> 💡 **这就是「自动」的真相：** 引入依赖 = classpath 有类 = 条件满足 = 自动配好。

### 五、自动配置加载流程

```
启动 Application.main()
  │
  ├─ @ComponentScan 扫描业务代码里的 @Component
  │
  └─ @EnableAutoConfiguration
        │
        ├─ 读取 AutoConfiguration.imports 清单（一百多个配置类）
        │
        ├─ 逐个判断 @ConditionalOnXxx 条件
        │     ├─ 满足 → 装配（如引入了 MySQL 驱动 → 创建数据源）
        │     └─ 不满足 → 跳过（如没引入 Redis → 不创建 RedisTemplate）
        │
        └─ 用户自定义的 @Bean 优先（@ConditionalOnMissingBean 保护）
```

> 💡 **核心原则：用户配置优先。** 自动配置永远是「兜底」——只要你自己定义了 Bean，自动配置就让位（靠 `@ConditionalOnMissingBean`）。

### 六、排查「为什么没自动配置」

启动时加 `--debug` 参数（或在 `application.yml` 设 `debug: true`），控制台会打印**自动配置报告**：

```
============================
CONDITIONS EVALUATION REPORT
============================

Positive matches:（生效的自动配置）
   DataSourceAutoConfiguration matched:
      - @ConditionalOnClass found required classes 'javax.sql.DataSource' (OnClassCondition)

Negative matches:（未生效及原因）
   RedisAutoConfiguration:
      - @ConditionalOnClass did not find required class 'redis...' (OnClassCondition)
        ↑ 没引入 Redis 依赖，所以没配
```

> 💡 **提示：** 排查「为什么某个 Bean 没有自动生成」，看 `Negative matches` 里的原因——是缺类、缺 Bean，还是缺配置，一目了然。

### 七、如何覆盖 / 排除自动配置

#### 1. 用户 Bean 覆盖

直接自己定义同类型 Bean，自动配置会让位：

```java
@Bean
public DataSource dataSource() {
    return new HikariDataSource();   // ✅ 自定义后，默认数据源不再创建
}
```

#### 2. 排除某个自动配置

```java
@SpringBootApplication(exclude = { DataSourceAutoConfiguration.class })  // 禁用数据源自动配置
public class Application { }
```

#### 3. 配置属性覆盖

大部分自动配置的参数都能在 `application.yml` 里覆盖，如数据源、Redis、端口。详见 [[配置文件与Profile]]。

### 八、常见问题与注意事项

> ⚠️ **注意：**
> - **2.7 是分水岭**：自定义 starter 注册自动配置类，新版必须用 `AutoConfiguration.imports` 文件，老的 `spring.factories` 已废弃。
> - 自动配置类用 `@AutoConfiguration`（Boot 2.7+）或 `@Configuration`，**不要在清单里乱写普通类**。
> - 自动配置**不是万能的**：复杂场景（多数据源、定制化逻辑）仍需手动配置。

> 💡 **提示：** 看到「自动配好了」别只觉得神奇，去对应的 `XxxAutoConfiguration` 类里看一眼——它就是个带条件注解的普通配置类，和你自己写的 `@Configuration` 没有本质区别。

### 九、总结

- **入口**：`@SpringBootApplication` = 配置类 + 组件扫描 + 自动配置。
- **加载机制**：从 `AutoConfiguration.imports`（2.7+）清单读取配置类。
- **按需装配**：靠 `@ConditionalOnClass`/`@ConditionalOnMissingBean` 等条件注解过滤。
- **用户优先**：自定义 Bean 永远覆盖自动配置（`@ConditionalOnMissingBean`）。
- **排查工具**：`--debug` 看 Conditions Evaluation Report。

理解了自动配置，Spring Boot 对你就不再有黑盒。下一篇看怎么管理配置：[[配置文件与Profile]]。
