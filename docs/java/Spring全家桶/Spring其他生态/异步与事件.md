### 一、概述

> 📖 [Spring - Core Technologies: Events](https://docs.spring.io/spring-framework/reference/core/beans/context-introduction.html#context-functionality-events)

本文讲两个让 Spring 应用「**更快、更解耦**」的特性：**`@Async` 异步执行** 和 **事件机制（ApplicationEvent）**。它们解决的是「**主流程不该等待的慢操作**」和「**做完一件事后要通知多个不相关方**」两类问题。

大白话：
- **`@Async`**：让一个方法「丢到后台线程跑」，主流程不用等它。像寄快递——你把包裹给快递员（异步方法）就回家，不用站那等他送到。
- **事件机制**：一个动作发生后，自动通知所有关心它的人，彼此不直接依赖。像「**广播通知**」——注册成功了，发短信、送积分、写日志这些事自动被触发，注册逻辑不用挨个调用。

| 你将学到 | 说明 |
| --- | --- |
| `@Async` | 异步方法怎么用、线程池怎么配 |
| 事件机制 | 发布事件 + 监听事件 |
| 解耦应用 | 注册后触发多个动作 |

> 📌 配合 [[定时任务]]、[[缓存抽象]] 阅读。

### 二、`@Async` 异步方法

#### 1. 开启 + 使用

```java
@SpringBootApplication
@EnableAsync          // ✅ 开启异步支持
public class Application { }
```

```java
@Service
public class SmsService {

    @Async                              // ✅ 这个方法会在另一个线程异步执行
    public void sendSms(String phone, String content) {
        // 耗时的发短信操作
        Thread.sleep(3000);
        System.out.println("短信已发送给 " + phone);
    }
}

// 调用方
@Autowired
private SmsService smsService;

public void register(User user) {
    saveUser(user);
    smsService.sendSms(user.getPhone(), "注册成功");   // ✅ 立即返回，不等短信发完
    System.out.println("注册流程结束");                // 先打印这行，3 秒后才打印"短信已发送"
}
```

> 💡 **核心原理：** `@Async` 底层是 [[../Spring框架核心/AOP面向切面]]——给方法套代理，调用时把任务提交到**线程池**，主线程立即返回。所以它也有「**自调用失效**」的坑（见后文）。

#### 2. 返回值

```java
@Async
public void asyncVoid() { }                        // 无返回值（fire-and-forget）

@Async
public Future<String> asyncWithResult() {          // 有返回值，用 Future
    return new AsyncResult<>("完成");               // AsyncResult 是 Future 的实现
}

@Async
public CompletableFuture<String> asyncBetter() {   // 推荐 CompletableFuture（更强大）
    return CompletableFuture.completedFuture("完成");
}
```

### 三、【重点】必须自定义线程池

> ⚠️ **重大陷阱：** 不配线程池时，Spring 默认用 `SimpleAsyncTaskExecutor`——它**每次调用都新建一个线程**，没有复用、没有上限，**高并发下会创建大量线程，把系统搞垮**。生产环境**绝不能**用默认的。

#### 自定义线程池

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Bean("taskExecutor")
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);          // 核心线程数
        executor.setMaxPoolSize(20);          // 最大线程数
        executor.setQueueCapacity(100);       // 队列容量
        executor.setKeepAliveSeconds(60);     // 空闲线程存活时间
        executor.setThreadNamePrefix("async-");   // 线程名前缀（便于排查）
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy());   // 拒绝策略：满了由调用线程执行
        executor.initialize();
        return executor;
    }
}
```

```java
@Async("taskExecutor")    // 指定用哪个线程池（可省略，默认找名为 taskExecutor 的）
public void sendSms(String phone) { }
```

> 💡 **提示：** 通过实现 `AsyncConfigurer`，`getAsyncExecutor()` 返回的线程池会成为 `@Async` 的**默认线程池**，省去每次指定名字。

### 四、`@Async` 的注意事项

> ⚠️ **注意：**
> - **自调用失效**：同类内部 `a()` 调 `b()`（b 上有 `@Async`），异步失效（走 `this` 不走代理）。解决：拆到不同类，或注入自身代理。
> - **异常不会抛给调用方**：`@Async` 方法抛异常，调用方感知不到（在另一个线程）。无返回值的需配 `AsyncUncaughtExceptionHandler`；有返回值的从 `Future.get()` 拿异常。
> - **方法必须是 `public`**：AOP 代理只对 public 方法生效。
> - **别滥用异步**：只有「**不需要立即拿到结果、且耗时**」的操作才值得异步（发短信、发邮件、写日志）。

#### 配置全局异常处理

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {
    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) -> {
            System.err.println("异步方法 " + method.getName() + " 出异常: " + ex.getMessage());
        };
    }
}
```

### 五、事件机制（解耦利器）

#### 场景：注册成功后要做一堆事

不用事件时，注册逻辑里塞满各种调用：

```java
public void register(User user) {
    saveUser(user);
    sendSms(user.getPhone());       // ❌ 注册逻辑和这些后续动作耦合在一起
    sendEmail(user.getEmail());     // ❌ 新增一个动作就要改这里
    givePoints(user.getId());       // ❌
    writeLog(user.getId());         // ❌
}
```

问题：注册逻辑和「注册后要做什么」**强耦合**，加一个动作就要改注册方法。

#### 用事件解耦

事件机制三要素：**发布事件、定义事件、监听事件**。

```java
// ① 定义事件（继承 ApplicationEvent，或直接用普通类）
public class UserRegisterEvent extends ApplicationEvent {
    private final Long userId;
    public UserRegisterEvent(Object source, Long userId) {
        super(source);
        this.userId = userId;
    }
    public Long getUserId() { return userId; }
}

// ② 发布事件
@Service
public class UserService {
    @Autowired
    private ApplicationEventPublisher publisher;   // Spring 自动注入

    public void register(User user) {
        saveUser(user);
        publisher.publishEvent(new UserRegisterEvent(this, user.getId()));  // ✅ 只发布，不管谁来处理
    }
}

// ③ 监听事件（可以有多个，互相独立）
@Component
public class RegisterListeners {

    @EventListener
    public void sendSms(UserRegisterEvent event) {       // 发短信的监听器
        System.out.println("给用户 " + event.getUserId() + " 发短信");
    }

    @EventListener
    public void givePoints(UserRegisterEvent event) {    // 送积分的监听器
        System.out.println("给用户 " + event.getUserId() + " 送积分");
    }

    @EventListener
    @Async                                              // 异步监听，不阻塞发布方
    public void writeLog(UserRegisterEvent event) {      // 写日志的监听器
        System.out.println("记录注册日志");
    }
}
```

**效果**：注册方法只管「发布事件」，发短信/送积分/写日志各自独立监听。**新增一个后续动作**，只需加一个 `@EventListener` 方法，**完全不用动注册逻辑**。

> 💡 **核心价值：解耦。** 发布方不关心「谁在听、有几个监听器」；监听方不关心「事件从哪来」。这就是「**观察者模式**」的 Spring 实现。

### 六、`@EventListener` 同步 vs 异步

| | 默认行为 | 加 `@Async` |
| --- | --- | --- |
| 执行线程 | 和发布方**同线程**（同步） | **独立线程**（异步） |
| 发布方是否等待 | **等待**所有监听器执行完 | 不等待 |
| 异常影响 | 监听器抛异常会**中断**发布方 | 不影响发布方 |

```java
@EventListener
public void syncListener(UserRegisterEvent e) { }      // 同步：register() 会等它执行完

@EventListener
@Async                                                  // 异步：register() 不等它
public void asyncListener(UserRegisterEvent e) { }
```

> 💡 **提示：** 核心后续动作（必须成功的，如送积分）用同步；非核心动作（可异步、可失败的，如发营销邮件）用 `@Async`。

### 七、实际应用场景

1. **`@Async` 场景**：发短信、发邮件、推送通知、生成报表——主流程不用等的慢操作。
2. **事件解耦场景**：注册成功后触发一系列动作；订单创建后通知库存、营销、积分系统。
3. **状态变更通知**：订单状态变化，通知多个关注方（统计、推送、日志）。
4. **审计日志**：重要操作后异步记录审计日志（不阻塞主流程）。

---

### 八、常见问题与注意事项

> ⚠️ **注意：**
> - **`@Async` 默认线程池是坑**：生产必须自定义线程池（见第三节）。
> - **自调用失效**：`@Async` 和 `@EventListener`（在同类里）都受 AOP 代理限制。
> - **事件监听器同步执行**：默认会阻塞发布方，慢监听器拖慢主流程。需异步就加 `@Async`。
> - **事件不要承载业务参数过多**：事件应传递「**发生了什么**」（事件名 + 必要 id），不该把一堆业务对象塞进去。
> - **事务与事件**：想「事务提交后再发事件」（避免事件发了但事务回滚），用 `@TransactionalEventListener`。

> 💡 **提示：** 事件机制是**解耦的核心手段**，但别过度设计——简单的直接调用更清晰。当「一个动作要触发多个**独立且可能变化**的后续动作」时，事件才真正 shines。

---

### 九、总结

- **`@Async`**：方法异步执行，主流程不等。生产**必须自定义线程池**（默认的会创建无上限线程）。
- **`@Async` 注意**：自调用失效、异常不抛给调用方、方法须 public。
- **事件机制**：发布事件（`ApplicationEventPublisher`）+ 监听事件（`@EventListener`），实现观察者模式解耦。
- **监听器默认同步**（阻塞发布方），加 `@Async` 变异步。
- **价值**：`@Async` 让慢操作不阻塞；事件让「触发」和「后续处理」解耦。
