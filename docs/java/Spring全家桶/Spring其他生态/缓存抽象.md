### 一、概述

> 📖 [Spring - Cache Abstraction](https://docs.spring.io/spring-framework/reference/integration/cache.html)

**Spring Cache** 是 Spring 提供的一套**缓存接口抽象**。业务代码只写几个注解（`@Cacheable` 等），底层缓存实现可以自由切换 Redis、Caffeine、EhCache——**换缓存不用改业务代码**。

大白话：不用 Spring Cache 时，每个查询方法都得手写「先查缓存、没有就查库、再放缓存」这套重复逻辑。Spring Cache 让你**只加个 `@Cacheable` 注解**，框架自动帮你查缓存、存缓存，业务方法干干净净。至于缓存存 Redis 还是本地内存，那是配置的事，跟业务无关。

| 你将学到 | 说明 |
| --- | --- |
| 缓存抽象的意义 | 业务与缓存实现解耦 |
| 四个核心注解 | `@Cacheable`/`@CachePut`/`@CacheEvict`/`@CacheConfig` |
| 切换底层 | Redis / Caffeine |
| 缓存三大问题 | 穿透 / 击穿 / 雪崩 |

> 📌 配合 [[../SpringData/SpringDataRedis]] 阅读（缓存常存 Redis）。

### 二、为什么需要「缓存抽象」

#### 不用抽象时的重复代码

```java
public User findById(Long id) {
    User user = redis.get("user:" + id);     // ❌ 每个方法都重复这套
    if (user != null) return user;
    user = userDao.findById(id);
    if (user != null) {
        redis.set("user:" + id, user, 30, MINUTES);   // ❌ 缓存逻辑侵入业务
    }
    return user;
}
```

每个查询方法都要写一遍缓存逻辑，又长又重复。

#### 用 Spring Cache 后

```java
@Cacheable(cacheNames = "user", key = "#id")   // ✅ 一个注解搞定
public User findById(Long id) {
    return userDao.findById(id);    // 业务方法干净
}
```

> 💡 **核心价值：** 「**业务代码只声明要缓存**」「**缓存用什么实现是配置决定的**」。今天用 Redis，明天想换 Caffeine（本地缓存），业务代码一行不改。

### 三、四个核心注解

| 注解 | 作用 | 执行时机 |
| --- | --- | --- |
| `@Cacheable` | 查缓存，命中则不执行方法；未命中则执行并缓存 | 方法**执行前** |
| `@CachePut` | 始终执行方法，把结果更新进缓存 | 方法**执行后** |
| `@CacheEvict` | 删除缓存 | 方法**执行后** |
| `@Caching` | 组合多个缓存操作 | — |
| `@CacheConfig` | 类级别公共配置（cacheNames 等） | — |

#### 1. `@Cacheable`（查询，最常用）

```java
@Cacheable(cacheNames = "user", key = "#id")
// 第一次调用：查库 → 把结果存缓存（key=user::1）
// 之后调用：直接返回缓存，不进方法
public User findById(Long id) {
    return userDao.findById(id);
}
```

#### 2. `@CachePut`（更新，配合修改方法）

```java
@CachePut(cacheNames = "user", key = "#user.id")
// 始终执行方法，把新结果覆盖进缓存
public User update(User user) {
    return userDao.save(user);   // 数据库更新 + 缓存同步更新
}
```

#### 3. `@CacheEvict`（删除，配合删除方法）

```java
@CacheEvict(cacheNames = "user", key = "#id")
// 执行后删除对应缓存
public void deleteById(Long id) {
    userDao.deleteById(id);
}

@CacheEvict(cacheNames = "user", allEntries = true)   // 清空整个 user 缓存
public void deleteAll() { }
```

#### key 的 SpEL 表达式

| 表达式 | 含义 |
| --- | --- |
| `#id` | 方法参数名为 id 的值 |
| `#p0` / `#a0` | 第 0 个参数 |
| `#result` | 方法返回值（`@CachePut` 常用） |
| `#user.id` | 参数 user 的 id 属性 |
| `T(java.lang.Math).random()` | 静态方法调用 |

### 四、开启支持

启动类加 `@EnableCaching`：

```java
@SpringBootApplication
@EnableCaching          // ✅ 开启缓存支持
public class Application { }
```

### 五、完整示例

```java
@Service
@CacheConfig(cacheNames = "user")   // 类级别公共配置，方法上省略 cacheNames
public class UserService {

    @Cacheable(key = "#id")
    public User findById(Long id) {
        System.out.println("查数据库...");
        return userDao.findById(id);          // 只在缓存未命中时执行
    }

    @Cacheable(key = "#name", unless = "#result == null")   // 结果为 null 不缓存
    public User findByName(String name) {
        return userDao.findByName(name);
    }

    @CachePut(key = "#user.id")
    public User save(User user) {
        return userDao.save(user);            // 更新数据库 + 更新缓存
    }

    @CacheEvict(key = "#id")
    public void deleteById(Long id) {
        userDao.deleteById(id);               // 删数据库 + 删缓存
    }
}
```

> 💡 **提示：** `@Cacheable` 的 `unless`/`condition` 能控制是否缓存。如 `unless = "#result == null"` 表示查询结果为 null 不缓存（避免缓存穿透时存一堆空值）。

### 六、切换底层实现

#### 1. 默认：ConcurrentHashMap（内存，重启丢失）

不引入任何缓存依赖时，Spring Boot 用内置的简单缓存（ConcurrentHashMap）。**仅适合开发测试**——重启即失效、不能跨实例共享。

#### 2. 整合 Redis（最常用）

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-cache</artifactId>
</dependency>
```

引入 Redis 依赖后，Spring Boot **自动用 Redis 作为缓存实现**。自定义 TTL 和序列化：

```java
@Configuration
public class CacheConfig {
    @Bean
    public RedisCacheManager cacheManager(RedisConnectionFactory factory) {
        RedisCacheConfiguration config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(30))                          // 默认 TTL 30 分钟
            .serializeKeysWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new StringRedisSerializer()))          // key 用 String
            .serializeValuesWith(RedisSerializationContext.SerializationPair
                .fromSerializer(new GenericJackson2JsonRedisSerializer())); // value 用 JSON
        return RedisCacheManager.builder(factory).cacheDefaults(config).build();
    }
}
```

> 💡 **提示：** 详情见 [[../SpringData/SpringDataRedis]]。配好序列化，缓存进 Redis 的内容才是可读 JSON。

#### 3. 整合 Caffeine（本地高性能缓存）

```xml
<dependency>
    <groupId>com.github.ben-manes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
</dependency>
```

引入后 Spring Boot 自动用 Caffeine（基于内存的本地缓存，性能极高）。适合**单机、读多写少、对延迟敏感**的场景。

| | Redis 缓存 | Caffeine 缓存 |
| --- | --- | --- |
| 位置 | 远程（独立服务） | 本地内存 |
| 性能 | 网络开销，毫秒级 | **纳秒级，极快** |
| 跨实例共享 | ✅ 能 | ❌ 各实例独立 |
| 容量 | 受 Redis 内存 | 受 JVM 内存 |
| 适用 | 分布式、大数据量 | 单机、热点数据 |

> 💡 **提示：** 大型系统常「**多级缓存**」：先查 Caffeine（本地，极快），没有再查 Redis（分布式），最后查数据库。

### 七、缓存经典三问：穿透 / 击穿 / 雪崩

这是缓存设计的核心知识（不止 Spring Cache，所有缓存都通用）：

| 问题 | 现象 | 原因 | 解决 |
| --- | --- | --- | --- |
| **缓存穿透** | 查一个**根本不存在**的数据，每次都打到数据库 | 缓存不存 null | 缓存空值 / **布隆过滤器**拦截非法 key |
| **缓存击穿** | 一个**热点 key** 突然失效，瞬间大量请求打到数据库 | 热点 key 过期 | **互斥锁**（只让一个线程查库）/ 热点 key 永不过期 |
| **缓存雪崩** | **大量 key 同时失效**，请求全打到数据库 | 大批 key 同一时刻过期 | TTL **加随机值**（避免同时失效）/ 多级缓存 / 限流降级 |

```
穿透：查 id=-1（不存在）→ 缓存没有 → 查库 → 没有 → 不缓存 → 下次还是查库
击穿：爆款商品 key 失效瞬间 → 1万个请求同时查库 → 数据库扛不住
雪崩：缓存集群重启/大量 key 同时过期 → 所有请求涌向数据库
```

> ⚠️ **注意：** Spring Cache 默认**不解决**这三个问题，需要你自己在使用时注意（如配随机 TTL、缓存空值、用 `unless` 控制不缓存 null 的反过来——主动缓存 null）。

### 八、常见问题与注意事项

> ⚠️ **注意：**
> - **自调用失效**：`@Cacheable` 等注解基于 AOP 代理，同类内部方法互调会失效（和 [[../Spring框架核心/AOP面向切面]] 一样）。
> - **缓存与数据库一致性**：常用「**先更新数据库、再删缓存**」（Cache Aside）。Spring Cache 的 `@CachePut`/`@CacheEvict` 配合修改方法实现。
> - **`@Cacheable` 的 key 要唯一**：不同方法的 key 撞了会串数据。习惯用 `cacheNames` 隔离（`user::1`、`order::1`）。
> - **缓存对象要可序列化**：存 Redis 时对象需能序列化（实现 Serializable 或用 JSON 序列化器）。

> 💡 **提示：** Spring Cache 适合**规则简单**的缓存（按 id 缓存、查删改同步）。**复杂缓存逻辑**（条件缓存、多级、动态 TTL）建议直接用 [[../SpringData/SpringDataRedis]] 手写，更灵活可控。

### 九、总结

- **Spring Cache = 缓存接口抽象**，业务写注解，底层可切换 Redis/Caffeine/EhCache。
- **四个注解**：`@Cacheable`（查）、`@CachePut`（更）、`@CacheEvict`（删）、`@CacheConfig`（类公共配置）。
- **切换底层**：引依赖即可（Redis 分布式、Caffeine 本地极速）。
- **缓存三问**：穿透（查不存在）、击穿（热点失效）、雪崩（大批失效）——靠空值缓存、互斥锁、随机 TTL 解决。
- **铁律**：自调用失效、注意缓存与数据库一致性。
