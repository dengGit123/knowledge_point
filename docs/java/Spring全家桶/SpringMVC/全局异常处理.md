### 一、概述

> 📖 [Spring MVC - Exception Handling](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-ann-rest-exceptions.html)

**全局异常处理** 让你用「**一处捕获，处处生效**」的方式处理所有接口的异常，避免每个方法里写一堆 `try-catch`，同时保证错误返回格式统一。

大白话：没有全局异常处理，每个接口都得自己 try-catch，写出来的代码又丑又重复；有了它，异常直接抛，会有一个「**全局守门员**」统一接住、统一格式返回。Controller 代码干干净净。

| 你将学到 | 说明 |
| --- | --- |
| `@RestControllerAdvice` | 声明全局异常处理类 |
| `@ExceptionHandler` | 处理特定异常 |
| 自定义业务异常 | 用异常 + 错误码表达业务错误 |
| 统一返回结构 | 所有错误格式一致 |

> 📌 配合 [[拦截器]]、[[请求与响应处理]] 阅读。

### 二、为什么需要全局异常处理

#### 没有全局处理时的痛点

```java
@GetMapping("/{id}")
public Result<User> get(@PathVariable Long id) {
    try {
        User user = userService.findById(id);
        if (user == null) {
            return Result.error(404, "用户不存在");   // ❌ 每个方法都要判断、try-catch
        }
        return Result.success(user);
    } catch (IllegalArgumentException e) {
        return Result.error(400, "参数错误");         // ❌ 重复的异常处理
    } catch (Exception e) {
        log.error("系统异常", e);
        return Result.error(500, "系统繁忙");         // ❌ 每个接口都来一遍
    }
}
```

每个接口都重复一遍异常处理，又长又容易遗漏。

#### 全局处理后

```java
@GetMapping("/{id}")
public User get(@PathVariable Long id) {              // ✅ Controller 干净，异常直接抛
    return userService.findById(id);                  // 找不到就抛 BizException，全局会接住
}
```

### 三、`@RestControllerAdvice` + `@ExceptionHandler`

两个核心注解：

| 注解 | 作用 |
| --- | --- |
| `@RestControllerAdvice` | 声明一个**全局异常处理类**（= `@ControllerAdvice` + `@ResponseBody`，返回值转 JSON） |
| `@ExceptionHandler(XxxException.class)` | 标在方法上，处理**指定的异常类型** |

---

### 四、自定义业务异常

用异常 + 错误码来表达「业务错误」，比返回错误标志更清晰：

```java
// 1. 自定义业务异常
public class BizException extends RuntimeException {
    private final int code;

    public BizException(int code, String message) {
        super(message);
        this.code = code;
    }

    public int getCode() { return code; }
}

// 2. 错误码枚举（推荐用枚举集中管理）
public enum ErrorCode {
    USER_NOT_FOUND(1001, "用户不存在"),
    PARAM_ERROR(1002, "参数错误"),
    NO_PERMISSION(1003, "无权限");

    private final int code;
    private final String message;
    // 构造器、getter...
}
```

业务里直接抛：

```java
public User findById(Long id) {
    User user = userDao.findById(id);
    if (user == null) {
        throw new BizException(ErrorCode.USER_NOT_FOUND);   // 直接抛，不用管怎么返回
    }
    return user;
}
```

### 六、完整示例：处理多种异常

一个生产可用的全局异常处理器：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    // 1. 业务异常：返回业务错误码
    @ExceptionHandler(BizException.class)
    public Result<Void> handleBiz(BizException e) {
        log.warn("业务异常: code={}, msg={}", e.getCode(), e.getMessage());
        return Result.error(e.getCode(), e.getMessage());
    }

    // 2. 参数校验异常（@Valid 校验失败）：返回 400
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public Result<Void> handleValid(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldError().getDefaultMessage();
        return Result.error(400, "参数错误: " + msg);
    }

    // 3. 请求方式不对（如 GET 请求了 POST 接口）
    @ExceptionHandler(HttpRequestMethodNotSupportedException.class)
    public Result<Void> handleMethod(HttpRequestMethodNotSupportedException e) {
        return Result.error(405, "请求方法不支持: " + e.getMethod());
    }

    // 4. 兜底：所有未捕获的异常，返回 500（最宽泛，放最后）
    @ExceptionHandler(Exception.class)
    public Result<Void> handleAll(Exception e) {
        log.error("系统异常", e);
        return Result.error(500, "系统繁忙，请稍后再试");   // 不把堆栈暴露给前端
    }
}
```

```
抛出异常类型                    匹配的处理器
─────────────────────────────────────────────
BizException               ──►  handleBiz       (业务码)
MethodArgumentNotException ──►  handleValid     (400)
其他所有 Exception          ──►  handleAll       (500 兜底)
```

### 七、异常处理优先级

当多个 `@ExceptionHandler` 都能匹配时，**更具体的异常类型优先**：

```java
@ExceptionHandler(Exception.class)          // 父类，宽泛
@ExceptionHandler(RuntimeException.class)   // 子类，更具体 → 优先匹配
```

- 异常类型**越具体，优先级越高**（Java 继承关系）。
- 可用 `@Order` 调整不同 `@RestControllerAdvice` 类之间的优先级。

> 💡 **提示：** 把最宽泛的 `Exception` 处理器放一个作为「**兜底**」，保证任何异常都不会以原始堆栈暴露给前端。

---

### 八、实际应用场景

1. **统一错误格式**：前端只需按 `code` 处理错误，逻辑统一。
2. **业务错误码体系**：用 `BizException` + 错误码枚举，错误信息集中管理。
3. **参数校验失败**：自动捕获 `MethodArgumentNotValidException`，返回具体哪个字段错了。
4. **隐藏系统细节**：兜底处理器把堆栈转成「系统繁忙」，避免泄露技术细节（也是安全要求）。

---

### 九、常见问题与注意事项

> ⚠️ **注意：**
> - **别把异常堆栈直接返回给前端**——既不友好，也可能泄露数据库结构等敏感信息。生产环境只返回友好提示。
> - **`@ExceptionHandler` 的返回类型**：前后端分离统一返回 `Result<T>` 对象（自动转 JSON）。
> - **拦截器/Filter 里抛的异常**：发生在 DispatcherServlet **之外**的异常，`@RestControllerAdvice` **可能捕获不到**（如 Filter 链早期）。这些需要在 Filter 内部处理或用 `ErrorController`。
> - **检查异常要主动抛**：业务方法抛 `RuntimeException` 子类（如 `BizException`）更方便，避免到处 `throws`。

> 💡 **提示：** 好的异常处理 = **业务异常用 `BizException` 明确表达 + 全局兜底统一格式**。Controller 里基本不该出现 `try-catch`。

---

### 十、总结

- **全局异常处理** = `@RestControllerAdvice` + `@ExceptionHandler`，一处捕获处处生效。
- **自定义业务异常**（`BizException` + 错误码枚举）表达业务错误，比返回标志更清晰。
- **更具体的异常优先**，最宽泛的 `Exception` 作兜底。
- **统一返回 `Result<T>`**，错误信息不暴露堆栈。
- Controller 里别写 `try-catch`，异常直接抛，交给全局处理。
