### 一、概述

> 📖 [Spring MVC - Interception](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-handlermapping-interceptor.html)

**拦截器（Interceptor）** 让你在 Controller 方法执行的**前后**插入通用逻辑（登录校验、权限、日志），而不污染业务代码。它是 Spring MVC 提供的机制，底层就是 [[SpringMVC工作原理]] 流程里的一环。

大白话：拦截器像高级会所的「**门禁 + 迎宾**」——客人（请求）进来前先验身份（`preHandle`），通过才放行；服务完后送客、收尾（`afterCompletion`）。整个过程发生在 Controller 前后，业务方法完全无感。

| 你将学到 | 说明 |
| --- | --- |
| 拦截器 vs 过滤器 | 两者区别 |
| 三个拦截方法 | `preHandle`/`postHandle`/`afterCompletion` |
| 注册拦截器 | `WebMvcConfigurer` |
| 执行顺序 | 多个拦截器怎么排 |

> 📌 拦截器是 [[SpringMVC工作原理]] 流程的一部分，常配合 [[全局异常处理]] 使用。

### 二、拦截器 vs 过滤器（Filter）

Java Web 里有两种「拦截」机制，容易混淆：

| 对比 | 拦截器（Interceptor） | 过滤器（Filter） |
| --- | --- | --- |
| 来源 | **Spring** 机制 | **Servlet** 规范 |
| 作用范围 | 只拦截进入 Controller 的请求 | 拦截**所有**请求（含静态资源） |
| 能否访问 Spring 上下文 | ✅ 能（可注入 Bean） | ❌ 较难 |
| 触发时机 | Controller **前后** | Servlet **前后**（更靠前） |
| 典型场景 | 登录校验、权限、业务日志 | 字符编码、跨域、请求包装 |

**执行顺序**（过滤器在外层，拦截器在内层）：

```
请求 ──► Filter（前）──► Interceptor.preHandle ──► Controller
                                                      │
响应 ◄── Filter（后）◄── Interceptor.postHandle ◄────┘
              ◄── Interceptor.afterCompletion
```

> 💡 **提示：** 业务相关的拦截（登录、权限）用 **拦截器**；和具体业务无关的通用处理（编码、跨域）用 **过滤器**。

### 三、`HandlerInterceptor` 的三个方法

实现 `HandlerInterceptor` 接口，重写三个方法：

| 方法 | 时机 | 用途 | 返回值 |
| --- | --- | --- | --- |
| `preHandle` | Controller **执行前** | 登录校验、权限检查 | `false` 则**中断**，`true` 放行 |
| `postHandle` | Controller **执行后**、视图渲染前 | 修改 Model | `void` |
| `afterCompletion` | **请求完成**（含异常） | 资源清理、记日志 | `void` |

```java
public class LoginInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse resp, Object handler) {
        // Controller 执行前：检查登录
        String token = req.getHeader("Authorization");
        if (token == null) {
            resp.setStatus(401);
            return false;                 // ✅ 返回 false 中断请求，Controller 不执行
        }
        return true;                      // 放行
    }

    @Override
    public void postHandle(HttpServletRequest req, HttpServletResponse resp, Object handler, ModelAndView mv) {
        // Controller 执行后（前后端分离一般用不到，因为直接返回 JSON）
    }

    @Override
    public void afterCompletion(HttpServletRequest req, HttpServletResponse resp, Object handler, Exception ex) {
        // 请求结束，可用于记录耗时、清理资源
    }
}
```

> ⚠️ **注意：** `preHandle` 返回 `false` 后，`postHandle` **不会执行**，但 `afterCompletion` **会执行**（已执行的拦截器的）。

### 四、注册拦截器

光写拦截器类还不够，要在配置里**注册**它，并指定拦截哪些路径：

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Autowired
    private LoginInterceptor loginInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(loginInterceptor)
                .addPathPatterns("/api/**")           // 拦截 /api/ 下所有请求
                .excludePathPatterns("/api/login");   // 排除登录接口（否则登录都进不来）
    }
}
```

| 方法 | 作用 |
| --- | --- |
| `addInterceptor(...)` | 注册一个拦截器 |
| `addPathPatterns(...)` | 拦截哪些路径（如 `/api/**`） |
| `excludePathPatterns(...)` | 排除哪些路径（如登录、公开接口） |
| `order(n)` | 设置优先级（数字小先执行） |

### 五、实战：登录 Token 校验拦截器

一个完整的、生产可用的登录拦截器：

```java
@Component
public class AuthInterceptor implements HandlerInterceptor {

    @Autowired
    private JwtUtil jwtUtil;   // 注入 Spring Bean，过滤器做不到这点

    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse resp, Object handler) throws IOException {
        // 1. 取 token
        String token = req.getHeader("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            writeError(resp, 401, "未登录");
            return false;
        }

        // 2. 校验 token
        try {
            String jwt = token.substring(7);
            Long userId = jwtUtil.parseUserId(jwt);   // 解析失败会抛异常
            req.setAttribute("userId", userId);       // 放进请求，Controller 可取
            return true;
        } catch (Exception e) {
            writeError(resp, 401, "token 无效或已过期");
            return false;
        }
    }

    private void writeError(HttpServletResponse resp, int code, String msg) throws IOException {
        resp.setStatus(code);
        resp.setContentType("application/json;charset=UTF-8");
        resp.getWriter().write("{\"code\":" + code + ",\"message\":\"" + msg + "\"}");
    }
}
```

注册时排除登录、注册等公开接口：

```java
registry.addInterceptor(authInterceptor)
        .addPathPatterns("/**")
        .excludePathPatterns("/login", "/register", "/swagger-ui/**");
```

### 六、多个拦截器的执行顺序

注册多个拦截器时，顺序是个重点：

```
假设注册了 拦截器A、拦截器B（按注册顺序）：

preHandle：A → B            （正序，依次放行）
   │
   ▼ Controller 执行
postHandle：B → A           （倒序！）
   │
   ▼
afterCompletion：B → A      （倒序！）
```

```
preHandle(A) ─► preHandle(B) ─► Controller
                                      │
afterCompletion(B) ◄─ postHandle(B) ◄─┤
afterCompletion(A) ◄─ postHandle(A) ◄─┘
```

> 💡 **记忆：** `preHandle` 像「**洋葱皮的外层→内层**」（先进），`postHandle`/`afterCompletion` 像「**内层→外层**」（后进先出）。和过滤器的执行顺序规律一致。

> ⚠️ **注意：** 如果 `preHandle(B)` 返回 `false`，则它**前面**已通过的拦截器（A）的 `afterCompletion` **仍会执行**，但 B 自己和后面的都不执行。

### 七、实际应用场景

1. **登录 / Token 校验**：最经典的用途，拦截需要登录的接口。
2. **权限控制**：校验当前用户是否有权访问该资源。
3. **接口耗时日志**：`preHandle` 记开始时间，`afterCompletion` 算耗时。
4. **限流**：在 `preHandle` 里做简单的接口限流。

### 八、常见问题与注意事项

> ⚠️ **注意：**
> - 拦截器**只拦截进入 Controller 的请求**；静态资源默认不经过 Controller，一般不会被拦截（除非配置了 `/**`）。
> - **`preHandle` 返回 false 后记得自己写响应**（如返回 401 JSON），否则客户端拿到空响应。
> - 拦截器里抛出的异常，能被 [[全局异常处理]] 的 `@RestControllerAdvice` 捕获吗？**可以**，因为发生在 DispatcherServlet 处理过程中。但若在更早的 Filter 抛异常，`@RestControllerAdvice` 捕获不到。

> 💡 **提示：** 拦截器、过滤器、AOP（见 [[../Spring框架核心/AOP面向切面]]）都能做「方法增强」，区别在「**作用范围和层级**」：过滤器（Servlet 层，最广）> 拦截器（Controller 层）> AOP（任意 Spring Bean 方法）。

---

### 九、总结

- **拦截器作用在 Controller 前后**，基于 Spring 机制，能访问 Spring 上下文。
- **三个方法**：`preHandle`（前，可中断）、`postHandle`（后）、`afterCompletion`（完成，含异常）。
- **注册**：`WebMvcConfigurer.addInterceptors`，配 `addPathPatterns`/`excludePathPatterns`。
- **多拦截器**：`preHandle` 正序，`postHandle`/`afterCompletion` 倒序。
- **层级关系**：过滤器（外）> 拦截器（中）> AOP（内）。
