### 一、概述

> 📖 [Nacos 官方文档](https://nacos.io/zh-cn/docs/v2/quickstart/quick-start.html) ｜ [Spring Cloud Netflix Eureka](https://github.com/Netflix/eureka)（已停更）

**服务注册与发现** 是微服务的「**通讯录**」——服务启动时把自己的地址登记上去，需要调用别人时去通讯录里查。解决了「服务实例 IP 动态变化、调用方怎么找到服务方」的问题。

大白话：没有注册中心时，订单服务想调用户服务，得把用户服务的 IP 写死在配置里——可用户服务一重启/扩容 IP 就变，全得改。注册中心像「**公司通讯录**」：每个员工（服务）报到时登记座位号，要找人时查通讯录，人换了座位通讯录自动更新。

| 实战 | 服务怎么注册、怎么发现 |

### 二、为什么需要注册中心

单体应用里，所有模块在一个进程内，直接方法调用。微服务里，服务在不同机器上，得通过网络调用，**问题是怎么拿到对方的地址**：

| 方案 | 问题 |
| --- | --- |
| 写死 IP | 服务重启/扩容 IP 变，全得改；无法负载均衡 |
| DNS | 切换慢、不够灵活、不知道服务健康状态 |
| **注册中心** | **服务自动注册、自动发现、自动剔除故障实例** ✅ |

注册中心的工作模式（**C/S 架构**）：

```
① 服务提供方启动 → 向注册中心「注册」自己的地址
② 服务消费方启动 → 从注册中心「拉取」服务列表（订阅）
③ 提供方变化（扩容/宕机）→ 注册中心通知消费方更新
④ 消费方调用时 → 从本地缓存的服务列表里选一个地址调用（负载均衡）
```

### 三、Eureka 架构（Netflix）

Eureka 是 Netflix 开源的注册中心，经典的「**AP 模式**」注册中心。

```
        ┌─────────────────────┐
        │    Eureka Server     │  ← 注册中心（独立部署）
        │   （服务注册表）       │
        └──┬───────────┬──────┘
     注册 │           │ 注册
   心跳续约│           │心跳续约
           ▼           ▼
   ┌──────────┐   ┌──────────┐
   │ Provider  │   │ Consumer  │
   │（服务提供方）│   │（服务消费方）│
   └──────────┘   └─────┬────┘
                         │ 拉取服务列表（缓存到本地）
                         ▼
                   调用 Provider（负载均衡选一个）
```

| 机制 | 说明 |
| --- | --- |
| **服务注册** | 服务启动时把自己的 IP+端口发给 Server |
| **心跳续约** | 服务每 30s 发一次心跳，告诉 Server「我还活着」 |
| **服务剔除** | 90s 没收到心跳，Server 认为服务挂了，从注册表移除 |
| **服务发现** | Consumer 拉取注册表并缓存本地，定期更新 |
| **自我保护** | Server 短时间内丢失大量心跳时，**不再剔除**（宁可保留可能失效的，也不误删）——AP 思想 |

> 💡 **提示：** Eureka 的**自我保护机制**体现了 AP 原则——宁可返回可能过时的服务列表（可用），也不因网络抖动误删正常服务。但 Eureka 已停止维护，新项目建议用 Nacos。

### 四、Nacos（Alibaba，推荐）

**Nacos**（Naming and Configuration Service）是阿里开源的，**注册中心 + 配置中心二合一**，国内微服务的事实标准。

| 特性 | 说明 |
| --- | --- |
| 一站式 | 注册中心 + 配置中心，部署一个 Nacos 全搞定 |
| AP/CP 可切换 | 默认 AP（像 Eureka），可切 CP（像 Zookeeper） |
| 控制台 | 自带 Web 控制台，可视化看服务列表、配置 |
| 健康检查 | 支持服务端探测 + 客户端心跳两种模式 |

#### 与 Eureka 对比

| 对比 | Eureka | Nacos |
| --- | --- | --- |
| CAP 模式 | 仅 AP | AP / CP 可切换 |
| 一致性协议 | — | Raft（CP）/ Distro（AP） |
| 配置中心 | 无，要配 Spring Cloud Config | **自带** |
| 控制台 | 简陋 | **友好** |
| 维护状态 | **停更** | 活跃 |

### 五、实战：Nacos 服务注册与发现

#### 1. 启动 Nacos Server

下载 Nacos 安装包，单机模式启动（默认端口 8848）：

```bash
sh startup.sh -m standalone
```

访问 `http://localhost:8848/nacos`（默认账号密码 nacos/nacos）。

#### 2. 服务提供方：注册到 Nacos

```xml
<!-- 依赖 -->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

```yaml
# application.yml
spring:
  application:
    name: user-service            # 服务名（消费方靠它找服务）
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848   # Nacos 地址
```

```java
@SpringBootApplication
public class UserServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(UserServiceApplication.class, args);
    }
}
// 启动后，user-service 自动注册到 Nacos（新版无需 @EnableDiscoveryClient）
```

#### 3. 服务消费方：发现并调用

```yaml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
```

```java
@Service
public class OrderService {

    @Autowired
    private DiscoveryClient discoveryClient;   // Spring Cloud 提供的发现客户端

    public void callUser() {
        // 按服务名拿到所有实例（多个实例 → 负载均衡选一个）
        List<ServiceInstance> instances = discoveryClient.getInstances("user-service");
        ServiceInstance instance = instances.get(0);
        String url = "http://" + instance.getHost() + ":" + instance.getPort() + "/user/1";

        // 用 RestTemplate 调用（简化演示，实际用 OpenFeign）
        User user = restTemplate.getForObject(url, User.class);
    }
}
```

> 💡 **提示：** 实际开发不会像上面这样手动选实例 + 拼 URL——太麻烦。配合 [[服务间调用OpenFeign]]，用「**服务名**」直接声明式调用，负载均衡自动完成。

### 六、服务发现的调用流程

```
① user-service 启动 → 注册到 Nacos（user-service → 192.168.1.10:8081）
② user-service 扩容 → 新实例也注册（user-service → 192.168.1.11:8081）
③ order-service 启动 → 从 Nacos 订阅 user-service，缓存实例列表到本地
④ order-service 调用 user-service：
   - 从本地缓存的实例列表里选一个（负载均衡）
   - 直接 HTTP 调用该实例
⑤ 某实例宕机 → Nacos 收不到心跳 → 剔除 → 通知 order-service 更新本地缓存
```

整个过程**对业务代码透明**，调用方只认「服务名」。

### 七、常见问题与注意事项

> ⚠️ **注意：**
> - **服务名要唯一**：消费方靠服务名找服务，重名会导致调错。
> - **健康检查**：服务优雅停机时要主动注销，否则要等心跳超时才剔除（有延迟）。
> - **本地缓存**：消费方会缓存服务列表，Nacos 短暂挂掉时仍能用缓存的列表调用（AP 优势），但新建/宕机实例短期感知不到。
> - **跨命名空间/分组访问**：Nacos 用 Namespace（环境隔离）+ Group（业务分组）+ dataId 三层隔离，跨层访问会找不到服务。

> 💡 **提示：** 注册中心解决的是「**找到服务**」的问题，调用时怎么负载均衡、怎么熔断兜底，是 [[服务间调用OpenFeign]]、[[熔断降级]] 的事。

### 八、总结

- **注册中心 = 微服务通讯录**，服务自动注册、自动发现、自动剔除故障实例。
- **Eureka**（Netflix，AP，停更）是经典学习案例；**Nacos**（Alibaba，AP/CP 可切，自带配置中心）是国内主流。
- **工作流程**：提供方注册 → 消费方订阅拉取 → 调用时负载均衡选实例。
- 消费方只认**服务名**，IP 变化对业务透明。
- Nacos 用 **Namespace + Group + dataId** 做环境/业务隔离。
