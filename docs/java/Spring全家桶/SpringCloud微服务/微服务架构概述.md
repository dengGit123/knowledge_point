### 一、概述

> 📖 [Spring Cloud 官方](https://spring.io/projects/spring-cloud) ｜ [Spring Cloud Alibaba](https://sca.aliyun.com/) ｜ [Martin Fowler 谈微服务](https://martinfowler.com/articles/microservices.html)

**微服务（Microservices）** 是一种架构风格——把一个大型单体应用，拆分成多个**小而独立**的服务，每个服务负责单一业务，独立部署、独立演进。**Spring Cloud** 则是 Java 生态下构建微服务的一整套工具集。

大白话：单体架构像「**一栋什么都塞在里面的大楼**」（所有功能挤在一起）；微服务像「**一个小区**」——住宅楼、超市、物业各自独立，各干各的，但要有人协调（注册中心、网关、配置中心等），这个「物业系统」就是 Spring Cloud。

### 二、单体 vs 微服务

| 对比 | 单体架构 | 微服务架构 |
| --- | --- | --- |
| 部署 | 一个应用打包部署 | 拆成多个服务，**各自独立部署** |
| 扩展 | 整体扩（为一个小模块也得复制整套） | **按需扩**（哪个服务压力大就扩哪个） |
| 技术栈 | 统一一种 | 各服务**可自由选型** |
| 团队协作 | 全在一个代码库，易冲突 | **服务即边界**，团队各管一块 |
| 复杂度 | 低 | **高**（分布式带来的各种难题） |
| 运维 | 一个应用好管 | 几十上百个服务，**运维成本陡增** |
| 数据一致性 | 单库事务即可 | **分布式事务**很难 |

> 💡 **提示：** 微服务**不是银弹**。它用「运维复杂度」换取「扩展性和团队效率」。小项目（几个人、几万行代码）用单体更划算，**别为了微服务而微服务**。

### 三、微服务带来的核心问题

应用一旦拆开，立刻冒出一堆单体时代不存在的问题：

| 问题 | 说明 | Spring Cloud 的解法 |
| --- | --- | --- |
| **服务怎么互相找到** | 服务实例 IP 动态变化（重启、扩容） | **注册中心**（服务注册与发现） |
| **配置怎么统一管理** | 几十个服务，配置散落各处 | **配置中心**（集中配置 + 动态刷新） |
| **外部请求怎么进来** | 外部不该直连内部各服务 | **API 网关**（统一入口、路由、鉴权） |
| **调用失败怎么办** | 一个慢调用可能拖垮整条链路（雪崩） | **熔断降级** + 限流 |
| **服务间怎么调用** | 不想手写 HTTP、手写负载均衡 | **声明式调用**（OpenFeign） |
| **调用链怎么追踪** | 请求跨多个服务，出问题难定位 | **链路追踪**（Sleuth + Zipkin） |
| **分布式事务怎么办** | 跨服务的数据一致性 | **分布式事务**（Seata） |

### 四、Spring Cloud 是什么

**Spring Cloud 不是一个框架，而是一整套微服务治理工具的「**集合**」**。它建立在 [[../SpringBoot/SpringBoot入门]] 之上——每个微服务，本质上都是一个 Spring Boot 应用。

```
Spring Boot（让单个服务快速开发、独立运行）
        │
        │  + 微服务治理能力
        ▼
Spring Cloud（服务注册、配置中心、网关、熔断、调用……）
        │
        ▼
微服务集群（几十个 Spring Boot 应用协同工作）
```

> ⚠️ **注意：** Spring Cloud **依赖** Spring Boot，两者**版本必须匹配**。它本身不提供具体实现，而是定义标准接口，由不同的「实现」（Netflix、Alibaba）来填充。

### 五、两套主流生态

Spring Cloud 的标准接口下，有两套主流实现：

| | Spring Cloud Netflix | Spring Cloud Alibaba |
| --- | --- | --- |
| 注册中心 | Eureka | **Nacos**（注册+配置一体） |
| 配置中心 | Spring Cloud Config | **Nacos Config** |
| 熔断降级 | Hystrix（**已停更**） | **Sentinel** |
| 服务调用 | Feign / Ribbon | OpenFeign / LoadBalancer |
| 网关 | Zuul（**已停更**） | Spring Cloud Gateway（官方） |
| 分布式事务 | — | **Seata** |
| 消息队列 | — | RocketMQ |
| 现状 | 多数组件进入维护模式 | **国内主流、活跃** |

> 💡 **提示：**
> - **国内新项目首选 Spring Cloud Alibaba**（Nacos + Sentinel + Seata），生态完整、文档中文友好、阿里在维护。
> - Netflix 体系（Eureka/Hystrix/Zuul）部分已停更，但**概念是通用的**，面试仍会问，学习价值在。

#### 新一代官方组件（取代 Netflix）

| Netflix 老组件 | 新组件（官方推荐） |
| --- | --- |
| Ribbon（负载均衡） | **Spring Cloud LoadBalancer** |
| Hystrix（熔断） | **Resilience4j**（或 Sentinel） |
| Zuul（网关） | **Spring Cloud Gateway** |
| Feign | **Spring Cloud OpenFeign**（仍活跃） |

### 六、版本对应关系

> ⚠️ **重要：** Spring Cloud 和 Spring Boot 版本**必须严格对应**，否则一堆莫名其妙的错。

| Spring Cloud 版本 | 对应 Spring Boot |
| --- | --- |
| 2023.x | Boot 3.2.x |
| 2022.x | Boot 3.0.x / 3.1.x |
| 2021.x | Boot 2.6.x / 2.7.x |
| 2020.x | Boot 2.4.x / 2.5.x |

> 💡 **提示：** Spring Cloud 版本号是「**发布列车名（Release Train）**」（如 2023.0.x），不像 Boot 那样的语义化版本。具体对应关系查 [Spring Cloud Release Train](https://spring.io/projects/spring-cloud#release-trains) 官方表格。**Spring Cloud Alibaba** 还要看它支持哪个 Cloud 版本。

### 七、微服务组件全景图

把前面的问题和解法拼成一张全景图：

```
                          外部请求（浏览器/APP）
                                │
                                ▼
                        ┌───────────────┐
                        │   API 网关     │  ← 统一入口、路由、鉴权、限流
                        │ (Gateway)     │
                        └───────┬───────┘
                                │ 路由转发
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
        ┌──────────┐      ┌──────────┐      ┌──────────┐
        │ 订单服务  │◄────►│ 用户服务  │◄────►│ 商品服务  │  ← 各服务独立部署
        └────┬─────┘ OpenFeign ┌─────┴────┘      └────┬─────┘
             │                  │                    │
             │ 都向注册中心注册、从注册中心拉取服务列表
             │                  │                    │
             └──────────────────┼────────────────────┘
                                │
                   ┌────────────┼─────────────┐
                   ▼            ▼             ▼
             ┌──────────┐ ┌──────────┐  ┌──────────┐
             │ 注册中心  │ │ 配置中心  │  │ 熔断/限流 │
             │Nacos/    │ │ Nacos    │  │ Sentinel │
             │Eureka    │ │          │  │          │
             └──────────┘ └──────────┘  └──────────┘
```

一个典型微服务请求链路：

```
用户 → 网关（Gateway，鉴权+路由）→ 目标服务
        → 该服务用 OpenFeign 调用其他服务（从 Nacos 拿地址，Sentinel 熔断兜底）
        → 配置都从 Nacos Config 拉取
```

### 八、常见问题与注意事项

> ⚠️ **注意：**
> - **别过早微服务化**：项目初期不确定边界，先用单体快速迭代，等业务稳定、团队规模上来再拆。
> - **拆得太细也是灾难**：每个服务都有部署、监控、运维成本，服务太多反而拖累。
> - **分布式事务成本极高**：能用最终一致性就别强求强一致性（Seata 也有性能代价）。
> - **本地开发调试难**：几十个服务起不起来？用 Mock、或只起核心服务。

> 💡 **提示：** 学微服务的最佳路径：**先懂单体里的 Spring Boot / MVC / Data（本知识库前面的章节），再学微服务的「治理」**。微服务本质上就是「多个 Boot 应用 + 一套协调机制」。

---

### 九、总结

- **微服务 = 把单体拆成多个独立小服务**，换取扩展性和团队效率，代价是运维复杂度。
- **微服务的核心难题**：注册发现、配置、网关、熔断、调用、追踪、分布式事务。
- **Spring Cloud = 微服务治理工具集**，建立在 Spring Boot 之上，版本需严格对应。
- **两套生态**：Netflix（部分停更）vs Alibaba（国内主流）。
- **选型**：国内新项目用 **Spring Cloud Alibaba（Nacos + Sentinel + Seata）**。
