### 一、概述

> 📖 [Nacos Config 官方文档](https://nacos.io/zh-cn/docs/v2/quickstart/quick-start.html) ｜ [Spring Cloud Config](https://docs.spring.io/spring-cloud-config/reference/)

**配置中心** 把分散在各个微服务里的配置**集中管理**，并支持**动态刷新**——改个配置不用重启服务，在线生效。

大白话：单体应用一个 `application.yml` 就够了。微服务有几十上百个实例，每个都有自己的配置（数据库、限流阈值），散落在各处。改个数据库密码，要改几十个文件、重启几十个服务——崩溃。配置中心像「**统一配置仓库**」：所有服务从仓库读配置，改仓库就全网生效，还不用重启。

| 你将学到 | 说明 |
| --- | --- |
| 为什么需要配置中心 | 集中管理 + 动态刷新 |
| 两种方案 | Spring Cloud Config vs Nacos Config |
| Nacos Config 实战 | 怎么接、怎么动态刷新 |
| 隔离模型 | Namespace / Group / dataId |

> 📌 本文承接 [[服务注册与发现]]（Nacos 注册+配置一体），是 [[微服务架构概述]] 的一部分。

### 二、为什么需要配置中心

| 痛点 | 配置中心怎么解决 |
| --- | --- |
| 配置散落几十个服务，难统一管理 | **集中存储**，一处管理 |
| 改配置要改文件 + 重启服务 | **动态刷新**，在线生效不重启 |
| 不同环境（dev/test/prod）配置不同 | **环境隔离**（Namespace/Profile） |
| 敏感配置（密码）散在代码库 | 集中托管 + 权限控制 |
| 灰度发布要动态调参数 | 控制台实时改，立即生效 |

> 💡 **提示：** 配置中心 vs 本地配置（[[../SpringBoot/配置文件与Profile]]）：本地配置改了要重启；配置中心的核心增量是「**动态刷新**」。

---

### 三、Spring Cloud Config vs Nacos Config

| 对比 | Spring Cloud Config | Nacos Config |
| --- | --- | --- |
| 配置存储 | **Git 仓库**（或本地文件） | **Nacos 服务端**（带数据库） |
| 动态刷新 | 需配合 **Spring Cloud Bus** + 消息队列广播 | **自带推送**，开箱即用 |
| 注册中心 | 无（要另搭） | **自带**（注册+配置一体） |
| 控制台 | 无，得直接操作 Git | **自带 Web 控制台** |
| 部署 | 较重（Config Server + Bus + MQ） | 一个 Nacos 搞定 |
| 推荐 | 老项目沿用 | **国内新项目首选** ✅ |

### 四、Nacos Config 实战

#### 1. 在 Nacos 控制台新建配置

登录 Nacos 控制台 → 配置管理 → 新建配置：

| 项 | 值 | 说明 |
| --- | --- | --- |
| **dataId** | `user-service-dev.yaml` | `${服务名}-${环境}.${格式}` |
| **Group** | `DEFAULT_GROUP` | 业务分组 |
| **配置格式** | YAML | |
| **内容** | 你的配置 | 如数据库、限流参数 |

```yaml
# user-service-dev.yaml（写在 Nacos 控制台里）
server:
  port: 8081
db:
  url: jdbc:mysql://prod-mysql:3306/user
  max-size: 50
```

> 💡 **dataId 命名约定**：`${prefix}-${profile}.${file-extension}`。`prefix` 默认是 `spring.application.name`，`profile` 是环境。所以 `user-service-dev.yaml` 表示「user-service 服务的 dev 环境配置」。

#### 2. 服务端接入配置中心

```yaml
# application.yml（或 bootstrap.yml）
spring:
  application:
    name: user-service                  # 决定 dataId 的 prefix
  cloud:
    nacos:
      config:
        server-addr: localhost:8848     # Nacos 地址
        file-extension: yaml            # 配置格式
  config:
    import:                             # Spring Cloud 2020+ 推荐写法
      - optional:nacos:user-service-dev.yaml
```

> ⚠️ **注意：** Spring Cloud 2020.x（Boot 2.4+）起，默认不再读 `bootstrap.yml`，改用 `spring.config.import`。若仍想用 `bootstrap.yml`，需引入 `spring-cloud-starter-bootstrap` 依赖。

#### 3. 读取配置

和读本地配置完全一样：

```java
@Service
public class DbService {
    @Value("${db.url}")
    private String url;       // 从 Nacos 读到 jdbc:mysql://prod-mysql:3306/user
}
```

### 五、动态刷新：`@RefreshScope`

默认情况下，`@Value` 在**启动时绑定一次**，运行时改 Nacos 配置不会生效。要**动态刷新**，加 `@RefreshScope`：

```java
@Service
@RefreshScope                         // ✅ 加上这个注解
public class DbService {
    @Value("${db.max-size}")
    private int maxSize;              // Nacos 里改 db.max-size → 自动刷新（无需重启）
}
```

```
① 在 Nacos 控制台修改 db.max-size = 50 → 100
② Nacos 推送变更到 user-service
③ user-service 销毁 @RefreshScope 标记的 Bean
④ 下次访问时重建 Bean，max-size 已是 100
```

> 💡 **提示：** `@RefreshScope` 的原理是给 Bean 套一层**代理**，配置变更时销毁原 Bean、下次访问时重建——本质上也是 [[../Spring框架核心/AOP面向切面]] 代理的思路。只有标了 `@RefreshScope` 的 Bean 才会动态刷新，没标的仍是启动时的值。

### 六、配置优先级

Nacos 的配置和本地 `application.yml` 会合并，有优先级（高覆盖低）：

```
Nacos 中 profile 配置（user-service-dev.yaml）     ← 高
        │
Nacos 中默认配置（user-service.yaml，无 profile）   │
        │                                          ▼
本地 application.yml                              低
```

> 💡 **提示：** 即「**远程优先于本地**」「**带环境的优先于默认**」。本地 `application.yml` 通常只放引导配置（Nacos 地址、应用名），业务配置全放 Nacos。

### 七、隔离模型：Namespace + Group + dataId

Nacos 用**三层**结构组织配置（也用于服务隔离）：

| 层级 | 作用 | 典型用法 |
| --- | --- | --- |
| **Namespace（命名空间）** | 物理隔离 | dev / test / prod 环境 |
| **Group（分组）** | 逻辑归类 | 不同业务线（订单组、用户组），默认 `DEFAULT_GROUP` |
| **dataId** | 具体配置集 | `user-service-dev.yaml` |

```
Nacos
├── Namespace: dev
│     ├── Group: DEFAULT_GROUP
│     │     ├── dataId: user-service.yaml
│     │     └── dataId: order-service.yaml
│     └── Group: PAYMENT_GROUP
│           └── dataId: pay-service.yaml
└── Namespace: prod
      └── ...
```

> 💡 **提示：** 环境隔离用 **Namespace**（dev/test/prod 完全隔离，互不可见）；同环境内不同业务用 **Group**。命名空间要在 `spring.cloud.nacos.config.namespace` 里指定（填命名空间 ID）。

### 八、实际应用场景

1. **多环境配置**：dev/test/prod 各一个 Namespace，代码不变、配置不同。
2. **动态调参**：限流阈值、日志级别、线程池大小，运行时在控制台改，立即生效。
3. **灰度开关**：用配置项做功能开关（`feature.newPay.enabled=true`），按需开启。
4. **统一数据库配置**：所有微服务从 Nacos 读数据源配置，改一处全网生效。

### 九、常见问题与注意事项

> ⚠️ **注意：**
> - **`@RefreshScope` 别滥用**：每次刷新都重建 Bean，频繁刷新有开销。常量类、配置类用，业务核心类慎用。
> - **Nacos 宕机会怎样**：服务会**用本地缓存的快照**继续运行（Nacos Client 会缓存配置），不会立即崩，但改不了配置。
> - **Bootstrap 与 import**：Spring Cloud 2020+ 默认弃用 bootstrap，用 `spring.config.import`；老项目注意区分。
> - **敏感配置**：数据库密码等，Nacos 也支持加密配置或对接密钥管理（如 KMS）。

> 💡 **提示：** 配置中心 + 注册中心（[[服务注册与发现]]）是微服务的「**基础设施双雄**」。Nacos 把两者合一，是国内微服务部署的标配。

### 十、总结

- **配置中心 = 集中配置 + 动态刷新**，解决微服务配置散落、改配置要重启的痛点。
- **Nacos Config**（推荐，自带推送、注册中心一体）vs **Spring Cloud Config**（基于 Git，需 Bus）。
- **dataId 约定**：`${服务名}-${profile}.${格式}`。
- **动态刷新**：`@Value` + `@RefreshScope`，配置变更自动生效。
- **隔离模型**：Namespace（环境）+ Group（业务）+ dataId（配置集）。
