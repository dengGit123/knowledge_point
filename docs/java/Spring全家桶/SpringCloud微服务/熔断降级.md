### 一、概述

> 📖 [Sentinel 官方文档](https://sentinelguard.io/zh-cn/docs/introduction.html) ｜ [Resilience4j](https://resilience4j.readme.io/)

**熔断降级** 是微服务里保护系统不「**雪崩**」的关键机制。当下游服务出现故障或响应变慢时，**主动切断调用、快速失败、返回兜底数据**，避免故障像雪球一样越滚越大，把整个集群拖垮。

大白话：家里电路负荷过大，保险丝会**自动跳闸**保护电器不被烧毁——这就是「熔断」。微服务里的熔断器也一样：发现下游服务挂了，就「跳闸」快速失败，不让调用方一直傻等，保护整条调用链。

| 你将学到 | 说明 |
| --- | --- |
| 雪崩效应 | 为什么需要熔断 |
| 熔断器状态机 | CLOSED/OPEN/HALF_OPEN |
| 熔断 vs 降级 vs 限流 | 三个易混概念 |
| 三种方案 | Hystrix / Sentinel / Resilience4j |

> 📌 本文配合 [[服务间调用OpenFeign]] 使用（Feign 失败时的兜底就靠它）。

### 二、为什么需要：雪崩效应

微服务调用是一条链：A → B → C。如果 C 挂了或变慢：

```
场景：订单服务 ─► 库存服务 ─► 仓库服务，仓库服务挂了

① 仓库服务挂掉，请求超时
② 库存服务调仓库一直等（请求线程被占住）
③ 大量请求堆积 → 库存服务的线程池被占满
④ 库存服务也响应不了 → 订单服务调它也超时
⑤ 订单服务线程池被占满 → 订单服务也挂
⑥ 整条链路全部瘫痪 = 雪崩
```

**根因**：一个下游故障，通过「**线程占用 + 级联超时**」向上蔓延。熔断降级就是**在故障蔓延前切断它**。

### 三、熔断器的三种状态

熔断器（Circuit Breaker）像个「**智能保险丝**」，有三个状态：

```
            错误率/慢调用率超阈值
   ┌─────────────────────────────┐
   │                             ▼
[CLOSED 关闭]                [OPEN 打开]
正常放行请求                 熔断中：直接快速失败
（统计错误）                 不再调用下游
   ▲                             │
   │                             │ 等待一段时间（如 10s）
   │ 试探成功                     ▼
   │                        [HALF_OPEN 半开]
   └─────────────────────── 放行少量试探请求
                                 │
                          ┌──────┴──────┐
                    试探成功          试探失败
                    → CLOSED          → OPEN
```

| 状态 | 行为 |
| --- | --- |
| **CLOSED（关闭）** | 正常放行，同时统计错误率/慢调用率 |
| **OPEN（打开）** | **熔断**，请求不往下走，直接走降级逻辑（快速失败） |
| **HALF_OPEN（半开）** | 放行少量请求试探下游是否恢复，成功则 CLOSED，失败则继续 OPEN |

> 💡 **核心思想：** 与其让大量请求傻等下游（拖垮自己），不如**主动快速失败**，给下游恢复的时间。

### 四、熔断、降级、限流（别混淆）

三个概念经常混着说，但有区别：

| 机制 | 触发条件 | 做什么 | 解决 |
| --- | --- | --- | --- |
| **熔断（Circuit Breaking）** | 下游错误率/慢调用率高 | 切断调用，快速失败 | 下游故障的自我保护 |
| **降级（Fallback/Degradation）** | 调用失败/熔断时 | 返回兜底数据（默认值/缓存） | 失败时仍给可用响应 |
| **限流（Rate Limiting）** | 请求速率超阈值 | 拒绝多余请求 | 防止自己被流量压垮 |

```
限流：入口控制流量（"一次只能进 100 人"）
熔断：故障时切断调用（"这条路塌方了，别走了"）
降级：失败时给替代方案（"走不了原路，给你走小路/返回默认"）

实际中三者配合：限流防压垮 → 熔断防雪崩 → 降级保可用
```

> 💡 **提示：**

> - **国内新项目首选 Sentinel**（功能全 + 有可视化控制台）。
> - **Resilience4j** 是 Hystrix 的轻量继任者，函数式风格，Spring 官方推荐，国外更流行。

### 六、Sentinel 实战

#### 1. 引入依赖

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

```yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080    # Sentinel 控制台地址
```

#### 2. 用注解定义资源 + 降级

```java
@Service
public class OrderService {

    @SentinelResource(value = "createOrder",        // 资源名（控制台配规则用它）
                       blockHandler = "blockHandler",     // 限流/熔断时走这个方法
                       fallback = "fallback")             // 调用异常时走这个方法
    public Order createOrder(Long userId) {
        return orderClient.create(userId);   // 可能远程调用失败
    }

    // 限流/熔断兜底（方法签名要一致 + BlockException 参数）
    public Order blockHandler(Long userId, BlockException ex) {
        return new Order(-1L, "系统繁忙，请稍后再试");    // 限流了，返回友好提示
    }

    // 调用异常兜底
    public Order fallback(Long userId, Throwable e) {
        return new Order(-1L, "创建失败，使用默认订单");  // 出错了，返回兜底
    }
}
```

#### 3. 控制台配流控规则

打开 Sentinel 控制台，对资源 `createOrder` 配置：

| 规则类型 | 含义 |
| --- | --- |
| 流控规则 | 限制 QPS 或并发线程数 |
| 熔断规则 | 慢调用比例/异常比例超阈值时熔断 |
| 热点规则 | 对某个参数单独限流（如对热门商品 id 限流） |
| 系统规则 | 按 CPU、Load 等系统指标限流 |

> 💡 **Sentinel 限流的几种策略**：
> - **阈值类型**：QPS（每秒请求数）/ 并发线程数。
> - **流控行为**：直接拒绝（快速失败）/ Warm Up（预热，逐步放开）/ 排队等待（匀速通过）。

### 七、Hystrix 现状说明

Hystrix 是熔断降级概念的鼻祖，但**已停止更新**：

- 不建议新项目使用。
- 但其**核心思想（熔断状态机、舱壁隔离、降级 fallback）** 仍是面试重点，所有方案（Sentinel/Resilience4j）都借鉴了它。
- 代码里 `@HystrixCommand(fallbackMethod="xxx")` 的写法和 Sentinel 的 `@SentinelResource` 思路一致。

### 八、实际应用场景

1. **保护下游故障蔓延**：调用失败时熔断 + 降级，不让一个挂掉的服务拖垮调用方。
2. **接口限流**：秒杀、大促场景，控制 QPS 保护系统。
3. **热点参数限流**：对爆款商品 id 单独限流，不影响其他商品。
4. **Feign 降级**：[[服务间调用OpenFeign]] 的 `fallback`，远程调用失败返回默认值。

### 九、常见问题与注意事项

> ⚠️ **注意：**
> - **熔断时间别设太长**：OPEN 状态持续太久，下游恢复了你也不知道。合理设「半开试探」间隔。
> - **降级方法要轻量**：降级逻辑不能再依赖可能也挂掉的服务，最好返回本地默认值/缓存。
> - **降级 ≠ 业务正确**：降级返回的是兜底数据，要告知用户「数据可能不是最新」（如展示「热门推荐加载失败，请稍后」）。
> - **Sentinel 规则默认存内存**，重启丢失。生产要配**规则持久化**（Nacos/文件）。

> 💡 **提示：** 熔断降级是微服务「**容错**」的核心。原则是「**快速失败优于长时间等待**」——宁可给用户一个降级响应，也别让请求无限期挂起拖垮系统。

---

### 十、总结

- **雪崩**：一个下游故障，通过线程占用级联蔓延，拖垮整条链路。
- **熔断器三状态**：CLOSED（正常）→ OPEN（熔断快速失败）→ HALF_OPEN（试探恢复）。
- **三机制**：限流（防压垮）+ 熔断（防雪崩）+ 降级（保可用）。
- **三种方案**：Hystrix（停更，鼻祖）／ **Sentinel**（国内主流，有控制台）／ Resilience4j（官方推荐）。
- **铁律**：生产环境远程调用必配熔断 + 降级，别裸调。
