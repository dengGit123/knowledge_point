# agents/ 目录使用说明

## 什么是 agents/

`agents/` 目录用于存放**子代理（Subagents）**配置。每个子代理就像一个专业的"小助手"，专注于处理特定领域的任务。

> **通俗理解**：想象一个装修队，有电工、水工、油漆工等不同工种的工人。子代理就像这些专业工人，每个代理专注做一件事，比如"代码审查专家"、"测试专家"、"文档专家"等。

## 文件位置

```
项目/.claude/agents/*.md
全局/.claude/agents/*.md
```

## 为什么需要子代理

### 问题场景

有时候你希望 Claude 用特定的方式处理任务：
- 代码审查时按照特定的标准和格式
- 编写文档时使用统一的风格
- 性能优化时关注特定的指标

### 解决方案

通过子代理，你可以：
1. **专业化**：每个代理专注于一个领域
2. **标准化**：同一类任务始终使用相同的处理方式
3. **工具限制**：可以限制代理使用的工具（如只读不改）

## 文件格式

### 基本结构

```markdown
---
description: 代理的简短描述
model: claude-opus-4-7
tools: [Read, Edit, Bash]
timeout: 300000
---

# 代理名称

## 你的职责
<!-- 代理的职责描述 -->

## 工作方式
<!-- 如何完成任务 -->

## 输出格式
<!-- 结果应该怎样呈现 -->
```

### Frontmatter 字段说明

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| description | string | 代理的简短描述 | "代码审查专家" |
| model | string | 可选，指定使用的模型 | "claude-opus-4-7" |
| tools | string[] | 可选，允许使用的工具 | `[Read, Edit, Bash]` |
| disable | string[] | 可选，禁用的工具 | `[Write]` |
| timeout | number | 可选，超时时间（毫秒） | `300000`（5分钟） |

### 可用工具列表

| 工具 | 说明 |
|------|------|
| Read | 读取文件内容 |
| Write | 创建新文件或完全覆写文件 |
| Edit | 编辑文件的特定部分 |
| Bash | 执行命令行命令 |
| Agent | 调用其他代理 |
| WebSearch | 网络搜索 |
| AskUserQuestion | 向用户提问 |

## 如何使用子代理

### 1. 通过对话触发

当你提出的需求与某个代理的职责匹配时，会自动调用该代理：

```
# 这会触发代码审查代理
帮我审查一下这段代码

# 这会触发文档专家代理
帮我写一个 API 文档
```

### 2. 指定代理调用

你可以明确指定使用某个代理：

```
使用代码审查代理检查 src/auth.ts
```

## 常用代理示例

### 1. 代码审查代理

专注于发现代码问题和提供改进建议。

```markdown
---
description: 专业的代码审查专家
model: claude-opus-4-7
tools: [Read, Edit, Bash]
disable: [Write]
---

# 代码审查专家

## 你的职责

你是一个专业的代码审查专家，专注于发现代码问题和提供改进建议。

## 审查维度

### 1. 正确性
- 逻辑错误和边界条件
- 错误处理是否完善
- 并安全和线程安全

### 2. 安全性
- SQL 注入、XSS 等常见漏洞
- 敏感信息泄露风险
- 权限检查是否完整

### 3. 性能
- 算法效率和复杂度
- 资源使用和内存泄漏
- 缓存策略是否合理

### 4. 可维护性
- 代码是否清晰易懂
- 命名是否语义化
- 是否需要添加注释

## 输出格式

### ✅ 亮点
- 代码设计合理，使用了工厂模式
- 错误处理完善，考虑了边界情况

### ⚠️ 改进建议
- `getUserData` 函数建议添加缓存
- 建议将魔法数字提取为常量 `MAX_RETRY_COUNT = 3`

### ❌ 必须修复
- SQL 查询存在注入风险，需要参数化
- 缺少对 `user.id` 为空的处理
```

### 2. 测试专家代理

专注于设计测试策略和编写测试代码。

```markdown
---
description: 测试策略和实现专家
model: claude-opus-4-7
tools: [Read, Write, Edit, Bash]
---

# 测试专家

## 你的职责

你是一个测试专家，负责设计测试策略和编写测试代码。

## 测试类型

### 1. 单元测试
- 测试单个函数/组件
- 覆盖正常情况和边界情况
- Mock 外部依赖

### 2. 集成测试
- 测试模块间的交互
- 测试 API 接口
- 测试数据库操作

### 3. 端到端测试
- 测试完整的用户流程
- 验证业务逻辑
- 检查关键路径

## 测试原则

- **独立性**：每个测试应该独立运行
- **快速性**：测试应该快速执行
- **可维护**：测试代码应该清晰易懂
- **描述性**：测试名称应该说明测试内容

## 输出格式

```typescript
describe('用户服务', () => {
  describe('getUserInfo', () => {
    it('应该返回用户信息', async () => {
      // Given
      const userId = 1

      // When
      const result = await getUserInfo(userId)

      // Then
      expect(result).toBeDefined()
      expect(result.id).toBe(userId)
    })

    it('用户不存在时应该抛出错误', async () => {
      await expect(getUserInfo(999)).rejects.toThrow('用户不存在')
    })
  })
})
```
```

## 与 Skills 的区别

| 特性 | agents/ | skills/ |
|------|---------|---------|
| 独立性 | 完全独立的代理 | 共享主会话上下文 |
| 工具 | 可以限制可用工具 | 继承主会话的所有工具 |
| 提示 | 独立的系统提示 | 任务流程提示 |
| 用途 | 专业化任务处理 | 可重用的工作流程 |
| 通信 | 与主代理隔离 | 可以访问主会话信息 |

> **通俗理解**：
> - **agents** 是"请专家"：比如请一位代码审查专家来审查你的代码
> - **skills** 是"按流程"：比如按照"代码审查流程"一步步检查代码

## 最佳实践

### 1. 职责单一

每个代理只专注于一个领域：

```markdown
# ✅ 好的设计
- code-reviewer.md   # 只做代码审查
- security.md        # 只做安全检查
- performance.md      # 只做性能分析

# ❌ 不好的设计
- full-stack.md      # 什么都能做，太复杂
```

### 2. 清晰的职责定义

在代理文件中明确说明职责和输出格式：

```markdown
## 你的职责
<!-- 清晰描述代理做什么 -->

## 不做的内容
<!-- 明确说明代理不做的事情 -->
```

### 3. 合理的工具限制

根据任务需要限制工具：

```markdown
# 只读代理（安全审查）
tools: [Read]
disable: [Write, Edit]

# 完整代理（重构）
tools: [Read, Write, Edit, Bash]
```

### 4. 合适的模型选择

- 复杂任务使用 `claude-opus-4-7`
- 一般任务使用 `claude-sonnet-4-6`
- 简单任务使用 `claude-haiku-4-5`

### 5. 版本控制

代理定义应该提交到代码库，让团队共享：

```bash
# .gitignore
# 不忽略代理文件

# 提交代理
git add .claude/agents/
git commit -m "docs: 添加代码审查代理"
```

## 注意事项

1. **代理数量适中**：不要创建太多代理，3-5 个常用代理即可
2. **定期更新**：根据团队反馈调整代理配置
3. **测试效果**：创建新代理后测试其输出是否符合预期
4. **团队共享**：有效的代理配置应该提交到代码库
5. **避免重叠**：不同代理的职责应该明确区分

## 常见问题

### Q：代理和主会话有什么区别？

**A**：代理是独立的"子会话"，有自己的系统提示和工具限制。主会话可以调用代理来完成特定任务。

### Q：一个代理可以调用另一个代理吗？

**A**：可以。如果代理配置中允许使用 `Agent` 工具，它就可以调用其他代理。

### Q：代理可以访问主会话的上下文吗？

**A**：默认不可以。代理是独立的，只能访问传递给它的信息。

### Q：如何测试代理是否有效？

**A**：在实际场景中调用代理，观察其输出是否符合预期。必要时调整代理的提示内容。

### Q：代理配置文件应该提交到代码库吗？

**A**：应该。团队共享的代理配置应该提交，让所有成员使用一致的专业代理。
