# skills/ 目录使用说明

## 概述

`skills/` 目录用于存放可重用的技能文件，通过 `/skill-name` 调用或根据上下文自动触发。

## 文件位置

```
项目/.claude/skills/<name>/SKILL.md
全局/.claude/skills/<name>/SKILL.md
```

## 目录结构

```
.claude/skills/
├── review/
│   └── SKILL.md
├── test/
│   └── SKILL.md
├── deploy/
│   └── SKILL.md
└── docs/
    └── SKILL.md
```

## 技能文件格式

### 基本结构

```markdown
---
description: 代码审查技能
trigger:
  - "review"
  - "审查"
---

# 代码审查

请执行以下步骤：
1. 检查代码风格
2. 查找潜在 bug
3. 建议优化方案
4. 验证测试覆盖
```

### Frontmatter 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| description | string | 技能描述 |
| trigger | string[] | 触发关键词列表 |
| model | string | 可选，指定使用的模型 |
| timeout | number | 可选，超时时间（毫秒） |

## 使用方式

### 1. 斜杠命令调用

```
/review
```

### 2. 自动触发

当消息包含触发关键词时自动调用：

```
帮我审查一下这段代码
```

### 3. 参数传递

```
/deploy --env=production --skip-tests
```

## 技能示例

### 代码审查技能

```markdown
---
description: 审查代码变更
trigger: ["review", "审查", "code review"]
---

# 代码审查

## 审查步骤

1. **变更概览**
   - 查看所有变更的文件
   - 理解变更的目的

2. **代码质量**
   - 检查代码风格
   - 查找潜在 bug
   - 验证错误处理

3. **最佳实践**
   - 检查安全性问题
   - 验证性能考虑
   - 确认可维护性

4. **测试**
   - 检查测试覆盖
   - 验证测试质量

5. **文档**
   - 确认文档更新
   - 检查注释质量

## 输出格式

### ✅ 通过项
- 项目 1
- 项目 2

### ⚠️ 建议改进
- 问题 1
- 问题 2

### ❌ 必须修复
- 严重问题 1
- 严重问题 2
```

### 测试技能

```markdown
---
description: 运行和修复测试
trigger: ["test", "测试", "fix test"]
---

# 测试执行

## 步骤

1. 运行测试套件
2. 分析失败原因
3. 修复失败测试
4. 重新运行验证
5. 报告结果

## 处理逻辑

- 单个测试失败：修复该测试
- 多个测试失败：分析共同原因
- 超时失败：检查异步逻辑
- 断言失败：验证预期行为
```

### 部署技能

```markdown
---
description: 部署应用
trigger: ["deploy", "部署", "release"]
---

# 部署流程

## 前置检查

1. ✅ 所有测试通过
2. ✅ 代码已合并到主分支
3. ✅ 环境变量已配置
4. ✅ 数据库迁移已准备

## 部署步骤

1. 拉取最新代码
2. 安装依赖
3. 运行构建
4. 执行数据库迁移
5. 重启服务
6. 运行冒烟测试
7. 监控服务状态

## 回滚计划

如果部署失败：
1. 停止新版本
2. 恢复到上一版本
3. 检查错误日志
4. 修复问题后重试
```

生成 Markdown 格式的文档，保存到 `docs/api.md`

## 与 commands/ 的区别

| 特性 | skills/ | commands/ |
|------|---------|-----------|
| 目录结构 | `<name>/SKILL.md` | `*.md` |
| 支持文件 | 可以包含辅助文件 | 单文件 |
| 复杂度 | 适合复杂技能 | 适合简单命令 |
| 命名 | 目录名 | 文件名 |

## 最佳实践

1. **单一职责**：每个技能只做一件事
2. **清晰描述**：description 准确说明功能
3. **合理触发**：trigger 设置合适的关键词
4. **参数处理**：明确说明需要的参数
5. **错误处理**：包含错误情况的说明

## 注意事项

- 技能名称不能与内置命令冲突
- 触发词要具有辨识度
- 复杂技能考虑拆分
- 技能文件应该提交到代码库
- 更新技能时同步更新文档

## 技能开发工作流

```
1. 确定技能需求
2. 创建技能目录
3. 编写 SKILL.md
4. 测试技能功能
5. 提交到代码库
6. 团队共享使用
```
