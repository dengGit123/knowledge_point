# skills/ 目录使用说明

## 什么是 skills/

`skills/` 目录用于存放**可重用的技能文件**，通过 `/skill-name` 调用或根据上下文自动触发。

> **通俗理解**：就像给 Claude 定义"技能包"或"工作流程模板"。每个技能是一个完整的多步骤任务流程，比如"代码审查流程"、"部署流程"等。

## 文件位置

```
项目/.claude/skills/<name>/SKILL.md
全局/.claude/skills/<name>/SKILL.md
```

## 目录结构

```
.claude/skills/
├── review/
│   └── SKILL.md
├── test/
│   └── SKILL.md
├── deploy/
│   └── SKILL.md
└── docs/
    └── SKILL.md
```

## 为什么需要 skills/

### 问题场景

有些任务复杂且重复，每次都要详细说明：
- 代码审查：检查风格、查找 bug、验证测试、检查文档...
- 部署流程：测试、构建、上传、重启、验证...
- 文档生成：分析代码、提取接口、生成文档、检查格式...

### 解决方案

通过技能定义，这些任务只需一行：
- `/review` — 执行代码审查流程
- `/deploy` — 执行部署流程
- `/docs` — 生成文档

## 与 commands/ 的区别

| 特性 | skills/ | commands/ |
|------|---------|-----------|
| 目录结构 | `<name>/SKILL.md` | `*.md` |
| 支持文件 | 可包含辅助文件 | 单文件 |
| 复杂度 | 适合复杂技能 | 适合简单命令 |
| 命名 | 目录名 | 文件名 |

> **通俗理解**：
> - **commands** 是"快捷方式"：一键完成简单任务
> - **skills** 是"工作流程"：复杂的多步骤任务流程

## 文件格式

### 基本结构

```markdown
---
description: 技能的简短描述
trigger: ["触发关键词1", "触发关键词2"]
model: claude-opus-4-7
timeout: 300000
---

# 技能名称

## 技能说明
<!-- 技能的详细说明 -->

## 执行步骤
1. 第一步
2. 第二步
3. 第三步

## 输出格式
<!-- 结果应该怎样呈现 -->
```

### Frontmatter 字段

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| description | string | ✅ | 技能的简短描述 |
| trigger | string[] | ❌ | 触发关键词列表 |
| model | string | ❌ | 指定使用的模型 |
| timeout | number | ❌ | 超时时间（毫秒） |

## 如何使用技能

### 1. 斜杠命令调用

```
/review
/test
/deploy
```

### 2. 自然语言触发

如果设置了 `trigger`，也可以用自然语言触发：

```
# 触发 review 技能（如果 trigger 包含"审查"）
帮我审查一下这段代码

# 触发 deploy 技能（如果 trigger 包含"部署"）
准备部署到生产环境
```

### 3. 参数传递

```
/deploy --env=production --skip-tests
```

## 技能示例

### 1. 代码审查技能

```markdown
---
description: 审查代码变更
trigger: ["review", "审查", "code review"]
---

# 代码审查

## 审查步骤

### 1. 变更概览
- 查看所有变更的文件
- 理解变更的目的
- 确认变更范围

### 2. 代码质量
- 检查代码风格
- 查找潜在 bug
- 验证错误处理

### 3. 最佳实践
- 检查安全性问题
- 验证性能考虑
- 确认可维护性

### 4. 测试
- 检查测试覆盖
- 验证测试质量

### 5. 文档
- 确认文档更新
- 检查注释质量

## 输出格式

### ✅ 通过项
- 项目 1
- 项目 2

### ⚠️ 建议改进
- 问题 1
- 问题 2

### ❌ 必须修复
- 严重问题 1
- 严重问题 2
```

### 2. 测试技能

```markdown
---
description: 运行和修复测试
trigger: ["test", "测试"]
---

# 测试执行

## 步骤

### 1. 运行测试
- 运行完整测试套件
- 收集测试结果

### 2. 分析失败
- 定位失败原因
- 分类失败类型

### 3. 修复问题
- 修复代码
- 更新测试

### 4. 验证修复
- 重新运行测试
- 确认全部通过

## 处理逻辑
- 单个测试失败：修复该测试
- 多个测试失败：分析共同原因
- 超时失败：检查异步逻辑
```

### 3. 部署技能

```markdown
---
description: 部署应用
trigger: ["deploy", "部署", "release"]
---

# 部署流程

## 前置检查
- [ ] 所有测试通过
- [ ] 代码已合并到主分支
- [ ] 环境变量已配置

## 部署步骤
1. 拉取最新代码
2. 安装依赖
3. 运行构建
4. 执行数据库迁移
5. 重启服务
6. 运行冒烟测试

## 回滚计划
如果部署失败：
1. 停止新版本
2. 恢复到上一版本
3. 检查错误日志
```

### 4. 文档生成技能

```markdown
---
description: 生成 API 文档

# API 文档生成

## 内容

### 1. 概览
- API 用途
- 认证方式

### 2. 端点列表
- 方法 + 路径
- 描述
- 参数

### 3. 示例
- 请求示例
- 响应示例
```

## 命名规范

### 技能目录命名

使用小写字母和连字符：

```
# ✅ 好的命名
code-review/
run-tests/
deploy-prod/

# ❌ 不好的命名
CodeReview/
runTests/
deploy_prod/
```

### 技能文件

每个技能目录下必须有一个 `SKILL.md` 文件：

```
skills/
├── my-skill/
│   └── SKILL.md      # 必需文件
├── another-skill/
│   ├── SKILL.md      # 必需文件
│   └── config.json   # 可选辅助文件
```

## 最佳实践

### 1. 单一职责

每个技能只做一件事：

```
# ✅ 好的设计
review/     # 只做代码审查
test/       # 只做测试
deploy/     # 只做部署

# ❌ 不好的设计
everything/  # 什么都能做
```

### 2. 清晰描述

description 准确说明功能：

```markdown
---
description: 审查代码质量和安全性    # ✅ 清晰
description: 做一些审查工作          # ❌ 模糊
---
```

### 3. 合理触发

trigger 设置合适的关键词：

```markdown
---
trigger: ["review", "审查", "code review"]   # ✅ 多种表达
trigger: ["执行审查操作"]                  # ❌ 太正式
---
```

### 4. 参数处理

明确说明需要的参数：

```markdown
## 参数
- `--scope`：审查范围（all/changed/file）
- `--strict`：严格模式（true/false）

## 使用示例
/review --scope=changed --strict=true
```

### 5. 错误处理

包含错误情况的说明：

```markdown
## 错误处理
- 如果找不到文件，提示用户检查路径
- 如果测试失败，提供修复建议
- 如果构建错误，显示完整错误信息
```

## 选择建议

### 使用 skills/ 的场景

- 复杂的多步骤流程
- 需要配置或辅助文件
- 需要独立的文档说明
- 团队共享的复杂功能

### 使用 commands/ 的场景

- 简单的单功能命令
- 不需要辅助文件
- 快速定义和修改
- 频繁使用的操作

## 注意事项

1. **技能名称不能与内置命令冲突**
2. **触发词要具有辨识度**
3. **复杂技能考虑拆分**
4. **技能文件应该提交到代码库**
5. **更新技能时同步更新文档**

## 工作流程

### 创建新技能

```
1. 确定技能需求
   ↓
2. 创建技能目录
   ↓
3. 编写 SKILL.md
   ↓
4. 测试技能功能
   ↓
5. 提交到代码库
```

### 更新技能

```
1. 修改 SKILL.md
   ↓
2. 更新触发词（如需要）
   ↓
3. 测试变更
   ↓
4. 提交更新
```

## 常见问题

### Q：技能和代理有什么区别？

**A**：技能是可重用的工作流程，共享主会话上下文；代理是独立的子会话，有独立的系统提示。

### Q：技能可以调用其他技能吗？

**A**：可以。在技能描述中说明需要调用的其他技能。

### Q：技能修改后需要重启吗？

**A**：不需要。技能修改后立即生效。

### Q：如何测试技能是否工作？

**A**：直接使用 `/技能名` 测试，观察输出是否符合预期。

### Q：技能文件应该提交到代码库吗？

**A**：应该。技能是团队共享的工作流程，应该提交到代码库。
