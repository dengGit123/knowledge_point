# commands/ 目录使用说明

## 什么是 commands/

`commands/` 目录用于存放**自定义命令**，让你可以通过简单的斜杠命令（`/命令名`）快速执行常用的任务。

> **通俗理解**：就像给 Claude 定义"快捷指令"或"模板"。比如你经常需要"清理缓存"，就可以定义一个 `/cleanup` 命令，以后直接输入就能执行，不用每次都详细说明。

## 文件位置

```
项目/.claude/commands/*.md
全局/.claude/commands/*.md
```

## 为什么需要自定义命令

### 问题场景

有些任务你会反复做，每次都要详细说明：
- "帮我清理所有缓存文件，包括 node_modules/.cache、.next/cache 和 dist/"
- "帮我运行测试、检查类型、然后检查代码风格"
- "帮我构建项目、检查构建结果、报告文件大小"

### 解决方案

通过自定义命令，这些操作只需一行：
- `/cleanup` — 清理缓存
- `/check` — 完整检查
- `/build` — 构建项目

## 文件格式

### 基本结构

```markdown
---
description: 命令的简短描述
trigger: ["触发关键词1", "触发关键词2"]
---

# 命令标题

请执行以下操作：
1. 第一步
2. 第二步
3. 第三步
```

### Frontmatter 字段

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| description | string | ✅ | 命令的简短描述 |
| trigger | string[] | ❌ | 触发关键词列表，支持自然语言触发 |

## 如何使用命令

### 1. 斜杠命令

使用 `/命令名` 调用：

```
/cleanup
/build
/test
```

### 2. 自然语言触发

如果设置了 `trigger`，也可以用自然语言触发：

```
# 触发 cleanup 命令（如果 trigger 包含"清理"）
帮我清理一下缓存

# 触发 format 命令（如果 trigger 包含"格式化"）
把代码格式化一下
```

## 常用命令示例

### 1. 清理命令

清理缓存和临时文件。

```markdown
---
description: 清理缓存和临时文件
trigger: ["清理", "缓存", "cleanup"]
---

# 清理缓存

请执行以下清理操作：

## 1. 缓存目录
- 删除 `node_modules/.cache/`
- 删除 `.next/cache/`
- 删除 `dist/`
- 删除 `.vite/`

## 2. 临时文件
- 删除 `*.log` 文件
- 删除 `.DS_Store`
- 删除 `*.tmp` 文件

## 3. 确认清理
- 显示删除的文件列表
- 报告释放的磁盘空间
```

### 2. 格式化命令

格式化项目代码。

```markdown
---
description: 格式化代码
trigger: ["格式化", "format"]
---

# 代码格式化

请按以下步骤格式化代码：

## 1. 检查工具
- 确保 Prettier 已安装
- 检查 `.prettierrc` 配置文件

## 2. 执行格式化
- 运行 `npm run format` 或 `npx prettier --write .`
- 格式化所有源代码文件

## 3. 验证结果
- 检查修改的文件列表
- 确认没有语法错误
- 运行 `npm run lint` 确认通过
```

### 3. 构建命令

构建生产版本。

```markdown
---
description: 构建项目
trigger: ["构建", "build"]
---

# 项目构建

请执行以下构建流程：

## 前置检查
- [ ] 检查环境变量
- [ ] 确认依赖已安装

## 构建步骤
1. 清理旧的构建产物
2. 执行 `npm run build`
3. 检查构建输出目录

## 构建验证
- 确认构建成功
- 报告构建文件大小
- 检查是否有警告

## 错误处理
- 如果构建失败，显示错误信息
- 分析失败原因
- 提供修复建议
```

### 4. 检查命令

检查代码质量。

```markdown
---
description: 检查代码质量
trigger: ["检查", "check", "lint"]
---

# 代码质量检查

请执行以下检查：

## 1. 类型检查
- 运行 TypeScript 检查：`npm run type-check`
- 报告所有类型错误

## 2. 代码风格
- 运行 ESLint：`npm run lint`
- 报告风格问题

## 3. 测试覆盖
- 运行测试：`npm test`
- 显示测试覆盖率

## 输出格式
### ✅ 通过
- 类型检查通过
- 代码风格检查通过
- 测试覆盖率：85%

### ❌ 失败
- 类型错误：src/utils.ts:23 - 类型不匹配
- 风格问题：src/components/Header.vue - 缺少分号
- 测试失败：2 个测试用例失败
```

### 5. 发布命令

发布新版本。

```markdown
---
description: 发布新版本
trigger: ["发布", "release", "deploy"]
---

# 版本发布

## 前置检查
- [ ] 所有测试通过
- [ ] CHANGELOG 已更新
- [ ] 版本号已更新

## 发布步骤
1. 确认发布版本号
2. 运行完整测试套件
3. 构建生产版本
4. 创建 git tag
5. 推送到远程仓库
6. 触发 CI/CD 流程

## 发布后验证
- 检查部署状态
- 验证主要功能
- 监控错误日志

## 回滚计划
如果发布失败：
1. 停止新版本
2. 恢复到上一版本
3. 检查错误日志
4. 修复后重试
```

### 6. 测试命令

运行测试并处理失败。

```markdown
---
description: 运行测试并修复
trigger: ["测试", "test"]
---

# 测试执行

## 测试步骤
1. 运行完整测试套件
2. 分析失败原因
3. 修复失败测试
4. 重新运行验证
5. 报告最终结果

## 处理逻辑
- 单个测试失败：定位并修复该测试
- 多个测试失败：分析共同原因
- 超时失败：检查异步逻辑
- 断言失败：验证预期行为

## 输出格式
测试结果：
- 通过：45 个
- 失败：2 个
- 跳过：1 个
- 覆盖率：87%
```

## 与 skills/ 的区别

| 特性 | commands/ | skills/ |
|------|-----------|---------|
| 文件结构 | 单文件 `*.md` | 目录 `name/SKILL.md` |
| 支持文件 | 不支持 | 可包含配置、辅助文件 |
| 适用场景 | 简单单功能命令 | 复杂多步骤流程 |
| 复杂度 | 轻量级 | 重量级 |

> **通俗理解**：
> - **commands** 是"快捷方式"：一键完成简单任务
> - **skills** 是"工作流"：复杂的多步骤任务流程

## 命名规范

### 文件命名

使用小写字母和连字符：

```
# ✅ 好的命名
cleanup.md
code-format.md
run-tests.md

# ❌ 不好的命名
Cleanup.md
codeFormat.md
runTests.md
```

### 描述规范

简洁明了，说明命令做什么：

```markdown
---
description: 清理缓存文件        # ✅ 好的描述
description: 帮你清理各种缓存    # ❌ 不要用"帮你"这种口语
---
```

### 触发词选择

选择用户可能使用的自然词汇：

```markdown
---
trigger: ["清理", "cleanup", "清除"]  # ✅ 包含多种表达方式
trigger: ["执行清理操作"]              # ❌ 太正式
---
```

## 最佳实践

### 1. 命名清晰

文件名要能清楚描述命令功能：

```bash
# ✅ 好的命名
cleanup.md      # 清理
format.md       # 格式化
deploy.md       # 部署

# ❌ 不好的命名
cmd1.md         # 含义不明
do-it.md        # 不清晰
stuff.md        # 太随意
```

### 2. 步骤明确

命令执行步骤要清晰详细：

```markdown
# ✅ 好的命令
## 步骤
1. 第一步具体做什么
2. 第二步具体做什么
3. 第三步验证结果

# ❌ 不好的命令
## 步骤
- 做一些操作
- 然后做其他操作
- 最后检查
```

### 3. 错误处理

包含错误情况的处理说明：

```markdown
## 错误处理
- 如果命令执行失败，显示错误信息
- 分析失败原因
- 提供修复建议
- 不要忽略错误继续执行
```

### 4. 输出友好

提供清晰的输出格式：

```markdown
## 输出格式
### ✅ 成功
清理完成！
- 删除文件：23 个
- 释放空间：156MB

### ❌ 失败
清理失败：权限不足
请使用 `sudo` 重试
```

### 5. 合理拆分

复杂任务考虑拆分成多个命令：

```bash
# ✅ 合理拆分
build.md      # 构建
test.md       # 测试
deploy.md     # 部署

# ❌ 一个命令做太多
everything.md # 构建+测试+部署+发布
```

## 选择建议

### 使用 commands/ 的场景

- 简单的单功能命令
- 不需要辅助文件
- 快速定义和修改
- 频繁使用的操作

### 使用 skills/ 的场景

- 复杂的多步骤流程
- 需要配置或辅助文件
- 需要独立的文档说明
- 团队共享的复杂功能

## 注意事项

1. **文件名即命令名**：`cleanup.md` 对应 `/cleanup`
2. **避免内置冲突**：不要使用与内置命令冲突的名称
3. **提交到代码库**：命令文件应该提交，让团队共享
4. **同步更新文档**：更新命令时同步更新说明
5. **复杂命令升级**：过于复杂的命令考虑升级为 skill

## 常见问题

### Q：命令和内置命令有冲突怎么办？

**A**：自定义命令会覆盖同名内置命令。建议使用不同的名称避免冲突。

### Q：命令可以接受参数吗？

**A**：目前不支持参数传递。如果需要参数，考虑使用 skills/。

### Q：命令可以调用其他命令吗？

**A**：可以。在命令描述中可以说明需要调用的其他命令。

### Q：命令修改后需要重启吗？

**A**：不需要。命令修改后立即生效。

### Q：如何测试命令是否工作？

**A**：直接使用 `/命令名` 测试，观察输出是否符合预期。
