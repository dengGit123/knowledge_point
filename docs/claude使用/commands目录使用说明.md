# commands/ 目录使用说明

## 概述

`commands/` 目录用于存放单文件提示命令，与 skills 使用相同的机制，但更简单直接。

## 文件位置

```
项目/.claude/commands/*.md
全局/.claude/commands/*.md
```

## 与 skills/ 的区别

| 特性 | commands/ | skills/ |
|------|-----------|---------|
| 文件结构 | 单文件 `*.md` | 目录 `name/SKILL.md` |
| 支持文件 | 不支持 | 可包含辅助文件 |
| 适用场景 | 简单命令 | 复杂技能 |
| 命名 | 文件名 | 目录名 |

## 文件格式

### 基本结构

```markdown
---
description: 清理缓存文件
---

请执行以下操作：
1. 删除 node_modules/.cache
2. 删除 .next/cache
3. 清理构建产物
```

### Frontmatter 字段

| 字段 | 类型 | 说明 |
|------|------|------|
| description | string | 命令描述 |
| trigger | string[] | 触发关键词（可选） |

## 使用方式

### 1. 斜杠命令

```
/cleanup
```

### 2. 自然语言

如果设置了 trigger，也可以自然语言触发：

```
帮我清理一下缓存
```

## 命令示例

### 清理命令

```markdown
---
description: 清理缓存和临时文件
---

# 清理缓存

请执行以下清理操作：

1. **缓存目录**
   - rm -rf node_modules/.cache
   - rm -rf .next/cache
   - rm -rf dist/

2. **临时文件**
   - rm -rf *.log
   - rm -rf .DS_Store

3. **确认清理**
   - 显示清理结果
   - 报告释放的空间
```

### 格式化命令

```markdown
---
description: 格式化代码
---

# 代码格式化

请按以下步骤格式化代码：

1. **检查格式化工具**
   - 确保 prettier 已安装
   - 检查配置文件

2. **执行格式化**
   - 运行 prettier
   - 检查修改的文件

3. **验证**
   - 确认没有语法错误
   - 运行 lint 检查
```

### 构建命令

```markdown
---
description: 构建项目
---

# 项目构建

## 步骤

1. 清理旧的构建产物
2. 安装依赖
3. 执行构建
4. 检查构建结果
5. 报告构建大小

## 错误处理

- 如果构建失败，显示错误信息
- 分析失败原因
- 提供修复建议
```

### 检查命令

```markdown
---
description: 检查代码质量
---

# 代码质量检查

## 检查项

1. **类型检查**
   - 运行 TypeScript 检查
   - 报告类型错误

2. **Lint 检查**
   - 运行 ESLint
   - 报告风格问题

3. **测试覆盖**
   - 运行测试
   - 显示覆盖率

## 输出格式

- ✅ 通过项
- ❌ 失败项
- ⚠️ 警告项
```

### 发布命令

```markdown
---
description: 发布新版本
---

# 版本发布

## 前置条件

- [ ] 所有测试通过
- [ ] CHANGELOG 已更新
- [ ] 版本号已更新

## 发布步骤

1. 确认发布版本号
2. 运行完整测试套件
3. 构建生产版本
4. 创建 git tag
5. 推送到远程仓库
6. 触发 CI/CD 流程

## 发布后验证

- 检查部署状态
- 验证主要功能
- 监控错误日志
```

## 目录示例

```
.claude/commands/
├── build.md          # 构建命令
├── test.md           # 测试命令
├── lint.md           # 代码检查命令
├── format.md         # 格式化命令
├── cleanup.md        # 清理命令
├── deploy.md         # 部署命令
└── release.md        # 发布命令
```

## 最佳实践

1. **命名清晰**：文件名要能描述命令功能
2. **描述准确**： description 简洁说明用途
3. **步骤明确**：命令执行步骤要清晰
4. **错误处理**：包含错误情况的处理
5. **输出友好**：提供清晰的输出信息

## 选择建议

### 使用 commands/：

- 简单的单功能命令
- 不需要辅助文件
- 快速定义和修改

### 使用 skills/：

- 复杂的多步骤流程
- 需要配置或辅助文件
- 需要独立的文档说明
- 团队共享的复杂功能

## 注意事项

- 文件名即是命令名
- 不要与内置命令冲突
- 命令文件应该提交到代码库
- 更新命令时同步更新文档
- 复杂命令考虑升级为 skill
