# output-styles/ 目录使用说明

## 什么是 output-styles/

`output-styles/` 目录用于存放**自定义输出风格文件**，控制 Claude Code 的响应方式、输出格式和详细程度。

> **通俗理解**：就像给 Claude 设置"说话风格模板"，可以让它在不同场景下用不同方式回应——有时简洁直接，有时详细教学，有时像审计报告一样严谨。

## 文件位置

```
项目/.claude/output-styles/*.md
全局/.claude/output-styles/*.md
```

## 目录结构

```
.claude/output-styles/
├── concise.md       # 简洁风格
├── detailed.md      # 详细风格
├── teaching.md      # 教学风格
├── audit.md         # 审计风格
├── report.md        # 报告风格
└── debug.md         # 调试风格
```

## 为什么需要 output-styles/

### 问题场景

在不同场景下，需要不同的输出风格：
- 代码审查时：需要严谨的清单格式
- 学习新知识时：需要循序渐进的教学风格
- 快速问答时：需要简洁明了的回答
- 问题排查时：需要结构化的调试信息

### 解决方案

通过输出风格文件，可以：
- 预定义多种响应模板
- 匹配不同使用场景
- 保持输出格式一致
- 提高沟通效率

## 作用

| 作用 | 说明 |
|------|------|
| 自定义响应风格 | 控制 Claude 说话的语气和方式 |
| 控制输出格式 | 决定使用列表、表格、代码块等格式 |
| 调整详细程度 | 控制解释的详细程度 |
| 定义场景规范 | 为特定场景制定专属输出格式 |

## 文件格式

### 基本结构

```markdown
---
description: 简洁输出风格
trigger: ["简洁", "快速", "简短"]
---

请使用简洁的风格回应：
- 直接给出答案
- 避免冗余说明
- 使用列表形式
- 省略背景信息
```

### Frontmatter 字段

| 字段 | 类型 | 必需 | 说明 |
|------|------|------|------|
| description | string | ✅ | 输出风格的简短描述 |
| trigger | string[] | ❌ | 触发该风格的关键词列表 |

## 如何使用输出风格

### 1. 自动应用

根据文件名或 trigger 关键词自动加载对应的输出风格。

```markdown
# concise.md
---
description: 简洁输出风格
trigger: ["简洁", "快速", "简短"]
---

请使用简洁的风格回应...
```

当用户说"简洁地告诉我..."时，自动应用此风格。

### 2. 手动指定

在对话中明确指定需要的输出风格：

```
请用 teaching 风格解释闭包的概念
```

## 输出风格详解

适用于快速问答、命令执行结果等场景。

```markdown

description: 极简输出风格
trigger: ["简洁", "快速", "简短", "concise"]
---

# 输出规范

## 原则
- 只输出结果，不解释
- 使用最少的文字
- 省略所有客套话
- 直接给出核心信息

## 格式模板
```
结果: xxx
状态: xxx
```

2. user.name 属性不存在

```typescript
// 方案1: 添加防御性检查
const name = user?.name?.toLowerCase() || 'Unknown'

// 方案2: 添加类型守卫
if (!user || !user.name) {
  return <div>No user data</div>
}

## 最佳实践

### 1. 明确场景定位

每个风格对应特定使用场景，避免混淆：

```
✅ 好的设计
concise.md    → 快速问答
teaching.md   → 学习教学
audit.md      → 代码审查

❌ 不好的设计
style1.md     → 含义不明确
template.md   → 场景模糊
```

### 2. 使用一致的格式

同类输出使用统一格式，便于解析和理解。

### 3. 保持结构清晰

输出应该有清晰的组织结构，使用标题、列表、表格等。

### 4. 适当详细程度

根据场景调整详细程度：
- 快速问答: 简洁
- 教学场景: 详细
- 审计场景: 全面

### 5. 考虑机器可读性

如果需要自动化处理，考虑使用结构化格式（表格、代码块）。

## 工作流程

### 创建新输出风格

```
1. 确定使用场景
   ↓
2. 设计输出结构
   ↓
3. 编写风格文件
   ↓
4. 测试风格效果
   ↓
5. 根据反馈调整
```

## 注意事项

1. **输出风格会叠加到系统提示**：不会覆盖项目规则，只影响输出格式
2. **多个风格文件可能冲突**：避免在同一场景中应用多个风格
3. **风格文件应该提交到代码库**：团队共享统一的输出风格
4. **定期审查和更新**：根据团队反馈调整风格定义
5. **trigger 关键词要具有辨识度**：避免与其他常用词冲突

## 常见问题

### Q：输出风格和项目规则有什么区别？

**A**：
- **项目规则 (CLAUDE.md)**：定义"做什么"，比如"使用 TypeScript 编写"
- **输出风格 (output-styles/)**：定义"怎么说"，比如"用列表格式输出"

### Q：可以同时使用多个输出风格吗？

**A**：不建议。多个风格可能产生冲突，导致输出格式混乱。

### Q：输出风格支持嵌套吗？

**A**：不支持。每个输出风格文件是独立的，不能引用其他风格。

### Q：如何测试输出风格是否生效？

**A**：
1. 创建风格文件
2. 使用 trigger 关键词测试
3. 观察输出格式是否符合预期
