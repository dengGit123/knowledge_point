# 执行上下文与执行栈

> 官方文档：[MDN - 执行上下文](https://developer.mozilla.org/zh-CN/docs/Web/JavaScript/Reference/Execution_model)

---

## 一、核心概念

执行上下文（Execution Context）是 JavaScript 引擎执行代码时的**环境抽象**，包含了代码运行所需的变量、作用域和 `this` 指向等信息。

> **通俗理解**：执行上下文就像一个"工作台"，每执行一段代码就搭一个工作台，上面放着这段代码需要的所有工具（变量、函数、this 等）。

---

## 二、执行上下文的类型

| 类型 | 说明 | 创建时机 |
|------|------|---------|
| **全局执行上下文** | 最外层的执行环境 | 页面加载 / 脚本启动时 |
| **函数执行上下文** | 每个函数调用时创建 | 函数被调用时 |
| **eval 执行上下文** | `eval()` 函数内的代码 | 执行 `eval()` 时（不推荐使用） |

---

## 三、执行栈（调用栈）

执行栈是一个**后进先出（LIFO）**的栈结构，用于管理执行上下文的创建和销毁。

```
执行过程：

① 页面加载 → 创建全局执行上下文 → 入栈
                │
② 调用 foo() → 创建 foo 执行上下文 → 入栈
                │
③ foo 中调用 bar() → 创建 bar 执行上下文 → 入栈
                │
④ bar 执行完毕 → bar 执行上下文出栈
                │
⑤ foo 执行完毕 → foo 执行上下文出栈
                │
⑥ 页面关闭 → 全局执行上下文出栈

栈状态变化：
[全局] → [全局, foo] → [全局, foo, bar] → [全局, foo] → [全局] → []
```

```javascript
function bar() {
  console.log('bar')
}

function foo() {
  console.log('foo')
  bar()
}

foo()

// 执行栈变化：
// 1. [全局EC]
// 2. [全局EC, fooEC]
// 3. [全局EC, fooEC] → 输出 'foo'
// 4. [全局EC, fooEC, barEC]
// 5. [全局EC, fooEC, barEC] → 输出 'bar'
// 6. [全局EC, fooEC]  → barEC 出栈
// 7. [全局EC]          → fooEC 出栈
```

## 四、执行上下文的创建过程

每个执行上下文的创建分为两个阶段：

```
┌─ 执行上下文生命周期 ────────────────────────────────────┐
│                                                         │
│  ① 创建阶段（Creation Phase）                           │
│     ├── 创建变量对象（VO / AO）                          │
│     ├── 建立作用域链（Scope Chain）                      │
│     └── 确定 this 指向                                  │
│                                                         │
│  ② 执行阶段（Execution Phase）                          │
│     └── 逐行执行代码，变量赋值、函数调用等               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 五、变量对象（Variable Object）

### 全局执行上下文：变量对象（VO）

全局执行上下文中的变量对象就是**全局对象**（浏览器中是 `window`）。

```javascript
// 全局代码
var globalVar = '全局'
function globalFunc() {}
console.log(window.globalVar)    // '全局'
console.log(window.globalFunc)   // ƒ globalFunc() {}
```

### 函数执行上下文：活动对象（AO）

函数执行上下文中的变量对象称为**活动对象（Activation Object）**，在函数被调用时创建。

```javascript
function foo(a, b) {
  var c = 10
  function bar() {}
  var d = function() {}
}

foo(1, 2)

// foo 的活动对象（AO）创建过程：
// AO = {
//   arguments: { 0: 1, 1: 2, length: 2 },
//   a: 1,           // 参数
//   b: 2,           // 参数
//   c: undefined,   // var 声明（初始 undefined）
//   bar: function,  // 函数声明（已赋值）
//   d: undefined    // var 声明（初始 undefined）
// }
```

## 六、变量提升（Hoisting）

### 提升的本质

变量提升的本质是：在**创建阶段**，引擎先扫描代码，将变量和函数声明提前记录到变量对象中。

### var 的变量提升

```javascript
console.log(a)    // undefined（已提升，但未赋值）
var a = 1
console.log(a)    // 1

// 等价于：
// var a                ← 声明被提升
// console.log(a)       // undefined
// a = 1                ← 赋值留在原位
// console.log(a)       // 1
```

### 函数声明提升

```javascript
// 函数声明 —— 整体提升（包括函数体）
console.log(foo)   // ƒ foo() { ... }
foo()              // 'hello' ✅

function foo() {
  console.log('hello')
}

// 函数表达式 —— 只提升变量声明
console.log(bar)   // undefined
// bar()           // TypeError: bar is not a function

var bar = function() {
  console.log('world')
}
```

### let/const 的"提升"

```javascript
// let/const 也会提升，但存在暂时性死区（TDZ）
{
  // console.log(x)  // ReferenceError（TDZ）
  let x = 1
  console.log(x)     // 1
}

// 对比 var
{
  console.log(y)      // undefined（var 提升，初始化为 undefined）
  var y = 1
}
```

> **关键区别**：`var` 提升时初始化为 `undefined`；`let/const` 提升但**不初始化**，在声明语句之前处于 TDZ，访问会抛出 `ReferenceError`。

### 提升优先级

```javascript
// 函数声明 > 变量声明
var foo = 1
function foo() {}

console.log(typeof foo)  // 'number'
// 解析：函数声明先提升，然后 var foo 发现 foo 已存在，跳过声明
// 执行阶段：foo = 1 覆盖了函数

// 同名函数声明，后面的覆盖前面的
function bar() { return 1 }
function bar() { return 2 }
console.log(bar())  // 2
```

## 七、作用域链的建立

在创建阶段，每个执行上下文会建立自己的**作用域链**：

```javascript
var global = 'global'

function outer() {
  var outerVar = 'outer'

  function inner() {
    var innerVar = 'inner'
    console.log(innerVar)  // 'inner'
    console.log(outerVar)  // 'outer'
    console.log(global)    // 'global'
  }

  inner()
}

outer()

// inner 的作用域链：
// [inner.AO, outer.AO, global.VO]
// 查找变量时从左到右依次查找
```

## 八、完整的执行流程示例

```javascript
var a = 1
function foo() {
  var b = 2
  function bar() {
    var c = 3
    console.log(a + b + c)  // 6
  }
  bar()
}
foo()
```

**执行过程**：

```
① 创建全局执行上下文
   GlobalEC = {
     VO: { a: undefined, foo: ƒ },
     ScopeChain: [GlobalVO],
     this: window
   }
   入栈: [GlobalEC]

② 执行全局代码
   a = 1, 调用 foo()

③ 创建 foo 执行上下文
   FooEC = {
     AO: { arguments: {}, b: undefined, bar: ƒ },
     ScopeChain: [FooAO, GlobalVO],
     this: window
   }
   入栈: [GlobalEC, FooEC]

④ 执行 foo 代码
   b = 2, 调用 bar()

⑤ 创建 bar 执行上下文
   BarEC = {
     AO: { arguments: {}, c: undefined },
     ScopeChain: [BarAO, FooAO, GlobalVO],
     this: window
   }
   入栈: [GlobalEC, FooEC, BarEC]

⑥ 执行 bar 代码
   c = 3
   console.log(a + b + c)
   → 查找 a: BarAO ❌ → FooAO ❌ → GlobalVO ✅ → 1
   → 查找 b: BarAO ❌ → FooAO ✅ → 2
   → 查找 c: BarAO ✅ → 3
   → 输出: 6

⑦ bar 执行完毕 → BarEC 出栈
⑧ foo 执行完毕 → FooEC 出栈
⑨ 全局代码执行完毕 → GlobalEC 保留（直到页面关闭）
```

## 九、面试常见问题

### Q1：变量提升是什么？为什么会发生？

变量提升是指在代码执行前，`var` 声明的变量和 `function` 声明的函数会被**提前到作用域顶部**。这是因为 JavaScript 引擎在创建执行上下文时，会先扫描代码收集所有声明（创建阶段），然后再逐行执行（执行阶段）。

### Q2：let/const 是否存在变量提升？

**存在**。`let`/`const` 声明的变量在创建阶段就会被记录，但不会被初始化。从作用域开始到声明语句之间称为**暂时性死区（TDZ）**，在此期间访问会抛出 `ReferenceError`。这与 `var` 的行为（提升并初始化为 `undefined`）不同。

### Q3：执行上下文和作用域的区别？

- **执行上下文**：**运行时**概念，每次函数调用创建新的执行上下文，包含 VO/AO、作用域链、this
- **作用域**：**词法（编写时）**概念，在代码编写时就确定了，定义了变量的可访问范围

### Q4：为什么函数声明会覆盖同名的变量声明？

在创建阶段，引擎按以下顺序处理声明：
1. **创建 arguments 对象**（仅函数上下文）
2. **检查函数声明**，创建并赋值（整体提升，优先级高）
3. **检查变量声明**，如果名称已存在则跳过（只提升声明，不赋值）

因此同名情况下，函数声明先创建并赋值，后续 `var` 发现同名就跳过，最终函数声明"胜出"。
