# TypeScript 声明文件（Declaration Files）

> 官方文档：[TypeScript Handbook - Declaration Files](https://www.typescriptlang.org/docs/handbook/declaration-files/introduction.html)

---

## 一、什么是声明文件

### 1.1 通俗解释

想象你买了一台进口电器，说明书是外文的，你看不懂。这时候有人给你翻译了一份中文说明书——**声明文件就是这个"翻译版说明书"**。

JavaScript 库本身只有运行时代码，TypeScript 编译器看不懂它们有哪些属性、方法、参数类型。声明文件（`.d.ts`）就是给这些 JS 库配的"类型说明书"，告诉 TypeScript 编译器：这个库长什么样，有哪些 API，参数和返回值分别是什么类型。

```
┌─────────────────────────────────────────────────┐
│                  你的 TS 代码                      │
│          import { foo } from 'some-lib'           │
└──────────────────┬──────────────────────────────┘
                   │ TypeScript 编译器需要知道
                   │ foo 的类型信息
                   ▼
┌─────────────────────────────────────────────────┐
│              some-lib.d.ts                       │
│   ┌───────────────────────────────────────┐     │
│   │  declare function foo(x: number): str │     │  ◄── 类型说明书
│   └───────────────────────────────────────┘     │
└──────────────────┬──────────────────────────────┘
                   │ 类型校验 & 智能提示
                   ▼
┌─────────────────────────────────────────────────┐
│              some-lib.js                         │
│   ┌───────────────────────────────────────┐     │
│   │  function foo(x) { return String(x) } │     │  ◄── 真正的运行时代码
│   └───────────────────────────────────────┘     │
└─────────────────────────────────────────────────┘
```

### 1.2 .d.ts 文件的作用

- **纯类型声明**：`.d.ts` 文件只包含类型信息，不包含任何可执行代码，编译后不会产出 `.js` 文件
- **零运行时开销**：声明文件仅在编译期使用，对运行时性能没有任何影响
- **提供智能提示**：IDE（如 VSCode）根据 `.d.ts` 文件提供自动补全、参数提示和文档悬浮

```typescript
// math-utils.d.ts —— 纯类型声明，没有实现
declare function add(a: number, b: number): number;
declare function multiply(a: number, b: number): number;
declare const PI: number;

// 上面这段代码编译后不会产生任何 .js 输出
```

### 1.3 为什么需要声明文件

| 场景 | 没有声明文件 | 有声明文件 |
|------|-------------|-----------|
| 引入 JS 库 | 全部推断为 `any`，没有提示 | 完整的类型检查和智能提示 |
| 调用 API | 不知道参数类型，容易传错 | 编译期就能发现参数错误 |
| 重构 | 不敢改，怕运行时报错 | 编译器帮你检查所有引用 |
| 团队协作 | 每个人都要翻文档或源码 | IDE 直接显示用法和类型 |

## 二、declare 关键字

`declare` 是声明文件的核心关键字，它的作用是**告诉 TypeScript 编译器：某个变量/函数/类/模块已经存在于运行时环境中，你不需要去查找它的定义，只需要按照我声明的类型来使用就行**。

可以类比为：`declare` 就像在派出所"登记户口"——你告诉编译器"这个人（变量/函数）住在这一带，信息我都报给你了，别再查了"。

### 2.1 declare var / let / const

用于声明全局变量。`var` / `let` / `const` 的语义与普通变量一致（可重赋值 vs 不可重赋值）。

```typescript
// global.d.ts

// 声明一个全局只读常量
declare const APP_VERSION: string;
declare const ENV: 'development' | 'production' | 'test';

// 声明一个全局可变变量
declare var __DEV__: boolean;

// 声明一个全局 Webpack 注入的变量
declare const __WEBPACK_ENV__: {
  API_BASE_URL: string;
  CDN_PATH: string;
};
```

使用示例：

```typescript
// app.ts —— 直接使用，无需 import
console.log(APP_VERSION); // string，有完整的类型提示
console.log(ENV);          // 'development' | 'production' | 'test'

if (__DEV__) {
  console.log('开发模式');
}
```

### 2.2 declare function

声明全局函数，包括参数类型和返回值类型。

```typescript
// global.d.ts

// 普通函数声明
declare function formatCurrency(amount: number, currency?: string): string;

// 函数重载
declare function createElement(tag: string): HTMLElement;
declare function createElement(tag: 'div'): HTMLDivElement;
declare function createElement(tag: 'canvas'): HTMLCanvasElement;

// 利用函数重载实现更精确的类型推断
declare function querySelector(selector: string): Element | null;
declare function querySelector<E extends Element>(selector: string): E | null;
```

### 2.3 declare class

声明一个全局类的完整类型结构。

```typescript
// global.d.ts

declare class Animal {
  name: string;
  age: number;
  constructor(name: string, age: number);
  speak(): void;
  static create(name: string): Animal;
}

declare class Dog extends Animal {
  breed: string;
  constructor(name: string, age: number, breed: string);
  fetch(item: string): void;
}
```

使用：

```typescript
const dog = new Dog('旺财', 3, '柴犬');
dog.speak();   // void
dog.fetch('飞盘'); // void
Dog.create('小白'); // Animal
```

### 2.4 declare enum

声明一个全局枚举。注意：声明文件中的 `declare enum` 要求运行时确实存在对应的枚举实现，如果不存在应使用 `const enum`。

```typescript
// global.d.ts

// 运行时存在该枚举
declare enum HttpStatusCode {
  OK = 200,
  BadRequest = 400,
  Unauthorized = 401,
  NotFound = 404,
  InternalServerError = 500
}

// 如果运行时不存在，用 const enum（编译时内联）
declare const enum Color {
  Red = '#FF0000',
  Green = '#00FF00',
  Blue = '#0000FF'
}
```

### 2.5 declare namespace

声明一个全局命名空间，用于组织相关的类型、变量和函数。在早期 JavaScript 库中很常见（如 jQuery、 lodash 的 UMD 版本）。

```typescript
// global.d.ts

declare namespace MyApp {
  interface User {
    id: number;
    name: string;
    email: string;
  }

  interface Config {
    apiUrl: string;
    timeout: number;
  }

  function getUser(id: number): Promise<User>;
  function setConfig(config: Partial<Config>): void;

  namespace Utils {
    function debounce<T extends (...args: any[]) => any>(
      fn: T,
      delay: number
    ): T;
    function throttle<T extends (...args: any[]) => any>(
      fn: T,
      limit: number
    ): T;
  }
}
```

使用：

```typescript
const user: MyApp.User = { id: 1, name: '张三', email: 'zhangsan@example.com' };
MyApp.getUser(1);
MyApp.Utils.debounce(() => {}, 300);
```

### 2.6 declare module

用于为整个模块声明类型。最常见的场景是为没有类型的第三方库补充声明。

```typescript
// modules.d.ts

// 为没有类型的 JS 库声明模块类型
declare module 'some-legacy-lib' {
  export function init(options: {
    apiKey: string;
    debug?: boolean;
  }): void;
  export function track(event: string, data?: Record<string, unknown>): void;
  export function destroy(): void;
}

// 为 CSS Modules 声明
declare module '*.module.css' {
  const classes: Record<string, string>;
  export default classes;
}

// 为 SCSS Modules 声明
declare module '*.module.scss' {
  const classes: Record<string, string>;
  export default classes;
}

// 为图片资源声明
declare module '*.png' {
  const src: string;
  export default src;
}
declare module '*.svg' {
  const src: string;
  export default src;
}

// 为 Vue 单文件组件声明
declare module '*.vue' {
  import type { DefineComponent } from 'vue';
  const component: DefineComponent<{}, {}, any>;
  export default component;
}

// 通配模块声明 —— 任何模块都当作 any（不推荐，应急用）
declare module 'no-types-lib';
```

Vite 项目中的静态资源声明：

```typescript
// env.d.ts
/// <reference types="vite/client" />

// Vite 默认已经通过 vite/client 提供了以下声明：
// - import.meta.env 的类型
// - *.module.css / *.module.scss
// - *.png / *.svg / *.json 等静态资源

// 如果需要额外声明自定义的资源类型：
declare module '*.glb' {
  const src: string;
  export default src;
}

declare module '*.gltf' {
  const data: Record<string, unknown>;
  export default data;
}
```

### 2.7 declare global

在模块上下文中（文件中有 `import` 或 `export`），使用 `declare global` 来向全局作用域添加声明。

```typescript
// augmentations.d.ts —— 注意：这是一个模块文件（有 export）

// 为 Window 扩展自定义属性
declare global {
  interface Window {
    // 自定义全局变量
    __APP_CONFIG__: {
      version: string;
      apiBaseUrl: string;
    };
    // 第三方 SDK 全局注入的对象
    gtag: (...args: any[]) => void;
  }
}

export {}; // 确保 TypeScript 将此文件视为模块
```

使用：

```typescript
// 任意 .ts 文件中
console.log(window.__APP_CONFIG__.version);
window.gtag('event', 'page_view');
```

### 2.8 declare 关键字速查表

| 语法 | 用途 | 使用场景 |
|------|------|---------|
| `declare var` | 声明全局变量 | CDN 引入的库暴露的全局变量 |
| `declare function` | 声明全局函数 | 全局工具函数 |
| `declare class` | 声明全局类 | 全局构造函数/类 |
| `declare enum` | 声明全局枚举 | 全局常量枚举 |
| `declare namespace` | 声明全局命名空间 | UMD 库的全局命名空间 |
| `declare module` | 声明模块类型 | 为无类型 JS 库补充声明 |
| `declare global` | 在模块中扩展全局 | 扩展 Window / 全局接口 |

## 三、三斜线指令

三斜线指令（Triple-Slash Directives）是 TypeScript 特有的编译器指令，用于告诉编译器在编译时需要引入的其他文件或类型包。可以类比为 C 语言中的 `#include` 或 HTML 中的 `<link>` 标签——把别的文件拉进来。

### 3.1 `/// <reference path="..." />`

指定当前文件依赖的另一个声明文件的路径。

```typescript
// base.d.ts
interface BaseEntity {
  id: number;
  createdAt: Date;
  updatedAt: Date;
}
```

```typescript
// user.d.ts
/// <reference path="base.d.ts" />

interface User extends BaseEntity {
  name: string;
  email: string;
}
```

```typescript
// app.d.ts
/// <reference path="base.d.ts" />
/// <reference path="user.d.ts" />

declare function getUser(id: number): User;
```

编译顺序示意：

```
编译器发现 app.d.ts
    │
    ├── 读取 /// <reference path="base.d.ts" />
    │   └── 先编译 base.d.ts（BaseEntity）
    │
    ├── 读取 /// <reference path="user.d.ts" />
    │   └── 再编译 user.d.ts（User extends BaseEntity）
    │
    └── 最后编译 app.d.ts（getUser: User）
```

### 3.2 `/// <reference types="..." />`

指定当前文件依赖某个 `@types` 包。和 `path` 的区别是：`types` 引用的是 npm 包名，`path` 引用的是文件路径。

```typescript
// env.d.ts

/// <reference types="vite/client" />
// 引入 vite 自带的类型声明，包括 import.meta.env、静态资源模块等

/// <reference types="node" />
// 引入 Node.js 的类型声明（如 process、Buffer 等）
```

Vite 项目中 `env.d.ts` 的典型内容：

```typescript
/// <reference types="vite/client" />

// Vite 的 vite/client 声明提供了：
// - ImportMeta 接口的扩展（import.meta.env）
// - 各种静态资源模块声明（*.css, *.png, *.svg 等）
// - CSS Modules 的类型声明
```

### 3.3 使用场景对比

| 指令 | 引用目标 | 典型场景 |
|------|---------|---------|
| `/// <reference path="..." />` | 相对/绝对路径的 `.d.ts` 文件 | 项目内部多个声明文件之间的依赖管理 |
| `/// <reference types="..." />` | `@types` 包或内置类型包 | 引入 Vite/Node 等环境的类型声明 |

### 3.4 注意事项

```
┌────────────────────────────────────────────────────────────┐
│                   三斜线指令使用注意                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. 必须位于文件最顶部（前面只能有注释和空行）                   │
│  2. 只在声明文件（.d.ts）中使用                               │
│  3. 如果文件有 import/export，三斜线指令通常不需要               │
│     （模块系统会自动处理依赖）                                 │
│  4. 在 tsconfig.json 配置了相应选项后，                        │
│     某些三斜线指令可以省略                                    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```typescript
// ❌ 错误：三斜线指令必须在文件顶部
interface Foo {}
/// <reference types="vite/client" />  // 编译报错

// ✅ 正确
/// <reference types="vite/client" />
interface Foo {}
```

## 四、@types 生态

### 4.1 DefinitelyTyped 介绍

[DefinitelyTyped](https://github.com/DefinitelyTyped/DefinitelyTyped) 是 GitHub 上最大的 TypeScript 类型声明仓库，由社区维护。它为数千个没有内置 TypeScript 支持的 JavaScript 库提供类型声明包（`@types/xxx`）。

```
┌───────────────────────────────────────────────────────┐
│                  类型声明来源                            │
├───────────────────────────────────────────────────────┤
│                                                       │
│  来源 1：库自带类型（内置 .d.ts）                       │
│  ├── vue、vuex、vue-router、vite                       │
│  ├── axios、lodash-es、zod                             │
│  └── package.json 中有 "types" 字段指向声明文件          │
│                                                       │
│  来源 2：@types 社区维护                                │
│  ├── @types/node                                      │
│  ├── @types/lodash                                    │
│  ├── @types/jquery                                    │
│  └── npm install @types/xxx 安装                       │
│                                                       │
│  来源 3：自己编写                                       │
│  └── 项目中的 *.d.ts 文件                               │
│                                                       │
└───────────────────────────────────────────────────────┘
```

官方文档：[DefinitelyTyped](https://github.com/DefinitelyTyped/DefinitelyTyped)

### 4.2 安装和使用 @types/xxx

```bash
# 安装 Node.js 类型声明
npm install -D @types/node

# 安装 jQuery 类型声明
npm install -D @types/jquery

# 安装 lodash 类型声明
npm install -D @types/lodash

# 安装多个常用 @types 包
npm install -D @types/node @types/jest
```

安装后，TypeScript 编译器会自动从 `node_modules/@types/` 目录下查找并加载类型声明，无需额外配置。

```
node_modules/
└── @types/
    ├── node/           ← Node.js 类型声明
    │   ├── index.d.ts
    │   └── package.json
    ├── lodash/         ← Lodash 类型声明
    │   ├── index.d.ts
    │   └── package.json
    └── jquery/         ← jQuery 类型声明
        ├── index.d.ts
        └── package.json
```

在 `tsconfig.json` 中可以通过 `types` 和 `typeRoots` 控制 `@types` 的加载行为：

```json
{
  "compilerOptions": {
    // 默认值：自动加载 node_modules/@types 下的所有包
    // 指定后：只加载列出的 @types 包（注意：会覆盖默认行为）
    "types": ["node", "vite/client"],

    // 自定义 @types 的查找目录
    "typeRoots": [
      "./node_modules/@types",
      "./src/types"
    ]
  }
}
```

### 4.3 常见 @types 包

| @types 包 | 说明 | 安装命令 |
|-----------|------|---------|
| `@types/node` | Node.js API 类型 | `npm i -D @types/node` |
| `@types/jest` | Jest 测试框架类型 | `npm i -D @types/jest` |
| `@types/jquery` | jQuery 类型 | `npm i -D @types/jquery` |
| `@types/lodash` | Lodash 工具库类型 | `npm i -D @types/lodash` |
| `@types/express` | Express 框架类型 | `npm i -D @types/express` |
| `@types/ws` | WebSocket 库类型 | `npm i -D @types/ws` |

> 注意：现代库通常已经内置 TypeScript 支持，不需要额外安装 `@types` 包。安装前可以先检查库的 `package.json` 中是否有 `types` 或 `typings` 字段。

### 4.4 查看已安装的类型声明

```bash
# 方法一：列出 node_modules/@types 目录
ls node_modules/@types

# 方法二：通过 npm 查看已安装的 @types 包
npm ls @types/node

# 方法三：查看某个包是否自带类型
# 检查 package.json 中的 types / typings 字段
cat node_modules/vue/package.json | grep -E '"types"|"typings"'
```

快速判断一个库是否需要安装 `@types`：

```typescript
// 引入库后，鼠标悬停在导入语句上
// 如果显示 any 或没有类型提示 → 需要安装 @types/xxx
import _ from 'lodash'; // 鼠标悬停：any ← 需要 @types/lodash

import { ref } from 'vue'; // 鼠标悬停：function ref<T>(value: T) ← 自带类型，无需 @types
```

## 五、编写自定义声明文件

### 5.1 为无类型的 JS 库编写声明

假设有一个名为 `myMathLib` 的 JS 库，没有任何类型声明：

```javascript
// node_modules/myMathLib/index.js
function add(a, b) {
  return a + b;
}
function clamp(value, min, max) {
  return Math.min(Math.max(value, min), max);
}
const PI_APPROX = 3.14159;
module.exports = { add, clamp, PI_APPROX };
```

步骤一：创建声明文件

```typescript
// src/types/myMathLib.d.ts

// 方法 A：使用 declare module（推荐）
declare module 'myMathLib' {
  export function add(a: number, b: number): number;
  export function clamp(value: number, min: number, max: number): number;
  export const PI_APPROX: number;
}
```

步骤二：确保 TypeScript 能找到声明文件

```json
// tsconfig.json
{
  "compilerOptions": {
    // 方式 1：自动包含 src/types 目录
    "include": ["src/**/*.ts", "src/**/*.d.ts"]
  }
}
```

如果声明文件不在默认搜索路径内，也可以在 `tsconfig.json` 中配置：

```json
{
  "compilerOptions": {
    "typeRoots": [
      "./node_modules/@types",
      "./src/types"
    ]
  }
}
```

使用效果：

```typescript
import { add, clamp, PI_APPROX } from 'myMathLib';

add(1, 2);        // ✅ number → 3
clamp(10, 0, 5);  // ✅ number → 5
PI_APPROX;        // ✅ 3.14159
add('1', '2');    // ❌ 类型错误：不能将 string 分配给 number
```

### 5.2 声明文件的组织结构

推荐的项目结构：

```
src/
├── types/                        ← 集中管理声明文件
│   ├── index.d.ts                ← 统一入口（可选）
│   ├── global.d.ts               ← 全局声明（Window 扩展等）
│   ├── modules.d.ts              ← 模块声明（第三方库补充）
│   ├── env.d.ts                  ← 环境变量声明
│   ├── shims-vue.d.ts            ← Vue 单文件组件声明
│   └── vite-env.d.ts             ← Vite 环境声明
│       /// <reference types="vite/client" />
├── components/
│   └── ...
├── App.vue
└── main.ts
```

Vite + Vue 项目的典型声明文件配置：

```typescript
// src/vite-env.d.ts
/// <reference types="vite/client" />

// Vue 单文件组件声明（Vite 项目通常已内置，无需手动添加）
declare module '*.vue' {
  import type { DefineComponent } from 'vue';
  const component: DefineComponent<{}, {}, any>;
  export default component;
}
```

```typescript
// src/types/global.d.ts
// 全局类型扩展

declare global {
  interface Window {
    __APP_VERSION__: string;
  }
}

export {};
```

```typescript
// src/types/modules.d.ts
// 第三方模块声明

declare module 'some-lib' {
  export function init(options: { apiKey: string }): void;
}

declare module '*.md' {
  const content: string;
  export default content;
}
```

### 5.3 全局声明 vs 模块声明

这是编写声明文件时最容易混淆的概念：

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   全局声明（Script）          模块声明（Module）               │
│   ├── 文件中无 import/export  ├── 文件中有 import/export     │
│   ├── 声明在全局作用域        ├── 声明在模块作用域            │
│   ├── 所有文件直接可见        ├── 需要 import 才能使用        │
│   └── 传统库的全局暴露方式    └── 现代 ESM 库的标准方式       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**全局声明**（文件中没有任何 `import` / `export`）：

```typescript
// global.d.ts —— 全局声明文件（没有 import/export）

// 直接声明，所有 .ts 文件都可直接使用
declare const APP_VERSION: string;

declare function formatDate(date: Date): string;

declare namespace Utils {
  function debounce(fn: Function, ms: number): Function;
}
```

```typescript
// app.ts —— 无需 import，直接使用
console.log(APP_VERSION);   // ✅ string
formatDate(new Date());      // ✅ string
Utils.debounce(() => {}, 300); // ✅ Function
```

**模块声明**（文件中有 `import` / `export`，或使用 `declare module`）：

```typescript
// modules.d.ts —— 模块声明文件

// declare module 是模块声明
declare module 'some-lib' {
  export function init(): void;
}

// 如果想在此文件中同时做全局声明，需要 declare global
// 但前提是文件中必须有 import/export
declare global {
  interface Window {
    myLib: any;
  }
}

export {}; // 这个 export 使文件成为模块
```

**关键区别**：

```typescript
// ❌ 错误示范：在全局声明文件中使用了 import
// global.d.ts
import { Ref } from 'vue'; // 这让文件变成了模块
declare const count: Ref<number>; // 不再是全局的！其他文件看不到

// ✅ 正确做法：使用 declare global
// global.d.ts
import { Ref } from 'vue';
declare global {
  const count: Ref<number>; // 在全局作用域中声明
}
export {};
```

## 六、扩展第三方类型

模块扩充（Module Augmentation）是 TypeScript 中非常强大的特性，它允许你在不修改原始库源码的情况下，扩展或修改已有类型。就像给别人的房子加盖一个阳台——不改主体结构，但增加了新功能。

官方文档：[TypeScript Handbook - Module Augmentation](https://www.typescriptlang.org/docs/handbook/declaration-merging.html#module-augmentation)

### 6.1 模块扩充基础

```typescript
// 扩充的基本模式
declare module '目标模块名' {
  // 在这里扩展该模块的类型
}
```

### 6.2 扩展 vue-router 的 RouteMeta

这是 Vue 项目中最常见的模块扩充场景——为路由元信息（meta）添加自定义字段。

```typescript
// src/types/router.d.ts
import 'vue-router';

// 扩充 vue-router 的 RouteMeta 接口
declare module 'vue-router' {
  interface RouteMeta {
    // 自定义 meta 字段
    requiresAuth?: boolean;        // 是否需要登录
    title?: string;                // 页面标题
    icon?: string;                 // 菜单图标
    keepAlive?: boolean;           // 是否缓存组件
    permissions?: string[];        // 权限列表
    breadcrumb?: Array<{           // 面包屑配置
      title: string;
      path?: string;
    }>;
  }
}

export {};
```

在路由配置中使用：

```typescript
// src/router/index.ts
import { createRouter, createWebHistory, RouteRecordRaw } from 'vue-router';

const routes: RouteRecordRaw[] = [
  {
    path: '/dashboard',
    component: () => import('@/views/Dashboard.vue'),
    meta: {
      requiresAuth: true,    // ✅ 有类型提示
      title: '仪表盘',       // ✅ 有类型提示
      icon: 'dashboard',     // ✅ 有类型提示
      keepAlive: true,       // ✅ 有类型提示
      permissions: ['admin'], // ✅ string[]
      breadcrumb: [
        { title: '首页', path: '/' },
        { title: '仪表盘' }
      ]
    }
  },
  {
    path: '/login',
    component: () => import('@/views/Login.vue'),
    meta: {
      requiresAuth: false,
      title: '登录'
    }
  }
];

const router = createRouter({
  history: createWebHistory(),
  routes
});

// 路由守卫中使用 meta
router.beforeEach((to, from, next) => {
  // to.meta.requiresAuth 完全有类型提示
  if (to.meta.requiresAuth) {
    const token = localStorage.getItem('token');
    if (!token) {
      next('/login');
      return;
    }
    // 权限检查
    if (to.meta.permissions?.length) {
      const userPermissions = getUserPermissions();
      const hasPermission = to.meta.permissions.every(p =>
        userPermissions.includes(p)
      );
      if (!hasPermission) {
        next('/403');
        return;
      }
    }
  }
  next();
});
```

### 6.4 扩展 Vite 的 ImportMetaEnv

在 Vite 项目中，`import.meta.env` 默认只有少量内置变量。自定义环境变量需要通过模块扩充来声明类型。

```typescript
// src/vite-env.d.ts
/// <reference types="vite/client" />

// 扩展 ImportMetaEnv 接口
interface ImportMetaEnv {
  // 内置变量（Vite 已声明，这里列出便于理解）
  readonly VITE_USER_NODE_ENV: 'development' | 'production' | 'test';
  readonly BASE_URL: string;
  readonly MODE: string;
  readonly DEV: boolean;
  readonly PROD: boolean;
  readonly SSR: boolean;

  // 自定义环境变量（必须以 VITE_ 开头才能暴露给客户端）
  readonly VITE_APP_TITLE: string;
  readonly VITE_API_BASE_URL: string;
  readonly VITE_ENABLE_MOCK: string;       // 'true' | 'false'
  readonly VITE_UPLOAD_MAX_SIZE: string;   // 字节数，如 '5242880'
  readonly VITE_SENTRY_DSN: string;
}

// 扩展 ImportMeta 接口
interface ImportMeta {
  readonly env: ImportMetaEnv;
}
```

`.env` 文件示例：

```bash
# .env.development
VITE_APP_TITLE=我的应用（开发）
VITE_API_BASE_URL=http://localhost:3000/api
VITE_ENABLE_MOCK=true
VITE_UPLOAD_MAX_SIZE=5242880
VITE_SENTRY_DSN=

# .env.production
VITE_APP_TITLE=我的应用
VITE_API_BASE_URL=https://api.example.com
VITE_ENABLE_MOCK=false
VITE_UPLOAD_MAX_SIZE=5242880
VITE_SENTRY_DSN=https://xxx@sentry.io/123
```

使用时有完整的类型提示：

```typescript
// src/utils/request.ts
const baseURL = import.meta.env.VITE_API_BASE_URL;   // ✅ string
const isMock = import.meta.env.VITE_ENABLE_MOCK === 'true'; // ✅ string
const maxUpload = Number(import.meta.env.VITE_UPLOAD_MAX_SIZE); // ✅ number

import.meta.env.VITE_UNKNOWN_VAR; // ❌ 类型错误：不存在该属性
```

## 七、声明文件的发布

### 7.1 与 npm 包一起发布

如果你的库是用 TypeScript 编写的，最简单的方式是将编译生成的 `.d.ts` 文件随 npm 包一起发布。

```
my-lib/
├── dist/
│   ├── index.js          ← 编译后的 JS 代码
│   ├── index.d.ts        ← 编译后的类型声明
│   ├── utils.js
│   └── utils.d.ts
├── src/                  ← TS 源码（不发布）
├── package.json
└── tsconfig.json
```

`package.json` 配置：

```json
{
  "name": "my-lib",
  "version": "1.0.0",
  "main": "dist/index.js",
  "types": "dist/index.d.ts",
  "files": ["dist"]
}
```

`tsconfig.json` 中需要开启声明文件生成：

```json
{
  "compilerOptions": {
    "declaration": true,           // 生成 .d.ts 文件
    "declarationMap": true,        // 生成 .d.ts.map（可选，便于调试）
    "outDir": "./dist",
    "rootDir": "./src"
  }
}
```

### 8.1 找不到模块声明

**问题现象**：

```typescript
import foo from 'some-js-lib';
// ❌ 报错：Could not find a declaration file for module 'some-js-lib'.
// '/path/to/some-js-lib/index.js' implicitly has an 'any' type.
```

**解决方案**（按推荐程度排序）：

```typescript
// 方案 1：安装 @types 包（最佳）
// npm install -D @types/some-js-lib

// 方案 2：创建自定义声明文件（推荐）
// src/types/modules.d.ts
declare module 'some-js-lib' {
  export function foo(x: number): string;

}

// 方案 3：快速声明为 any（应急，不推荐长期使用）
// src/types/modules.d.ts
declare module 'some-js-lib';

// 方案 4：关闭 noImplicitLint（不推荐）
// tsconfig.json
{
  "compilerOptions": {
    "noImplicitAny": false  // 全局关闭，影响范围太大
  }
}
```

### 8.2 类型冲突处理

**问题现象**：不同版本的 `@types` 包之间存在类型不兼容。

```typescript
// 同时安装了 @types/node@18 和依赖要求 @types/node@16
// 导致类型签名不匹配
```

```bash

```

```json
// package.json

      "@types/node": "20.11.0"
    }
  },
  "resolutions": {
    "@types/node": "20.11.0"
  }
}
```

```typescript
// 2. 使用类型断言处理局部冲突
import type { Request, Response } from 'express';

app.get('/api/users', (req: Request, res: Response) => {
  // 局部类型不匹配时，使用 as 进行类型断言
  const userId = (req.query.id as string) || '';
});
```

```json

  }
}
```

### 8.3 any 污染

**问题现象**：一个 `any` 类型的声明会"污染"所有使用它的地方，导致类型安全失效。

```typescript
// ❌ 危险：整个模块都是 any
declare module 'some-lib'; // 所有导出都是 any

import foo from 'some-lib';
foo.anything();       // 不报错
foo.nothing.here.ever(); // 不报错 —— any 的"黑洞效应"
const result = foo(); // result 也是 any
result.whatever;      // 还是 any —— 污染蔓延
```

```
any 的"黑洞效应"：

  any ──→ 变量 ──→ 函数参数 ──→ 返回值 ──→ 另一个变量
   │                                                  │
   └────────────── 所有类型检查全部失效 ─────────────────┘
```

**解决方案**：

```typescript
// ✅ 尽可能精确地声明类型
declare module 'some-lib' {
  export function init(config: { apiKey: string }): void;
  export function getData<T>(key: string): Promise<T>;
  export function destroy(): void;
}

// ✅ 如果实在不知道类型，至少用 unknown 代替 any
declare module 'unsafe-lib' {
  export function doSomething(input: unknown): unknown;
  // unknown 比 any 安全：使用前必须进行类型检查
}

import { doSomething } from 'unsafe-lib';
const result = doSomething('test');

// result 是 unknown，必须先检查才能使用
if (typeof result === 'string') {
  console.log(result.toUpperCase()); // ✅ 安全
}
result.toUpperCase(); // ❌ 报错：Object is of type 'unknown'
```

**any vs unknown 对比**：

| 特性 | `any` | `unknown` |
|------|-------|-----------|
| 赋值给其他类型 | 直接可以 | 需要类型检查 |
| 调用方法 | 可以 | 不可以 |
| 参与运算 | 可以 | 不可以 |
| 类型安全 | 完全丧失 | 需要收窄后才安全 |
| 推荐程度 | 避免使用 | 安全的 any 替代品 |

## 九、面试常见问题

### Q1：什么是 `.d.ts` 文件？它和普通 `.ts` 文件有什么区别？

`.d.ts` 文件是 TypeScript 的类型声明文件，只包含类型信息，不包含可执行代码。区别如下：

- `.ts` 文件既有类型也有实现，编译后生成 `.js` 文件
- `.d.ts` 文件只有类型声明，编译后不产生任何输出
- `.d.ts` 文件用于描述已有的 JavaScript 代码的形状（shape），让 TypeScript 编译器能理解和检查这些代码

### Q2：declare 关键字的作用是什么？

`declare` 告诉 TypeScript 编译器某个实体（变量、函数、类、模块等）已经在运行时环境中存在，不需要编译器去查找它的实现。它用于：

- 声明全局变量或函数的类型
- 为没有类型的第三方 JS 库提供类型信息
- 扩展现有类型定义

### Q3：如何为没有 TypeScript 支持的第三方库添加类型？

三种方式（按优先级）：

1. 查找并安装 `@types/xxx` 社区维护包
2. 在项目中创建 `declare module 'lib-name' { ... }` 自定义声明文件
3. 创建 `declare module 'lib-name';` 快速声明为 any（应急方案）

### Q4：全局声明和模块声明有什么区别？

| 维度 | 全局声明 | 模块声明 |
|------|---------|---------|
| 文件特征 | 无 import/export | 有 import/export 或使用 declare module |
| 作用域 | 全局，所有文件可见 | 模块内，需 import |
| 使用场景 | CDN 引入的全局库 | npm 包、ES 模块 |
| 在模块文件中做全局声明 | 需要 `declare global { }` | — |

### Q5：什么是模块扩充（Module Augmentation）？举一个实际应用场景。

模块扩充是指在模块上下文中使用 `declare module` 来扩展现有模块的类型定义。最经典的应用场景是扩展 vue-router 的 `RouteMeta`：

```typescript
declare module 'vue-router' {
  interface RouteMeta {
    requiresAuth?: boolean;
    title?: string;
  }
}
```

这样在路由配置中使用 `meta` 字段时就能获得完整的类型提示和检查。

### Q6：三斜线指令 `/// <reference types="vite/client" />` 的作用是什么？

这条指令告诉 TypeScript 编译器引入 Vite 客户端类型声明，它提供了：

- `import.meta.env` 的类型定义
- 静态资源（`*.css`, `*.png`, `*.svg` 等）的模块声明
- CSS Modules 的类型声明

### Q7：TypeScript 是如何查找模块的类型声明的？

查找顺序：

1. `package.json` 的 `exports` 字段中的 `types` 条件
2. `package.json` 的 `types` 或 `typings` 字段指向的 `.d.ts` 文件
3. 包根目录下的 `index.d.ts`
4. `node_modules/@types/` 目录下查找
5. 如果都找不到，报错（或隐式为 `any`）

### Q8：`skipLibCheck` 选项有什么作用？应该在生产项目中使用吗？

`skipLibCheck: true` 会跳过对所有 `.d.ts` 文件的类型检查。

优点：
- 显著加快编译速度（尤其在大型项目中）
- 避免 `@types` 包之间的版本冲突

缺点：
- 可能遗漏声明文件中的类型错误
- 你的代码与声明文件的类型不匹配不会被检查

建议：在大型项目中可以开启以提升性能，但需要确保自己编写的代码类型正确。对于库的作者，不建议开启。

### Q9：如何扩展 Vite 的 `import.meta.env` 类型？

通过扩展 `ImportMetaEnv` 接口：

```typescript
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_API_URL: string;
  readonly VITE_APP_TITLE: string;
}
```

注意：不需要 `declare global`，因为 `ImportMetaEnv` 是一个顶层接口，TypeScript 会自动合并同名接口。

### Q10：`any` 污染是什么？如何避免？

`any` 污染是指一个 `any` 类型的值会将其"无类型"特性传播到所有使用它的地方，导致整个类型链失效。避免方法：

1. 为第三方库编写精确的声明文件，而不是使用空的 `declare module`
2. 用 `unknown` 代替 `any` —— `unknown` 强制要求使用前进行类型检查
3. 开启 `noImplicitAny` 严格模式，防止隐式 `any`
4. 使用类型守卫（Type Guard）收窄类型
