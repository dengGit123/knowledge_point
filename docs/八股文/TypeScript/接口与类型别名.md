# 接口与类型别名

> TypeScript 中定义对象形状的两种核心方式：**interface（接口）** 和 **type（类型别名）**。
> 它们各有千秋，理解差异才能在项目中做出正确选择。

官方文档：
- [Interfaces](https://www.typescriptlang.org/docs/handbook/2/objects.html)
- [Type Aliases](https://www.typescriptlang.org/docs/handbook/2/everyday-types.html#type-aliases)
- [Differences Between Type Aliases and Interfaces](https://www.typescriptlang.org/docs/handbook/2/everyday-types.html#differences-between-type-aliases-and-interfaces)

## 一、接口（interface）

接口是 TypeScript 中用来**描述对象形状**的契约。可以把它想象成一份"蓝图"或"合同"——它规定了对象必须拥有哪些属性、方法，但不管你怎么实现。

### 1.1 基本语法

```typescript
// 定义一个描述"用户"的接口
interface User {
  name: string
  age: number
  email: string
}

// 使用接口：对象必须满足接口定义的形状
const user: User = {
  name: '张三',
  age: 25,
  email: 'zhangsan@example.com'
}

// ❌ 缺少属性会报错
// const wrongUser: User = {
//   name: '张三'
//   // Error: Property 'age' is missing
// }

// ❌ 多余属性也会报错（在直接赋值时）
// const extraUser: User = {
//   name: '张三',
//   age: 25,
//   email: 'zhangsan@example.com',
//   phone: '13800138000' // Error: 对象文字可以只指定已知属性
// }
```

> **通俗类比**：接口就像一份"租房合同"，上面写清楚了你必须交的押金、月租、租期。你住进去之后必须按合同来，少交一个字段都不行。

### 1.2 可选属性与只读属性

```typescript
interface Product {
  id: number
  name: string
  price: number
  // 可选属性：用 ? 标记，可以有也可以没有
  description?: string
  tags?: string[]
  // 只读属性：只能在创建时赋值，之后不可修改
  readonly createdAt: Date
}

const product: Product = {
  id: 1,
  name: 'TypeScript 实战教程',
  price: 99.9,
  // description 和 tags 可以不写
  createdAt: new Date()
}

// ✅ 可选属性可以后续赋值
product.description = '一本好书'
product.tags = ['编程', 'TypeScript']

// ❌ 只读属性不可修改
// product.createdAt = new Date() // Error: Cannot assign to 'createdAt' because it is a read-only property
```

**`readonly` vs `const` 的区别**：

| 特性 | `readonly` | `const` |
|------|-----------|---------|
| 作用对象 | 接口/类型的属性 | 变量声明 |
| 粒度 | 对象的某个字段 | 整个变量 |
| 深层只读 | 仅第一层（浅只读） | 整个变量不可重赋值 |

```typescript
// readonly 只保护第一层
interface Config {
  readonly database: {
    host: string
    port: number
  }
}

const config: Config = {
  database: { host: 'localhost', port: 3306 }
}

// ❌ 不能重赋值整个 database
// config.database = { host: '127.0.0.1', port: 5432 }

// ✅ 但可以修改内部属性！readonly 是浅层的
config.database.port = 5432 // 这是允许的！

// 如果需要深层只读，使用 Readonly 泛型或自定义递归类型
type DeepReadonly<T> = {
  readonly [P in keyof T]: T[P] extends object ? DeepReadonly<T[P]> : T[P]
}
```

### 1.3 函数类型接口

接口不仅能描述对象，还能描述**函数的签名**。

```typescript
// 方式一：用接口描述函数类型
interface MathOperation {
  (a: number, b: number): number
}

const add: MathOperation = (a, b) => a + b
const multiply: MathOperation = (a, b) => a * b

console.log(add(2, 3))      // 5
console.log(multiply(4, 5)) // 20

// 方式二：混合类型 —— 既是对象又是函数（Vue 插件常见模式）
interface PluginInstaller {
  (app: any, options?: Record<string, unknown>): void
  version: string
  install: (app: any) => void
}

// 类似 Vue 插件的写法
function createPlugin(): PluginInstaller {
  const plugin = ((app: any, options?: Record<string, unknown>) => {
    // 插件安装逻辑
  }) as PluginInstaller

  plugin.version = '1.0.0'
  plugin.install = (app: any) => { /* ... */ }

  return plugin
}
```

### 1.4 可索引签名

当对象的属性名不确定，但值的类型确定时，使用可索引签名。

```typescript
// 字符串索引签名
interface StringMap {
  [key: string]: string
}

const headers: StringMap = {
  'Content-Type': 'application/json',
  'Authorization': 'Bearer token123'
}

// 数字索引签名（常用于数组模拟）
interface NumberArray {
  [index: number]: string
}

const fileNames: NumberArray = ['a.ts', 'b.ts', 'c.ts']

// 混合使用：索引签名 + 具名属性
// 注意：具名属性的返回值必须是索引签名返回值的子类型
interface AppConfig {
  [key: string]: string | number | boolean
  version: string          // ✅ string 是 string | number | boolean 的子类型
  debug: boolean           // ✅ boolean 是 string | number | boolean 的子类型
  maxRetryCount: number    // ✅
}

const appConfig: AppConfig = {
  version: '2.0.0',
  debug: true,
  maxRetryCount: 3,
  // 可以动态添加任意 string | number | boolean 值的属性
  apiBaseUrl: 'https://api.example.com'
}
```

```
┌────────────────────────────────────────┐
│         索引签名注意事项               │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │ [key: string]  → 最灵活          │  │
│  │ [key: number]  → 常用于数组/列表  │  │
│  │ [key: symbol]  → 较少使用         │  │
│  └──────────────────────────────────┘  │
│                                        │
│  ⚠️ 具名属性必须是索引签名值的子类型    │
│  ⚠️ 索引签名会限制所有属性的返回类型   │
│  ⚠️ 尽量不要滥用，会丢失类型精确性     │
└────────────────────────────────────────┘
```

## 二、接口继承（extends）

就像 class 可以继承一样，interface 也能继承，从而实现类型定义的复用与组合。

> **通俗类比**：接口继承就像"抄别人的作业"——你可以抄一个人的（单继承），也可以把好几个人的拼在一起抄（多继承），甚至还能在抄的时候改改答案（覆盖属性）。

### 2.1 单继承

```typescript
interface Animal {
  name: string
  age: number
  move(): void
}

interface Dog extends Animal {
  breed: string
  bark(): void
}

// Dog 拥有 Animal 的所有属性 + 自己的属性
const dog: Dog = {
  name: '旺财',
  age: 3,
  breed: '柴犬',
  move() {
    console.log('跑步中...')
  },
  bark() {
    console.log('汪汪汪！')
  }
}
```

### 2.2 多继承

接口可以同时继承多个接口，实现类型的组合。

```typescript
interface Serializable {
  serialize(): string
}

interface Loggable {
  log(message: string): void
}

interface Entity {
  id: number
  createdAt: Date
}

// 同时继承三个接口
interface UserModel extends Entity, Serializable, Loggable {
  username: string
  email: string
}

const user: UserModel = {
  id: 1,
  createdAt: new Date(),
  username: 'zhangsan',
  email: 'zhangsan@example.com',
  serialize() {
    return JSON.stringify({ id: this.id, username: this.username })
  },
  log(message: string) {
    console.log(`[${this.username}] ${message}`)
  }
}
```

### 2.3 覆盖父接口属性

子接口可以重新定义父接口的属性，但必须是父类型的**子类型**（协变）。

```typescript
interface BaseEvent {
  timestamp: Date | number
  data: unknown
}

interface ClickEvent extends BaseEvent {
  // ✅ 缩窄类型：unknown → { x: number; y: number }
  data: { x: number; y: number }
  // ✅ 缩窄类型：Date | number → number
  timestamp: number
  // 新增属性
  target: string
}

// ❌ 不能将类型扩大
// interface BadEvent extends BaseEvent {
//   data: string  // 如果父类型是 unknown，这其实是可以的
//                  // 但如果父类型是 string，子类型改成 number 就会报错
// }

// 具体例子：类型缩窄（协变）
interface Response {
  status: number | string
}

interface SuccessResponse extends Response {
  status: 200 | 201  // ✅ 字面量类型是 number 的子类型
}
```

### 2.4 继承的合并规则

当多继承导致冲突时，TypeScript 会检查冲突属性是否兼容。

```typescript
interface A {
  value: string | number
}

interface B {
  value: string | boolean
}

// ✅ 交叉后 value: string（两个联合类型的交集）
// 但 interface extends 不会自动做交集，而是要求子类型兼容
interface C extends A, B {
  value: string  // ✅ string 同时是 A.value 和 B.value 的子类型
}

// ❌ 如果不处理冲突
// interface D extends A, B {
//   // Error: 同时继承的 value 类型不兼容，必须显式覆盖
// }
```

## 三、声明合并

**声明合并（Declaration Merging）** 是 interface 独有的能力，也是它与 type 最本质的区别之一。

> **通俗类比**：声明合并就像多个设计师各自画了一部分图纸，最终这些图纸会自动拼成一张完整的图纸。type 则像一张写死的纸，重复写就会报错。

### 3.1 同名接口自动合并

```typescript
interface Window {
  title: string
}

interface Window {
  width: number
  height: number
}

// 等价于：
// interface Window {
//   title: string
//   width: number
//   height: number
// }

const win: Window = {
  title: '主窗口',
  width: 1920,
  height: 1080
}
```

**合并规则**：

```
┌─────────────────────────────────────────────────┐
│              声明合并规则                         │
│                                                  │
│  1. 非函数属性 → 必须类型一致，否则报错           │
│                                                  │
│     interface Box { size: number }               │
│     interface Box { size: string }  // ❌ 冲突!  │
│                                                  │
│  2. 函数属性 → 变成重载（按后定义的优先）          │
│                                                  │
│     interface Printer {                          │
│       print(msg: string): void                   │
│     }                                            │
│     interface Printer {                          │
│       print(msg: string, count: number): void    │
│     }                                            │
│     // print 有两个重载签名                       │
│                                                  │
│  3. 合并顺序：后面的接口排在重载列表前面           │
│                                                  │
└─────────────────────────────────────────────────┘
```

### 3.2 扩展第三方类型

这是声明合并最实用的场景——给第三方库添加自定义属性或方法。

```typescript
// 场景一：扩展 Vue Router 的 Route Meta
// router/index.ts
import 'vue-router'

// 利用声明合并扩展 RouteMeta
declare module 'vue-router' {
  interface RouteMeta {
    // 添加自定义字段
    requiresAuth?: boolean
    title?: string
    permissions?: string[]
  }
}

// 现在可以在路由配置中安全使用
const routes = [
  {
    path: '/dashboard',
    component: () => import('@/views/Dashboard.vue'),
    meta: {
      requiresAuth: true,      // ✅ 有类型提示
      title: '仪表盘',         // ✅ 有类型提示
      permissions: ['admin']   // ✅ 有类型提示
    }
  }
]
```

```typescript
// 场景二：扩展 Vite 的 ImportMetaEnv
// env.d.ts
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_API_BASE_URL: string
  readonly VITE_APP_TITLE: string
  readonly VITE_ENABLE_MOCK: string
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}

// 在代码中使用
const apiUrl = import.meta.env.VITE_API_BASE_URL // ✅ 完整类型提示
```

```typescript
// 场景三：扩展全局 Window（非 Vue 场景）
interface Window {
  __VUE_DEVTOOLS_GLOBAL_HOOK__: any
  myCustomSDK: {
    init(config: { apiKey: string }): void
    track(event: string, data?: Record<string, unknown>): void
  }
}

// 现在可以安全访问
window.myCustomSDK.init({ apiKey: 'abc123' })
```

> **注意**：`type` 不支持声明合并！这是 interface 最独特的优势。

```typescript
type Animal = { name: string }
type Animal = { age: number }  // ❌ Error: Duplicate identifier 'Animal'

interface Plant { name: string }
interface Plant { age: number } // ✅ 自动合并为 { name: string; age: number }
```

## 四、类型别名（type）

type 用来给任何类型起一个新名字。它的能力范围比 interface 更广——不仅能描述对象形状，还能给联合类型、元组、基本类型等起别名。

### 4.1 基本语法

```typescript
// 给对象类型起别名（看起来和 interface 很像）
type Point = {
  x: number
  y: number
}

type User = {
  name: string
  age: number
  email: string
}

const point: Point = { x: 10, y: 20 }
const user: User = { name: '张三', age: 25, email: 'zhangsan@example.com' }
```

### 4.2 给各种类型起别名

```typescript
// 1. 给原始类型起别名（interface 做不到）
type ID = string
type Timestamp = number
type IsActive = boolean

const userId: ID = 'user_001'
const loginTime: Timestamp = Date.now()
const active: IsActive = true

// 2. 联合类型别名（interface 做不到）
type Status = 'pending' | 'active' | 'inactive' | 'deleted'
type Method = 'GET' | 'POST' | 'PUT' | 'DELETE' | 'PATCH'
type Result<T> = SuccessResult<T> | ErrorResult
type StringOrNumber = string | number

// 3. 元组别名（interface 做不到）
type Coordinate = [number, number]        // [经度, 纬度]
type RGB = [number, number, number]       // [红, 绿, 蓝]
type NameAge = [string, number]           // [姓名, 年龄]

const shanghai: Coordinate = [121.47, 31.23]
const red: RGB = [255, 0, 0]
const person: NameAge = ['张三', 25]

// 4. 函数类型别名
type Callback = (data: unknown) => void
type CompareFn<T> = (a: T, b: T) => number
type AsyncFunction<T> = (...args: unknown[]) => Promise<T>

// 5. 映射类型（interface 做不到）
type Readonly<T> = {
  readonly [P in keyof T]: T[P]
}

type Optional<T, K extends keyof T> = Omit<T, K> & Partial<Pick<T, K>>

// 6. 条件类型（interface 做不到）
type IsString<T> = T extends string ? true : false
type NonNullable<T> = T extends null | undefined ? never : T
```

### 4.3 type 与 interface 的直观对比

```typescript
// ======== 描述对象形状：两者都能做 ========

// interface 写法
interface PersonInterface {
  name: string
  age: number
}

// type 写法
type PersonType = {
  name: string
  age: number
}

// 效果完全一样
const p1: PersonInterface = { name: '张三', age: 25 }
const p2: PersonType = { name: '张三', age: 25 }

// ======== 以下功能只有 type 能做 ========

// 联合类型
type Status = 'active' | 'inactive'

// 元组
type Pair = [string, number]

// 基本类型别名
type ID = string | number

// 条件类型
type Unwrap<T> = T extends Promise<infer U> ? U : T
```

## 五、interface vs type 全面对比

### 5.1 完整对比表格

| 对比维度 | `interface` | `type` |
|---------|------------|--------|
| **基本用法** | 描述对象形状 | 给任何类型起别名 |
| **对象类型** | `interface Foo { x: number }` | `type Foo = { x: number }` |
| **继承方式** | `extends` 关键字 | `&` 交叉类型 |
| **声明合并** | 支持（同名自动合并） | 不支持（重复定义报错） |
| **联合类型** | 不支持 | `type A = B \| C` |
| **交叉类型** | 通过 extends 实现 | `type A = B & C` |
| **元组类型** | 不支持 | `type Pair = [string, number]` |
| **基本类型别名** | 不支持 | `type ID = string` |
| **字面量类型** | 不支持 | `type Dir = 'left' \| 'right'` |
| **条件类型** | 不支持 | `type A<T> = T extends B ? C : D` |
| **映射类型** | 不支持 | `type Readonly<T> = { readonly [P in keyof T]: T[P] }` |
| **infer 关键字** | 不支持 | `type R<T> = T extends (...args: any[]) => infer R ? R : never` |
| **计算属性** | 不支持 | `type Getter<T> = { [P in keyof T as \`get${Capitalize<string & P>}\`]: () => T[P] }` |
| **泛型** | 支持 | 支持 |
| **extends 约束** | `interface Foo<T extends Bar>` | `type Foo<T extends Bar> = ...` |
| **类实现** | `class A implements Foo` | `class A implements Foo` |
| **函数类型** | `interface Fn { (x: number): string }` | `type Fn = (x: number) => string` |
| **错误提示** | 接口名直接出现在错误信息中 | 错误信息可能展开为字面量形式 |
| **性能** | 大量属性/继承时略优 | 复杂类型可能稍慢（极少数情况） |

### 5.2 各自适用场景

```
┌──────────────────────────────────────────────────────┐
│                                                       │
│   优先使用 interface 的场景：                          │
│   ┌─────────────────────────────────────────────┐    │
│   │ ✓ 定义 API 响应、数据模型的"形状"            │    │
│   │ ✓ 需要声明合并（扩展第三方类型）              │    │
│   │ ✓ 面向对象设计，需要 extends 继承链           │    │
│   │ ✓ 团队/社区规范约定使用 interface             │    │
│   └─────────────────────────────────────────────┘    │
│                                                       │
│   优先使用 type 的场景：                               │
│   ┌─────────────────────────────────────────────┐    │
│   │ ✓ 联合类型、交叉类型、条件类型                │    │
│   │ ✓ 元组、基本类型别名                          │    │
│   │ ✓ 映射类型、工具类型                          │    │
│   │ ✓ 函数类型（更简洁的写法）                    │    │
│   │ ✓ 需要利用模板字面量类型                      │    │
│   └─────────────────────────────────────────────┘    │
│                                                       │
└──────────────────────────────────────────────────────┘
```

### 5.3 开发建议

1. **没有绝对的对错**：interface 和 type 在描述对象形状时功能几乎等价，选择更多是团队约定
2. **官方推荐**：TypeScript 官方文档建议"优先使用 interface，type 作为补充"
3. **实际项目中**：通常两种混用——interface 描述数据模型，type 处理类型运算
4. **保持一致**：在同一个项目中，对同类事物保持统一的风格

```typescript
// 推荐的混用模式

// 数据模型 → interface
interface UserInfo {
  id: number
  name: string
  email: string
  role: UserRole
}

// 联合/交叉 → type
type UserRole = 'admin' | 'editor' | 'viewer'
type Nullable<T> = T | null
type UserResponse = ApiResponse<UserInfo> | ApiError

// 工具类型 → type
type ApiResponse<T> = {
  code: number
  message: string
  data: T
}
```

## 六、实际应用场景

### 6.1 API 响应类型定义

```typescript
// api/types.ts —— 统一管理 API 类型

// 基础响应结构
interface ApiResponse<T = unknown> {
  code: number
  message: string
  data: T
  timestamp: number
}

// 分页响应

// composables/useUser.ts —— 组合式函数的类型定义
interface UseUserOptions {
  immediate?: boolean
  includeDeleted?: boolean
}

interface UseUserReturn {
  user: Ref<UserInfo | null>
  loading: Ref<boolean>
  error: Ref<string | null>
  fetchUser: (id: number) => Promise<void>
  updateUser: (data: Partial<UserInfo>) => Promise<void>
}

function useUser(options: UseUserOptions = {}): UseUserReturn {
  const user = ref<UserInfo | null>(null)
  const loading = ref(false)
  const error = ref<string | null>(null)

  const fetchUser = async (id: number) => {
    loading.value = true
    error.value = null
    try {
      const res = await fetch(`/api/users/${id}`)
      const data: ApiResponse<UserInfo> = await res.json()
      user.value = data.data
    } catch (e) {
      error.value = (e as Error).message
    } finally {
      loading.value = false
    }
  }

  const updateUser = async (data: Partial<UserInfo>) => {
    if (!user.value) return
    loading.value = true
    try {
      await fetch(`/api/users/${user.value.id}`, {
        method: 'PATCH',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(data)
      })
    } finally {
      loading.value = false
    }
  }

  if (options.immediate) {
    fetchUser(0) // 默认取第一个用户
  }

  return { user, loading, error, fetchUser, updateUser }
}
```

### 6.3 函数签名定义

```typescript
// 工具函数的类型签名

// 事件处理函数
type EventHandler<T = Event> = (event: T) => void | boolean
type AsyncEventHandler<T = Event> = (event: T) => Promise<void>

// 防抖函数
type DebouncedFn<T extends (...args: any[]) => any> = {
  (...args: Parameters<T>): void
  cancel(): void
  flush(): void
}

function debounce<T extends (...args: any[]) => any>(
  fn: T,
  delay: number
): DebouncedFn<T> {
  let timer: ReturnType<typeof setTimeout> | null = null

  const debounced = ((...args: Parameters<T>) => {
    if (timer) clearTimeout(timer)
    timer = setTimeout(() => fn(...args), delay)
  }) as DebouncedFn<T>

  debounced.cancel = () => {
    if (timer) clearTimeout(timer)
  }

  debounced.flush = () => {
    if (timer) {
      clearTimeout(timer)
      fn()
    }
  }

  return debounced
}

// 策略模式：用 interface 定义策略接口
interface SortStrategy<T> {
  sort(items: T[]): T[]
  compare(a: T, b: T): number
}

class NumberSort implements SortStrategy<number> {
  compare(a: number, b: number): number {
    return a - b
  }
  sort(items: number[]): number[] {
    return [...items].sort(this.compare)
  }
}

class StringSort implements SortStrategy<string> {
  compare(a: string, b: string): number {
    return a.localeCompare(b, 'zh-CN')
  }
  sort(items: string[]): string[] {
    return [...items].sort(this.compare)
  }
}
```

## 七、面试常见问题

### Q1：interface 和 type 有什么区别？

**答案要点**：

1. **声明合并**：interface 支持同名声明自动合并，type 不支持（重复定义会报错）
2. **类型范围**：type 可以给联合类型、元组、基本类型、条件类型起别名，interface 只能描述对象形状
3. **继承方式**：interface 用 `extends` 继承，type 用 `&` 交叉类型组合
4. **扩展性**：interface 可以被 class `implements`，type 也支持
5. **计算属性/映射类型**：只有 type 支持
6. **错误提示**：interface 在错误信息中显示接口名，更直观

**总结一句话**：描述对象形状优先用 interface，需要类型运算（联合、交叉、条件、映射）时用 type。

---

### Q2：什么是声明合并？有什么实际用途？

**答案要点**：

声明合并是 TypeScript 的编译器特性——当多个同名的 interface 出现在同一作用域时，它们的成员会自动合并为一个接口。

**规则**：
- 非函数属性必须类型一致，否则报错
- 函数属性会变成重载，后定义的排在前面

**实际用途**：
1. 扩展第三方库的类型（如 `vue-router` 的 `RouteMeta`、Vite 的 `ImportMetaEnv`）
2. 扩展全局类型（如 `Window`）
3. 模块 augmentation（`declare module 'xxx'` 扩展已有模块）

> 注意：`type` 没有声明合并能力。

---

### Q3：接口继承时属性冲突怎么处理？

**答案要点**：

当子接口通过 `extends` 继承父接口时：
- 子接口可以**缩窄**父属性的类型（协变），这是合法的
- 如果多继承导致同一属性类型冲突，子接口必须显式覆盖该属性为兼容类型

```typescript
interface A { value: string | number }
interface B { value: string | boolean }

// 必须显式覆盖，string 是两个联合的交集
interface C extends A, B {
  value: string
}
```

关键原则：子类型的属性必须是父类型属性的**子类型**（里氏替换原则）。

---

### Q4：什么时候用 interface？什么时候用 type？

**答案要点**：

| 场景 | 推荐 |
|------|------|
| 定义 API 响应、数据模型 | `interface` |
| 需要声明合并（扩展第三方类型） | `interface` |
| 面向对象，需要继承链 | `interface` |
| 联合类型（`\|`） | `type` |
| 元组类型 | `type` |
| 条件类型 | `type` |
| 映射类型/工具类型 | `type` |
| 函数类型签名 | 均可，`type` 写法更简洁 |

TypeScript 官方建议优先使用 `interface`，需要高级类型运算时用 `type`。实际项目中两者混用是常态，关键是团队保持一致。

---

### Q5：readonly 属性能完全防止修改吗？如何实现深层只读？

**答案要点**：

`readonly` 是**浅层**的，只保护第一层属性不可重赋值，不会递归保护嵌套对象。

```typescript
interface Config {
  readonly db: { host: string; port: number }
}

const config: Config = { db: { host: 'localhost', port: 3306 } }
// config.db = { ... }  // ❌ 被保护
// config.db.port = 5432 // ✅ 可以修改！
```

实现深层只读的方法：
1. 使用 TypeScript 内置的 `Readonly<T>`（但也只保护一层）
2. 自定义递归类型：`type DeepReadonly<T> = { readonly [P in keyof T]: T[P] extends object ? DeepReadonly<T[P]> : T[P] }`
3. 使用 `as const` 断言（编译期常量）
4. 运行时使用 `Object.freeze()` 递归冻结
