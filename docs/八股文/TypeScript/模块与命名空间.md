## 一、模块系统概述

### 1.1 通俗解释

把代码组织想象成一座公寓楼：

```
┌─────────────────────────────────────────────────┐
│                   公寓楼（项目）                    │
│                                                   │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐      │
│  │ 房间 A   │   │ 房间 B   │   │ 房间 C   │      │
│  │ utils.ts │   │ user.ts  │   │ app.ts   │      │
│  │          │   │          │   │          │      │
│  │ 🪟 export│──>│ 🪟 export│──>│ 🚪 import│      │
│  │  窗户：  │   │  窗户：  │   │  敲门：  │      │
│  │ 对外提供 │   │ 对外提供 │   │ 请求使用 │      │
│  │ 功能     │   │ 功能     │   │ 功能     │      │
│  └──────────┘   └──────────┘   └──────────┘      │
│                                                   │
│  没有 export 的东西 = 房间里的私有物品，外人看不到     │
│  没有 import 的东西 = 没去敲门，拿不到别人的东西       │
└─────────────────────────────────────────────────┘
```

- **export** = 打开窗户，告诉别人"我这有你想要的东西"
- **import** = 敲门请求，告诉别人"我想用你的东西"
- 没有这两者的文件 = 一间封闭的房间，里面的东西全部是全局的（脚本文件）

### 1.2 ES Module vs CommonJS

JavaScript 有两套模块系统，在 TypeScript 中都可能遇到：

```
┌─────────────────────┬──────────────────────────┬──────────────────────────┐
│       特性          │     ES Module (ESM)      │      CommonJS (CJS)      │
├─────────────────────┼──────────────────────────┼──────────────────────────┤
│  导出语法           │  export / export default │  module.exports / exports│
│  导入语法           │  import                  │  require()               │
│  加载方式           │  编译时静态分析            │  运行时动态加载            │
│  Tree-shaking      │  ✅ 支持                  │  ❌ 不支持               │
│  浏览器原生支持      │  ✅ `<script type="module">`│  ❌ 不支持             │
│  Node.js 支持       │  ✅ .mjs 或 "type":"module"│  ✅ 默认支持            │
│  TypeScript 推荐    │  ✅ 推荐                  │  不推荐新项目使用          │
│  引用机制           │  引用（绑定）             │  值拷贝（基本类型）         │
│  循环依赖处理       │  生成部分绑定             │  得到未完成的 exports      │
└─────────────────────┴──────────────────────────┴──────────────────────────┘
```

```typescript
// ES Module 方式（推荐）
export const name = 'Tom'
export function greet() { /* ... */ }
import { name, greet } from './user'

// CommonJS 方式（不推荐新项目使用）
const name = 'Tom'
module.exports = { name }
const { name } = require('./user')
```

### 1.3 TypeScript 中的模块判断规则

TypeScript 将 `.ts` 文件分为两种：**模块** 和 **脚本**，判断规则极其简单：

```
  .ts 文件
      │
      ▼
  文件中是否存在顶层
  import 或 export？
      │
      ├── 存在 ──→ 该文件是【模块】
      │            所有变量都是文件内私有的
      │            只有 export 的内容才能被外部访问
      │
      └── 不存在 ──→ 该文件是【脚本】
                   所有顶层声明都在全局命名空间中
                   可能导致命名冲突！
```

```typescript
// ---------- 模块（Module） ----------
// math.ts —— 有 export，所以是模块
export function add(a: number, b: number): number {
  return a + b
}
const secret = 123 // 私有，外部无法访问

// ---------- 脚本（Script） ----------
// legacy.ts —— 没有 import/export，所以是脚本
const globalVar = '我在全局命名空间中，其他脚本也能看到我'
// ⚠️ 如果另一个脚本也声明了 globalVar，就会冲突！
```

> **最佳实践**：在现代项目中，每个 `.ts` 文件都应该是模块。即使只是导出一个空对象 `export {}`，也比脚本文件更安全。

## 二、ES Module 语法

### 2.1 export 导出

#### 命名导出（Named Export）

一个文件可以有多个命名导出，像一扇窗户上标明了每样物品的名字：

```typescript
// user.ts
export interface User {
  id: number
  name: string
  email: string
}

export function createUser(name: string, email: string): User {
  return { id: Date.now(), name, email }
}

export const MAX_USERS = 1000

// 也可以先声明，再统一导出
function validateEmail(email: string): boolean {
  return email.includes('@')
}

function formatUserName(user: User): string {
  return `${user.name} <${user.email}>`
}

export { validateEmail, formatUserName }
```

#### 默认导出（Default Export）

每个文件最多只有一个默认导出，就像房间门口挂了一个"主推产品"的牌子：

```typescript
// logger.ts
export default class Logger {
  constructor(private prefix: string) {}

  log(message: string): void {
    console.log(`[${this.prefix}] ${message}`)
  }
}

// 注意：一个文件只能有一个 export default
// ❌ 错误：不能有多个默认导出
// export default function anotherOne() {} // Error!
```

#### 重导出（Re-export）

像中间商一样，从别的模块拿来东西再转手卖出去：

```typescript
// barrel.ts —— 桶文件，统一导出
export { User, createUser } from './user'
export { default as Logger } from './logger'
export { validateEmail } from './user'

// 全部重导出（慎用）
export * from './utils' // 导出 utils 中的所有命名导出

// 重导出时重命名
export { createUser as createNewUser } from './user'
```

### 2.2 import 导入

```
┌───────────────────────────────────────────────────────────┐
│                     导入方式总览                           │
├───────────────────┬───────────────────────────────────────┤
│  命名导入          │  import { X, Y } from './mod'        │
│  默认导入          │  import X from './mod'               │
│  命名空间导入      │  import * as mod from './mod'        │
│  副作用导入        │  import './mod'                      │
│  动态导入          │  const mod = await import('./mod')   │
│  类型导入          │  import type { X } from './mod'      │
└───────────────────┴───────────────────────────────────────┘
```

#### 命名导入

```typescript
// 按名字精确导入
import { User, createUser, MAX_USERS } from './user'

// 导入时重命名
import { createUser as createUserAccount } from './user'

// 一次导入多个
import {
  User,
  createUser,
  validateEmail,
} from './user'
```

#### 默认导入

```typescript
// 默认导出的东西，导入时可以随意起名
import Logger from './logger'
import MyLogger from './logger' // 一样的，随便叫什么

// 同时导入默认和命名
import Logger, { User } from './user'
//     ^默认    ^命名
```

#### 命名空间导入

```typescript
// 把整个模块作为对象导入
import * as UserModule from './user'

UserModule.createUser('Tom', 'tom@example.com')
UserModule.MAX_USERS
```

#### 动态导入

动态导入返回一个 Promise，适合按需加载（路由懒加载等场景）：

```typescript
// Vue 路由懒加载的经典用法
const routes = [
  {
    path: '/dashboard',
    component: () => import('./views/Dashboard.vue') // 动态导入
  },
  {
    path: '/settings',
    component: () => import('./views/Settings.vue')
  }
]

// 也可以用 await
async function loadModule() {
  const { createUser } = await import('./user')
  return createUser('Tom', 'tom@test.com')
}
```

### 2.3 type-only 导入导出

这是 TypeScript 特有的语法，明确告诉编译器"这只是类型，编译后要删掉"：

```typescript
// ---------- import type ----------
// ✅ 推荐：明确表示只导入类型
import type { User } from './user'

// 在运行时完全不存在，编译后被完全移除
function processUser(u: User) { // OK
  // console.log(u.name) // OK，但 User 本身只是类型标注
}

// ---------- 内联 type 修饰符 ----------
// 混合导入：value 是值，User 是类型
import { createUser, type User } from './user'

// ---------- export type ----------
// 只导出类型，不允许在运行时使用
export type { User }
export type { CreateUserResult } from './user'

// ---------- 对比 ----------
import type { User } from './user'      // ✅ 纯类型导入
import { type User } from './user'      // ✅ 内联类型导入（TS 4.5+）
import { User } from './user'           // ⚠️ 也行，但如果 User 只是类型，用 type 更安全
```

> **为什么要用 `import type`？** 配合 `isolatedModules` 选项（Vite/esbuild 等工具需要），确保每个文件可以被独立编译，不用分析整个项目。

## 三、模块解析策略

### 3.1 什么是模块解析？

当你写 `import { foo } from './utils'` 时，TypeScript 需要找到 `./utils` 到底对应哪个文件。这个查找过程就叫**模块解析（Module Resolution）**。

### 3.2 node 解析策略

模拟 Node.js 的模块查找逻辑，这是最常用的策略：

```
import { foo } from './utils'
                          │
                          ▼
            ┌─────────────────────────────┐
            │  1. 检查相对路径              │
            │     ./utils.ts  存在？──是──→ 找到！
            │                │
            │               否
            │                ▼
            │     ./utils.tsx 存在？──是──→ 找到！
            │                │
            │               否
            │                ▼
            │     ./utils.d.ts 存在？──是──→ 找到！
            │                │
            │               否
            │                ▼
            │     ./utils/package.json    │
            │       中有 "types" 字段？    │
            │                │
            │               否
            │                ▼
            │     ./utils/index.ts 存在？  │
            │     ./utils/index.tsx 存在？ │
            └─────────────────────────────┘
```

对于非相对路径（如 `import { ref } from 'vue'`）：

```
import { ref } from 'vue'
                      │
                      ▼
  ┌─────────────────────────────────────────────────┐
  │  在 node_modules 中逐级向上查找：                  │
  │                                                   │
  │  /project/node_modules/vue/                      │
  │    ├── package.json → "types" 或 "exports" 字段   │
  │    ├── index.d.ts                                │
  │    └── ...                                       │
  │                                                   │
  │  如果没找到，继续向上：                              │
  │  /node_modules/vue/                              │
  │  ... 直到文件系统根目录                             │
  └─────────────────────────────────────────────────┘
```

### 3.3 bundler 解析策略

TypeScript 5.0 新增，专门为 Vite、esbuild、webpack 等打包工具设计：

```json
// tsconfig.json
{
  "compilerOptions": {
    "moduleResolution": "bundler"
  }
}
```

**与 node 策略的区别**：

```
┌──────────────────────┬──────────────────┬──────────────────────┐
│       特性           │     node         │     bundler          │
├──────────────────────┼──────────────────┼──────────────────────┤
│ 相对路径省略后缀      │ ❌ 需要精确匹配   │ ✅ 自动尝试 .ts/.tsx  │
│ package.json exports │ ✅ 支持          │ ✅ 支持              │
│ 目录导入 (./dir)     │ ✅ 查找 index    │ ✅ 查找 index        │
│ 打包工具特性兼容     │ ❌               │ ✅ 原生兼容           │
│ 适合 Vite 项目       │ ⚠️ 可以但非最优  │ ✅ 推荐              │
└──────────────────────┴──────────────────┴──────────────────────┘
```

### 3.4 经典解析策略（Classic）

早期 TypeScript 的解析方式，现在几乎不用了：

```typescript
// import { foo } from './utils'
// Classic 策略只查找：
//   ./utils.ts
//   ./utils.d.ts
// 不会查找 node_modules，也不会查找 package.json
// 仅用于向后兼容
```

### 3.5 模块解析流程总览

```
  tsconfig.json 中 moduleResolution 设置
      │
      ├── "classic" ─── 老项目兼容，不推荐
      │
      ├── "node10" / "node"
      │       │
      │       ├── 模拟 Node.js 10 的解析逻辑
      │       ├── 支持 node_modules 查找
      │       └── 不支持 package.json 的 exports
      │
      ├── "nodenext"
      │       │
      │       ├── 最接近 Node.js 真实行为
      │       ├── 完全支持 package.json exports
      │       ├── 强制要求写文件扩展名
      │       └── 适合纯 Node.js 项目
      │
      └── "bundler"  ← ✅ Vite / Vue 项目推荐
              │
              ├── 兼容打包工具的解析方式
              ├── 支持省略扩展名
              ├── 支持 exports 字段
              └── 不强制要求写扩展名
```

**推荐配置**（Vite + Vue 项目）：

```json
// tsconfig.json
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "bundler"
  }
}
```

## 四、路径映射

### 4.1 为什么需要路径映射？

当项目结构变深时，相对路径会变得难以维护：

```typescript
// 😫 嵌套地狱
import { formatDate } from '../../../shared/utils/date'
import { useAuth } from '../../../composables/useAuth'
import type { User } from '../../../types/user'

// 😊 路径映射后
import { formatDate } from '@shared/utils/date'
import { useAuth } from '@composables/useAuth'
import type { User } from '@types/user'
```

### 4.2 baseUrl 配置

`baseUrl` 设置模块解析的根目录起点：

```json
// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": "."
  }
}
```

```typescript
// 设了 baseUrl 为 "." 后，可以这样写
import { foo } from 'shared/utils'  // 从项目根目录开始找
```

> 注意：单独使用 `baseUrl` 不太推荐，因为容易和 `node_modules` 中的包名混淆。更推荐使用 `paths`。

### 4.3 paths 配置

`paths` 允许你定义路径别名，映射到实际文件路径：

```json
// tsconfig.json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@components/*": ["src/components/*"],
      "@composables/*": ["src/composables/*"],
      "@utils/*": ["src/utils/*"],
      "@types/*": ["src/types/*"]
    }
  }
}
```

```typescript
// 使用路径别名
import MyButton from '@components/MyButton.vue'
import { useAuth } from '@composables/useAuth'
import { formatDate } from '@utils/date'
import type { User } from '@types/user'
```

### 4.4 与 Vite 的 alias 配合

TypeScript 的 `paths` 只是**编译时**的路径映射，Vite 在运行时需要单独配置别名：

```typescript
// vite.config.ts
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import { resolve } from 'path'

export default defineConfig({
  plugins: [vue()],
  resolve: {
    alias: {
      '@': resolve(__dirname, 'src'),
      '@components': resolve(__dirname, 'src/components'),
      '@composables': resolve(__dirname, 'src/composables'),
      '@utils': resolve(__dirname, 'src/utils'),
    }
  }
})
```

> **关键点**：`tsconfig.json` 的 `paths` 和 `vite.config.ts` 的 `alias` 必须保持一致。TypeScript 用前者做类型检查，Vite 用后者做打包。两者缺一不可。

```
  ┌──────────────────────────────────────────────┐
  │              路径别名配置流程                   │
  │                                                │
  │  tsconfig.json paths    →  TypeScript 类型检查  │
  │       ↕ 必须一致                                │
  │  vite.config.ts alias   →  Vite 打包/开发服务器  │
  └──────────────────────────────────────────────┘
```

## 五、命名空间（Namespace）

### 5.1 什么是命名空间？

命名空间是 TypeScript **早期**提供的代码组织方式，用于在全局范围内避免命名冲突。类比成一个大箱子里面套小箱子：

```
┌─────────────────────────────────────────────┐
│  namespace Animals                          │
│  ├── Dog                                   │
│  ├── Cat                                   │
│  └── namespace Zoo                         │
│      ├── Tiger                             │
│      └── Lion                              │
└─────────────────────────────────────────────┘
```

### 5.2 基本语法

```typescript
// validation.ts
namespace Validation {
  // 不加 export 的是私有的
  const lettersRegexp = /^[A-Za-z]+$/
  const numberRegexp = /^[0-9]+$/

  // 加了 export 的才能在外部访问
  export interface StringValidator {
    isAcceptable(s: string): boolean
  }

  export class ZipCodeValidator implements StringValidator {
    isAcceptable(s: string): boolean {
      return s.length === 5 && numberRegexp.test(s)
    }
  }

  export class LettersOnlyValidator implements StringValidator {
    isAcceptable(s: string): boolean {
      return lettersRegexp.test(s)
    }
  }
}

// 使用
const validator = new Validation.ZipCodeValidator()
console.log(validator.isAcceptable('12345')) // true
```

### 5.3 嵌套命名空间

```typescript
namespace App {
  export namespace Models {
    export interface User {
      id: number
      name: string
    }
  }

  export namespace Services {
    export function getUser(id: number): Models.User {
      return { id, name: 'Tom' }
    }
  }

  export namespace Utils {
    export function formatUser(user: Models.User): string {
      return `#${user.id} ${user.name}`
    }
  }
}

// 使用时需要完整的限定路径
const user: App.Models.User = App.Services.getUser(1)
const formatted = App.Utils.formatUser(user)
```

### 5.4 命名空间合并

同名命名空间会自动合并，这和接口的声明合并类似：

```typescript
// animal.ts
namespace Animal {
  export function walk(): void {
    console.log('walking...')
  }
}

// animal-extra.ts（可以是另一个文件）
namespace Animal {
  export function run(): void {
    console.log('running...')
  }
}

// 使用时两个功能都可用
Animal.walk() // walking...
Animal.run()  // running...
```

**跨文件的命名空间合并**需要用三斜线指令 `/// <reference path="..." />` 来告知编译器依赖关系：

```typescript
// animal.ts
namespace Animal {
  export interface Pet {
    name: string
  }
}

// animal-dog.ts
/// <reference path="animal.ts" />
namespace Animal {
  export class Dog implements Pet {
    constructor(public name: string) {}
    bark(): void {
      console.log(`${this.name}: Woof!`)
    }
  }
}
```

### 5.5 命名空间与模块结合

命名空间可以放在模块内部，作为模块的导出：

```typescript
// shapes.ts —— 这是一个模块（有 export）
export namespace Shapes {
  export class Circle {
    constructor(public radius: number) {}
    area(): number {
      return Math.PI * this.radius ** 2
    }
  }

  export class Rectangle {
    constructor(public width: number, public height: number) {}
    area(): number {
      return this.width * this.height
    }
  }
}

// app.ts
import { Shapes } from './shapes'

const circle = new Shapes.Circle(5)
console.log(circle.area()) // 78.54...
```

> 这种用法是合理的：模块提供文件级别的隔离，命名空间提供模块内部的逻辑分组。

## 六、模块 vs 命名空间对比

### 6.1 大表格对比

```
┌─────────────────────┬──────────────────────┬──────────────────────┐
│       维度          │   模块（Module）      │  命名空间（Namespace） │
├─────────────────────┼──────────────────────┼──────────────────────┤
│ 本质               │ 文件级别的代码隔离     │ 全局级别的名称分组     │
│ 声明方式           │ import / export      │ namespace 关键字      │
│ 文件对应关系       │ 一个文件 = 一个模块   │ 跨文件合并            │
│ 加载方式           │ 按需加载              │ 全部包含              │
│ Tree-shaking      │ ✅ 支持              │ ❌ 不支持             │
│ 循环依赖处理       │ 有成熟方案            │ 容易出问题            │
│ 浏览器/Node 支持   │ ✅ 原生支持           │ ❌ 编译后全局变量      │
│ 作用域隔离         │ ✅ 文件级自动隔离     │ 需要手动 export       │
│ 与打包工具兼容     │ ✅ 完美               │ ⚠️ 需要额外处理       │
│ 第三方库使用       │ ✅ npm 生态标准       │ ❌ 需要声明文件转换    │
│ 异步/懒加载        │ ✅ 动态 import()     │ ❌ 不支持             │
│ 依赖关系           │ ✅ 显式声明           │ ⚠️ 隐式（reference）  │
│ 现代 TypeScript    │ ✅ 推荐方式           │ ⚠️ 遗留方案           │
│ 学习成本           │ 低（就是 JS 标准）    │ 中（TypeScript 特有） │
└─────────────────────┴──────────────────────┴──────────────────────┘
```

### 6.2 为什么推荐使用模块

1. **JavaScript 标准**：ES Module 是 ECMAScript 官方标准，所有现代浏览器和 Node.js 都支持
2. **打包工具友好**：Vite、webpack、Rollup 等都基于 ES Module，支持 tree-shaking
3. **依赖显式化**：`import` 明确声明了文件间的依赖关系，便于理解和维护
4. **npm 生态**：所有第三方包都以模块形式发布
5. **异步加载**：`import()` 支持代码分割和懒加载

### 6.3 命名空间的适用场景（极少数）

命名空间在现代项目中**几乎不需要使用**，但在以下极少数场景可能有用：

- **.d.ts 类型声明文件**：为没有类型的全局库添加类型声明
- **全局类型扩展**：扩展第三方库的类型（配合 `declare global`）
- **旧项目迁移**：逐步将老旧的命名空间代码迁移到模块

```typescript
// 扩展 Vue Router 的类型 —— 这是命名空间的合理用法
import 'vue-router'

declare module 'vue-router' {
  interface RouteMeta {
    requiresAuth?: boolean
    title?: string
    permissions?: string[]
  }
}

// 使用时就能获得类型提示
// router.push({ name: 'admin', meta: { requiresAuth: true, title: '管理后台' } })
```

## 七、循环依赖

### 7.1 什么是循环依赖？

A 依赖 B，B 又依赖 A，形成了一个环：

```
  ┌──────────┐         ┌──────────┐
  │   A.ts   │ ──────> │   B.ts   │
  │          │  import │          │
  │          │ <────── │          │
  └──────────┘  import └──────────┘
       ▲                        │
       │         ┌──────────┐   │
       └─────────│   C.ts   │<──┘
          import │          │
                 └──────────┘
```

```typescript
// a.ts
import { b } from './b'
export const a = 'A'
console.log(b) // ⚠️ 可能是 undefined！

// b.ts
import { a } from './a'
export const b = 'B'
console.log(a) // ⚠️ 可能是 undefined！
```

### 7.2 如何检测循环依赖

**工具方法**：

- **TypeScript 编译器**：部分情况下会给出警告
- **ESLint**：`import/no-cycle` 规则
- **Madge**：可视化依赖图，专门检测循环依赖

```bash
# 使用 madge 检测循环依赖
npx madge --circular src/

# 生成依赖图
npx madge --image graph.svg src/
```

```javascript
// .eslintrc.cjs
module.exports = {
  rules: {
    'import/no-cycle': ['error', { maxDepth: 5 }]
  }
}
```

### 7.3 解决方案

#### 方案一：重构，提取共享部分到独立模块

```
  循环依赖：                    重构后：
  A ←→ B                      A → Shared ← B

  ┌───┐   ┌───┐              ┌───┐   ┌───────┐   ┌───┐
  │ A │←─→│ B │      →      │ A │──→│Shared │←──│ B │
  └───┘   └───┘              └───┘   └───────┘   └───┘
```

```typescript
// shared/types.ts —— 提取共享的类型和常量
export interface Config {
  name: string
  value: number
}

export const DEFAULT_CONFIG: Config = { name: 'default', value: 0 }

// a.ts —— 只从 shared 导入
import type { Config } from './shared/types'
export function processA(config: Config) { /* ... */ }

// b.ts —— 只从 shared 导入
import type { Config } from './shared/types'
export function processB(config: Config) { /* ... */ }
```

#### 方案二：延迟导入（Lazy Import）

将 import 放到函数内部，在实际使用时才导入：

```typescript
// a.ts
export const a = 'A'

export function useB() {
  // 延迟到函数调用时才导入，避免模块加载阶段的循环
  const { b } = require('./b') // CommonJS 风格
  return b
}

// 或者用动态 import（ESM 风格）
export async function useBAsync() {
  const { b } = await import('./b')
  return b
}
```

#### 方案三：接口抽离（依赖倒置）

通过接口解耦，让双方都依赖接口而不是对方：

```typescript
// interfaces.ts
export interface IUserData {
  getName(): string
  getEmail(): string
}

export interface INotificationService {
  send(userId: string, message: string): void
}

// user.ts —— 只依赖接口
import type { IUserData, INotificationService } from './interfaces'

export class User implements IUserData {
  constructor(
    private name: string,
    private email: string,
    private notifier?: INotificationService // 注入，不直接导入
  ) {}

  getName(): string { return this.name }
  getEmail(): string { return this.email }

  notify(message: string): void {
    this.notifier?.send(this.name, message)
  }
}
```

## 八、模块的最佳实践

### 8.1 文件组织方式

推荐的项目结构（以 Vue + Vite + TypeScript 项目为例）：

```
src/
├── api/              # API 请求模块
│   ├── index.ts      # barrel 文件
│   ├── user.ts       # 用户相关 API
│   └── product.ts    # 产品相关 API
├── components/       # 公共组件
│   ├── index.ts
│   ├── Button.vue
│   └── Modal.vue
├── composables/      # 组合式函数
│   ├── index.ts
│   ├── useAuth.ts
│   └── useTheme.ts
├── types/            # 类型定义
│   ├── index.ts
│   ├── user.ts
│   └── api.ts
├── utils/            # 工具函数
│   ├── index.ts
│   ├── date.ts
│   └── format.ts
├── views/            # 页面
│   ├── Home.vue
│   └── Dashboard.vue
├── stores/           # 状态管理
│   ├── index.ts
│   └── user.ts
├── router/           # 路由
│   └── index.ts
├── App.vue
└── main.ts           # 入口文件
```

### 8.2 barrel 文件（index.ts）

barrel 文件将一个目录下的所有模块统一导出，简化导入路径：

```typescript
// composables/index.ts —— barrel 文件
export { useAuth } from './useAuth'
export { useTheme } from './useTheme'
export { useLoading } from './useLoading'
export type { AuthState } from './useAuth' // type-only 重导出

// 使用时只需要导入目录名
import { useAuth, useTheme } from '@/composables'
// 而不是：
// import { useAuth } from '@/composables/useAuth'
// import { useTheme } from '@/composables/useTheme'
```

> **注意**：barrel 文件过度使用可能影响 tree-shaking。如果你的项目使用 `isolatedModules` 或者对包体积非常敏感，可以考虑直接导入具体文件。

### 8.3 type-only 导入的好处

```typescript
// ✅ 推荐：使用 import type
import type { User, ApiResponse } from '@/types'

// 为什么推荐？
// 1. 语义清晰：一看就知道这只是类型，运行时不存在
// 2. isolatedModules 兼容：Vite/esbuild 等工具要求每个文件可独立编译
// 3. 防止误导入值：不会意外导入运行时代码
// 4. 编译更快：编译器知道可以直接跳过

// ✅ 内联 type（混合导入时）
import { createUser, type User } from '@/api/user'
//      ^^^^^^^^^^^ 值     ^^^^ 类型
```

```json
// tsconfig.json 中开启严格导入
{
  "compilerOptions": {
    "isolatedModules": true,
    "verbatimModuleSyntax": true // TS 5.0+，强制要求 type-only 导入
  }
}
```

### 8.4 tree-shaking 友好的导出

Tree-shaking 就像摇树让枯叶掉落——打包工具会移除未使用的代码：

```
  你写的代码                 打包后
  ┌─────────────┐          ┌─────────────┐
  │ export A    │          │ A           │
  │ export B    │ ──shake──→│ C           │
  │ export C    │          │             │
  │ export D    │          │ B 和 D 从未被导入，被"摇掉"了
  └─────────────┘          └─────────────┘
```

```typescript
// ✅ tree-shaking 友好：命名导出
export function add(a: number, b: number) { return a + b }
export function multiply(a: number, b: number) { return a * b }
// 打包工具能精确识别哪个函数被使用了

// ❌ tree-shaking 不友好：默认导出 + 对象
export default {
  add,
  multiply,
  divide,
}
// 打包工具难以判断对象中哪些属性被使用了

// ❌ 命名空间导出：不利于 tree-shaking
namespace Math {
  export function add() {}
  export function multiply() {}
}
export { Math }
```

**tree-shaking 最佳实践总结**：

1. 使用命名导出而非默认导出对象
2. 避免使用命名空间
3. 避免副作用（模块顶层不要有 `window.xxx = ...` 之类的操作）
4. 使用 `import type` 导入纯类型
5. 配置 `sideEffects: false`（package.json 中）

## 九、面试常见问题

### Q1：TypeScript 中模块和脚本的区别是什么？

**答**：TypeScript 根据文件中是否存在顶层的 `import` 或 `export` 语句来判断：
- 有 `import/export` → **模块**：所有声明都是文件私有的，只有 export 的才能被外部访问
- 没有 `import/export` → **脚本**：所有顶层声明都在全局命名空间中，可能命名冲突

现代项目中应该确保每个 `.ts` 文件都是模块，即使内容不需要导出，也建议加一个 `export {}`。

---

### Q2：moduleResolution 中的 "bundler" 和 "node" 有什么区别？Vite 项目应该用哪个？

**答**：

| 特性 | node | bundler |
|------|------|---------|
| 省略扩展名 | 需精确匹配 | 自动尝试 .ts/.tsx |
| package.json exports | 部分支持 | 完全支持 |
| 打包工具兼容 | 非最优 | 原生兼容 |
| 扩展名要求 | 不强制 | 不强制 |

Vite 项目推荐使用 `"bundler"`，因为它与 Vite 的解析逻辑更匹配，减少类型检查和实际运行时的不一致。

---

### Q3：namespace 和 module 有什么区别？为什么现代项目推荐用模块？

**答**：

- **namespace** 是 TypeScript 早期提供的全局级别代码组织方式，编译后变成全局变量
- **module** 基于 ES Module 标准，每个文件是一个模块，有真正的作用域隔离

推荐模块的原因：
1. ES Module 是 JavaScript 标准，浏览器和 Node.js 原生支持
2. 打包工具（Vite/webpack）基于模块工作，支持 tree-shaking
3. 模块的依赖关系是显式的（import），更易维护
4. 支持异步加载（`import()`），namespace 不支持
5. npm 生态全部基于模块

namespace 仅在类型声明文件（`.d.ts`）中扩展第三方库类型时可能用到。

---

### Q4：如何解决模块的循环依赖问题？

**答**：三种主要方案：

1. **提取共享模块**：把 A 和 B 共同依赖的部分抽到独立的模块 C 中，让 A → C ← B
2. **延迟导入**：将 `import` 放在函数内部（动态 `import()`），在实际使用时才加载
3. **接口抽离**：通过接口实现依赖倒置，双方都依赖抽象而非具体实现

最佳方案是第一种，从架构层面消除循环。第三种是设计模式层面的解耦。第二种是临时解决方案。

---

### Q5：`import type` 和普通 `import` 有什么区别？为什么要用 `import type`？

**答**：

- `import type` 导入的内容**只存在于类型层面**，编译后会被完全移除
- 普通 `import` 导入的内容在运行时也存在

使用 `import type` 的原因：
1. 语义更清晰，明确告诉读者"这只是类型"
2. 配合 `isolatedModules: true`（Vite 项目必须开启），确保每个文件可独立编译
3. 防止打包工具误将类型代码打包进产物
4. 开启 `verbatimModuleSyntax` 后，TypeScript 强制要求纯类型导入使用 `import type`

```typescript
import type { User } from './types'        // 纯类型，编译后移除
import { createUser, type User } from './api' // 混合导入，TS 4.5+
```
