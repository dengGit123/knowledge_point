# 问题一：使用 axios 封装请求时，如何定义接口返回的类型？

## 1. 先理解 axios 的响应结构

axios 每次请求都会返回一个 `AxiosResponse` 对象，它的核心结构是：

```ts
interface AxiosResponse<T = any> {
  data: T;          // 👈 后端真正返回的数据在这里
  status: number;       // HTTP 状态码，如 200
  statusText: string;   // 状态文本，如 "OK"
  headers: any;         // 响应头
  config: any;          // 请求配置
  request?: any;        // 原始请求对象
}
```

> 💡 **关键点：** 业务数据藏在 `response.data` 里。如果不指定类型，`data` 默认是 `any`，等于**丢掉了类型保护**。

## 2. 最基础：给请求方法传泛型

axios 的所有请求方法（`get`/`post`/`put`/`delete` 等）都支持泛型。给方法传一个类型参数，就能告诉它「`data` 是什么类型」：

```ts
import axios from 'axios';

// 定义接口返回的数据类型
interface User {
  id: number;
  name: string;
  email: string;
}

// 给 get 传 <User>，response.data 就会被识别为 User 类型
const res = await axios.get<User>('/api/user/1');

res.data.id;      // ✅ number
res.data.name;    // ✅ string
res.data.age;     // ❌ 报错：User 上没有 age 属性
```

**原理**：`axios.get<User>()` 的返回类型是 `` `Promise<AxiosResponse<User>>` ``，所以 `res.data` 的类型就是 `User`。

> ⚠️ **注意：** 这是「**相信接口返回的格式**」的标注方式。axios 不会真正校验返回数据，它只是按你给的类型做**类型假设**。如果后端实际返回和类型不一致，TS 不会报错，但运行时可能出问题。

## 3. 进阶：封装统一的 `request` 方法（只返回业务数据）

实际项目里，每次都拿 `res.data.data` 很啰嗦。更常见的做法是**封装一个方法，直接返回业务数据**。

### 3.1 定义后端统一响应格式

大多数后端会返回统一的结构：

```ts
// 后端统一响应格式
interface ApiResponse<T = unknown> {
  code: number;      // 业务状态码，0 表示成功
  message: string;   // 提示信息
  data: T;           // 真正的业务数据
}
```

### 3.2 封装请求实例 + 拦截器

```ts
import axios, {
  AxiosInstance,
  AxiosRequestConfig,
  AxiosResponse,
  AxiosError,
} from 'axios';

// 1. 创建 axios 实例
const service: AxiosInstance = axios.create({
  baseURL: '/api',
  timeout: 10000,
});

// 2. 请求拦截器：自动带上 token
service.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
  },
  (error) => Promise.reject(error),
);

// 3. 响应拦截器：统一处理业务错误码
service.interceptors.response.use(
  (response) => {
    const res = response.data as ApiResponse;
    // 业务约定：code 不为 0 视为失败
    if (res.code !== 0) {
      // 这里可以统一弹错误提示
      console.error(res.message);
      return Promise.reject(new Error(res.message || '请求失败'));
    }
    return response; // 成功则原样返回完整 response
  },
  (error: AxiosError) => {
    // HTTP 层面错误（如 404、500）
    if (error.response?.status === 401) {
      console.error('未登录或登录过期');
    }
    return Promise.reject(error);
  },
);
```

### 3.3 封装 `request`：泛型透传，只返回业务数据

```ts
// 封装的请求方法：直接返回 Promise<T>（业务数据）
export const request = {
  get<T>(url: string, config?: AxiosRequestConfig): Promise<T> {
    return service
      .get<ApiResponse<T>>(url, config)   // 指定 data 的类型是 ApiResponse<T>
      .then((res) => res.data.data);      // 两层 data：外层是响应体，内层是业务数据
  },

  post<T>(url: string, data?: unknown, config?: AxiosRequestConfig): Promise<T> {
    return service
      .post<ApiResponse<T>>(url, data, config)
      .then((res) => res.data.data);
  },

  put<T>(url: string, data?: unknown, config?: AxiosRequestConfig): Promise<T> {
    return service
      .put<ApiResponse<T>>(url, data, config)
      .then((res) => res.data.data);
  },

  delete<T>(url: string, config?: AxiosRequestConfig): Promise<T> {
    return service
      .delete<ApiResponse<T>>(url, config)
      .then((res) => res.data.data);
  },
};

export default request;
```

> 💡 **类型链路解析**（这是最核心的部分）：
> - `service.get<ApiResponse<T>>()` → 返回 `` `Promise<AxiosResponse<ApiResponse<T>>>` ``；
> - `.then(res => ...)` 里 `res` 的类型是 `AxiosResponse<ApiResponse<T>>`；
> - `res.data` → `ApiResponse<T>`（响应体）；
> - `res.data.data` → `T`（业务数据）。
>
> 最终 `request.get<User>()` 返回 `` `Promise<User>` ``，业务代码拿到的就是干净的业务数据。

## 4. 业务侧使用

```ts
import request from '@/utils/request';

// 定义业务接口类型
interface User { id: number; name: string; }
interface UserList { list: User[]; total: number; }

// 调用时传入类型，返回值自动推断
const user = await request.get<User>('/user/1');
user.name;   // ✅ string

// 分页列表
const result = await request.get<UserList>('/user/list', {
  params: { page: 1, size: 10 },
});
result.list.forEach((u) => console.log(u.name));
```

> 💡 **提示：** 建议把每个接口的类型集中放在 `types/` 目录或就近的 `api/user.ts` 里，便于复用和管理。

## 5. 常见坑与注意事项

| 问题 | 说明 | 建议 |
| --- | --- | --- |
| `res.data` 是 `any` | 没给方法传泛型 | 始终传泛型 `get<User>()` |
| 类型是「骗」TS 的 | TS 不校验真实数据，后端改了字段 TS 不知道 | 关键接口可用运行时校验（如 [zod](https://zod.dev)）兜底 |
| 拦截器 `return res.data` 类型报错 | 响应拦截器要求返回 `AxiosResponse` | 把解包放到封装方法里（如本例），拦截器只做错误处理 |
| `data` 字段两层含义 | axios 的 `data` 和后端的 `data` 都叫 data | 心里分清：axios 的 `res.data` 是响应体，再 `.data` 才是业务数据 |
| 泛型默认值报错 | `` `ApiResponse<T>` `` 里 `T` 没约束 | 给默认值 `T = unknown`，避免 `T` 未传时类型丢失 |

# 问题二：不确定字段个数时，类型该如何定义？

## 1. 先分清是哪种「不确定」

「不确定字段个数」其实有好几种情况，对应的方案完全不同：

| 情况 | 例子 | 合适的方案 |
| --- | --- | --- |
| 字段名和数量都完全未知 | 一个 `Map` 式配置对象 | **索引签名** / `Record` |
| 有几个固定字段，但也可能多出别的 | 后端返回的扩展字段 | **固定字段 + 索引签名** |
| 字段名固定，但某些可能不存在 | 可选的表单字段 | **`Partial`** |
| 键可以枚举，数量不定 | 状态映射、字典 | **`Record<枚举键, 值>`** |
| 运行时频繁增删键 | 缓存对象 | **`Map` 类型** |

下面逐一讲解。

## 2. 方案一：索引签名（字段名完全未知）

**索引签名**（index signature）的写法是 `` `[键名: 键类型]: 值类型` ``，表示「这个对象可以有任意数量的键，只要键是某种类型、值是某种类型」。

```ts
// 表示：一个对象，可以有任意个 string 类型的键，值都是 number
interface Scores {
  [subject: string]: number;
}

const scores: Scores = {
  math: 90,
  english: 85,
  physics: 78,
  // 想加多少科加多少科
};

scores.math;       // number
scores['history']; // number（即使没定义，类型上也是 number）
```

> ⚠️ **注意（重要陷阱）：** 索引签名会让**任何键访问都返回值类型**，哪怕这个键根本不存在。`scores.unknown` 类型上是 `number`，但运行时其实是 `undefined`。需要自己注意空值。

## 3. 方案二：`Record` 工具类型（更简洁）

`Record<K, V>` 是 TypeScript 内置的工具类型，等价于「键为 `K`、值为 `V` 的对象」。它比手写索引签名更简洁。

```ts
// 等价于 { [key: string]: number }
const scores: Record<string, number> = {
  math: 90,
  english: 85,
};

// ⭐ 更强大的用法：键是一个具体的联合类型（枚举）
type Status = 'pending' | 'success' | 'error';

const statusText: Record<Status, string> = {
  pending: '处理中',
  success: '成功',
  error: '失败',
};
```

> 💡 **提示：** 当「键可以枚举、但值统一」时，`Record<联合类型, V>` 是**最安全**的写法——它会在编译期检查你有没有漏掉某个键（见下方对比）。

**索引签名 vs `Record` 对比：**

| 写法 | 键的约束 | 会校验是否遗漏键吗 |
| --- | --- | --- |
| `` `{ [k: string]: V }` `` | 任意 string | ❌ 不会 |
| `Record<string, V>` | 任意 string | ❌ 不会 |
| `Record<'a' \| 'b', V>` | 只能是 `a`/`b` | ✅ 会（少写/多写都报错） |

```ts
type Status = 'pending' | 'success' | 'error';

// ❌ 报错：缺少 error 属性
const bad: Record<Status, string> = {
  pending: '处理中',
  success: '成功',
};

// ✅ 正确：三个键都写全了
const good: Record<Status, string> = {
  pending: '处理中',
  success: '成功',
  error: '失败',
};
```

## 4. 方案三：固定字段 + 索引签名（有已知字段，也允许额外字段）

接口返回的数据，经常是「**有几个明确字段，还可能带一些扩展字段**」。这时可以**把已知字段写出来，再加一个索引签名兜底**。

```ts
interface UserResponse {
  // 已知的固定字段
  id: number;
  name: string;
  // 兜底：允许任何额外的 string 键，值为 unknown
  [key: string]: unknown;
}

const user: UserResponse = {
  id: 1,
  name: 'Tom',
  // 扩展字段也允许
  avatar: 'https://...',
  nickname: '汤姆',
  vipLevel: 3,
};

user.id;       // number
user.avatar;   // unknown（扩展字段只能拿到 unknown）
```

> ⚠️ **注意一个限制：** 当同时存在固定字段和索引签名时，**固定字段的类型必须能赋给索引签名的值类型**。比如下面这样会报错：

```ts
// ❌ 报错：id 是 number，但索引签名要求值是 string，二者冲突
interface Bad {
  id: number;
  [key: string]: string;
}
```

解决办法是把索引签名的值类型放宽（如 `unknown` 或 `number | string`）。

## 5. 方案四：`Partial`（字段名固定，但可能不存在）

如果字段是**已知的**，只是**每个都可能不存在**，用 `Partial<T>` 最合适——它把所有字段变成可选的。

```ts
interface UpdateUser {
  name: string;
  age: number;
  email: string;
}

// Partial<UpdateUser> 等价于 { name?: string; age?: number; email?: string }
function updateUser(id: number, data: Partial<UpdateUser>) {
  // 只传想改的字段即可
}

updateUser(1, { name: '新名字' });       // ✅ 只改 name
updateUser(1, { name: 'x', age: 20 });    // ✅ 改多个
updateUser(1, {});                          // ✅ 啥都不改也行
```

> 💡 **提示：** 这和索引签名的区别在于——`Partial` 的**字段名是固定**的，你访问 `data.xxx`（不在原类型里）会报错；而索引签名允许任意字段名。**「字段固定但可能缺失」用 `Partial`，「字段名都不确定」用索引签名/`Record`。**

## 6. 方案五：`Map` 类型（运行时频繁增删键）

如果是**运行时**需要频繁增删的键值对（缓存、计数器等），用 `Map` 比用普通对象更合适，类型也更清晰。

```ts
// 键是 string，值是 number 的 Map
const visitCount = new Map<string, number>();

visitCount.set('home', 1);
visitCount.set('about', 3);
visitCount.delete('home');

const count = visitCount.get('about'); // number | undefined
```

**对象（索引签名）vs `Map`：**

| 特性 | 对象 `{ [k]: V }` | `Map<K, V>` |
| --- | --- | --- |
| 键的类型 | 只能 `string` / `number` / `symbol` | 任意（包括对象） |
| 运行时增删 | `delete obj.x`（性能一般） | `set` / `delete`（高效） |
| 遍历顺序 | 基本按插入顺序 | 严格按插入顺序 |
| 序列化（JSON） | ✅ 方便 | ❌ 需手动转换 |
| 适用场景 | 配置、接口数据 | 运行时动态键值对、缓存 |

## 7. 方案选择速查

```
字段情况？
│
├─ 字段名完全不知道 / 任意键
│    ├─ 键是任意 string  →  索引签名 [key: string]: V  或  Record<string, V>
│    └─ 键可枚举（如状态） →  Record<'a'|'b'|'c', V>（推荐，有校验）
│
├─ 有几个固定字段，但也可能多出别的
│     →  固定字段 + [key: string]: unknown
│
├─ 字段名固定，但可能缺失
│     →  Partial<T>
│
└─ 运行时频繁增删键（缓存等）
      →  Map<K, V>
```

## 8. 常见坑与注意事项

1. **索引签名访问不存在的键不报错**：`obj.someUnknownKey` 类型上是值类型，运行时却是 `undefined`。访问前最好用 `in` 判断，或把值类型写成 `V | undefined`。

2. **JSON 反序列化后没有类型**：`JSON.parse(...)` 返回 `any`，再复杂的类型定义也无法保证运行时结构。重要数据建议用 [zod](https://zod.dev) / [ajv](https://ajv.js.org/) 做运行时校验。

3. **不要滥用 `Record<string, any>`**：它和 `any` 一样会**关闭类型检查**。如果某些字段明确，就单独写出来，只对真正不确定的部分用索引签名。

4. **键值同构 vs 异构**：索引签名要求**所有值同类型**。如果不同键的值类型不同（如 `{ name: string, age: number, ... }`），要么逐个定义字段，要么用联合类型 `` `V = string | number` ``（会损失精度）。

# 问题三：联合类型怎么收窄（Narrowing）？

## 1. 什么是「收窄」

联合类型（如 `string | number`）表示一个值可能是几种类型之一。**收窄**（narrowing）就是：在某个代码分支里，TypeScript 根据**运行时条件**，把这个联合类型「**缩小**」成更具体的类型，从而能安全访问该类型独有的属性。

```ts
function padLeft(value: string | number) {
  // 这里 value 是 string | number
  if (typeof value === 'string') {
    // 👉 收窄：此分支里 value 被识别为 string
    return value.padStart(10, ' '); // ✅ 可以调用字符串方法
  }
  // 这里 value 被识别为 number
  return value.toFixed(2); // ✅ 可以调用数字方法
}
```

> 💡 **通俗理解：** 联合类型像「**这可能是 A，也可能是 B**」。收窄就是「**通过一道检查后，TS 确定它现在只可能是 A 了**」，于是放行 A 独有的操作。

## 2. 六种常用收窄方式

### 2.1 `typeof`（原始类型）

适用于 `string` / `number` / `boolean` / `undefined` / `symbol` / `bigint` 等。

```ts
function format(value: string | number) {
  if (typeof value === 'string') {
    value.toUpperCase(); // ✅ string
  } else {
    value.toFixed(2);    // ✅ number
  }
}
```

> ⚠️ **注意：** `typeof null === 'object'`，所以用 `typeof` 区分对象和 `null` 不可靠，要用 `=== null`。

### 2.2 真值收窄（Truthiness）

对 `0`、`''`、`null`、`undefined`、`NaN` 等「假值」做判断。

```ts
function printAll(strs: string | string[] | null) {
  if (strs) {
    // 排除了 null（以及空字符串、空数组等假值）
    strs.forEach((s) => console.log(s)); // ✅ string[]
  }
}
```

### 2.3 相等性收窄（`===` / `==` / `!==`）

通过和具体值比较来收窄，TS 会取两个类型的交集。

```ts
function example(x: string | number, y: string | boolean) {
  if (x === y) {
    // x 和 y 在此都是 string（它们的交集）
    x.toUpperCase(); // ✅
    y.toUpperCase(); // ✅
  }
}
```

### 2.4 `in` 操作符（判断属性是否存在）

适用于有不同属性的对象联合类型。

```ts
interface Fish { swim: () => void }
interface Bird { fly: () => void }

function move(animal: Fish | Bird) {
  if ('swim' in animal) {
    animal.swim(); // ✅ Fish
  } else {
    animal.fly();  // ✅ Bird
  }
}
```

### 2.5 `instanceof`（类实例）

判断对象是不是某个类的实例。

```ts
function logValue(x: Date | string) {
  if (x instanceof Date) {
    x.getTime(); // ✅ Date
  } else {
    x.toUpperCase(); // ✅ string
  }
}
```

### 2.6 判别联合（Discriminated Union）⭐ 最推荐

给联合类型的每个成员加一个**公共的字面量类型字段**（通常是 `type` 或 `kind`），TS 就能根据这个字段精确收窄。这是**最强大、最常用**的模式。

```ts
// 每个 shape 都有一个 kind 字段作为「判别标识」
interface Circle { kind: 'circle'; radius: number }
interface Square { kind: 'square'; size: number }

type Shape = Circle | Square;

function area(shape: Shape) {
  switch (shape.kind) {
    case 'circle':
      return Math.PI * shape.radius ** 2; // ✅ shape 是 Circle
    case 'square':
      return shape.size ** 2;             // ✅ shape 是 Square
  }
}
```

> 💡 **提示：** 判别联合配合 `switch` 是处理「多种形态数据」（接口返回的不同消息类型、不同表单控件）的最佳实践。再加 `never` 做穷尽检查（见下方），还能在新增类型时强制你补全分支。

## 3. 自定义类型守卫（Type Predicate）

当内置手段不够用时，可以写一个**返回类型为 `参数 is 类型` 的函数**，让 TS 把它当作「类型判断器」。

```ts
interface Fish { swim: () => void }
interface Bird { fly: () => void }

// 返回值用「参数 is 类型」声明，这就是类型守卫
function isFish(pet: Fish | Bird): pet is Fish {
  return (pet as Fish).swim !== undefined;
}

const pet = (Math.random() > 0.5
  ? { swim() {} }
  : { fly() {} }) as Fish | Bird;

if (isFish(pet)) {
  pet.swim(); // ✅ TS 知道这里是 Fish
} else {
  pet.fly();  // ✅ Bird
}
```

> 💡 **提示：** 项目里常见的 `isNonEmptyArray`、`isError(e): e is Error` 等都是自定义类型守卫，适合封装复杂的判断逻辑。

## 4. 常见坑

| 坑 | 说明 |
| --- | --- |
| 收窄在函数调用后**失效** | 把变量传进函数再读出来，TS 不保证类型仍收窄着。复杂场景把收窄结果先存到 `const` |
| `typeof` 拦不住 `null` | `typeof null === 'object'`，对象判空要用 `!== null` |
| 联合数组遍历类型是「并集」 | `(string \| number)[]` 遍历时每个元素仍是 `string \| number`，需逐个收窄 |
| 漏判分支 | 用判别联合 + `never` 穷尽检查，可在漏掉 `case` 时让 TS 报错 |

**用 `never` 做穷尽检查**（确保所有分支都被处理）：

```ts
function area(shape: Shape): number {
  switch (shape.kind) {
    case 'circle': return Math.PI * shape.radius ** 2;
    case 'square': return shape.size ** 2;
    default:
      // 将来新增 Triangle 却忘了加 case 时，
      // 这里的 shape 不再是 never，编译报错——强制你补全分支
      const _exhaustive: never = shape;
      return _exhaustive;
  }
}
```

# 问题四：泛型组件怎么写（Vue 3.3+）？

## 1. 为什么需要泛型组件

普通组件的 props 类型是**固定**的。但有些组件希望**类型跟随使用场景变化**——比如一个 `Select` 组件，选项值可能是 `string`、也可能是 `number`；一个 `Table` 组件，每行数据类型由调用方决定。这时就需要**泛型组件**：让组件像泛型函数一样，接收一个类型参数。

## 2. 推荐写法：`<script setup>` + `generic` 属性（Vue 3.3+）

在 `<script>` 标签上加 `generic` 属性，就能声明泛型类型参数。它的值就是一组泛型参数，写法和 TypeScript 的泛型参数列表完全一致：

```vue
<!-- 泛型选择器组件：值的类型 T 由调用方决定 -->
<script setup lang="ts" generic="T">
import { ref } from 'vue'

// props 用到泛型 T
defineProps<{
  options: T[]          // 选项数组，元素类型是 T
  modelValue: T | null  // 当前选中值，类型也是 T
}>()

const emit = defineEmits<{
  'update:modelValue': [value: T] // 触发事件时值类型是 T
}>()
</script>
```

**使用时，T 会根据传入的 props 自动推断**：

```vue
<script setup lang="ts">
import Select from './Select.vue'
import { ref } from 'vue'

// 传入 string[]，T 自动推断为 string
const city = ref<string | null>(null)
</script>

<template>
  <!-- modelValue 只接受 string | null，传 number 会报错 ✅ -->
  <Select v-model="city" :options="['北京', '上海', '广州']" />
</template>
```

## 3. 多个泛型参数、约束与默认值

`generic` 支持 TypeScript 泛型的全部语法——多个参数、`extends` 约束、默认类型、引用导入的类型：

```vue
<script
  setup
  lang="ts"
  generic="T extends string | number, U extends Item"
>
import type { Item } from './types'

defineProps<{
  id: T     // T 被约束为 string 或 number
  list: U[] // U 被约束为 Item 类型
}>()
</script>
```

> 💡 **提示：**
> - `T extends xxx` 表示「T 必须是 xxx 的子类型」，相当于给泛型加限制；
> - 可以引用 `import type` 进来的类型作为约束（如 `U extends Item`）。

## 4. 典型场景：泛型列表组件

```vue
<!-- List.vue：通用列表，渲染任意类型的列表项 -->
<script setup lang="ts" generic="T">
defineProps<{
  items: T[]
  labelKey: keyof T // labelKey 必须是 T 的某个键
}>()
</script>

<template>
  <ul>
    <li v-for="(item, index) in items" :key="index">
      {{ item[labelKey] }}
    </li>
  </ul>
</template>
```

```vue
<!-- 使用：items 是 User[]，labelKey 只能是 User 的某个键 -->
<script setup lang="ts">
import List from './List.vue'

interface User { id: number; name: string; age: number }
const users: User[] = [
  { id: 1, name: 'Tom', age: 20 },
  { id: 2, name: 'Lucy', age: 22 },
]
</script>

<template>
  <!-- label-key="name" ✅；写成 label-key="xxx" 会报错（不是 User 的键）-->
  <List :items="users" label-key="name" />
</template>
```

## 5. 用 `defineComponent` 写泛型组件（不推荐）

用选项式 `defineComponent` 写泛型组件**比较繁琐**，且 `T` 难以从 props 自动推断。Vue 官方推荐用 **`<script setup>` + `generic` 属性**。只在不能用 `<script setup>` 的特殊场景（如纯 `.ts` 文件里的函数式组件），才考虑下面的工厂函数写法：

```ts
import { defineComponent, PropType } from 'vue'

// 用泛型工厂函数包裹
function defineGenericList<T>() {
  return defineComponent({
    props: {
      // 用 PropType 标注涉及 T 的 props
      items: Array as PropType<T[]>,
    },
    setup(props) {
      // props.items 类型是 T[]
      return () => null
    },
  })
}
```

> ⚠️ **注意：** 这种写法在很多场景下类型推导会失真，实际开发**优先用 `<script setup generic>`**。

## 6. 注意事项

| 项 | 说明 |
| --- | --- |
| 版本要求 | `generic` 属性需要 **Vue 3.3+** |
| 类型推断失败时 | 模板里可用 `@vue-generic {类型}` 指令**显式指定**泛型实参 |
| `ref` 引用泛型组件 | `InstanceType` 不生效，需借助 `vue-component-type-helpers` 库的 `ComponentExposed` |
| 不支持复杂条件类型 | `generic` 内的类型→运行时转换基于 AST，单个 prop 内可用条件类型，但整个 props 对象不行 |

# 问题五：第三方库没有类型怎么办？

当 `import` 一个库报「**找不到模块 xxx 的声明文件**（`Could not find a declaration file for module 'xxx'`）」时，按下面的顺序排查解决。

## 1. 第一步：找官方/社区的类型包（首选）

绝大多数流行库都有对应的类型声明。用 [TypeSearch](https://www.typescriptlang.org/dt/search) 搜一下，或直接装 `@types/xxx`：

```bash
# 比如用 lodash 但没类型
npm install -D @types/lodash
```

> 💡 **提示：** `@types` 包来自社区项目 [DefinitelyTyped](https://github.com/DefinitelyTyped/DefinitelyTyped)，安装后会自动被 TS 识别（位于 `node_modules/@types/` 下）。这是**最省心**的办法。

## 2. 第二步：检查库自带类型

很多现代库**自带类型**（在 `package.json` 里有 `types` 或 `typings` 字段，或直接发布 `.d.ts`）。如果还报错，检查：

- 库版本太旧 → 升级到带类型的版本；
- 子路径导入没类型 → 看文档是否需要换导入方式。

## 3. 第三步：自己写声明文件（最灵活）

如果既没自带类型、也没有 `@types`，就**手写一份 `.d.ts` 声明**。完整写法见 [[声明模块]]，这里给最常用的几种：

### 3.1 快速消错：最简声明（退化为 any）

```ts
// shim.d.ts 或 types/global.d.ts
declare module 'some-untyped-lib'; // 整个模块被当作 any
```

> ⚠️ **注意：** 这相当于**关掉类型检查**，仅作临时消错，不推荐长期使用。

### 3.2 精确声明：定义真实的类型

```ts
declare module 'some-lib' {
  export interface Options { timeout?: number; retries?: number }
  export function init(options: Options): void;
  export const version: string;
}
```

### 3.3 为非 JS 资源声明（图片、Markdown 等）

```ts
declare module '*.svg' {
  const src: string;
  export default src;
}
```

> 💡 **提示：** 声明文件放在 `tsconfig.json` 的 `include` 范围内（或默认的 `src` 下）就会被自动加载。详见 [[声明模块]] 和 [[tsconfig配置文件]]。

## 4. 第四步：临时绕过（应急，不推荐）

实在不想写类型，可用注释**单行屏蔽报错**：

```ts
// @ts-expect-error 该库暂无类型声明
import oldLib from 'legacy-lib';

// 或
// @ts-ignore
import other from 'no-types';
```

两者区别：

| 注释 | 行为 |
| --- | --- |
| `// @ts-expect-error` | **期望下一行有错**；若下一行其实没错，它反而会报错（提醒你该删掉这个屏蔽了） |
| `// @ts-ignore` | 无脑屏蔽下一行错误；下一行没错也不提醒 |

> 💡 **提示：** 推荐用 `@ts-expect-error`，它不会「悄悄掩盖」已经解决的问题。

## 5. 第五步：从 JS 自动生成声明（针对自己的/纯 JS 库）

如果库是你自己的纯 JS 代码，可让 `tsc` 帮忙生成声明文件：

```bash
# 把 .js 编译出对应的 .d.ts
tsc --declaration --emitDeclarationOnly --allowJs --outDir types src/index.js
```

> 💡 **提示：** 配合 JSDoc 注释能让生成的 `.d.ts` 带上类型信息，质量更高。也可用微软官方的 [dts-gen](https://github.com/microsoft/dts-gen) 工具为 npm 包生成骨架。

## 6. 决策流程

```
报错「找不到声明文件」
│
├─ 1. 能搜到 @types/xxx 吗？  →  装 @types 包 ✅（首选）
│
├─ 2. 库自带类型吗？         →  升级版本 / 检查导入方式
│
├─ 3. 都没有                 →  手写 .d.ts（精确声明优于 any 占位）
│
└─ 4. 临时应急               →  // @ts-expect-error（记得后续补类型）
```

# 总结

**问题一（axios 类型）**：

- axios 请求方法都支持泛型，`get<User>()` → `res.data` 即为 `User`；
- 实战中**封装 `request` 方法**，用 `` `ApiResponse<T>` `` 配合 `.then(res => res.data.data)`，对外只返回 `` `Promise<T>` ``（干净的业务数据）；
- 拦截器负责 token 注入和错误码处理，**解包逻辑放在封装方法里**避免类型冲突。

**问题二（不确定字段）**：

- 完全未知的键 → **索引签名** `` `[key: string]: V` `` 或 **`Record`**；
- 键可枚举 → **`Record<联合类型, V>`**（最安全，有遗漏校验）；
- 固定字段 + 扩展字段 → **固定字段 + 索引签名**；
- 字段固定但可能缺失 → **`Partial<T>`**；
- 运行时动态键值对 → **`Map<K, V>`**。

**问题三（联合类型收窄）**：

- 用 `typeof`、真值判断、相等性（`===`）、`in`、`instanceof` 等把联合类型**缩小**到具体类型；
- **判别联合**（公共字面量字段 + `switch`）是处理多形态数据的最佳实践；
- 复杂判断写**自定义类型守卫**（`参数 is 类型`）；
- 用 `never` 做**穷尽检查**，防止漏掉分支。

**问题四（泛型组件）**：

- Vue 3.3+ 用 `<script setup generic="T">` 声明泛型类型参数；
- `generic` 值支持多参数、`extends` 约束、默认类型、引用导入类型；
- **优先用 `generic`**，而非 `defineComponent` 工厂函数；
- 推断失败用 `@vue-generic`，`ref` 引用泛型组件用 `vue-component-type-helpers`。

**问题五（第三方库没类型）**：

- 优先装 **`@types/xxx`**（用 [TypeSearch](https://www.typescriptlang.org/dt/search) 查）；
- 其次检查库是否**自带类型**（升级版本 / 换导入路径）；
- 都没有就**手写 `.d.ts`**（精确声明优于 `declare module 'x';` 占位）；
- 临时应急用 `// @ts-expect-error`（**优于** `@ts-ignore`，解决问题时会主动提醒）。

> 💡 **一句话记忆：** axios 封装靠「泛型透传 + 统一响应格式」拿到 `` `Promise<T>` ``；不确定字段靠「索引签名 / `Record` 兜底任意键，`Partial` 表达可能缺失」——前者是「**告诉 axios 数据长什么样**」，后者是「**告诉 TS 键的开放程度**」。
