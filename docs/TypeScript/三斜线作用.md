# TypeScript 三斜线指令（Triple-Slash Directives）

> 📖 [Triple-Slash Directives —— TypeScript 官方文档](https://www.typescriptlang.org/docs/handbook/triple-slash-directives.html)

## 一、概述

**三斜线指令（Triple-Slash Directives）** 是一种写在文件顶部的特殊单行注释，注释里包含一个 XML 标签，用来给 TypeScript 编译器（`tsc`）下达指令。

简单说：它就是**写给编译器看的「注释」**，告诉编译器「编译这个文件时，还要带上谁」「这个文件依赖哪个类型包」等等。

```ts
// 形如这样：三个斜杠 + 一个 XML 标签
/// <reference path="./other.ts" />
```

**为什么需要它？**

在没有 ES Module（`import` / `export`）的年代，多个 `.ts` / `.d.ts` 文件之间没有显式的依赖声明，编译器不知道该把哪些文件一起编译。三斜线指令就是那个时候用来声明文件依赖的手段。

> ⚠️ **注意：** 现代 TypeScript 项目里，文件之间的依赖**优先用标准的 `import` / `export` 表达**，三斜线指令的使用已经大大减少。但在**编写声明文件（`.d.ts`）**、**按需引入内置 `lib`**、**AMD 模块命名**等场景下，它仍然是官方推荐的方式。

## 二、基本规则（最容易踩坑）

三斜线指令有严格的格式和位置要求，不符合要求时它会被当成普通注释、**失去作用**。

### 2.1 必须写在文件顶部

三斜线指令**只能出现在文件顶部**。它的前面只能有单行注释、多行注释或其他三斜线指令。一旦它前面出现了任何语句或声明（哪怕是一个 `import`），它就会变成普通注释，不再有指令效果。

```ts
// ✅ 正确：写在最顶部
/// <reference path="./utils.ts" />

import { foo } from './bar'; // import 在三斜线之后，没问题
```

```ts
// ❌ 错误：出现在 import 之后，指令失效
import { foo } from './bar';

/// <reference path="./utils.ts" />  // 这里它只是普通注释，编译器不会理会
```

> 💡 **提示：** 三斜线指令必须用一个**单行注释**承载，且必须是**三个斜杠** `///`，写成 `//` 或 `/**/` 都不行。多个指令可以连续写。

### 2.2 编译器对它的处理（TypeScript 5.5+）

从 **TypeScript 5.5** 起，编译器有了一个重要变化：

- 编译器**不再自动生成**三斜线 `reference` 指令；
- 对于你手写的三斜线指令，编译器**默认会从输出中删除**它们，除非你显式加上 `preserve="true"`。

```ts
// 不加 preserve，编译产物里会被删除
/// <reference path="./other.ts" />

// 加上 preserve，会保留到输出
/// <reference path="./other.ts" preserve="true" />
```

这点对写库/发布 `.d.ts` 的人尤其重要——如果你的产物依赖这些指令被下游工具读到，记得加 `preserve="true"`。

## 三、各类指令详解

### 3.1 `/// <reference path="..." />`

这是**最常用**的一个，用来声明**文件之间的依赖**：告诉编译器「编译当前文件时，请把 `path` 指向的文件也一起加进来编译」。

```ts
// 引用同级目录下的 utils.ts，编译器会把它一并纳入编译
/// <reference path="./utils.ts" />

const result = add(1, 2); // add 定义在 utils.ts 中
```

**几个要点：**

- `path` 是相对路径时，**相对于当前文件所在目录**解析；
- 它还会影响输出顺序：使用 `outFile` 把多个文件合并成一个时，文件按预处理后的依赖顺序输出；
- 编译器会对输入文件做一次**预处理**：从根文件（命令行或 `tsconfig.json` 的 `files` 指定的文件）出发，**深度优先**地解析三斜线引用，把被引用的文件也加入编译；
- 引用不存在的文件、或引用文件自身，都会**报错**。

> 💡 **提示：** 如果编译时加了 `--noResolve` 标志，三斜线引用会被忽略——既不新增文件，也不改变文件顺序。

### 3.2 `/// <reference types="..." />`

它声明对**一个类型包（通常是 `@types/xxx`）**的依赖，可以理解为「**给类型包写的 `import`**」。

```ts
// 声明：本文件用到了 @types/node 里定义的类型（如 process、Buffer）
/// <reference types="node" />

process.env.HOME; // 有了这行引用，process 才能被识别
```

**使用场景区分：**

| 文件类型 | 推荐 |
| --- | --- |
| `.d.ts` 声明文件 | 用 `/// <reference types="node" />` |
| 普通 `.ts` 文件 | 不要用这个，改用命令行参数或 `tsconfig.json` 的 `types` 选项 |

> ⚠️ **注意：** 在普通 `.ts` 源码文件里声明 `@types` 依赖，正确的做法是配置 `tsconfig.json` 的 `types` / `typeRoots`，而不是写三斜线指令。

### 3.3 `/// <reference lib="..." />`

让某个文件**单独引入一个内置 `lib`**，效果等同于给整个编译加 `--lib xxx`，但作用范围更小——只影响当前文件。

内置 `lib` 的写法和 `tsconfig.json` 的 `lib` 选项一致，比如写 `es2017.string` 而不是 `lib.es2017.d.ts`。

```ts
// 在这个文件里启用 ES2017 的字符串方法（如 padStart）
/// <reference lib="es2017.string" />

'foo'.padStart(4); // 不会报错
```

> 💡 **提示：** 编写声明文件（`.d.ts`）时，如果依赖 `DOM` API、`Symbol`、`Iterable` 这类内置运行时类型，官方推荐用 `reference lib` 指令显式声明依赖，而不是自己重复定义一遍这些类型。

### 3.4 `/// <reference no-default-lib="true"/>`

把当前文件**标记为「默认库」**。你会看到 `lib.d.ts` 及其各种变体文件顶部都有这行。

它的作用是**告诉编译器不要把默认库（`lib.d.ts`）自动包含进来**，效果类似于命令行的 `--noLib`。

```ts
/// <reference no-default-lib="true"/>
```

> 💡 **提示：** 这个指令还和 `skipDefaultLibCheck` 配合：只有带这行标记的文件，才会被「跳过类型检查」。日常项目开发几乎用不到，主要是给 TypeScript 自带的 `lib` 文件用的。

### 3.5 `/// <amd-module name="..." />`

给编译出的 **AMD 模块**起一个名字。默认情况下生成的 AMD 模块是**匿名**的，这会让一些打包工具（如 `r.js`）处理起来有困难，这个指令可以为它命名。

```ts
// amdModule.ts
/// <amd-module name="NamedModule" />

export class C {}
```

编译后：

```js
define("NamedModule", ["require", "exports"], function (require, exports) {
  var C = (function () {
    function C() {}
    return C;
  })();
  exports.C = C;
});
```

> ⚠️ **注意：** AMD 是一种较老的模块规范，现代项目基本都用 ES Module。这个指令只有在需要输出 AMD 格式（`"module": "amd"`）时才用得上。

### 3.6 `/// <amd-dependency />`（已废弃）

```ts
// 已废弃 ❌
/// <amd-dependency path="legacy/moduleA" name="moduleA" />

declare var moduleA: MyType;
moduleA.callStuff();
```

它原本用来声明对一个**非 TS 模块**的依赖，让编译器把它注入到产物的 `require` 调用里。

> ⚠️ **注意：** 该指令**已被废弃**，官方建议改用 `import "moduleName";` 语句替代。

### 3.7 `preserve="true"`

如前文 2.2 所述，从 TypeScript 5.5 起，默认会从输出中删除三斜线指令。给 `reference` 类指令加上 `preserve="true"`，可以让它**保留在编译产物里**：

```ts
// 会被保留到输出产物
/// <reference path="./other.ts" preserve="true" />
/// <reference types="node" preserve="true" />
/// <reference lib="es2017.string" preserve="true" />
```

## 四、指令速查表

| 指令 | 作用 | 适用场景 |
| --- | --- | --- |
| `/// <reference path="..." />` | 声明对另一个文件的依赖，将其纳入编译 | 文件间依赖（旧代码 / `.d.ts`） |
| `/// <reference types="..." />` | 声明对类型包（`@types/xxx`）的依赖 | `.d.ts` 声明文件 |
| `/// <reference lib="..." />` | 单文件级引入内置 `lib` | `.d.ts` 中声明依赖 DOM / 内置运行时类型 |
| `/// <reference no-default-lib="true"/>` | 标记为默认库，不自动引入 `lib.d.ts` | TS 自带 `lib` 文件（日常不用） |
| `/// <amd-module name="..." />` | 为 AMD 模块命名 | 输出 AMD 格式模块 |
| `/// <amd-dependency />` | 声明非 TS 模块依赖 | **已废弃**，用 `import "x";` 替代 |
| 任意 `reference` + `preserve="true"` | 保留指令到编译产物 | TS 5.5+ 需要下游工具读取指令时 |

## 五、实际应用场景

### 5.1 编写声明文件，依赖其他声明文件

假设你在为一个库手写类型声明，需要引用另一个声明文件里的类型：

```ts
// my-lib.d.ts
/// <reference path="./base-types.d.ts" />

// base-types.d.ts 里定义的 Person 类型现在可以用
declare function getUser(id: number): Person;
```

> 💡 **提示：** `.d.ts` 文件通常不写运行时的 `import` / `export`（否则会变成「模块」而非「全局声明」），所以这种文件间依赖就用三斜线指令来表达。

### 5.2 单个文件按需启用内置类型

某个声明文件用到了 `padStart`，但项目整体的 `lib` 没包含 ES2017，可以只在这个文件引入：

```ts
// polyfill-types.d.ts
/// <reference lib="es2017.string" />

interface String {
  // 这里可以放心引用/扩展 ES2017 的字符串能力
}
```

### 5.3 声明文件中依赖 `@types` 包

```ts
// 依赖 Node.js 的全局类型（如 process、Buffer）
/// <reference types="node" />

declare function getEnv(key: string): string | undefined;
```

### 5.4 AMD 模块命名（打包兼容）

当编译产物要交给 `r.js` 等 AMD 打包工具处理，且需要具名模块时：

```ts
/// <amd-module name="app/main" />

export function bootstrap() {
  console.log('app started');
}
```

## 六、常见问题与注意事项

### 6.1 现代 TS 项目还要用三斜线吗？

**大部分情况下不需要。** 文件间依赖用 `import` / `export` 即可，编译器会自动追踪依赖。三斜线指令主要保留给：

- 编写 `.d.ts` 声明文件；
- 按需引入内置 `lib`；
- AMD 模块命名等特殊场景。

`typescript-eslint` 甚至提供了 [`@typescript-eslint/triple-slash-reference`](https://typescript-eslint.io/rules/triple-slash-reference/) 规则，用来限制甚至禁止使用三斜线引用，鼓励改用 ES 模块 `import`。

### 6.2 三斜线指令 vs `import`

| 对比项 | 三斜线指令 | `import` |
| --- | --- | --- |
| 出现时代 | TS 早期（无 ES Module） | ES Module 标准方式 |
| 作用范围 | 声明文件依赖 / 类型包 / lib | 模块间导入 |
| 位置 | 只能在文件顶部 | 可以在文件顶部（模块作用域） |
| 现代推荐 | 仅在 `.d.ts` / 特殊场景 | 日常开发首选 |

> ⚠️ **注意：** 如果一个 `.d.ts` 里有 `import` 或 `export`，它就变成了一个**模块**，里面的声明不再自动成为全局类型。这一点决定了你有时不得不在三斜线指令和 `import` 之间做选择。

### 6.3 位置错误导致指令「不生效」

最常见的坑：指令没写在最顶部，结果编译器根本没认它。

```ts
// ❌ 指令前面出现了声明，失效
const config = { port: 3000 };
/// <reference path="./env.d.ts" />
```

**解决：** 始终把三斜线指令放在文件最前面（其他注释除外）。

### 6.4 相对路径基准容易搞错

`path` 是相对路径时，基准是**当前文件所在目录**，不是 `tsconfig.json` 所在目录，也不是项目根目录。

### 6.5 TypeScript 5.5 的产物变化

如果你发布库，下游工具依赖三斜线指令（比如需要从 `.d.ts` 里读到 `reference types`），记得：

- 升级到 TS 5.5+ 后指令默认会被删掉；
- 用 `preserve="true"` 显式保留，避免下游丢失类型依赖。

## 七、总结

- 三斜线指令是**写给编译器的顶部注释**，格式是 `/// <XML 标签 />`；
- 它只能出现在**文件顶部**，否则失效；
- 核心用途有三类：**声明文件/类型包依赖**（`path` / `types`）、**按需引入内置 `lib`**（`lib`）、**AMD 模块命名**（`amd-module`）；
- 现代 TypeScript 项目里，文件依赖**优先用 `import` / `export`**，三斜线指令主要服务于 `.d.ts` 声明文件等特殊场景；
- **TypeScript 5.5+** 默认会从产物删除三斜线指令，需要保留时加 `preserve="true"`。

> 💡 **一句话记忆：** 三斜线指令 = ES Module 诞生前，TypeScript 用来表达「依赖关系」的注释指令；今天它退居二线，主要在声明文件和内置 `lib` 的场景里发挥作用。
