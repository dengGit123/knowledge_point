# TypeScript 泛型（Generic）详解

> 📖 [泛型 —— TypeScript 官方手册](https://www.typescriptlang.org/docs/handbook/2/generics.html)
> 📖 [类型参数 —— TypeScript 官方手册（进阶）](https://www.typescriptlang.org/docs/handbook/2/types-from-extraction.html)

## 一、概述

**泛型（Generic）** 就是「**类型的参数化**」——把「类型」本身当成一个可变的参数，写一段不预先固定具体类型、又能保证类型安全的代码（函数、接口、类）。

> 💡 **通俗理解：** 普通函数的参数是「**值**的占位符」，泛型则是「**类型**的占位符」。就像函数接收一个值返回它本身，泛型函数接收一个**类型**，让同一段逻辑能套用在多种类型上，且每种用法都保留各自的精确类型。

```ts
// T 是「类型占位符」，调用时才确定 T 具体是什么类型
function identity<T>(value: T): T {
  return value;
}

identity<string>('hello'); // T = string，返回 string
identity<number>(42);      // T = number，返回 number
```

泛型要解决的核心矛盾是：**既要复用代码（不每种类型写一遍），又不能为了复用而牺牲类型安全（退化成 `any`）。**

## 二、为什么需要泛型：从 `any` 的痛点说起

假设要写一个「原样返回输入」的函数。

**方案 1：写死类型** —— 只能服务于一种类型，不复用：

```ts
function returnString(value: string): string {
  return value;
}
returnString(123); // ❌ 报错，number 不能传给 string
```

**方案 2：用 `any`** —— 能复用，但**完全失去类型保护**（等于关掉了 TS）：

```ts
function returnAny(value: any): any {
  return value;
}
const result = returnAny('hello');
result.toUpperCase(); // TS 不报错，但万一实际是 number，运行时就崩
result.toFixed();     // TS 不报错，但 string 没有 toFixed
```

**方案 3：泛型** —— 既复用，又保留精确类型：

```ts
function identity<T>(value: T): T {
  return value;
}

const r1 = identity('hello'); // T 推断为 string，r1: string
r1.toUpperCase(); // ✅ 合法
r1.toFixed();     // ❌ 报错，string 没有 toFixed —— 这正是我们想要的保护
```

> 💡 **提示：** 泛型的精髓在于「**输入和输出之间的类型关联**」。`identity` 的签名 `T → T` 告诉 TS：「返回值的类型，一定和入参相同」。这种关联是 `any` 给不了的。

## 三、泛型函数

### 3.1 基本语法

在函数名后用尖括号声明一个或多个**类型变量**（约定大写字母，最常用 `T`）：

```ts
function identity<T>(value: T): T {
  return value;
}
```

`<T>` 表示「这里有一个类型参数 `T`，函数内部先用它当占位符，调用时再确定」。

### 3.2 两种调用方式

```ts
// 方式一：显式指定类型参数
identity<string>('hello');

// 方式二：不写，让 TS 根据实参自动推断（更常用）
identity('hello'); // TS 看到 'hello'，推断 T = string
```

> 💡 **提示：** 绝大多数时候用**方式二（类型推断）**，代码更简洁。只有 TS 推断得不够精确、或你想强制约束时，才显式写类型参数。

### 3.3 箭头函数 / 函数表达式

箭头函数同样可以声明泛型：

```ts
const identity = <T>(value: T): T => value;
```

> ⚠️ **`.tsx` 文件中的坑（Vue JSX / React 常遇）：** 在 `.tsx` 里写 `<T>(...)` 会被解析器误当成 **JSX 标签**而报错。两种解法：

```tsx
// 解法一：加一个尾逗号，告诉解析器这是泛型不是 JSX
const identity = <T,>(value: T): T => value;

// 解法二：给 T 加约束（推荐，语义更明确）
const identity = <T extends unknown>(value: T): T => value;
```

## 四、多个类型参数

一个泛型可以有多个类型变量，用逗号分隔。常见命名：`T`（Type）、`U`/`V`（第二个、第三个类型）、`K`（Key）、`V`/`Val`（Value）、`E`（Element）、`R`（Return）。

```ts
// 输入两种类型，组成一个 pair
function pair<K, V>(key: K, value: V): { key: K; value: V } {
  return { key, value };
}

const p = pair('id', 1);
// 推断：{ key: string; value: number }
```

```ts
// 合并两个对象，保留各自的键
function merge<A, B>(a: A, b: B): A & B {
  return { ...a, ...b };
}
```

## 五、泛型约束（`extends`）

### 5.1 为什么需要约束

默认情况下，类型参数 `T` 可以是**任何东西**，所以函数体内**不能随意访问它的属性**：

```ts
function printLength<T>(value: T): void {
  console.log(value.length); // ❌ 报错：T 不一定有 length
}
```

`T` 可能是 `number`、`boolean`，它们没有 `.length`，所以 TS 拒绝。

### 5.2 用 `extends` 给类型参数加约束

`extends` 在这里**不是「类继承」**，而是「**约束**」——要求 `T` 必须满足某个形状（含有某些属性）：

```ts
interface Lengthwise {
  length: number;
}

// T extends Lengthwise：T 至少要有 length 属性
function printLength<T extends Lengthwise>(value: T): T {
  console.log(value.length); // ✅ 现在可以安全访问 length
  return value;
}

printLength('hello');    // ✅ string 有 length
printLength([1, 2, 3]);  // ✅ 数组有 length
printLength(123);        // ❌ 报错，number 没有 length
```

> 💡 **提示：** 同一个 `extends` 关键字在 TS 里有两层含义——**接口/类型上下文**里是「约束」（本节），**类上下文**里是「继承」。看到 `<T extends Xxx>` 就理解为「T 受 Xxx 约束」。

### 5.3 用 `keyof` 约束：类型安全的属性访问

最经典的泛型 + `keyof` 组合，保证「访问的键一定存在」（详见 [[keyof]]）：

```ts
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { id: 1, name: 'Alice', age: 25 };

getProperty(user, 'name'); // ✅ 返回 string
getProperty(user, 'age');  // ✅ 返回 number
getProperty(user, 'email'); // ❌ 报错，'email' 不是 user 的键
```

这里有两个泛型协作：`K extends keyof T` 表示「键 `K` 必须是对象 `T` 的某个属性名」，返回类型 `T[K]` 则**精确地**随键变化（访问 `name` 返回 `string`，访问 `age` 返回 `number`）。

### 5.4 约束另一个类型参数

类型参数之间也可以互相约束：

```ts
// 要求 newObj 的字段必须是 target 已有字段的子集
function copyField<T, K extends keyof T>(target: T, newObj: Pick<T, K>): void {
  Object.assign(target, newObj);
}
```

## 六、泛型接口与类型别名

### 6.1 泛型接口

接口名后加 `<T>`，整个接口就能参数化：

```ts
// 一个通用的「API 响应」结构
interface ApiResponse<T> {
  code: number;
  message: string;
  data: T; // 真正的数据，类型由 T 决定
}

// 使用时指定 T
type UserResp = ApiResponse<{ id: number; name: string }>;
// 等价于 { code: number; message: string; data: { id: number; name: string } }
```

### 6.2 泛型类型别名

类型别名（`type`）同样支持，且更灵活：

```ts
type Pair<A, B> = { first: A; second: B };

const p: Pair<string, number> = { first: 'a', second: 1 };
```

## 七、泛型类

类名后加 `<T>`，类的属性和方法就都能用这个类型参数：

```ts
class Stack<T> {
  private items: T[] = [];

  push(item: T): void {
    this.items.push(item);
  }

  pop(): T | undefined {
    return this.items.pop();
  }
}

const numberStack = new Stack<number>();
numberStack.push(1);     // ✅
numberStack.push('a');   // ❌ 报错，只能放 number
const top = numberStack.pop(); // top: number | undefined
```

> ⚠️ **注意：类的静态成员不能使用类型参数。** 因为类型参数属于「实例」层面，而静态成员属于「类」本身，在实例化之前就存在，那时 `T` 还没确定：

```ts
class Box<T> {
  static defaultValue: T; // ❌ 报错：静态成员不能引用类型参数 T
  content: T;             // ✅ 实例属性可以
}
```

## 八、默认类型参数

从 TS 2.3 起，类型参数可以给**默认值**。调用/使用时若不指定，就用默认值（规则和函数默认参数一致：有默认值的必须放后面）：

```ts
interface PaginatedData<T = any> {
  list: T[];
  total: number;
}

type UserPage = PaginatedData;      // 用默认值 any
type ArticlePage = PaginatedData<{ title: string }>; // 指定 T
```

```ts
function createArray<T = string>(length: number, fill: T): T[] {
  return Array(length).fill(fill);
}
```

> 💡 **提示：** 默认类型参数常用于库的设计——给使用者一个合理的「不传也行」默认行为，降低上手成本。

## 九、实际应用场景

### 9.1 封装统一的 API 请求

前端最经典的泛型场景：一个请求函数，返回值的类型随接口变化：

```ts
import axios from 'axios';

interface ApiResponse<T> {
  code: number;
  data: T;
}

// T 是「这个接口返回的数据类型」，由调用方指定
async function request<T>(url: string): Promise<T> {
  const res = await axios.get<ApiResponse<T>>(url);
  return res.data.data;
}

// 调用时声明各自的 data 类型，返回值就有精确提示
interface User {
  id: number;
  name: string;
}
const user = await request<User>('/api/user/1');
user.name; // ✅ 有类型提示，string
```

### 9.2 通用的状态管理（`useState` 风格）

```ts
function createState<T>(initial: T) {
  let state = initial;
  return {
    get: () => state,
    set: (next: T) => {
      state = next;
    },
  };
}

const count = createState(0);     // 推断 T = number
count.set(10);                    // ✅
count.set('ten');                 // ❌ 报错，必须是 number
```

### 9.3 Vue 组件 Props 与 `ref` 的泛型

Vue 的 `ref`、`reactive`、`defineProps` 本身就是泛型函数，类型参数让它们适配各种数据：

```ts
import { ref } from 'vue';

const count = ref<number>(0);            // Ref<number>
count.value = 5;                         // ✅
count.value = '5';                       // ❌ 报错

interface User {
  id: number;
  name: string;
}
const currentUser = ref<User | null>(null);
currentUser.value = { id: 1, name: 'Alice' }; // ✅
```

### 9.4 工具函数：数组去重

```ts
function unique<T>(arr: T[]): T[] {
  return [...new Set(arr)];
}

unique([1, 2, 2, 3]);          // number[]
unique(['a', 'b', 'b']);       // string[]
```

## 十、常见坑与注意事项

### 10.1 泛型是「编译时」概念，运行时不存在

泛型只在**类型检查**阶段有意义，编译成 JS 后**类型信息全部被擦除**。所以你**无法在运行时判断 `T`**：

```ts
function log<T>(value: T): void {
  // ❌ 运行时 T 已经不存在，没法这样判断
  if (value instanceof T) {
  }
}
```

> 💡 **提示：** 这也是为什么泛型函数里通常需要额外传入一个「类型守卫」或「构造函数」来实现运行时区分，而不是直接靠 `T`。

### 10.2 类型推断失败时，会退化成 `any`/宽松类型

TS 推断不出来时，类型参数可能落到 `any`（受 `noImplicitAny` 控制）或宽泛类型。这时要**手动显式指定类型参数**：

```ts
function merge<T, U>(a: T, b: U) {
  return { ...a, ...b };
}
// 传入两个对象能推断，但复杂场景下建议显式：
merge<{ id: number }, { name: string }>({ id: 1 }, { name: 'a' });
```

### 10.3 命名约定

单字母大写是惯例，但**有意义的多字母名字更易读**：

```ts
// 可读性更好（推荐在复杂泛型里用）
function getProperty<ObjectType, KeyType extends keyof ObjectType>(
  obj: ObjectType,
  key: KeyType,
): ObjectType[KeyType] {
  return obj[key];
}
```

常见约定字母：`T`（类型）、`K`（键）、`V`（值）、`E`（元素）、`R`（返回值）、`U`/`V`（辅助类型）。

### 10.4 约束 `extends` 不会改变推断的精确性

```ts
function printLength<T extends { length: number }>(value: T): T {
  return value;
}

const x = printLength('hello'); // x 仍是 string，不是 { length: number }
```

`extends` 只是「准入门槛」，传入 `string` 后 `T` 仍是精确的 `string`，不会塌缩成约束类型。

### 10.5（进阶）`const` 类型参数：TS 5.0+

想让推断结果更「窄」（比如推断成字面量类型而非宽类型），可以用 `const` 类型参数：

```ts
function tuple<const T>(value: T): T {
  return value;
}

const a = tuple([1, 2, 3]);
// 不加 const：T 推断为 number[]
// 加 const：T 推断为 readonly [1, 2, 3]（字面量元组）
```

> ⚠️ 这是 TS 5.0+ 特性，老版本不支持。普通业务多数用不到，了解即可。

## 十一、与相关概念的关系

| 概念 | 关系 | 说明 |
| --- | --- | --- |
| `any` | 泛型的反面 | `any` 放弃类型，泛型保留类型。详见 [[any和unkonw]] |
| `extends` | 泛型约束 | `<T extends X>` 给类型参数加约束，不是类继承 |
| `keyof` | 常与泛型配合 | `<K extends keyof T>` 做类型安全的键访问，详见 [[keyof]] |
| 工具类型 | 泛型的应用 | `Partial<T>`、`Pick<T,K>` 等本质是泛型类型别名，详见 [[内置的工具类型]] |
| 条件类型 | 泛型 + 类型判断 | `T extends U ? X : Y`，详见 [[条件类型]] |

## 十二、总结

- **泛型 = 类型的参数化**，把类型当占位符，让一段逻辑复用于多种类型且保留各自精确类型；
- **解决 `any` 痛点**：在「复用」和「类型安全」之间取得平衡，关键在于建立「输入→输出」的类型关联；
- **基本写法**：函数/接口/类名后加 `` `<T>` ``，约定大写字母（`T`、`K`、`V`、`U`…）；
- **调用方式**：多数靠**类型推断**，推断不准时再显式 `` `fn<Type>(...)` ``；
- **约束**：用 `` `<T extends Xxx>` `` 限制类型范围、安全访问属性；配合 `keyof` 实现类型安全的属性访问；
- **典型场景**：API 请求封装、`ref`/状态管理、数组工具、通用数据结构（`Stack<T>`）等；
- **牢记**：泛型是**编译时**概念，运行时被擦除，无法在运行时判断类型参数。

> 💡 **一句话记忆：** 普通函数参数化的是「值」，泛型参数化的是「类型」——用 `` `<T>` `` 占位、用 `extends` 约束、用类型推断省去手写，从而写出既复用又类型安全的代码。
