# TypeScript 声明文件与模块声明（`.d.ts` / `declare`）

> 📖 [Declaration Files —— TypeScript 官方文档](https://www.typescriptlang.org/docs/handbook/declaration-files/introduction.html)
> 📖 [.d.ts Templates —— TypeScript 官方文档](https://www.typescriptlang.org/docs/handbook/declaration-files/templates.html)

## 一、概述

当你在 TypeScript 项目里 `import` 一个第三方库、一个 `.vue` 文件、一张 `.png` 图片时，TypeScript 要能「看懂」这些东西的类型，否则就会报错。**类型声明**就是用来告诉 TypeScript「这些东西长什么样」的。

TypeScript 的类型声明体系主要靠两样东西：

- **声明文件（`.d.ts`）**：专门存放**类型信息**的文件，不包含可执行代码，后缀是 `.d.ts`（d 代表 declaration）；
- **`declare` 关键字**：在「**环境上下文**」（ambient context）里声明「某个东西已经存在，这是它的类型」。

```ts
// 告诉 TS：运行时会存在一个全局变量 jQuery，类型是这个函数
declare function jQuery(selector: string): any;
```

> 💡 **通俗理解：** `.d.ts` 文件就像一份「**说明书**」——它不实现功能，只描述「这个东西有哪些属性、方法、是什么类型」，让 TypeScript 能据此做类型检查。

## 二、核心概念：环境声明（Ambient Declaration）

`declare` 关键字的作用是做**环境声明**（ambient declaration）：声明一个变量/函数/类/模块「**在运行时已经存在**」，TypeScript 不需要、也不会为它生成实现代码。

```ts
// declare 声明的 jQuery，编译后不会产生任何 JS 代码
// 它只是告诉 TS「运行时会有个叫 jQuery 的东西」
declare const jQuery: (selector: string) => HTMLElement;

jQuery('#app'); // TS 不会报错了
```

**`declare` 和普通声明的区别：**

| 对比项 | 普通声明 `const a = 1` | 环境声明 `declare const a` |
| --- | --- | --- |
| 是否产生运行时代码 | 会（`const a = 1`） | 不会（纯类型） |
| 由谁提供实际值 | 声明语句本身 | 外部（宿主环境、第三方脚本等） |
| 典型场景 | 你自己写的逻辑 | 引用全局变量、为 JS 库补类型 |

## 三、环境声明的几种形式

### 3.1 `declare var` / `let` / `const`

声明一个已存在的全局变量。`var`/`let`/`const` 的区别和普通变量一致（`const` 表示只读）。

```ts
// 全局配置对象，由 <script> 标签或运行时注入
declare const APP_CONFIG: {
  version: string;
  env: 'dev' | 'prod';
};

console.log(APP_CONFIG.version); // TS 能识别类型
```

### 3.2 `declare function`

声明一个全局函数。

```ts
// 声明全局函数，支持重载
declare function formatPrice(value: number, currency?: string): string;

const price = formatPrice(99.5, 'CNY'); // "￥99.50"
```

### 3.3 `declare class`

声明一个全局可用的类。

```ts
declare class Animal {
  constructor(name: string);
  name: string;
  speak(): void;
}

const dog = new Animal('Rex');
dog.speak();
```

### 3.4 `declare enum`

声明一个全局枚举（**环境枚举**）。注意：环境枚举的成员**必须显式写值**，不能依赖自动递增推断。

```ts
declare enum Color {
  Red = 0,
  Green = 1,
  Blue = 2,
}

const c: Color = Color.Red;
```

### 3.5 `declare namespace`

声明一个包含子属性的全局对象。常用于「**有结构配置的全局变量**」，比如传统的全局配置对象。

```ts
// 假设运行时挂了一个全局的 myApp 配置对象
declare namespace myApp {
  const version: string;
  const config: { debug: boolean };
  function init(): void;
}

myApp.init();
console.log(myApp.config.debug);
```

## 四、声明文件 `.d.ts` 的两种形态

一个 `.d.ts` 文件是**全局的**还是**模块的**，取决于一个关键规则：

> ⭐ **核心规则：** 顶层是否有 `import` 或 `export`。
> - **没有** `import`/`export` → 它是**全局脚本**，里面的声明对整个项目可见；
> - **有** `import`/`export`（哪怕只导出一个空的东西）→ 它是**一个模块**，里面的声明需要通过 `import` 才能使用。

### 4.1 全局声明文件

没有任何顶层 `import`/`export` 时，声明的类型自动成为全局可见。

```ts
// globals.d.ts —— 全局声明文件
declare const API_BASE_URL: string;
declare function gtag(...args: any[]): void;
```

这两个声明在整个项目的任何文件里都能直接用，无需 `import`。

> 💡 **提示：** 全局声明很方便，但也容易造成命名冲突和「类型从哪来的」困惑。**能用模块就不要用全局**。

### 4.2 模块声明文件

只要顶层有 `import`/`export`，文件就变成了模块，里面的声明只对外导出的部分可见。

```ts
// types/math.d.ts —— 模块声明文件
export interface Point {
  x: number;
  y: number;
}

export declare function distance(a: Point, b: Point): number;
```

使用时必须 `import`：

```ts
import { Point, distance } from './types/math';
```

> ⚠️ **注意：** 这是新手最常踩的坑——在 `.d.ts` 里不小心写了个 `import`，结果原本想做成全局的类型突然「找不到」了。原因就是文件因此变成了模块。

## 五、`declare module`：声明 / 增强模块

`declare module` 用来为一个**模块名**写类型声明。它有三种典型用法。

### 5.1 为没有类型的库声明类型

当某个 npm 包没有自带类型、也没有 `@types/xxx` 时，可以用 `declare module '包名'` 手动补一份（哪怕是 `any`，也能消除报错）。

```ts
// shim.d.ts
// 给一个没有类型声明的小工具库补类型
declare module 'query-string' {
  export function parse(str: string): Record<string, string>;
  export function stringify(obj: Record<string, any>): string;
}
```

> 💡 **提示：** 如果只是想临时消除「找不到类型」的报错，也可以写最简单的 `declare module 'xxx';`，此时该模块会被当作 `any`。但**不推荐**长期这样用，等于关掉了类型检查。

### 5.2 为非 JS 资源声明类型

Vite / Webpack 项目里经常 `import` 图片、CSS 等非 JS 资源，它们没有类型，需要用通配符 `declare module` 补声明（详见第七节）。

```ts
// 声明：所有 .png 导入都返回一个 URL 字符串
declare module '*.png' {
  const src: string;
  export default src;
}
```

### 5.3 模块增强（Module Augmentation）

`declare module` 还能**扩展已有模块导出的类型**。比如给 Vue 扩展自定义的全局属性、给 `express` 的 `Request` 加自定义字段。

```ts
// 扩展 Vue 3 的组件实例属性类型
import type { ComponentCustomProperties } from 'vue';

declare module 'vue' {
  interface ComponentCustomProperties {
    $translate: (key: string) => string;
  }
}
```

> ⚠️ **注意：** 模块增强必须**先 `import` 目标模块**，再用 `declare module '同名'` 扩展，且文件本身要是模块（有 `import`/`export`）。这样才会被当作「增强」而不是「覆盖」。

## 六、`declare global`：在模块里扩展全局类型

有时你想在**模块文件**（有 `import`/`export` 的文件）里修改**全局类型**，比如给 `Window`、`Array` 加字段。这时就要用 `declare global`。

```ts
// global-properties.d.ts
export {}; // 关键：让本文件成为模块，否则用不了 declare global

declare global {
  interface Window {
    // 给 window 加一个自定义字段
    __APP_VERSION__: string;
  }

  // 给所有数组扩展一个自定义方法（仅类型层面）
  interface Array<T> {
    last(): T | undefined;
  }
}

window.__APP_VERSION__; // 不再报错
[1, 2, 3].last();       // 类型上可用
```

> ⚠️ **注意：** `declare global` 只能在**模块文件**里用。如果你的文件没有 `import`/`export`，它本身就是全局脚本，直接写声明即可，反而**不能**套 `declare global`。

## 七、实战场景

### 7.1 为 `.vue` 单文件组件写声明（Vite/Vue 项目必备）

TypeScript 本身不认识 `.vue` 文件，直接 `import App from './App.vue'` 会报「找不到模块」的错。需要补一份通配符声明：

```ts
// vite-env.d.ts 或 env.d.ts
/// <reference types="vite/client" />

declare module '*.vue' {
  import type { DefineComponent } from 'vue';
  // 定义组件的默认导出类型为一个 Vue 组件
  const component: DefineComponent<{}, {}, any>;
  export default component;
}
```

> 💡 **提示：** Vite 项目用 `npm create vite` 创建时，会自动生成一个 `env.d.ts`，里面通过 `/// <reference types="vite/client" />` 引入了 Vite 自带的类型声明（包括 `.vue`、静态资源等）。一般你不需要手写 7.1 / 7.2，知道原理即可。

### 7.2 为静态资源声明类型

图片、CSS Modules、字体等资源也需要声明：

```ts
// 静态资源返回 URL 字符串
declare module '*.png' {
  const src: string;
  export default src;
}
declare module '*.svg' {
  const src: string;
  export default src;
}

// CSS Modules 返回一个类名映射对象
declare module '*.module.css' {
  const classes: { readonly [key: string]: string };
  export default classes;
}
```

使用：

```ts
import logo from './assets/logo.png';        // string
import styles from './App.module.css';       // 类名对象
```

### 7.3 给 Vue 3 扩展全局属性类型

当你通过 `app.config.globalProperties` 注入自定义方法时（如 `this.$translate`），需要在类型层面声明它，否则组件里 `this.$translate` 会报错。

```ts
// src/types/vue.d.ts
import type { ComponentCustomProperties } from 'vue';

declare module 'vue' {
  interface ComponentCustomProperties {
    $translate: (key: string) => string;
    $loading: boolean;
  }
}
```

配合 `main.ts`：

```ts
import { createApp } from 'vue';
import App from './App.vue';

const app = createApp(App);
// 运行时注入
app.config.globalProperties.$translate = (key: string) => `翻译-${key}`;
app.mount('#app');
```

### 7.4 为没有类型的 npm 包补声明

方案一：装 DefinitelyTyped 的类型包（**首选**）。

```bash
npm install -D @types/lodash
```

方案二：包既没自带类型、也没有 `@types`，则手动写。

```ts
// types/shims.d.ts
declare module 'some-untyped-lib' {
  export function greet(name: string): string;
  export const version: string;
}
```

### 7.5 让某个库可被全局 `import` 时带上类型（UMD 全局）

如果一个库既支持 `import` 又通过 `<script>` 挂到了全局，可以用 `export as namespace` 声明它的全局名：

```ts
// my-lib.d.ts
export as namespace MyLib;

export function doSomething(): void;
```

这样无论 `import { doSomething } from 'my-lib'` 还是直接用全局的 `MyLib.doSomething()`，都有类型。

## 八、常见问题与注意事项

### 8.1 「声明文件是全局还是模块」最容易搞错

记住一句话：**顶层有 `import`/`export` 就是模块，否则就是全局。** 这是绝大多数「我写的声明不生效」「类型找不到」问题的根源。

```ts
// ❌ 想做成全局，却无意中写了 export，变成模块导致全局失效
interface GlobalType { foo: string }
export {}; // 这一行让文件变成了模块
```

### 8.2 `declare` 不会生成实现，别指望它创建变量

```ts
// ❌ 以为这会创建变量
declare const foo: string;
foo; // 运行时 foo 是 undefined，甚至直接报错
```

`declare` 只是类型声明，**真正的值必须由别处（HTML 脚本、运行时注入、第三方库）提供**。

### 8.3 找不到类型声明怎么办

按以下顺序排查：

1. 包是否自带类型（`package.json` 里有 `types` / `typings` 字段）；
2. 是否有对应的 `@types/包名`（用 [TypeSearch](https://www.typescriptlang.org/dt/search) 查找）；
3. 都没有 → 手写 `declare module '包名'`。

### 8.4 `declare` 与 `interface` / `type` 的区别

| 写法 | 作用 | 是否产生值 |
| --- | --- | --- |
| `type` / `interface` | 定义**纯类型** | 否（部分 interface 可声明合并） |
| `declare` | 声明**已存在的外部实体**的类型 | 否（只声明，不创建） |

简单说：`interface` 是「定义一个类型的形状」，`declare` 是「告诉你某个东西已经存在并且长这样」。

### 8.5 声明合并（Declaration Merging）

同名的 `interface`、`namespace`、`declare module` 会自动**合并**。模块增强（5.3）正是利用了 `interface` 合并的机制——你写的同名 `interface` 会并入原有的，而不是覆盖。

```ts
// 第一次定义
interface User {
  name: string;
}
// 另一处「增强」
interface User {
  age: number;
}
// 最终 User 同时拥有 name 和 age
```

> 💡 **提示：** 这也是为什么模块增强里用 `interface ComponentCustomProperties { ... }` 能给已有接口「加字段」而不报重复错误。

### 8.6 声明文件放在哪里会被自动识别

- **项目内**：`.d.ts` 文件只要在 `tsconfig.json` 的 `include` 范围内（或默认的 `src` 下），就会被自动加载，无需手动 `import` 全局声明；
- **`@types` 包**：放在 `node_modules/@types/` 下，由 `typeRoots` 自动包含；
- **自定义路径**：可在 `tsconfig.json` 的 `compilerOptions.types` / `typeRoots` 里配置。

## 九、速查对比表

| 关键字 / 概念 | 作用 | 典型场景 |
| --- | --- | --- |
| `.d.ts` 文件 | 存放类型声明，无运行时代码 | 为 JS 库 / 资源补类型 |
| `declare var/const` | 声明全局变量 | 全局配置、`window` 注入的值 |
| `declare function` | 声明全局函数 | 全局工具函数 |
| `declare class` | 声明全局类 | 全局可用的类 |
| `declare namespace` | 声明带结构的全局对象 | 传统命名空间式全局配置 |
| `declare module '名'` | 声明一个模块的类型 | 为无类型库 / 非 JS 资源补类型 |
| `declare module '名'`（增强） | 扩展已有模块类型 | 给 Vue / express 加字段 |
| `declare global` | 在模块中扩展全局类型 | 扩展 `Window`、`Array` |
| `export as namespace` | 声明 UMD 全局名 | 同时支持 import 和全局使用 |
| `/// <reference types="..." />` | 引入类型包依赖 | 引入 `@types` 包（见 [[三斜线作用]]） |

## 十、总结

- **声明文件 `.d.ts`** 是 TypeScript 的「类型说明书」，只描述类型、不含可执行代码；
- **`declare`** 做环境声明，告诉 TS「这个东西运行时已存在」；
- 判断 `.d.ts` 是全局还是模块，只看**顶层有没有 `import`/`export`**；
- **`declare module`** 负责声明/增强模块（含 `.vue`、图片等非 JS 资源）；
- **`declare global`** 负责在模块文件里扩展全局类型（如 `Window`）；
- Vite + Vue 3 项目里，`.vue` 和静态资源的声明通常由 `vite/client` 自带，了解原理即可；
- 找不到类型时，优先用 `@types` 包，其次手写 `declare module`。

> 💡 **一句话记忆：** `.d.ts` 是「说明书」，`declare` 是「我保证运行时会有这个东西」，`declare module` / `declare global` 则是这两把钥匙分别用来打开「模块」和「全局」两扇门的用法。
