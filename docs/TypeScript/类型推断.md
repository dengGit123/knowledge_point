# TypeScript 类型推断（Type Inference）

> 📖 [Type Inference —— TypeScript 官方文档](https://www.typescriptlang.org/docs/handbook/type-inference.html)
> 📖 [satisfies —— TypeScript 官方文档](https://www.typescriptlang.org/docs/handbook/release-notes/typescript-4-9.html#the-satisfies-operator)

## 一、概述

在 TypeScript 中，**类型推断**是指：当你在代码里**没有显式写类型注解**时，TypeScript 会根据上下文**自动推断**出变量、参数、返回值等的类型。

```ts
// 没有写 :number，但 TS 推断出 count 是 number
let count = 10;          // 👉 number
const name = 'Tom';      // 👉 'Tom'（字面量类型）
function add(a: number, b: number) {
  return a + b;          // 👉 返回类型推断为 number
}
```

> 💡 **通俗理解：** 类型推断就像「**自动识别**」——你不用每次都贴标签（类型注解），TypeScript 看一眼赋的值就能猜出类型。它让代码既**简洁**又**类型安全**。

> ⚠️ **注意：** 类型推断不等于「随便」。`strict` 模式下，当 TS **推断不出来**时会报错（而不是默认 `any`），这正是 `noImplicitAny` 的作用。

## 二、推断发生在哪些地方

### 2.1 变量与成员初始化

```ts
let x = 3;          // number
let text = 'hello'; // string
let isActive = true; // boolean

class Point {
  x = 0;             // 推断为 number（类属性也会推断）
  y = 0;
}
```

### 2.2 函数返回值

```ts
function greet(name: string) {
  return `Hello, ${name}`; // 推断返回类型为 string
}
// 等价于 function greet(name: string): string
```

### 2.3 解构与对象字面量

```ts
const { a, b } = { a: 1, b: 'x' };
// a: number, b: string

const user = { name: 'Tom', age: 18 };
// user: { name: string; age: number }
```

## 三、拓宽（Widening）

这是类型推断里**最容易被忽视**的点：字面量值在赋给 `let` 变量时，会被**拓宽**成更宽的基础类型。

```ts
const c = 'hello';  // 👉 类型是字面量 'hello'
let v = 'hello';    // 👉 类型被拓宽成 string（因为 let 可重新赋值）

const n = 3;        // 👉 3
let m = 3;          // 👉 number
```

**原因**：`let` 变量可以重新赋值，所以 TS 假设它将来可能变成同类的其它值，于是拓宽成基础类型；`const` 不会重新赋值，所以保留精确的字面量类型。

> 💡 **提示：** 对象字面量的属性默认也会拓宽：

```ts
const obj = { x: 1 };
// obj.x 的类型是 number（不是字面量 1），因为对象属性可变
```

## 四、最佳通用类型（Best Common Type）

当需要从多个值推断一个类型时（如数组），TypeScript 会找这些值的**最佳通用类型**。如果没有共同父类型，就用**联合类型**。

```ts
let arr = [0, 1, null]; // 👉 (number | null)[]
// 因为 number 和 null 没有更具体的共同类型

let pets = [new Dog(), new Cat()]; // 👉 (Dog | Cat)[]
// 如果有共同父类 Animal，可用显式注解：let pets: Animal[] = [...]
```

> 💡 **提示：** 最佳通用类型只看**直接**的成员类型，不会自动上溯到共同父类。需要父类时要自己写类型注解。

## 五、上下文类型（Contextual Typing）

前面几种都是「**从右往左**」推断（根据赋的值推断变量类型）。上下文类型则相反——「**从左往右**」推断：根据变量的**使用位置**反向推断出表达式类型。

最典型的例子是函数参数：

```ts
// onmousedown 的类型决定了 e 的类型，无需手动标注
window.onmousedown = function (e) {
  console.log(e.button); // 👉 e 自动推断为 MouseEvent
};

// 数组 map 的回调参数 x 自动推断为 number
[1, 2, 3].map((x) => x * 2);
```

**对象字面量也会被上下文类型约束**：

```ts
interface User { name: string; age: number }

// 赋值给已类型化的变量时，TS 按目标类型检查并提示
const u: User = { name: 'Tom', age: 18, extra: true };
// ❌ 报错：对象字面量只能指定已知属性，'extra' 不在 User 中
```

> 💡 **提示：** 上下文类型能大幅减少样板代码——你不必给每个回调参数都写类型，TS 会从「使用处」推断出来。

## 六、如何控制类型推断

有时 TS 推断的类型**不够精确**或**不够宽**，你需要主动控制。

### 6.1 `as const`（const 断言）

让推断结果保留**最窄的字面量类型**，常用于配置、常量、联合。

```ts
// 不加 as const：status 是 string
let status = 'open';

// 加 as const：保留字面量
const status = 'open' as const;        // 👉 'open'
const arr = [1, 2, 3] as const;        // 👉 readonly [1, 2, 3]
const config = { endpoint: '/api' } as const;
// 👉 { readonly endpoint: '/api' }
```

> 💡 **提示：** `as const` 在定义「常量联合」时极其有用：

```ts
const ROLES = ['admin', 'user', 'guest'] as const;
type Role = (typeof ROLES)[number]; // 👉 'admin' | 'user' | 'guest'
```

### 6.2 `satisfies` 操作符（TS 4.9+）

`satisfies` 用于「**校验表达式符合某类型，但保留推断出的更精确类型**」，弥补了「直接标注会拓宽类型」的不足。

```ts
// 目标：保证每个值都是 string 或 number[]，同时保留具体类型
const palette = {
  red: [255, 0, 0],
  green: '#00ff00',
  blue: [0, 0, 255],
} satisfies Record<string, string | number[]>;

palette.red.push(1);     // ✅ red 被推断为 number[]，可以 push
palette.green.toUpperCase(); // ✅ green 被推断为 string
```

对比直接标注会拓宽类型：

```ts
// ❌ 直接标注会把 red 拓宽为 string | number[]，丢失 number[] 的精确性
const palette2: Record<string, string | number[]> = { /* ... */ };
palette2.red.push(1); // ❌ 报错：string 上没有 push
```

> 💡 **提示：** `satisfies` 的口诀——「**要校验、不要拓宽**」。它在不牺牲类型精度的前提下，保证值满足某个约束。

### 6.3 显式类型注解

当推断结果不符合预期（或推断不出来），直接写注解：

```ts
let arr: Animal[] = [new Dog(), new Cat()]; // 显式指定父类型
function handler(e: Event) { /* ... */ }     // 参数推断不出时必须标注
```

## 七、函数中的推断

### 7.1 返回类型推断

```ts
function pick(arr: number[], i: number) {
  return arr[i]; // 推断返回 number
}
```

> 💡 **提示：** 大多数情况让 TS 推断返回类型即可。只有当推断结果对外暴露不够清晰、或想强制约束返回类型时，才显式标注返回类型。

### 7.2 泛型函数的类型推断

调用泛型函数时，类型参数会**根据实参自动推断**：

```ts
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

first([1, 2, 3]);    // 👉 T 推断为 number，返回 number | undefined
first(['a', 'b']);   // 👉 T 推断为 string
```

### 7.3 `noImplicitAny`

当参数推断不出来（且没有注解）时，`noImplicitAny` 会**报错**，而不是静默当作 `any`：

```ts
function fn(s) {
  // ❌ 报错：参数 s 隐式为 any
  console.log(s.substring(0));
}
```

这是 `strict` 模式的一部分，强制你给无法推断的参数显式标注类型。

## 八、注意事项与陷阱

| 陷阱 | 说明 | 解决 |
| --- | --- | --- |
| **空数组推断为 `any[]`** | `let arr = []` 会被推断为 `any[]`（之后 push 什么就变什么） | 显式标注 `let arr: number[] = []` |
| **字面量被拓宽** | `let x = 'a'` 得到 `string` 而非 `'a'` | 用 `as const` 或 `const` |
| **对象属性拓宽** | `const o = { n: 1 }` 里 `o.n` 是 `number` | 用 `as const` 或显式类型 |
| **联合类型太宽** | `[1, null]` 得到 `(number \| null)[]` | 视需要过滤或标注 |
| **回调参数没标注就报错** | 上下文类型无法应用时，参数推断不出 | 给参数显式标注 |
| **跨文件/模块边界推断变弱** | 复杂场景下推断可能不如预期精确 | 关键公共 API 显式标注 |

> ⚠️ **注意（最佳实践）：** 对**对外暴露的公共 API**（库的导出函数、组件 props 类型），建议**显式标注返回类型和参数类型**，不要过度依赖推断——这样能保证类型契约稳定，不因内部实现变动而改变对外类型。

## 九、总结

- 类型推断让代码**少写注解**的同时保持**类型安全**；
- 推断主要发生在：变量初始化、函数返回值、解构/对象字面量；
- **拓宽**：`let` 和对象属性会把字面量拓宽成基础类型，`const` 保留字面量；
- **最佳通用类型**：多值推断时取共同类型，没有就用联合；
- **上下文类型**：从「使用处」反向推断（回调参数、对象字面量）；
- 控制推断：`as const`（收窄）、`satisfies`（校验不拓宽）、显式注解；
- **公共 API 显式标注**，内部代码善用推断。

> 💡 **一句话记忆：** 类型推断是 TS 的「**自动识别**」——平时让它自己猜（省事），猜得不准时用 `as const` 收窄、`satisfies` 校验、注解修正；**对外接口别偷懒，显式写清楚**。
