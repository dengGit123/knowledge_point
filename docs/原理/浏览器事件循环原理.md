# 浏览器事件循环原理

## 概述

JavaScript 是单线程语言，浏览器通过事件循环（Event Loop）机制实现非阻塞异步操作。事件循环是浏览器处理异步代码的核心机制，它不断从任务队列中取出回调函数执行，同时协调渲染、用户交互、网络响应等多种任务。

理解事件循环对于编写高效的 JavaScript 代码、避免阻塞、优化性能至关重要。事件循环涉及调用栈（Call Stack）、任务队列（Task Queue）、微任务队列（Microtask Queue）多个组件的协作。

## 一、调用栈

### 1.1 调用栈概念

调用栈（Call Stack）是一种后进先出（LIFO）的数据结构，用于存储代码执行时的函数调用信息。每次调用一个函数时，浏览器会将该函数的执行上下文压入栈顶；函数执行完成后，上下文从栈顶弹出。

执行上下文包含：

- 函数的局部变量
- 函数的作用域链
- this 绑定
- 当前执行位置

### 1.2 执行过程

```javascript
function greet(name) {
  return `Hello, ${name}`;
}

function sayGoodbye(name) {
  const message = greet(name);
  console.log(`${message}, goodbye!`);
}

sayGoodbye('Alice');
```

执行过程：

1. 调用 `sayGoodbye('Alice')`，将函数压入栈
2. 在 `sayGoodbye` 内部调用 `greet(name)`，将 `greet` 压入栈
3. `greet` 执行完成并返回，从栈弹出
4. `sayGoodbye` 执行 `console.log`，从栈弹出
5. 所有函数执行完毕，栈为空

调用栈的深度是有限的，递归调用过深会导致"栈溢出"（Stack Overflow）。

## 二、任务队列与事件循环

### 2.1 任务队列

任务队列（Task Queue）也称为宏任务队列（Macrotask Queue），用于存放需要异步执行的回调函数。当异步操作完成时（如 setTimeout 计时结束、网络请求响应），对应的回调被推入任务队列。

常见的宏任务来源：

- setTimeout / setInterval 的回调
- I/O 操作完成后的回调
- UI 交互事件（click、keydown）的回调
- requestAnimationFrame 的回调
- 整个脚本执行

### 2.2 事件循环机制

事件循环（Event Loop）是一个持续运行的机制，其核心逻辑：

```
while (true) {
  // 1. 执行完当前调用栈中的所有同步代码
  while (callStack is not empty) {
    executeTask(callStack.pop());
  }

  // 2. 执行所有微任务
  while (microtaskQueue is not empty) {
    executeTask(microtaskQueue.shift());
  }

  // 3. 执行一个宏任务
  if (taskQueue is not empty) {
    executeTask(taskQueue.shift());
  }

  // 4. 如果需要渲染，执行渲染步骤
  if (needsRender) {
    render();
  }
}
```

事件循环的每次迭代中：

1. 首先执行完调用栈中的所有同步代码
2. 然后清空微任务队列（全部执行）
3. 然后从宏任务队列取出一个任务执行
4. 重复以上过程

### 2.3 执行顺序示例

```javascript
console.log('1 - script start');

setTimeout(() => {
  console.log('2 - setTimeout');
}, 0);

Promise.resolve().then(() => {
  console.log('3 - Promise.then');
});

console.log('4 - script end');
```

输出顺序：

```
1 - script start
4 - script end
3 - Promise.then
2 - setTimeout
```

执行过程分析：

1. `console.log('1')` 同步执行
2. `setTimeout` 回调注册到宏任务队列
3. `Promise.resolve().then` 回调注册到微任务队列
4. `console.log('4')` 同步执行
5. 同步代码执行完毕，开始执行微任务：`console.log('3')`
6. 微任务队列清空，执行一个宏任务：`console.log('2')`

## 三、微任务队列

### 3.1 微任务概念

微任务（Microtask）是比宏任务优先级更高的异步任务。微任务队列在每次事件循环迭代中会被完全清空，而宏任务队列每次只取出一个执行。

常见的微任务来源：

- Promise 的 then / catch / finally 回调
- MutationObserver 的回调
- queueMicrotask() 添加的任务
- async/await（实质是 Promise + 微任务）

### 3.2 Promise 与微任务

Promise 的回调被注册为微任务：

```javascript
Promise.resolve()
  .then(() => console.log('then1'))
  .then(() => console.log('then2'))
  .then(() => console.log('then3'));
```

Promise 链的每个 `.then()` 都会产生一个微任务，它们按顺序执行。

### 3.3 async/await 与微任务

async 函数内部，await 之后的代码会被包装成微任务执行：

```javascript
async function fetchData() {
  console.log('1 - before await');
  await Promise.resolve();
  console.log('2 - after await');
}

console.log('3 - before call');
fetchData();
console.log('4 - after call');
```

输出顺序：

```
3 - before call
1 - before await
4 - after call
2 - after await
```

执行过程：

1. `console.log('3')` 同步执行
2. 调用 `fetchData()`，`console.log('1')` 同步执行
3. 遇到 `await Promise.resolve()`，暂停函数执行，后续代码注册为微任务
4. `console.log('4')` 同步执行
5. 同步代码执行完毕，执行微任务：`console.log('2')`

## 四、宏任务与微任务对比

### 4.1 任务优先级

| 类型 | 来源 | 特点 |
|------|------|------|
| 同步代码 | - | 立即执行，阻塞后续任务 |
| 微任务 | Promise.then, queueMicrotask | 当前宏任务结束后、下个渲染前全部执行 |
| 宏任务 | setTimeout, setInterval, I/O, 事件 | 每个事件循环迭代执行一个 |

### 4.2 执行顺序规则

事件循环的执行顺序规则：

1. 执行所有同步代码（调用栈清空）
2. 执行所有微任务（队列清空）
3. 执行一个宏任务
4. 重复步骤 2-3（如果有新任务加入）

这意味着微任务总是在下一个宏任务之前执行，而且微任务可以继续添加微任务，形成微任务链。

### 4.3 综合示例

```javascript
console.log('1 - script start');

setTimeout(() => {
  console.log('2 - outer setTimeout');
  Promise.resolve().then(() => {
    console.log('3 - inner Promise.then');
  });
}, 0);

Promise.resolve().then(() => {
  console.log('4 - first Promise.then');
  setTimeout(() => {
    console.log('5 - inner setTimeout');
  }, 0);
});

console.log('6 - script end');
```

输出顺序：

```
1 - script start
6 - script end
4 - first Promise.then
2 - outer setTimeout
3 - inner Promise.then
5 - inner setTimeout
```

执行过程：

1. 执行 `console.log('1')`
2. 注册外层 setTimeout 回调到宏任务队列
3. 注册第一个 Promise.then 回调到微任务队列
4. 执行 `console.log('6')`
5. 同步代码执行完毕
6. 执行微任务：`console.log('4')`，注册 setTimeout 回调到宏任务队列
7. 微任务队列清空
8. 执行宏任务：`console.log('2')`，注册 Promise.then 回调到微任务队列
9. 微任务队列不为空，执行微任务：`console.log('3')`
10. 微任务队列清空
11. 执行宏任务：`console.log('5')`

## 五、定时器与延迟

### 5.1 setTimeout 原理

setTimeout 的延迟时间并不精确，它只是表示"最少延迟"时间。回调被加入宏任务队列，等待事件循环调度执行。如果主线程繁忙，回调可能延迟更久。

```javascript
const start = Date.now();
setTimeout(() => {
  const delay = Date.now() - start;
  console.log(`Delayed by ${delay}ms`);
}, 100);

// 同步阻塞操作
for (let i = 0; i < 100000000; i++) {}
```

上述代码中，即使 setTimeout 设置了 100ms 延迟，由于 for 循环的同步阻塞，实际延迟会远超 100ms。

### 5.2 setInterval 与定时器漂移

setInterval 同样不保证精确执行，它只保证回调之间的"最小间隔"。如果某个回调执行时间过长，下一个回调会延迟执行。

```javascript
let count = 0;
const intervalId = setInterval(() => {
  console.log(`Interval tick: ${++count}`);
  if (count >= 3) {
    clearInterval(intervalId);
  }
}, 100);

// 同步阻塞操作
for (let i = 0; i < 100000000; i++) {}
```

定时器漂移的解决方案：

- 使用 requestAnimationFrame 实现精确动画
- 在回调中记录实际执行时间，动态调整下次延迟
- 使用时间戳比较而不是计数器

### 5.3 requestAnimationFrame

requestAnimationFrame（rAF）是在下次浏览器重排或重绘之前执行的回调，常用于动画。与 setTimeout 相比，rAF 与浏览器刷新频率同步，通常是 60FPS（每帧约 16.67ms）。

```javascript
function animate() {
  // 更新动画状态
  updateAnimation();

  // 绘制
  draw();

  // 请求下一帧
  requestAnimationFrame(animate);
}

requestAnimationFrame(animate);
```

rAF 的特点：

- 与浏览器刷新频率同步，避免丢帧
- 页面不可见时自动暂停，节省性能
- 页面回到可见时，会在下一帧自动恢复

## 六、事件循环与渲染

### 6.1 渲染时机

浏览器在事件循环中会检查是否需要渲染。渲染通常在宏任务执行完成后、微任务执行之前进行：

```
同步代码执行
    ↓
微任务执行（全部清空）
    ↓
是否需要渲染？ ──是──→ 执行渲染
    ↓
执行一个宏任务
    ↓
（重复循环）
```

渲染时机受 `requestAnimationFrame` 控制，浏览器会在每次重绘前执行 rAF 回调。

### 6.2 长任务与性能

如果某个宏任务执行时间过长（如复杂的计算、大量的 DOM 操作），会阻塞页面渲染和用户交互，造成卡顿。

Chrome DevTools Performance 面板会将执行时间超过 50ms 的任务标记为"Long Task"。

优化长任务的方法：

- 将大任务拆分到多个小任务，使用 setTimeout 或 requestAnimationFrame 分批执行
- 使用 Web Worker 将计算密集型任务移到后台线程
- 使用 requestIdleCallback 在浏览器空闲时执行低优先级任务

```javascript
function processTask(items) {
  const chunkSize = 100;
  let index = 0;

  function processChunk() {
    const chunk = items.slice(index, index + chunkSize);
    chunk.forEach(processItem);
    index += chunkSize;

    if (index < items.length) {
      setTimeout(processChunk, 0);  // 让出主线程
    }
  }

  processChunk();
}
```

### 6.3 批量 DOM 操作

频繁的 DOM 操作会触发多次重排/重绘，影响性能。应批量操作 DOM：

```javascript
// 低效：多次重排
element.style.width = '100px';
element.style.height = '100px';
element.style.margin = '10px';

// 高效：使用 CSS 类
element.classList.add('large');

// 高效：使用 requestAnimationFrame 批量更新
requestAnimationFrame(() => {
  element.style.width = '100px';
  element.style.height = '100px';
});
```

## 七、异步迭代与并发

### 7.1 任务调度

多个异步任务并发执行时，事件循环按顺序从宏任务队列取出任务执行：

```javascript
Promise.resolve().then(() => console.log('p1'));
Promise.resolve().then(() => console.log('p2'));
setTimeout(() => console.log('t1'), 0);
setTimeout(() => console.log('t2'), 0);
```

执行顺序：

```
p1
p2
t1
t2
```

微任务总是先于宏任务执行，微任务队列清空后才执行下一个宏任务。

### 7.2 async/await 顺序执行

async/await 本质是 Promise 语法糖，await 会暂停当前 async 函数执行，等待 Promise 解决：

```javascript
async function fetchUser(id) {
  const user = await getUser(id);
  const posts = await getPosts(user.id);
  return { user, posts };
}
```

上述代码中，`getPosts` 等待 `getUser` 完成才执行，实现顺序执行。

### 7.3 并行执行

使用 Promise.all 并行执行多个异步任务：

```javascript
async function fetchAll() {
  const [users, posts, comments] = await Promise.all([
    getUsers(),
    getPosts(),
    getComments()
  ]);
  return { users, posts, comments };
}
```

Promise.all 会并行等待所有 Promise，全部完成后才进入下一个微任务。

## 八、常见场景分析

### 8.1 setTimeout 0 与 Promise

```javascript
setTimeout(() => console.log('timeout'), 0);
Promise.resolve().then(() => console.log('promise'));
console.log('sync');
```

输出：`sync` → `promise` → `timeout`

原因：Promise.then 是微任务，setTimeout 是宏任务，同步代码先执行，然后微任务，然后宏任务。

### 8.2 async/await 链

```javascript
async function test() {
  console.log('1');
  await Promise.resolve();
  console.log('2');
  setTimeout(() => console.log('3'), 0);
  await Promise.resolve();
  console.log('4');
}

test();
console.log('5');
```

输出：`1` → `5` → `2` → `3` → `4`

执行过程：

1. `test()` 调用，`console.log('1')` 同步执行
2. `await Promise.resolve()` 暂停，后续代码注册为微任务
3. `console.log('5')` 同步执行
4. 执行微任务：`console.log('2')`，`setTimeout` 注册为宏任务
5. `await Promise.resolve()` 暂停，后续代码注册为微任务
6. 微任务队列清空后，执行一个宏任务：`console.log('3')`
7. 执行微任务：`console.log('4')`

### 8.3 循环中的异步

```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
```

输出：`3` → `3` → `3`

原因：var 没有块级作用域，setTimeout 回调执行时，循环已经结束，i 的值是 3。

解决方案：使用 let（块级作用域）或闭包：

```javascript
// 方案1：使用 let
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}

// 方案2：使用闭包
for (var i = 0; i < 3; i++) {
  (function(index) {
    setTimeout(() => console.log(index), 0);
  })(i);
}
```

## 九、开发者工具

### 9.1 Performance 面板

Chrome DevTools Performance 面板可以可视化事件循环：

- 录制页面操作，查看各个任务执行时长
- 识别 Long Task（超过 50ms 的任务）
- 查看微任务和宏任务的执行时间线

### 9.2 Console 定时

在 Console 中使用 `console.time` 和 `console.timeEnd` 测量代码执行时间：

```javascript
console.time('async operation');
Promise.resolve().then(() => {
  console.timeEnd('async operation');
});
```

### 9.3 Web Vitals

使用 Web Vitals 库测量核心性能指标：

- LCP (Largest Contentful Paint)
- FID (First Input Delay)
- CLS (Cumulative Layout Shift)

这些指标与事件循环的执行效率密切相关。

## 十、总结

| 概念 | 说明 | 执行时机 |
|------|------|----------|
| 调用栈 | 存储函数执行上下文 | 同步代码执行 |
| 宏任务队列 | setTimeout, setInterval, I/O, 事件 | 每个事件循环执行一个 |
| 微任务队列 | Promise.then, queueMicrotask | 每个事件循环全部清空 |
| 事件循环 | 协调任务执行的机制 | 持续运行 |

事件循环的核心执行顺序：

```
同步代码 → 微任务全部 → 宏任务一个 → （重复微任务+宏任务）
```

理解事件循环对于编写高效、无阻塞的 JavaScript 代码至关重要。
