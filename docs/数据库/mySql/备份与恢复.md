# MySQL 备份与恢复

> 官方文档：[MySQL 8.0 - Backup and Recovery](https://dev.mysql.com/doc/refman/8.0/en/backup-and-recovery.html) / [The Binary Log](https://dev.mysql.com/doc/refman/8.0/en/binary-log.html) / [Point-in-Time Recovery](https://dev.mysql.com/doc/refman/8.0/en/point-in-time-recovery.html)

## 为什么备份是数据库的"保命技能"？

```
数据丢失的常见原因：
  😱 程序 bug：UPDATE 忘了 WHERE，全表数据被改
  😱 人为误操作：手滑执行了 DROP DATABASE
  😱 硬件故障：磁盘损坏、机器宕机
  😱 软件故障：MySQL 崩溃、文件损坏
  😱 黑客攻击：勒索软件加密、删库跑路
  😱 自然灾害：机房断电、火灾

没有备份 = 一次事故就可能让公司倒闭
```

> 💡 **核心观念：备份不是"可选项"，而是"必选项"。而且——能恢复的备份，才叫备份。** 很多公司"有备份"，但从来没演练过恢复，真出事时发现备份是坏的、恢复不了。

### 两个关键指标

| 指标 | 含义 | 通俗解释 |
|-----|------|---------|
| **RPO**（恢复点目标） | 能容忍丢失多少数据 | "最多丢 1 小时的数据" |
| **RTO**（恢复时间目标） | 多久内必须恢复 | "1 小时内必须恢复服务" |

备份策略就是围绕这两个指标设计的：备份越频繁 → RPO 越小；恢复越快 → RTO 越小。

## 一、先建立全局认知：备份的分类

备份有很多种维度，先一次性理清：

### 1.1 按"备份内容"分

| 类型 | 含义 | 工具 | 特点 |
|-----|------|------|------|
| **逻辑备份** | 导出 SQL 语句（CREATE/INSERT） | `mysqldump` | 速度慢、可跨版本、文本可读 |
| **物理备份** | 直接拷贝数据文件（.ibd 等） | `XtraBackup`、拷文件 | 速度快、跨版本难、二进制 |

### 1.2 按"备份范围"分

| 类型 | 含义 | 特点 |
|-----|------|------|
| **全量备份** | 备份所有数据 | 恢复简单，但耗时占空间 |
| **增量备份** | 只备份"上次备份后变化"的数据 | 省空间，但恢复要串起来 |
| **差异备份** | 备份"上次全量后变化"的数据 | 折中方案 |

### 1.3 按"是否影响业务"分

| 类型 | 含义 | 影响 |
|-----|------|------|
| **冷备** | 停掉 MySQL 再备份 | 业务中断 |
| **温备** | 备份时只读不能写 | 部分影响 |
| **热备** | 业务正常运行下备份 | 无影响（InnoDB 支持） |

> 💡 **通俗理解：** 生产环境几乎都用**热备**（业务不能停）。InnoDB 引擎天然支持热备（通过 MVCC 一致性快照）。

## 二、逻辑备份：mysqldump（最常用）

`mysqldump` 是 MySQL 自带的逻辑备份工具，把数据库导成一堆 SQL 语句。

### 2.1 基本用法

```bash
# 导出整个数据库（结构 + 数据）
mysqldump -u root -p mydb > mydb.sql

# 导出特定表
mysqldump -u root -p mydb users orders > tables.sql

# 导出所有数据库
mysqldump -u root -p --all-databases > all.sql

# 只导出结构
mysqldump -u root -p --no-data mydb > struct.sql

# 只导出数据
mysqldump -u root -p --no-create-info mydb > data.sql
```

### 2.2 常用参数详解

| 参数 | 作用 |
|-----|------|
| `-u 用户名` | 指定用户 |
| `-p` | 提示输入密码（**别写在命令行**，会泄露） |
| `--single-transaction` | InnoDB 一致性备份，**不锁表**（重点） |
| `--master-data[=2]` | 记录备份时的 binlog 位点（注释里），用于 PITR |
| `--routines` | 导出存储过程和函数 |
| `--triggers` | 导出触发器（默认开启） |
| `--events` | 导出事件调度 |
| `--no-data` | 只导结构，不导数据 |
| `--no-create-info` | 只导数据，不导建表语句 |
| `--all-databases` | 导出所有库 |
| `--databases` | 导出指定多个库 |
| `--where="条件"` | 只导出符合条件的行 |
| `--quick` | 不缓存全表到内存，逐行导出（大表必备） |

### 2.3 生产级备份命令（推荐组合）

```bash
# 日常全量备份的"标准写法"
mysqldump -u root -p \
  --single-transaction \        # InnoDB 一致性快照，不锁表
  --master-data=2 \             # 记录 binlog 位点（用于 PITR）
  --routines \                  # 存储过程/函数
  --triggers \                  # 触发器
  --events \                    # 事件
  --quick \                     # 大表不占内存
  --default-character-set=utf8mb4 \  # 指定字符集
  mydb > mydb_$(date +%Y%m%d).sql
```

> 💡 **提示：** 这条命令是生产备份的"黄金模板"，记住 `--single-transaction`（不锁表）和 `--master-data=2`（记位点）这两个关键参数。

### 2.4 一致性备份原理：--single-transaction

```bash
# ❌ 没加 --single-transaction：备份期间表被锁，或数据不一致
mysqldump -u root -p mydb > b.sql

# ✅ 加了 --single-transaction：开启事务拿到一致性快照
mysqldump -u root -p --single-transaction mydb > b.sql
```

**原理：** `--single-transaction` 会在一个 `REPEATABLE READ` 事务里导出，利用 InnoDB 的 MVCC 机制，**导出的始终是同一时刻的数据快照**，期间其他会话的增删改不影响、也不被阻塞。

> ⚠️ **注意：** `--single-transaction` **只对 InnoDB 有效**（MyISAM 不支持事务，仍会锁表）。而且**不能和 `--lock-tables` 同时用**。

### 2.5 --master-data 的作用

```bash
# --master-data=2 会在导出文件开头记录 binlog 位点（注释掉）
mysqldump -u root -p --single-transaction --master-data=2 mydb > b.sql
```

打开 `b.sql`，开头能看到：

```sql
-- CHANGE MASTER TO MASTER_LOG_FILE='mysql-bin.000123', MASTER_LOG_POS=12345;
```

这一行记录了"备份完成时 binlog 在哪个文件的哪个位置"，是**做 PITR（按时间点恢复）的起点**。

| 值 | 含义 |
|----|------|
| `--master-data=1` | 记录位点，且不注释（直接生效，用于搭建从库） |
| `--master-data=2` | 记录位点，但注释掉（留作参考，常用） |

### 2.6 大库的 mysqldump 优化

```
问题：几十 GB 的库，mysqldump 要几小时，还占内存
优化建议：
  - 必加 --quick：不把全表加载到内存，逐行导出
  - 必加 --single-transaction：避免锁表
  - 分库分表分别导出，并行加速
  - 超大库考虑用物理备份（XtraBackup），比 mysqldump 快很多
  - 压缩输出：mysqldump ... | gzip > b.sql.gz
```

## 三、物理备份

### 3.1 直接拷贝文件（冷备）

```bash
# 1. 停掉 MySQL（必须！运行时拷贝会损坏数据）
mysqladmin -u root -p shutdown

# 2. 拷贝整个数据目录
cp -r /var/lib/mysql /backup/mysql_$(date +%Y%m%d)

# 3. 重启 MySQL
systemctl start mysqld
```

> ⚠️ **注意：** 直接拷贝文件**必须先停库**（冷备），运行时拷贝会得到损坏的、不一致的数据。这种方法简单粗暴，但要停业务，仅适合小项目或维护窗口。

### 3.2 Percona XtraBackup（热物理备份，推荐）

**XtraBackup** 是 Percona 公司开源的物理热备工具，**不停机、不锁表**（InnoDB），速度快，是生产环境主流方案。

```bash
# 全量备份
xtrabackup --backup --target-dir=/backup/full -u root -p

# 准备备份（让备份达到一致状态，恢复前必做）
xtrabackup --prepare --target-dir=/backup/full

# 恢复（把数据拷回 MySQL 数据目录）
xtrabackup --copy-back --target-dir=/backup/full

# 增量备份（基于上次全量）
xtrabackup --backup --target-dir=/backup/inc1 \
  --incremental-basedir=/backup/full -u root -p
```

| 对比 | mysqldump（逻辑） | XtraBackup（物理） |
|-----|-------------------|-------------------|
| 速度 | 慢 | **快** |
| 大库表现 | 几十 GB 就吃力 | TB 级也行 |
| 锁表 | InnoDB 不锁 | InnoDB 不锁 |
| 跨版本/跨平台 | ✅ 容易 | ❌ 较难 |
| 备份可读性 | ✅ 文本 SQL | ❌ 二进制 |
| 占空间 | 较小 | 较大 |
| 适用 | 小中型库、跨版本迁移 | 大型生产库 |

> 💡 **提示：** 小库用 `mysqldump`，大库（几十 GB 以上）用 `XtraBackup`。

## 四、二进制日志 binlog（恢复的灵魂）

**binlog（binary log）记录了所有"改数据"的操作**（INSERT/UPDATE/DELETE/DDL）。它是数据恢复、主从复制的核心。

```
备份 = 某一时刻的"照片"（静态快照）
binlog = 从那时起的"录像"（所有变更）
备份 + binlog = 能恢复到任意时刻的数据 ✅
```

> 💡 **通俗理解：** 全量备份是"存档点"，binlog 是"游戏录像"。从存档点开始，把录像重放到某个时刻，就能回到那个时刻的状态——这就是 **PITR（按时间点恢复）**。

### 4.1 开启 binlog

binlog **默认可能没开**，需要在配置文件 `my.cnf`（Linux）/ `my.ini`（Windows）里配置：

```ini
[mysqld]
# 开启 binlog
log_bin = mysql-bin
# 服务器唯一 ID（主从复制必填）
server_id = 1
# binlog 格式（推荐 ROW）
binlog_format = ROW
# 保留天数（避免占满磁盘）
binlog_expire_logs_seconds = 604800   # 7 天
```

改完**重启 MySQL** 生效。验证：

```sql
SHOW VARIABLES LIKE 'log_bin';        -- ON 表示已开启
SHOW VARIABLES LIKE 'binlog_format';  -- 查看格式
SHOW VARIABLES LIKE 'server_id';
```

### 4.2 binlog 的三种格式

| 格式 | 记录内容 | 优点 | 缺点 |
|-----|---------|------|------|
| **STATEMENT** | 记录 SQL 语句 | 日志小 | 某些函数（NOW、UUID）主从不一致 |
| **ROW** | 记录每行数据的变更 | **精确，主从一致**（推荐） | 日志较大 |
| **MIXED** | 混合 | 折中 | 复杂 |

> 💡 **提示：生产环境推荐 `ROW` 格式**，虽然日志大点，但数据精确、主从一致，恢复时最可靠。

### 4.3 查看 binlog

```sql
-- 查看所有 binlog 文件
SHOW BINARY LOGS;
-- 或 SHOW MASTER STATUS;  （查看当前正在写的 binlog 和位点）

-- 查看某个 binlog 里的事件
SHOW BINLOG EVENTS IN 'mysql-bin.000001';

-- 查看从指定位置开始的事件
SHOW BINLOG EVENTS IN 'mysql-bin.000001' FROM 12345 LIMIT 10;
```

**`SHOW BINLOG EVENTS` 输出的关键列：**

| 列 | 含义 |
|---|------|
| `Log_name` | binlog 文件名 |
| `Pos` | 事件起始位置 |
| `Event_type` | 事件类型（Query、Write_rows、Update_rows...） |
| `End_log_pos` | 事件结束位置 |
| `Info` | 具体内容（SQL 或影响的表） |

### 4.4 mysqlbinlog 工具

`mysqlbinlog` 是命令行工具，用来**查看和解析 binlog 文件**：

```bash
# 查看 binlog 内容（原始）
mysqlbinlog mysql-bin.000001

# 解码 ROW 格式（变成可读 SQL）
mysqlbinlog --base64-output=DECODE-ROWS -v mysql-bin.000001

# 按时间范围提取
mysqlbinlog --start-datetime="2026-07-02 10:00:00" \
            --stop-datetime="2026-07-02 12:00:00" \
            mysql-bin.000001

# 按位置范围提取
mysqlbinlog --start-position=12345 --stop-position=23456 mysql-bin.000001
```

## 五、数据恢复

### 5.1 全量恢复（从备份恢复）

```bash
# 方式一：命令行导入
mysql -u root -p mydb < mydb_backup.sql

# 方式二：在 mysql 客户端里用 SOURCE
mysql -u root -p
mysql> CREATE DATABASE IF NOT EXISTS mydb;
mysql> USE mydb;
mysql> SOURCE /path/to/mydb_backup.sql;
```

> ⚠️ **注意：全量恢复只能恢复到"备份那一刻"**。备份之后的变更需要靠 binlog 重放（见 PITR）。

### 5.2 基于时间点恢复 PITR（核心！）

**场景：** 早上 2 点做了全量备份，上午 11 点有人误删了库。想恢复到 11 点误删前的状态。

**PITR 三步走：**

```
步骤 1：恢复最近的全量备份（回到今早 2 点）
步骤 2：找出全量备份对应的 binlog 位点（--master-data 记录的）
步骤 3：用 mysqlbinlog 重放"备份位点 → 误删前"之间的 binlog
```

**具体操作：**

```bash
# 步骤 1：先恢复全量备份
mysql -u root -p mydb < mydb_20260702.sql

# 步骤 2：查看备份文件里的 binlog 位点
head -50 mydb_20260702.sql | grep "CHANGE MASTER"
# 得到：MASTER_LOG_FILE='mysql-bin.000123', MASTER_LOG_POS=12345

# 步骤 3：重放 binlog 到误删前（假设误删发生在 10:59:00）
mysqlbinlog --start-position=12345 \
            --stop-datetime="2026-07-02 10:59:00" \
            mysql-bin.000123 mysql-bin.000124 \
  | mysql -u root -p
```

> 💡 **提示：** `--stop-datetime` 设成"误操作前一秒"，这样误删那行就不会被重放。

### 5.3 基于位置点恢复（更精确）

时间不够精确时（同一秒内有多个操作），用**位置点**精确定位：

```bash
# 1. 先查看 binlog，找到误操作的精确位置
mysqlbinlog --base64-output=DECODE-ROWS -v mysql-bin.000124 | grep -A5 -B5 "DROP"

# 假设发现误删在 position 50000，下一条正常操作在 50500

# 2. 重放：位点 12345 → 50000（误删前）
mysqlbinlog --start-position=12345 --stop-position=50000 mysql-bin.000124 \
  | mysql -u root -p

# 3. 跳过误删，继续重放：50500 → 结束（如果误删后还有正常操作）
mysqlbinlog --start-position=50500 mysql-bin.000124 \
  | mysql -u root -p
```

**PITR vs 位置点恢复：**

| 方式 | 精确度 | 适用 |
|-----|--------|------|
| 按时间 `--stop-datetime` | 秒级 | 大致恢复 |
| 按位置 `--stop-position` | 操作级 | 精确跳过误操作 |

## 六、误删抢救实战流程

### 6.1 误删库/表（DROP DATABASE/TABLE）

```
1️⃣ 第一时间：不要再写入！把库设为只读，防止 binlog 被覆盖
   FLUSH TABLES WITH READ LOCK;
   SET GLOBAL read_only = ON;

2️⃣ 确认有全量备份 + binlog 都在
   ls -l /backup/        # 找最近的备份
   SHOW BINARY LOGS;     # 确认 binlog 没丢

3️⃣ 在备用机器上演练恢复（别直接在生产上操作！）
   - 恢复全量备份
   - 用 mysqlbinlog 重放到 DROP 之前

4️⃣ 验证恢复的数据正确

5️⃣ 把恢复好的数据导回生产
```

> ⚠️ **注意：抢救的第一原则是"保护现场"**——别再往库里写东西，别覆盖 binlog。binlog 一旦丢失或被覆盖，就无法 PITR 了。

### 6.2 误删数据行（DELETE 没带 WHERE）

```
如果是 DELETE 误删（没 DROP 表）：
  - 同样用 PITR：恢复全量 + 重放 binlog 到误删前
  - 或更轻量：用 binlog 找出被删的数据，单独 INSERT 回去

如果是 UPDATE 误改：
  - binlog（ROW 格式）记录了"改前改后"的值
  - 可以从 binlog 提取"改前的值"反向恢复
```

### 6.3 完全没有备份怎么办？

```
最后的希望：
  1. binlog：如果 binlog 还在，至少能重放历史变更
  2. undolog：InnoDB 的回滚日志（短期，用于事务回滚，时效短）
  3. 数据恢复工具：如 undrop-for-innodb（从 ibd 文件提取）
  4. 专业数据恢复服务：找磁盘恢复公司

所以——一定要提前做好备份！别等出事才后悔。
```

## 七、备份策略（3-2-1 原则）

### 7.1 经典的 3-2-1 备份原则

```
3 份数据：1 份原件 + 2 份备份
2 种介质：存在 2 种不同的存储介质上
1 份异地：至少 1 份存到异地（防机房故障）
```

### 7.2 推荐的备份组合

```
每日：mysqldump 全量逻辑备份（小快灵，便于恢复单表）
持续：开启 binlog（用于 PITR，恢复到秒级）
每周/大库：XtraBackup 物理全量 + 增量（大库快速恢复）
异地：备份文件同步到对象存储（OSS/S3）/ 另一个机房
演练：每月做一次"恢复演练"，验证备份真的能恢复
```

### 7.3 备份保留策略

| 备份类型 | 建议保留 |
|---------|---------|
| 每日全量 | 最近 7~30 天 |
| 每周全量 | 最近 4~12 周 |
| 每月全量 | 最近 12 个月 |
| binlog | 7~30 天（按 RPO 定） |

### 7.4 备份脚本示例（配合 cron 定时任务）

```bash
#!/bin/bash
# /opt/scripts/backup_mysql.sh

DATE=$(date +%Y%m%d_%H%M)
BACKUP_DIR=/backup/mysql
DB_USER=root
DB_PASS='你的密码'
KEEP_DAYS=7

# 全量备份
mysqldump -u$DB_USER -p$DB_PASS \
  --single-transaction --master-data=2 --routines --triggers --quick \
  mydb | gzip > $BACKUP_DIR/mydb_$DATE.sql.gz

# 删除 7 天前的备份
find $BACKUP_DIR -name "mydb_*.sql.gz" -mtime +$KEEP_DAYS -delete
```

```bash
# 加入 crontab，每天凌晨 2 点执行
crontab -e
# 输入：
0 2 * * * /opt/scripts/backup_mysql.sh >> /var/log/mysql_backup.log 2>&1
```

> ⚠️ **注意：脚本里密码直接写明文有安全风险。** 生产环境建议用 MySQL 配置文件存凭证（`~/.my.cnf` 的 `[client]` 段，设权限 600），而不是写在命令行/脚本里。

## 八、主从复制简介（binlog 的延伸应用）

binlog 除了恢复，还是**主从复制**的基础——把主库的 binlog 在从库上重放，实现数据同步。

```
主库（Master）                从库（Slave）
  写入数据                      实时同步
    │                            │
    ▼                            ▼
  binlog  ──── 读取重放 ────→  relay log → 应用到数据
```

**核心价值：**

| 用途 | 说明 |
|-----|------|
| **读写分离** | 主库写，从库读，分担压力 |
| **数据备份** | 从库就是一份实时副本 |
| **高可用** | 主库挂了，从库顶上 |
| **异地容灾** | 从库在另一个机房 |

**配置步骤概览：**

```ini
# 主库 my.cnf
[mysqld]
log_bin = mysql-bin
server_id = 1
```

```ini
# 从库 my.cnf
[mysqld]
server_id = 2
relay_log = relay-bin
```

```sql
-- 主库：创建复制账号
CREATE USER 'repl'@'%' IDENTIFIED BY '密码';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';

-- 主库：查看当前 binlog 位点
SHOW MASTER STATUS;

-- 从库：配置主库信息并启动复制
CHANGE REPLICATION SOURCE TO
  SOURCE_HOST='主库IP',
  SOURCE_USER='repl',
  SOURCE_PASSWORD='密码',
  SOURCE_LOG_FILE='mysql-bin.000123',
  SOURCE_LOG_POS=12345;

START REPLICA;          -- MySQL 8.0+（旧版 START SLAVE）
SHOW REPLICA STATUS\G   -- 查看复制状态（关键看 Slave_IO/SQL_Running）
```

> 💡 **提示：** 主从复制是另一个大主题，本文只做简介。其底层就是"持续地做 binlog 重放"，和 PITR 原理相通。详细的复制拓扑、延迟优化、高可用方案可单独成篇。

## 九、常见问题与注意事项

### 问题 1：备份文件能恢复，但中文乱码

```bash
# 原因：备份或恢复时字符集不一致
# 解决：备份和恢复都显式指定 utf8mb4

# 备份时
mysqldump --default-character-set=utf8mb4 ... mydb > b.sql

# 恢复时
mysql --default-character-set=utf8mb4 mydb < b.sql
```

### 问题 2：binlog 占满磁盘怎么办

```sql
-- 1. 查看所有 binlog
SHOW BINARY LOGS;

-- 2. 手动清理旧的 binlog（只保留最新的）
PURGE BINARY LOGS BEFORE '2026-07-01';

-- 3. 或删除指定文件之前的
PURGE BINARY LOGS TO 'mysql-bin.000130';

-- 4. 长期方案：配置自动过期
-- binlog_expire_logs_seconds = 604800  （7天）
```

> ⚠️ **注意：删 binlog 前确认从库已经同步完，且不需要这些 binlog 做 PITR，否则删了就恢复不了了。**

### 问题 3：PITR 恢复后数据对不上

```
排查：
  1. 全量备份的 binlog 位点找对了吗？（--master-data 记录的）
  2. start-position 是备份位点，不是 0
  3. 时间点（stop-datetime）设准了吗？（要在误操作之前）
  4. binlog 格式是 ROW 吗？（ROW 最精确）
  5. 中间有没有跨多个 binlog 文件？（都要包含进去）
```

### 问题 4：从备份恢复要多久？

```
取决于：
  - 备份大小（几十 GB vs 几 TB）
  - 恢复方式（mysqldump 重新执行 SQL 慢；物理备份拷贝快）
  - binlog 重放量（PITR 要重放所有变更，量大了很慢）

优化：
  - 大库用物理备份恢复（比 mysqldump 快 10 倍以上）
  - 提高 innodb_flush_log_at_trx_commit 恢复时的写入速度（恢复后改回）
```

### 问题 5：mysqldump 备份卡住不动

```
排查：
  1. 是不是没加 --single-transaction，被锁了
  2. 是不是没加 --quick，大表占满内存在交换
  3. 网络问题
  4. 磁盘满了（备份文件写不进去）
```

### 问题 6：如何只恢复某一张表？

```bash
# 从全量备份里提取单表的建表和数据（备份是文本，可以编辑）
# 方法：用 sed 截取备份文件中该表的部分
sed -n '/-- Current Database: `mydb`/,/-- Current Database:/p' all.sql > mydb.sql

# 更好的做法：备份时就分表单独导出，恢复时直接用对应的表文件
mysqldump -u root -p mydb users > users.sql
```

## 十、快速参考

### 备份命令速查

```bash
# 逻辑全量备份（生产模板）
mysqldump -u root -p --single-transaction --master-data=2 \
  --routines --triggers --events --quick \
  --default-character-set=utf8mb4 库名 > 备份.sql

# 压缩备份
mysqldump -u root -p --single-transaction 库名 | gzip > 备份.sql.gz

# 物理备份（XtraBackup）
xtrabackup --backup --target-dir=/backup/full -u root -p
xtrabackup --prepare --target-dir=/backup/full
```

### 恢复命令速查

```bash
# 全量恢复
mysql -u root -p 库名 < 备份.sql
gunzip < 备份.sql.gz | mysql -u root -p 库名   # 解压恢复

# PITR 按时间点恢复
mysqlbinlog --start-position=位点 --stop-datetime="时间" binlog文件 \
  | mysql -u root -p

# 按位置点恢复
mysqlbinlog --start-position=起始 --stop-position=结束 binlog文件 \
  | mysql -u root -p
```

### binlog 常用命令

```sql
SHOW VARIABLES LIKE 'log_bin';          -- 是否开启
SHOW VARIABLES LIKE 'binlog_format';    -- 格式
SHOW BINARY LOGS;                       -- 所有 binlog 文件
SHOW MASTER STATUS;                     -- 当前 binlog 位点
SHOW BINLOG EVENTS IN '文件名';         -- 查看 binlog 事件
PURGE BINARY LOGS BEFORE '日期';        -- 清理旧 binlog
```

### 备份恢复知识体系

```
备份
├── 逻辑备份：mysqldump（小库、可读、跨版本）
├── 物理备份：XtraBackup（大库、快、热备）
└── binlog：增量记录所有变更（PITR 的基础）

恢复
├── 全量恢复：mysql < backup.sql
├── PITR：全量备份 + binlog 重放（回到任意时刻）
└── 位置点恢复：精确跳过误操作

策略
├── 3-2-1 原则：3 份数据 / 2 介质 / 1 异地
├── 全量 + binlog 组合
└── 定期恢复演练（能恢复才算数！）
```

### 备份恢复自检清单

```
□ 开启了 binlog 吗？（log_bin=ON，格式 ROW）
□ 有定期全量备份吗？（mysqldump --single-transaction）
□ 备份文件存到异地了吗？（不止一份）
□ 知道怎么从备份恢复吗？（演练过吗？）
□ 知道怎么做 PITR 吗？（全量 + binlog 重放）
□ binlog 保留时间够吗？（满足 RPO）
□ 误删时知道第一时间保护现场吗？（只读、别覆盖 binlog）
□ 应用账号是最小权限吗？（防止误 DROP）
```
