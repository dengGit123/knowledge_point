# MySQL 视图与触发器

> 官方文档：[MySQL 8.0 - Views](https://dev.mysql.com/doc/refman/8.0/en/views.html) / [Triggers](https://dev.mysql.com/doc/refman/8.0/en/triggers.html)

## 这篇讲什么？

本文介绍两个"数据库里的便捷工具"：

1. **视图（VIEW）**：把一段复杂查询"存成一张虚拟表"，以后像查表一样用它
2. **触发器（TRIGGER）**：在增删改时**自动执行**的一段逻辑，像事件监听

```
视图 = 复杂查询的"快捷方式"，是个虚拟表
触发器 = 数据变更时的"自动回调函数"
```

> 💡 **提示：** 视图在实际项目中**比较常用**（简化查询、权限控制）；触发器**要慎用**（隐蔽的副作用、难调试），现代开发更倾向用应用层逻辑代替。

---

# 第一部分：视图（VIEW）

## 一、什么是视图？

**视图是一张"虚拟表"**——它本身不存数据，而是保存了一条 `SELECT` 查询语句。每次查询视图，就等于执行它背后的那条 SELECT。

```
真实表 users、orders（存数据）
        ↓
   视图 v_user_orders = 一条 JOIN 查询
        ↓
   查询视图 = 自动执行 JOIN，返回结果

视图本身不占数据存储空间（只存定义）
```

> 💡 **通俗理解：** 视图就像"预设的搜索条件"。你把"北京用户 + 他的订单"这个复杂查询存成视图 `v_beijing_orders`，以后直接 `SELECT * FROM v_beijing_orders` 就行，不用每次重写 JOIN。

---

## 二、创建视图

### 2.1 基本语法

```sql
CREATE VIEW 视图名 AS
SELECT 查询语句;
```

### 2.2 示例：把多表 JOIN 存成视图

```sql
-- 原本要写复杂 JOIN 查"用户 + 订单"
SELECT u.id, u.name, o.id AS order_id, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- 把它存成视图
CREATE VIEW v_user_orders AS
SELECT u.id, u.name, o.id AS order_id, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- 以后直接查视图，像查表一样
SELECT * FROM v_user_orders;
SELECT * FROM v_user_orders WHERE name = '张三';
```

### 2.3 创建或替换视图

```sql
-- 如果视图已存在就替换，不存在就创建
CREATE OR REPLACE VIEW v_user_orders AS
SELECT id, name FROM users;

-- 或用 ALTER 修改视图
ALTER VIEW v_user_orders AS
SELECT id, name, age FROM users;
```

### 2.4 删除视图

```sql
DROP VIEW IF EXISTS v_user_orders;
```

### 2.5 查看视图

```sql
-- 查看视图定义（看到背后的 SELECT）
SHOW CREATE VIEW v_user_orders;

-- 查看所有表和视图
SHOW FULL TABLES WHERE Table_type = 'VIEW';
```

## 三、视图的用途

### 3.1 简化复杂查询

```sql
-- 把常用的复杂统计查询存成视图
CREATE VIEW v_city_stats AS
SELECT city,
       COUNT(*)   AS 人数,
       AVG(age)   AS 平均年龄,
       SUM(balance) AS 总余额
FROM users
GROUP BY city;

-- 业务代码直接用，不用写一堆聚合 SQL
SELECT * FROM v_city_stats WHERE 人数 > 100;
```

### 3.2 权限控制（数据隔离）

```sql
-- 只暴露部分列给某个角色（隐藏敏感字段）
CREATE VIEW v_user_public AS
SELECT id, name, age FROM users;   -- 不含 email、balance 等敏感字段

-- 给某个账号只授权访问视图，不授权访问原表
GRANT SELECT ON v_user_public TO 'app_read'@'%';
-- app_read 只能看到 id/name/age，看不到 email
```

### 3.3 屏蔽底层表结构变化

```sql
-- 底层表改名/重构后，只要视图定义更新，上层查询代码不用改
-- 比如 users 表拆成 user_base + user_ext
CREATE VIEW users AS        -- 视图名保持不变
SELECT b.id, b.name, e.age, e.balance
FROM user_base b
JOIN user_ext e ON b.id = e.id;
```

## 四、可更新视图（通过视图改数据）

某些简单视图，可以直接 `INSERT/UPDATE/DELETE`，操作会作用到底层表。

```sql
-- 简单视图（单表、无聚合）通常是可更新的
CREATE VIEW v_user_minor AS
SELECT id, name, age FROM users WHERE age < 18;

-- 通过视图更新（实际改的是 users 表）
UPDATE v_user_minor SET age = 17 WHERE id = 1;

-- 通过视图插入（注意：插入的数据会进 users 表，但不一定满足 age<18 条件）
INSERT INTO v_user_minor(id, name, age) VALUES(10, '小王', 15);
```

### 可更新视图的条件

视图**必须满足**以下条件才能更新：

- 只引用**一张表**（无 JOIN 多表）
- 不包含**聚合函数**（COUNT、SUM...）
- 不包含 `DISTINCT`、`GROUP BY`、`HAVING`
- 不包含计算列（如 `amount * 2`）

> ⚠️ **注意：** 复杂视图（带 JOIN、聚合）**不可更新**。实际开发中**很少通过视图改数据**，视图主要用于查询。

### WITH CHECK OPTION

```sql
CREATE VIEW v_user_minor AS
SELECT id, name, age FROM users WHERE age < 18
WITH CHECK OPTION;

-- 此后插入/更新必须满足 age < 18，否则报错
INSERT INTO v_user_minor VALUES(11, '小张', 20);  -- ❌ 报错，age 不满足 < 18
```

---

## 五、视图的优缺点

| 优点 | 缺点 |
|-----|------|
| 简化复杂查询，复用 SQL | 视图嵌套太多时性能差、难优化 |
| 提供数据安全（隐藏敏感列） | 复杂视图每次查询都重算，可能慢 |
| 屏蔽表结构变化 | 不能建索引（视图本身无索引） |
| 对外接口稳定 | 可更新视图有限制，易出错 |

> 💡 **提示：** 视图适合**读多写少、查询复用、权限隔离**的场景。别为了用视图而用——简单的单表查询没必要建视图。

# 第二部分：触发器（TRIGGER）

## 六、什么是触发器？

**触发器是绑定在表上的"自动动作"**：当对这张表执行 `INSERT/UPDATE/DELETE` 时，会**自动触发**一段预先写好的 SQL，无需手动调用。

```
普通函数/存储过程：你主动 CALL，才会执行
触发器：你对表增删改时，它自动执行（你拦不住）

类似前端的"事件监听"：
  users 表 INSERT → 自动执行 trigger 里的逻辑（比如写日志）
```

> ⚠️ **注意：触发器是"隐式执行"的**，调试困难，容易出现"不知道为什么数据变了"的问题。**慎用。**

---

## 七、创建触发器

### 7.1 语法

```sql
CREATE TRIGGER 触发器名
{BEFORE | AFTER} {INSERT | UPDATE | DELETE}
ON 表名
FOR EACH ROW
BEGIN
    -- 触发时执行的逻辑
END;
```

**四个关键要素：**

| 要素 | 取值 | 说明 |
|-----|------|------|
| 时机 | `BEFORE` / `AFTER` | 在操作**之前**还是**之后**触发 |
| 事件 | `INSERT` / `UPDATE` / `DELETE` | 监听哪种操作 |
| 表 | `ON 表名` | 绑定哪张表 |
| 粒度 | `FOR EACH ROW` | 行级触发（每改一行触发一次） |

### 7.2 示例：用户注册时自动写日志

```sql
DELIMITER $$

-- 用户插入后，自动往日志表写一条记录
CREATE TRIGGER trg_user_insert
AFTER INSERT ON users
FOR EACH ROW
BEGIN
    INSERT INTO user_log(user_id, action, log_time)
    VALUES(NEW.id, '注册', NOW());
END $$

DELIMITER ;
```

此后，只要执行 `INSERT INTO users ...`，就会**自动**往 `user_log` 写一条日志。

### 7.3 NEW 和 OLD 关键字

触发器里用 `NEW` 和 `OLD` 访问数据：

| 关键字 | 含义 | 适用事件 |
|-------|------|---------|
| `NEW` | **新**数据（插入/更新后的值） | INSERT、UPDATE |
| `OLD` | **旧**数据（更新/删除前的值） | UPDATE、DELETE |

```sql
-- INSERT：只有 NEW（新插入的行）
-- DELETE：只有 OLD（被删除的行）
-- UPDATE：NEW（新值）和 OLD（旧值）都有

-- 示例：用户改名字时，记录旧名字和新名字
DELIMITER $$
CREATE TRIGGER trg_user_update_name
AFTER UPDATE ON users
FOR EACH ROW
BEGIN
    IF OLD.name != NEW.name THEN
        INSERT INTO name_change_log(user_id, old_name, new_name, changed_at)
        VALUES(OLD.id, OLD.name, NEW.name, NOW());
    END IF;
END $$
DELIMITER ;
```

### 7.4 BEFORE 触发器：修改或校验数据

`BEFORE` 触发器在数据**真正写入前**执行，可以**修改待写入的值**或**拦截操作**。

```sql
-- 插入前自动校验/补全数据
DELIMITER $$
CREATE TRIGGER trg_user_before_insert
BEFORE INSERT ON users
FOR EACH ROW
BEGIN
    -- 如果没传 age，默认填 18
    IF NEW.age IS NULL THEN
        SET NEW.age = 18;
    END IF;

    -- 名字统一转大写
    SET NEW.name = UPPER(NEW.name);
END $$
DELIMITER ;
```

> 💡 **提示：** `BEFORE` 触发器可以**修改 NEW 的值**（如补全、格式化），`AFTER` 触发器不能改（数据已写入）。

### 7.5 删除触发器

```sql
DROP TRIGGER IF EXISTS trg_user_insert;
```

### 7.6 查看触发器

```sql
SHOW TRIGGERS;                         -- 查看所有触发器
SHOW TRIGGERS WHERE `Table` = 'users'; -- 查看某张表的触发器
```

## 八、BEFORE vs AFTER 触发器

| 时机 | 执行点 | 能改数据吗 | 典型用途 |
|-----|--------|:--------:|---------|
| `BEFORE` | 操作**前** | ✅ 能改 NEW | 数据校验、补全、格式化 |
| `AFTER` | 操作**后** | ❌ | 写日志、同步到其他表、统计 |

```sql
-- BEFORE：拦截非法数据
CREATE TRIGGER check_age BEFORE INSERT ON users
FOR EACH ROW
BEGIN
    IF NEW.age < 0 OR NEW.age > 150 THEN
        SIGNAL SQLSTATE '45000'   -- 抛出错误，阻止插入
        SET MESSAGE_TEXT = '年龄不合法';
    END IF;
END;

-- AFTER：记录操作日志
CREATE TRIGGER log_after_delete AFTER DELETE ON users
FOR EACH ROW
BEGIN
    INSERT INTO delete_log(table_name, record_id, deleted_at)
    VALUES('users', OLD.id, NOW());
END;
```

## 九、触发器的优缺点与"坑"

### 优点

- 自动执行，无需业务代码调用
- 保证数据联动一致性（如自动写日志、自动更新统计）
- 集中管理，所有应用共用

### 缺点（为什么慎用）

| 问题 | 说明 |
|-----|------|
| **隐蔽性强** | 数据莫名变化，新手很难发现是触发器干的 |
| **难调试** | 没有 print、断点，出错难排查 |
| **性能影响** | 每行增删改都触发，大批量操作时性能差 |
| **递归/死循环风险** | 触发器里又操作表，可能引发递归 |
| **增加锁** | 触发器内 SQL 也会加锁，影响并发 |
| **迁移困难** | 换数据库时触发器语法不兼容 |

### 触发器 vs 应用层逻辑

```
触发器：
  ✅ 自动、强一致（绕不开）
  ❌ 隐蔽、难维护、影响性能

应用层（Java）：
  ✅ 易调试、易维护、可单测
  ✅ 易扩展（可分布式）
  ❌ 需要在每个调用点都写逻辑（可能遗漏）

现代开发趋势：
  核心业务逻辑放应用层（Java/Service）
  触发器只用于"数据强约束/审计日志"等少数场景
```

## 十、视图与触发器对比

| 对比项 | 视图（VIEW） | 触发器（TRIGGER） |
|-------|-------------|------------------|
| 本质 | 虚拟表（存查询定义） | 自动执行的逻辑 |
| 触发方式 | 手动 SELECT | 增删改时自动触发 |
| 存数据吗 | ❌ 不存 | ❌ 不存（存定义） |
| 主要用途 | 简化查询、权限控制 | 数据联动、审计日志 |
| 能改数据吗 | 简单视图可以 | 通过 NEW 改待写入值 |
| 推荐程度 | ✅ 适度使用 | ⚠️ 慎用 |

## 十一、常见问题与注意事项

### 问题 1：视图能加索引吗？

```
❌ 不能。视图本身没有索引，它背后的查询走的是底层表的索引。
优化视图查询 = 优化它背后的 SELECT（给底层表加索引）。
```

### 问题 2：视图查询很慢怎么办？

```
- 检查视图背后的 SELECT，用 EXPLAIN 分析
- 确保底层表的关联列、条件列有索引
- 避免视图层层嵌套（视图查视图）
- 复杂统计视图，可考虑改用"物化表"（定时刷新的实体表）
```

### 问题 3：触发器为什么不生效？

```
排查：
  1. 事件类型对吗？（INSERT 触发器不会在 UPDATE 时触发）
  2. BEFORE/AFTER 写对了吗？
  3. 触发器有没有报错被吞掉？
  4. 用 SHOW TRIGGERS 确认触发器存在
```

### 问题 4：触发器里能不能操作"自己所在的表"？

```sql
-- ❌ 会报错或递归
CREATE TRIGGER t AFTER INSERT ON users
FOR EACH ROW
BEGIN
    INSERT INTO users ...   -- 在 users 触发器里又写 users → 递归
END;

-- 解决：触发器操作"别的表"，不要操作自身表
```

### 问题 5：同一事件能建多个触发器吗？

```
MySQL 5.7.2 之前：同一表同一事件同一时机，只能有 1 个触发器。
MySQL 5.7.2+：可以建多个，用 FOLLOWS/PRECEDES 指定执行顺序。
但建议合并成一个，避免执行顺序混乱。
```

### 问题 6：视图和临时表的区别

```
视图：永久存在（直到 DROP），所有会话共享，不存数据
临时表（TEMPORARY）：会话级，会话结束自动删除，存真实数据
```

## 十二、快速参考

### 视图语法

```sql
-- 创建
CREATE VIEW 视图名 AS SELECT ...;
CREATE OR REPLACE VIEW 视图名 AS SELECT ...;
ALTER VIEW 视图名 AS SELECT ...;

-- 使用（像表一样）
SELECT * FROM 视图名 WHERE ...;

-- 查看 / 删除
SHOW CREATE VIEW 视图名;
DROP VIEW IF EXISTS 视图名;
```

### 触发器语法

```sql
-- 创建
DELIMITER $$
CREATE TRIGGER 触发器名
{BEFORE|AFTER} {INSERT|UPDATE|DELETE}
ON 表名
FOR EACH ROW
BEGIN
    -- 用 NEW / OLD 访问数据
    -- AFTER：写日志、同步表
    -- BEFORE：校验、补全（可 SET NEW.列 = 值）
END $$
DELIMITER ;

-- 查看 / 删除
SHOW TRIGGERS;
DROP TRIGGER IF EXISTS 触发器名;
```

### 使用建议

```
视图（适度用）：
  ✅ 复杂查询复用
  ✅ 敏感字段权限隔离
  ✅ 屏蔽表结构变化
  ❌ 不要为简单查询建视图
  ❌ 不要视图嵌视图

触发器（慎用）：
  ⚠️ 仅用于数据强约束、审计日志等少数场景
  ⚠️ 不要在里面写复杂业务逻辑
  ⚠️ 核心业务逻辑优先放应用层（Java/Service）
  ⚠️ 注意性能（逐行触发）和递归风险
```
