# MySQL 用户与权限管理

> 官方文档：[MySQL 8.0 - User Account Management](https://dev.mysql.com/doc/refman/8.0/en/user-account-management.html) / [GRANT Statement](https://dev.mysql.com/doc/refman/8.0/en/grant.html) / [Roles](https://dev.mysql.com/doc/refman/8.0/en/roles.html)

## 为什么需要用户与权限管理？

```
如果所有应用都用 root 账号连数据库：
  😱 一个 bug → UPDATE 没带 WHERE → 全表数据被改
  😱 一个 bug → DROP DATABASE → 库没了
  😱 代码泄露 → root 密码泄露 → 整个数据库被控制
  😱 内部人员 → 任意查看/修改所有数据

用户与权限管理的作用：
  ✅ 给不同应用/人员分配专用账号
  ✅ 每个账号只给"刚好够用"的权限（最小权限原则）
  ✅ 即使出问题，影响范围也有限
```

> 💡 **核心原则：最小权限原则**——每个账号只拥有完成任务所必需的最小权限。生产应用账号**绝不用 root**，通常只给 `SELECT/INSERT/UPDATE/DELETE`。

## 一、理解 MySQL 账号：user@host

这是新手最容易误解的点：**MySQL 的账号是"用户名 + 主机"的组合，不是只有用户名。**

```
'root'@'localhost'    → 只能从本机连接的 root
'root'@'%'            → 能从任意主机连接的 root
'root'@'192.168.1.5'  → 只能从 192.168.1.5 连接的 root

这三个是【三个不同的账号】！可以有不同的密码和权限。
```

| 主机写法 | 含义 |
|---------|------|
| `'localhost'` | 只能从数据库本机连接 |
| `'127.0.0.1'` | 只能从本机（IPv4）连接 |
| `'192.168.1.%'` | 192.168.1.x 网段的所有主机 |
| `'%'` | **任意主机**（最宽松，慎用） |
| `'%.example.com'` | example.com 域名下的主机 |

> ⚠️ **注意：** `'localhost'` 和 `'127.0.0.1'` 在 MySQL 里**不完全等价**（涉及 socket vs TCP）。`%` 不包含 localhost，如果想让本机也能连，要单独建 `@'localhost'` 账号。

## 二、用户管理

### 2.1 创建用户

```sql
-- 创建用户，指定密码
CREATE USER 'appuser'@'%' IDENTIFIED BY 'StrongPwd123!';

-- 只允许从内网连接（更安全）
CREATE USER 'appuser'@'192.168.1.%' IDENTIFIED BY 'StrongPwd123!';

-- 本机管理员
CREATE USER 'admin'@'localhost' IDENTIFIED BY 'AdminPwd456!';
```

> 💡 **提示：密码要强**——至少 8 位，含大小写字母、数字、特殊字符。别用 `123456`、`password`。

### 2.2 修改密码

```sql
-- MySQL 8.0 推荐写法
ALTER USER 'appuser'@'%' IDENTIFIED BY 'NewPwd789!';

-- 修改当前登录用户自己的密码
ALTER USER USER() IDENTIFIED BY 'NewPwd789!';

-- 旧写法（5.7，8.0 已废弃，了解即可）
SET PASSWORD FOR 'appuser'@'%' = 'NewPwd789!';
```

### 2.3 删除用户

```sql
-- 删除指定账号
DROP USER 'appuser'@'%';

-- 删除时该用户正在连接？默认会等它断开
-- 加上可选：DROP USER 'appuser'@'%'（8.0 暂无强制断开参数，需 KILL）
```

### 2.4 查看用户

```sql
-- 查看所有用户（用户信息存在 mysql 库的 user 表里）
SELECT user, host FROM mysql.user;

-- 查看当前登录的用户
SELECT CURRENT_USER();   -- 返回 'appuser'@'%'（权限判定用的账号）
SELECT USER();           -- 返回 'appuser'@'连接来源IP'（你登录时用的）
```

### 2.5 锁定/解锁用户（临时禁用）

```sql
-- 锁定账号（无法登录，但账号和权限保留）
ALTER USER 'appuser'@'%' ACCOUNT LOCK;

-- 解锁
ALTER USER 'appuser'@'%' ACCOUNT UNLOCK;
```

> 💡 **提示：** 员工离职时，**先锁定账号再处理**，比直接删除更安全（保留审计痕迹，可恢复）。

## 三、权限体系

### 3.1 权限的层级

MySQL 权限分**四个层级**，从大到小：

| 层级 | 作用范围 | 写法 | 示例权限 |
|-----|---------|------|---------|
| **全局** | 所有库所有表 | `*.*` | 超级管理员权限 |
| **数据库** | 某个库的所有表 | `库名.*` | 库级权限 |
| **表** | 某张表 | `库名.表名` | 表级权限 |
| **列** | 某几列 | `库名.表名(列1,列2)` | 列级权限 |
| **例程** | 存储过程/函数 | `库名.过程名` | 执行存储过程 |

```sql
-- 全局：所有库所有表
GRANT SELECT ON *.* TO 'user'@'%';

-- 库级：mydb 库的所有表
GRANT SELECT ON mydb.* TO 'user'@'%';

-- 表级：mydb 库的 users 表
GRANT SELECT ON mydb.users TO 'user'@'%';

-- 列级：只能看 users 表的 name 和 age 列
GRANT SELECT (name, age) ON mydb.users TO 'user'@'%';
```

### 3.2 常用权限一览

| 权限 | 含义 | 给应用账号吗 |
|-----|------|:----------:|
| `SELECT` | 查询 | ✅ |
| `INSERT` | 插入 | ✅ |
| `UPDATE` | 更新 | ✅ |
| `DELETE` | 删除 | ✅ |
| `CREATE` | 建库/建表 | ⚠️ 视情况 |
| `DROP` | 删库/删表 | ❌ 慎给 |
| `ALTER` | 改表结构 | ❌ 慎给 |
| `INDEX` | 创建/删除索引 | ⚠️ 视情况 |
| `REFERENCES` | 外键 | ⚠️ 视情况 |
| `ALL [PRIVILEGES]` | 所有权限 | ❌ 只给管理员 |
| `GRANT OPTION` | 能把权限授给别人 | ❌ 慎给 |
| `RELOAD` | 刷新日志/权限等 | ❌ 管理员 |
| `REPLICATION CLIENT` | 查看主从状态 | ⚠️ 监控用 |
| `REPLICATION SLAVE` | 复制用户（从库用） | 仅从库 |
| `PROCESS` | 查看进程列表 | ❌ 管理员 |
| `SHOW DATABASES` | 查看所有库 | ⚠️ 视情况 |
| `FILE` | 读写文件（LOAD DATA） | ❌ 高危 |

### 3.3 应用账号的标准配置

```sql
-- 绝大多数 Web 应用只需要这 4 个权限
GRANT SELECT, INSERT, UPDATE, DELETE ON mydb.* TO 'appuser'@'%';
```

> 💡 **提示：** 应用账号给 `CRUD`（增删改查）就够了。`CREATE/DROP/ALTER` 这类**改结构**的权限留给 DBA 或运维，避免程序 bug 误删库表。

## 四、授权 GRANT

### 4.1 基本语法

```sql
-- GRANT 权限 ON 范围 TO 账号;
GRANT SELECT, INSERT, UPDATE, DELETE ON mydb.* TO 'appuser'@'%';
```

### 4.2 授予多个权限

```sql
-- 多个权限用逗号隔开
GRANT SELECT, INSERT, UPDATE ON mydb.* TO 'appuser'@'%';

-- 授予某表的所有权限（慎用）
GRANT ALL ON mydb.users TO 'admin'@'localhost';

-- 全局所有权限（超级管理员，慎用）
GRANT ALL PRIVILEGES ON *.* TO 'admin'@'localhost' WITH GRANT OPTION;
```

### 4.3 WITH GRANT OPTION（能否再授权）

```sql
-- 加上 WITH GRANT OPTION，该用户就能把自己的权限授给别人
GRANT SELECT ON mydb.* TO 'admin'@'%' WITH GRANT OPTION;

-- 这相当于"二级管理员"权限，非常敏感，慎给！
```

### 4.4 创建用户并授权一步到位

```sql
-- MySQL 8.0 推荐：CREATE USER + GRANT 分开写（更清晰）
CREATE USER 'appuser'@'%' IDENTIFIED BY 'pwd';
GRANT SELECT, INSERT, UPDATE, DELETE ON mydb.* TO 'appuser'@'%';

-- ⚠️ 旧版可以 GRANT 时直接 IDENTIFIED BY 建用户，但 8.0 已不推荐
```

> ⚠️ **注意：** MySQL 8.0 起，`GRANT` **不再自动创建用户**。必须先 `CREATE USER`，再 `GRANT`。否则报错。

## 五、撤销权限 REVOKE

```sql
-- REVOKE 权限 ON 范围 FROM 账号;
REVOKE DELETE ON mydb.* FROM 'appuser'@'%';

-- 撤销多个权限
REVOKE INSERT, UPDATE ON mydb.* FROM 'appuser'@'%';

-- 撤销所有权限（但保留账号）
REVOKE ALL PRIVILEGES, GRANT OPTION FROM 'appuser'@'%';
```

| 操作 | 关键字 |
|-----|-------|
| 授权 | `GRANT ... TO` |
| 撤权 | `REVOKE ... FROM` |

> 💡 **提示：** `REVOKE` 撤销权限，但**不删除用户**。要彻底移除用 `DROP USER`。

## 六、查看权限

### 6.1 查看某用户的权限

```sql
-- 查看指定账号的所有权限
SHOW GRANTS FOR 'appuser'@'%';

-- 查看当前登录用户的权限
SHOW GRANTS;

-- 输出示例：
-- GRANT SELECT, INSERT, UPDATE, DELETE ON `mydb`.* TO `appuser`@`%`
```

### 6.2 查看所有用户的权限分布

```sql
-- 查看 mysql.user 表（全局权限）
SELECT user, host, Select_priv, Insert_priv, Update_priv, Delete_priv
FROM mysql.user;
```

## 七、刷新权限 FLUSH PRIVILEGES

```sql
-- 刷新权限，让修改生效
FLUSH PRIVILEGES;
```

**什么时候需要 FLUSH PRIVILEGES？**

| 情况 | 需要 FLUSH 吗 |
|-----|:----------:|
| 用 `GRANT/REVOKE/CREATE USER` 修改 | ❌ 不需要（自动刷新） |
| 用 `ALTER USER` 改密码 | ❌ 不需要 |
| **直接 UPDATE mysql.user 表** | ✅ **需要** |

> ⚠️ **注意：** 只有**直接改 `mysql` 系统表**（如 `UPDATE mysql.user SET ...`）时才需要 `FLUSH PRIVILEGES`。用标准的 `GRANT/REVOKE/CREATE/ALTER USER` 语句**会自动刷新，不要多此一举**。日常**推荐用标准语句**，避免直接操作系统表。

## 八、角色 Role（MySQL 8.0+）

**角色是一组权限的集合**。把权限打包成角色，再把角色赋给多个用户，方便统一管理。

```
没有角色：100 个用户，每个都要单独 GRANT 一堆权限，改一次要改 100 次
有角色：定义 1 个角色（含一堆权限），赋给 100 个用户，改权限只改角色

类似"用户组"的概念。
```

### 8.1 创建角色并授权

```sql
-- 创建角色
CREATE ROLE 'app_read', 'app_write';

-- 给角色授权
GRANT SELECT ON mydb.* TO 'app_read';
GRANT INSERT, UPDATE, DELETE ON mydb.* TO 'app_write';
```

### 8.2 把角色赋给用户

```sql
-- 创建用户
CREATE USER 'reader1'@'%' IDENTIFIED BY 'pwd';
CREATE USER 'reader2'@'%' IDENTIFIED BY 'pwd';
CREATE USER 'writer1'@'%' IDENTIFIED BY 'pwd';

-- 把角色赋给用户
GRANT 'app_read' TO 'reader1'@'%', 'reader2'@'%';
GRANT 'app_read', 'app_write' TO 'writer1'@'%';
```

### 8.3 设置默认角色

```sql
-- 用户登录后默认激活的角色（否则角色的权限不生效）
SET DEFAULT ROLE ALL TO 'reader1'@'%';

-- 或只激活部分角色
SET DEFAULT ROLE 'app_read' TO 'reader1'@'%';
```

### 8.4 角色相关操作

```sql
-- 查看角色权限
SHOW GRANTS FOR 'app_read';

-- 查看用户被授予的角色
SHOW GRANTS FOR 'reader1'@%' USING 'app_read';

-- 撤销用户的角色
REVOKE 'app_write' FROM 'writer1'@'%';

-- 删除角色
DROP ROLE 'app_read';
```

> 💡 **提示：** 用户多的系统用角色能大幅简化管理。给角色定义权限，用户只管"属于哪个角色"。

## 九、认证插件与密码策略

### 9.1 认证插件

MySQL 用认证插件决定"怎么验证密码"：

| 插件 | 版本 | 特点 |
|-----|------|------|
| `caching_sha2_password` | **8.0 默认** | SHA-256 加密，更安全 |
| `mysql_native_password` | 5.7 默认 | 老插件，8.0 已降级 |

```sql
-- 查看用户的认证插件
SELECT user, host, plugin FROM mysql.user;

-- 创建用户时指定认证插件（兼容老客户端时）
CREATE USER 'appuser'@'%' IDENTIFIED WITH mysql_native_password BY 'pwd';

-- 修改现有用户的认证插件
ALTER USER 'appuser'@'%' IDENTIFIED WITH mysql_native_password BY 'pwd';
```

> ⚠️ **注意：** MySQL 8.0 默认 `caching_sha2_password`，但**一些老版本客户端/驱动不支持**，会报错 `Authentication plugin 'caching_sha2_password' cannot be loaded`。解决办法：升级驱动，或临时改回 `mysql_native_password`。

### 9.2 密码策略（validate_password）

MySQL 可以强制密码复杂度：

```sql
-- MySQL 8.0 安装密码校验组件
INSTALL COMPONENT 'file://component_validate_password';

-- 查看密码策略配置
SHOW VARIABLES LIKE 'validate_password%';
```

| 配置 | 含义 | 建议值 |
|-----|------|-------|
| `validate_password.length` | 最小长度 | 8 |
| `validate_password.policy` | 策略等级 | MEDIUM |
| `validate_password.mixed_case_count` | 大小写字母数 | 1 |
| `validate_password.number_count` | 数字数 | 1 |
| `validate_password.special_char_count` | 特殊字符数 | 1 |

```
validate_password.policy 等级：
  LOW    只检查长度
  MEDIUM 检查长度 + 数字 + 小写 + 大写 + 特殊字符（默认）
  STRONG 检查长度 + 复杂度 + 字典词
```

```sql
-- 设置密码策略
SET GLOBAL validate_password.policy = MEDIUM;
SET GLOBAL validate_password.length = 8;
```

## 十、安全实践（生产环境必读）

### 10.1 账号安全清单

```
✅ 不要用 root 连接应用数据库
✅ 每个应用用独立账号（A 应用不能访问 B 应用的库）
✅ 应用账号只给 CRUD，不给 CREATE/DROP/ALTER
✅ 密码强复杂度，定期更换
✅ 限制连接主机（@'内网IP段'，别用 @'%'）
✅ 离职员工账号及时锁定/删除
✅ 用角色统一管理权限
✅ 敏感操作（DROP/TRUNCATE）走 DBA 审批
❌ 不要把密码硬编码在代码/脚本里（用配置中心/密钥管理）
❌ 不要给应用账号 GRANT OPTION
❌ 不要给应用账号 FILE 权限（防 LOAD DATA 读写文件）
```

### 10.2 连接加密

```ini
# my.cnf 强制 SSL 加密连接（防密码被抓包）
[mysqld]
require_secure_transport = ON
```

```sql
-- 创建强制 SSL 的用户
CREATE USER 'appuser'@'%' IDENTIFIED BY 'pwd' REQUIRE SSL;
```

### 10.3 定期审计

```sql
-- 审计：谁有什么权限
SELECT user, host FROM mysql.user;
SHOW GRANTS FOR 'appuser'@'%';

-- 审计：当前有哪些连接
SHOW PROCESSLIST;

-- 审计：谁在用超级权限
SELECT user, host, Super_priv FROM mysql.user WHERE Super_priv = 'Y';
```

### 10.4 典型账号划分

| 账号 | 权限 | 用途 |
|-----|------|------|
| `root` | 全部 | 仅本机维护，不连应用 |
| `app_rw` | SELECT/INSERT/UPDATE/DELETE | 应用日常读写 |
| `app_read` | SELECT | 报表/只读应用 |
| `dba` | 全部 + GRANT OPTION | DBA 维护 |
| `repl` | REPLICATION SLAVE | 从库复制 |
| `backup` | SELECT + LOCK TABLES | 备份用 |
| `monitor` | REPLICATION CLIENT + PROCESS | 监控用 |

```sql
-- 备份专用账号（够用就行）
CREATE USER 'backup'@'localhost' IDENTIFIED BY 'pwd';
GRANT SELECT, LOCK TABLES, RELOAD, REPLICATION CLIENT, SHOW VIEW ON *.* TO 'backup'@'localhost';

-- 只读报表账号
CREATE USER 'report'@'192.168.1.%' IDENTIFIED BY 'pwd';
GRANT SELECT ON analytics.* TO 'report'@'192.168.1.%';
```

## 十一、常见问题与注意事项

### 问题 1：用 root 创建了用户，还是登录不了？

```
检查账号的 host 部分：
  CREATE USER 'appuser'@'localhost' ...   只能本机连
  但你从 192.168.1.100 连 → 登录失败

解决：明确允许的来源
  CREATE USER 'appuser'@'192.168.1.%' ... 内网
  CREATE USER 'appuser'@'%' ...           任意（不安全）
```

### 问题 2：GRANT 报错 "You are not allowed to create a user with GRANT"

```
MySQL 8.0：GRANT 不再自动建用户。
解决：先 CREATE USER，再 GRANT。
```

### 问题 3：报错 "Access denied for user"

```
排查：
  1. 用户名/密码对吗？
  2. host 匹配吗？（'%' 不含 'localhost'）
  3. 账号被锁定了吗？（ACCOUNT LOCK）
  4. 密码过期了吗？（ALTER USER ... PASSWORD EXPIRE）
  5. 认证插件客户端支持吗？（caching_sha2_password 老客户端不支持）
  6. 有该库的权限吗？（SHOW GRANTS）
```

### 问题 4：忘记了 root 密码怎么办？

```bash
# 1. 停掉 MySQL
systemctl stop mysqld

# 2. 跳过权限验证启动
mysqld --skip-grant-tables --skip-networking &

# 3. 免密登录
mysql -u root

# 4. 改密码（8.0）
FLUSH PRIVILEGES;
ALTER USER 'root'@'localhost' IDENTIFIED BY '新密码';
FLUSH PRIVILEGES;
exit;

# 5. 正常重启 MySQL
systemctl restart mysqld
```

> ⚠️ **注意：** `--skip-grant-tables` 会让数据库**完全无密码、无权限校验**，操作完务必立即重启恢复正常模式。且配合 `--skip-networking` 禁止远程连接，防止安全风险。

### 问题 5：FLUSH PRIVILEGES 到底什么时候用？

```
用标准语句（CREATE USER / GRANT / REVOKE / ALTER USER）→ 自动刷新，不用 FLUSH
直接 UPDATE mysql 系统表改权限 → 必须手动 FLUSH PRIVILEGES

结论：日常用标准语句，几乎不需要手动 FLUSH。
```

### 问题 6：如何让某个用户只能看到部分库？

```sql
-- 1. 先撤销全局的 SHOW DATABASES 权限
REVOKE SHOW DATABASES ON *.* FROM 'user'@'%';

-- 2. 只给目标库的权限
GRANT SELECT ON mydb.* TO 'user'@'%';

-- 配合启动参数：SHOW DATABASES 只显示有权限的库
-- my.cnf: skip-show-database（更严格）
```

## 十二、快速参考

### 账号管理速查

```sql
-- 创建
CREATE USER '用户'@'主机' IDENTIFIED BY '密码';

-- 改密码
ALTER USER '用户'@'主机' IDENTIFIED BY '新密码';

-- 锁定/解锁
ALTER USER '用户'@'主机' ACCOUNT LOCK;   -- 锁定
ALTER USER '用户'@'主机' ACCOUNT UNLOCK; -- 解锁

-- 删除
DROP USER '用户'@'主机';

-- 查看
SELECT user, host FROM mysql.user;
SELECT CURRENT_USER();
```

### 权限管理速查

```sql
-- 授权（4 个范围）
GRANT 权限 ON *.*              TO '用户'@'主机';  -- 全局
GRANT 权限 ON 库名.*           TO '用户'@'主机';  -- 库级
GRANT 权限 ON 库名.表名        TO '用户'@'主机';  -- 表级
GRANT 权限 (列1,列2) ON 库.表 TO '用户'@'主机';  -- 列级

-- 撤权
REVOKE 权限 ON 范围 FROM '用户'@'主机';
REVOKE ALL PRIVILEGES, GRANT OPTION FROM '用户'@'主机';

-- 查看
SHOW GRANTS FOR '用户'@'主机';
SHOW GRANTS;  -- 当前用户

-- 刷新（仅直接改系统表后需要）
FLUSH PRIVILEGES;
```

### 角色速查（8.0+）

```sql
CREATE ROLE '角色名';                          -- 建角色
GRANT 权限 ON 范围 TO '角色名';                 -- 给角色授权
GRANT '角色名' TO '用户'@'主机';                -- 角色赋给用户
SET DEFAULT ROLE ALL TO '用户'@'主机';          -- 设默认角色
REVOKE '角色名' FROM '用户'@'主机';             -- 撤角色
DROP ROLE '角色名';                            -- 删角色
```

### 权限层级与典型账号

```
全局  *.*              → root / dba
库级  mydb.*           → 应用账号（CRUD）
表级  mydb.users       → 细粒度控制
列级  mydb.users(列)   → 只允许看部分列

生产账号划分：
  app_rw    → SELECT/INSERT/UPDATE/DELETE（应用）
  app_read  → SELECT（报表）
  dba       → ALL（DBA 维护）
  repl      → REPLICATION SLAVE（主从复制）
  backup    → SELECT/LOCK TABLES/RELOAD（备份）
```
