# 垃圾回收机制（Garbage Collection）

> JavaScript 的垃圾回收（GC）是引擎**自动**完成的内存管理机制。开发者不需要手动分配和释放内存，引擎会自动追踪内存的使用情况，回收那些**不再被引用**的对象所占的内存空间。

## 内存生命周期

无论使用哪种编程语言，内存的生命周期几乎都是一样的：

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│  1. 分配内存  │ ──→ │  2. 使用内存  │ ──→ │  3. 释放内存  │
│  (Allocate)  │     │  (Use)       │     │  (Release)  │
└─────────────┘     └──────────────┘     └─────────────┘
```

| 阶段 | JavaScript 中的表现 |
|------|-------------------|
| **分配** | 引擎自动为变量、对象、函数等分配内存，开发者无需关心 |
| **使用** | 对内存进行读写操作（赋值、调用函数、访问属性等） |
| **释放** | 引擎的垃圾回收器自动检测并释放不再使用的内存 |

```javascript
// 1. 分配 — 引擎自动为对象分配内存
let user = { name: 'Alice', age: 25 } // 分配内存存储对象

// 2. 使用 — 读写操作
console.log(user.name) // "Alice"
user.age = 26

// 3. 释放 — 当 user 不再引用该对象时，GC 自动回收
user = null // 解除引用，对象变为"不可达"，等待 GC 回收
```

## 核心概念：可达性

垃圾回收的核心判断标准是**可达性（Reachability）**。

### 什么是可达性

一个值是"可达的"，意味着存在某种方式可以通过**引用链**从**根（Root）**访问到它。

```
根（Root）：
  ┌─────────────────────────────────────────────────┐
  │  • 全局变量（window / globalThis）               │
  │  • 当前正在执行的函数的局部变量和参数              │
  │  • 当前调用栈上所有函数的局部变量                  │
  │  • 被 Root 引用的所有对象                         │
  └─────────────────────────────────────────────────┘
```

### 可达性判定流程

```
                    Root（根）
                   ╱        ╲
                 ╱            ╲
               objA            objB ←─── 可达！
              ╱    ╲            │
           prop1  prop2       prop3
                              │
                            objC ←─── 可达！（通过 objB → prop3）

              objD ←─── 不可达！没有任何引用链能从 Root 到达它
```

```javascript
// 可达的示例
let global = '全局变量'       // 根（全局变量）直接可达

function foo() {
  let local = { x: 1 }       // 根（调用栈上的局部变量）可达
  return local
}

const result = foo()          // result 引用了 local 指向的对象 → 可达

// 不可达的示例
let obj = { data: '一些数据' }
obj = null                    // 原来的 { data: '一些数据' } 不再可达 → 将被 GC 回收
```

### 引用计数 vs 可达性

| 方法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **引用计数** | 计算每个对象被引用的次数，为 0 时回收 | 即时回收 | 无法处理**循环引用** |
| **可达性分析**（标记-清除） | 从根出发遍历，标记所有可达对象，未标记的回收 | 解决循环引用 | 需要 STW（Stop-The-World）暂停 |

```javascript
// 引用计数无法处理的循环引用问题
function createCycle() {
  let a = {}
  let b = {}

  a.ref = b  // a 引用 b，b 的引用计数 = 1
  b.ref = a  // b 引用 a，a 的引用计数 = 1

  // 函数执行完毕，a、b 离开作用域
  // 但 a、b 的引用计数都不为 0（互相引用）
  // 引用计数法无法回收 → 内存泄漏！
}

// 标记-清除（可达性分析）可以正确处理
// 因为 createCycle 执行完后，a、b 都无法从根到达 → 被回收 ✅
```

> **早期浏览器**（如 IE6/7）的 JavaScript 引擎使用引用计数，容易因循环引用导致内存泄漏。**现代浏览器**全部采用基于可达性的标记-清除算法。

## 垃圾回收算法

### 1. 标记-清除（Mark-and-Sweep）

这是**现代 JavaScript 引擎的标准算法**，也是 V8、SpiderMonkey、JavaScriptCore 共同采用的基础。

#### 执行过程

```
阶段一：标记（Mark）                  阶段二：清除（Sweep）

   从 Root 开始遍历                    扫描堆内存
   所有可达对象打上标记                   ┌─────────────────┐
                                      │ ✓ 已标记 → 保留   │
   Root                               │ ✗ 未标记 → 回收   │
    │                                 └─────────────────┘
    ├── objA  ✓
    │    ├── prop1  ✓
    │    └── prop2  ✓
    ├── objB  ✓
    └── objC  ✗  ← 不可达，将被清除
```

```javascript
// 标记-清除示例
function process() {
  const reachable = { id: 1 }   // 从调用栈可达
  const unreachable = { id: 2 } // 从调用栈可达

  return reachable
  // 函数返回后：
  //   reachable 被 return 引用 → 仍可达（被调用者持有）
  //   unreachable 不再被引用 → 不可达 → GC 回收
}

const obj = process()
// { id: 1 } 存活，{ id: 2 } 被回收
```

#### 优点

- 解决了循环引用问题
- 算法实现相对简单
- 是后续所有高级 GC 算法的基础

#### 缺点

- **内存碎片化**：清除后内存中存在不连续的空闲块
- **STW 暂停**：标记和清除期间需要暂停 JavaScript 执行（会造成卡顿）

### 2. 标记-整理（Mark-Compact）

在标记-清除的基础上增加了**整理（压缩）**步骤，解决内存碎片问题。

```
标记 → 清除 → 整理（移动存活对象，使其紧密排列）

整理前（碎片化）：                  整理后（连续）：
┌──────────────────────┐          ┌──────────────────────┐
│ ✓ │ ✗ │ ✓ │ ✗ │ ✓ │ ✗ │  →    │ ✓ │ ✓ │ ✓ │     空闲    │
└──────────────────────┘          └──────────────────────┘
```

- **优点**：消除内存碎片，提高内存分配效率
- **缺点**：移动对象需要更新所有引用，开销较大

### 3. 分代回收（Generational Collection）

详见 [V8 引擎的垃圾回收策略](#v8-引擎的垃圾回收策略) 章节。

### 4. 增量标记（Incremental Marking）

将一次完整的标记任务**拆分成多个小步骤**，穿插在 JavaScript 执行之间，避免长时间暂停。

```
传统标记（长时间暂停）：
├── JS 执行 ──────┣━━━━━━━ 标记（暂停）━━━━━━━┫── JS 执行 ──→

增量标记（拆分成小步骤）：
├── JS ──┣━标记━┫── JS ──┣━标记━┫── JS ──┣━标记━┫── JS ──→
```

### 5. 并发回收（Concurrent GC）

GC 的部分工作在**后台线程**完成，不阻塞主线程的 JavaScript 执行。

```
主线程：    ├── JS 执行 ──────────────────────── JS 执行 ──→
GC 线程：   ┣━━━━━━━━ 后台标记/清除 ━━━━━━━━━━━━┫
```

## V8 引擎的垃圾回收策略

Chrome 和 Node.js 使用的 V8 引擎采用**分代回收**策略，将堆内存分为**新生代**和**老生代**，分别使用不同的算法。

### 堆内存整体结构

```
┌──────────────────────────────────────────────────────────┐
│                      V8 堆内存                            │
│                                                          │
│  ┌────────────────────┐    ┌─────────────────────────┐  │
│  │    新生代（Young）   │    │     老生代（Old）         │  │
│  │                    │    │                         │  │
│  │  ┌──────┬──────┐  │    │  ┌─────────────────────┐│  │
│  │  │From  │ To   │  │    │  │  老生代对象          ││  │
│  │  │(活动) │(空闲) │  │    │  │  (长期存活的对象)    ││  │
│  │  └──────┴──────┘  │    │  │                     ││  │
│  │                    │    │  │  采用标记-清除/整理   ││  │
│  │  容量较小（1~8MB） │    │  └─────────────────────┘│  │
│  │  采用 Scavenge 算法│    │  容量较大（数百 MB+）    │  │
│  └────────────────────┘    └─────────────────────────┘  │
│                                                          │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  大对象区（Large Object Space）                      │ │
│  │  存放体积较大的对象，直接分配在此，不经过新生代        │ │
│  └─────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

### 新生代回收（Scavenge 算法）

新生代采用 **Cheney 算法**（一种 Scavenge 算法），使用**半空间复制**策略。

#### 核心思想

将新生代内存分为**两个等大的半空间**（Semi-space）：

| 区域 | 用途 |
|------|------|
| **From 空间**（活动空间） | 新对象分配在这里 |
| **To 空间**（空闲空间） | GC 时作为复制目标 |

#### 回收流程

```
步骤一：对象分配在 From 空间

┌────────────────────────────┐
│  From 空间（活动）          │    To 空间（空闲）
│  ┌───┬───┬───┬───┬───┐   │    ┌──────────────────┐
│  │ A │ B │ C │ D │ E │   │    │                  │
│  └───┴───┴───┴───┴───┘   │    └──────────────────┘
└────────────────────────────┘

步骤二：GC 触发，标记可达对象，复制到 To 空间

  From 空间                      To 空间
  ┌───┬───┬───┬───┬───┐       ┌───┬───┬───┐
  │ A │ B │✗C│ D │✗E│ ──→   │ A │ B │ D │     （C、E 不可达，被丢弃）
  └───┴───┴───┴───┴───┘       └───┴───┴───┘

步骤三：交换 From 和 To 角色

  新 From 空间（活动）           新 To 空间（空闲）
  ┌───┬───┬───┐                ┌──────────────────┐
  │ A │ B │ D │                │  （旧 From 被清空）│
  └───┴───┴───┘                └──────────────────┘
```

#### 对象晋升（Promotion）

满足以下条件之一的对象会被**移动到老生代**：

| 晋升条件 | 说明 |
|---------|------|
| **经历过一次 Scavenge 回收** | 已经存活过一轮 GC，说明生命周期较长 |
| **To 空间使用率超过 25%** | 为了保证后续分配有足够空间 |

```
新生代（From）  ──晋升──→  老生代
  短命对象                 长命对象
```

#### Scavenge 的特点

- **速度快**：只处理少量存活对象，大部分新对象都是"朝生夕死"
- **无碎片**：复制算法天然保证内存连续
- **牺牲空间**：只能使用一半的新生代内存（用空间换时间）

### 老生代回收

老生代存放生命周期较长的对象，使用**标记-清除**和**标记-整理**的组合策略。

#### 三种回收方式

| 方式 | 说明 | 时机 |
|------|------|------|
| **标记-清除** | 标记可达对象，清除不可达对象 | 常规回收 |
| **标记-整理** | 在清除后整理内存碎片 | 空间碎片较多时 |
| **增量标记** | 将标记阶段拆分为小步骤穿插执行 | 避免长时间暂停 |

#### 老生代回收流程

```
1. 标记阶段（Marking）
   ├── 从根集合开始
   ├── 递归遍历所有可达对象
   ├── 每个对象标记为"存活"
   └── 增量执行，穿插在 JS 主线程中

2. 清除阶段（Sweeping）
   ├── 遍历堆内存
   ├── 未标记的对象被视为垃圾
   └── 回收其内存空间

3. 整理阶段（Compaction，按需执行）
   ├── 将存活对象移动到内存一端
   ├── 释放连续的空闲空间
   └── 消除内存碎片
```

### 全流程总结

```
新对象分配
    │
    ▼
┌─────────────┐
│  新生代 From  │
│  空间已满？    │──── 否 ──→ 继续分配
└──────┬──────┘
       │ 是（触发 Scavenge GC）
       ▼
┌──────────────────┐
│  Scavenge 回收    │
│  复制存活对象到 To │
└──────┬───────────┘
       │
       ├── 存活对象满足晋升条件？
       │       │ 是
       │       ▼
       │  ┌─────────────┐
       │  │  晋升到老生代  │
       │  └──────┬──────┘
       │         │
       │         ▼
       │  ┌──────────────────┐
       │  │  老生代空间不足？  │──── 否 ──→ 继续
       │  └──────┬───────────┘
       │         │ 是（触发 Major GC）
       │         ▼
       │  ┌──────────────────────┐
       │  │  标记-清除 / 标记-整理  │
       │  └──────────────────────┘
       │
       └── 不满足晋升条件 → 留在新生代（To 变为新的 From）
```

### V8 GC 的优化技术

V8 引擎使用了多种优化技术来减少 GC 对性能的影响：

| 技术 | 说明 |
|------|------|
| **增量标记（Incremental Marking）** | 将标记阶段拆分成小步骤，避免长时间暂停 |
| **并发标记（Concurrent Marking）** | 标记工作在后台线程执行，不阻塞主线程 |
| **并发清除（Concurrent Sweeping）** | 清除工作在后台线程执行 |
| **并行标记（Parallel Marking）** | 多个辅助线程同时进行标记 |
| **写屏障（Write Barrier）** | 追踪对象引用变化，保证增量标记的正确性 |

```
主线程：    ├── JS ──┣┫── JS ──┣┫── JS ──┣┫── JS ──→
辅助线程1：                  ┣━━标记━━━━━━━━━━━┫
辅助线程2：                  ┣━━标记━━━━━━━━━━━┫
写屏障：           ↑ 捕获引用变更，通知 GC 线程
```

## 内存泄漏

内存泄漏是指**不再需要的内存没有被释放**，导致可用内存持续减少。JavaScript 的 GC 只回收不可达的对象，如果对象仍然被可达的引用持有，就不会被回收。

### 1. 意外的全局变量

```javascript
// ❌ 忘记声明变量 → 变成全局变量（window 的属性），永远不会被回收
function foo() {
  bar = '意外的全局变量'    // 没有 var/let/const，自动挂到全局
  this.baz = '也是全局变量' // this 指向 window（非严格模式）
}
foo()

// ✅ 修复：使用严格模式 + 始终声明变量
'use strict'
function foo() {
  const bar = '局部变量'  // 明确声明
}
```

### 2. 被遗忘的定时器

```javascript
// ❌ 定时器引用了外部数据，永远不会被回收
function startPolling() {
  const hugeData = loadHugeData()

  setInterval(() => {
    // hugeData 被闭包引用，永远不会被回收
    process(hugeData)
  }, 1000)
}

// ✅ 修复：保存定时器 ID，在不需要时清除
function startPolling() {
  const hugeData = loadHugeData()
  const timerId = setInterval(() => {
    process(hugeData)
  }, 1000)

  // 在适当时机清除
  return () => clearInterval(timerId)
}

const stopPolling = startPolling()
// 不再需要时
stopPolling()
```

### 3. 被遗忘的事件监听器

```javascript
// ❌ 挂载后从不移除
function setupListener() {
  const element = document.getElementById('button')
  const data = { /* 大量数据 */ }

  element.addEventListener('click', () => {
    console.log(data) // data 被闭包持有
  })
  // 即使 element 从 DOM 中移除，事件监听器仍持有引用
}

// ✅ 修复：适时移除监听器
function setupListener() {
  const element = document.getElementById('button')
  const data = { /* 大量数据 */ }

  const handler = () => {
    console.log(data)
  }

  element.addEventListener('click', handler)

  // 提供清理方法
  return () => element.removeEventListener('click', handler)
}
```

### 4. 闭包引用

```javascript
// ❌ 闭包持有不需要的变量
function createClosure() {
  const hugeArray = new Array(1000000).fill('*') // 1M 个元素
  const name = 'Alice'

  return function () {
    // 只用到了 name，但整个闭包作用域包含 hugeArray
    // hugeArray 不会被回收
    return name
  }
}

// ✅ 修复：只保留需要的值
function createClosure() {
  const hugeArray = new Array(1000000).fill('*')
  const name = 'Alice'

  // 提前释放不需要的引用
  // hugeArray = null // 但如果在闭包外层，仍可能被捕获

  // 更好的方式：将需要的值单独提取
  const nameOnly = name
  return function () {
    return nameOnly // 闭包只捕获 nameOnly
  }
}
```

### 5. 脱离 DOM 的引用

```javascript
// ❌ JS 变量引用了已从 DOM 树中移除的元素
const elements = {
  button: document.getElementById('myButton'),
  image: document.getElementById('myImage')
}

function removeButton() {
  document.body.removeChild(elements.button)
  // elements.button 仍然引用着这个 DOM 节点
  // DOM 节点不会被 GC 回收
}

// ✅ 修复：移除 DOM 后同时清除 JS 引用
function removeButton() {
  document.body.removeChild(elements.button)
  elements.button = null // 解除引用
}
```

### 6. 未清理的 Map / Set 引用

```javascript
// ❌ 作为缓存使用，但从不清理
const cache = new Map()

function process(id) {
  if (!cache.has(id)) {
    const result = expensiveComputation(id)
    cache.set(id, result) // 只增不减
  }
  return cache.get(id)
}

// ✅ 修复方式 1：使用 WeakMap（键是对象时）
const cache = new WeakMap()

function process(obj) {
  if (!cache.has(obj)) {
    const result = expensiveComputation(obj)
    cache.set(obj, result) // 当 obj 被回收时，缓存条目自动消失
  }
  return cache.get(obj)
}

// ✅ 修复方式 2：手动实现 LRU 缓存，限制大小
class LRUCache {
  constructor(maxSize = 100) {
    this.maxSize = maxSize
    this.cache = new Map()
  }

  get(key) {
    if (!this.cache.has(key)) return undefined
    const value = this.cache.get(key)
    // 移到末尾（最近使用）
    this.cache.delete(key)
    this.cache.set(key, value)
    return value
  }

  set(key, value) {
    if (this.cache.has(key)) this.cache.delete(key)
    this.cache.set(key, value)
    // 超出容量，删除最老的（Map 按插入顺序）
    if (this.cache.size > this.maxSize) {
      const firstKey = this.cache.keys().next().value
      this.cache.delete(firstKey)
    }
  }
}
```

### 内存泄漏排查工具

| 工具 | 使用方式 | 适用场景 |
|------|---------|---------|
| **Chrome DevTools Memory 面板** | Performance Monitor → Memory → Take Heap Snapshot | 定位泄漏对象 |
| **Allocation Timeline** | Memory → Record Allocation Timeline | 观察内存分配趋势 |

```
判断泄漏的经验法则：

  JS Heap 曲线呈锯齿状上升 → ✅ 正常（GC 在回收，但有波动）
  JS Heap 曲线持续上升不回落 → ❌ 疑似泄漏
```

## WeakRef 与 FinalizationRegistry

ES2021 引入了两个与垃圾回收密切相关的新特性。

### WeakRef — 弱引用

`WeakRef` 创建对一个对象的**弱引用**，不会阻止 GC 回收该对象。

```javascript
// 强引用 — 阻止 GC 回收
let obj = { data: '重要数据' }
const strongRef = obj // strongRef 持有强引用
obj = null            // obj 解除了，但 strongRef 还在引用 → 对象不会被回收

// 弱引用 — 不阻止 GC 回收
let target = { data: '重要数据' }
const weakRef = new WeakRef(target)

// 通过 .deref() 获取原始对象
console.log(weakRef.deref()) // { data: '重要数据' }

target = null // 解除强引用
// GC 可能在某个时刻回收 target 指向的对象
// 之后调用:
// weakRef.deref() → undefined（对象已被回收）
```

#### 使用场景

```javascript
// 场景：缓存 — 不想缓存阻止对象的回收
class WeakCache {
  #cache = new Map()

  get(key) {
    const ref = this.#cache.get(key)
    if (!ref) return undefined

    const value = ref.deref()
    if (!value) {
      // 对象已被 GC 回收，清理缓存条目
      this.#cache.delete(key)
      return undefined
    }
    return value
  }

  set(key, value) {
    this.#cache.set(key, new WeakRef(value))
  }
}
```

### FinalizationRegistry — 回收回调

注册一个回调，在对象被 GC 回收时执行。

```javascript
const registry = new FinalizationRegistry((heldValue) => {
  console.log(`对象被回收了，关联值: ${heldValue}`)
  // 可以在这里做清理工作
})

let obj = { data: '临时数据' }
registry.register(obj, 'obj-001') // 注册 obj，关联值为 'obj-001'

obj = null
// 当 GC 回收该对象后，回调可能被触发
// 输出: "对象被回收了，关联值: obj-001"
```

#### 综合示例

```javascript
// 可被 GC 自动清理的缓存
class GCAwareCache {
  #cache = new Map()
  #registry = new FinalizationRegistry((key) => {
    // 对象被回收后，清理缓存条目
    this.#cache.delete(key)
  })

  set(key, value) {
    this.#cache.set(key, new WeakRef(value))
    this.#registry.register(value, key) // value 被回收时通知我们
  }

  get(key) {
    const ref = this.#cache.get(key)
    if (!ref) return undefined

    const value = ref.deref()
    if (!value) {
      this.#cache.delete(key)
      return undefined
    }
    return value
  }
}
```

> ⚠️ **重要提示**：`WeakRef` 和 `FinalizationRegistry` 是底层 API，回调的执行时机**不确定**（取决于 GC 的调度），不应依赖它们来实现核心业务逻辑。

## 内存优化实践

### 1. 及时解除引用

```javascript
function process() {
  const bigData = loadBigData()

  // 处理完毕后及时解除引用
  const result = transform(bigData)
  bigData = null // 允许 GC 尽早回收

  return result
}
```

### 2. 使用 WeakMap / WeakSet

```javascript
// ❌ 强引用 — 阻止回收
const metadata = new Map()
function addMeta(obj, meta) {
  metadata.set(obj, meta) // obj 被 Map 强引用，无法被回收
}

// ✅ 弱引用 — 允许回收
const metadata = new WeakMap()
function addMeta(obj, meta) {
  metadata.set(obj, meta) // obj 被 WeakMap 弱引用，不阻止回收
}
```

### 3. 避免创建不必要的闭包

```javascript
// ❌ 每次循环创建新的闭包
function setupAll() {
  for (let i = 0; i < 1000; i++) {
    document.getElementById(`btn-${i}`).addEventListener('click', () => {
      console.log(i) // 每个闭包捕获 i
    })
  }
}

// ✅ 使用事件委托，只创建一个处理函数
function setupAll() {
  document.addEventListener('click', (e) => {
    const id = e.target.id
    if (id.startsWith('btn-')) {
      console.log(id.replace('btn-', ''))
    }
  })
}
```

### 4. 复用对象和数组

```javascript
// ❌ 频繁创建新对象
function processItems(items) {
  return items.map(item => {
    return { result: item.value * 2 } // 每次创建新对象
  })
}

// ✅ 对象池模式（适用于高频创建/销毁的场景）
class ObjectPool {
  #pool = []
  #factory

  constructor(factory, initialSize = 10) {
    this.#factory = factory
    for (let i = 0; i < initialSize; i++) {
      this.#pool.push(factory())
    }
  }

  acquire() {
    return this.#pool.length > 0 ? this.#pool.pop() : this.#factory()
  }

  release(obj) {
    this.#pool.push(obj)
  }
}
```

### 5. 注意 DOM 操作的内存影响

```javascript
// ❌ 频繁操作 DOM 导致大量中间对象
function updateList(items) {
  const list = document.getElementById('list')
  for (const item of items) {
    const li = document.createElement('li')
    li.textContent = item.name
    list.appendChild(li) // 每次触发回流
  }
}

// ✅ 使用 DocumentFragment 批量操作
function updateList(items) {
  const list = document.getElementById('list')
  const fragment = document.createDocumentFragment()

  for (const item of items) {
    const li = document.createElement('li')
    li.textContent = item.name
    fragment.appendChild(li) // 先添加到 fragment，不触发回流
  }

  list.appendChild(fragment) // 一次性添加，只触发一次回流
}
```

## 常见误区

### 误区一：手动设置 `null` 等于立即回收

```javascript
let obj = { data: 'test' }
obj = null // ❌ 这并不会立即触发 GC

// ✅ 只是解除了引用，让对象变为"不可达"
//    实际的回收时机由引擎的 GC 调度器决定
//    可能在下次 GC 周期时回收，也可能延迟更久
```

### 误区二：GC 可以精确控制时机

```javascript
// ❌ JavaScript 没有提供触发 GC 的标准 API
// 虽然 V8（Node.js）提供了：
if (typeof globalThis.gc === 'function') {
  globalThis.gc() // 仅在 --expose-gc 启动时可用
}
// 但这只是调试工具，不应在生产环境依赖
```

### 误区三：`delete` 操作能帮助 GC

```javascript
// ❌ delete 操作会降低 V8 的隐藏类优化效果
const obj = { a: 1, b: 2, c: 3 }
delete obj.b // V8 会降低对此对象属性的访问速度

// ✅ 使用 null 替代 delete
const obj2 = { a: 1, b: 2, c: 3 }
obj2.b = null // 保留隐藏类结构，同时释放 b 的值
```

### 误区四：闭包一定导致内存泄漏

```javascript
// ❌ 误解：所有闭包都会泄漏
function outer() {
  const data = 'hello'
  return function inner() {
    return data // 闭包持有 data 的引用
  }
}
const fn = outer()

// ✅ 正确理解：
//    闭包持有引用是**正常行为**，不等于"泄漏"
//    只有当闭包持有**不再需要的数据**且**无法被释放**时，才是泄漏
//    上面的 fn 仍然需要 data，所以不是泄漏
```

### 误区五：现代浏览器不存在内存泄漏

```javascript
// ❌ 现代 GC 很强大，但仍然存在泄漏的可能
// 以下场景在 2024 年的 Chrome 中仍然会泄漏：

// 1. 未清理的 setInterval
const timer = setInterval(() => {
  const data = fetchData() // data 的引用被闭包持有
  process(data)
}, 1000)
// 忘记 clearInterval(timer) → 泄漏

// 2. 闭包中意外捕获大对象
function setup() {
  const hugeData = new ArrayBuffer(100 * 1024 * 1024) // 100MB
  const name = 'Alice'

  return function () {
    return name // 只用了 name，但 hugeData 也被闭包捕获
  }
}
// hugeData 无法被回收 → 泄漏
```

## 总结

| 层面 | 要点 |
|------|------|
| **基本原理** | GC 基于**可达性**判断，从根出发遍历引用链，回收不可达对象 |
| **核心算法** | 标记-清除（Mark-and-Sweep）是现代 JS 引擎的基础算法 |
| **V8 分代回收** | 新生代用 Scavenge（复制算法），老生代用标记-清除/整理 |
| **回收触发** | 新生代空间不足 → Scavenge；老生代空间不足 → Major GC |
| **优化技术** | 增量标记、并发回收、并行标记、写屏障 |
| **内存泄漏** | 全局变量、定时器、事件监听、闭包、DOM 引用、Map/Set |
| **弱引用** | WeakMap/WeakSet/WeakRef 不阻止 GC，适合缓存场景 |
| **最佳实践** | 及时解除引用、使用弱引用集合、事件委托、对象复用 |

### 记忆口诀

> **可达就存活，不可达就回收。新生代用复制，老生代用标记。**
