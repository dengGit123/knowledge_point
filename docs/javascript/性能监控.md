### 一、概述

**浏览器性能监控**，指的是用浏览器提供的一系列 API，**自动采集页面加载速度、运行时流畅度、用户交互响应等性能数据**，然后上报到日志平台，帮助开发者发现和定位性能瓶颈。

为什么要做性能监控？

- **加载慢、卡顿会直接流失用户**：研究显示，页面加载时间每增加 1 秒，转化率会显著下降。
- **本地测不出真实体验**：你的电脑很快、网络很好，但用户的设备可能很差，只有采集「真实用户数据（RUM）」才能反映真实情况。
- **性能是动态的**：随着代码迭代、用户量增长，性能会持续变化，必须长期监控而不是一次性测试。

从监控维度看，浏览器性能监控主要回答三类问题：

| 维度 | 回答的问题 | 关键指标 |
| --- | --- | --- |
| 加载性能 | 页面加载快不快？ | TTFB、FCP、LCP、Load |
| 交互响应 | 点击/输入响应快不快？ | INP（原 FID） |
| 视觉稳定 | 页面会不会"乱跳"？ | CLS |
| 运行时 | 动画卡不卡？内存泄不泄漏？ | FPS、Long Task、JS 堆内存 |

> 💡 **提示：** 浏览器性能 API 的底层都是 `Performance` 接口。理解了 `performance.now()` 和 `PerformanceObserver`，剩下的都是「换一种 entry 类型」而已。

### 二、核心基础：高精度时间戳

#### 1. 为什么不用 `Date.now()`

监控性能时，我们需要精确到毫秒甚至更小单位的时间。但 `Date.now()` 有两个致命缺点：

- **精度低**：只能精确到毫秒，且受系统时钟影响（系统时间被改了，数据就错了）。
- **不是单调递增**：依赖系统时钟，可能出现"时间倒流"。

```js
// ❌ 监控性能不要用 Date.now()
const start = Date.now();
doSomething();
const cost = Date.now() - start; // 精度差，且可能为负
```

#### 2. 用 `performance.now()`

`performance.now()` 返回一个**相对于"页面起始时间点"的、单调递增的高精度时间戳**，精度可达微秒级，是性能监控的基石。

```js
// ✅ 正确做法
const start = performance.now();
doSomething();
const cost = performance.now() - start; // 单位：毫秒，高精度
console.log(`耗时：${cost.toFixed(2)}ms`);
```

`performance.now()` 的起点是 `performance.timeOrigin`，它表示当前文档开始导航（navigation）的时间。任意时刻：

```
performance.now() ≈ Date.now() - performance.timeOrigin
```

> ⚠️ **注意（精度限制）：** 出于防范「Spectre 时序攻击」的安全考虑，在不满足安全上下文（非 HTTPS、跨域、未设置 `cross-origin-isolated` 头）的页面里，浏览器会把 `performance.now()` 的精度**强制降低到 1ms 甚至更粗**。做 RUM 监控完全够用，但如果你要做亚毫秒级的精确测量，需要配置跨源隔离。

### 三、`PerformanceObserver`：性能数据的"统一观察者"

#### 1. 为什么需要它

浏览器内部会自动产生大量性能记录，比如"首屏绘制时刻""某个图片加载完毕""发生了一次布局偏移"等。这些记录叫做**性能条目（PerformanceEntry）**。

获取这些条目有两种方式：

- **主动拉取**：`performance.getEntries()` —— 只能拿到"已经发生"的记录，且要不停轮询，容易漏。
- **被动观察**：`PerformanceObserver` —— **订阅某种类型的条目，一旦产生就自动回调通知你**，这是官方推荐的方式。

#### 2. 基本用法

```js
// 1. 创建观察者，传入回调函数
const observer = new PerformanceObserver((list) => {
  // list.getEntries() 返回本次新产生的所有条目
  list.getEntries().forEach((entry) => {
    console.log(entry.name, entry.startTime, entry.duration);
  });
});

// 2. 指定要观察的条目类型
observer.observe({ type: 'largest-contentful-paint', buffered: true });
```

#### 3. 关键配置项

| 选项 | 类型 | 说明 |
| --- | --- | --- |
| `type` | string | 要观察的条目类型，如 `'paint'`、`'layout-shift'`、`'largest-contentful-paint'` 等 |
| `buffered` | boolean | 是否在观察开始时，**回放性能缓冲区里已有的历史条目**（非常重要，见下方说明） |

> ⚠️ **注意（一定要加 `buffered: true`）：** 性能事件往往发生得比你的代码执行还早（比如 FCP 在脚本加载时就已经产生了）。如果不加 `buffered: true`，你会**错过那些已经发生的事件**，采集到空数据。

#### 4. 停止观察

```js
observer.disconnect(); // 不再需要时，断开观察，避免内存泄漏
```

> 💡 **提示：** `PerformanceObserver` 的设计思路和 `IntersectionObserver`、`MutationObserver` 一样——都是"告诉浏览器我关心什么，事件来了再叫我"，比手动轮询高效得多。

### 四、页面加载性能：Navigation Timing API

#### 1. 它能做什么

`PerformanceNavigationTiming` 记录了**当前页面从导航开始到加载完成的完整时间线**，包括 DNS 解析、TCP 连接、请求、响应、DOM 解析、事件触发等每一个阶段。

#### 2. 完整的时间线节点

```js
const [nav] = performance.getEntriesByType('navigation');
console.log(nav);
```

一个 `PerformanceNavigationTiming` 对象包含以下关键时间戳（单位都是毫秒，相对于 `timeOrigin`）：

| 属性 | 含义 |
| --- | --- |
| `startTime` | 导航开始，值为 0 |
| `redirectStart` / `redirectEnd` | 重定向阶段 |
| `fetchStart` | 开始拉取资源 |
| `domainLookupStart` / `domainLookupEnd` | **DNS 解析**阶段 |
| `connectStart` / `connectEnd` | **TCP 连接**阶段 |
| `secureConnectionStart` | TLS 握手开始 |
| `requestStart` | 请求已发出 |
| `responseStart` | 收到第一个字节（**TTFB**） |
| `responseEnd` | 响应体接收完毕 |
| `domInteractive` | DOM 可交互 |
| `domContentLoadedEventStart` / `End` | DOMContentLoaded 事件 |
| `domComplete` | DOM 解析完成 |
| `loadEventStart` / `End` | load 事件 |

#### 3. 计算各阶段耗时

阶段耗时 = 结束时间戳 − 开始时间戳：

```js
function getNavigationTiming() {
  const [nav] = performance.getEntriesByType('navigation');
  if (!nav) return null;

  return {
    // 重定向耗时
    redirect: nav.redirectEnd - nav.redirectStart,
    // DNS 解析耗时
    dns: nav.domainLookupEnd - nav.domainLookupStart,
    // TCP 连接耗时
    tcp: nav.connectEnd - nav.connectStart,
    // SSL 握手耗时（https 才有）
    ssl: nav.secureConnectionStart
      ? nav.connectEnd - nav.secureConnectionStart
      : 0,
    // 首字节时间（请求发出 → 收到第一个字节）
    ttfb: nav.responseStart - nav.requestStart,
    // 响应内容下载耗时
    contentDownload: nav.responseEnd - nav.responseStart,
    // DOM 解析耗时
    domParse: nav.domInteractive - nav.responseEnd,
    // 白屏时间（导航开始 → DOM 可交互）
    whiteScreen: nav.domInteractive - nav.startTime,
    // 完整加载耗时
    fullLoad: nav.loadEventEnd - nav.startTime,
  };
}

// 注意：要在 load 事件之后读取，否则 loadEventEnd 为 0
window.addEventListener('load', () => {
  // load 事件触发后，loadEventEnd 还没赋值，用 setTimeout 推迟一帧
  setTimeout(() => {
    console.table(getNavigationTiming());
  }, 0);
});
```

> ⚠️ **注意：** `loadEventEnd` 表示 load 事件处理函数**全部执行完**的时刻。在 `load` 回调里直接读它是 0，必须用 `setTimeout` 延迟到事件处理结束后再读。

#### 4. 几个高频引用的指标

| 指标 | 计算方式 | 含义 |
| --- | --- | --- |
| **TTFB**（首字节时间） | `responseStart - requestStart` | 服务器响应快慢，越快越好 |
| **白屏时间** | `domInteractive - startTime` | 用户看到页面前等了多久 |
| **首屏可交互** | `domInteractive` | DOM 可以响应交互 |
| **完全加载** | `loadEventEnd - startTime` | 整个页面加载完毕 |

### 五、资源加载性能：Resource Timing API

`PerformanceResourceTiming` 和 Navigation Timing 结构几乎一样，但它记录的是**每一个具体资源**（图片、脚本、样式表、接口请求等）的加载耗时。

```js
// 获取所有资源的加载记录
const resources = performance.getEntriesByType('resource');

resources.forEach((res) => {
  console.log(res.name);              // 资源 URL
  console.log(res.initiatorType);     // 发起类型：script / link / img / fetch ...
  console.log(res.duration);          // 总耗时
  console.log(res.transferSize);      // 实际传输大小（含压缩）
  console.log(res.encodedBodySize);   // 压缩后大小
  console.log(res.decodedBodySize);   // 解压后大小
});
```

典型用途——找出加载最慢的资源：

```js
// 找出耗时前 5 的资源
const slowest = performance
  .getEntriesByType('resource')
  .map((r) => ({ name: r.name, duration: r.duration, size: r.transferSize }))
  .sort((a, b) => b.duration - a.duration)
  .slice(0, 5);

console.table(slowest);
```

> ⚠️ **注意（跨域限制）：** 对于跨域资源，默认只能看到时间，看不到 `transferSize` 等详细字段（值为 0）。需要资源服务器返回 `Timing-Allow-Origin: *` 响应头，才能拿到完整数据。

### 六、Core Web Vitals：用户体验核心指标

这是 **Google 定义的一组"用户体验"核心指标**，直接影响搜索排名。理解它们是做前端性能监控的重中之重。

当前（2024 年 3 月起）的三大核心指标：

| 指标 | 全称 | 衡量什么 | 目标值 |
| --- | --- | --- | --- |
| **LCP** | Largest Contentful Paint | 加载性能（最大内容渲染时刻） | ≤ 2.5s |
| **INP** | Interaction to Next Paint | 交互响应（2024 年替代 FID） | ≤ 200ms |
| **CLS** | Cumulative Layout Shift | 视觉稳定性（布局偏移程度） | ≤ 0.1 |

下面逐个讲解如何采集。

#### 1. FP 与 FCP（首次绘制 / 首次内容绘制）

虽然不是"三大核心"，但它们是加载体验的早期信号：

- **FP（First Paint）**：浏览器把**任何内容**绘制到屏幕上的时刻（包括背景色）。
- **FCP（First Contentful Paint）**：浏览器绘制**第一个有意义的文字/图片/背景**的时刻。目标 ≤ 1.8s。

```js
new PerformanceObserver((list) => {
  list.getEntries().forEach((entry) => {
    // entry.name: 'first-paint' 或 'first-contentful-paint'
    // entry.startTime: 绘制发生的时刻（毫秒）
    console.log(entry.name, entry.startTime);
  });
}).observe({ type: 'paint', buffered: true });
```

#### 2. LCP（最大内容绘制）

LCP 记录**视口内最大那个可见元素（大图、标题、文字块）被完整渲染的时刻**，比传统的 Load 事件更贴近用户对"页面加载好了"的感知。

```js
new PerformanceObserver((list) => {
  const entries = list.getEntries();
  const lastEntry = entries[entries.length - 1]; // 取最后一次（LCP 会不断更新）
  console.log('LCP:', lastEntry.startTime, 'ms');
}).observe({ type: 'largest-contentful-paint', buffered: true });
```

> 💡 **提示：** LCP 在页面生命周期内会**多次触发**（每当出现更大的元素就更新），所以要取**最后一次**的值作为最终 LCP。`web-vitals` 库内部已经处理了这个逻辑。

#### 3. INP（Interaction to Next Paint，2024 替代 FID）

**INP 衡量用户与页面的所有交互（点击、按键、拖拽）的响应速度**，取页面整个生命周期内"接近最差"的那次交互耗时，反映整体响应性。

- 旧指标 **FID** 只衡量**第一次**交互的延迟，不能代表整体体验，已于 2024 年 3 月被 INP 取代。
- INP 通过监听 `event` 类型的性能条目来计算，每次交互的耗时 ≈ 该交互所有事件的 `processingEnd - startTime`。

```js
let maxInteraction = 0;

new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    // 单个事件的耗时（处理结束 - 开始）
    const duration = entry.processingEnd - entry.startTime;
    if (duration > maxInteraction) {
      maxInteraction = duration;
    }
  }
}).observe({ type: 'event', buffered: true });

// 页面隐藏时，把"接近最差"的交互耗时作为 INP 上报
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    console.log('INP（近似）:', maxInteraction, 'ms');
  }
});
```

> ⚠️ **注意：** 上面的手写版是简化版（取最大值）。官方 INP 算法是基于所有交互的"高百分位（约 P98）"，且涉及交互聚合逻辑。**生产环境强烈建议直接用 `web-vitals` 库**，见本章末尾。

#### 4. CLS（累计布局偏移）

CLS 衡量**页面内容"跳动"的严重程度**——比如你正要点按钮，结果图片加载完把按钮挤走了，点空了。值越小越稳定。

```js
let clsScore = 0;

new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    // hadRecentInput: true 表示偏移是用户输入（如点击）引起的，应忽略
    if (!entry.hadRecentInput) {
      clsScore += entry.value; // entry.value 是这次偏移的"影响分数"
    }
  }
}).observe({ type: 'layout-shift', buffered: true });
```

> ⚠️ **注意（不要简单累加）：** 早期 CLS 算法是把所有偏移累加，但 2020 年起官方改为**"会话窗口"算法**——把 5 秒内、间隔不超过 1 秒的连续偏移合并为一个会话，取最大会话的总分。简单累加会高估 CLS。同样，**生产环境用 `web-vitals` 库**才是准确的。

#### 5. 强烈推荐：用 `web-vitals` 库统一采集

手写每个指标很容易出错（LCP 取最后值、CLS 会话窗口、INP 百分位、上报时机……）。Google 官方提供了 [`web-vitals`](https://github.com/GoogleChrome/web-vitals) 库，**封装了所有正确逻辑**，生产环境应当直接用它。

安装：

```bash
npm install web-vitals
```

统一采集（v4+ API）：

```js
import { onCLS, onINP, onLCP, onFCP, onTTFB } from 'web-vitals';

// metric 对象包含：name(指标名)、value(数值)、rating(评分 good/needs-improvement/poor)、id 等
function report(metric) {
  // 上报到你的分析平台（见第十节）
  const body = JSON.stringify({
    name: metric.name,
    value: metric.value,
    rating: metric.rating,
    page: location.pathname,
  });
  navigator.sendBeacon('/api/perf', body);
}

onCLS(report);   // 布局稳定性
onINP(report);   // 交互响应
onLCP(report);   // 最大内容绘制
onFCP(report);   // 首次内容绘制
onTTFB(report);  // 首字节时间
```

> 💡 **提示（归因信息）：** 如果你还想知道"是哪个元素导致了 LCP 慢""是哪次交互导致 INP 高"，可以引入 `web-vitals/attribution`，它会附带 `attribution` 字段，包含具体的 DOM 元素、选择器等信息，非常便于定位问题。

```js
import { onLCP } from 'web-vitals/attribution';

onLCP((metric) => {
  console.log('慢的元素是：', metric.attribution.elementSelector);
});
```

### 七、用户自定义测量：User Timing API

前面都是浏览器自动产生的条目。如果你想知道**某段业务代码的执行耗时**（比如"数据请求 + 渲染"总共花了多久），用 User Timing API 手动打点。

#### 1. 标记与测量

```js
// 打一个"起点"标记
performance.mark('start-fetch');

fetch('/api/data')
  .then((r) => r.json())
  .then((data) => {
    // 打一个"终点"标记
    performance.mark('end-fetch');
    // 测量两个标记之间的耗时
    performance.measure('fetch-duration', 'start-fetch', 'end-fetch');

    // 读取测量结果
    const [measure] = performance.getEntriesByName('fetch-duration');
    console.log('请求耗时：', measure.duration, 'ms');
  });
```

#### 2. 清理标记

```js
performance.clearMarks('start-fetch'); // 清除指定标记
performance.clearMarks();              // 清除所有标记
performance.clearMeasures();           // 清除所有测量
```

### 八、长任务监控：Long Tasks API

**"长任务"是指主线程连续占用超过 50ms 的任务**。因为浏览器渲染和响应用户输入都在主线程上，长任务会让页面"卡住"——50ms 是用户能感知到延迟的临界值。

```js
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.log(
      `发现长任务！耗时：${entry.duration.toFixed(0)}ms`,
      `来源：${entry.attribution[0]?.name}`
    );
  }
}).observe({ type: 'longtask', buffered: true });
```

> ⚠️ **注意：** Long Tasks API 目前在 Firefox/Safari 支持有限。它只能告诉你"有长任务"，但定位长任务具体在执行什么代码比较困难。新一代的替代方案是 **`performance.measureUserAgentSpecificMemory`**（内存）和基于 `event` timing 的 INP 监控。

### 九、运行时性能监控

加载完成后，页面运行过程中的流畅度也需要监控。

#### 1. FPS（帧率）监控

页面流畅的标准是 **60 FPS**（每帧 16.6ms）。低于这个值用户就会感觉卡顿。浏览器没有直接提供 FPS API，但可以用 `requestAnimationFrame` 统计。

```js
let lastTime = performance.now();
let frames = 0;
let rafId;

function calculateFPS(now) {
  frames++;
  // 每经过 1 秒，统计一次
  if (now - lastTime >= 1000) {
    const fps = Math.round((frames * 1000) / (now - lastTime));
    console.log('当前 FPS：', fps);
    if (fps < 50) {
      console.warn('⚠️ 检测到卡顿，FPS 偏低');
    }
    frames = 0;
    lastTime = now;
  }
  rafId = requestAnimationFrame(calculateFPS);
}

// 开始监控
rafId = requestAnimationFrame(calculateFPS);

// 停止监控
// cancelAnimationFrame(rafId);
```

#### 2. JS 内存使用监控

`performance.memory`（仅 Chrome 支持，非标准）可以读取 JavaScript 堆内存使用情况：

```js
if (performance.memory) {
  setInterval(() => {
    const mem = performance.memory;
    console.log({
      jsHeapSizeLimit: (mem.jsHeapSizeLimit / 1048576).toFixed(1) + 'MB', // 堆大小上限
      totalJSHeapSize: (mem.totalJSHeapSize / 1048576).toFixed(1) + 'MB', // 已分配堆
      usedJSHeapSize: (mem.usedJSHeapSize / 1048576).toFixed(1) + 'MB',   // 实际使用
    });
  }, 5000);
}
```

> ⚠️ **注意：** `performance.memory` 是非标准 API，只能在 Chrome/Edge 用。如果你的应用内存持续增长不下降，大概率存在**内存泄漏**，详细原理见 [内存管理](./内存管理.md) 和 [垃圾回收机制](./垃圾回收机制.md)。

#### 3. 网络环境监控

`navigator.connection`（实验性 API）能获取用户的网络状况，用于做"弱网降级"：

```js
const conn = navigator.connection;
if (conn) {
  console.log('网络类型：', conn.effectiveType); // '4g' / '3g' / '2g' / 'slow-2g'
  console.log('下行带宽：', conn.downlink, 'Mbps');
  console.log('往返延迟：', conn.rtt, 'ms');
  console.log('省流模式：', conn.saveData); // 用户开了"流量节省"

  // 网络变化时通知
  conn.addEventListener('change', () => {
    console.log('网络环境变化：', conn.effectiveType);
  });
}
```

典型应用：在弱网下降低图片质量、禁用自动播放视频：

```js
if (navigator.connection?.effectiveType?.startsWith('2g')) {
  // 弱网环境，加载低清图
}
```

### 十、数据上报

采集到性能数据后，需要发送到服务器。性能数据上报有两个特殊要求：**不打扰用户**、**页面关闭时也能发出去**。

#### 1. `navigator.sendBeacon`（推荐用于页面卸载时）

`sendBeacon` 专门为"页面即将关闭"的场景设计，**浏览器保证即使在 unload 阶段也能把数据发出去**，且不会阻塞页面关闭。

```js
window.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') {
    const data = JSON.stringify({ lcp: lcpValue, path: location.pathname });
    // 异步、可靠、不阻塞
    navigator.sendBeacon('/api/perf', data);
  }
});
```

#### 2. `fetch` 的 `keepalive` 选项

`fetch` 加上 `keepalive: true` 也能达到类似效果，且支持自定义请求头：

```js
fetch('/api/perf', {
  method: 'POST',
  body: JSON.stringify(data),
  headers: { 'Content-Type': 'application/json' },
  keepalive: true, // 即使页面关闭也保证发出
}).catch(() => {});
```

> 💡 **提示：** 推荐用 `visibilitychange` + `hidden` 作为上报时机（而不是 `beforeunload`），因为移动端很多场景下 `beforeunload` 不可靠。`hidden` 表示用户切走了标签页/最小化了，这是上报"会话结束类指标"的最佳时机。

#### 3. 批量上报与采样

为避免每个指标都发一次请求造成请求风暴，应当**攒一批一起发**，并对数据采样：

```js
const queue = [];

function enqueue(metric) {
  queue.push(metric);
  if (queue.length >= 10) flush();
}

function flush() {
  if (!queue.length) return;
  const body = JSON.stringify(queue.splice(0));
  navigator.sendBeacon('/api/perf/batch', body);
}

// 页面隐藏时兜底上报
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'hidden') flush();
});
```

### 十一、完整监控 SDK 示例

把前面的内容整合成一个可直接使用的轻量性能监控模块：

```js
/**
 * 轻量性能监控 SDK
 * 用法：import { initPerfMonitor } from './perf'; initPerfMonitor({ endpoint: '/api/perf' });
 */
export function initPerfMonitor({ endpoint, sampleRate = 1 }) {
  // 采样：随机丢弃部分样本，减轻上报压力
  if (Math.random() > sampleRate) return;

  const queue = [];

  const report = (data) => {
    queue.push({ ...data, path: location.pathname, ts: Date.now() });
  };

  const flush = () => {
    if (!queue.length) return;
    const body = JSON.stringify(queue.splice(0));
    if (navigator.sendBeacon) {
      navigator.sendBeacon(endpoint, body);
    } else {
      fetch(endpoint, { method: 'POST', body, keepalive: true }).catch(() => {});
    }
  };

  // 1. 页面加载各阶段耗时
  window.addEventListener('load', () => {
    setTimeout(() => {
      const [nav] = performance.getEntriesByType('navigation');
      if (nav) {
        report({
          type: 'navigation',
          ttfb: Math.round(nav.responseStart - nav.requestStart),
          fullLoad: Math.round(nav.loadEventEnd - nav.startTime),
        });
      }
    }, 0);
  });

  // 2. 资源加载（只报慢资源）
  new PerformanceObserver((list) => {
    for (const e of list.getEntries()) {
      if (e.duration > 1000) {
        report({ type: 'slow-resource', url: e.name, duration: Math.round(e.duration) });
      }
    }
  }).observe({ type: 'resource', buffered: true });

  // 3. Core Web Vitals
  new PerformanceObserver((list) => {
    const entries = list.getEntries();
    report({ type: 'lcp', value: Math.round(entries[entries.length - 1].startTime) });
  }).observe({ type: 'largest-contentful-paint', buffered: true });

  let clsScore = 0;
  new PerformanceObserver((list) => {
    for (const e of list.getEntries()) {
      if (!e.hadRecentInput) clsScore += e.value;
    }
  }).observe({ type: 'layout-shift', buffered: true });

  let maxInp = 0;
  new PerformanceObserver((list) => {
    for (const e of list.getEntries()) {
      maxInp = Math.max(maxInp, e.processingEnd - e.startTime);
    }
  }).observe({ type: 'event', buffered: true });

  // 4. 长任务
  new PerformanceObserver((list) => {
    for (const e of list.getEntries()) {
      report({ type: 'longtask', duration: Math.round(e.duration) });
    }
  }).observe({ type: 'longtask', buffered: true });

  // 5. 页面隐藏时上报 CLS / INP / 队列
  document.addEventListener('visibilitychange', () => {
    if (document.visibilityState === 'hidden') {
      if (clsScore > 0) report({ type: 'cls', value: +clsScore.toFixed(3) });
      if (maxInp > 0) report({ type: 'inp', value: Math.round(maxInp) });
      flush();
    }
  });

  // 6. 定时批量上报（兜底）
  setInterval(flush, 30000);
}
```

> 💡 **提示：** 上面是教学版的精简实现（CLS/INP 用了简化算法）。**真实项目请直接用 `web-vitals` 库**替换 Core Web Vitals 部分，它在边界情况处理上经过大量验证。

### 十二、注意事项与最佳实践

#### 1. 采集时机

- **加载类指标（LCP、TTFB、Load）**：在页面隐藏（`visibilitychange → hidden`）时上报最终值。
- **交互类指标（CLS、INP）**：同样在页面隐藏时上报，因为它们在用户离开前都可能在变化。
- **避免用 `unload`**：移动端不可靠，且会破坏 bfcache（前进后退缓存），统一用 `visibilitychange`。

#### 2. 常见陷阱

| 陷阱 | 说明 | 解决 |
| --- | --- | --- |
| 漏掉 `buffered: true` | 错过已发生的事件，采集到空 | 观察历史条目时务必加 `buffered: true` |
| `loadEventEnd` 为 0 | 在 load 回调里直接读 | 用 `setTimeout` 推迟一帧 |
| CLS 简单累加 | 高估偏移值 | 用 `web-vitals` 的会话窗口算法 |
| INP 取最大值 | 不够准确 | 用 `web-vitals` 的高百分位算法 |
| 跨域资源无大小 | `transferSize` 为 0 | 资源加 `Timing-Allow-Origin` 响应头 |
| 精度被限制 | 非 HTTPS 下 `performance.now()` 精度降低 | 部署 HTTPS，必要时配置跨源隔离 |

#### 3. 最佳实践

- **先有指标，再有优化**：明确监控 LCP/INP/CLS，再针对差项优化。
- **采集真实用户数据（RUM）**：实验室数据（Lighthouse）≠ 真实用户体验，两者结合。
- **关注 P75 而非平均值**：Google 用第 75 百分位评判，平均值会被少数极值带偏。
- **数据要能定位问题**：光知道"LCP 慢"没用，要带上"是哪个元素慢"（用 attribution）。

### 十三、相关概念对比

| API / 方案 | 监控对象 | 典型用途 |
| --- | --- | --- |
| `performance.now()` | 任意代码段耗时 | 手动打点测耗时 |
| `PerformanceObserver` | 各类性能条目 | 订阅式采集所有指标 |
| Navigation Timing | 当前页面加载 | 分析加载各阶段瓶颈 |
| Resource Timing | 每个资源 | 找出慢资源、大资源 |
| `web-vitals` 库 | Core Web Vitals | 生产环境采集 RUM 指标 |
| Long Tasks API | 主线程长任务 | 定位卡顿原因 |
| Lighthouse | 实验室综合评分 | 上线前本地评估 |

---

### 十四、总结

- **性能监控的基石**是 `performance.now()`（高精度计时）和 `PerformanceObserver`（订阅式采集）。
- **加载性能**用 Navigation Timing 分析各阶段；**资源性能**用 Resource Timing 找慢资源。
- **Core Web Vitals（LCP / INP / CLS）** 是用户体验的核心，生产环境用 `web-vitals` 库采集最省心最准确。
- **运行时性能**关注 FPS、长任务、内存、网络环境。
- **数据上报**用 `sendBeacon` 或 `fetch + keepalive`，在 `visibilitychange → hidden` 时上报，移动端最可靠。
- 记住两个关键细节：观察历史事件要加 `buffered: true`，读 `loadEventEnd` 要用 `setTimeout` 推迟。
