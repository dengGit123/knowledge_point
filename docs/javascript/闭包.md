#### 闭包是一个函数能够‌记住并访问‌其词法作用域中的变量，即使该函数在其作用域之外执行。简单来说，闭包就是函数与其所声明的词法环境的结合。

### 1.闭包的形成机制
* 需要同时满足三个条件
* 当内部函数被返回或传递到外部时，会携带定义时的作用域链
1. 函数嵌套： 一个函数内部定义`另一个函数`
2. 内部函数引用外部函数的变量： 内部函数使用了`外部函数的局部变量`
3. 内部函数被导出到外部作用域： 内部函数被`返回`、`传递到外部作用域`或作为`回调函数`使用

```js
function outer() {
    let outerVar = "我是外部变量";
    function inner() {
        console.log(outerVar);
    }
    return inner;
}
const closureFunc = outer();
closureFunc(); // 输出:我是外部变量
```
#### `inner`函数形成了一个闭包，因为它持续持有对外部变量`outerVar`的引用，即使`outer`函数已经执行完毕

### 2.闭包与作用域链的关系
* 函数的作用域在<strong>‌定义时</strong>就已确定，而不是在执行时
* 当函数被创建时，它会保存对其外层作用域的引用，形成一条作用域链
* 在正常情况下，函数执行完毕后，其局部变量会被垃圾回收机制回收。但当这些变量被闭包引用时，它们就不会被回收，因为闭包保持着对这些变量的引用

### 3.闭包的实际应用
##### 计数器示例展示了闭包如何保持状态：
```js
function createCounter() {
    let count = 0;
    return function() {
        count++;
        return count;
    };
}
const counter = createCounter();
console.log(counter()); // 1
console.log(counter()); // 2
```
#### 即使`createCounter`函数已经执行完毕，返回的匿名函数仍然能够访问count变量，因为闭包保留了对其外部作用域的引用

### 4.如何避免闭包导致的内存泄漏？
#### 理解问题本质
* 闭包导致内存泄漏的本质，是它保留了对外部作用域变量的引用，使得这些变量无法被垃圾回收
* 现代浏览器的垃圾回收主要采用“标记-清除”算法，只要对象从根（如全局window对象）出发不可达，就会被回收。如果闭包长期持有对某些变量的引用（例如作为回调或在事件监听器中），即使外部函数已执行完毕，这些变量也不会被垃圾收集

#### 核心解决方法
1. <strong>及时解除事件监听和定时器</strong>
* 闭包常被用于事件回调或定时器函数中，若这些闭包未被正确移除，会长期持有对外部变量的引用
###### 解决方案‌：在不需要时主动移除事件监听或清除定时器。
```js
function setup() {
    const data = { value: "需要被释放的数据" };
    const handleClick = () => { console.log(data.value); };
    document.addEventListener('click', handleClick);
    // 提供清理方法：解除引用
    return () => {
        document.removeEventListener('click', handleClick);
    };
}
const cleanup = setup();
// 当不再需要时（如组件卸载），调用清理方法
cleanup(); // 此时handleClick闭包及data可被GC回收
```
2. #### 避免在循环中创建闭包或及时释放引用
<p>循环中频繁创建闭包（如为多个DOM元素绑定事件）会产生大量无法回收的引用</p>
<strong>解决方案‌：</strong>
* 改用事件委托减少闭包数量
* 循环结束后手动清除闭包引用

3. #### 手动断开引用
<p>对于不再需要的闭包，可以主动将其设置为null，从而打破引用链。一旦将闭包引用设置为null，如果没有其他引用指向相关变量，JavaScript就能对其进行垃圾回收
</p>
