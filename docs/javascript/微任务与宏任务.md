# 微任务与宏任务（Microtask & Macrotask）

## 一、概述

JavaScript 是**单线程**语言，同一时刻只能执行一段代码。但实际开发中充满异步操作（定时器、网络请求、用户事件）。为了高效调度这些"不立即执行的任务"，JS 把它们分成了两类队列：**宏任务队列**和**微任务队列**，并由**事件循环（Event Loop）**统一调度。

搞懂微任务和宏任务，就能解释清楚一道道经典的"输出顺序"面试题，也能理解 `async/await`、`Vue.nextTick`、`setTimeout` 到底谁先谁后、为什么。

## 二、前置知识：单线程与异步

### 1. 为什么 JS 是单线程

JS 最初设计用于浏览器中操作 DOM。如果多线程同时操作同一个 DOM，会产生冲突（一个线程删除节点，另一个线程要修改它）。为避免复杂度，JS 被设计成**单线程**。

### 2. 单线程如何处理异步

单线程意味着所有任务要排队，前一个结束才执行后一个。如果某个任务很耗时（如网络请求），后面所有任务都得等——页面就卡死了。解决方法是**异步**：

- **同步任务**：在**主线程调用栈**上排队执行，前一个结束才执行下一个。
- **异步任务**：交给**宿主环境**（浏览器的 Web APIs）去等待，等"时机成熟"（如定时器到点、请求返回），把回调函数放入**任务队列**，等主线程空了再执行。

异步任务又细分为 **宏任务** 和 **微任务**，它们进入不同的队列，优先级不同。

## 三、事件循环（Event Loop）机制

事件循环就是"不停地从任务队列里取任务，放到调用栈执行"的循环过程。浏览器的每一次循环（tick）遵循以下流程：

```
        ┌──────────────────────────────────────┐
        │            调用栈 Call Stack           │  ← 执行同步代码
        └────────────────┬─────────────────────┘
                         │  栈空了
                         ▼
        ┌──────────────────────────────────────┐
        │   ① 取一个宏任务执行（script/setTimeout等）│
        └────────────────┬─────────────────────┘
                         │
                         ▼
        ┌──────────────────────────────────────┐
        │   ② 清空所有微任务（Promise.then 等）     │  ← 微任务必须全部执行完
        └────────────────┬─────────────────────┘
                         │
                         ▼
        ┌──────────────────────────────────────┐
        │   ③ 渲染（UI Render，按需，约 16.7ms/帧） │  ← requestAnimationFrame 回调在此前后
        └────────────────┬─────────────────────┘
                         │
                         └────────── 回到 ①，循环往复
```

### 核心规则（记住这三条）

1. **每次循环，只取一个宏任务**执行；执行完后，**清空所有微任务**（包括微任务执行过程中新产生的微任务，也要一并清空）。
2. **微任务优先级高于宏任务**：微任务在"当前宏任务结束后、下一个宏任务开始前"全部执行完。
3. **渲染时机**：清空微任务后、下一个宏任务前，浏览器会判断是否需要重绘（rAF 回调也在这个阶段附近执行）。

> 💡 **提示：** 最外层的整段 `<script>` 同步代码，本身就被视为**第一个宏任务**。所以执行顺序是：**同步代码（宏任务） → 微任务 → 下一个宏任务 → 微任务 → ……**

## 四、宏任务（MacroTask）

### 1. 定义与特点

- 宏任务由**宿主环境**（浏览器 / Node）发起，进入**宏任务队列**。
- 宏任务队列是**先进先出（FIFO）**，但每次事件循环只取**一个**。

### 2. 包含的 API

| 宏任务来源 | 示例 |
|------|------|
| **script 整体代码** | 最外层同步代码（第一个宏任务） |
| **定时器** | `setTimeout`、`setInterval` |
| **I/O 操作** | 网络请求回调、文件读写回调 |
| **UI 事件** | `click`、`scroll`、`keydown` 等事件回调 |
| **消息** | `postMessage`、`MessageChannel` |
| **Node 专属** | `setImmediate` |

```js
// 典型的宏任务：setTimeout 的回调
setTimeout(() => {
  console.log('我是一个宏任务')
}, 0)
```

> ⚠️ **注意：** `requestAnimationFrame`（rAF）比较特殊——它不是标准的宏任务，也不算微任务，它的回调在**浏览器渲染前**执行，时机介于"清空微任务后、下一个宏任务前"。下文会专门说明。

---

## 五、微任务（MicroTask）

### 1. 定义与特点

- 微任务由 **JS 引擎**自身发起，进入**微任务队列**。
- 每次宏任务结束后，**微任务队列会被一次性清空**（全部执行）。

### 2. 包含的 API

| 微任务来源 | 示例 |
|------|------|
| **Promise** | `Promise.then` / `catch` / `finally` 的回调 |
| **async/await** | `await` 之后的代码（本质是 Promise） |
| **MutationObserver** | 监听 DOM 变化的回调 |
| **queueMicrotask** | `queueMicrotask(fn)` 显式添加 |
| **Node 专属** | `process.nextTick`（优先级甚至高于普通微任务） |

```js
// 典型的微任务：Promise.then 的回调
Promise.resolve().then(() => {
  console.log('我是一个微任务')
})

// 显式添加微任务
queueMicrotask(() => {
  console.log('我也是微任务')
})
```

> 💡 **提示：** `new Promise((resolve, reject) => {...})` **构造函数内的代码是同步执行的**，只有 `.then` / `.catch` 的回调才是微任务。这是常见考点。

---

## 六、微任务 vs 宏任务 对比

| 对比维度 | 宏任务（MacroTask） | 微任务（MicroTask） |
|------|------|------|
| **发起方** | 宿主环境（浏览器/Node） | JS 引擎 |
| **队列** | 宏任务队列 | 微任务队列 |
| **每次循环执行数量** | 取 **1 个** | **全部**清空 |
| **优先级** | 较低（在微任务之后） | 较高（在下一个宏任务之前） |
| **执行时机** | 每轮事件循环取一个 | 当前宏任务结束后立刻全部执行 |
| **典型代表** | `setTimeout`、UI 事件、I/O | `Promise.then`、`queueMicrotask`、`await` 后 |
| **是否影响渲染** | 渲染发生在两个宏任务之间 | 渲染前会先清空微任务 |

## 七、执行顺序详解

### 1. 最简示例

```js
console.log('1')                        // 同步

setTimeout(() => {
  console.log('2')                      // 宏任务
}, 0)

Promise.resolve().then(() => {
  console.log('3')                      // 微任务
})

console.log('4')                        // 同步

// 输出顺序：1 → 4 → 3 → 2
```

**逐步解析：**

| 步骤 | 动作 | 输出 |
|------|------|------|
| 1 | 执行同步代码 | `1`、`4` |
| 2 | 遇到 `setTimeout`，回调进**宏任务队列** | —— |
| 3 | 遇到 `Promise.then`，回调进**微任务队列** | —— |
| 4 | 同步代码执行完（第一个宏任务结束），**清空所有微任务** | `3` |
| 5 | 取下一个**宏任务**（setTimeout 回调）执行 | `2` |

### 2. 微任务会被"清空到底"

微任务执行过程中如果又产生了新的微任务，这些新微任务也必须在当前微任务队列清空前执行完，**不会拖到下一轮**。

```js
Promise.resolve()
  .then(() => {
    console.log('A')
    return Promise.resolve()             // 返回新 Promise
  })
  .then(() => {
    console.log('B')
  })

Promise.resolve().then(() => {
  console.log('C')
})
// 输出：A → C → B
// 解析：A 是第一批微任务；A 返回的 Promise 的 then(B) 需要"等一个微任务周期"；
//      C 也在第一批；所以先 A、C，再 B。
```

> 💡 **提示：** 当 `.then` 的回调**返回一个 Promise** 时，规范会额外安排一次微任务（PromiseResolveThenableJob）来"展开"它，因此后续 `.then` 会被推迟——这就是上面 `B` 排在 `C` 之后的原因。返回普通值则没有这个额外步骤。

## 八、async/await 与事件循环

### 1. 本质

`async/await` 是 Promise 的语法糖。**`await` 之后的代码，等价于放到 `.then` 的回调里**，因此是**微任务**。

```js
async function foo() {
  console.log('1')          // 同步（进入函数立即执行）
  await bar()               // 遇到 await，交出执行权
  console.log('3')          // ← 这是 await 之后，属于微任务！
}
async function bar() {
  console.log('2')          // 同步
}

foo()
console.log('4')

// 输出：1 → 2 → 4 → 3
// 解析：1、2、4 是同步；await 之后的 3 是微任务，在同步代码之后执行。
```

### 2. ⚠️ await 的"新旧版本"差异（经典考点）

`await` 的执行时机在不同 V8 版本（Chrome 73 / Node 11 为分界线）**曾有过差异**，这是因为 V8 对 await 做了性能优化（减少一次微任务调度）。现代浏览器已统一，但老面试题的"标准答案"可能基于旧版本。

| 版本 | `await` 一个已 settled 的 Promise | 表现 |
|------|------|------|
| **新版（Chrome 73+ / Node 11+）** | 减少 1 次微任务调度 | await 后续更早执行 |
| **旧版（Node 10 及更早）** | 多一次微任务 | await 后续更晚执行 |

> 💡 **提示：** 结论记住一条即可——**现代浏览器中，`await` 之后的代码按微任务调度，与同一批次的 `Promise.then` 一起排队**。下文面试题以现代版本为准。

## 九、经典输出题解析

这是最经典的"综合题"，涵盖了同步代码、`setTimeout`、`async/await`、`Promise.then` 全部知识点：

```js
async function async1() {
  console.log('async1 start')
  await async2()
  console.log('async1 end')          // ← await 之后，微任务
}
async function async2() {
  console.log('async2')
}

console.log('script start')          // 同步

setTimeout(() => {
  console.log('setTimeout')          // 宏任务
}, 0)

async1()                             // 调用 async1

new Promise((resolve) => {
  console.log('promise')             // 构造函数内：同步
  resolve()
}).then(() => {
  console.log('promise then')        // ← 微任务
})

console.log('script end')            // 同步
```

**输出顺序（现代浏览器 Chrome 73+ / Node 11+）：**

```text
script start
async1 start
async2
promise
script end
async1 end
promise then
setTimeout
```

**逐步解析：**

| 阶段 | 执行内容 | 输出 |
|------|------|------|
| **同步阶段** | 执行整段 script（第一个宏任务） | `script start` |
| | 进入 `async1()`，执行到 `await` 前 | `async1 start` |
| | 执行 `await async2()`，即调用 `async2()` | `async2` |
| | `await` 暂停，把 `async1 end` 推入微任务队列，返回主流程 | —— |
| | 执行 `new Promise` 构造函数（同步） | `promise` |
| | `.then` 回调推入微任务队列 | —— |
| | 继续同步代码 | `script end` |
| **微任务阶段** | 清空微任务：先 `async1 end`，再 `promise then` | `async1 end`、`promise then` |
| **宏任务阶段** | 取下一个宏任务（setTimeout 回调） | `setTimeout` |

> 💡 **提示：** 关键在于识别哪些是"同步"、哪些是"微任务"、哪些是"宏任务"。`new Promise(fn)` 的 `fn` 同步执行；`.then` 是微任务；`await` 之后是微任务；`setTimeout` 是宏任务。

## 十、Vue.nextTick 的本质

在 Vue 中，修改响应式数据后，**DOM 不会立即更新**——Vue 会把更新放到**异步队列**中，在下一个"微任务"时机统一刷新（批量更新，提升性能）。所以"刚修改完数据，立刻读 DOM"拿到的还是旧值，必须用 `nextTick` 等待 DOM 更新完成。

```vue
<script setup>
import { ref, nextTick } from 'vue'

const count = ref(0)

async function increment() {
  count.value++
  console.log(document.getElementById('num').textContent)   // ❌ 旧值（DOM 还没更新）

  await nextTick()                                          // ✅ 等 DOM 更新完
  console.log(document.getElementById('num').textContent)   // ✅ 新值
}
</script>

<template>
  <button id="num" @click="increment">{{ count }}</button>
</template>
```

### Vue.nextTick 的实现与微任务

| 版本 | 内部实现 | 类型 |
|------|------|------|
| **Vue 3** | `Promise.resolve().then()` | **微任务** |
| Vue 2（降级链） | Promise → MutationObserver → setImmediate → setTimeout | 优先**微任务**，无 Promise 才降级宏任务 |

> 💡 **提示：** 因为 `nextTick` 用的是**微任务**（Promise），它会在当前同步代码后、下一个宏任务（如 `setTimeout`）**之前**执行。这就是"为什么 `nextTick` 比 `setTimeout(() => {}, 0)` 更早触发"。

## 十一、Node.js 事件循环（简介）

Node 的事件循环基于 **libuv**，与浏览器**不完全相同**，它分为多个**阶段（phases）**，循环往复：

```
   ┌───────────────────────────┐
┌─▶│         timers            │  setTimeout / setInterval 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │  系统级回调（如 TCP 错误）
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │  内部使用
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           poll             │  获取新 I/O 事件，执行 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           check            │  setImmediate 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │      close callbacks       │  close 事件回调
│  └─────────────┬─────────────┘
│                │  每个阶段切换前：清空微任务（process.nextTick 优先于 Promise）
└────────────────┘
```

### 关键差异（Node vs 浏览器）

| 对比维度 | 浏览器 | Node.js |
|------|------|------|
| **模型** | 宏任务 + 微任务循环 | 多阶段（timers/poll/check…）循环 |
| **微任务清空时机** | 每个宏任务后 | 每个阶段切换之间 |
| **特有微任务** | 无 | `process.nextTick`（优先级高于 Promise） |
| **特有宏任务** | 无 | `setImmediate` |
| **定时器宏任务**（Node 11+） | 每个回调后清空微任务 | 同样对齐（每个回调后清空微任务） |

> 💡 **提示：** Node 11+ 之后，`setTimeout`/`setInterval` 这类"定时器宏任务"的微任务处理已与浏览器**对齐**（每个回调后清空微任务）；但 `setImmediate`、I/O 回调等阶段级行为仍有差异。

### Node 中 `process.nextTick` 优先级最高

```js
process.nextTick(() => console.log('nextTick'))     // 优先级最高的微任务
Promise.resolve().then(() => console.log('promise'))
// Node 输出：nextTick → promise（nextTick 先于普通微任务）
```

## 十二、常见问题与注意事项

### 1. `new Promise(fn)` 的 `fn` 是同步的

```js
console.log('start')
new Promise((resolve) => {
  console.log('promise 内部')   // ✅ 同步执行（不是微任务）
  resolve()
}).then(() => {
  console.log('then')           // 微任务
})
console.log('end')
// 输出：start → promise 内部 → end → then
```

### 2. `setTimeout(fn, 0)` 不是真的 0ms

浏览器对 `setTimeout` 有**最小延迟限制**（嵌套调用时通常约 4ms），且要等当前宏任务和微任务都执行完才会触发。所以 `setTimeout(fn, 0)` 的实际触发时间通常 > 0ms，且一定**晚于同批次的微任务**。

### 3. `requestAnimationFrame` 不是标准宏任务

rAF 的回调在**浏览器准备重绘前**执行，时机在"清空微任务之后、下一个宏任务之前"。它常用于动画，确保回调与屏幕刷新同步（约 60fps / 16.7ms）。

```js
// 微任务先于 rAF，rAF 先于 setTimeout
Promise.resolve().then(() => console.log('微任务'))
requestAnimationFrame(() => console.log('rAF（渲染前）'))
setTimeout(() => console.log('宏任务'), 0)
// 通常输出：微任务 → rAF → 宏任务
```

> ⚠️ **注意：** rAF 的精确触发顺序依赖浏览器渲染时机，在标签页隐藏时 rAF 会被暂停（节能），而微任务/宏任务不受影响。

### 4. 微任务太多会"饿死"宏任务

如果微任务执行过程中**不断产生新的微任务**，微任务队列永远清不完，后续的宏任务（包括 UI 渲染、用户事件）就永远得不到执行——页面会卡死。所以**不要在微任务里无限递归地产生微任务**。

### 5. 不要用 `setTimeout(0)` 来"等 DOM 更新"

在 Vue 中等待 DOM 更新要用 `nextTick`（微任务），它比 `setTimeout(0)`（宏任务）**更早**触发，也更可靠。

## 十三、总结

| 知识点 | 一句话记忆 |
|------|------|
| **宏任务** | 宿主环境发起（`setTimeout`、UI 事件、I/O），每轮取一个 |
| **微任务** | 引擎发起（`Promise.then`、`await` 后、`queueMicrotask`），每次清空全部 |
| **执行顺序** | 同步代码 → 清空所有微任务 → 渲染 → 取一个宏任务 → 循环 |
| **微任务优先** | 微任务在下一个宏任务前**全部**执行完 |
| **await 后** | 是微任务（现代浏览器） |
| **nextTick** | Vue 用 `Promise.then` 实现，是**微任务**，早于 `setTimeout` |
| **rAF** | 渲染前后执行，非标准宏/微任务 |
| **Node 差异** | 多阶段循环，`process.nextTick` 优先级最高 |

**核心理解**：事件循环的精髓是**"一个宏任务 → 清空所有微任务 → 渲染"**的循环。宏任务是"粗粒度"的、由浏览器安排的外部事件（定时器、I/O、交互）；微任务是"细粒度"的、由 JS 自己产生的、优先级更高的内部任务（Promise、await）。记住"**微任务一定插队在所有宏任务之前执行完**"，再识别清楚每个 API 属于哪一类，就能破解几乎所有的执行顺序问题。
