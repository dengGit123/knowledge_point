# 词法作用域（Lexical Scope）

> 词法作用域是指**函数的作用域在函数定义时就已经确定**，而不是在函数调用时确定。它也被称为**静态作用域**（Static Scope）。JavaScript 采用的就是词法作用域。

## 核心概念

| 概念 | 说明 |
|------|------|
| **定义** | 作用域由**代码书写位置**决定，在**编译阶段**（词法分析阶段）就已确定 |
| **别称** | 静态作用域（Static Scope） |
| **核心特征** | 函数内部可以访问**定义时**所在作用域的变量，无论函数在哪里被调用 |
| **对立面** | 动态作用域（Dynamic Scope）— 作用域由**调用位置**决定 |

```
一句话总结：你在哪里定义，就决定你能访问谁，跟你被谁调用无关。
```

## 作用域的本质

在深入词法作用域之前，先理解 JavaScript 中**作用域的三种层次**：

```
┌─────────────────────────────────────────────────────┐
│  全局作用域（Global Scope）                          │
│  ┌───────────────────────────────────────────────┐  │
│  │  函数作用域（Function Scope）                  │  │
│  │  ┌─────────────────────────────────────────┐  │  │
│  │  │  块级作用域（Block Scope）               │  │  │
│  │  │  { let/const 声明的变量 }               │  │  │
│  │  └─────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### 各级作用域的创建方式

```javascript
// 全局作用域 — 最外层
var global = '全局变量'

function outer() {
  // 函数作用域 — function 声明创建
  var outerVar = '外层函数变量'

  function inner() {
    // 函数作用域 — 嵌套函数
    var innerVar = '内层函数变量'

    if (true) {
      // 块级作用域 — {} + let/const
      let blockVar = '块级变量'
      const blockConst = '块级常量'
    }
    // blockVar、blockConst 在这里不可访问
  }
}
```

## 词法作用域 vs 动态作用域

这是理解词法作用域最关键的一步——搞清楚它**不是什么**。

### 对比表

| 维度 | 词法作用域（JavaScript） | 动态作用域（如 Bash） |
|------|------------------------|---------------------|
| **确定时机** | 代码**定义时** | 函数**调用时** |
| **依赖关系** | 嵌套结构（词法位置） | 调用栈（运行时） |
| **可预测性** | 高 — 看代码就知道变量来源 | 低 — 取决于运行时的调用链 |
| **代表语言** | JavaScript、Python、Java | Bash、Perl（可选） |

### 代码对比

```javascript
// ===== JavaScript（词法作用域） =====
let value = '全局'

function foo() {
  console.log(value) // 输出什么？
}

function bar() {
  let value = 'bar 内部'
  foo() // foo 在 bar 中被调用
}

bar()  // 输出: "全局"
// ✅ foo 定义在全局，所以它访问全局的 value
//    与 foo 在哪里被调用（bar 内部）无关
```

```bash
# ===== Bash（动态作用域） =====
value="全局"

foo() {
  echo $value
}

bar() {
  local value="bar 内部"
  foo   # foo 在 bar 中被调用
}

bar    # 输出: "bar 内部"
# ✅ foo 在 bar 内被调用，所以它访问 bar 的 value
#    取决于调用链，而非定义位置
```

> **核心区别**：同样的代码结构，JavaScript 输出 `"全局"`，Bash 输出 `"bar 内部"`。这就是词法作用域 vs 动态作用域的根本差异。

## 作用域链

词法作用域通过**作用域链**（Scope Chain）实现变量查找。

### 什么是作用域链

当一个函数被定义时，引擎会保存一个**内部属性 `[[Environment]]`**，引用它**定义时**的外层词法环境。多个词法环境通过这个引用串联起来，就形成了作用域链。

```
查找规则：
  当前作用域 → 外层作用域 → 更外层作用域 → ... → 全局作用域
  找到就停止，找不到就继续往外找，直到全局作用域还找不到则抛出 ReferenceError
```

### 图示

```javascript
let a = '全局'

function outer() {
  let b = '外层'

  function middle() {
    let c = '中间层'

    function inner() {
      let d = '内层'
      console.log(a, b, c, d) // 可以访问所有外层变量
    }

    inner()
  }

  middle()
}
```

```
作用域链（从内到外）：

inner 的作用域链:
  inner(LexicalEnvironment) ──→ middle(LexicalEnvironment) ──→ outer(LexicalEnvironment) ──→ Global(LexicalEnvironment)

  查找变量 d → inner 作用域    ✅ 找到
  查找变量 c → inner → middle  ✅ 找到
  查找变量 b → inner → middle → outer ✅ 找到
  查找变量 a → inner → middle → outer → Global ✅ 找到
  查找变量 x → inner → middle → outer → Global ❌ ReferenceError
```

### 关键特征

```javascript
function outer() {
  let x = 10

  function inner() {
    // inner 定义在 outer 内部
    // 所以 inner 的作用域链是: inner → outer → Global
    console.log(x) // 10 — 沿作用域链向外查找，在 outer 中找到
  }

  inner()
}

// ⚠️ 即使把 inner 传到别处调用，作用域链不变
function run(fn) {
  let x = 999
  fn() // 仍然输出 10，不是 999
}

run(inner) // 10 — 因为 inner 的作用域链在定义时就已固定
```

## 词法作用域的查找规则

### 规则一：从内到外，逐层查找

```javascript
const name = '全局'

function outer() {
  const name = '外层'

  function inner() {
    const name = '内层'
    console.log(name) // "内层" — 先在当前作用域找，找到了就停止
  }

  inner()
}

outer()
```

### 规则二：只查找标识符，不关心对象属性

```javascript
const obj = { name: '对象属性' }

function foo() {
  console.log(name)  // ReferenceError! — 作用域查找的是标识符 "name"
  console.log(obj.name) // "对象属性" — obj 是标识符可以找到，.name 是属性访问
}
```

> 词法作用域只管**变量（标识符）的查找**，对象属性的访问是另一套机制（原型链）。

### 规则三：跨级查找，不会跳过外层

```javascript
let a = '全局 a'

function level1() {
  let a = 'L1 a'
  let b = 'L1 b'

  function level2() {
    let a = 'L2 a'

    function level3() {
      console.log(a) // "L2 a" — 向上找到最近的 level2 中的 a
      console.log(b) // "L1 b" — 跳过 level2（没有 b），在 level1 找到
    }

    level3()
  }

  level2()
}

level1()
```

## 遮蔽效应（Shadowing）

当内层作用域声明了与外层**同名**的变量时，内层变量会**遮蔽**外层变量。

```javascript
let message = '外层消息'

function shadow() {
  let message = '内层消息' // 遮蔽了外层的 message
  console.log(message)    // "内层消息"
}

shadow()
console.log(message)      // "外层消息" — 外层变量不受影响
```

### 遮蔽的几种形式

```javascript
// 1. 函数参数遮蔽外层变量
const x = '全局'
function foo(x) {       // 参数 x 遮蔽全局 x
  console.log(x)        // 取决于传入的值
}
foo('参数值')            // "参数值"

// 2. 块级变量遮蔽函数变量
function bar() {
  const name = '函数层'
  if (true) {
    const name = '块级层' // 遮蔽外层 name
    console.log(name)    // "块级层"
  }
  console.log(name)      // "函数层"
}

// 3. var 不会创建块级作用域，但 let/const 会
function baz() {
  var i = '函数层 i'
  for (let i = 0; i < 3; i++) { // let i 遮蔽外层 i
    // 这里的 i 是循环变量
  }
  console.log(i) // "函数层 i" — 外层 i 不受影响
}
```

### 无法直接访问被遮蔽的全局变量（但 window 可以）

```javascript
const color = '蓝色'

function paint() {
  const color = '红色' // 遮蔽了全局 color
  console.log(color)   // "红色"

  // 在浏览器中可以通过 window 访问全局变量
  // console.log(window.color) // "蓝色"（仅限 var 声明的全局变量）
}
```

## 与闭包的关系

词法作用域是闭包的**基础**。闭包之所以能够"记住"外层变量，正是因为词法作用域在**定义时**就绑定了外层环境。

```javascript
function createCounter() {
  let count = 0 // 这个变量属于 createCounter 的作用域

  // 返回的函数"记住"了 count 所在的作用域
  // 这就是闭包 — 函数 + 它的词法作用域
  return {
    increment() { return ++count },
    decrement() { return --count },
    getCount()  { return count }
  }
}

const counter = createCounter()
// createCounter 已经执行完毕，但 count 不会被回收
// 因为返回的函数仍然引用着 count 所在的词法作用域

counter.increment() // 1
counter.increment() // 2
counter.getCount()  // 2
```

### 闭包保留的是**引用**，不是快照

```javascript
function createFunctions() {
  const funcs = []
  let i

  for (i = 0; i < 3; i++) {
    funcs.push(function () {
      return i // 闭包保留的是对变量 i 的引用，不是 i 的快照值
    })
  }

  return funcs
}

const fns = createFunctions()
fns[0]() // 3 — 循环结束后 i = 3，所有函数共享同一个 i
fns[1]() // 3
fns[2]() // 3
```

> 这正是词法作用域的体现：每个函数在定义时绑定的是**同一个 `i` 变量的引用**，而不是当时的值。

**修复方式** — 使用 `let` 创建块级作用域：

```javascript
function createFunctions() {
  const funcs = []

  for (let i = 0; i < 3; i++) {  // let 在每次迭代创建新的作用域
    funcs.push(function () {
      return i // 每个函数绑定的是各自迭代中的 i
    })
  }

  return funcs
}

const fns = createFunctions()
fns[0]() // 0
fns[1]() // 1
fns[2]() // 2
```

## 欺骗词法作用域

JavaScript 提供了一些机制可以"绕过"词法作用域的规则，但**强烈不推荐使用**。

### 1. `eval` — 在运行时修改作用域

```javascript
function foo(str) {
  eval(str)           // 在 foo 的作用域内执行代码
  console.log(a, b)   // a 是 eval 动态注入的
}

var b = 2
foo('var a = 1')      // "1" "2" — eval 在 foo 内声明了 a
```

> `eval` 可以在运行时将字符串作为代码执行，**修改当前的词法作用域**。这打破了"作用域在定义时确定"的规则。

**问题**：引擎无法在编译阶段优化 `eval` 所在的函数，因为不知道它会修改什么变量，导致性能下降。

### 2. `with` — 在运行时创建作用域

```javascript
const obj = { x: 1, y: 2 }

function foo() {
  with (obj) {
    console.log(x) // 1 — 在 with 块中，obj 的属性被视为局部变量
    x = 10         // 修改了 obj.x
  }
  console.log(obj.x) // 10
}

foo()
```

**更危险的情况**：

```javascript
const obj = { x: 1 }

function foo() {
  with (obj) {
    y = 10 // obj 没有 y 属性 → 泄漏到全局作用域！
  }
}

foo()
console.log(y) // 10 — 全局变量被意外创建
```

> `with` 已在**严格模式**下被禁用。

### 严格模式禁止这些行为

```javascript
'use strict'

eval('var x = 1') // x 不会泄漏到外层作用域
// with (obj) {}  // SyntaxError: Strict mode code may not include a with statement
```

## 块级作用域与词法作用域

`let` 和 `const` 引入了块级作用域，这是 ES6 对词法作用域体系的**补充**，而非替代。

### `var` vs `let/const` 的作用域差异

```javascript
// var — 函数作用域，忽略块级边界
function varExample() {
  if (true) {
    var x = 1 // x 属于整个 varExample 函数作用域
  }
  console.log(x) // 1 — 可以访问，因为 var 不受 {} 限制
}

// let/const — 块级作用域
function letExample() {
  if (true) {
    let y = 1   // y 只属于这个 if 块
    const z = 2 // z 只属于这个 if 块
  }
  console.log(y) // ReferenceError: y is not defined
}
```

### 块级作用域同样是词法的

```javascript
{
  let outer = '外层块'

  {
    let inner = '内层块'
    console.log(outer) // "外层块" — 块级作用域同样遵循词法作用域规则
  }

  // console.log(inner) // ReferenceError
}
```

> 块级作用域没有改变词法作用域的本质——作用域仍然在**定义时**确定，只是多了一层**块级**的划分粒度。

### 循环中的块级作用域

```javascript
// 经典问题：var 在循环中
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100)
}
// 输出: 3, 3, 3 — 只有一个 i（函数作用域），循环结束后 i = 3

// 修复：let 在每次迭代创建独立作用域
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100)
}
// 输出: 0, 1, 2 — 每次迭代都有独立的 i
```

### `try/catch` 中的块级作用域

```javascript
try {
  throw new Error('测试')
} catch (err) {
  // err 只在 catch 块内有效（ES2019+）
  console.log(err.message) // "测试"
}
// console.log(err) // ReferenceError — err 不存在
```

## 词法作用域在引擎中的实现

### 编译阶段：确定作用域结构

JavaScript 引擎在**编译阶段**（准确说是解析阶段）会进行词法分析，构建作用域的结构：

```
源代码 → 词法分析（Tokenizer/Lexer） → 语法分析（Parser） → AST（抽象语法树）

在构建 AST 的过程中，引擎会：
  1. 遇到变量声明 → 在当前作用域中注册该变量
  2. 遇到函数声明 → 创建新的子作用域
  3. 遇到块级声明（let/const）→ 创建新的块级作用域
  4. 记录每个函数的 [[Environment]] 引用（指向定义时的外层作用域）
```

### 执行阶段：变量查找

```
引擎查找变量 foo 的过程：

1. 在当前 LexicalEnvironment 中查找 foo
   ├─ 找到 → 返回值 ✅
   └─ 没找到 → 继续

2. 通过 [[OuterEnv]] 引用跳到外层 LexicalEnvironment
   ├─ 找到 → 返回值 ✅
   └─ 没找到 → 继续

3. 重复步骤 2 直到全局作用域
   ├─ 全局作用域找到 → 返回值 ✅
   └─ 全局也没找到 → ReferenceError ❌
```

### 示例：引擎的视角

```javascript
let x = 10             // ① 全局作用域注册 x

function outer() {     // ② 全局作用域注册 outer，创建 outer 的函数作用域
  let y = 20           // ③ outer 作用域注册 y

  function inner() {   // ④ outer 作用域注册 inner，创建 inner 的函数作用域
    let z = 30         // ⑤ inner 作用域注册 z
    console.log(x + y + z) // ⑥ 查找 x、y、z
  }

  inner()              // ⑦ 调用 inner
}

outer()
```

```
步骤⑥的变量查找过程：

查找 z: inner 作用域           → 找到 (30) ✅
查找 y: inner → outer 作用域   → 找到 (20) ✅
查找 x: inner → outer → 全局   → 找到 (10) ✅
```

### LHS 与 RHS 查找

引擎在词法作用域中的查找分为两种：

| 查找类型 | 含义 | 何时发生 |
|---------|------|---------|
| **LHS**（Left-Hand Side） | 找到变量的**容器**（内存地址），对其进行赋值 | 变量出现在赋值操作左侧 |
| **RHS**（Right-Hand Side） | 找到变量的**值** | 变量出现在赋值操作右侧或表达式中 |

```javascript
let a = b + c

// LHS 查找：a（找到 a 的容器，把值放进去）
// RHS 查找：b（取出 b 的值）
// RHS 查找：c（取出 c 的值）
```

**查找失败的结果**：

| 查找类型 | 失败结果 |
|---------|---------|
| RHS 查找失败 | `ReferenceError: x is not defined` |
| LHS 查找失败（非严格模式） | 自动创建全局变量 |
| LHS 查找失败（严格模式） | `ReferenceError` |

## 常见陷阱与误区

### 陷阱一：循环中的函数引用

```javascript
// ❌ 错误预期：输出 0, 1, 2
for (var i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i)
  }, 0)
}
// 实际输出: 3, 3, 3

// 原因：var i 属于函数作用域，3 个回调共享同一个 i
//       循环结束时 i = 3，所有回调引用的都是 3

// ✅ 修复方式 1：使用 let
for (let i = 0; i < 3; i++) {
  setTimeout(function () {
    console.log(i)
  }, 0)
}
// 输出: 0, 1, 2

// ✅ 修复方式 2：IIFE 创建独立作用域
for (var i = 0; i < 3; i++) {
  (function (j) {
    setTimeout(function () {
      console.log(j)
    }, 0)
  })(i)
}
// 输出: 0, 1, 2
```

### 陷阱二：误以为作用域在调用时确定

```javascript
let name = '全局'

function greet() {
  console.log(name)
}

function runGreet() {
  let name = '局部'
  greet() // 你可能期望输出 "局部"
}

runGreet() // 实际输出: "全局"
// greet 定义在全局作用域，它的作用域链是: greet → Global
// 与在哪里被调用（runGreet 内部）无关
```

### 陷阱三：函数表达式 vs 函数声明的作用域差异

```javascript
// 函数声明 — 提升到当前作用域顶部
console.log(foo()) // "hello" — 可以在声明前调用

function foo() {
  return 'hello'
}

// 函数表达式 — 不会提升函数本身
// console.log(bar()) // TypeError: bar is not a function

var bar = function () {
  return 'world'
}

// 关键区别：
// 函数声明：整个函数体随作用域注册一起提升
// 函数表达式：只有变量声明提升（var bar），赋值留在原位
```

### 陷阱四：IIFE 与作用域隔离

```javascript
// IIFE（立即调用函数表达式）创建独立作用域
// 在 ES6 之前是唯一的块级作用域替代方案

(function () {
  var privateVar = '不会污染全局'
  // ...
})()

// console.log(privateVar) // ReferenceError

// 现代 JavaScript 推荐使用块级作用域替代
{
  let privateVar = '同样不会污染全局'
}
```

---

## 总结

| 特性 | 说明 |
|------|------|
| **定义时机** | 作用域在**代码编写/定义时**确定（词法分析阶段） |
| **查找方向** | 从当前作用域向**外层**逐级查找，直到全局 |
| **核心机制** | 作用域链（Scope Chain）通过 `[[Environment]]` / `[[OuterEnv]]` 引用串联 |
| **闭包基础** | 函数"记住"定义时的词法环境，这就是闭包的原理 |
| **块级补充** | `let`/`const` 增加了块级粒度，但不改变词法作用域的本质 |
| **不可运行时修改** | `eval`/`with` 可以欺骗词法作用域，但应避免使用 |
| **JavaScript 采用** | 词法作用域（静态作用域） |

### 记忆口诀

> **词法作用域 = 写在哪，就访问哪 → 定义决定，调用无关**
