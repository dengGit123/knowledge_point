# 事件流（Event Flow）

## 一、概述

当你点击页面上的一个按钮时，这个"点击"并不是只让按钮知道——它会沿着一条固定的路径**传播**：先从最外层一路向下"找到"这个按钮，再从按钮一路向上"返回"。这条传播路径，就是 **事件流**。

事件流描述的是：**事件从触发开始，到所有监听器执行完毕，所经过的完整传播过程**。它是 DOM 事件机制的核心，理解了它，才能搞懂事件委托、阻止冒泡、`target`/`currentTarget` 等关键概念。

> 📖 [MDN 官方文档 — DOM 事件介绍](https://developer.mozilla.org/zh-CN/docs/Web/API/Document_Object_Model/Events)
>
> 📖 [MDN — Event.eventPhase（事件阶段）](https://developer.mozilla.org/zh-CN/docs/Web/API/Event/eventPhase)
>
> 📖 [DOM 标准 — Dispatching events（事件派发）](https://dom.spec.whatwg.org/#dispatching-events)

## 二、历史背景：为什么会有两种传播方向

理解事件流，最好先知道它的来历。早期浏览器大战时期，两家厂商对"事件怎么传播"有两套截然不同的设计：

| 阵营 | 设计 | 传播方向 |
|------|------|------|
| 网景（Netscape） | **事件捕获** | 从最外层（`window`）向下传播到目标元素 |
| 微软（IE） | **事件冒泡** | 从目标元素向上传播到最外层（`document`） |

后来 W3C 制定标准时，把**两套方案都保留**了，于是有了现在的 **DOM 事件流 = 捕获阶段 + 目标阶段 + 冒泡阶段**。

> 💡 **提示：** 这就是为什么今天一个事件会经历"先捕获、后冒泡"两个方向——它是两家厂商方案的折中结果。

---

## 三、核心概念：事件流的三个阶段

DOM 事件流分为三个阶段，依次发生：

```
                  捕获阶段(从外到内)            冒泡阶段(从内到外)
                  ───────────────▶            ◀───────────────
           ┌─────────────────────────────────────────────────────┐
   window  │ ①                                                   │ ⑨
 document  │   ②                                             ⑧   │
      html │     ③                                       ⑦       │
      body │       ④                                 ⑥           │
     parent│         ⑤  ─────▶ 目标阶段  target ◀─────            │
   (button)│              （事件真正发生在 button 上）              │
           └─────────────────────────────────────────────────────┘
```

### 1. 捕获阶段（Capture Phase）

事件从最外层的 `window` 出发，沿着 DOM 树**逐层向下**传递，直到目标元素的**父节点**为止。

> 注意：标准里捕获阶段**严格不包括目标元素本身**——到达目标时的处理属于"目标阶段"。

### 2. 目标阶段（Target Phase）

事件到达了**真正的目标元素**（即你实际点击的那个元素）。在目标元素上注册的监听器会被触发。

### 3. 冒泡阶段（Bubbling Phase）

事件从目标元素开始，**逐层向上**冒泡，经过父节点，一直到 `window`。

> 💡 **提示：** 大多数时候我们写的事件监听器，都在**冒泡阶段**触发（这也是默认行为）。

## 四、用代码看清三个阶段

### 1. 基础结构

```html
<div id="grandparent">
  <div id="parent">
    <button id="btn">点击我</button>
  </div>
</div>
```

### 2. 给每个元素同时注册「捕获」和「冒泡」监听器

`addEventListener` 的第三个参数 `useCapture`：`true` 表示在**捕获阶段**触发，`false`（默认）表示在**冒泡阶段**触发。

```js
const grandparent = document.getElementById('grandparent')
const parent = document.getElementById('parent')
const btn = document.getElementById('btn')

// 冒泡阶段监听（第三个参数为 false 或省略）
grandparent.addEventListener('click', () => console.log('grandparent 冒泡'))
parent.addEventListener('click', () => console.log('parent 冒泡'))
btn.addEventListener('click', () => console.log('btn 冒泡'))

// 捕获阶段监听（第三个参数为 true）
grandparent.addEventListener('click', () => console.log('grandparent 捕获'), true)
parent.addEventListener('click', () => console.log('parent 捕获'), true)
btn.addEventListener('click', () => console.log('btn 捕获'), true)
```

点击 `<button>` 后，控制台输出顺序：

```text
grandparent 捕获   // ① 捕获阶段：从外到内
parent 捕获        // ② 捕获阶段
btn 冒泡           // ③ 目标阶段（btn 是目标，捕获和冒泡监听器都触发，按注册顺序）
btn 捕获           // ④ 目标阶段
parent 冒泡        // ⑤ 冒泡阶段：从内到外
grandparent 冒泡   // ⑥ 冒泡阶段
```

> ⚠️ **注意（高频考点）：** 在**目标阶段**，目标元素上注册的监听器**不再区分捕获还是冒泡**，而是严格按照**注册顺序**执行。所以上面 `btn 冒泡` 先于 `btn 捕获` 输出——因为"冒泡监听器"先注册。

### 3. 用 `eventPhase` 实时查看阶段

事件对象的 `event.eventPhase` 属性告诉你当前处于哪个阶段：

```js
btn.addEventListener('click', (e) => {
  console.log(e.eventPhase)   // 2
})
```

| 值 | 常量 | 含义 |
|:--:|------|------|
| `0` | `Event.NONE` | 未分发 |
| `1` | `Event.CAPTURING_PHASE` | 捕获阶段 |
| `2` | `Event.AT_TARGET` | 目标阶段 |
| `3` | `Event.BUBBLING_PHASE` | 冒泡阶段 |

## 五、`event.target` 与 `event.currentTarget`

这是事件流里最容易混淆的两个属性，但理解了事件流就豁然开朗：

| 属性 | 含义 | 是否变化 |
|------|------|------|
| `event.target` | **真正触发事件的元素**（事件最初发生在谁身上） | 传播过程中**不变** |
| `event.currentTarget` | **当前正在执行监听器的元素**（即绑定了该监听器的元素） | 随传播**不断变化** |

```html
<div id="parent">
  <button id="btn">点击我</button>
</div>

<script>
  const parent = document.getElementById('parent')

  parent.addEventListener('click', function (e) {
    console.log('target:', e.target.id)            // 'btn'（点的是按钮）
    console.log('currentTarget:', e.currentTarget.id) // 'parent'（监听器绑在父元素上）
    console.log('this:', this.id)                   // 'parent'（this 指向 currentTarget）
  })
</script>
```

点击按钮时，事件冒泡到 `parent`，`parent` 的监听器执行：
- `e.target` 永远是 `btn`（真正被点中的那个）
- `e.currentTarget` 是 `parent`（当前监听器绑在哪，就是谁）

> 💡 **提示：** 在监听器里，`this` 通常指向 `currentTarget`（除非用了箭头函数，箭头函数的 `this` 由外层决定）。

## 六、阻止事件传播与默认行为

事件对象提供了几个控制方法，注意它们的区别：

| 方法 | 作用 | 是否阻止同元素其他监听器 | 是否阻止默认行为 |
|------|------|:---:|:---:|
| `e.stopPropagation()` | 停止事件继续传播（捕获/冒泡中断） | ❌ 否 | ❌ 否 |
| `e.stopImmediatePropagation()` | 停止传播 **+** 阻止同元素剩余同类型监听器 | ✅ 是 | ❌ 否 |
| `e.preventDefault()` | 取消默认行为（如表单提交、链接跳转） | ❌ 否 | ✅ 是 |

### 1. 阻止冒泡（阻止事件继续向上传播）

```js
document.getElementById('btn').addEventListener('click', (e) => {
  e.stopPropagation()   // 事件不再向上冒泡，父元素的 click 监听器不会触发
  console.log('按钮被点击，但不冒泡了')
})
```

> ⚠️ **注意：** `stopPropagation()` 只阻止传播，**不影响当前元素上其他同类型监听器的执行**。如果想让当前元素上其余监听器也不执行，用 `stopImmediatePropagation()`。

### 2. `stopImmediatePropagation()` 的区别

```js
const btn = document.getElementById('btn')

btn.addEventListener('click', (e) => {
  e.stopImmediatePropagation()   // 阻止传播 + 阻止下面这个监听器
  console.log('第一个监听器')
})
btn.addEventListener('click', () => {
  console.log('第二个监听器')     // ❌ 不会执行
})
```

### 3. 阻止默认行为

```js
// 阻止链接跳转
document.querySelector('a').addEventListener('click', (e) => {
  e.preventDefault()   // 链接不会跳转，但事件仍会冒泡
})

// 阻止表单提交
document.querySelector('form').addEventListener('submit', (e) => {
  e.preventDefault()
  // 自定义校验 / AJAX 提交
})
```

> 💡 **提示：** `preventDefault()` 和 `stopPropagation()` 是两件事：前者取消默认动作，后者取消传播。可以只阻止默认行为而不阻止冒泡，反之亦然。

## 七、事件委托（Event Delegation）

事件流最经典的应用就是**事件委托**：利用**事件冒泡**，把本该绑在多个子元素上的监听器，统一绑在它们的**共同父元素**上。

### 1. 为什么需要事件委托

假设有一个长列表，有 100 个 `<li>`：

```html
<ul id="list">
  <li>苹果</li>
  <li>香蕉</li>
  <li>橙子</li>
  <!-- ...100 个 -->
</ul>
```

```js
// ❌ 笨办法：给每个 li 都绑定监听器（100 个监听器，性能差）
document.querySelectorAll('#list li').forEach((li) => {
  li.addEventListener('click', () => console.log(li.textContent))
})
```

```js
// ✅ 事件委托：只给父元素绑定一个监听器（1 个监听器）
document.getElementById('list').addEventListener('click', (e) => {
  // e.target 才是真正被点击的 li
  if (e.target.tagName === 'LI') {
    console.log(e.target.textContent)
  }
})
```

### 2. 事件委托的优势

| 优势 | 说明 |
|------|------|
| **减少监听器数量** | 只在父元素绑一个，节省内存 |
| **动态元素自动生效** | 后来添加的子元素也能响应事件，无需重新绑定 |
| **方便统一管理** | 修改逻辑只需改一处 |

### 3. 事件委托 + `data-*` 属性的常见模式

```html
<div id="toolbar">
  <button data-action="save">保存</button>
  <button data-action="delete">删除</button>
  <button data-action="share">分享</button>
</div>

<script>
  document.getElementById('toolbar').addEventListener('click', (e) => {
    const btn = e.target.closest('button')   // closest 向上查找最近的匹配元素
    if (!btn) return                          // 点击的不是按钮则忽略
    const action = btn.dataset.action
    console.log(`执行：${action}`)             // 执行：save / delete / share
  })
</script>
```

> 💡 **提示：** `closest()` 是事件委托的好搭档——当 `target` 是按钮内部的子元素（如 `<span>`）时，用 `e.target.closest('button')` 能向上找到真正的按钮。

## 八、哪些事件不冒泡

**大部分事件会冒泡**（`click`、`mousedown`、`keydown` 等），但有一类事件**天生不冒泡**，无法用事件委托处理：

| 不冒泡的事件 | 冒泡的替代事件 |
|------|------|
| `focus` | `focusin` |
| `blur` | `focusout` |
| `mouseenter` | `mouseover` |
| `mouseleave` | `mouseout` |
| `load`（资源加载） | `error`（部分情况） |

```js
// ❌ focus 不冒泡，事件委托失效
parent.addEventListener('focus', (e) => { /* 子元素 focus 时不会触发 */ })

// ✅ 用 focusin（会冒泡）实现委托
parent.addEventListener('focusin', (e) => {
  console.log('子元素获得焦点：', e.target)
})
```

> ⚠️ **注意：** 给不冒泡的事件做"事件委托"时，要改用它的"冒泡版本"（如 `focusin`/`focusout`、`mouseover`/`mouseout`）。注意 `mouseover`/`mouseout` 在子元素间移动时也会触发，行为与 `mouseenter`/`mouseleave` 略有不同。

## 九、`addEventListener` 的 options 配置（进阶）

第三个参数除了布尔值（`useCapture`），还可以传一个**配置对象**：

```js
element.addEventListener('click', handler, {
  capture: false,   // 是否在捕获阶段触发
  once: true,       // 是否只触发一次，触发后自动移除
  passive: true     // 声明该监听器永不调用 preventDefault（用于优化滚动性能）
})
```

| 选项 | 作用 | 典型场景 |
|------|------|------|
| `capture` | 在捕获阶段触发 | 需要先于子元素处理事件 |
| `once` | 触发一次后自动移除 | 引导提示、首次点击处理 |
| `passive` | 承诺不调用 `preventDefault` | `touchmove`/`wheel` 优化滚动流畅度 |

### `passive` 为什么能提升性能

移动端滚动时，浏览器遇到 `touchmove` 监听器，必须等监听器执行完，才知道是否调用了 `preventDefault()`（决定要不要阻止滚动）。这会导致滚动卡顿。声明 `passive: true` 告诉浏览器"我不会阻止默认行为"，浏览器就能**立即滚动**而不等待。

```js
// ✅ 推荐做法：移动端滚动监听加 passive
document.addEventListener('touchmove', handleTouch, { passive: true })
```

> 💡 **提示：** 现代浏览器对 `document`/`body` 上的 `touchstart`/`touchmove`/`wheel` 默认就是 `passive`，即使你没写。在这些监听器里调用 `preventDefault()` 会失效并报警告。

## 十、实际应用场景

### 场景 1：弹窗点击遮罩层关闭

点击遮罩层关闭弹窗，但点击弹窗内部不能关闭——靠 `stopPropagation` 实现。

```html
<div id="overlay">
  <div id="modal">弹窗内容</div>
</div>

<script>
  const overlay = document.getElementById('overlay')
  const modal = document.getElementById('modal')

  // 点击遮罩层（冒泡到 overlay）→ 关闭
  overlay.addEventListener('click', () => closeModal())

  // 点击弹窗内部 → 阻止冒泡，不关闭
  modal.addEventListener('click', (e) => e.stopPropagation())
</script>
```

### 场景 2：Vue 的事件修饰符对应原生事件流

Vue 3 的模板修饰符其实就是对事件流控制的封装：

```vue
<template>
  <!-- .stop   → e.stopPropagation() -->
  <button @click.stop="onClick">阻止冒泡</button>

  <!-- .prevent → e.preventDefault() -->
  <a href="/xx" @click.prevent="onClick">阻止默认跳转</a>

  <!-- .capture → 在捕获阶段触发 -->
  <div @click.capture="onClick">捕获阶段触发</div>

  <!-- .once   → 触发一次后解绑 -->
  <button @click.once="onClick">只触发一次</button>

  <!-- .self   → 只有 e.target === e.currentTarget 时才触发 -->
  <div @click.self="onClick">仅点击自身触发</div>
</template>
```

> 💡 **提示：** `.self` 修饰符就是通过判断 `e.target === e.currentTarget` 实现的——这也是理解事件流的价值所在。

### 场景 3：全局搜索框（捕获阶段优先拦截）

想让某个监听器**早于所有冒泡监听器**执行（比如全局快捷键拦截），可以注册在捕获阶段：

```js
// 在捕获阶段监听，确保比目标元素的冒泡监听器先执行
document.addEventListener('keydown', (e) => {
  if (e.key === '/' && document.activeElement.tagName !== 'INPUT') {
    e.preventDefault()
    searchInput.focus()   // 按 / 自动聚焦搜索框（类似 GitHub 搜索）
  }
}, true)   // ← 捕获阶段
```

### 场景 4：动态列表的事件委托（Vue 3）

列表项是动态渲染的，用事件委托避免给每个项绑监听器：

```vue
<script setup>
import { ref } from 'vue'
const items = ref(['苹果', '香蕉', '橙子'])

function handleClick(e) {
  const li = e.target.closest('li')
  if (!li) return
  console.log('点击了：', li.textContent)
}
</script>

<template>
  <!-- 监听器绑在 ul 上，统一处理所有 li 的点击 -->
  <ul @click="handleClick">
    <li v-for="(item, i) in items" :key="i">{{ item }}</li>
  </ul>
</template>
```

## 十一、常见问题与注意事项

### 1. 目标阶段的执行顺序最容易出错

很多人以为"捕获一定先于冒泡"，于是在目标元素上也期望"先捕获后冒泡"。**错！** 在目标元素上，监听器按**注册顺序**执行，与捕获/冒泡无关。（见第四节示例）

### 2. `removeEventListener` 要"对得上"

移除监听器时，必须传入**同一个函数引用**和**相同的 `useCapture`**，否则移除不掉。

```js
const handler = () => console.log('hi')

btn.addEventListener('click', handler)
btn.removeEventListener('click', handler)   // ✅ 能移除

btn.addEventListener('click', () => console.log('hi'))
btn.removeEventListener('click', () => console.log('hi'))  // ❌ 移除不掉（不同的匿名函数引用）
```

> ⚠️ **注意：** 如果用了 `{ once: true }`，监听器执行一次后会**自动移除**，无需手动 `removeEventListener`。

### 3. 阻止冒泡要谨慎

随意 `stopPropagation()` 会破坏事件委托和其他基于冒泡的逻辑。优先用判断 `e.target` 或 `.self` 的方式精确控制，而不是粗暴地阻止冒泡。

### 4. `this` 在箭头函数里不一样

```js
btn.addEventListener('click', function () {
  console.log(this)   // ✅ 指向 btn（currentTarget）
})
btn.addEventListener('click', () => {
  console.log(this)   // ⚠️ 指向外层（不是 btn）—— 箭头函数没有自己的 this
})
```

需要拿到绑定元素时，优先用 `e.currentTarget`，比依赖 `this` 更可靠。

### 5. 事件传播的起点是 `window`

现代标准中，捕获阶段从 `window` 开始（而非旧的 `document`）。冒泡阶段也会一路冒到 `window`。

```
window → document → <html> → <body> → ... → 目标元素（捕获）
目标元素 → ... → <body> → <html> → document → window（冒泡）
```

## 十二、总结

| 知识点 | 一句话记忆 |
|------|------|
| **三个阶段** | 捕获（外→内）→ 目标 → 冒泡（内→外） |
| **默认触发阶段** | 冒泡阶段（`addEventListener` 第三参默认 `false`） |
| **目标阶段顺序** | 目标元素上的监听器按**注册顺序**执行，不区分捕获/冒泡 |
| **target / currentTarget** | `target` 是真正触发的元素（不变）；`currentTarget` 是绑定监听器的元素（随传播变化） |
| **阻止传播** | `stopPropagation()` 停传播，`stopImmediatePropagation()` 还阻止同元素其他监听器 |
| **阻止默认行为** | `preventDefault()`（与阻止传播是两件事） |
| **事件委托** | 利用冒泡，在父元素统一监听子元素事件（配 `closest` + `data-*`） |
| **不冒泡事件** | `focus`/`blur`/`mouseenter`/`mouseleave` 等，委托要用冒泡版本 |
| **passive** | 声明不阻止默认行为，优化 `touchmove`/`wheel` 滚动性能 |

**核心理解**：事件流就是事件在 DOM 树上的"一来一回"——先从 `window` 向下**捕获**到目标，再从目标向上**冒泡**回 `window`。绝大多数业务逻辑都发生在冒泡阶段，事件委托正是对冒泡的巧妙利用；而捕获阶段适合做"全局优先拦截"。搞清 `target`/`currentTarget` 和三阶段的执行顺序，就掌握了事件流的全部精髓。
